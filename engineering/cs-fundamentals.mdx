## Overview

Understanding computer science fundamentals helps you write better code, debug efficiently, and make informed architectural decisions.

## Operating Systems

### Process vs Thread

<CardGroup cols={2}>
  <Card title="Process" icon="window-maximize">
    - Independent execution unit
    - Own memory space
    - Higher overhead to create
    - Inter-process communication needed
  </Card>
  <Card title="Thread" icon="layer-group">
    - Lightweight execution unit
    - Shared memory with parent process
    - Lower overhead to create
    - Direct memory sharing
  </Card>
</CardGroup>

### Memory Management

```
┌─────────────────────────────────────┐

```

### Key Concepts

| Concept | Description | Interview Relevance |
|---------|-------------|---------------------|
| **Deadlock** | Circular wait for resources | High |
| **Race Condition** | Unpredictable behavior from timing | Very High |
| **Mutex/Semaphore** | Synchronization primitives | High |
| **Virtual Memory** | Abstraction over physical memory | Medium |
| **Context Switching** | Saving/restoring process state | Medium |

## Networking

### OSI Model (Simplified)

```
Layer 7: Application  (HTTP, FTP, DNS)
Layer 4: Transport    (TCP, UDP)
Layer 3: Network      (IP, Routing)
Layer 2: Data Link    (Ethernet, MAC)
Layer 1: Physical     (Cables, Signals)
```

### TCP vs UDP

<CardGroup cols={2}>
  <Card title="TCP" icon="handshake">
    - Connection-oriented
    - Reliable delivery
    - Ordered packets
    - Flow control
    - Use: Web, Email, File transfer
  </Card>
  <Card title="UDP" icon="bolt">
    - Connectionless
    - Best-effort delivery
    - No ordering guarantee
    - Low latency
    - Use: Gaming, Streaming, DNS
  </Card>
</CardGroup>

### HTTP Methods & Status Codes

```python
# Common HTTP Methods
GET     # Retrieve resource (idempotent)
POST    # Create resource
PUT     # Update/Replace resource (idempotent)
PATCH   # Partial update
DELETE  # Remove resource (idempotent)

# Status Code Categories
1xx  # Informational
2xx  # Success (200 OK, 201 Created, 204 No Content)
3xx  # Redirection (301 Moved, 304 Not Modified)
4xx  # Client Error (400 Bad Request, 401 Unauthorized, 404 Not Found)
5xx  # Server Error (500 Internal, 502 Bad Gateway, 503 Unavailable)
```

## Database Fundamentals

### ACID Properties

| Property | Description | Example |
|----------|-------------|---------|
| **Atomicity** | All or nothing | Bank transfer: debit AND credit |
| **Consistency** | Valid state transitions | Balance can't be negative |
| **Isolation** | Concurrent transactions don't interfere | Parallel reads/writes |
| **Durability** | Committed data persists | Survives crashes |

### SQL vs NoSQL

<CardGroup cols={2}>
  <Card title="SQL (Relational)" icon="table">
    - Structured schema
    - ACID compliant

    - Vertical scaling
    - Use: Transactions, Analytics
  </Card>
  <Card title="NoSQL" icon="database">
    - Flexible schema
    - BASE (Eventually consistent)
    - Simple queries
    - Horizontal scaling
    - Use: Real-time, Big Data
  </Card>
</CardGroup>

### Indexing

```sql
-- B-Tree Index (default for most DBs)
-- Good for: range queries, sorting, equality
CREATE INDEX idx_user_email ON users(email);

-- Hash Index
-- Good for: equality comparisons only
CREATE INDEX idx_user_id ON users USING HASH(id);

-- Composite Index
-- Order matters! (leftmost prefix rule)
CREATE INDEX idx_user_name ON users(last_name, first_name);
```

## Practice Questions

<AccordionGroup>
  <Accordion title="What happens when you type a URL in a browser?">
    1. DNS resolution to get IP address
    2. TCP connection (3-way handshake)
    3. TLS handshake (if HTTPS)
    4. HTTP request sent
    5. Server processes request
    6. HTTP response received
    7. Browser renders HTML/CSS/JS
  </Accordion>
  
  <Accordion title="How does a database index work?">
    Indexes are data structures (usually B-trees) that maintain sorted references to rows. Instead of scanning all rows (O(n)), the database can use binary search on the index (O(log n)) to find matching rows quickly.
  </Accordion>
  
  <Accordion title="Explain deadlock and how to prevent it">
    Deadlock occurs when processes wait for each other's resources in a circular manner. Prevention strategies:
    - Lock ordering (always acquire locks in same order)
    - Lock timeout
    - Deadlock detection and recovery
    - Avoid nested locks
  </Accordion>
</AccordionGroup>
