# Device Drivers & Hardware I/O

A device driver is a specialized kernel module that acts as a translator between the operating system's generic I/O requests and the hardware's specific register-level interface. Think of drivers as interpreters at the United Nations: the kernel speaks one language ("read 4KB from offset X"), and the hardware speaks another (specific register writes, DMA descriptor formats, interrupt protocols). The driver translates between them so neither side needs to know the other's language.

For a systems engineer, the driver layer is where software meets the raw physics of the machine -- and where the most subtle bugs (memory corruption, data loss, kernel panics) tend to hide.

## 1. Talking to Hardware: Ports and MMIO

The CPU has two primary ways to communicate with a hardware device's registers. This is the most fundamental concept in driver development -- everything else builds on top of these two mechanisms.

### 1.1 Port-Mapped I/O (PMIO)
- **Mechanism**: Uses a separate address space and specialized instructions (`in`, `out` on x86). The CPU has a dedicated I/O bus with its own 16-bit address space, completely separate from memory addresses.
- **Scope**: Limited (usually 64KB on x86). Primarily used for legacy devices like the keyboard controller (port `0x60`), serial ports (`0x3F8`), and the PIC interrupt controller.
- **Why it is fading**: The separate I/O bus does not scale to modern high-bandwidth devices. You cannot do DMA through port I/O.

### 1.2 Memory-Mapped I/O (MMIO)
- **Mechanism**: The hardware registers are mapped directly into the CPU's **Physical Address Space**. The device's control registers appear as ordinary memory addresses.
- **Usage**: To the CPU, reading a status register is just a `MOV` instruction from a specific address. This is elegant because it means the full power of the memory bus (caching, prefetching, burst transfers) is available -- though for device registers, you typically want caching *disabled* (write-through or uncacheable) to ensure every read actually hits the hardware.
- **Barriers**: Because the compiler or CPU might reorder memory accesses, drivers must use **Memory Barriers** (`wmb()`, `rmb()`) to ensure the hardware receives commands in the correct order. Without barriers, the compiler might reorder "write command to device" before "write data to device," causing the device to process garbage.

**Practical tip**: When reading driver code, `ioread32()` / `iowrite32()` are the correct kernel APIs for MMIO -- they handle endianness, barriers, and platform differences. Direct pointer dereferences to MMIO regions are a common bug source because they skip these safety checks.

## 2. Character vs. Block Devices

Linux classifies most hardware into three categories: Character, Block, and Network.

| Feature | Character Devices | Block Devices |
|---------|-------------------|---------------|
| **Access** | Sequential (Stream of bytes) | Random (Addressable blocks) |
| **Buffering** | None (Direct) | Heavy (Page Cache) |
| **Example** | Serial ports, Mice, Keyboards | Hard drives, SSDs, SD cards |
| **Interface** | `open`, `read`, `write` | Block Layer / Request Queues |

### The Major and Minor Numbers
- **Major Number**: Identifies the **Driver** (e.g., all SATA disks use the same major number).
- **Minor Number**: Identifies the **Device Instance** (e.g., `sda` vs `sdb`).

## 3. Interrupt Handling: The Split-Half Model

Hardware is slow; the CPU is fast. To avoid wasting CPU cycles "polling" for data, devices use **Interrupts**. The analogy is a restaurant kitchen bell: the chef (hardware) rings the bell (interrupt) when the food is ready, rather than having the waiter (CPU) stand at the counter asking "is it done yet?" every second.

### 3.1 The Top Half (Interrupt Service Routine)
When an interrupt occurs, the kernel stops everything and runs the **Top Half**. This is the waiter grabbing the plate from the pass -- it must be instant because every other waiter (and customer) is frozen until it is done.

- **Constraint**: Must be extremely fast. Typical target: under 10 microseconds. Every microsecond spent here is a microsecond that *no other interrupt* can be serviced.
- **Rule**: **CANNOT SLEEP**. You cannot call `kmalloc(GFP_KERNEL)`, wait on a mutex, or do anything that might block. The top half runs in "Interrupt Context" -- there is no process backing it, so the scheduler cannot context-switch away from it. If it sleeps, the kernel panics.

### 3.2 The Bottom Half (Deferred Work)
The top half does the bare minimum (acknowledge the hardware, copy data from the device's buffer) and schedules a **Bottom Half** to do the heavy lifting (parse a network packet, update protocol state, deliver data to user space). This split is the key insight: keep the interrupt-disabled window as short as possible, and defer everything else.

| Mechanism | Context | Characteristics |
|-----------|---------|-----------------|
| **Softirqs** | Interrupt | High priority, statically defined (e.g., Networking/Timer). |
| **Tasklets** | Interrupt | Built on Softirqs. Easier to use but only run on one CPU at a time. |
| **Workqueues** | **Process** | Runs in a kernel thread. **CAN SLEEP**. Best for long-running tasks like disk I/O. |

## 4. DMA: Zero-Copy Hardware Transfers

**Direct Memory Access (DMA)** allows hardware to read/write directly to System RAM without involving the CPU. Think of it like a delivery truck that has its own key to the warehouse -- it can unload directly into the right bay without the warehouse manager (CPU) having to move each box by hand.

### 4.1 Descriptor Rings
Modern high-performance devices (like NVMe or 100G NICs) use **Descriptor Rings** -- circular queues in RAM shared between the driver and the device. Think of it like a conveyor belt sushi restaurant: the driver places orders (descriptors) on the belt, the device picks them up and places completed plates (completions) back.

1. The Driver puts a "Descriptor" (a command + a pointer to a buffer in RAM) into the submission ring and advances the tail pointer.
2. The Driver writes to an MMIO register ("Doorbell") to notify the device that new work is available.
3. The Device reads the ring, performs the DMA transfer, and writes a "Completion" entry to a second ring (the completion queue). It then optionally raises an interrupt.

### 4.2 Scatter-Gather I/O
If a data buffer is scattered across non-contiguous physical pages, the driver provides a **Scatterlist**. The DMA controller can then "gather" these pieces into a single transfer, avoiding expensive memory copies.

## 5. The Linux Driver Model

The kernel uses a unified model to track hardware.
- **`struct bus_type`**: Represents the transport (PCI, USB, I2C).
- **`struct device`**: Represents a physical piece of hardware.
- **`struct device_driver`**: Represents the code that can manage a device.

**The Matchmaking**: When a new device is plugged into the PCI bus, the PCI bus core iterates through all registered `device_driver` objects. If a driver's "Vendor ID" and "Device ID" match the hardware, the kernel calls the driver's `probe()` function.

## 6. Driver Development Lifecycle

Understanding drivers in the abstract is useful, but real systems follow a concrete lifecycle from **prototype** to **production**.

### 6.1 Out-of-Tree vs In-Tree Drivers

- **Out-of-tree (external)**:
  - Built as separate modules against the kernel headers.
  - Often shipped by hardware vendors; updated independently from the main kernel.
  - Managed via tools like **DKMS** so they rebuild automatically on kernel upgrades.
- **In-tree (mainline)**:
  - Maintained directly in the kernel source tree.
  - Benefit from continuous testing, refactoring, and API stability guarantees.
  - Preferred for long-term maintenance and security.

### 6.2 Typical Development Flow

1. **Prototype**:
   - Start with a minimal `probe`/`remove` driver that can identify the device and print logs.
   - Use `dmesg`, `lspci`, and `/sys/bus/*` to introspect resources.
2. **Basic I/O and interrupts**:
   - Map MMIO regions, configure basic registers.
   - Wire up interrupt handlers and verify they fire correctly.
3. **DMA and performance**:
   - Implement descriptor rings and DMA paths.
   - Use `perf`, `/proc/interrupts`, and custom tracing to validate throughput and latency.
4. **Error handling and recovery**:
   - Handle hotplug, reset, timeouts, and error paths robustly.
   - Ensure resources are released correctly on failure and module unload.
5. **Upstreaming and maintenance**:
   - Conform to subsystem style (netdev, block, DRM, etc.).
   - Add documentation and selftests where possible.

### 6.3 Common Pitfalls

- Forgetting to handle **suspend/resume** paths, leading to devices that break after sleep.
- Using blocking operations or sleeping in **interrupt context** instead of deferring to workqueues.
- Leaking DMA mappings or neglecting IOMMU interactions, causing subtle memory corruption.

Keep this lifecycle in mind when reading or writing real drivers; the concepts in this chapter map directly onto these stages.

## 6.5 Caveats and Common Pitfalls

<Warning>
**Production traps that surface as kernel panics, data corruption, or ABI breakage:**

1. **Shared interrupts -- the handler must check it is its device.** On legacy PCI (and even some modern systems), an IRQ line is shared among multiple devices. Every shared handler runs on every interrupt; each must read its device's status register and return `IRQ_HANDLED` or `IRQ_NONE`. A handler that always returns `IRQ_HANDLED` "for safety" hides bugs in *other* drivers (the kernel thinks the IRQ was serviced when it was not) and can cause interrupt storms. A handler that does work without checking can corrupt state when another device's IRQ fires.

2. **DMA coherence -- invalidate cache before/after transfer.** On x86 with cache-coherent DMA, the hardware handles this; you barely notice. On ARM, MIPS, and many embedded platforms, DMA is *not* coherent: the device writes to RAM but the CPU cache holds stale copies. You must call `dma_sync_single_for_cpu` (invalidate) before reading device-written data, and `dma_sync_single_for_device` (clean) after CPU writes the device will read. Forgetting these produces intermittent corruption that depends on cache pressure -- the worst class of bug.

3. **`ioctl` is unstructured -- easy to introduce ABI bugs.** The `ioctl` interface is a giant typeless switch on integer commands. There is no schema, no versioning, no automatic backward compatibility. Common bugs: defining an ioctl with `_IOR(x, y, struct foo)` and later adding a field to `struct foo` -- old userspace passes the smaller struct, the kernel reads past the end, returns garbage or panics. Or using a `long` in a 32-bit-vs-64-bit struct, breaking ioctl on compat tasks. Modern alternatives: netlink, sysfs, character device with versioned binary protocols.

4. **Module unload while in use -- refcounts must be tight.** If a driver's code is in use (a thread is executing in the module, an open fd references the device, a timer is queued) and the module is unloaded, the kernel jumps into freed memory -- instant panic. The fix is `try_module_get`/`module_put` plus reference counting on every entry point. Forgetting one path is enough. This is why `rmmod` typically fails for active drivers; some lazy authors paper over it with `MODULE_PUT_AND_RETURN_THREAD` instead of fixing the actual refcount path.
</Warning>

<Tip>
**Solutions and patterns that prevent these failure modes:**

- **Always read your device's status register at the top of the IRQ handler.** Return `IRQ_NONE` if the interrupt is not for you. This is a 5-line invariant that prevents most shared-IRQ misery. Modern kernels prefer MSI/MSI-X (per-device interrupts, no sharing) -- request them via `pci_alloc_irq_vectors` if your device supports it.
- **Use the streaming DMA API correctly: `dma_map_single` plus `dma_sync_single_for_*` around touches.** Or use coherent allocations (`dma_alloc_coherent`) for descriptors and small buffers. Coherent DMA is more expensive (uncached memory on some platforms) but eliminates the synchronization burden. For per-transfer buffers, streaming DMA wins.
- **Prefer netlink, sysfs, configfs, or a versioned character-device protocol over `ioctl` for new interfaces.** If you must use ioctl, define structs with explicit padding, version fields, and reserved bytes. Run `compat_ioctl` checks for 32-on-64 builds. Document the ABI in `Documentation/userspace-api/ioctl/` -- if you cannot write the spec, you cannot ship the interface.
- **Use the device model's `kref` and module reference patterns; do not invent your own.** `request_module`, `try_module_get`, `module_put`, `kref_init`, `kref_get`, `kref_put` -- these are the building blocks. When a release function fires (refcount hits zero), you can finally tear down. Test by `rmmod` while a fuzzer or stress harness is hammering the device.
- **Run KASAN, KCSAN, KFENCE, lockdep, and `slub_debug` on every driver in CI.** These sanitizers catch memory bugs, data races, lock inversions, and slab corruption before they hit production. They slow the kernel, but driver bugs caught at CI cost minutes; the same bugs in production cost weekend escalations.
</Tip>

## 7. Interview Deep Dive: Senior Level

<AccordionGroup>
  <Accordion title="Why must the Top Half of an interrupt handler be non-blocking?">
    The Top Half runs in "Interrupt Context." In this state, there is no associated process (no `task_struct`). If the handler were to sleep (e.g., waiting for a mutex), the scheduler would try to switch to another process, but it wouldn't know how to "return" to the interrupt handler later because it has no stack or state in the scheduler's run-queue. This would lead to an immediate system hang or panic.
  </Accordion>

  <Accordion title="Explain 'Interrupt Storm' and how NAPI mitigates it.">
    An interrupt storm occurs when a device (like a 10Gbps NIC) generates interrupts faster than the CPU can handle them, causing the CPU to spend 100% of its time in the Top Half. **NAPI (New API)** mitigates this by switching from Interrupts to **Polling** under high load. After the first interrupt, the driver disables interrupts and tells the kernel to "poll" the device in a bottom-half until the data buffer is empty.
  </Accordion>

  <Accordion title="What is 'IOMMU' and why is it used?">
    An IOMMU (Input-Output Memory Management Unit) is to a device what an MMU is to a process. it translates **Device Virtual Addresses** to **Host Physical Addresses**. It is used for:
    1. **Security**: Preventing a malicious device from DMA-ing into kernel memory.
    2. **Virtualization**: Allowing a Guest VM to have direct access to a physical PCI device (PCI Passthrough).
  </Accordion>
</AccordionGroup>

## Interview Deep-Dive

<AccordionGroup>
  <Accordion title="Walk me through writing a character device driver">
    **Strong Answer Framework:**

    1. **Frame the interface contract.** A character device exposes byte-stream semantics through `open`, `read`, `write`, `release`, optionally `ioctl`, `mmap`, `poll`. Userspace sees a node in `/dev` (e.g., `/dev/mychar`); the kernel routes calls through the driver's `file_operations`.
    2. **Allocate a major/minor number.** `alloc_chrdev_region(&dev, 0, 1, "mychar")` requests a dynamic major from the kernel. Hardcoding majors is fragile; let the kernel allocate.
    3. **Initialize a `cdev` and add it.** `cdev_init(&my_cdev, &my_fops)` ties the cdev to the `file_operations` struct. `cdev_add(&my_cdev, dev, 1)` makes the device visible to the VFS.
    4. **Create the device node.** Either udev rules + sysfs entry (production), or `device_create(class, NULL, dev, NULL, "mychar")` to spawn `/dev/mychar` automatically.
    5. **Implement `file_operations`.**
       - `open`: increment refcount, allocate per-fd state, return 0 or `-E*`.
       - `read`: copy from internal buffer to user with `copy_to_user`. Handle EOF, partial reads, signal interruption.
       - `write`: validate size, `copy_from_user`, push into hardware or buffer.
       - `release`: free per-fd state, decrement refcount.
       - `unlocked_ioctl`: switch on cmd codes; validate arg pointer; `copy_from_user`/`copy_to_user` for structured args.
    6. **Handle concurrency.** Multiple processes can open the device simultaneously. Use a mutex for ordering, or per-fd state with no shared mutable data.
    7. **Tear down on module exit.** `cdev_del`, `unregister_chrdev_region`, free buffers. Make sure no `release` paths can race with module unload.
    8. **Add error paths.** Every `goto err_X` label, every refcount decrement on failure, every freed allocation.

    **Real-World Example:** The misc device subsystem (`miscdevice`) exists precisely because character devices have so much boilerplate. Drivers like `/dev/null`, `/dev/random`, and `/dev/kvm` use `miscdevice` for a one-call registration. When studying real character drivers, read `drivers/char/misc.c` and `drivers/char/random.c` -- they show the full pattern in a few hundred lines. The 2017 `/dev/random` rewrite by Ted T'so is a great study in how a "simple" character driver hides deep semantics around blocking, entropy estimation, and security.

    <Note>
    **Senior Follow-up 1:** "How does `mmap` change the picture?"

    Implementing `mmap` lets userspace map device buffers directly. The driver provides a `vm_operations_struct` with `fault` (for demand-fill) or maps physical MMIO via `remap_pfn_range`. This is how high-perf drivers (VFIO, DRM, RDMA) avoid `read`/`write` syscalls entirely. The catch: lifecycle management gets harder -- you must keep mappings valid until userspace munmaps, even on driver shutdown.
    </Note>

    <Note>
    **Senior Follow-up 2:** "How do you support `select`/`poll` on your character device?"

    Implement `.poll = my_poll`. The handler calls `poll_wait(file, &my_waitqueue, wait)` to register the wait queue, then returns a bitmask: `EPOLLIN | EPOLLRDNORM` if readable, `EPOLLOUT` if writable, etc. When data arrives, your code calls `wake_up_interruptible(&my_waitqueue)` to wake all waiters. Without proper poll support, userspace must busy-loop, defeating the purpose of having a kernel driver.
    </Note>

    <Note>
    **Senior Follow-up 3:** "What is the difference between `unlocked_ioctl` and the older `ioctl`?"

    Pre-2.6.36, ioctl was called with the BKL (Big Kernel Lock) held -- a global mutex that serialized everything. `unlocked_ioctl` runs without BKL; the driver is responsible for its own locking. The old form is gone. For 32-bit userspace on 64-bit kernels, also implement `compat_ioctl` to translate struct layouts.
    </Note>

    **Common Wrong Answers:**

    1. *"Just register the major and you are done."* Without `cdev_add`, the device exists in the table but VFS will not route to it.
    2. *"Use `printk` to communicate with userspace."* That is a log, not an interface. Use the file_ops.
    3. *"`copy_to_user` is just memcpy."* It is not. It handles user-page faults, validates the pointer is a valid user range, and may sleep. Inside an atomic context, you must use `copy_to_user_inatomic` or pre-fault.

    **Further Reading:**
    - Linux Device Drivers, 3rd ed (Corbet, Rubini, Kroah-Hartman) -- canonical, available free online
    - `drivers/char/misc.c` and `drivers/char/random.c` in the kernel source
    - "Writing your first character device driver" -- Greg Kroah-Hartman talks
  </Accordion>

  <Accordion title="Top half vs bottom half -- why this split?">
    **Strong Answer Framework:**

    1. **State the constraint.** When an interrupt fires, the kernel preempts whatever was running, masks further interrupts on the current CPU, and runs the handler. Until that handler returns, no other interrupt at the same or lower priority can be serviced -- including timer interrupts that drive scheduling. A long handler effectively freezes the CPU.
    2. **Quantify the budget.** Target the top half at single-digit microseconds. At 100Gbps line rate, packets arrive every ~6ns; even a 100us top half means dropping 16,000 packets. Hard real-time systems demand even tighter budgets (sub-microsecond).
    3. **Define the work split.**
       - Top half: acknowledge hardware (clear interrupt status), copy minimal volatile state out of device registers, schedule a bottom half. Done.
       - Bottom half: protocol parsing, packet routing, copy to user buffers, anything that needs memory allocation, anything that might sleep.
    4. **Enumerate the bottom-half mechanisms.**
       - **Softirqs:** statically defined (NET_RX, TIMER, BLOCK), per-CPU, run in interrupt context, cannot sleep. Highest priority.
       - **Tasklets:** built on softirqs (HI_SOFTIRQ/TASKLET_SOFTIRQ). Same constraints, but a given tasklet runs on at most one CPU at a time -- simpler concurrency. Being deprecated in favor of...
       - **Threaded IRQs:** `request_threaded_irq` lets the kernel run the handler in a kernel thread. Can sleep. Costs a context switch but trades off vs latency.
       - **Workqueues:** run in kernel threads, can sleep, allocate with `GFP_KERNEL`, do I/O. The right choice when bottom-half work is heavy or needs memory.
    5. **Show the war story.** A senior engineer once put a 50ms I2C bus transaction in the top half. The system survived in light load and crashed under high IRQ rate from another device because watchdog timer interrupts were starved. Moving the I2C read to a workqueue fixed it. The lesson: anything that can block, must not be in the top half.

    **Real-World Example:** The 2014 introduction of NAPI on Linux explicitly addressed the top-half problem for high-rate NICs. Pre-NAPI, every packet caused an interrupt, and the top half plus softirq were per-packet. Under 10Gbps load this saturated the CPU before useful work could happen. NAPI reframes the problem: after the first interrupt, switch to polling in softirq context, drain the receive ring, and only re-arm interrupts when traffic subsides. The same pattern (the "deferred polling" model) is now in NVMe (CQs) and io_uring.

    <Note>
    **Senior Follow-up 1:** "How do you choose between a tasklet and a workqueue?"

    Tasklet: needs to run soon (next softirq), cannot sleep, small amount of work. Workqueue: long-running, may sleep, may allocate large buffers, may do I/O. Latency-critical and quick: tasklet. Heavy and async-friendly: workqueue. Rule of thumb: if the bottom half could ever take longer than 100us, use a workqueue or threaded IRQ.
    </Note>

    <Note>
    **Senior Follow-up 2:** "What happens if a softirq runs too long?"

    `ksoftirqd/N` (a per-CPU kernel thread) takes over. After a fixed number of softirq invocations from the IRQ exit path, the kernel defers remaining softirq work to ksoftirqd, which runs at normal priority and can be preempted. This prevents a runaway softirq from starving userland entirely. You can see this on heavy network systems -- `ksoftirqd` near top of `top`.
    </Note>

    <Note>
    **Senior Follow-up 3:** "What is `request_threaded_irq` and when do you use it?"

    `request_threaded_irq(irq, hard_handler, thread_handler, flags, name, dev)`. The hard handler runs in interrupt context (top half), returns `IRQ_WAKE_THREAD` to invoke the thread handler. The thread handler runs in a kernel thread, can sleep, can allocate. Use this when your driver needs to do meaningful work on every IRQ but cannot afford to block the CPU.
    </Note>

    **Common Wrong Answers:**

    1. *"Just make the top half faster."* You can shave microseconds, but you cannot make `kmalloc(GFP_KERNEL)` non-sleeping. Some work belongs in a bottom half regardless.
    2. *"Always use workqueues."* For latency-critical work (network RX), a workqueue's context-switch cost is too high. Use a softirq or threaded IRQ.
    3. *"Bottom halves are slower."* They are deferred but may run nanoseconds later. The split is about *where* the work runs, not about speed.

    **Further Reading:**
    - "Linux Kernel Development" by Robert Love, ch. 8 (Bottom Halves and Deferring Work)
    - LWN, "The new way of ignoring interrupts" -- the threaded IRQ history
    - "NAPI: New API" -- Documentation/networking/napi.rst in the kernel source
  </Accordion>

  <Accordion title="DMA -- explain coherent vs streaming, with an example">
    **Strong Answer Framework:**

    1. **Establish the problem.** DMA lets a device read/write RAM directly, but the CPU has caches. Without coordination, the CPU sees stale cached data after a device write, or the device sees stale RAM after a CPU write. The DMA API exists to manage this.
    2. **Coherent (consistent) DMA.** `dma_alloc_coherent(dev, size, &dma_handle, GFP_KERNEL)` returns a pair: a kernel virtual address and a DMA address. The memory is allocated such that CPU and device always see the same data -- on x86, this is normal cached memory plus snoop logic; on ARM/MIPS, it is uncached or specially mapped memory. Use for: descriptor rings, control structures, anything accessed concurrently by both CPU and device.
    3. **Streaming DMA.** `dma_map_single(dev, ptr, size, direction)` takes existing memory and prepares it for one DMA transfer in one direction. After the transfer, `dma_unmap_single` releases. For partial syncs during the transfer (rare), `dma_sync_single_for_cpu` and `dma_sync_single_for_device`. Use for: per-packet network buffers, per-block disk buffers -- anything where the lifetime is bounded and direction is known.
    4. **Pick the right one.**
       - Long-lived, both-direction, small (under 1 page): coherent.
       - Per-IO, one-direction, possibly large: streaming.
       - Coherent is more expensive per access on non-x86 (uncached reads), so do not use it for bulk data.
    5. **Concrete example: NVMe.** The SQ and CQ ring buffers are allocated coherent (driver and controller both poll them constantly). Each I/O's data buffer is mapped streaming -- direction known, lifetime bounded by the I/O.
    6. **Misuse symptoms.** Skipping `dma_map_single`: device DMAs to physical address that may be a virtual page that has been swapped, freed, or remapped -- corruption or fault. Using coherent for a 1MB I/O buffer on ARM: 10x slower because of uncached reads. Forgetting `dma_unmap_single`: leaks IOMMU mappings, eventually exhausts the IOMMU page table.

    **Real-World Example:** The Raspberry Pi 4 ships with a USB controller (the VLI VL805) that initially had a buggy DMA implementation -- on certain access patterns, it would fetch from cache instead of physical memory. The fix involved both firmware updates and Linux driver patches in 2020 to ensure coherent allocations were used in specific paths. This kind of subtle hardware/software co-bug is why driver authors must know the DMA API cold.

    <Note>
    **Senior Follow-up 1:** "What is the IOMMU and how does it interact with DMA mapping?"

    The IOMMU is a hardware unit that translates device-issued DMA addresses to physical addresses, analogous to an MMU for the CPU. With an IOMMU, `dma_map_single` allocates an I/O virtual address and installs a translation. The device sees the I/OVA; the IOMMU translates to the actual physical page. Benefits: device can only DMA to memory the OS authorized, scattered physical pages can be presented contiguously to the device, and a malicious or buggy device cannot corrupt arbitrary memory.
    </Note>

    <Note>
    **Senior Follow-up 2:** "What is `dma_sync_single_for_*` actually doing?"

    Cache maintenance. `for_cpu` (after device write) invalidates the CPU cache line for the buffer so the next CPU read goes to RAM. `for_device` (after CPU write) flushes the cache line so the device reads the latest data. On coherent platforms (x86), these are no-ops. On non-coherent (ARM, MIPS), they are real cache maintenance instructions and matter for correctness.
    </Note>

    <Note>
    **Senior Follow-up 3:** "What is bounce buffering and when does it kick in?"

    If a device cannot DMA to certain physical addresses (e.g., a 32-bit device on a 64-bit machine with high memory), the kernel allocates a bounce buffer in addressable memory, copies to/from it around the DMA. Slow but transparent. Configurable via `dma_set_mask_and_coherent`. Modern devices (PCIe with 64-bit addressing) rarely need bouncing.
    </Note>

    **Common Wrong Answers:**

    1. *"DMA is automatic; the kernel handles caching."* True on x86 if you use the API. On other architectures, you must call sync functions correctly. Always go through the DMA API.
    2. *"Just always use coherent DMA."* Performance disaster on non-x86 platforms. Streaming is the right default for bulk per-IO buffers.
    3. *"Skip `dma_unmap_single` to save time."* IOMMU mappings leak and IOMMU page tables fill, eventually halting the system.

    **Further Reading:**
    - `Documentation/core-api/dma-api.rst` and `dma-api-howto.rst` in the kernel source
    - "DMA on Linux" -- LWN article series
    - PCIe specification, ch. 2 (Transaction Layer) for the wire-level view of DMA
  </Accordion>

  <Accordion title="A kernel module causes random crashes only under memory pressure -- how would you debug?">
    **Strong Answer Framework:**

    1. **Form initial hypotheses.** Memory-pressure-only failures point at: GFP_KERNEL allocation in atomic context (direct reclaim attempts to sleep), unhandled `kmalloc` returning NULL, use-after-free that surfaces when slab recycling speeds up, DMA buffer leaks, or race conditions exposed under increased concurrency.
    2. **Capture the crash.** Get a kernel panic log via netconsole, kdump, or serial console. `dmesg` shows the offending instruction pointer and stack. `addr2line` against the .ko file maps IP to source.
    3. **Enable the sanitizers.**
       - **KASAN:** detects out-of-bounds and use-after-free. Slows the kernel ~3x but invaluable.
       - **KFENCE:** lightweight, production-friendly use-after-free detection (statistical sampling).
       - **CONFIG_SLUB_DEBUG / `slub_debug=FZPU`:** poisons freed slab memory, validates redzones.
       - **lockdep:** reveals deadlock-prone lock ordering.
       - **CONFIG_DEBUG_ATOMIC_SLEEP:** warns when sleeping primitives are called in atomic context.
    4. **Force the failing path.** Use `/sys/kernel/debug/failslab/` and `failpage_alloc` to inject allocation failures. If your error path has a bug, you reproduce in seconds instead of waiting for natural pressure.
    5. **Bisect.** If the crash is regression-induced, `git bisect` between known good and bad kernels. Combine with KASAN to catch the actual bug, not just the symptom.
    6. **Static analysis.** Run `sparse`, `smatch`, and `coccinelle` scripts against the driver. These catch missing error checks, double frees, and locking errors.

    **Real-World Example:** In 2019, a series of crashes on Google's production fleet running a custom NIC driver were traced to a `GFP_KERNEL` allocation in a softirq path. The bug had been latent for months -- under normal load, the allocation always succeeded from the per-CPU cache. Under memory pressure, it triggered direct reclaim, which tried to sleep, which paniced. KASAN did not catch it (no memory corruption); `CONFIG_DEBUG_ATOMIC_SLEEP` would have. The post-incident playbook now requires both for all driver CI.

    <Note>
    **Senior Follow-up 1:** "What is the difference between GFP_KERNEL and GFP_ATOMIC?"

    GFP_KERNEL allows sleeping: direct reclaim, OOM kill, writeback. Use in process context. GFP_ATOMIC cannot sleep -- pulls only from existing free pool plus emergency reserves. Returns NULL immediately on failure. Required in interrupt context and with spinlocks held. The bug that hides in plain sight: code uses GFP_KERNEL "because it usually works" until pressure makes it sleep at the wrong time.
    </Note>

    <Note>
    **Senior Follow-up 2:** "How do you reproduce a memory-pressure bug in CI?"

    `stress-ng --vm 4 --vm-bytes 95% --timeout 5m` saturates RAM. Inside the test container or VM, run your driver workload concurrently. Combine with `failslab` injection to fail allocations randomly. For tail-latency-sensitive paths, also force compaction via `/proc/sys/vm/compact_memory`.
    </Note>

    <Note>
    **Senior Follow-up 3:** "What is the OOM killer's interaction with module memory?"

    Module-allocated memory does not look like userspace memory -- it is kernel memory (slab/vmalloc) and is not directly reclaimable. If a driver leaks, the OOM killer cannot help; eventually the system locks up or panics. This is why driver leaks are uniquely dangerous and why every driver should expose its memory consumption via `/proc` or sysfs counters.
    </Note>

    **Common Wrong Answers:**

    1. *"Add more RAM."* Treats the symptom; the bug recurs at higher scale.
    2. *"Always use GFP_ATOMIC to be safe."* GFP_ATOMIC is for atomic context only; in process context it makes failures more likely (smaller pool to draw from).
    3. *"Just check if `kmalloc` returned NULL."* Necessary but not sufficient -- the error path itself often has bugs (double free, missed unwind).
