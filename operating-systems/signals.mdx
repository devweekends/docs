# Signal Architecture

Signals are the Unix equivalent of hardware interrupts for user-space processes. While they seem simple (send a signal, catch it in a handler), their internal implementation is a masterpiece of kernel-to-user-space context switching.

Think of a signal like a fire alarm in an office building. You are at your desk working (executing instructions). The alarm goes off (signal delivered). You stop what you are doing, save your place (signal frame on stack), follow the evacuation procedure (signal handler), and then return to your desk and resume exactly where you left off (rt_sigreturn restores context). The complexity arises because the "alarm" can fire at any instant -- even in the middle of writing a sentence.

## 1. How Signals Are Delivered
When a process receives a signal (e.g., `SIGSEGV` or `SIGINT`), the kernel doesn't just call a function. It must force the user process to execute a handler at an **arbitrary point in its execution**.

### The Delivery Flow
1.  **Generation**: A process calls `kill()` or the hardware triggers an exception (e.g., divide-by-zero generates `SIGFPE`, invalid memory access generates `SIGSEGV`).
2.  **Pending State**: The kernel marks the signal as "pending" in the process's `task_struct`. For standard signals (1-31), only one instance is recorded -- if three `SIGINT` arrive before the first is delivered, the process sees one. For real-time signals (32-64), the kernel queues each instance.
3.  **Check on Return**: The kernel checks for pending signals every time it is about to return to user-space (e.g., after a syscall or an interrupt). This is the critical detail: signals are never delivered in the middle of kernel execution. The kernel finishes its work, then inspects the pending mask right before the `sysret`/`iret` instruction.
4.  **Signal Delivery**: If a signal is pending and not masked, the kernel initiates the delivery by rewriting the user-space stack and instruction pointer.

```c
// Simplified from kernel/signal.c -- the "check on return" logic
static void do_signal(struct pt_regs *regs) {
    struct ksignal ksig;

    // Dequeue the next pending, unblocked signal
    if (get_signal(&ksig)) {
        // Rewrite the user stack to execute the handler
        handle_signal(&ksig, regs);
        return;
    }

    // No signal pending -- restore original user context and return
    restore_saved_sigmask();
}
```

## 2. Signal Frames & The User Stack
The most complex part of signal delivery is managing the user stack. The kernel cannot execute the signal handler in kernel mode; it must do it in user mode.

### Setting up the Frame
1.  **Save User Context**: The kernel saves the current user registers (RIP, RSP, RFLAGS, etc.) onto the **User Stack**. This data is called a **Signal Frame**.
2.  **Modify RIP**: The kernel changes the saved Instruction Pointer (RIP) in the kernel-trap frame to point to the start of the user's signal handler.
3.  **Return to User**: The kernel performs a "return from interrupt" (e.g., `sysret` or `iret`). The CPU resumes execution at the signal handler.

### The Return: `rt_sigreturn`
When the signal handler finishes, how does the process go back to what it was doing?
-   The kernel places a small piece of code (the **Restorer**) on the stack or in a fixed location.
-   The signal handler "returns" to this restorer.
-   The restorer calls the `rt_sigreturn` system call.
-   The kernel sees the `rt_sigreturn`, reads the saved context from the signal frame on the user stack, restores the registers, and resumes the original program.

## 3. Async-Signal Safety
Signals are **asynchronous**. They can arrive while your program is in the middle of a `malloc()` or a `printf()`.

### The Problem
If your program is inside `malloc()` (which holds a global lock) and a signal arrives whose handler also calls `malloc()`, you will **deadlock** with yourself.

### The Rules
-   Only call **Async-Signal Safe** functions in a signal handler (e.g., `write`, `read`, `_exit`). The full list is defined by POSIX and includes about 120 functions -- when in doubt, check `man 7 signal-safety`.
-   Never use standard I/O (`printf`, `fopen`) or memory allocation (`malloc`, `free`). These hold internal locks that your main code may already be holding when the signal arrives.
-   Common practice: Set a global `volatile sig_atomic_t` flag and handle the logic in your main loop. This is the "self-pipe trick" generalized: the handler does the absolute minimum (set a flag or write a byte to a pipe), and the main loop does the real work.

**Practical tip**: The `volatile` keyword alone is not enough. You must use `sig_atomic_t` because the C standard only guarantees atomic access for this type in signal contexts. Using `volatile int` on a platform where `int` writes are not atomic would create a data race between the handler and the main thread.

## 4. Signal Stacks: `sigaltstack`
If a process suffers a **Stack Overflow**, it receives a `SIGSEGV`. However, to handle that signal, the kernel needs to push a signal frame onto the stack. If the stack is full, the kernel cannot push the frame, and the process is killed instantly -- a catch-22.
-   *Solution*: `sigaltstack()`. This allows a process to define a separate, dedicated stack (typically 8-32 KB) for signal handlers. The handler then runs on a completely different memory region, independent of the overflowed stack.

**Practical tip**: This is how crash-reporting libraries (Google Breakpad, Sentry Native) capture stack traces even on stack overflow. They register a `SIGSEGV` handler with `SA_ONSTACK` and a pre-allocated alternate stack, allowing them to dump diagnostics before the process dies.

```c
// Setting up an alternate signal stack
stack_t ss;
ss.ss_sp = malloc(SIGSTKSZ);   // SIGSTKSZ is typically 8192 bytes
ss.ss_size = SIGSTKSZ;
ss.ss_flags = 0;
sigaltstack(&ss, NULL);

// Register handler with SA_ONSTACK so it uses the alternate stack
struct sigaction sa;
sa.sa_handler = crash_handler;
sa.sa_flags = SA_ONSTACK;       // Use alternate stack for this handler
sigemptyset(&sa.sa_mask);
sigaction(SIGSEGV, &sa, NULL);
```

## 5. Signals vs. Threads
In a multi-threaded process, signals interact with the threading model in subtle ways. Think of it like a phone call to an office: a call to a specific extension (synchronous signal) reaches one person, while a call to the main line (asynchronous signal) is routed to whoever picks up first.

-   **Synchronous / Thread-directed Signals**: Signals caused by a specific thread's own execution (e.g., `SIGSEGV` from a bad pointer dereference, `SIGFPE` from a divide-by-zero, `SIGILL` from an invalid instruction) are delivered only to that thread. The thread caused the fault, so only it can meaningfully handle it.
-   **Asynchronous / Process-wide Signals**: Signals sent from outside (e.g., `SIGINT` from Ctrl+C, `SIGTERM` from `kill`) are delivered to **one arbitrary thread** that does not have the signal masked. The kernel picks the first eligible thread it finds.
-   **Thread-targeted Signals**: `pthread_kill(thread, sig)` sends a signal to a specific thread regardless of whether it is synchronous or asynchronous.

**Practical tip**: The standard pattern for multi-threaded daemons is to block all signals in every worker thread using `pthread_sigmask`, then dedicate one thread to call `sigwait()` (or use `signalfd` with `epoll`). This eliminates the unpredictability of "which thread gets the signal?" and avoids async-signal-safety concerns entirely in worker code.

```c
// Worker thread setup: block all signals so the dedicated thread handles them
void *worker(void *arg) {
    sigset_t all;
    sigfillset(&all);
    pthread_sigmask(SIG_BLOCK, &all, NULL);  // Block everything in this thread

    // ... do work safely without worrying about signal interruption ...
    return NULL;
}

// Dedicated signal-handling thread
void *signal_thread(void *arg) {
    sigset_t wait_set;
    sigemptyset(&wait_set);
    sigaddset(&wait_set, SIGINT);
    sigaddset(&wait_set, SIGTERM);

    int sig;
    // sigwait blocks until one of the masked signals arrives
    sigwait(&wait_set, &sig);
    printf("Caught signal %d, initiating graceful shutdown\n", sig);
    // ... coordinate shutdown with worker threads ...
    return NULL;
}
```

## 6. Modern Alternative: `signalfd`
Handling signals is painful due to async-safety issues. Linux provides `signalfd`, which allows you to receive signals as data from a file descriptor.
-   You mask the signals normally (block them from being delivered as interrupts).
-   You `read()` from the signalfd -- this returns a `struct signalfd_siginfo` containing the signal number, sender PID, and other metadata.
-   This converts the asynchronous signal into a **synchronous event**, making it safe to use with `epoll` and standard libraries.

The result is that signals become just another file descriptor event, composable with your existing event loop. No more worrying about async-signal safety, no more self-pipe tricks.

```c
// Block SIGINT and SIGTERM from default delivery
sigset_t mask;
sigemptyset(&mask);
sigaddset(&mask, SIGINT);
sigaddset(&mask, SIGTERM);
sigprocmask(SIG_BLOCK, &mask, NULL);

// Create a file descriptor that delivers these signals as readable data
int sfd = signalfd(-1, &mask, SFD_NONBLOCK);

// Now add sfd to your epoll set alongside sockets, timers, etc.
// When SIGINT arrives, epoll_wait returns sfd as readable.
struct signalfd_siginfo info;
read(sfd, &info, sizeof(info));
printf("Received signal %d from PID %d\n", info.ssi_signo, info.ssi_pid);
```

## 7. Interview Deep Dive

<AccordionGroup>
  <Accordion title="How does the kernel deliver a signal to a user-space process?">
    The kernel does not "call" the handler. It rewrites the user stack:
    
    1. Saves the full register context (RIP, RSP, RFLAGS, general-purpose registers) onto the user stack as a **signal frame**.
    2. Overwrites the saved RIP in the trap frame to point to the user's signal handler.
    3. Returns to user space via `sysret`/`iret`. The CPU resumes at the handler address.
    4. When the handler returns, it falls through to a **restorer** stub that calls `rt_sigreturn`.
    5. The kernel reads the saved context from the signal frame, restores all registers, and resumes the original code path.
    
    A senior engineer would note that this is also a security-sensitive area: if an attacker can corrupt the signal frame on the stack, they can hijack `rt_sigreturn` to set arbitrary register values (a technique known as SROP -- Sigreturn-Oriented Programming).
  </Accordion>

  <Accordion title="What is async-signal safety and why does it matter?">
    Async-signal safety means a function can be safely called from within a signal handler. The danger: a signal can arrive while the main code is holding an internal lock (e.g., inside `malloc`). If the handler also calls `malloc`, the thread deadlocks against itself -- it is waiting for a lock it already holds, but it cannot return to the main code to release it.
    
    POSIX defines about 120 async-signal-safe functions (e.g., `write`, `read`, `_exit`, `signal`). Everything else -- `printf`, `malloc`, `syslog`, any function that takes a lock -- is unsafe.
    
    The production-grade pattern: do the minimum in the handler (set a `volatile sig_atomic_t` flag or write a byte to a self-pipe), then handle the real logic in your main event loop.
  </Accordion>

  <Accordion title="Compare signalfd to traditional signal handlers. When would you choose each?">
    **Traditional handlers** are necessary when:
    - You need to handle synchronous signals like `SIGSEGV` (signalfd cannot catch these).
    - You are writing a library that must not impose an event-loop architecture on the caller.
    
    **signalfd** is better when:
    - You already have an event loop (`epoll`/`select`/`poll`).
    - You want to avoid async-signal-safety headaches entirely.
    - You need to handle process-lifecycle signals (`SIGTERM`, `SIGCHLD`, `SIGHUP`).
    
    In modern service architectures, signalfd (or the equivalent `signal_fd` in Go via `os/signal.Notify`) is the standard approach. It turns an asynchronous interrupt into a synchronous, pollable event -- composable with all other I/O.
  </Accordion>
</AccordionGroup>

## Summary for Senior Engineers
-   **Signal handlers are not normal functions**: They run on a modified stack with a kernel-injected return path.
-   **The `rt_sigreturn` syscall** is the only way to safely exit a signal handler and restore the machine state.
-   **Async-safety** is non-negotiable. If you ignore it, your program will flake out under load, producing deadlocks that only reproduce in production.
-   **Use `signalfd`** in modern applications to avoid the complexities of asynchronous handlers.
-   **Block signals in worker threads** and handle them in a dedicated thread or event loop -- this is the pattern used by every serious multi-threaded daemon.
-   **Always set up `sigaltstack`** if you need to handle `SIGSEGV` for crash reporting, because the default stack may already be overflowed.

## Interview Deep-Dive

<AccordionGroup>
  <Accordion title="A multi-threaded server occasionally hangs under load. You suspect a signal handler is calling malloc(), which deadlocks with the main thread's malloc(). How would you diagnose this, and what is the correct way to handle signals in a multi-threaded application?">
    **Strong Answer:**

    This is a classic async-signal-safety violation, and it is one of the most insidious bugs because it only manifests under specific timing conditions -- the signal must arrive exactly while the main thread holds the heap lock.

    Diagnosis:
    - **Attach GDB to the hung process**: `gdb -p PID`, then `thread apply all bt`. Look for two stack frames on the same thread stuck in malloc/free-related functions. One frame will be the normal malloc call, and above it the signal handler's malloc call waiting for the same lock. Since signals run on the interrupted thread's stack, both frames belong to the same OS thread, creating a self-deadlock.
    - **strace -e signal**: Run the server under strace to see which signals are being delivered and when.
    - **Core dump analysis**: If the process was killed, examine the core dump for a thread whose stack shows a signal handler frame interleaved with a malloc frame.

    The correct approach for multi-threaded signal handling:

    1. **Block all signals in all worker threads**: At thread creation, use `pthread_sigmask(SIG_BLOCK, ...)` to block all signals (except synchronous ones like SIGSEGV/SIGBUS that cannot be meaningfully blocked).
    2. **Dedicate one thread for signal handling**: Create a thread that calls `sigwait()` in a loop. This converts asynchronous signals into synchronous events -- `sigwait` returns the signal number, and the handler thread can safely call any function because it is running in a normal execution context, not a signal context.
    3. **Alternative: signalfd**: On Linux, use `signalfd()` to receive signals as file descriptor events. Integrate with your epoll loop. This is even better for event-driven architectures because signals become just another event source.

    If you absolutely must use a signal handler (e.g., for SIGSEGV crash handling), the handler must only call async-signal-safe functions. The safest pattern: set a `volatile sig_atomic_t` flag in the handler and check it in the main loop.

    **Follow-up: Why can SIGSEGV not be handled via sigwait() or signalfd()?**

    SIGSEGV is a synchronous signal -- it is generated by the specific instruction that caused the fault, and the kernel delivers it to the specific thread that executed that instruction. Blocking SIGSEGV and waiting for it with sigwait in a different thread does not work because the faulting thread is stuck at the faulting instruction and cannot proceed. The signal must be handled on the faulting thread's stack so the handler can either fix the condition (e.g., map the missing page for a JIT compiler) or terminate gracefully. This is fundamentally different from asynchronous signals like SIGINT, which are not tied to a specific instruction.
  </Accordion>

  <Accordion title="Explain the signal frame mechanism. When the kernel delivers a signal, how does it ensure the process returns to exactly where it was after the handler completes?">
    **Strong Answer:**

    Signal delivery is one of the most elegant kernel-to-user-space interactions in Unix. The mechanism is essentially: the kernel hijacks the process's return to user space, forces it to execute the handler, and then the handler "returns" through a kernel-provided trampoline that restores the original state.

    Step by step:
    - **Signal becomes pending**: When a signal is generated (via `kill()`, hardware exception, etc.), the kernel sets a bit in the target process's `pending` signal mask.
    - **Check on return to user space**: Every time the kernel is about to return to user space (after a syscall, interrupt, or page fault), it checks for pending unmasked signals via `do_signal()`.
    - **Build the signal frame**: The kernel saves the entire user-space register context (all general-purpose registers, RFLAGS, RIP, RSP, FPU/SSE state) onto the user's stack as a `struct rt_sigframe`. This frame also contains a small piece of code: the "restorer" trampoline (or a pointer to it in the vDSO).
    - **Modify the return context**: The kernel changes the return RIP to point to the user's signal handler function. The return RSP is adjusted to account for the signal frame. Arguments are set up per the calling convention (signal number in RDI, `siginfo_t` pointer in RSI, `ucontext_t` in RDX for `SA_SIGINFO` handlers).
    - **Return to user space**: The kernel does its normal return (SYSRET or IRET). The CPU resumes in user space, but at the signal handler instead of the original code.
    - **Handler executes and returns**: The handler's `ret` instruction transfers control to the restorer trampoline.
    - **Restorer calls rt_sigreturn**: The trampoline code executes the `rt_sigreturn` syscall. The kernel reads the `rt_sigframe` from the user stack, validates it, and restores all registers. The process resumes exactly where it was when the signal arrived.

    The security implication: if an attacker can corrupt the signal frame on the stack, they can forge arbitrary register values via `rt_sigreturn` -- a technique called SROP (Sigreturn-Oriented Programming). Modern kernels add cookies/canaries to the signal frame to detect tampering.

    **Follow-up: What happens if a second signal arrives while the first signal's handler is executing?**

    By default, the kernel blocks the signal being handled (same signal number) during handler execution -- this prevents recursive handler invocation. Other signals are delivered normally, which can interrupt the running handler. If the second signal's handler also modifies shared state, you get nested signal frames on the stack. Each handler returns through its own `rt_sigreturn`, unwinding in LIFO order. The `SA_NODEFER` flag disables the automatic blocking, allowing recursive delivery of the same signal -- this is almost never what you want and leads to stack overflow under signal storms.
  </Accordion>

  <Accordion title="Your application uses signalfd to handle SIGTERM for graceful shutdown. During testing, you notice that SIGTERM sometimes arrives but the signalfd never becomes readable. What could cause this?">
    **Strong Answer:**

    This is a subtle bug that I have seen in production, and it comes down to the signal mask configuration.

    The most likely cause: **the signal is not blocked in all threads.** For signalfd to work correctly, the signal must be blocked in every thread. If any thread does not have SIGTERM blocked, the kernel delivers the signal to that thread using the default disposition (terminate the process) instead of queuing it for the signalfd. The signalfd never sees it because the signal was already consumed.

    Common scenarios where this happens:
    - **Thread created before signal mask setup**: If you create threads before calling `pthread_sigmask(SIG_BLOCK, ...)`, those threads inherit the original (unblocked) signal mask. The signal gets delivered to one of those early threads.
    - **Library spawns threads**: Some libraries (glibc's DNS resolver, OpenSSL, logging frameworks) create helper threads that inherit the signal mask from the creating thread. If you blocked SIGTERM in your main thread after the library initialized, the library's threads have the old mask.
    - **Race condition during setup**: If SIGTERM arrives between the `signalfd()` call and the `pthread_sigmask()` call, the signal is delivered with default behavior.

    The fix: block signals as the very first action in `main()`, before creating any threads or initializing any libraries:

    ```c
    int main() {
        sigset_t mask;
        sigemptyset(&mask);
        sigaddset(&mask, SIGTERM);
        sigaddset(&mask, SIGINT);
        pthread_sigmask(SIG_BLOCK, &mask, NULL);  // Block FIRST
        int sfd = signalfd(-1, &mask, SFD_NONBLOCK);
        // Now create threads, initialize libraries, etc.
    }
    ```

    **Follow-up: Can signalfd be used across a fork()? What about after exec()?**

    The signalfd file descriptor is inherited across fork() and refers to the same kernel signalfd object. Both parent and child can read signals from it, but a given signal is consumed by whichever reader gets it first. The child also inherits the signal mask. After exec(), the signal mask is preserved but the signalfd file descriptor is closed if it was marked `O_CLOEXEC` (which `SFD_CLOEXEC` sets). Without `O_CLOEXEC`, the fd survives exec but the new program likely does not know about it. The practical advice: for services that fork workers, each worker should create its own signalfd rather than sharing the parent's.
  </Accordion>
</AccordionGroup>

## Production Caveats &amp; Common Pitfalls

Signals are deceptively simple to call but treacherous to use correctly. Here are four traps senior engineers must internalize.

<Warning>
**Caveat 1: The async-signal-safe function list is tiny -- printf is NOT safe**

POSIX defines exactly which functions can be safely called from a signal handler. The list (`man 7 signal-safety`) has about 120 functions. `printf`, `fprintf`, `malloc`, `free`, `syslog`, `dlopen`, almost every standard library function -- none are on it. The reason: these functions hold internal locks. If a signal arrives while the main thread holds the malloc lock, and the handler calls printf (which calls malloc internally to grow stdio buffers), the thread deadlocks against itself. This is the most common signal bug in production C code.
</Warning>

<Tip>
**The fix:** Do the absolute minimum in the handler. The "self-pipe trick" is canonical: write a single byte to a pipe FD, return. The main loop's epoll/select reads the pipe and does real work. Even better: use `signalfd` so signals come through your event loop directly, eliminating async-signal concerns entirely. If you must format output from a handler, use `write(2)` with a pre-formatted buffer -- never printf. For numeric formatting in the handler, write your own minimal int-to-string routine (it is 10 lines) rather than calling snprintf.
</Tip>

<Warning>
**Caveat 2: EINTR -- most syscalls return early on signal, must retry**

When a signal arrives during a slow syscall (read, write, accept, connect, select), most syscalls return -1 with errno = EINTR. This is the kernel saying "I interrupted your syscall to deliver a signal; you decide whether to retry." Naive code that doesn't check for EINTR sees random failures whenever a signal fires. The classic example: a server that runs fine until SIGCHLD arrives (because a child exited), at which point its `accept()` returns EINTR and the server thinks the listening socket failed.
</Warning>

<Tip>
**The fix:** Wrap blocking syscalls in a retry loop: `while ((n = read(fd, buf, len)) == -1 &amp;&amp; errno == EINTR) {}`. Or, register handlers with `SA_RESTART` flag in `sigaction` -- the kernel will automatically restart most syscalls (read, write, accept). Some syscalls (`select`, `pselect`, `nanosleep`) cannot be auto-restarted because they have a timeout that would be wrong; for these, always handle EINTR explicitly. In Go and Rust, the runtime handles EINTR for you. In Python, the interpreter retries until 3.5 -- after which signal-handling was reformed and most stdlib I/O is EINTR-safe.
</Tip>

<Warning>
**Caveat 3: Signal masks inherited by threads -- pthread_sigmask vs sigprocmask**

In a multi-threaded program, each thread has its own signal mask. `sigprocmask` is undefined behavior in multi-threaded code -- you must use `pthread_sigmask`. Worse, when you create threads, they inherit the signal mask from the *creating* thread. If your main thread blocks SIGTERM at startup before creating workers, all workers inherit the block, and your sigwait/signalfd thread receives SIGTERM correctly. But if you create threads first and block SIGTERM later, the workers have an inconsistent mask and SIGTERM may be delivered to a worker that does not expect it.
</Warning>

<Tip>
**The fix:** Block all signals in `main()` before creating any threads: `sigfillset(&amp;all); pthread_sigmask(SIG_BLOCK, &amp;all, NULL);`. Then create a dedicated signal-handling thread that calls `sigwait()` or reads from a `signalfd`. All worker threads inherit the "everything blocked" mask. Synchronous signals (SIGSEGV, SIGFPE, SIGBUS) cannot be blocked meaningfully -- they are delivered to the thread that caused the fault regardless of mask. Always use `pthread_sigmask` in multi-threaded code; `sigprocmask` works in single-threaded code only.
</Tip>

<Warning>
**Caveat 4: signalfd / sigwait for synchronous handling**

Traditional signal handlers run at unpredictable times in the middle of arbitrary code. They are async-signal-restricted. They can interrupt syscalls. They may be delivered to any thread. signalfd flips this: signals become readable events on a file descriptor. sigwait blocks one thread waiting for a signal. Both convert async signals into synchronous events you can handle with normal code -- no async-signal restrictions, no EINTR, no thread targeting surprises.
</Warning>

<Tip>
**The fix:** For new code, prefer `signalfd` integrated with your event loop. Block signals globally, create a signalfd for the signals you care about, add it to epoll. When a signal arrives, epoll wakes up, you read the signalfd, and dispatch synchronously. For sigwait, dedicate one thread that calls `sigwait` in a loop and dispatches into normal code. Both approaches eliminate async-signal-safety as a concern. Caveat: synchronous signals (SIGSEGV) cannot be handled this way -- the faulting thread has no way to proceed past the fault, so you still need a traditional handler with `sigaltstack` for crash reporting.
</Tip>

## Senior-Level Interview Questions

<AccordionGroup>
  <Accordion title="Implement a signal-safe logger. What can you call inside the handler, and how do you format output?">
    **Strong Answer Framework:**

    1. **State the constraint**: Inside a signal handler, you can only call async-signal-safe functions. The list excludes printf, malloc, snprintf, and most stdio. The safe set includes: write, read, _exit, sigaction, kill, getpid, time, localtime_r is NOT safe (it touches locale state); only `time()` itself is.
    2. **Sketch the design**: Pre-allocate a fixed-size buffer at process start (no malloc in handler). Write a tiny format function that handles only the formatters you need (typically %s, %d, %x). Use `write(STDERR_FILENO, buf, len)` for output. For timestamps, use `time(NULL)` and a hand-coded epoch-to-string conversion.
    3. **Implement the int-to-string**: Standard C trick -- write digits backwards into a buffer, then reverse. About 10 lines. Handles negatives by checking sign first.
    4. **Discuss alternatives**: For most use cases, the better answer is "don't log from a signal handler." Use the self-pipe trick: handler writes one byte to a pipe; main loop reads the pipe and uses normal logging. signalfd is the cleanest -- signals become a normal event source, you log normally from your event handler. The "implement signal-safe logger" exercise is worth doing once for understanding, then never again.

    **Real-World Example:**

    Google's Breakpad and Sentry's native crash handler both implement signal-safe stack capture. Breakpad's `LinuxDumper` pre-allocates a buffer at startup, uses `getcontext()` (one of the few safe context-snapshot functions), and writes the minidump via direct write(2) calls. They explicitly avoid malloc by using a pre-allocated arena. The code (`src/client/linux/handler/exception_handler.cc` in the Breakpad repo) is a fascinating read on what async-signal-safe code looks like in production.

    <Note>
    **Senior follow-up 1:** Can you call write() to a pipe in a handler if the pipe might be full? Yes, write is async-signal-safe -- but it can return EAGAIN if the pipe is non-blocking and full, or it can return early with EINTR. You must check the return value. For a self-pipe, set it non-blocking and ignore EAGAIN; the byte just signals "something happened" so a missed byte is fine if there are already pending notifications.
    </Note>

    <Note>
    **Senior follow-up 2:** What about logging from a SIGSEGV handler when the stack itself may be corrupted? Two things: register the handler with `SA_ONSTACK` and provide a separate stack via `sigaltstack`. The handler runs on the alternate stack, not the corrupted main stack. This is how crash reporters survive stack overflow -- `SIGSEGV` due to stack overflow normally cannot be handled because there is no stack space for the frame; with sigaltstack, there is.
    </Note>

    <Note>
    **Senior follow-up 3:** Why is `localtime_r` not async-signal-safe even though it has _r in the name? The _r means thread-safe (reentrant), not async-signal-safe. localtime_r reads timezone state which may be modified by `setenv("TZ", ...)`. If a signal arrives during setenv, localtime_r in the handler sees inconsistent state. For signal-safe time formatting, use only `time()` plus your own arithmetic, or pre-compute the offset at startup.
    </Note>

    **Common Wrong Answers:**
    - "Use snprintf with a fixed buffer in the handler." Wrong: snprintf is not async-signal-safe.
    - "Lock a mutex around the logger." Wrong: mutex acquisition is not async-signal-safe; you can deadlock.
    - "It is fine because it is a 'simple' printf." Wrong: stdio holds a lock that the main thread may be holding.

    **Further Reading:**
    - `man 7 signal-safety` -- the canonical list.
    - "Signal Safety: AS-Safe" in glibc manual -- gnu.org/software/libc/manual/.
    - Breakpad source -- chromium.googlesource.com/breakpad/.
  </Accordion>

  <Accordion title="What does SIGKILL do that SIGTERM doesn't? Why is SIGKILL uncatchable?">
    **Strong Answer Framework:**

    1. **State the difference**: SIGTERM (signal 15) is the standard "please terminate" -- it can be caught, blocked, or ignored. The application can do graceful cleanup. SIGKILL (signal 9) is the kernel's nuclear option -- it cannot be caught, blocked, or ignored. The kernel terminates the process immediately without giving it a chance to run any code.
    2. **Explain why SIGKILL is uncatchable**: The kernel enforces this in `do_signal()`. When SIGKILL is pending, the kernel calls `do_group_exit(SIGKILL)` directly without checking for installed handlers. If userspace could catch SIGKILL, there would be no way to forcibly terminate a stuck process. This is an OS-level guarantee, not a userspace policy.
    3. **State the limit of SIGKILL**: SIGKILL kills the *process* immediately, but the kernel still has to wait for any in-progress system calls to complete. A process stuck in TASK_UNINTERRUPTIBLE (D state) does not die from SIGKILL -- the kernel waits for the I/O to complete first. This is why "kill -9 a stuck NFS process" doesn't work.
    4. **Discuss the graceful shutdown pattern**: Standard practice: send SIGTERM first, wait a grace period (e.g., 30 seconds), then SIGKILL if the process is still running. Kubernetes does this with `terminationGracePeriodSeconds`. systemd does the same with `TimeoutStopSec`.

    **Real-World Example:**

    Kubernetes' default `terminationGracePeriodSeconds` is 30. When a pod is deleted, the kubelet sends SIGTERM, waits 30s, then sends SIGKILL. In 2018, a high-profile incident at HBO during the Game of Thrones finale: a database container was stuck in D state due to a slow EBS volume, SIGTERM did nothing, SIGKILL did nothing (D state ignored signals), and the kubelet logged "container failed to stop" while the database held a critical lock. The fix was to fail over to a healthy node; the underlying issue was the EBS volume's I/O credit exhaustion.

    <Note>
    **Senior follow-up 1:** What is SIGSTOP and how is it different from SIGKILL? SIGSTOP also cannot be caught/blocked/ignored, but instead of terminating, it pauses the process indefinitely. It is what `Ctrl+Z` (technically SIGTSTP) escalates to. The process stays in TASK_STOPPED state until SIGCONT resumes it. This is what `kill -STOP` does. Useful for debugging or pausing rogue processes without killing them.
    </Note>

    <Note>
    **Senior follow-up 2:** A process is in TASK_UNINTERRUPTIBLE waiting for an NFS server. SIGKILL does nothing. What changed in modern kernels? Linux 2.6.25 (2008) added `TASK_KILLABLE` -- like TASK_UNINTERRUPTIBLE but allows SIGKILL specifically to break the sleep. NFS code path was migrated to use TASK_KILLABLE, so on modern kernels you can SIGKILL a stuck NFS process. There are still some kernel paths using pure TASK_UNINTERRUPTIBLE (e.g., critical journal flushes), but they should be brief.
    </Note>

    <Note>
    **Senior follow-up 3:** What signal does `docker stop` use? `docker stop` sends SIGTERM to PID 1 of the container, waits 10 seconds (configurable with `--time`), then sends SIGKILL. The 10 second default has caught many people: web servers with long-running requests get killed mid-request. Either increase the timeout or implement faster shutdown. `docker kill` skips straight to SIGKILL.
    </Note>

    **Common Wrong Answers:**
    - "SIGKILL is just a stronger SIGTERM." Wrong: SIGKILL is fundamentally different -- it bypasses userspace entirely.
    - "You can catch SIGKILL with SA_NODEFER." Wrong: SIGKILL cannot be caught regardless of flags.
    - "SIGKILL kills any process instantly." Wrong: D-state processes resist even SIGKILL on older kernels.

    **Further Reading:**
    - `man 7 signal` -- table of signals and which can be caught.
    - "Killable system calls" on LWN -- lwn.net/Articles/288056/.
    - Linux kernel `kernel/signal.c` -- read `do_signal()` for the SIGKILL fast path.
  </Accordion>

  <Accordion title="Why is printf() unsafe in a signal handler? What would you use instead, and why is the self-pipe trick the canonical solution?">
    **Strong Answer Framework:**

    1. **Identify the specific danger**: printf calls into stdio (`stdout` is a `FILE *`), which uses a mutex (`flockfile`) to serialize writes from multiple threads. Inside printf, there is also potentially a malloc to grow internal buffers, and locale lookups for formatting %d/%f. The mutex is the killer: if the main thread held the stdio lock when the signal arrived, and the handler calls printf, the handler tries to acquire a lock its own thread holds -- self-deadlock.
    2. **Generalize the rule**: Any function that takes a lock or modifies global state may deadlock when called from a signal handler that interrupted code holding that lock. POSIX's async-signal-safe list defines exactly which functions are guaranteed not to do this.
    3. **Introduce the self-pipe trick**: Create a pipe at startup (`pipe2(fd, O_NONBLOCK | O_CLOEXEC)`). Add the read end to your event loop (epoll/select). Install a signal handler that does only one thing: `write(pipe[1], "x", 1)` -- write is async-signal-safe. The main loop wakes up on the readable pipe, drains the byte, and dispatches the signal in normal code where any function is fair game.
    4. **Mention the modern alternative**: signalfd (Linux 2.6.22+) skips the pipe entirely. signalfd creates an FD that becomes readable when a queued signal arrives. read(signalfd, &amp;info, sizeof(info)) returns siginfo_t with details. This is what Go's `signal.Notify` and Rust's `tokio::signal` use under the hood.

    **Real-World Example:**

    Nginx's signal handling, documented in `src/os/unix/ngx_process_cycle.c`, uses the self-pipe trick. A signal handler sets a `sig_atomic_t` flag and writes to a pipe. The master process's main loop polls the pipe; when readable, it checks the flag and dispatches to handler functions in normal context. This pattern was canonical Unix daemon code from the early 2000s; signalfd in 2007 made it obsolete on Linux but the pattern remains portable across BSDs and macOS.

    <Note>
    **Senior follow-up 1:** Why not just use a `volatile sig_atomic_t` flag and check it in the main loop without a pipe? You can, but the main loop would need to check the flag at every iteration of every blocking syscall, which is impractical. The pipe makes the signal *visible* to your event loop -- when a signal arrives, the pipe becomes readable and your epoll wakes up. Without a pipe, you would need to use pselect/ppoll which atomically apply a signal mask, but those are more complex.
    </Note>

    <Note>
    **Senior follow-up 2:** Can a signal handler call write() on the same FD that the main thread is writing to? Yes, write is async-signal-safe. But the OUTPUT may be interleaved (the main thread's write to stderr and the handler's write to stderr can produce mixed output if they happen concurrently from different threads). For a single-threaded program, since the signal handler runs on the main thread, writes are serialized but the main thread's write may be partially complete -- so you can get incomplete lines. For human-readable logs from a handler, prefer all-at-once writes with a fixed-size buffer.
    </Note>

    <Note>
    **Senior follow-up 3:** What is `sigsafe_printf` and where do you find it? There are several open-source implementations of an async-signal-safe printf-like function, but none are in stdlib. They typically reimplement format parsing, write digits manually, and use only write(2). Examples: glibc's `_dl_debug_printf` (used by the dynamic linker) and Postgres' `write_stderr`. If you need this functionality, take one of these as a reference rather than rolling your own.
    </Note>

    **Common Wrong Answers:**
    - "printf is not signal-safe because it does I/O." Wrong: write() does I/O and is signal-safe. The issue is locks and global state.
    - "Use sprintf instead of printf." Wrong: sprintf is also not async-signal-safe (it uses locale state).
    - "Add a mutex to printf." Wrong: that is what makes it unsafe.

    **Further Reading:**
    - "The self-pipe trick" by D. J. Bernstein -- cr.yp.to/docs/selfpipe.html.
    - "signalfd, eventfd, timerfd: New ways to handle async events" on LWN -- lwn.net/Articles/253793/.
    - "Async-Signal-Safe Functions" in glibc manual.
  </Accordion>

  <Accordion title="Walk through the entire flow when a process accesses an unmapped address. What does the kernel do, and how does the application get SIGSEGV?">
    **Strong Answer Framework:**

    1. **The fault**: User code executes `mov rax, [rdi]` where rdi points to address 0x42 (unmapped). The CPU's MMU walks the page table, finds no entry, raises a #PF (page fault) exception. The CPU pushes the faulting RIP, RSP, error code, and CR2 (the faulting address) onto the kernel stack and jumps to the kernel's page fault handler.
    2. **The kernel page fault handler**: `do_page_fault()` (arch/x86/mm/fault.c) reads CR2 to get the faulting address. It walks the process's VMAs (Virtual Memory Areas) -- the linked list of mapped regions in mm_struct. If no VMA contains the address, this is a true segfault.
    3. **Sending the signal**: The kernel calls `force_sig_fault(SIGSEGV, SEGV_MAPERR, addr)`. This sets the SIGSEGV bit in the task's pending mask along with the siginfo_t describing the fault (signal number, fault address, fault type).
    4. **Signal delivery**: On the way back to user space, `do_signal()` dequeues SIGSEGV. If a handler is installed (e.g., for crash reporting), the kernel sets up a signal frame on the user stack (or alt stack if SA_ONSTACK), modifies the saved RIP to point at the handler, and returns. The handler runs. When it returns, rt_sigreturn restores the original context. If no handler, default action is to terminate the process and (typically) write a core dump.
    5. **The subtle bit**: If the handler returns normally (without exiting or longjmping), the kernel restores the original RIP and RSP -- which means the same instruction that faulted is re-executed. If the underlying condition hasn't changed (still unmapped), it faults again, the handler runs again, infinite loop. This is why crash handlers either exit or longjmp out, never return.

    **Real-World Example:**

    JIT compilers (V8, JVM) use a clever trick: they intentionally let SIGSEGV happen on a guard page, catch it in a handler, and use the fault as a signal that "this VM region needs to be allocated/extended now." The handler maps the missing memory via mmap and returns; the original instruction re-executes successfully. This is how Go's growable goroutine stacks work -- a guard page at the end of the current stack triggers a fault, the runtime grows the stack, and execution resumes. This requires SA_ONSTACK because the original stack is exactly what overflowed.

    <Note>
    **Senior follow-up 1:** What is `SEGV_MAPERR` vs `SEGV_ACCERR`? Both are SIGSEGV with different siginfo si_code. `SEGV_MAPERR` (1) means "the address is not mapped at all" (no VMA covers it). `SEGV_ACCERR` (2) means "the address is mapped but you don't have permission for this access type" -- e.g., writing to a read-only page. Crash handlers can read si_code to distinguish, useful for debugging vs intentional guard page faults.
    </Note>

    <Note>
    **Senior follow-up 2:** Why does running out of stack cause SIGSEGV instead of SIGBUS or some other signal? On Linux, the stack is implemented as a VMA that grows automatically when accessed below the current stack pointer (within the rlimit). When you access an address below the rlimit-allowed stack, the kernel extends the VMA. Beyond the rlimit, no VMA covers the address, so it is SEGV_MAPERR. On some BSDs, stack-related faults are SIGBUS. POSIX is loose on this -- portable code should handle both.
    </Note>

    <Note>
    **Senior follow-up 3:** Can a SIGSEGV handler safely call malloc to allocate a logging buffer? No. malloc is not async-signal-safe. If the SIGSEGV happened while the main thread held the malloc lock, the handler deadlocks. For crash reporters, allocate the buffer at startup (when the process is healthy) and use it from the handler.
    </Note>

    **Common Wrong Answers:**
    - "The process exits immediately." Wrong: a handler can intervene.
    - "SIGSEGV is delivered to all threads." Wrong: it is synchronous, delivered to the faulting thread only.
    - "The kernel handles all page faults silently." Only "valid" faults (page-in, COW, stack growth) are silent; "invalid" faults become SIGSEGV.

    **Further Reading:**
    - `arch/x86/mm/fault.c` in the Linux kernel -- the actual page fault handler.
    - "Page faults explained" by Mathieu Desnoyers -- LWN article on page fault paths.
    - "How V8 handles guard pages" in the V8 design docs.
  </Accordion>
