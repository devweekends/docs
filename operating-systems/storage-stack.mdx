# The Advanced Storage Stack

When an application calls `write()`, it begins a journey through multiple kernel layers: the Virtual File System (VFS), the Page Cache, the Block Layer, and finally the hardware driver. For high-performance systems, understanding how the kernel batches, schedules, and dispatches these requests to modern SSDs is critical.

## 1. The Block I/O Layer: The `bio` and `request`

The Block Layer's job is to translate filesystem-level operations into hardware-level disk commands. Think of it as a postal sorting office: mail (bios) arrives from various departments (filesystems), gets grouped by destination (adjacent sectors), sorted for efficient delivery (I/O scheduling), and dispatched to delivery trucks (hardware queues).

### 1.1 The `struct bio`
The `bio` is the basic unit of I/O in the kernel. It represents an I/O operation that is in flight.
- **Segments**: A `bio` can point to multiple non-contiguous memory pages (Scatter-Gather) that should be written to contiguous sectors on disk. This is critical because `malloc()`-allocated user buffers may be physically scattered, but the disk needs to know "write these bytes to sector 12345."
- **Lifecycle**: Created by the filesystem, passed to the block layer, and finally completed by the driver. The completion callback (`bio_endio()`) fires asynchronously when the hardware reports success (or failure).

### 1.2 The `struct request`
To improve efficiency, the kernel does not send every `bio` to the disk immediately. It groups one or more `bio`s into a `request`.
- **Merging**: If two `bio`s target adjacent sectors, they are merged into a single `request` to reduce the number of commands sent to the hardware. On rotational disks, this is essential -- a single 128 KB sequential write is orders of magnitude faster than 32 separate 4 KB random writes because the disk head does not need to seek between them. On SSDs, merging still helps because it reduces command overhead and enables the controller's internal parallelism.

## 2. Multiqueue Block I/O (`blk-mq`)

In the era of HDDs, the kernel used a single request queue protected by a single lock. With modern SSDs that can handle millions of IOPS, this lock became a massive bottleneck -- like funneling all traffic through a single toll booth on a 16-lane highway.

### 2.1 The `blk-mq` Architecture
Modern Linux uses a two-level queueing system:
1. **Software Staging Queues**: Per-CPU queues where requests are initially placed. Each CPU deposits requests in its own queue without needing to coordinate with other CPUs -- this eliminates lock contention entirely. The scheduler runs per-queue, merging and sorting locally.
2. **Hardware Dispatch Queues**: These map to the actual hardware queues of the device (e.g., NVMe queues). After the software queue applies scheduling policy, requests are "dispatched" into the hardware queue, which the device controller drains via DMA.

```

     CPU 0              CPU 1              CPU 2              CPU 3
  ┌──────────┐       ┌──────────┐       ┌──────────┐       ┌──────────┐
  │ SW Queue │       │ SW Queue │       │ SW Queue │       │ SW Queue │
  │ (no lock │       │ (no lock │       │ (no lock │       │ (no lock │
  │  needed) │       │  needed) │       │  needed) │       │  needed) │
  └────┬─────┘       └────┬─────┘       └────┬─────┘       └────┬─────┘
       │                  │                  │                  │
       │    ┌─────────────┼──────────────────┘                  │
       │    │             │                                     │
       ▼    ▼             ▼                                     ▼
  ┌──────────────┐   ┌──────────────┐
  │  HW Queue 0  │   │  HW Queue 1  │   (maps to NVMe SQ pairs)
  └──────┬───────┘   └──────┬───────┘
         │                  │
         ▼                  ▼

  └──────────────────────────────┘
```

**Practical tip**: You can inspect the queue mapping on your system with `cat /sys/block/nvme0n1/mq/*/cpu_list` to see which CPUs are mapped to which hardware queues. On a well-configured server, you want at least as many hardware queues as you have active CPUs submitting I/O.

## 3. I/O Schedulers: Sorting the Stream

Schedulers decide the order in which requests are sent to the hardware.

| Scheduler | Type | Best For |
|-----------|------|----------|
| **MQ-Deadline** | Fixed | Predictable latency; default for most systems. Guarantees reads are served within a deadline (500 ms default) even under heavy write pressure. |
| **Kyber** | Dynamic | High-performance NVMe; prioritizes read latency over write throughput. Self-tuning based on observed latencies -- requires no manual configuration. |
| **BFQ** | Complex | Desktop responsiveness; ensures no process is starved of I/O. Tracks per-process I/O budgets to prevent a backup job from freezing your IDE. |
| **None** | Passthrough | High-end NVMe where the hardware itself is the scheduler. Used when the SSD's internal firmware does a better job of ordering commands than the kernel can. |

**Practical tip**: For servers running databases, use `mq-deadline` or `none`. For desktops where interactive responsiveness matters, `bfq` is the right choice. Check your current scheduler with `cat /sys/block/nvme0n1/queue/scheduler` and change it with `echo mq-deadline > /sys/block/nvme0n1/queue/scheduler`.

## 4. NVMe Internals: Performance at Scale

NVMe (Non-Volatile Memory Express) is a protocol designed specifically for NAND flash over the PCIe bus.

### 4.1 Submission and Completion Queues (SQ/CQ)

The NVMe SQ/CQ model mirrors the io_uring pattern (see [Modern Features](/operating-systems/modern-features)): a pair of ring buffers in host RAM, shared between the driver and the device controller. Think of it as a restaurant order system: the waiter (driver) places order slips on a conveyor belt (SQ), and the kitchen (NVMe controller) places finished dishes on a return belt (CQ).

- **Submission Queue (SQ)**: The driver writes a 64-byte command (containing the opcode, Logical Block Address, data pointer, and length) to a ring buffer in RAM and "rings a doorbell" -- a single MMIO write to a register on the device that says "new command available at index N."
- **Completion Queue (CQ)**: The device reads the command via DMA, performs the I/O, then writes a 16-byte completion entry (status code, command ID, result) back to another ring buffer in RAM and triggers an MSI-X interrupt.

```c
// Simplified NVMe command submission (from drivers/nvme/host/pci.c)
static blk_status_t nvme_queue_rq(struct blk_mq_hw_ctx *hctx,
                                  const struct blk_mq_queue_data *bd) {
    struct nvme_queue *nvmeq = hctx->driver_data;
    struct request *req = bd->rq;
    struct nvme_command cmd;

    // Build the 64-byte NVMe command from the block-layer request
    nvme_setup_cmd(req->q->queuedata, req, &cmd);

    // Write command to the next slot in the Submission Queue ring buffer
    memcpy(&nvmeq->sq_cmds[nvmeq->sq_tail], &cmd, sizeof(cmd));

    // Advance tail pointer (wrap around when we reach the end)
    nvmeq->sq_tail = (nvmeq->sq_tail + 1) % nvmeq->q_depth;

    // Ring the doorbell: one MMIO write tells the controller
    // "there are new commands up to index sq_tail"
    writel(nvmeq->sq_tail, nvmeq->q_db);

    return BLK_STS_OK;
}
```

### 4.2 Scaling with MSI-X
NVMe supports up to 64,000 queue pairs. In a multi-core system, each CPU can have its own dedicated SQ/CQ pair. Using **MSI-X interrupts** (Message Signaled Interrupts - Extended), the completion interrupt for a specific CPU's queue is delivered directly to that CPU, maximizing cache locality -- the data structures the interrupt handler touches are already in that CPU's L1 cache.

**Why this matters for performance**: In the old SATA/AHCI world, a single command queue with a single interrupt line meant all I/O was serialized through one CPU. NVMe's per-CPU queues enable true parallelism: 16 CPUs can each submit and complete I/O independently, with zero lock contention and perfect cache locality. This is how modern NVMe SSDs achieve millions of IOPS.

## 5. Device Mapper: LVM and RAID

The Device Mapper (DM) is a kernel framework that allows for "Virtual Block Devices." It sits between the filesystem and the physical block device, intercepting and transforming I/O. Think of it as a layer of indirection -- just as virtual memory decouples process addresses from physical RAM, Device Mapper decouples logical volumes from physical disks.

- **LVM (Logical Volume Manager)**: Allows for resizing volumes on a live system, thin provisioning (allocate storage on demand rather than upfront), and snapshots (point-in-time copies of a volume for backups).
- **dm-raid (Software RAID)**: Implements redundancy in software. Common configurations: RAID 1 (mirror for reliability), RAID 5/6 (striping with parity for capacity + protection), RAID 10 (mirror + stripe for performance + reliability).
- **dm-multipath**: Sends I/O over multiple physical cables to the same storage array for redundancy and load balancing. If one HBA (Host Bus Adapter) or cable fails, I/O automatically routes through the surviving path.
- **dm-crypt**: Transparent full-disk encryption. LUKS volumes use dm-crypt under the hood.

**Practical tip**: You can inspect the Device Mapper topology of your system with `dmsetup table` or `lsblk` to see how logical volumes map to physical devices. Understanding this stack is essential when diagnosing I/O latency -- an extra DM layer adds microseconds of overhead and can introduce unexpected serialization points.

## 6. End-to-End: From `write()` to NVMe

Follow a single `write(fd, buf, 4096)` call through the entire storage stack:

```
┌─────────────────────────────────────────────────────────────────────┐
│              WRITE() JOURNEY: USER SPACE TO NAND FLASH              │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  1. USER SPACE                                                      │
│  ┌────────────────────────────────────────────────────────────────┐ │
│  │  write(fd, buf, 4096);                                          │ │
│  │  → Triggers syscall: mov rax, 1; syscall                        │ │
│  └──────────────────────────────┬─────────────────────────────────┘ │
│                                 │ SYSCALL                           │
│  2. VFS LAYER                   ▼                                   │
│  ┌────────────────────────────────────────────────────────────────┐ │
│  │  ksys_write() → vfs_write() → ext4_file_write_iter()            │ │

│  │  • Check permissions, file position                             │ │
│  └──────────────────────────────┬─────────────────────────────────┘ │
│                                 ▼                                   │
│  3. PAGE CACHE                                                      │

│  │  • Triggers MSI-X interrupt                                     │ │

│                                 ▼                                   │
```

### Key Latency Points

| Stage | Typical Latency | Bottleneck |
|-------|-----------------|------------|
| Syscall entry | ~100 ns | Mode switch |
| Page Cache copy | ~500 ns | Memory bandwidth |
| Block layer | ~1 μs | Software queuing |
| NVMe command | ~10-100 μs | Flash access |
| **Total (cached)** | **under 1 μs** | Returns after Page Cache |
| **Total (fsync)** | **~50-200 μs** | Waits for NVMe completion |

## 6.5 Caveats and Common Pitfalls

<Warning>
**Production traps in the storage stack:**

1. **Page cache vs disk -- your write is not durable until `fsync` returns successfully.** A successful `write()` only guarantees the bytes were copied into kernel memory. They sit as dirty pages until the writeback thread flushes them, which can be seconds later. A power loss between `write()` returning and the writeback completing means your data is gone, even though the application has already told the user "saved." Every system that claims durability must call `fsync` (or `fdatasync`) and check its return value.

2. **I/O schedulers (BFQ, mq-deadline, none) -- NVMe usually wants `none`.** The kernel ships sensible defaults for spinning rust: `mq-deadline` enforces read deadlines, `BFQ` provides fairness for desktops. Both add CPU work and reordering latency that NVMe does not need -- the device firmware schedules better than the kernel can. On NVMe, `none` (passthrough) often improves throughput by 10-30 percent and reduces tail latency. Distros pick one default; the right value is workload- and device-specific.

3. **Block size mismatches cause read-modify-write amplification.** Modern NVMe drives have 4K (or 16K, on Optane) physical sectors. If your filesystem uses 512-byte logical sectors and your application writes unaligned 1KB chunks, the drive must read the surrounding 4K, modify, and write the whole sector back -- often 4-8x the actual data. Worse, this happens silently; throughput halves and you do not see it in `iostat` because the drive accepts the writes. Align your filesystem block size, your application writes, and your partition offsets.

4. **RAID5 write hole -- battery-backed cache or ZFS solves it.** In RAID 5/6, a write touches a data block plus parity blocks. If power fails between writing data and writing parity, the parity becomes inconsistent. On the next disk failure, reconstruction uses corrupted parity to "rebuild" garbage. Hardware RAID solves this with a battery-backed (or flash-backed) write cache that journals the write atomically. ZFS solves it via copy-on-write -- new data goes to a fresh block, parity is computed, then the pointer is atomically swung. Software RAID5 without a journal device is a data-loss event waiting to happen.
</Warning>

<Tip>
**Solutions and patterns that earn their keep:**

- **Treat `fsync` return as the durability boundary.** Every transaction commit, every "save successful" UI confirmation, every replication ack must be downstream of a successful `fsync`. Check the return value -- on EIO, your data is gone (and on Linux pre-4.13, the error is also lost on subsequent fsyncs from other fds; rotate the file or restart).
- **For NVMe, switch to `none` and benchmark.** `echo none > /sys/block/nvme0n1/queue/scheduler` is reversible; measure with your real workload. If you see latency or throughput improve, keep it. If not, revert. Document the choice in your provisioning code so it survives a node rebuild.
- **Match your filesystem block size to device physical sectors and your write pattern.** For databases, align WAL writes to 4K (or whatever the drive reports via `BLKPBSZGET`). For object stores, larger blocks (64K-1MB) reduce metadata overhead. `xfs_info` and `tune2fs -l` reveal current settings; resizing is harder than getting it right at format time.
- **For RAID 5/6, use ZFS or a journal/log device.** mdadm RAID5 supports `--bitmap=internal` and a write-intent bitmap, but the ultimate fix is a journal device (`--write-journal`) on a fast SSD, or just use ZFS RAIDZ which sidesteps the hole entirely. Better: skip parity RAID for write-heavy workloads; mirrors (RAID 10) avoid the problem at the cost of capacity.
- **Diagnose with the right tool, not the loudest one.** `iostat -x 1` for high-level throughput/latency. `biosnoop` (bcc) for per-I/O traces. `biolatency` for histogram view. `nvme smart-log` and `nvme error-log` for device health. Before guessing, read.
</Tip>

## 7. Interview Deep Dive: Senior Level

<AccordionGroup>
  <Accordion title="What is 'Write Amplification' and how does the OS mitigate it?">
    Write Amplification occurs because SSDs can only overwrite data by erasing entire "Blocks" (which are larger than "Pages"). If the OS modifies a small file, the SSD might have to read, erase, and rewrite a huge block. 
    
    The OS mitigates this using the **TRIM** command. When a file is deleted, the OS tells the SSD: "These sectors are no longer needed." The SSD's garbage collector can then skip those sectors, reducing unnecessary writes and extending the drive's life.
  </Accordion>

  <Accordion title="Explain the difference between 'Synchronous' and 'Asynchronous' I/O from a kernel perspective.">
    - **Synchronous (`read`/`write`)**: The calling process is put into "Uninterruptible Sleep" (D state) until the hardware interrupt confirms the I/O is complete.
    - **Asynchronous (`io_uring` / `AIO`)**: The process submits a request and continues executing immediately. It later checks a completion queue or receives a signal when the data is ready. This is critical for high-concurrency servers.
  </Accordion>

  <Accordion title="How does `fsync()` work across the storage stack?">
    When you call `fsync()`:
    1. VFS identifies all dirty pages in the Page Cache for that file.
    2. The Block Layer creates `bio`s with the `REQ_PREFLUSH` and `REQ_FUA` flags.
    3. The Driver sends a specialized command to the disk telling it to flush its **Internal Volatile Cache** to the persistent NAND. 
    4. The call only returns once the hardware confirms the data is physically on the platter/flash.
  </Accordion>
</AccordionGroup>

## Interview Deep-Dive

<AccordionGroup>
  <Accordion title="Explain the storage stack from VFS to disk -- every layer a write() touches">
    **Strong Answer Framework:**

    1. **Set scope.** A buffered `write(fd, buf, 4096)` to a regular file on ext4 over LVM over NVMe. No `O_DIRECT`, no `O_SYNC`. The file already exists; we are appending.
    2. **Userland to syscall.** glibc thin wrapper -> `syscall` instruction -> `__NR_write`. Mode switch ~150ns.
    3. **VFS.** `ksys_write` validates fd via `fdget`, takes a reference on `struct file`, dispatches to `f_op->write_iter`. For ext4 this is `ext4_file_write_iter`. Position is resolved (or `f_pos_lock` taken for shared fds).
    4. **Filesystem layer.** ext4 takes its inode lock, may need to allocate blocks. `generic_perform_write` walks pages, calls `write_begin` (which prepares the page in cache), `copy_from_user` into the cached page, `write_end` (mark dirty, possibly extend file). For a 4K aligned write, no read-modify-write of partial pages.
    5. **Page cache.** Page is now dirty. The syscall returns to userland here -- the application sees "write complete" but nothing has hit the device.
    6. **Writeback.** Later, the per-bdi writeback thread (`bdi_writeback`) wakes when dirty bytes exceed `dirty_background_ratio` or `dirty_expire_centisecs` elapses. It calls `ext4_writepages` -> `mpage_writepages` -> assembles `bio` structures.
    7. **Device mapper (LVM).** `submit_bio` enters the LVM target's map function, which translates logical to physical block addresses on the underlying PV. Adds a few hundred nanoseconds.
    8. **Block layer (blk-mq).** Bio is wrapped in a `request`, placed on per-CPU software queue. I/O scheduler (mq-deadline or none) merges adjacent requests, applies fairness/deadline policy, dispatches to a hardware queue.
    9. **NVMe driver.** `nvme_queue_rq` builds a 64-byte command, places it in the SQ ring, rings the doorbell with one MMIO write.
    10. **Hardware.** Controller DMAs the command, then DMAs the data from page cache, writes it to NAND (FTL handles wear leveling and GC), writes a 16-byte CQE to the completion queue, raises MSI-X.
    11. **Completion.** ISR runs `nvme_irq` -> `nvme_process_cq` -> `bio_endio` walks the chain back up: clear PageDirty, wake any `fsync` waiter, release the request slot.

    **Real-World Example:** In 2017, Cloudflare diagnosed a 200ms tail-latency spike on writes to their config service that turned out to be `dm-crypt` synchronously encrypting on a single CPU during writeback bursts. Each layer in the stack -- VFS, filesystem, device mapper, block layer, driver -- can hide latency, and "the fix" required end-to-end tracing with `bpftrace` to attribute the time correctly. Their write-up emphasizes: when latency moves, you must trace the whole path, not just the layer you suspect.

    <Note>
    **Senior Follow-up 1:** "Where in this path does `fsync` change behavior?"

    `fsync` forces the writeback to happen synchronously before returning. It also issues a `REQ_PREFLUSH | REQ_FUA` to the device, which flushes the volatile write cache to NAND (or platter). The path is identical until step 6, where instead of waiting for the bdi thread, the calling thread drives the writeback itself and blocks until completion.
    </Note>

    <Note>
    **Senior Follow-up 2:** "How does `O_DIRECT` shorten this path?"

    Skips the page cache entirely. Bio points at user pages (pinned via `get_user_pages`); the rest of the stack from blk-mq downward is identical. No writeback thread, no dirty pages, no `copy_from_user`. The cost is alignment requirements and loss of caching.
    </Note>

    <Note>
    **Senior Follow-up 3:** "Where is the latency budget on a small synchronous write?"

    For a 4KB `O_SYNC` write to NVMe: ~150ns syscall, ~1us VFS+ext4 (including journal write for metadata), ~1us blk-mq, ~50-100us NVMe write+flush, ~150ns return path. The NAND write dominates by 50-100x. Software is rarely the bottleneck on modern devices unless you are doing something pathological.
    </Note>

    **Common Wrong Answers:**

    1. *"`write()` writes to disk."* No -- buffered writes hit the page cache. Many production bugs come from believing this.
    2. *"The block layer just passes bios through."* It merges, sorts, applies QoS, splits oversized requests, and dispatches to per-CPU hardware queues. It is one of the most complex pieces of the kernel.
    3. *"DMA copies once."* The CPU did `copy_from_user` (cache-to-cache), then the device DMAed (memory-to-device). Two motions.

    - Jens Axboe's blog and LSFMM (Linux Storage and Filesystems Summit) talks
    - Brendan Gregg, "Linux Storage Stack Diagram" -- the canonical visual reference
  </Accordion>

  <Accordion title="fsync -- what guarantees does it actually give?">
    **Strong Answer Framework:**

    1. **State the contract precisely.** `fsync(fd)` must not return until all in-flight data and metadata for that file are durable on stable storage. "Durable" means the data will survive a power loss.
    2. **Walk the layers it touches.** VFS flushes dirty pages for the file's address_space. Filesystem flushes the journal so metadata is recoverable. Block layer issues `REQ_PREFLUSH` (flush write cache before this op) and may use `REQ_FUA` (force this op to bypass cache). Driver translates that into device-specific commands -- for NVMe, the Flush admin command.
    3. **State the conditions for the guarantee to hold.** The drive must honor the flush command. Most enterprise drives do; some consumer SSDs lie about flush completion to look fast in benchmarks. The kernel and filesystem must propagate errors correctly. The application must call `fsync` and check the return value.
    4. **Acknowledge the dark history (fsync-gate).** Pre-Linux 4.13, an I/O error during writeback was reported on the *first* fsync after the error, then *cleared*. A subsequent fsync (possibly from a different fd held by a different process) would return success despite the data being lost. Postgres community uncovered this in 2018; Linux now keeps an "errseq" counter so each fd sees the error at least once.
    5. **`fsync` does NOT guarantee:** ordering with other files (use `fsync` on each), durability of the directory entry itself (must `fsync` the parent directory after creating a file), or that subsequent writes after the fsync are durable.
    6. **`fdatasync` is a weaker barrier** that skips metadata not needed for read consistency (e.g., does not flush atime updates). For WAL append patterns it is sufficient and faster.

    **Real-World Example:** In 2018, Andres Freund (Postgres committer) presented findings at PGCon revealing that Postgres's longstanding fsync handling was unsound on Linux. After an EIO during writeback, Postgres was acknowledging commits that were not actually durable. The fix took months: a panic-on-error mode, retry policies, kernel patches across multiple distros. The 2024 incident at MongoDB Atlas (post-mortem published) similarly traced data loss to a misunderstanding of when fsync errors clear. Treat fsync as a contract you must read carefully, not a reflex.

    <Note>
    **Senior Follow-up 1:** "If I write to a new file then fsync the file, is the file guaranteed to exist after a crash?"

    Not necessarily. The file's data and inode are durable, but the *directory entry* in the parent directory may still be in the page cache. If the parent's directory block was not journaled and flushed, after a crash the inode exists but no directory points at it -- it goes to lost+found. You must `fsync(parent_dir_fd)` after `fsync(file_fd)` for create-then-rename or create-then-fsync patterns.
    </Note>

    <Note>
    **Senior Follow-up 2:** "What is the difference between `REQ_PREFLUSH` and `REQ_FUA`?"

    `REQ_PREFLUSH` says "before doing this op, flush all previously-acknowledged but volatile writes." `REQ_FUA` (Force Unit Access) says "this op must bypass the write cache entirely." Postgres-style WAL uses both: PREFLUSH ensures previous writes are durable before the new one; FUA on the WAL write itself ensures it is durable on completion.
    </Note>

    <Note>
    **Senior Follow-up 3:** "How do you test fsync correctness?"

    Crash injection. Kill the VM mid-fsync, pull the cord, use `dm-flakey` to drop writes after a barrier. Tools like ALICE (from the ext4 paper) systematically explore crash states. If your "durable" data survives every injected crash, you are doing fsync right. If it does not, your bug is in your understanding of the stack, not in the storage device.
    </Note>

    **Common Wrong Answers:**

    1. *"fsync flushes everything."* It flushes data and metadata for one file, not unrelated dirty pages elsewhere on the system.
    2. *"If fsync returns 0, my data is on disk forever."* It is durable as of that moment. Subsequent writes need their own fsync.
    3. *"I do not need to check the return value."* You absolutely do. EIO from fsync means data loss.

    **Further Reading:**
    - "PostgreSQL's fsync surprise" -- LWN, 2018; the canonical write-up of fsync-gate
    - "Files are hard" -- Dan Luu's survey of crash-consistency research
    - The fsync(2) man page on a recent Linux kernel
  </Accordion>

  <Accordion title="Diagnose 100% disk busy with low throughput -- what tool, what to look at?">
    **Strong Answer Framework:**

    1. **First, define what "100% busy" means.** `iostat -x` shows `%util` which is the fraction of wall time the device had at least one outstanding request. On a single-queue HDD this maps to utilization. On NVMe with 64 hardware queues, `%util` can hit 100 percent with the device 5 percent loaded -- one request in flight at all times is enough. So `%util` is misleading for modern devices.
    2. **Pivot to the right metrics.** Look at `await` (avg I/O latency), `aqu-sz` (avg queue depth), `r/s` and `w/s` (IOPS), `rkB/s` and `wkB/s` (throughput), and `%iowait` from `vmstat`. The combination of high `await`, low IOPS, and low throughput tells you "queue is stuck behind something slow."
    3. **Hypothesize and trace.**
       - **Synchronous flushes:** Frequent `fsync` from a single writer serializes the queue. `bpftrace -e 'tracepoint:block:block_rq_complete { @[args->cmd_flags] = count(); }'` shows flush counts.
       - **Garbage collection on the SSD:** Bimodal latency histogram (most fast, occasional 100ms+ stalls). `nvme smart-log` and `bcc`'s `biolatency` reveal it.
       - **Saturated tail latency on one drive in a RAID:** A single bad drive can stall an entire `md` array. `iostat -x` per-device shows the outlier.
       - **dm-crypt CPU saturation:** Encryption serialized on one CPU. `top` shows kworker pegged; switch to `--perf-no_read_workqueue --perf-no_write_workqueue` or use larger key sizes with AES-NI hardware.
       - **Filesystem journal contention:** Many concurrent writers serialize at the journal commit. ext4 `data=journal` doubles write traffic. Check with `iostat` per-partition vs the journal device.
    4. **Confirm with eBPF, not guesses.** `biosnoop` shows per-I/O latency and PID. `funccount 'submit_bio'` and friends show submission rates. `iolatency` gives histograms. Modern Linux gives you near X-ray vision into the storage stack -- use it.
    5. **Fix the root cause, not the symptom.** Adding more disks does not fix CPU-bound dm-crypt. Throwing RAM at SSD GC does not stop the GC pauses. Diagnose first, then act.

    **Real-World Example:** In 2019, GitHub had a multi-hour incident where Postgres replicas fell behind during a primary writeback storm. `%util` showed 100 percent on the WAL volume. Initial reaction: scale up the disks. Real cause (per their post-mortem): a synchronous fsync inside a single-threaded checkpoint codepath, plus a large `wal_buffers`. The fix was tuning the checkpoint, not the hardware. Trusting `%util` would have cost them another six figures in over-provisioned IOPS.

    <Note>
    **Senior Follow-up 1:** "Why is `%util` misleading on NVMe?"

    `%util` is "fraction of time the request queue had at least one in-flight request." On a single-queue device this is utilization. On multi-queue NVMe with thousands of in-flight slots, one outstanding request at all times produces 100 percent `%util` while utilizing maybe 1 percent of the device. Use `aqu-sz` (queue depth) divided by max queue depth as a real utilization proxy, or look at IOPS vs the device's spec sheet.
    </Note>

    <Note>
    **Senior Follow-up 2:** "What is the difference between `iowait` and the disk being busy?"

    `iowait` (in `top`) is "CPU was idle but at least one task was blocked on I/O." A system with `iowait=80%` is not necessarily disk-bound; it just means *something* is waiting. If that something is a single low-priority cron job, the disk may be 5 percent busy. Conversely, a saturated disk on a busy CPU shows low `iowait` because the CPU is never idle. Cross-reference with `iostat`.
    </Note>

    <Note>
    **Senior Follow-up 3:** "How do you tell SSD garbage collection apart from filesystem journal contention?"

    GC produces a bimodal latency distribution -- most I/Os fast, periodic outliers. `biolatency -m` (millisecond histogram) reveals it. Journal contention produces a steady-high distribution under load and disappears when writes pause. Also, GC is per-device; journal contention is per-filesystem. Move the workload to a fresh device -- if the bimodal pattern follows, it is GC; if it disappears, it was journaling.
    </Note>

    **Common Wrong Answers:**

    1. *"100% util means we need more disks."* Often false on NVMe. Diagnose first.
    2. *"It is just slow disk; nothing we can do."* Almost never. Software is the usual culprit on modern hardware.
    3. *"`dd if=/dev/zero of=...` to test."* Sequential writes from `dd` test almost nothing relevant for typical random workloads. Use `fio` with realistic patterns.

    **Further Reading:**
    - Brendan Gregg, *Systems Performance* -- the disk chapter is the textbook
    - bcc-tools `biolatency`, `biosnoop`, `iolatency` man pages and examples
    - "USE Method" methodology by Brendan Gregg for systematic diagnosis
  </Accordion>

  <Accordion title="Explain blk-mq architecture and why the old single-queue design was a bottleneck for NVMe">
    **Strong Answer Framework:**

    1. **Set historical context.** The old request queue (single-queue, single spinlock) was designed in the HDD era. With 200 IOPS on a spinning disk and 5ms seek time, the lock was held for microseconds out of every 5ms -- contention was negligible.
    2. **Show why it failed for NVMe.** A single NVMe drive does 1M+ random read IOPS at p99 latencies under 100us. With 32 CPUs all submitting through a single lock, the lock itself becomes the bottleneck -- cache-line bouncing alone adds hundreds of nanoseconds per I/O, dwarfing the device latency.
    3. **Describe the two-level architecture.** Software staging queues are per-CPU (or per-NUMA-node), so submission requires no cross-CPU coordination. Bios are merged into requests at this level. Hardware dispatch queues map to device hardware queue pairs (NVMe SQ/CQ). The framework dispatches with CPU affinity so completion interrupts return to the originating CPU, maximizing cache hits.
    4. **Quantify.** On a 32-core server doing 1M IOPS, each CPU drives ~31K IOPS through its private queue with zero lock contention. blk-mq overhead is roughly 0.5-1us per I/O, vs 5-10us for the old single-queue path. That is the difference between achieving 1M IOPS and hitting a software ceiling at ~200K.
    5. **Connect to MSI-X.** Per-CPU completion interrupts are essential -- without MSI-X and IRQ affinity, you funnel 1M completions back to a single CPU and re-create the bottleneck on the receive side.

    **Real-World Example:** Christoph Hellwig and Jens Axboe drove the blk-mq merge in Linux 3.13 (2014) after Fusion-io and Intel demonstrated that single-queue legacy block layer could not feed PCIe SSDs. Facebook's published numbers from the era showed Flashcache+legacy block at ~150K IOPS while raw blk-mq+nvme could hit 1M+ on the same hardware. The legacy block layer was finally removed in Linux 5.0 (2019), 5 years after blk-mq was introduced -- a long migration.

    <Note>
    **Senior Follow-up 1:** "When would you choose `none` vs `mq-deadline`?"

    `none` for NVMe with random workloads -- the device firmware schedules better than the kernel can. `mq-deadline` for SATA SSDs or spinning disks where read deadlines and merging help, and especially on Postgres-style mixed workloads where you want to bound read latency under heavy write pressure. Change per-device with `echo X > /sys/block/.../queue/scheduler`.
    </Note>

    <Note>
    **Senior Follow-up 2:** "How does the kernel decide which hardware queue to dispatch to?"

    `blk_mq_map_queues` builds a CPU-to-HW-queue map at init time, typically one HW queue per CPU when supported. On submission, the framework looks at the current CPU and picks the matching HW queue. This is why pinning your I/O-issuing threads to specific CPUs matters for maximum throughput -- crossing NUMA nodes mid-pipeline costs cache misses.
    </Note>

    <Note>
    **Senior Follow-up 3:** "What is `BLK_MQ_F_TAG_QUEUE_SHARED` and when is it used?"

    Some devices have fewer hardware tags than the kernel would like queue depth. Shared tag mode lets multiple HW queues draw from a single tag pool, trading queue independence for tag pressure. Common on virtual block devices (virtio-blk) and some RAID controllers. Modern NVMe usually has plenty of tags per queue (1024+), so you rarely see this.
    </Note>

    **Common Wrong Answers:**

    1. *"blk-mq is just a thread pool."* No -- it is per-CPU lockless queues with explicit dispatch policy. Threads are not the model.
    2. *"All schedulers are equivalent on NVMe."* Measurably false. `none` saves 5-10 percent CPU and reduces tail latency on most NVMe.
    3. *"More queues are always better."* Beyond CPU count, queues add metadata overhead without benefit. The sweet spot is one HW queue per CPU.

    - "What's new in storage for Linux" -- Christoph Hellwig's annual LSFMM talks
