# eBPF: Programmable Kernel Internals

eBPF (Extended Berkeley Packet Filter) is arguably the most significant change to the Linux kernel in the last decade. It allows developers to run sandboxed programs inside the kernel without changing kernel source code or loading a kernel module. It is the "JavaScript of the Kernel."

## 1. The eBPF Virtual Machine

eBPF is a **Register-Based Virtual Machine** that runs inside the kernel. Unlike traditional BPF which was limited to network packet filtering, eBPF is Turing-complete (with bounded loops) and can be used for tracing, security, networking, and performance monitoring.

### 1.1 The Register Set

eBPF has 11 64-bit registers mapped to native CPU registers for maximum performance:

```
┌─────────────────────────────────────────┐
│      eBPF Register Architecture         │
├─────────────────────────────────────────┤
│ R0  - Return value from functions       │
│       and program exit code             │
│                                         │
│ R1  - First argument to helpers         │
│ R2  - Second argument                   │
│ R3  - Third argument                    │
│ R4  - Fourth argument                   │
│ R5  - Fifth argument                    │
│                                         │
│ R6  - Callee-saved (preserved)          │
│ R7  - Callee-saved                      │
│ R8  - Callee-saved                      │
│ R9  - Callee-saved                      │
│                                         │
│ R10 - Read-only frame pointer           │
│       (points to 512-byte stack)        │
└─────────────────────────────────────────┘
```

**x86-64 Mapping:**
```
eBPF R0  → RAX
eBPF R1  → RDI
eBPF R2  → RSI
eBPF R3  → RDX
eBPF R4  → RCX
eBPF R5  → R8
eBPF R6  → RBX
eBPF R7  → R13
eBPF R8  → R14
eBPF R9  → R15
eBPF R10 → RBP
```

### 1.2 The Instruction Set

Each eBPF instruction is 64 bits (8 bytes):

```c
struct bpf_insn {
    __u8    code;        // opcode
    __u8    dst_reg:4;   // destination register (0-10)
    __u8    src_reg:4;   // source register (0-10)
    __s16   off;         // signed offset
    __s32   imm;         // signed immediate constant
};
```

**Example eBPF bytecode:**
```
// r1 = *(u64 *)(r10 - 8)   Load from stack
0x79 0x11 0xf8 0xff 0x00 0x00 0x00 0x00

// r0 = 0                    Set return value
0xb7 0x00 0x00 0x00 0x00 0x00 0x00 0x00

// exit                      Return from program
0x95 0x00 0x00 0x00 0x00 0x00 0x00 0x00
```

### 1.3 The JIT (Just-In-Time) Compiler

The kernel contains architecture-specific JIT compilers that translate eBPF bytecode to native machine code:

```
┌──────────────────────────────────────────────┐
│          eBPF Program Lifecycle              │
├──────────────────────────────────────────────┤
│                                              │
│  1. C Source Code                            │
│     ↓                                        │
│  2. Clang/LLVM Compilation                   │
│     ↓                                        │
│  3. eBPF Bytecode (.o ELF object)            │
│     ↓                                        │
│  4. Userspace: bpf() syscall                 │
│     ↓                                        │
│  ┌─────────────────────────────────┐         │
│  │  Kernel: Verifier               │         │
│  │  - Control flow analysis        │         │
│  │  - Memory safety checks         │         │
│  │  - Register tracking            │         │
│  │  - Bounded loop validation      │         │
│  └─────────────────────────────────┘         │
│     ↓                                        │
│  5. JIT Compilation                          │
│     - x86-64 codegen                         │
│     - ARM64 codegen                          │
│     - s390x codegen                          │
│     ↓                                        │
│  6. Native machine code in kernel memory     │
│     - Executable, non-writable pages         │
│     - Performance: ~95% of native C          │
│     ↓                                        │
│  7. Program attached to hook point           │
│     - Kprobe, Tracepoint, XDP, etc.          │
│                                              │
└──────────────────────────────────────────────┘
```

**JIT Benefits:**
- **Performance**: Near-native speed (no interpretation overhead)
- **Security**: JIT code is mapped as read-execute only
- **Efficiency**: Direct register mapping eliminates translation overhead

**Checking JIT status:**
```bash
# Enable JIT (requires root)
echo 1 > /proc/sys/net/core/bpf_jit_enable

# Enable JIT with debug output
echo 2 > /proc/sys/net/core/bpf_jit_enable

# View JIT'd code
bpftool prog dump jited id <prog_id>
```

## 2. The Verifier: Guaranteeing Safety

The BPF verifier is the gatekeeper that ensures loaded programs cannot crash the kernel, leak data, or cause security vulnerabilities. It performs static analysis on the eBPF bytecode before JIT compilation.

### 2.1 The Verifier Architecture

```
┌─────────────────────────────────────────────────────┐
│              BPF Verifier Pipeline                  │
├─────────────────────────────────────────────────────┤
│                                                     │
│  Step 1: DAG Validation                             │
│  ┌───────────────────────────────────────┐          │
│  │ • Detect infinite loops               │          │
│  │ • Verify all paths reach exit         │          │
│  │ • Check unreachable code              │          │
│  │ • Bounded loop complexity (5.3+)      │          │
│  └───────────────────────────────────────┘          │
│                ↓                                    │
│  Step 2: Register State Tracking                    │
│  ┌───────────────────────────────────────┐          │
│  │ For each instruction, track:          │          │
│  │ • Register types (scalar, ptr, etc.)  │          │
│  │ • Value ranges (min/max bounds)       │          │
│  │ • Null-ness                           │          │
│  │ • Alignment constraints               │          │
│  └───────────────────────────────────────┘          │
│                ↓                                    │
│  Step 3: Memory Access Validation                   │
│  ┌───────────────────────────────────────┐          │
│  │ • Verify pointer types                │          │
│  │ • Check bounds for map access         │          │
│  │ • Validate stack reads/writes         │          │
│  │ • Prevent kernel address leaks        │          │
│  └───────────────────────────────────────┘          │
│                ↓                                    │
│  Step 4: Helper Function Validation                 │
│  ┌───────────────────────────────────────┐          │
│  │ • Check argument types                │          │
│  │ • Verify capabilities                 │          │
│  │ • Ensure return values handled        │          │
│  └───────────────────────────────────────┘          │
│                ↓                                    │
│  Step 5: Complexity Checks                          │
│  ┌───────────────────────────────────────┐          │
│  │ • Instruction count limit (1M)        │          │
│  │ • Stack depth limit (512 bytes)       │          │
│  │ • Tail call chain depth (33)          │          │
│  │ • Map nesting depth                   │          │
│  └───────────────────────────────────────┘          │
│                                                     │
└─────────────────────────────────────────────────────┘
```

### 2.2 Register Type System

The verifier tracks sophisticated type information for each register:

```c
// Register types (kernel internal)
enum bpf_reg_type {
    NOT_INIT,              // Uninitialized register
    SCALAR_VALUE,          // Unknown scalar value
    PTR_TO_CTX,            // Pointer to bpf_context
    PTR_TO_MAP_VALUE,      // Pointer to map value
    PTR_TO_MAP_KEY,        // Pointer to map key
    PTR_TO_STACK,          // Pointer to stack frame
    PTR_TO_PACKET,         // Pointer to packet data (XDP)
    PTR_TO_PACKET_END,     // Pointer past last packet byte
    PTR_TO_SOCKET,         // Pointer to socket
    PTR_TO_TCP_SOCK,       // Pointer to TCP socket
    // ... many more types
};

// Register state tracking
struct bpf_reg_state {
    enum bpf_reg_type type;
    s64 smin_value;        // Minimum signed value
    s64 smax_value;        // Maximum signed value
    u64 umin_value;        // Minimum unsigned value
    u64 umax_value;        // Maximum unsigned value
    u32 id;                // Unique ID for this value
    u32 ref_obj_id;        // Reference tracking
    // ... more tracking fields
};
```

### 2.3 Verifier Example: Range Tracking

```c
// BPF program snippet
int my_prog(struct __sk_buff *skb) {
    __u32 index;

    // Read packet length
    index = skb->len;         // Verifier: index = [0, U32_MAX]

    // Bounds check
    if (index >= 100)         // Verifier: creates two branches
        return 0;

    // In this branch: index = [0, 99]
    // Now safe to use as array index
    return my_array[index];   // Verifier: OK! index < 100
}
```

**Verifier log output:**
```
0: (61) r1 = *(u32 *)(r1 +0)          // r1 = skb->len
1: (25) if r1 > 99 goto pc+2          // range split
2: (67) r1 <<= 32
3: (c7) r1 s>>= 32                     // sign extension
   R1_w=inv(id=0,umax_value=99)        // verified range!
4: (18) r2 = 0xffff888100001000        // map address
6: (0f) r2 += r1                       // safe pointer arithmetic
7: (71) r0 = *(u8 *)(r2 +0)            // verified memory access
8: (95) exit
```

### 2.4 Bounded Loops (Linux 5.3+)

Modern kernels allow loops if the verifier can prove they are bounded:

```c
// Valid bounded loop
#pragma unroll
for (int i = 0; i < 10; i++) {
    // Process data
}

// The verifier will:
// 1. Unroll the loop up to a limit
// 2. Track the loop variable range
// 3. Ensure no infinite paths exist
```

**Complexity limits:**
- Max verified instructions: 1 million
- Max loop iterations (unrolled): ~8192
- Max program size: 1 million instructions
- Stack size: 512 bytes

### 2.5 Memory Safety Guarantees

```
┌─────────────────────────────────────────┐
│     Memory Access Rules                 │
├─────────────────────────────────────────┤
│                                         │
│ 1. Stack Access                         │
│    ✓ Must be within [R10-512, R10)      │
│    ✗ Cannot read uninitialized data     │
│    ✗ Cannot leak stack pointers         │
│                                         │
│ 2. Map Access                           │
│    ✓ Must use bpf_map_lookup_elem()     │
│    ✓ Must check for NULL return         │
│    ✗ Cannot access out-of-bounds        │
│                                         │
│ 3. Packet Access (XDP)                  │
│    ✓ Must bounds-check against end      │
│    ✓ If (ptr + offset < end) then OK    │
│    ✗ Cannot access after packet end     │
│                                         │
│ 4. Context Access                       │
│    ✓ Only allowed fields (via BTF)      │
│    ✗ Cannot write to read-only fields   │
│    ✗ Cannot leak kernel pointers        │
│                                         │
└─────────────────────────────────────────┘
```

## 3. eBPF Maps: Persistent Data Structures

Maps are the primary mechanism for eBPF programs to store state and communicate with userspace.

### 3.1 Map Types

```c
// Hash map - key/value store
BPF_MAP_TYPE_HASH
- Use case: PID → counter, IP → stats
- Lookup: O(1) average
- Memory: Dynamic allocation

// Array map - fixed size indexed array
BPF_MAP_TYPE_ARRAY
- Use case: Per-CPU stats, histogram buckets
- Lookup: O(1)
- Memory: Pre-allocated

// Per-CPU hash/array
BPF_MAP_TYPE_PERCPU_HASH
BPF_MAP_TYPE_PERCPU_ARRAY
- Use case: Lock-free aggregation
- Benefit: No synchronization overhead

// Ring buffer (preferred over perf buffer)
BPF_MAP_TYPE_RINGBUF
- Use case: Streaming events to userspace
- Memory: Efficient page-aligned buffer
- Ordering: Maintains event ordering

// LRU Hash map
BPF_MAP_TYPE_LRU_HASH
- Use case: Caching, connection tracking
- Eviction: Automatic LRU policy

// Stack trace map
BPF_MAP_TYPE_STACK_TRACE
- Use case: Store kernel/user stack traces
- Format: Array of instruction pointers

// Program array (tail calls)
BPF_MAP_TYPE_PROG_ARRAY
- Use case: Jump table for complex programs
- Limit: 33 chained tail calls
```

### 3.2 Map Definition Example

```c
// Kernel-side BPF program
#include <linux/bpf.h>
#include <bpf/bpf_helpers.h>

// Define a hash map
struct {
    __uint(type, BPF_MAP_TYPE_HASH);
    __uint(max_entries, 10240);
    __type(key, u32);                 // PID
    __type(value, u64);               // Count
} syscall_count SEC(".maps");

// Trace syscall entry
SEC("tracepoint/raw_syscalls/sys_enter")
int trace_syscall(struct trace_event_raw_sys_enter *ctx) {
    u32 pid = bpf_get_current_pid_tgid() >> 32;
    u64 *count, one = 1;

    // Lookup or create entry
    count = bpf_map_lookup_elem(&syscall_count, &pid);
    if (count) {
        __sync_fetch_and_add(count, 1);  // Atomic increment
    } else {
        bpf_map_update_elem(&syscall_count, &pid, &one, BPF_ANY);
    }

    return 0;
}

char LICENSE[] SEC("license") = "GPL";
```

**Userspace interaction:**
```c
// Load and attach program (using libbpf)
#include <bpf/libbpf.h>

int main() {
    struct bpf_object *obj;
    struct bpf_link *link;
    int map_fd;

    // Load BPF object
    obj = bpf_object__open_file("program.o", NULL);
    bpf_object__load(obj);

    // Attach to tracepoint
    link = bpf_program__attach(
        bpf_object__find_program_by_name(obj, "trace_syscall")
    );

    // Get map file descriptor
    map_fd = bpf_object__find_map_fd_by_name(obj, "syscall_count");

    // Read map data
    u32 key, next_key;
    u64 value;

    key = 0;
    while (bpf_map_get_next_key(map_fd, &key, &next_key) == 0) {
        bpf_map_lookup_elem(map_fd, &next_key, &value);
        printf("PID %u: %llu syscalls\n", next_key, value);
        key = next_key;
    }

    return 0;
}
```

### 3.3 Ring Buffer Architecture

Ring buffers replaced perf buffers as the preferred event streaming mechanism:

```
┌────────────────────────────────────────────────────┐
│         BPF Ring Buffer Architecture               │
├────────────────────────────────────────────────────┤
│                                                    │
│  Kernel Space (BPF Program)                        │
│  ┌──────────────────────────────────────┐          │
│  │  Producer: bpf_ringbuf_reserve()     │          │
│  │           bpf_ringbuf_submit()       │          │
│  │                                      │          │
│  │  ┌────────────────────────────┐      │          │
│  │  │   Ring Buffer (mmap'd)     │      │          │
│  │  │                            │      │          │
│  │  │  [E1][E2][E3][ ][ ][ ][E7] │      │          │
│  │  │    ^           ^            │      │          │
│  │  │    consumer    producer     │      │          │
│  │  └────────────────────────────┘      │          │
│  └──────────────────────────────────────┘          │
│                ↓ mmap                              │
│  User Space                                        │
│  ┌──────────────────────────────────────┐          │
│  │  Consumer: ring_buffer__consume()    │          │
│  │           ring_buffer__poll()        │          │
│  │                                      │          │
│  │  Callback for each event             │          │
│  └──────────────────────────────────────┘          │
│                                                    │
│  Advantages over Perf Buffer:                      │
│  ✓ Better memory efficiency                        │
│  ✓ Event ordering guarantees                       │
│  ✓ No per-CPU fragmentation                        │
│  ✓ Variable-length records                         │
│                                                    │
└────────────────────────────────────────────────────┘
```

**Ring buffer usage:**
```c
// Kernel side
SEC("kprobe/do_sys_open")
int BPF_KPROBE(trace_open, const char *filename) {
    struct event {
        u32 pid;
        char filename[256];
    } *e;

    // Reserve space
    e = bpf_ringbuf_reserve(&events, sizeof(*e), 0);
    if (!e)
        return 0;

    // Fill event
    e->pid = bpf_get_current_pid_tgid() >> 32;
    bpf_probe_read_str(&e->filename, sizeof(e->filename), filename);

    // Submit to userspace
    bpf_ringbuf_submit(e, 0);

    return 0;
}

// Userspace side
static int handle_event(void *ctx, void *data, size_t len) {
    struct event *e = data;
    printf("PID %d opened: %s\n", e->pid, e->filename);
    return 0;
}

int main() {
    struct ring_buffer *rb;

    rb = ring_buffer__new(map_fd, handle_event, NULL, NULL);

    // Poll for events
    while (running) {
        ring_buffer__poll(rb, 100 /* timeout ms */);
    }

    return 0;
}
```

## 4. Hook Points and Program Types

eBPF programs are event-driven and attach to specific kernel hooks.

### 4.1 Tracing Program Types

### 4.2 Kprobes vs Tracepoints vs Fentry

```c
// KPROBE - Dynamic instrumentation (any function)
SEC("kprobe/do_sys_openat2")
int BPF_KPROBE(trace_openat_kprobe,
               int dfd, const char *filename,
               struct open_how *how)
{
    // Access via PT_REGS_PARM macros
    char fname[64];
    bpf_probe_read_str(fname, sizeof(fname), filename);
    bpf_printk("kprobe: %s\n", fname);
    return 0;
}

// TRACEPOINT - Stable ABI
SEC("tracepoint/syscalls/sys_enter_openat")
int trace_openat_tp(struct trace_event_raw_sys_enter *ctx) {
    // Structured access via BTF
    char *filename = (char *)ctx->args[1];
    bpf_printk("tracepoint: %s\n", filename);
    return 0;
}

// FENTRY - Fast and type-safe (5.5+)
SEC("fentry/do_sys_openat2")
int BPF_PROG(trace_openat_fentry,
             int dfd, const char *filename,
             struct open_how *how)
{
    // Direct argument access (type-checked!)
    bpf_printk("fentry: %s\n", filename);
    return 0;
}

// FEXIT - Access return value
SEC("fexit/do_sys_openat2")
int BPF_PROG(trace_openat_fexit,
             int dfd, const char *filename,
             struct open_how *how,
             long ret)  // Return value
{
    bpf_printk("fexit: returned %ld\n", ret);
    return 0;
}
```

**Performance comparison:**
```
Hook Type      | Overhead    | Stability | Type Safety
---------------|-------------|-----------|-------------
Kprobe         | ~1-2μs      | Unstable  | Manual
Kretprobe      | ~2-4μs      | Unstable  | Manual
Tracepoint     | ~500-800ns  | Stable    | Partial
Fentry         | ~200-400ns  | Unstable  | Full (BTF)
Fexit          | ~200-400ns  | Unstable  | Full (BTF)
```

## 5. XDP: eXpress Data Path

XDP is the fastest packet processing framework in Linux, running eBPF programs in the network driver.

### 5.1 XDP Architecture

```
┌───────────────────────────────────────────────────┐
│         XDP Packet Processing Pipeline            │
├───────────────────────────────────────────────────┤
│                                                   │
│  NIC Hardware                                     │
│  ┌─────────────────────────────┐                  │
│  │  Packet arrives → DMA        │                  │
│  │  into RX ring buffer         │                  │
│  └─────────────────────────────┘                  │
│              ↓                                    │
│  Driver: XDP Hook (EARLIEST possible point)       │
│  ┌─────────────────────────────────────────────┐  │
│  │  eBPF Program runs on raw packet buffer    │  │
│  │  - No sk_buff allocated yet!                │  │
│  │  - Linear packet data                       │  │
│  │  - Can modify packet in-place               │  │
│  │                                             │  │
│  │  Return Actions:                            │  │
│  │  ┌──────────────────────────────────────┐   │  │
│  │  │  XDP_PASS  → Continue to stack       │   │  │

│                                                   │
└───────────────────────────────────────────────────┘
```

### 5.2 XDP Program Example: DDoS Mitigation

```c
#include <linux/bpf.h>
#include <linux/if_ether.h>
#include <linux/ip.h>
#include <linux/tcp.h>
#include <bpf/bpf_helpers.h>

// Rate limiting map: IP → packet count
struct {
    __uint(type, BPF_MAP_TYPE_LRU_HASH);
    __uint(max_entries, 1000000);
    __type(key, __u32);         // Source IP
    __type(value, __u64);       // Packet count
} rate_limit SEC(".maps");

SEC("xdp")
int xdp_filter(struct xdp_md *ctx) {
    void *data_end = (void *)(long)ctx->data_end;
    void *data = (void *)(long)ctx->data;

    // Parse Ethernet header
    struct ethhdr *eth = data;
    if ((void *)(eth + 1) > data_end)
        return XDP_PASS;

    // Only process IPv4
    if (eth->h_proto != __constant_htons(ETH_P_IP))
        return XDP_PASS;

    // Parse IP header
    struct iphdr *ip = (void *)(eth + 1);
    if ((void *)(ip + 1) > data_end)
        return XDP_PASS;

    // Rate limit by source IP
    __u32 src_ip = ip->saddr;
    __u64 *count = bpf_map_lookup_elem(&rate_limit, &src_ip);

    if (count) {
        *count += 1;

        // Drop if exceeds 10K pps (simplified)
        if (*count > 10000) {
            return XDP_DROP;
        }
    } else {
        __u64 init = 1;
        bpf_map_update_elem(&rate_limit, &src_ip, &init, BPF_ANY);
    }

    // Pass to kernel stack
    return XDP_PASS;
}

char LICENSE[] SEC("license") = "GPL";
```

## 6. BTF and CO-RE: Portable eBPF

BTF (BPF Type Format) enables portable eBPF programs that work across kernel versions.

### 6.1 The Portability Problem

```
Problem: Kernel struct layout changes between versions

Linux 5.10:                    Linux 5.15:
struct task_struct {           struct task_struct {
    int pid;         // +0         int tgid;        // +0
    int tgid;        // +4         int pid;         // +4  ← MOVED!
    char comm[16];   // +8         int real_parent; // +8  ← NEW!
    ...                            char comm[16];   // +12 ← MOVED!
}                                  ...
                                }

Traditional BPF: Hardcoded offset → BREAKS!
CO-RE BPF: Runtime relocation → WORKS!
```

### 6.2 BTF Format

BTF is a compact metadata format describing kernel types:

```c
// Kernel exports BTF at /sys/kernel/btf/vmlinux

// Example: struct task_struct BTF representation
[1] STRUCT 'task_struct' size=9472 vlen=238
    'thread_info' type_id=2 bits_offset=0
    'state' type_id=3 bits_offset=192
    'stack' type_id=4 bits_offset=256
    'flags' type_id=5 bits_offset=320
    'pid' type_id=6 bits_offset=352        ← Offset in bits!
    'tgid' type_id=6 bits_offset=384
    ...
```

**Viewing BTF:**
```bash
# Dump kernel BTF
bpftool btf dump file /sys/kernel/btf/vmlinux format c

# Generate vmlinux.h (all kernel types)
bpftool btf dump file /sys/kernel/btf/vmlinux format c > vmlinux.h
```

### 6.3 CO-RE (Compile Once - Run Everywhere)

```c
#include "vmlinux.h"  // BTF-generated kernel headers
#include <bpf/bpf_core_read.h>

SEC("kprobe/wake_up_new_task")
int BPF_KPROBE(trace_new_task, struct task_struct *task) {
    // OLD WAY (breaks on kernel updates):
    // pid = *(int *)((void *)task + 1234);  // hardcoded offset

    // CO-RE WAY (portable across kernels):
    pid_t pid = BPF_CORE_READ(task, pid);
    char comm[16];
    BPF_CORE_READ_STR_INTO(&comm, task, comm);

    bpf_printk("New task: %s (PID %d)\n", comm, pid);
    return 0;
}
```

**How CO-RE works:**
```
1. Compile time (developer machine):
   - Clang embeds relocation info in .BTF.ext section
   - "Field 'pid' at offset UNKNOWN"

2. Load time (target machine):
   - libbpf reads /sys/kernel/btf/vmlinux
   - Finds actual offset of task_struct->pid
   - Patches eBPF instructions with correct offset

3. Runtime:
   - Program uses correct offset for this kernel!
```

**Core macros:**
```c
// Field read with NULL check
BPF_CORE_READ(src, field1, field2)

// String read
BPF_CORE_READ_STR_INTO(&dst, src, field)

// Field existence check (compile-time)
if (bpf_core_field_exists(task->some_new_field)) {
    // Use field only on newer kernels
}

// Type existence check
if (bpf_core_type_exists(struct some_new_struct)) {
    // Use type only if available
}
```

## 8. Security and LSM Hooks

eBPF can enforce security policies via LSM (Linux Security Module) hooks.

### 8.1 LSM BPF Program

```c
#include "vmlinux.h"
#include <bpf/bpf_helpers.h>
#include <bpf/bpf_core_read.h>

// Deny writes to /etc/shadow
SEC("lsm/file_open")
int BPF_PROG(restrict_file_open, struct file *file) {
    const char *filename;
    char path[256];

    filename = BPF_CORE_READ(file, f_path.dentry, d_name.name);
    bpf_probe_read_kernel_str(path, sizeof(path), filename);

    // Block access to /etc/shadow
    if (__builtin_memcmp(path, "/etc/shadow", 11) == 0) {
        return -EPERM;  // Permission denied
    }

    return 0;  // Allow
}

// Deny execution of /tmp binaries
SEC("lsm/bprm_check_security")
int BPF_PROG(restrict_exec, struct linux_binprm *bprm) {
    const char *filename;
    char path[256];

    filename = BPF_CORE_READ(bprm, file, f_path.dentry, d_name.name);
    bpf_probe_read_kernel_str(path, sizeof(path), filename);

    // Check if path starts with /tmp/
    if (path[0] == '/' && path[1] == 't' &&
        path[2] == 'm' && path[3] == 'p' && path[4] == '/') {
        bpf_printk("Blocked execution from /tmp: %s\n", path);
        return -EACCES;
    }

    return 0;
}
```

## 11. Common Pitfalls and Best Practices

The verifier and the runtime each impose constraints that look pedantic in isolation but quickly compound when you scale a real eBPF program past a tutorial. Below are the four traps that keep showing up in production code reviews, paired with the patterns that get past them cleanly.

<Warning>
**Pitfall 1: Verifier complexity limits -- programs over roughly one million instructions get rejected**

Older kernels capped verifier complexity at about 4,096 visited instructions; modern kernels (5.10+) raised it to one million. That sounds enormous until you write a real program. Each path the verifier explores counts independently. A nested loop with two branches doubles the path count per iteration. A switch statement over 16 protocols with two branches each adds 32 paths per visit. By the time you write a halfway-realistic packet parser, you can hit complexity limits even though the source code is small.

The error message is unhelpful: "BPF program is too large. Processed 1000001 insn." It tells you the verifier gave up but not where. You then have hours of binary-bisection through your program trying to find the function or loop the verifier could not collapse.
</Warning>

<Tip>
**Solution: tail calls, function-by-function verification, and inlining hints**

Three techniques compose well. First, split logically distinct stages into separate eBPF programs and chain them with `bpf_tail_call`. Each program is verified independently, so the per-program complexity budget resets at the boundary. Second, use `__attribute__((noinline))` on helper functions so the verifier treats them as separate units (kernel 5.10+ supports per-function verification). Third, use `bpf_loop` (kernel 5.17+) for bounded iteration -- the verifier specially handles it and avoids unrolling.

```c
// Tail call pattern for staged processing
SEC("xdp/parse_l2")
int parse_l2(struct xdp_md *ctx) {
    parse_ethernet(ctx);
    bpf_tail_call(ctx, &prog_array, STAGE_PARSE_L3);
    return XDP_PASS;  // fallthrough if tail call unavailable
}

SEC("xdp/parse_l3")
int parse_l3(struct xdp_md *ctx) {
    parse_ip(ctx);
    bpf_tail_call(ctx, &prog_array, STAGE_PARSE_L4);
    return XDP_PASS;
}

// bpf_loop pattern for bounded iteration without unrolling
static long process_one(u32 i, void *ctx_ptr) {
    struct my_ctx *c = ctx_ptr;
    /* ... do the work ... */
    return 0;
}

SEC("xdp")
int main_prog(struct xdp_md *ctx) {
    struct my_ctx c = { .ctx = ctx };
    bpf_loop(64, process_one, &c, 0);  // iterate up to 64 times
    return XDP_PASS;
}
```

Diagnostic: when you hit the limit, run the loader with `BPF_F_LOG_LEVEL=2` to dump the verifier log. The verifier prints which instruction it was processing when complexity ran out -- that points you at the function to split.
</Tip>

<Warning>
**Pitfall 2: 512 bytes of stack -- larger state belongs in maps, not locals**

eBPF programs have exactly 512 bytes of stack. That is not a soft limit; the verifier rejects any allocation that would exceed it. The trap: you write a program that parses TCP options into a local struct, then add a few SACK blocks, then add timestamp captures, and quietly cross 512 bytes. The verifier rejects with "combined stack size exceeds 512 bytes" and your only fix is restructuring data flow.

A second variant: large local arrays compiled with `-O0` or `-O1` end up on the stack instead of being optimized away. A 256-byte local string buffer for a packet payload, plus a 200-byte parsing context, plus your function's normal locals, and you are over budget. Optimization flags matter more than usual in eBPF because the compiler shapes what the verifier sees.
</Warning>

<Tip>
**Solution: per-CPU array maps as scratch space**

A single-entry per-CPU array map gives you a "thread-local" scratch buffer outside the stack. It costs one map lookup, but it has no size limit and is private to the executing CPU.

```c
struct {
    __uint(type, BPF_MAP_TYPE_PERCPU_ARRAY);
    __type(key, u32);
    __type(value, struct big_state);  // can be many KB
    __uint(max_entries, 1);
} scratch SEC(".maps");

SEC("xdp")
int handler(struct xdp_md *ctx) {
    u32 key = 0;
    struct big_state *state = bpf_map_lookup_elem(&scratch, &key);
    if (!state) return XDP_PASS;  // verifier requires the NULL check

    // Use state for parsing, accumulating, etc. -- no stack pressure
    parse_packet(ctx, state);
    return process(state);
}
```

The pattern doubles as a way to share state between tail-called programs: each stage looks up the same per-CPU scratch entry, accumulates into it, and the next stage sees what the previous wrote. Always check the lookup result for NULL even if you know the key exists -- the verifier requires it because the map could in principle be empty.

Always compile eBPF programs with `-O2`. Without it, the C compiler keeps locals the optimizer would have eliminated, and the verifier sees stack frames you did not intend.
</Tip>

<Warning>
**Pitfall 3: Tail calls vs function calls -- the lifecycle semantics are not the same**

`bpf_tail_call` and a regular function call look similar -- both transfer control to other code -- but they differ in fundamental ways that bite when you mix them. Tail calls do not return: the calling program's stack is replaced, not extended. Tail calls are limited (33 deep currently) and the limit is shared across the whole call chain, not per-program. Tail calls cannot pass arguments other than the original program context; everything else must go through maps.

The trap: engineers treat tail calls like function calls and write recursive-looking code. Three layers in, they hit the tail call limit. Or they assume that registers persist across the boundary, write code that depends on a value, and the value silently zeroes. Or they tail-call into a program that has different attach-type expectations and the kernel rejects the call at runtime, returning 0 from `bpf_tail_call` -- which the calling program often does not check.
</Warning>

<Tip>
**Solution: pick the right primitive, and check the return value**

Use a regular function call for "split this code for verifier complexity reasons" -- the function returns and the calling stack continues. Use a tail call for "transition to a new program with a fresh complexity budget and possibly a different attach point." Treat tail calls as a one-way ticket: anything that needs to come back must go through a map.

```c
// Function call -- regular, returns, shares stack budget
__attribute__((noinline))
static int parse_tcp(struct xdp_md *ctx, struct flow *out) {
    /* ... */
    return 0;
}

// Tail call -- one-way, fresh program, must check failure
SEC("xdp")
int dispatcher(struct xdp_md *ctx) {
    u8 protocol = sniff_protocol(ctx);
    bpf_tail_call(ctx, &handlers, protocol);
    // If we reach here, the tail call FAILED (no program registered, or limit hit).
    // Always have a fallthrough.
    return XDP_PASS;
}
```

Check tail call depth with `bpf_get_current_task` plus instrumentation if you suspect chains are growing. For complex stage machines, a `BPF_MAP_TYPE_PROG_ARRAY` indexed by a state field gives you cleaner control flow than nested tail calls -- each stage tail-calls based on the state map, instead of each stage hardcoding the next.
</Tip>

<Warning>
**Pitfall 4: bpf_probe_read for kernel pointers -- older kernels need exception handling**

Reading kernel memory in eBPF is dangerous because the address could be invalid (NULL, freed, or otherwise unmapped). `bpf_probe_read` is the safe primitive: it uses kernel exception handling to catch faults and return an error rather than crash. This is essential because eBPF programs run in atomic context where a kernel oops would take down the host.

The trap is version-specific. On kernels older than 5.5, there was a single `bpf_probe_read` for both user and kernel addresses, with the helper guessing which based on flags. This was fragile; a NULL kernel pointer dereference (which should return an error) sometimes accessed user memory at the same address (silently succeeding with garbage). Kernel 5.5 split it into `bpf_probe_read_kernel` and `bpf_probe_read_user`, removing the ambiguity. CO-RE programs that want to support old and new kernels need to handle this transition.

Newer kernels also added `bpf_core_read` (the CO-RE-aware variant that handles BTF relocations) and direct kernel memory reads via `BPF_PROG_TYPE_TRACING` (with the verifier proving the access is safe). Each has subtly different semantics and error behavior.
</Warning>

<Tip>
**Solution: use the right helper for the kernel version, default to BPF_CORE_READ for tracing**

For modern code (kernel 5.5+), use `bpf_probe_read_kernel` and `bpf_probe_read_user` explicitly -- never the deprecated `bpf_probe_read`. For tracing programs that need to traverse kernel structures portably, use `BPF_CORE_READ` from libbpf, which combines safe access, type relocation via BTF, and helpful error returns:

```c
#include <bpf/bpf_core_read.h>

SEC("kprobe/tcp_sendmsg")
int trace_tcp_send(struct pt_regs *ctx) {
    struct sock *sk = (struct sock *)PT_REGS_PARM1(ctx);

    // Old way (works on 5.5+ but no CO-RE relocation)
    u16 sport;
    bpf_probe_read_kernel(&sport, sizeof(sport), &sk->sk_num);

    // Modern, portable way: CO-RE handles field offset relocations
    u16 sport_core = BPF_CORE_READ(sk, sk_num);
    u32 saddr = BPF_CORE_READ(sk, sk_rcv_saddr);

    bpf_printk("sport=%u saddr=%u\n", sport_core, saddr);
    return 0;
}
```

For supporting really old kernels (4.x), libbpf provides `bpf_core_read` that falls back to `bpf_probe_read` and runtime relocation. Check the loader output -- libbpf logs which probe variant it selected, and that tells you whether you are getting the modern path.

Always check the return value of probe reads. A non-zero return means the read failed (page not mapped, address invalid). Most production bugs around `bpf_probe_read` are not crashes but silent data corruption from unchecked partial reads.
</Tip>

### 11.1 Verifier Rejection Issues

**Problem: Unbounded loops**
```c
// BAD - Verifier rejects
for (int i = 0; i < data_len; i++) {  // data_len unknown!
    process(data[i]);
}

// GOOD - Bounded loop
#define MAX_LOOP 100
for (int i = 0; i < MAX_LOOP && i < data_len; i++) {
    process(data[i]);
}
```

**Problem: Unverified NULL pointer**
```c
// BAD - Verifier rejects
value = bpf_map_lookup_elem(&map, &key);
*value += 1;  // Verifier: "R1 could be NULL!"

// GOOD - Check for NULL
value = bpf_map_lookup_elem(&map, &key);
if (!value)
    return 0;
*value += 1;  // Verifier: OK
```

**Problem: Invalid memory access**
```c
// BAD - Verifier rejects
char *str = (char *)ctx->arg1;
if (str[100] == 'x')  // Unchecked read!
    return 1;

// GOOD - Use helper
char str[101];
bpf_probe_read_kernel_str(str, sizeof(str), (char *)ctx->arg1);
if (str[100] == 'x')
    return 1;
```

### 11.2 Performance Best Practices

**Use per-CPU maps to avoid contention:**
```c
// BAD - Lock contention on shared map
struct {
    __uint(type, BPF_MAP_TYPE_HASH);
    // ...
} shared_map SEC(".maps");

// GOOD - No locks needed
struct {
    __uint(type, BPF_MAP_TYPE_PERCPU_HASH);
    // ...
} percpu_map SEC(".maps");
```

**Minimize helper calls:**
```c
// BAD - Helper call in loop
for (int i = 0; i < 10; i++) {
    u64 ts = bpf_ktime_get_ns();  // Expensive!
    process(ts);
}

// GOOD - Call once
u64 ts = bpf_ktime_get_ns();
for (int i = 0; i < 10; i++) {
    process(ts);
}
```

**Use tail calls for complex logic:**
```c
// Program exceeds instruction limit?
// Split into multiple programs:

SEC("xdp")
int stage1(struct xdp_md *ctx) {
    // Process part 1
    bpf_tail_call(ctx, &prog_array, 1);  // Jump to stage2
    return XDP_PASS;  // Fallback if tail call fails
}

SEC("xdp")
int stage2(struct xdp_md *ctx) {
    // Process part 2
    return XDP_PASS;
}
```

## 13. Interview Questions

<AccordionGroup>
  <Accordion title="Q1: What is the 'BTF' (BPF Type Format) and why is it important for CO-RE?">
    **BTF (BPF Type Format)** is a compact metadata format that describes the structure of kernel types. It's crucial for **CO-RE (Compile Once – Run Everywhere)** because:

    1. **Problem**: Kernel structures like `task_struct` change offsets between kernel versions
    2. **Solution**: BTF provides type information at runtime
    3. **Process**:
       - Developer compiles BPF program with relocations
       - libbpf reads BTF from `/sys/kernel/btf/vmlinux` at load time
       - Field accesses are patched with correct offsets for the running kernel

    **Example**: Access `task->pid` works on any kernel 4.18+ without recompiling, even though the offset of `pid` field varies.

    Without BTF/CO-RE, you'd need kernel headers and recompilation for each kernel version.
  </Accordion>

  <Accordion title="Q2: Explain the difference between XDP and standard Linux Networking.">
    **Standard Linux Networking:**
    - Packet arrives → Driver allocates `sk_buff` → Copies packet data → Network stack processing
    - `sk_buff` allocation is expensive (metadata, cloning, refcounting)
    - Processes through netfilter, routing, TCP stack
    - Throughput: ~1-5M pps per core

    **XDP (eXpress Data Path):**
    - Packet arrives → eBPF runs on raw DMA buffer → Decision (PASS/DROP/TX/REDIRECT)
    - No `sk_buff` allocation for dropped/redirected packets
    - Runs before any kernel processing
    - Throughput: ~25M pps per core (DROP), ~20M pps (TX)

    **Use cases where XDP wins:**
    - DDoS mitigation (drop at wire speed)
    - Load balancing (redirect without stack)
    - Packet filtering (faster than iptables)

    **When to use standard networking:**
    - Complex protocol processing
    - Need full TCP stack features
    - Application-level filtering
  </Accordion>

  <Accordion title="Q3: How does eBPF handle 'Tail Calls' and why are they useful?">
    **Tail Calls** allow one eBPF program to jump to another, replacing its execution context:

    ```c
    // Jump to program at index 1
    bpf_tail_call(ctx, &prog_array, 1);
    // This line never executes if tail call succeeds
    ```

    **How they work:**
    - Current program context is replaced (not a function call)
    - Same stack frame is reused
    - R1 (context pointer) preserved
    - No return to original program

    **Why useful:**
    1. **Overcome complexity limit**: Split large program into smaller pieces
    2. **Dynamic behavior**: Load different "stages" based on packet type
    3. **Verifier happiness**: Each program verified independently

    **Limitations:**
    - Max chain depth: 33 tail calls
    - Overhead: ~10-20ns per tail call
    - Debugging harder (no single stack trace)

    **Example use**: Packet parser with per-protocol handlers (stage1: L2 → stage2: IPv4 → stage3: TCP)
  </Accordion>

  <Accordion title="Q4: What are the main differences between kprobes, tracepoints, and fentry/fexit?">
    **Kprobes:**
    - Attach to any kernel function dynamically
    - Uses int3 breakpoint mechanism (slow)
    - Unstable: function signatures can change
    - Access args via `PT_REGS_PARM` macros (architecture-specific)
    - Overhead: ~1-2μs

    **Tracepoints:**
    - Predefined stable hooks in kernel code
    - Compiled into kernel (no breakpoint)
    - Stable ABI: safe across kernel versions
    - Structured arguments via BTF
    - Overhead: ~500-800ns
    - Limited to existing tracepoints (~1500 in kernel)

    **Fentry/Fexit (5.5+):**
    - Attach to any kernel function (like kprobes)
    - Uses function trampolines (no breakpoint)
    - Type-safe argument access via BTF
    - Fexit can access return value
    - Overhead: ~200-400ns
    - Requires BTF and CONFIG_DEBUG_INFO_BTF

    **Decision matrix:**
    - Production + stability needed → Tracepoints
    - Newest kernels + performance → Fentry/Fexit
    - Broad compatibility → Kprobes
  </Accordion>

  <Accordion title="Q5: How does the eBPF verifier prevent infinite loops while allowing bounded loops?">
    The verifier uses **static analysis** to ensure termination:

    **Pre-5.3 (No loops allowed):**
    - Verifier built DAG (Directed Acyclic Graph)
    - Any back-edge rejected
    - Solution: Loop unrolling via `#pragma unroll`

    **Post-5.3 (Bounded loops):**
    1. **Loop detection**: Identify back-edges in CFG
    2. **Iteration tracking**: Track loop counter in register state
    3. **Bounds proving**:
       ```c
       for (i = 0; i < 100; i++) { ... }
       // Verifier tracks: i ∈ [0, 99]
       ```
    4. **Complexity limit**: Max iterations before verifier gives up
    5. **Verification**: Simulates execution paths, pruning equivalent states

    **Requirements for loops:**
    - Loop variable must have provable upper bound
    - Bound must be constant or map value with known range
    - Total complexity stays under 1M instructions

    **Example rejection:**
    ```c
    while (condition) { ... }  // Rejected: no provable bound
    for (i = 0; i < len; i++)  // Rejected: len unknown
    ```

    **Workaround:**
    ```c
    #define MAX 100
    for (i = 0; i < MAX && i < len; i++)  // OK: bounded by MAX
    ```
  </Accordion>

  <Accordion title="Q6: Explain eBPF ring buffers and how they differ from perf buffers.">
    **Perf Buffer (older):**
    ```
    Architecture:
    - Per-CPU circular buffers
    - Each CPU has separate buffer
    - Events can be reordered across CPUs
    - Lost events if buffer full (can't reserve)

    Problems:
    - Memory fragmentation (must size all per-CPU buffers)
    - No ordering guarantees
    - Wasteful on CPU imbalance
    ```

    **Ring Buffer (modern, 5.8+):**
    ```
    Architecture:
    - Single shared buffer (multi-producer, single-consumer)
    - Lock-free using atomics
    - Maintains global ordering
    - Can reserve space before writing

    Advantages:
    ✓ Better memory efficiency
    ✓ Event ordering preserved
    ✓ Reserve/commit model (no lost events)
    ✓ Variable-length records
    ✓ Lower overhead
    ```

    **API difference:**
    ```c
    // Perf buffer
    e = bpf_perf_event_output(ctx, &events, BPF_F_CURRENT_CPU,
                               &data, sizeof(data));

    // Ring buffer
    e = bpf_ringbuf_reserve(&events, sizeof(*e), 0);
    if (e) {
        // fill e
        bpf_ringbuf_submit(e, 0);
    }
    ```

    **When to use ring buffer:** Always, unless kernel is under 5.8
  </Accordion>

  <Accordion title="Q7: How would you debug an eBPF program that passes the verifier but crashes the kernel?">
    This is rare but critical. Debugging steps:

    **1. Check Verifier Logs:**

    **2. Enable Debug Output:**

    **3. Check Kernel Logs:**
    ```bash
    dmesg | tail -100
    # Look for: "BPF:xxx", "RIP:", "Call Trace:"
    ```

    **4. Inspect JIT Code:**
    ```bash
    bpftool prog dump jited id <id>
    # Look for invalid memory access patterns
    ```

    **5. Common Crash Causes:**
    - Helper function called with invalid args (verifier missed edge case)
    - Race condition in map access


    **7. Kernel Debugging:**
    ```bash

    crash vmlinux vmcore
    ```

    **Prevention:**
    - Always check helper return values
    - Use bounded operations
    - Test on multiple kernel versions

  </Accordion>

  <Accordion title="Q8: What is the performance overhead of eBPF tracing compared to alternatives?">
    **Comparison of tracing overhead:**

    **strace (ptrace-based):**
    - Overhead: 100-300x slowdown
    - Mechanism: 2 context switches per syscall
    - Use case: Only for debugging, never production

    **SystemTap:**
    - Overhead: 10-50x slowdown
    - Mechanism: Kernel module with safety checks
    - Use case: Complex analysis, but risky in production

    **perf probe (kprobes):**
    - Overhead: 5-20x for traced functions
    - Mechanism: int3 breakpoint
    - Use case: Short-term analysis

    **eBPF (kprobes):**
    - Overhead: 2-10x for traced functions
    - Mechanism: JIT'd code at breakpoint
    - In-kernel aggregation: under 1% total overhead

    **eBPF (tracepoints):**
    - Overhead: 1-3x for traced functions
    - Mechanism: No breakpoint, direct call
    - In-kernel aggregation: under 0.5% total overhead

    **eBPF (fentry/fexit):**
    - Overhead: under 2x for traced functions
    - Mechanism: Direct function trampoline
    - In-kernel aggregation: under 0.3% total overhead

    **Key factors:**
    1. **Aggregation**: eBPF aggregates in kernel, others send every event to userspace
    2. **Context switches**: eBPF avoids userspace trips
    3. **JIT**: Near-native code execution

    **Real example:**
    ```
    Trace all open() syscalls on busy server:
    - strace: System becomes unusable
    - perf: 20% CPU overhead
    - eBPF: under 1% CPU overhead
    ```

## Interview Deep-Dive

<AccordionGroup>
  <Accordion title="What is the eBPF verifier, why does it exist, and what happens if you try to load a program that reads uninitialized memory?">
    **Strong Answer:**

    - The eBPF verifier is a static analyzer built into the Linux kernel that validates every eBPF program before it is allowed to execute. It exists because eBPF programs run inside the kernel with kernel privileges -- a bug in an eBPF program could crash the kernel, leak sensitive data, or cause infinite loops that hang the system. Unlike a kernel module (which is trusted, written by privileged users), eBPF programs can be loaded by unprivileged users (with appropriate capabilities), so they must be proven safe before execution.
    - The verifier performs several checks: it ensures all branches converge (no infinite loops -- it verifies that every loop is bounded), it tracks the type and bounds of every register (preventing out-of-bounds memory access), it ensures all memory reads go to valid locations (stack, map values, packet data with bounds-checked offsets), and it verifies that all code paths reach a return instruction.
    - If you try to load a program that reads uninitialized memory (say, reading from a stack slot that was never written), the verifier rejects the program at load time with a specific error message indicating which instruction and which register is uninitialized. The program never executes. This is fundamentally different from C/C++ where reading uninitialized memory is undefined behavior that silently produces wrong results or crashes at runtime.
    - The verifier is conservative -- it rejects some programs that would be safe in practice because it cannot prove they are safe statically. This is a common frustration when writing eBPF programs: you know your loop terminates, but the verifier cannot prove it. The workaround is to add explicit bounds checks or restructure the code to make the safety proof obvious to the verifier.

    **Follow-up: How does the eBPF JIT compiler work, and why is it important for performance?**

    After the verifier approves an eBPF program, the JIT (Just-In-Time) compiler translates the eBPF bytecode into native machine instructions for the host architecture (x86, ARM, etc.). Without JIT, the kernel would interpret eBPF bytecode instruction-by-instruction, which is 5-10x slower. With JIT, the eBPF program runs at near-native speed because it is compiled to the same machine code the CPU executes natively. The eBPF register model (11 64-bit registers) is designed to map directly to CPU registers, minimizing register spills. JIT is especially critical for high-frequency attach points like XDP (which runs on every packet) and kprobes on hot kernel functions. On modern kernels, JIT is enabled by default.
  </Accordion>

  <Accordion title="You want to trace all TCP connections that take longer than 100ms to establish on a production server with zero downtime. How would you do this with eBPF?">
    **Strong Answer:**

    - I would use bpftrace or a custom BCC/libbpf program that attaches to the tcp_v4_connect and inet_sock_set_state tracepoints (or kprobes). The approach is: when tcp_v4_connect is called, record the timestamp in an eBPF hash map keyed by the socket pointer. When the socket transitions to the ESTABLISHED state (detected via inet_sock_set_state), look up the start time, calculate the delta, and if it exceeds 100ms, emit an event to user space via a ring buffer or perf event array.
    - The specific implementation with bpftrace would look like: attach to kprobe:tcp_v4_connect to record start time in a map keyed by `tid` (thread ID), then attach to kretprobe:tcp_v4_connect to handle the return, and use the tcp:inet_sock_set_state tracepoint to detect when the state becomes TCP_ESTABLISHED. Calculate the delta and filter for connections exceeding 100ms.
    - This has effectively zero overhead in production because: eBPF programs are JIT-compiled to native code, the hash map lookup is O(1), and the program only runs when a TCP connection event fires (not on every packet). The filtering (>100ms) happens in kernel space, so only slow connections generate events that cross the kernel-user boundary.
    - For production use, I would add: the source and destination IP/port (extracted from the sock struct), the process name and PID (from bpf_get_current_comm and bpf_get_current_pid_tgid), and a stack trace (bpf_get_stackid) to identify which code path initiated the slow connection. I would output to a ring buffer and have a user-space daemon aggregate and send to our monitoring system.

    **Follow-up: What are eBPF maps, and why are they used instead of just passing data through function arguments?**

    eBPF maps are key-value data structures that persist across eBPF program invocations and are shared between eBPF programs and user space. They are necessary because eBPF programs are stateless by themselves -- they run, return a value, and are done. Maps provide the "memory" needed to correlate events across time (like recording a timestamp in one program and reading it in another), aggregate data (like histograms of latency), and communicate with user space (user space reads the map to get results). Common map types include hash maps, arrays, ring buffers (for streaming events to user space), LRU maps (for bounded caches), and per-CPU arrays (for lock-free per-CPU counters). The choice of map type affects performance: per-CPU maps avoid atomic operations but use more memory, while shared hash maps need locking overhead.
  </Accordion>

  <Accordion title="Compare eBPF to kernel modules for extending kernel functionality. When would you still choose a kernel module over eBPF?">
    **Strong Answer:**

    - eBPF is safer (verified before execution, cannot crash the kernel), easier to deploy (no need to compile against kernel headers or reboot), and portable across kernel versions (with CO-RE/BTF). Kernel modules have full, unrestricted access to all kernel APIs and data structures, can allocate arbitrary memory, sleep, and implement complex long-running logic.
    - I would choose a kernel module over eBPF when: (1) I need to implement a new device driver -- eBPF cannot register character devices, handle interrupts, or manage hardware directly. (2) I need to modify kernel behavior that eBPF hooks do not cover -- eBPF can only attach to predefined hook points (kprobes, tracepoints, XDP, tc, LSM hooks). If I need to change how the scheduler works or add a new filesystem, I need a module. (3) I need long-running operations that sleep -- eBPF programs must complete quickly and cannot sleep (they run in restricted execution contexts). (4) I need to allocate large amounts of kernel memory or create kernel threads.
    - eBPF is the right choice for: observability (tracing, profiling, metrics), networking (XDP packet processing, load balancing, firewalling -- Cilium does this), security enforcement (LSM hooks for policy decisions), and any case where I need to safely extend a production kernel without risking a crash.
    - The trend is clear: eBPF is replacing kernel modules for everything except hardware drivers and core kernel modifications. The Linux kernel is moving toward making more functionality implementable via eBPF (like scheduler extensions with sched_ext) to reduce the need for out-of-tree modules.

    **Follow-up: What is CO-RE (Compile Once, Run Everywhere) and how does it solve the eBPF portability problem?**

    Before CO-RE, eBPF programs that accessed kernel data structures needed to be compiled against the specific kernel headers of the target machine, because struct layouts change between kernel versions (field offsets move). CO-RE uses BTF (BPF Type Format) -- type information embedded in the kernel -- to relocate field accesses at load time. You compile your eBPF program once against any kernel's headers, and the loader (libbpf) reads the target kernel's BTF to adjust struct offsets before loading the program. This means a single compiled eBPF binary can run on kernels from 4.18 to 6.x without recompilation, as long as the fields it accesses exist (possibly at different offsets).
  </Accordion>

  <Accordion title="When does eBPF beat a kernel module, and when do you still reach for a kernel module?">
    **Strong Answer Framework:**

    1. **Safety profile.** Kernel modules can crash the kernel; eBPF cannot. The verifier rejects unsafe programs at load time -- no infinite loops, no out-of-bounds memory access, no uninitialized reads, no sleeping in atomic context. For any code path you do not personally trust to be bug-free, eBPF is the safer choice by construction.
    2. **Deployment friction.** Kernel modules require matching the running kernel's symbol versions, often require signed-module enforcement to be temporarily relaxed, and may need a reboot to install (or unload-reload at minimum). eBPF programs are loaded into a running kernel via syscall, no reboot, no kernel rebuild, no symbol compatibility issues with CO-RE. For SaaS products that ship across hundreds of customer kernels, this difference is enormous.
    3. **Capability surface.** Modules can do anything: register character devices, replace syscalls, allocate any kernel memory, sleep at will, hold spinlocks. eBPF programs are limited to predefined hook points (kprobes, tracepoints, XDP, tc, LSM hooks, etc.), bounded execution time, no sleeping, restricted memory access. If you need to extend kernel functionality outside those hooks -- a new filesystem, a new scheduler class, a new networking protocol -- you need a module.
    4. **Performance profile.** eBPF programs are JIT-compiled to native code; the runtime overhead is comparable to native kernel code for the same logic. The pure-overhead delta vs a kernel module doing the same thing is typically under 10 percent and often unmeasurable. Performance is rarely the deciding factor.
    5. **Choose a module when:** you are writing a device driver, you need to register a character device or kobject, you need to extend kernel functionality at a hook eBPF does not expose, you need long-running threads or sleepable code, or you are part of the upstream kernel community shipping a feature for everyone.
    6. **Choose eBPF when:** you are tracing, profiling, monitoring, or filtering. You are implementing networking dataplane logic at XDP/tc speed. You are writing security policy that needs to evolve faster than kernel release cycles. You need to ship across many kernel versions without per-kernel builds.

    **Real-World Example:** Cilium, the Kubernetes networking and security project, replaces kube-proxy and iptables with eBPF programs at the XDP and tc hooks. The 2021 announcement included specific numbers: at scale (10K+ services per cluster), eBPF replaced kube-proxy's iptables rules and reduced p99 connection setup latency from tens of milliseconds to single-digit milliseconds, and reduced CPU consumption per service-pod by an order of magnitude. The same project considered building a kernel module variant; they chose eBPF because Cilium ships into customer clusters running diverse kernel versions and could not realistically maintain modules per kernel. The project simply could not exist as a kernel module at the same fleet scale.

    <Note>
    **Senior follow-up 1: What does sched_ext bring to this comparison?**

    `sched_ext` (kernel 6.12, 2024) lets you implement custom CPU schedulers as eBPF programs rather than as kernel modules. Before sched_ext, scheduler experimentation required patching the kernel and rebuilding -- an expensive iteration loop that meant only a few researchers and Google did it. With sched_ext, you can prototype a scheduler in C, load it into a running kernel, and measure on production-realistic workloads. Meta has used this to ship workload-specific schedulers (CFS-derived but optimized for their server fleet) without committing to upstream-merge timelines. This is the trend: more historically module-only kernel features become eBPF-extensible over time.
    </Note>

    <Note>
    **Senior follow-up 2: Are there security cases where the kernel module is more secure than eBPF?**

    Counterintuitively, yes. A signed kernel module is loaded once at boot by a trusted operator, with all its capabilities up front. An eBPF program is often loaded by less-privileged operators (admin processes, observability daemons) and can attach to a wider set of hooks. If your threat model is "admin user is malicious," eBPF actually expands the attack surface because more code can attach to more places. The mitigation is `kernel.unprivileged_bpf_disabled=1` (default on most distros now) which restricts BPF program loading to root, narrowing the surface back to "as restrictive as module loading." For high-security environments, you also want `kernel.kptr_restrict=2` and proper LSM policies on `BPF_PROG_LOAD`.
    </Note>

    <Note>
    **Senior follow-up 3: Why does the Linux kernel community prefer eBPF over modules for new observability features?**

    Three reasons. First, ABI stability: kernel modules are tightly coupled to internal kernel APIs that change every release; out-of-tree modules become a maintenance treadmill. eBPF programs use a stable ABI (the BPF helpers and stable hook points) that the kernel team commits to maintaining. Second, distribution: a kernel feature that requires an out-of-tree module never reaches the long tail of users who run distro kernels. eBPF features ship the day the kernel does. Third, accountability: an unsafe module crashes the host with no easy recovery; an unsafe eBPF program is rejected at load. The kernel team prefers mechanisms that fail safe.
    </Note>

    **Common Wrong Answers:**
    - "eBPF is always faster than modules." It is not. JIT-compiled eBPF is comparable to native kernel C, sometimes a few percent slower because of the verifier-imposed code structure. Speed is rarely the discriminator.
    - "Use eBPF for everything." Many kernel features cannot be expressed in eBPF: device drivers, new syscalls, new filesystem types. Knowing where the line is matters more than picking a side.
    - "Kernel modules are dead." They are not. Every NIC driver, every storage driver, every kernel subsystem ships as a module. The honest framing is "eBPF replaces a particular subset of historical module use cases -- observability, networking dataplane, security policy -- not all of them."

    **Further Reading:**
    - "BPF: A New Type of Software" (Brendan Gregg) -- the definitive framing of where eBPF fits
    - The eBPF.io documentation, particularly "What is eBPF?" overview
    - Cilium architecture documents -- specifically the "Why eBPF" section explaining the kube-proxy replacement decision
    - Linux source: `Documentation/bpf/` and `kernel/sched/ext.c` for the sched_ext example
  </Accordion>

  <Accordion title="Walk me through using bpftrace to diagnose a process that has gotten mysteriously slow. What probes do you reach for first, and how do you narrow it down?">
    **Strong Answer Framework:**

    1. **Define "slow" before you trace.** Is it CPU-bound, I/O-bound, or stalled? `top` shows CPU; `iotop` and `/proc/[pid]/io` show I/O bytes; off-CPU time (the time the thread spent not running) is what matters for stalls and is harder to see. The first measurement decides which family of probes to use.
    2. **CPU-bound: profile on-CPU stacks.** `bpftrace -e 'profile:hz:99 /pid == $1/ { @[kstack, ustack] = count(); }' <pid>` samples the kernel and user stack at 99 Hz for the target PID. After 30 seconds, the most-frequent stacks tell you where the process is burning CPU. Pipe through `flamegraph.pl` for a visualization. This is the same data Brendan Gregg's CPU flame graphs are built from, just collected with one bpftrace one-liner instead of perf record + post-processing.
    3. **I/O-bound: trace block layer.** `bpftrace -e 'tracepoint:block:block_rq_issue /pid == $1/ { @[comm] = count(); }' <pid>` counts block requests issued by the target. For latency, `tracepoint:block:block_rq_complete` minus `tracepoint:block:block_rq_issue` keyed by request gives per-IO duration histograms. This catches the case where a process is making lots of small synchronous reads against slow storage.
    4. **Stalled (off-CPU): trace scheduler events.** `bpftrace -e 'kprobe:finish_task_switch /args->prev->pid == $1/ { @start[args->prev->pid] = nsecs; } kfunc:try_to_wake_up /args->p->pid == $1 && @start[$1]/ { @offcpu_us = hist((nsecs - @start[$1]) / 1000); delete(@start[$1]); }' <pid>` measures how long the process slept off-CPU each time. A histogram with a fat tail in milliseconds means the process is waiting on something -- a lock, an I/O, a network response.
    5. **Narrow by syscall.** Once you know the family, `bpftrace -e 'tracepoint:syscalls:sys_enter_* /pid == $1/ { @[probe] = count(); }' <pid>` tells you which syscalls the process is making heaviest. Add latency tracking with `sys_exit` to find which syscalls are slow. Tools like `funclatency` and `syscount` from BCC do exactly this.
    6. **Drill into the slow syscall.** If `read` is slow, you want filesystem-level traces: `vfs_read` latency, `ext4_file_read_iter` latency, page cache hit rate. If `futex` is slow, you have a lock contention issue and `bpftrace -e 'kprobe:futex_wait { @ustack[ustack] = count(); }'` shows which userspace stacks are waiting. If `connect` is slow, network latency or DNS.

    **Real-World Example:** Netflix's published Performance team workflow (Brendan Gregg's blog from 2019-2020) walks through diagnosing a JVM that mysteriously slowed down. The first bpftrace pass showed off-CPU time clustered in `futex_wait`. The second pass captured user-space stacks and revealed the wait was inside JVM safepoint synchronization. The third pass showed safepoints were triggered by a frequent GC-related lock acquisition. Total diagnostic time: under 10 minutes of bpftrace on a production host, no JVM restart, no debug build. Without eBPF, the same investigation would have required a dev-environment reproduction with JVM flight recording, which Netflix had previously done -- and which took days.

    <Note>
    **Senior follow-up 1: What is the cost of running these probes in production, and how do you decide if it is acceptable?**

    Profile probes at 99 Hz cost roughly 1-2 percent CPU per core sampled. Tracepoints that fire on every syscall cost 50-200 ns per syscall, which translates to 0.5-2 percent for typical workloads (10K-50K syscalls/sec) and 5-15 percent for syscall-heavy workloads. Block tracepoints are fine because block rate is low. The decision rule: short-duration profiling (under 60 seconds) is almost always acceptable in production. Long-duration always-on tracing needs careful attention to which probes are attached -- always-on `sys_enter_*` is a bad idea on a high-syscall service, but always-on `block:block_rq_complete` is fine. When in doubt, measure the overhead with the probe attached vs detached during a quiet window.
    </Note>

    <Note>
    **Senior follow-up 2: When does perf still beat bpftrace, and vice versa?**

    `perf record` excels at one-shot CPU profiling for offline flame graph generation -- it has decade-old optimizations for stack walking, symbol resolution, and the perf.data format integrates with mature post-processing. bpftrace excels at custom queries that combine multiple probes with custom aggregation. If you want "histogram of malloc sizes filtered by call site," perf cannot do that without significant scripting; bpftrace expresses it in one line. If you want a 10-minute CPU profile across the whole system, perf is more efficient and the post-processing tooling is richer. The honest answer in 2026: I use perf for CPU profiling and bpftrace for everything else.
    </Note>

    <Note>
    **Senior follow-up 3: What is `retsnoop` and when do you reach for it over plain bpftrace?**

    `retsnoop` is a tool that captures the full kernel call stack for any kernel function returning a specific error, with arguments and return values for each frame. It uses BPF kfuncs and BTF to walk stack frames programmatically -- something you cannot easily express in bpftrace. The use case: "this syscall returns -EINVAL sometimes; tell me exactly which kernel function originated the error." Plain bpftrace can attach to syscall entry and exit, but it cannot tell you the chain of internal calls between them. retsnoop fills that gap and is the right tool for "the kernel said no and I do not know why."
    </Note>

    **Common Wrong Answers:**
    - "Just run strace." strace is fine for low-rate, single-process syscall tracing, but it serializes the target via ptrace, slowing it 10-100x and changing timing. Many bugs disappear under strace.
    - "Use perf record for everything." perf record samples on-CPU stacks; it cannot trace off-CPU time, lock contention, or specific kernel events without additional tooling. It is one tool among several.
    - "Reproduce it in dev with a debugger." Sometimes correct, often not -- production issues frequently depend on production traffic patterns or hardware. eBPF's value is that you do not have to reproduce; you can observe in place.

    - The bpftrace one-liners reference: github.com/iovisor/bpftrace/blob/master/docs/tutorial_one_liners.md
    - Brendan Gregg's blog, especially "Linux Performance Analysis in 60 Seconds" and the off-CPU flame graph posts
    - Andrii Nakryiko's `retsnoop` tool: github.com/anakryiko/retsnoop

  <Accordion title="The verifier just rejected my eBPF program. Walk me through the common reasons and how to fix each one.">
    **Strong Answer Framework:**

    1. **First, get the actual error.** The verifier prints a detailed log if you ask for it. With libbpf, set `BPF_F_LOG_LEVEL=2` or pass `log_level=2` to the loader. With bpftool, use `log_level 2`. The log shows the instruction the verifier was on, the register state at that point, and the specific safety property that failed. Reading verifier output is a learnable skill -- the format is consistent and the messages, once you have seen each one twice, become diagnostic.
    2. **"R1 unbounded memory access" -- you read past a known-bounded buffer.** Common when parsing packets: you check `data + offset less than data_end`, then read more bytes than the check covered. Fix: re-check after every advance. The verifier requires the check to be visible in the register's known bounds at the read site; a check earlier in the function does not propagate if there is any branching between.
    3. **"R3 type=map_value_or_null expected=map_value" -- you forgot the NULL check on a map lookup.** `bpf_map_lookup_elem` returns NULL when the key is missing. Fix: explicit `if (!value) return 0;` immediately after the lookup. The verifier tracks pointer types and refuses to dereference a value that could be NULL.
    4. **"R5 type=inv expected=fp" -- you passed a non-stack pointer where a stack pointer was expected.** Helpers like `bpf_probe_read` require the destination to be on the stack so the verifier can prove the bounds. Fix: copy into a stack local first, or use a per-CPU map and read into the map value directly with helpers that accept map pointers.
    5. **"BPF program is too large. Processed 1000001 insn" -- complexity limit hit.** The verifier explored too many paths. Fix: split into tail-called sub-programs, mark helper functions `__attribute__((noinline))` so the verifier handles them as separate units, replace unrolled loops with `bpf_loop` (kernel 5.17+), or simplify control flow to reduce the path explosion.
    6. **"invalid bpf_context access" -- you accessed a field of the program context that is not allowed for this attach type.** XDP context exposes a different set of fields than tc context, even though they look similar. Fix: check the kernel's BTF for what is actually accessible at your attach point. Tools like `bpftool btf dump` show the accessible field set.
    7. **"R0 has unknown scalar with mixed signed bounds" -- the verifier cannot prove a value is non-negative.** Common with arithmetic on values returned from helpers. Fix: explicit bounds checks (`if (val less than 0 || val greater than MAX) return 0;`) so the verifier's bound tracking narrows the value's possible range.
    8. **"call to xxx is not allowed for program type yyy" -- the helper is not exposed for your program type.** Each program type (XDP, tc, kprobe, tracepoint, etc.) has its own set of allowed helpers. Fix: pick the right program type for the operation, or restructure to do the privileged operation in a different program connected via a map.

    **Real-World Example:** Cloudflare's writeup on debugging eBPF programs (2021) describes a specific case where their L7 load balancer's XDP program started failing the verifier after a kernel upgrade. The error was "R6 invalid mem access" on a packet pointer dereference that had worked for years. Root cause: a kernel patch tightened the verifier's tracking of packet pointer arithmetic, requiring an additional bounds check after a specific advance pattern. Fix took 30 minutes once they read the verifier log carefully; without the log, they had to bisect kernel versions to find the change. The team's takeaway, which they shared publicly: always treat verifier output as the primary debugging signal, not source-level reasoning about whether the code "should be safe."

    <Note>
    **Senior follow-up 1: How does the verifier track register bounds, and what are the limits of its analysis?**

    The verifier maintains per-register min and max bounds that narrow as the program executes through branches. After `if (x less than 100)`, the true branch knows `x` is in `[INT_MIN, 99]`. Bitwise ops like `x and 0xFF` collapse the bounds to `[0, 255]`. The limits show up around: arithmetic that could overflow (the verifier widens bounds to be conservative), values returned from helpers (often unbounded), and any operation the verifier's symbolic execution cannot model. Modern kernels have improved this -- the 5.4 kernel introduced "scalar precision tracking" which propagates exact values through more operations -- but pathologic cases still defeat analysis. The fix when you hit one is always the same: insert an explicit bounds check that narrows the value the verifier sees.
    </Note>

    <Note>
    **Senior follow-up 2: Why do verifier rejections sometimes happen on a kernel upgrade for a program that worked previously?**

    Two reasons, both legitimate. First, the verifier tightens checks over time as new bug classes are discovered. If the previous verifier was permissive in a way that allowed unsafe programs, the new verifier rejects programs that relied on that permissiveness -- even if those specific programs were not actually unsafe. Second, BTF and CO-RE rely on kernel data structure layout; if a struct field moves or changes type, a CO-RE program that used to relocate cleanly may fail. The fix is usually small (add a bounds check, update a CO-RE relocation hint), but it can be surprising. The lesson: pin your verifier-tested kernels in CI and run the verifier against them as a release-gate.
    </Note>

    <Note>
    **Senior follow-up 3: What does `bpf_assume` or its modern equivalent give you, and when should you use it?**

    On recent kernels, the `__builtin_assume` hint or libbpf's pattern of explicit `if (val greater than MAX) val = MAX;` (range narrowing) lets you give the verifier hints that an invariant holds. The verifier cannot prove arbitrary invariants on its own, but if you tell it "this value is at most 64," it will accept reads that depend on the bound. Use this when you genuinely have an out-of-band guarantee (a constant from a map you control, a value derived from kernel state with a known cap). Do not use it as a workaround -- if the assumption fails, you are back to the safety problems eBPF was supposed to prevent. The discipline: every assumption gets a comment explaining why it must hold.
    </Note>

    **Common Wrong Answers:**
    - "Add `__attribute__((unused))` and the verifier will skip it." There is no such escape hatch. The verifier checks every instruction the program executes; you cannot annotate it away.
    - "Just turn off the verifier." You cannot. The verifier is the safety boundary that lets eBPF run in the kernel at all. Disabling it would re-create the kernel-module risk profile without the kernel-module debugging tools.
    - "Compile with `-O0` to make it simpler." Often makes things worse. `-O0` produces verbose code with redundant locals that exceed stack budget and confuse the verifier's bounds tracking. Always use `-O2` for eBPF.
