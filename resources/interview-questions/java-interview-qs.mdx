## 1. Core Java Fundamentals

<Accordion title="1. JDK vs JRE vs JVM">
**Answer**:
The way I think about this is in layers — each one wraps the next.

*   **JVM (Java Virtual Machine)**: The execution engine at the bottom of the stack. It takes compiled bytecode (`.class` files) and interprets or JIT-compiles it into native machine instructions. The JVM is what makes Java "write once, run anywhere" — there are platform-specific JVM implementations for Windows, Linux, macOS, etc. It handles memory management, garbage collection, and the thread scheduler. Important: the JVM spec is an open standard — HotSpot (Oracle/OpenJDK), GraalVM, Eclipse OpenJ9, and Azul Zing are all different implementations with different performance characteristics.
*   **JRE (Java Runtime Environment)**: JVM + the standard class libraries (`java.lang`, `java.util`, `java.io`, etc.) + supporting files. This is everything you need to *run* a Java application but not to *build* one. Before Java 11, you could download a standalone JRE. Post-Java 11, Oracle stopped shipping separate JRE downloads — you use `jlink` to create custom runtime images instead.
*   **JDK (Java Development Kit)**: JRE + development tools — the compiler (`javac`), debugger (`jdb`), archiver (`jar`), documentation generator (`javadoc`), and since Java 9, the `jshell` REPL. In modern Java (11+), the JDK effectively *is* the distribution since standalone JREs are no longer shipped.

**Real-world nuance**: In containerized deployments, teams use tools like `jlink` to strip unused modules and create minimal runtime images — reducing Docker image size from ~300MB (full JDK) to ~40-80MB. This matters at scale when you have hundreds of microservices.

**What interviewers are really testing:** Whether you understand the layered architecture and can articulate why each layer exists — not just memorized definitions. Senior candidates should mention JVM implementations (HotSpot vs GraalVM), the post-Java-11 JRE change, or `jlink` for container optimization.

**Red flag answer:** "JDK is for development, JRE is for running, JVM is the virtual machine" — this is correct but surface-level and shows no real understanding of why the distinction matters or how it impacts deployment.

**Follow-up:**
1. "If your production Docker image is 400MB and ops wants it under 100MB, how do you slim it down without switching languages?"
2. "What's the difference between HotSpot and GraalVM, and when would you choose one over the other?"
3. "What changed about JRE distribution after Java 11, and how does `jlink` fit in?"
</Accordion>

<Accordion title="2. OOP Concepts in Java">
**Answer**:
Java's OOP model rests on four pillars, but what separates a strong answer from a textbook one is showing *why* each exists and *where* they break down.

*   **Encapsulation**: Bundling data (fields) with the methods that operate on that data, and restricting direct access via access modifiers (`private`, `protected`, `public`). The real purpose is to protect invariants — for example, a `BankAccount` class ensures the balance never goes negative by forcing all mutations through a `withdraw()` method that validates. The gotcha: Java's encapsulation is easily broken by reflection (`Field.setAccessible(true)`), which frameworks like Hibernate and Spring use heavily. Java 9+ modules (`module-info.java`) add a stronger encapsulation boundary.
*   **Inheritance**: The `extends` keyword creates an IS-A relationship. `super` calls parent constructors/methods. The key trade-off is **reuse vs coupling** — inheriting from a concrete class tightly couples you to its implementation. This is why the mantra "favor composition over inheritance" exists (Effective Java, Item 18). In practice, most experienced teams limit inheritance to 1-2 levels deep and prefer interfaces + delegation.
*   **Polymorphism**: Comes in two flavors. **Compile-time** (method overloading — same name, different parameter types) is resolved by the compiler. **Runtime** (method overriding — subclass provides different implementation) is resolved via the vtable at runtime. Runtime polymorphism is the one that matters for design — it enables the Strategy pattern, dependency injection, and basically all of Spring's proxy-based architecture.
*   **Abstraction**: Hiding implementation details and exposing only the contract. Achieved via interfaces (pure contract) and abstract classes (partial implementation + contract). Since Java 8, interfaces can have `default` methods, blurring the line. The rule of thumb: use interfaces for capabilities ("can do" — `Serializable`, `Comparable`), abstract classes for shared state/behavior in a type hierarchy.

**Real-world example**: In a payment processing system, you'd have a `PaymentProcessor` interface with implementations like `StripeProcessor`, `PayPalProcessor`, `BraintreeProcessor`. Runtime polymorphism lets you swap processors without changing calling code. Encapsulation ensures each processor's API keys and internal state are hidden. This is OOP earning its keep in production.

**What interviewers are really testing:** Can you go beyond definitions? Do you understand *when inheritance is harmful*? Can you articulate the composition-over-inheritance principle with a real scenario? Senior candidates should bring up SOLID principles naturally.

**Red flag answer:** Listing the four pillars without any nuance about trade-offs, or claiming "inheritance promotes code reuse" without mentioning the coupling problem.

**Follow-up:**
1. "Give me a real example where inheritance was the wrong choice and composition would have been better."
2. "How does runtime polymorphism actually work at the JVM level? What's a vtable?"
3. "Java doesn't support multiple inheritance of state — how do default methods in interfaces work around this, and what's the diamond problem?"
</Accordion>

<Accordion title="3. String vs StringBuilder vs StringBuffer">
**Answer**:
This question is really about immutability, thread safety, and performance trade-offs — three themes that recur throughout Java.

*   **String**: Immutable — once created, the character sequence never changes. Every "modification" creates a new `String` object. Strings are interned in the **String Pool** (part of the heap since Java 7, previously in PermGen). Because they're immutable, they're inherently thread-safe, safe as HashMap keys, and can be cached/shared freely. The gotcha: concatenating Strings in a loop (`s += "x"`) creates O(n) intermediate objects, leading to O(n^2) time complexity. The compiler optimizes simple concatenation (`"a" + "b"`) to `StringBuilder` calls since Java 5, and since Java 9 uses `invokedynamic`-based `StringConcatFactory` for even better optimization.
*   **StringBuilder**: Mutable character sequence backed by a resizable `char[]` (or `byte[]` since Java 9 with Compact Strings). Not thread-safe — no synchronization. This is the right choice 99% of the time when you need to build strings dynamically (loops, complex formatting). Default capacity is 16 characters; if you know the approximate size, pass it to the constructor to avoid resizing: `new StringBuilder(256)`.
*   **StringBuffer**: Identical API to `StringBuilder` but with `synchronized` methods. Introduced in Java 1.0 before `StringBuilder` existed. In practice, you almost never need `StringBuffer` — if multiple threads are building a string, you likely have a design problem. I've seen `StringBuffer` in codebases maybe twice in 10 years, and both times it was legacy code that should have been refactored.

**Performance numbers**: In a benchmark building a 100K-character string, `StringBuilder` typically runs in ~1ms, `StringBuffer` in ~3-5ms (synchronization overhead), and naive `String` concatenation in ~5000ms+ due to object churn and GC pressure.

**What interviewers are really testing:** Do you understand *why* immutability matters (thread safety, caching, security for class loading)? Can you explain the performance implications with specifics? Senior candidates should mention Compact Strings (Java 9), `StringConcatFactory`, or initial capacity sizing.

**Red flag answer:** "StringBuffer is thread-safe, StringBuilder is not, so use StringBuffer when you need thread safety." — This is technically correct but shows no judgment. The real answer is that you almost never want StringBuffer.

**Follow-up:**
1. "Why are Strings stored in a pool, and what are the implications for `==` vs `.equals()` on Strings?"
2. "What changed in Java 9 with Compact Strings, and why does it matter for memory?"
3. "If I'm processing 10 million log lines and building a report string, what's my strategy?"
</Accordion>

<Accordion title="4. equals() vs ==">
**Answer**:
This is one of the most common sources of bugs in Java, and the interviewer wants to see if you understand *why* the distinction exists and the contract that comes with `equals()`.

*   **`==` (Reference equality)**: Compares memory addresses. For primitives, it compares values (since primitives aren't objects). For objects, it answers: "Do these two references point to the exact same object on the heap?" This is almost never what you want when comparing objects semantically.
*   **`equals()` (Logical equality)**: Defined in `Object`, defaults to `==` (reference comparison). You override it to define what "same content" means for your class. For example, two `User` objects with the same `id` and `email` should be `equals()` even if they're different instances.

**The equals/hashCode contract** (this is critical):
If you override `equals()`, you **must** override `hashCode()`. The contract: if `a.equals(b)` is true, then `a.hashCode() == b.hashCode()` must be true. Violating this breaks `HashMap`, `HashSet`, and any hash-based collection — objects become "unfindable" after being inserted. I've debugged production issues where a missing `hashCode()` override caused a cache to grow unbounded because "equal" objects kept getting re-inserted.

**The five properties of `equals()`**: Reflexive, Symmetric, Transitive, Consistent, and null returns false. Getting symmetry wrong across class hierarchies is a classic trap — `Point.equals(ColorPoint)` returns true but `ColorPoint.equals(Point)` returns false.

**Modern approach**: Use `Objects.equals(a, b)` for null-safe comparisons. Use Lombok's `@EqualsAndHashCode` or Java 16+ Records (which auto-generate correct `equals()`/`hashCode()`) to avoid hand-writing these.

**What interviewers are really testing:** The `equals`/`hashCode` contract and real consequences of violating it. Senior candidates should mention the symmetry trap in inheritance, `Objects.equals()`, or Records auto-generating these.

**Red flag answer:** "equals compares content and == compares reference" without mentioning `hashCode()` — this is incomplete and dangerous.

**Follow-up:**
1. "What happens if you override `equals()` but not `hashCode()`? Walk me through the exact failure scenario with a HashMap."
2. "How would you implement `equals()` correctly in a class hierarchy where a parent and child class both add meaningful fields?"
3. "What's the advantage of using Records over manually implementing equals/hashCode?"
</Accordion>

<Accordion title="5. Java Collections Framework">
**Answer**:
The Collections Framework is the backbone of everyday Java programming. What matters in an interview is knowing *which* collection to pick for a given use case and *why* — understanding the internal data structures and their Big-O characteristics.

*   **List** (Ordered, allows duplicates):
    *   `ArrayList`: Backed by a dynamic array. O(1) random access by index, O(1) amortized append, O(n) insert/remove in the middle (shifting). Default capacity 10, grows by 50%. This is your default choice 90% of the time.
    *   `LinkedList`: Doubly-linked list. O(1) insert/remove at head/tail (if you have a reference), O(n) random access. Also implements `Deque`. In practice, `ArrayList` beats `LinkedList` in almost all scenarios due to CPU cache locality — the linked list's node objects are scattered across the heap.
*   **Set** (No duplicates):
    *   `HashSet`: Backed by a `HashMap` internally. O(1) add/remove/contains. No ordering.
    *   `LinkedHashSet`: Maintains insertion order. Slight overhead.
    *   `TreeSet`: Backed by a Red-Black tree (`TreeMap`). O(log n) operations. Sorted order. Elements must be `Comparable` or provide a `Comparator`.
    *   `EnumSet`: Extremely efficient for enum values — uses bitwise operations internally.
*   **Map** (Key-value pairs):
    *   `HashMap`: O(1) average get/put. Uses array of buckets + linked lists (pre-Java 8) or **balanced trees when bucket size exceeds 8** (Java 8+ treeification — degrades gracefully from O(n) to O(log n) on hash collisions). Load factor 0.75 triggers resize.
    *   `ConcurrentHashMap`: Thread-safe without locking the entire map. Uses segment-level locking (pre-Java 8) or CAS + `synchronized` on individual bins (Java 8+). Essential for concurrent applications.
    *   `TreeMap`: Red-Black tree. O(log n) operations. Sorted by key.
    *   `LinkedHashMap`: Insertion-ordered or access-ordered (useful for LRU caches — override `removeEldestEntry()`).
*   **Queue/Deque**:
    *   `PriorityQueue`: Min-heap. O(log n) insert/remove. Not thread-safe.
    *   `ArrayDeque`: Resizable circular array. Faster than `LinkedList` as a stack/queue.
    *   `BlockingQueue` (`LinkedBlockingQueue`, `ArrayBlockingQueue`): Thread-safe, used in producer-consumer patterns.

**What interviewers are really testing:** Can you reason about data structure trade-offs in context? Do you know about Java 8's HashMap treeification? Can you pick the right collection for a real scenario (e.g., "I need a thread-safe map with high read throughput" -> `ConcurrentHashMap`)?

**Red flag answer:** Just listing collection names without Big-O or internal structure knowledge. Or saying "LinkedList is better for insertions" without qualifying that ArrayList usually wins due to cache locality.

**Follow-up:**
1. "You're building an in-memory LRU cache with a max size of 1000 entries. Which collection do you use and why?"
2. "Explain what happens inside a HashMap when two keys have the same hashCode. What changed in Java 8?"
3. "Why is `ConcurrentHashMap` preferred over `Collections.synchronizedMap()`? What's the locking strategy difference?"
</Accordion>

<Accordion title="6. final keyword">
**Answer**:
`final` is Java's way of saying "this won't change after initialization" — but what "change" means is nuanced and trips up a lot of candidates.

*   **Variable**: Once assigned, cannot be reassigned. For primitives, the value is constant. For objects, the *reference* is constant but the object's internal state can still be mutated. This is the critical distinction: `final List<String> list = new ArrayList<>()` — you can't reassign `list` to a new ArrayList, but you can absolutely call `list.add("item")`. True immutability requires making the object itself unmodifiable (e.g., `Collections.unmodifiableList()`, `List.of()`).
*   **Method**: Cannot be overridden in subclasses. The JVM can inline final methods more aggressively since it knows the implementation won't change at runtime. Use this when you want to lock down behavior — for example, template method pattern where certain steps must not be overridden.
*   **Class**: Cannot be subclassed. `String`, `Integer`, and all wrapper classes are final. This is essential for immutability guarantees — if `String` weren't final, a subclass could override methods and break the immutability contract that the entire JVM relies on for security (class loading, string pooling).

**Effectively final (Java 8+)**: A variable that is never reassigned after initialization is "effectively final" even without the `final` keyword. This matters for lambdas — they can capture effectively final local variables. The compiler enforces this.

**What interviewers are really testing:** The reference-vs-object mutation distinction for `final` variables, and understanding of why `String` is final. Senior candidates should mention effectively final, or the security implications of final classes.

**Red flag answer:** "final makes a variable constant" without clarifying that for objects, only the reference is constant. Or saying "final class means immutable class" — finality prevents subclassing but doesn't make the class immutable by itself.

**Follow-up:**
1. "I have `final Map<String, String> config = new HashMap<>()`. Can I add entries? How do I make it truly immutable?"
2. "Why is `String` declared as final? What would break if it weren't?"
3. "What's the difference between `final`, `finally`, and `finalize()` — and which one should you never use?"
</Accordion>

<Accordion title="7. Exception Handling">
**Answer**:
Java's exception hierarchy and the checked/unchecked distinction is a design choice that sparks genuine debate — understanding the trade-offs shows maturity.

*   **Checked Exceptions** (`extends Exception`): The compiler forces you to either catch them or declare them with `throws`. Examples: `IOException`, `SQLException`, `ClassNotFoundException`. The philosophy: these represent *recoverable* conditions that the caller should handle. The trade-off: they create coupling — every layer in the call stack must either handle or propagate, leading to verbose code. This is why Spring wraps all checked exceptions (e.g., `SQLException` -> `DataAccessException`) into unchecked ones.
*   **Unchecked Exceptions** (`extends RuntimeException`): Not enforced by the compiler. Examples: `NullPointerException`, `IllegalArgumentException`, `ArrayIndexOutOfBoundsException`. These typically represent programming errors (bugs) that shouldn't be caught — they should be fixed. Modern Java development (and all Spring code) heavily favors unchecked exceptions.
*   **Errors** (`extends Error`): Severe JVM-level problems — `OutOfMemoryError`, `StackOverflowError`, `NoClassDefFoundError`. You generally should not catch these because the JVM is in an unreliable state. Exception: some applications catch `OutOfMemoryError` in specific scopes to gracefully degrade, but this is risky.

**Best practices in production**:
- Never catch `Exception` or `Throwable` broadly — you'll swallow errors you need to see.
- Use try-with-resources (Java 7+) for all `AutoCloseable` resources — eliminates a whole class of resource leak bugs.
- Log the stack trace: `log.error("Payment failed for order {}", orderId, exception)` — the exception as the last arg gives you the full trace in SLF4J.
- Create domain-specific exceptions: `InsufficientBalanceException`, `OrderNotFoundException` — these make error handling readable and give the API layer meaningful HTTP status codes.

**What interviewers are really testing:** Do you understand the philosophical debate between checked and unchecked? Can you articulate why Spring converts checked to unchecked? Do you follow try-with-resources religiously?

**Red flag answer:** "I always catch Exception to be safe" — this is an anti-pattern that masks bugs. Or not mentioning try-with-resources.

**Follow-up:**
1. "Why does Spring wrap checked exceptions like `SQLException` into unchecked `DataAccessException`? What's the design principle?"
2. "Show me a scenario where catching `OutOfMemoryError` is actually justified. What are the risks?"
3. "What's the problem with `catch (Exception e) { e.printStackTrace(); }` in production code?"
</Accordion>

<Accordion title="8. Interface vs Abstract Class">
**Answer**:
This question has gotten more nuanced since Java 8 blurred the lines — the interviewer wants to see if you understand the *current* state of things, not the Java 6 answer.

*   **Interface** (since Java 8+):
    *   Can have: abstract methods, `default` methods (with implementation), `static` methods, `private` methods (Java 9+), constants (`public static final` implicitly).
    *   Cannot have: instance fields (state), constructors.
    *   A class can implement multiple interfaces — this is Java's answer to multiple inheritance of *behavior* (without the state conflicts of C++ multiple inheritance).
    *   **Use when**: Defining a contract/capability. Think "-able" suffix: `Serializable`, `Comparable`, `Iterable`. Or when you need multiple inheritance: a class can be both `Cacheable` and `Auditable`.

*   **Abstract Class**:
    *   Can have: Everything an interface can, PLUS instance fields (state), constructors, non-public methods, mutable state.
    *   A class can extend only ONE abstract class.
    *   **Use when**: You have shared state or partial implementation in a type hierarchy. Example: `AbstractHttpServlet` shares request parsing logic, subclasses implement `doGet()`/`doPost()`. Template Method pattern is the classic use case.

**Decision framework**: Start with an interface. If you find yourself wanting shared state or a constructor, consider an abstract class. If you need both, use an interface with a companion abstract "skeletal" implementation (like `AbstractList` implementing `List`) — this is the pattern from Effective Java.

**What interviewers are really testing:** Do you know about default methods and how they changed the interface-vs-abstract-class calculus? Can you articulate the decision criteria? Senior candidates should mention the skeletal implementation pattern or the diamond problem with default methods.

**Red flag answer:** Giving the pre-Java-8 answer ("interfaces can't have implementations") or saying "always use interfaces" without nuance.

**Follow-up:**
1. "If two interfaces both define a `default` method with the same signature and a class implements both, what happens?"
2. "Explain the skeletal implementation pattern — `AbstractList` implements `List`. Why is this useful?"
3. "When would you refactor from an abstract class to an interface with default methods? What's the migration path?"
</Accordion>

<Accordion title="9. Generics">
**Answer**:
Generics are about type safety at compile time while maintaining backward compatibility with pre-generics Java code — and the way Java achieves this (type erasure) is both elegant and full of gotchas.

**Core concepts**:
*   **Type Parameters**: `class Box<T>`, `<K, V>`, method-level generics `<T> T findById(Class<T> type, Long id)`.
*   **Bounded types**: `<T extends Number>` (upper bound — T must be Number or subclass), `<T extends Comparable<T>>` for ensuring comparability.
*   **Wildcards**:
    *   `? extends Number` (upper bounded): Read-only, can get `Number` out. Used for producers (PECS: Producer Extends).
    *   `? super Integer` (lower bounded): Write-only, can put `Integer` in. Used for consumers (PECS: Consumer Super).
    *   `?` (unbounded): Maximum flexibility, but you can only read as `Object`.

**Type Erasure**: The compiler checks generic types, then **erases** them to their bounds (or `Object`) in the bytecode. So `List<String>` and `List<Integer>` are both just `List` at runtime. Consequences:
*   You cannot do `new T()` or `new T[]` — the type is erased.
*   You cannot do `instanceof List<String>` — generic type info is gone at runtime.
*   You can use `TypeToken` (Gson) or `ParameterizedTypeReference` (Spring) to work around erasure for frameworks that need runtime type info.

**PECS principle** (Producer Extends, Consumer Super): This is the rule for deciding which wildcard to use. `Collections.copy(List<? super T> dest, List<? extends T> src)` — the source produces elements (extends), the destination consumes them (super). Joshua Bloch's Effective Java made this canonical.

**What interviewers are really testing:** Understanding of type erasure and its consequences, PECS, and the difference between `<T>`, `<? extends T>`, and `<? super T>`. Senior candidates should explain *why* type erasure exists (backward compatibility with Java 4 bytecode).

**Red flag answer:** "Generics add type safety" without being able to explain type erasure or wildcards. Or confusing `<? extends T>` with `<? super T>`.

**Follow-up:**
1. "Why can't you create a generic array `new T[10]` in Java? What's the underlying issue?"
2. "Explain PECS with a real code example. When would you use `? super` vs `? extends`?"
3. "How do frameworks like Jackson and Spring work around type erasure to deserialize `List<MyDto>` correctly?"
</Accordion>

<Accordion title="10. Java 8+ Features (Up to Java 21)">
**Answer**:
Java 8 was a paradigm shift, but modern Java (17, 21) has continued evolving rapidly. A strong candidate covers the full arc.

**Java 8 (2014) — The Big Bang**:
*   **Lambdas**: `(x) -> x * 2`. Enables functional-style programming. Under the hood, lambdas use `invokedynamic` — they're NOT anonymous inner classes (no extra `.class` file, no `this` capture issues).
*   **Streams API**: Declarative data processing pipelines. `list.stream().filter(...).map(...).collect(...)`. Lazy evaluation — intermediate operations build a pipeline, terminal operations trigger execution. Parallel streams use the Fork/Join pool.
*   **Optional**: Explicit null handling. `Optional.ofNullable(user).map(User::getEmail).orElse("N/A")`. Prevents `NullPointerException` chains. Should NOT be used as method parameters or fields — only as return types.
*   **Functional Interfaces**: `@FunctionalInterface` — single abstract method. `Function<T,R>`, `Predicate<T>`, `Consumer<T>`, `Supplier<T>`.
*   **Default Methods**: Interface methods with implementation. Enabled API evolution without breaking existing implementations.
*   **New Date/Time API** (`java.time`): Immutable, thread-safe. `LocalDate`, `ZonedDateTime`, `Duration`, `Period`. Replaced the terrible `java.util.Date`/`Calendar`.

**Java 9-11**:
*   Modules (Project Jigsaw), `var` local type inference, HTTP Client API, `String` methods (`isBlank()`, `lines()`, `strip()`).

**Java 14-17 (LTS)**:
*   **Records**: `record Point(int x, int y) {}` — immutable data carriers with auto-generated `equals()`, `hashCode()`, `toString()`. Replaces 50 lines of boilerplate.
*   **Sealed Classes**: `sealed class Shape permits Circle, Square {}` — restricts which classes can extend. Enables exhaustive `switch` (compiler knows all subtypes).
*   **Pattern Matching for `instanceof`**: `if (obj instanceof String s)` — no cast needed.
*   **Text Blocks**: Multi-line strings with `"""`.

**Java 21 (LTS)**:
*   **Virtual Threads (Project Loom)**: Lightweight threads managed by the JVM, not the OS. You can create millions of them. `Thread.startVirtualThread(() -> ...)`. Eliminates the need for reactive frameworks in many I/O-bound scenarios.
*   **Pattern Matching for Switch**: Exhaustive pattern matching with type patterns, guarded patterns.
*   **Sequenced Collections**: `SequencedCollection`, `SequencedMap` — unified API for first/last element access.

**What interviewers are really testing:** Are you staying current with the language? Can you articulate the impact of Records, Sealed Classes, and Virtual Threads? Do you know the difference between LTS releases (11, 17, 21) and non-LTS?

**Red flag answer:** Only mentioning Java 8 features in 2025+ — the language has evolved massively. Or saying "Optional replaces null checks" without knowing the intended usage (return types only).

**Follow-up:**
1. "How do Virtual Threads differ from platform threads at the JVM level? When would you NOT use them?"
2. "You have a legacy codebase on Java 8. What's your migration strategy to Java 21? What breaks?"
3. "Why are Records NOT a replacement for entities/JPA classes? What's the fundamental incompatibility?"
</Accordion>

<Accordion title="11. Memory Model (Stack vs Heap)">
**Answer**:
Understanding the JVM memory layout is essential for debugging performance issues, `OutOfMemoryError`s, and concurrency bugs.

*   **Stack Memory**:
    *   One stack per thread — each thread gets its own stack, completely isolated.
    *   Stores: local primitives, local object references (the reference itself, not the object), method call frames (arguments, return address, local variables).
    *   LIFO structure — each method call pushes a new frame, return pops it.
    *   Fixed size per thread (default ~512KB-1MB, configurable with `-Xss`). Exceeding it throws `StackOverflowError` (infinite recursion is the classic cause).
    *   Auto-cleared when the method returns. No garbage collection needed.

*   **Heap Memory**:
    *   Shared by ALL threads — this is where concurrency gets interesting and dangerous.
    *   Stores: all objects and arrays, regardless of where they were instantiated.
    *   Managed by the Garbage Collector. Divided into regions:
        *   **Young Generation** (Eden + Survivor S0/S1): Newly created objects. Minor GC runs here frequently and is fast (most objects die young — "generational hypothesis").
        *   **Old Generation (Tenured)**: Objects that survived multiple Minor GCs. Major/Full GC runs here — slower and more disruptive.
    *   Sized with `-Xms` (initial) and `-Xmx` (max). A common production config might be `-Xms4g -Xmx4g` (equal to avoid resize pauses).

*   **Metaspace** (Java 8+): Stores class metadata, method bytecode, constant pool. Uses native memory (not heap). Replaced PermGen. Grows automatically but can be capped with `-XX:MaxMetaspaceSize`.

*   **Direct Memory (Off-Heap)**: Used by NIO `ByteBuffer.allocateDirect()`. Not subject to GC — must be manually managed. Frameworks like Netty use this heavily for zero-copy I/O.

**Escape Analysis** (important optimization): If the JIT compiler determines an object never "escapes" a method (isn't returned, stored in a field, or passed to another thread), it can allocate it on the stack instead of the heap — avoiding GC entirely. This is why micro-benchmarks in Java can be deceptive.

**What interviewers are really testing:** Can you explain where objects vs references live? Can you tune heap sizing? Do you know about Young/Old generation and why the generational hypothesis matters?

**Red flag answer:** "Objects go on the heap, primitives go on the stack" — this is incomplete. Primitives *inside* objects are on the heap. And escape analysis can put objects on the stack.

**Follow-up:**
1. "Your service is throwing `OutOfMemoryError: Metaspace`. Heap is fine. What's happening and how do you fix it?"
2. "Explain escape analysis. Why does it matter for performance?"
3. "You set `-Xms512m -Xmx4g` and see GC pauses during traffic spikes. What's the issue?"
</Accordion>

<Accordion title="12. Garbage Collection Roots and How GC Works">
**Answer**:
The fundamental question GC answers is: "Is this object still reachable?" Everything starts from GC roots.

**GC Roots** — these are the starting points for reachability analysis:
*   Local variables and parameters on active thread stacks
*   Active threads themselves
*   Static fields of loaded classes
*   JNI references (native code holding Java object references)
*   References held by the JVM internally (class objects, exception objects, system class loader)

**The Mark-and-Sweep algorithm** (simplified):
1. **Mark phase**: Starting from GC roots, traverse all reachable object references recursively. Mark every reachable object as "alive."
2. **Sweep phase**: Walk the entire heap. Any object NOT marked is unreachable (garbage) — reclaim its memory.
3. **Compact phase** (in some collectors): Move surviving objects together to eliminate fragmentation, update all references.

**Why "Mark and Sweep" alone isn't enough**: In a real JVM, this would require a full stop-the-world (STW) pause every time. Modern collectors (G1, ZGC, Shenandoah) use techniques like:
*   **Generational collection**: Collect young generation frequently (cheap), old generation rarely.
*   **Concurrent marking**: Mark while application threads run (using write barriers to track reference changes).
*   **Incremental compaction**: Compact small regions at a time rather than the whole heap.

**Key insight**: An object with no references pointing to it is NOT the definition of garbage. The correct definition is: an object not reachable from any GC root. Circular references (A -> B -> C -> A) get collected if none of them are reachable from a root — this is why reference counting (used by Python/Swift) isn't used in Java.

**What interviewers are really testing:** Understanding that GC is root-based, not reference-counting. The mark-and-sweep algorithm. Why circular references aren't a problem in Java.

**Red flag answer:** "GC collects objects with zero references" — this describes reference counting, not Java's GC. Or not knowing what GC roots are.

**Follow-up:**
1. "If two objects reference each other but nothing else references them, will they be collected? Why?"
2. "What's a write barrier in the context of concurrent GC? Why is it needed?"
3. "How would you identify which GC roots are preventing a large object from being collected? What tools would you use?"
</Accordion>

<Accordion title="13. GC Algorithms and Tuning">
**Answer**:
Choosing the right GC and tuning it is one of the highest-impact performance decisions in production Java. Here's the landscape:

*   **Serial GC** (`-XX:+UseSerialGC`): Single-threaded, stop-the-world for everything. Only appropriate for small heaps (&lt;200MB) or client-side apps. Simple, minimal overhead.

*   **Parallel GC** (`-XX:+UseParallelGC`): Multi-threaded for both Young and Old GC. Optimized for **throughput** (maximize application time vs GC time). Was the default before Java 9. Good for batch processing jobs where pause time doesn't matter — e.g., a nightly ETL job processing 10TB of data.

*   **G1 GC (Garbage First)** (`-XX:+UseG1GC`): Default since Java 9. Region-based — divides the heap into equal-sized regions (1-32MB each), classified as Eden, Survivor, Old, or Humongous. Key behaviors:
    *   Targets a pause-time goal (default 200ms, configurable with `-XX:MaxGCPauseMillis=100`).
    *   Prioritizes collecting regions with the most garbage first (hence the name).
    *   Mixed collections: can collect some Old regions alongside Young regions.
    *   Best general-purpose collector for heaps 4GB-64GB.

*   **ZGC** (`-XX:+UseZGC`): Ultra-low-latency collector. Guarantees sub-millisecond pauses (&lt;1ms) regardless of heap size (tested up to 16TB). Uses colored pointers and load barriers. Concurrent compaction. Production-ready since Java 15. Ideal for latency-sensitive services (trading platforms, real-time bidding, gaming servers).

*   **Shenandoah** (`-XX:+UseShenandoahGC`): Similar goals to ZGC — low latency, concurrent compaction. Developed by Red Hat. Uses Brooks forwarding pointers instead of colored pointers. Available in OpenJDK but not Oracle JDK.

**Tuning in practice**:
- Start with G1 and default settings — it's remarkably good out of the box.
- Monitor with tools: GC logs (`-Xlog:gc*`), GCViewer, GCEasy.io, Grafana + Prometheus JMX exporter.
- Key metrics: GC pause time (p99), GC frequency, throughput (% time not in GC), heap occupancy after GC.
- Common tuning: increase heap if Old Gen is constantly full, reduce `-XX:MaxGCPauseMillis` if latency matters, increase `-XX:G1HeapRegionSize` for heaps with many large objects.

**What interviewers are really testing:** Can you select the right GC for a given workload? Can you reason about the throughput-vs-latency trade-off? Have you actually tuned GC in production?

**Red flag answer:** Not knowing G1 is the default, or saying "just increase the heap" without understanding root cause analysis.

**Follow-up:**
1. "Your service has p99 latency spikes of 500ms every 30 seconds. GC logs show Full GC pauses. What's your debugging approach?"
2. "When would you choose ZGC over G1? What's the trade-off?"
3. "Explain what a 'Humongous allocation' is in G1 and why it's problematic."
</Accordion>

<Accordion title="14. ClassLoader Subsystem">
**Answer**:
The ClassLoader is how Java dynamically loads, links, and initializes classes — and understanding it is crucial for debugging `ClassNotFoundException`, `NoClassDefFoundError`, and the classloader leaks that plague application servers.

**Three phases**:
1. **Loading**: Reads the `.class` file (from disk, network, or generated at runtime) and creates a `java.lang.Class` object representing it in Metaspace.
2. **Linking**:
    *   *Verification*: Validates bytecode structure, type safety, stack depth. Prevents malicious bytecodes.
    *   *Preparation*: Allocates memory for static fields with default values.
    *   *Resolution*: Resolves symbolic references (class/method names in the constant pool) to direct references (memory addresses). Can be lazy.
3. **Initialization**: Executes static initializers and `static {}` blocks. Happens exactly once per class. This is where `ExceptionInInitializerError` comes from.

**ClassLoader hierarchy (delegation model)**:
*   **Bootstrap ClassLoader**: Loads core JDK classes (`java.lang.*`, `java.util.*`). Written in native code (C/C++). No parent.
*   **Platform ClassLoader** (Java 9+, was "Extension" before): Loads platform-specific classes.
*   **Application ClassLoader**: Loads classes from your classpath/modulepath.

**Delegation model**: When a class is requested, the classloader first delegates to its parent. Only if the parent can't load it does the child attempt. This ensures core classes (`String`, `Object`) are always loaded by the Bootstrap ClassLoader, preventing spoofing.

**Breaking delegation (custom classloaders)**: Frameworks like Tomcat, OSGi, and JBoss deliberately break the parent-first delegation to allow each WAR/bundle to have its own version of a library. Tomcat's classloader loads the WAR's classes first, then delegates to the parent — the opposite of the default model. This is how two WARs can use different versions of, say, Jackson.

**ClassLoader leaks**: In application servers, when a WAR is redeployed, the old classloader should be GC'd with all its loaded classes. But if anything holds a reference to an old class (static field, ThreadLocal, JDBC driver registration), the entire classloader and ALL its classes leak, causing Metaspace/PermGen exhaustion. This was one of the most common production issues in the pre-container era.

**What interviewers are really testing:** Understanding of the delegation model and why it exists. How classloader leaks happen. The difference between `ClassNotFoundException` (classloader can't find the class) and `NoClassDefFoundError` (class was found at compile time but not at runtime — usually a transitive dependency issue).

**Red flag answer:** Just listing "Bootstrap, Extension, Application" without explaining delegation or real-world implications.

**Follow-up:**
1. "What's the difference between `ClassNotFoundException` and `NoClassDefFoundError`? When do you see each?"
2. "How does Tomcat achieve classloader isolation between WARs? Why does it break the delegation model?"
3. "You're getting `OutOfMemoryError: Metaspace` after repeated redeployments in Tomcat. What's happening?"
</Accordion>

<Accordion title="15. Memory Leaks in Java">
**Answer**:
"Java has garbage collection, so it can't have memory leaks" — this is a myth. Java absolutely has memory leaks; they just look different from C/C++ leaks. In Java, a memory leak is an object that is still reachable from a GC root but will never be used again.

**Common sources of memory leaks (from most to least common in my experience)**:

1. **Static collections growing unbounded**: `static Map<String, Session> sessionCache = new HashMap<>()` — entries are added but never removed. I've seen this bring down production services holding 8GB+ of stale cache entries. Fix: use a bounded `LinkedHashMap` with `removeEldestEntry()`, Caffeine cache with TTL/max-size, or `WeakHashMap`.

2. **Unclosed resources**: `InputStream`, `Connection`, `ResultSet` that aren't closed in a `finally` block or try-with-resources. Each leaked DB connection holds heap memory AND a network socket. At scale, you exhaust the connection pool within hours. Fix: always use try-with-resources.

3. **Listeners/callbacks never deregistered**: An observer registers to receive events but is never unregistered. The event source holds a strong reference to the observer indefinitely. Common in Swing/JavaFX apps and event-driven architectures.

4. **Inner classes holding outer class reference**: Non-static inner classes implicitly hold a reference to their enclosing outer class instance. If the inner class outlives the outer class (e.g., returned as a callback), the entire outer class is retained. Fix: use static inner classes or lambdas (which don't capture `this` unless they reference instance members).

5. **ThreadLocal variables in thread pools**: `ThreadLocal` stores values per-thread. In a thread pool, threads are reused — the ThreadLocal value is never cleared unless you call `.remove()`. In a web server with a 200-thread pool, each thread can accumulate leaked context objects. Fix: always call `threadLocal.remove()` in a `finally` block.

6. **String.intern() abuse**: Interning every string puts them in the String Pool (which lives in the heap since Java 7). If you intern user-generated strings, the pool grows without bound.

7. **Class loader leaks**: Already covered in Q14 — application server redeployments where old classloaders can't be GC'd.

**Diagnostic tools**:
- `jmap -dump:live,format=b,file=heap.hprof <pid>` — capture heap dump
- Eclipse MAT (Memory Analyzer Tool) — find leak suspects with dominator tree analysis
- VisualVM — live monitoring and heap dump analysis
- `-XX:+HeapDumpOnOutOfMemoryError` — automatically dumps on OOM (always enable in production)

**What interviewers are really testing:** Real production experience with memory issues. Can you enumerate specific leak patterns? Do you know the diagnostic tools?

**Red flag answer:** "Java has garbage collection so memory leaks aren't possible" or only mentioning one type of leak.

**Follow-up:**
1. "Your service's heap usage grows linearly over 3 days and then OOMs. Walk me through your investigation process, tool by tool."
2. "What's the difference between a memory leak and high memory pressure? How do you distinguish them?"
3. "How does `WeakHashMap` help prevent cache-related leaks? What's the limitation?"
</Accordion>

<Accordion title="16. Just-In-Time (JIT) Compiler">
**Answer**:
The JIT compiler is Java's secret weapon for performance — it's why well-warmed-up Java can match or beat C++ in some benchmarks, despite starting as interpreted bytecode.

**How it works**:
1. The JVM starts by **interpreting** bytecode — executing it instruction by instruction. This is slow but has zero startup cost.
2. The JVM profiles which methods are called frequently ("hot spots" — hence the name HotSpot JVM).
3. When a method exceeds a threshold (default ~10,000 invocations for the C2 compiler), the JIT compiles it to **native machine code** specific to the hardware it's running on.
4. The compiled native code replaces the interpreted version — subsequent calls run at near-C speed.

**Two JIT compilers in HotSpot**:
*   **C1 (Client)**: Fast compilation, lower optimization. Used early to reduce startup latency.
*   **C2 (Server)**: Slower compilation, aggressive optimizations. Used for hot methods after profiling.
*   **Tiered compilation** (default since Java 8): Uses C1 first for quick wins, then promotes to C2 as more profiling data is available. Five levels (0-4).

**Key optimizations the JIT performs**:
*   **Method inlining**: Replaces method calls with the method body. Eliminates call overhead and enables further optimizations. Small methods are inlined aggressively (`-XX:MaxInlineSize=35` bytes by default).
*   **Loop unrolling**: Reduces loop overhead by duplicating the loop body.
*   **Escape analysis**: Determines if an object escapes the method — if not, it can be allocated on the stack or its fields can be scalar-replaced (stored in CPU registers).
*   **Dead code elimination**: Removes unreachable code paths.
*   **Null check elimination**: Removes redundant null checks based on control flow analysis.
*   **Speculative optimization with deoptimization**: The JIT can speculate (e.g., "this virtual call always goes to ClassA.method()") and compile a direct call. If the speculation is invalidated (ClassB is loaded), it *deoptimizes* — falls back to interpreted mode and recompiles.

**GraalVM**: A newer JIT compiler (and AOT compiler) written in Java itself. Can produce better optimized code than C2 for certain workloads. Also supports Ahead-of-Time compilation (native images) via `native-image`, which eliminates JIT warmup entirely but loses some runtime optimization capability.

**What interviewers are really testing:** Understanding of JIT warmup and why Java performance improves over time. Knowledge of key optimizations. Awareness of GraalVM/native-image trade-offs.

**Red flag answer:** "JIT compiles bytecode to machine code" — correct but shows no depth. Not knowing about warmup time, tiered compilation, or GraalVM.

**Follow-up:**
1. "Your service takes 30 seconds to reach peak performance after restart. How do you reduce this warmup time?"
2. "What's the trade-off between JIT compilation and Ahead-of-Time (AOT) compilation with GraalVM native-image?"
3. "How would speculative optimization and deoptimization affect a benchmark you're running?"
</Accordion>

This question tests whether you understand the fundamental architectural decisions that differentiate these languages and can reason about trade-offs, not just list features.

| Aspect | Java | C++ |

<Accordion title="18. Metaspace (Java 8+)">
**Answer**:
Metaspace replaced PermGen in Java 8 and changed how class metadata is stored — understanding this is essential for diagnosing a specific class of production issues.

**What changed from PermGen**:
*   **PermGen** (Java 7 and earlier): Fixed-size region inside the JVM heap. Stored class metadata, interned strings (pre-Java 7), static variables. Had a hard limit (`-XX:MaxPermSize`, default 64-256MB). When exhausted: `OutOfMemoryError: PermGen space`. This was a constant pain point — especially with hot-redeployment in application servers.
*   **Metaspace** (Java 8+): Allocated in **native memory** (outside the JVM heap). Grows automatically by default — no fixed limit unless you set one with `-XX:MaxMetaspaceSize`. When the OS runs out of native memory: `OutOfMemoryError: Metaspace`.

**What lives in Metaspace**:
*   Class metadata (field info, method info, constant pool)
*   Bytecode for methods
*   Annotations metadata
*   Interned strings were moved to the heap in Java 7 (they do NOT live in Metaspace)

**Production gotcha**: Because Metaspace auto-grows, it can silently consume all native memory if you have a class-loading leak. This commonly happens with:
*   Libraries using runtime bytecode generation (CGLIB, Javassist, dynamic proxies) — each generated proxy class creates new metadata.
*   Application server hot-redeployments where old classloaders aren't GC'd (see Q14).
*   Groovy/scripting engines that compile new classes per invocation.

**Monitoring and tuning**:
*   `jstat -gc <pid>` shows Metaspace usage.
*   Set `-XX:MaxMetaspaceSize=256m` as a safety net in production — better to get an early OOM than to have the JVM consume all OS memory and get killed by the OOM-killer.
*   `-XX:MetaspaceSize=128m` sets the initial trigger threshold for GC of dead classloaders.

**What interviewers are really testing:** Understanding why PermGen was problematic, how Metaspace solves it, and the new failure modes Metaspace introduces. Production awareness of class-loading leaks.

**Red flag answer:** "Metaspace replaced PermGen and uses native memory" without explaining why this matters or what problems it solves/creates.

**Follow-up:**
1. "Your application is consuming 2GB of native memory in Metaspace despite having only 100 classes. What's happening?"

<Accordion title="19. Reference Types (Strong, Weak, Soft, Phantom)">
**Answer**:
Java has four reference types that give you varying levels of control over garbage collection — understanding them is key to building caches, preventing leaks, and managing resources.

*   **Strong Reference**: The default. `Object obj = new Object()`. As long as a strong reference exists from a GC root, the object will NEVER be collected. This is what causes memory leaks — strong references to objects you no longer need.

*   **Weak Reference** (`java.lang.ref.WeakReference<T>`): Collected as soon as the GC runs if only weak references exist (no strong refs). Use case: `WeakHashMap` — keys are weak references. When the key object is no longer strongly referenced anywhere, the entry is automatically removed. Real-world: caching metadata about objects where the cache shouldn't prevent the object from being GC'd. ClassLoader to loaded-class mappings often use weak references.

*   **Soft Reference** (`java.lang.ref.SoftReference<T>`): Collected only when the JVM is running low on memory — the GC tries to keep soft-referenced objects around as long as possible. Use case: memory-sensitive caches. Example: an image thumbnail cache — keep thumbnails in memory when heap is plentiful, let them be reclaimed under memory pressure. The JVM guarantees: all soft references will be cleared before throwing `OutOfMemoryError`. Gotcha: the "when" is JVM-specific and hard to predict — in practice, Caffeine or Guava cache with explicit max-size is more predictable.

*   **Phantom Reference** (`java.lang.ref.PhantomReference<T>`): The object is already eligible for GC — you can never retrieve the referent (`get()` always returns null). Used exclusively for post-mortem cleanup via a `ReferenceQueue`. When the phantom-referenced object is collected, the phantom reference is enqueued. A cleanup thread polls the queue and performs resource cleanup (file handles, native memory). This is the safer replacement for `finalize()` (which is deprecated since Java 9 and removed in Java 18). Java's `Cleaner` API (Java 9+) is built on phantom references.

**ReferenceQueue**: All weak/soft/phantom references can optionally be registered with a `ReferenceQueue`. When the referenced object is collected (or about to be, for phantom), the reference object itself is added to the queue. This enables cleanup logic without relying on `finalize()`.

**What interviewers are really testing:** Practical understanding of when to use each type, especially the distinction between Soft (cache) and Weak (don't prevent GC). Knowledge of `ReferenceQueue` and why `finalize()` is deprecated.

**Red flag answer:** Not knowing the difference between Weak and Soft, or thinking Phantom references let you "resurrect" objects.

**Follow-up:**
1. "You're building a cache for expensive-to-compute results. Should you use `SoftReference` or Caffeine with max-size? What's the trade-off?"
2. "Why is `finalize()` deprecated? What replaces it and how does it work internally?"
3. "How does `WeakHashMap` actually remove entries? When does the removal happen?"
</Accordion>

<Accordion title="20. Volatile Keyword and Java Memory Model">
**Answer**:
`volatile` is one of the most misunderstood keywords in Java. It provides visibility guarantees, not atomicity — and understanding the difference requires grasping the Java Memory Model (JMM).

**The problem `volatile` solves**:
Without `volatile`, each thread can cache a variable's value in its CPU register or L1/L2 cache. Thread A might write `flag = true`, but Thread B never sees the update because it's reading from its cached copy. This is not a theoretical concern — it happens in production, especially on multi-core CPUs.

**What `volatile` guarantees**:
1. **Visibility**: Every read of a volatile variable goes to main memory (not a thread-local cache). Every write is immediately flushed to main memory. When Thread A writes a volatile variable and Thread B subsequently reads it, Thread B sees the write.
2. **Happens-before relationship**: A write to a volatile variable *happens-before* every subsequent read of that same variable. This also extends transitively — all writes by Thread A before the volatile write are visible to Thread B after the volatile read. This is how the "Double-Checked Locking" pattern for singletons works: the volatile write of the instance reference publishes all the constructor's field writes.
3. **Prevents instruction reordering**: The compiler and CPU cannot reorder reads/writes of volatile variables with respect to other reads/writes (it acts as a memory barrier).

**What `volatile` does NOT guarantee**:
*   **Atomicity of compound operations**: `volatile int count; count++` is NOT thread-safe. `count++` is read-modify-write (three steps). Two threads can read the same value, increment, and write back — losing an increment. For atomic read-modify-write, use `AtomicInteger` or `synchronized`.

**Common uses of `volatile`**:
*   **Flag variables**: `volatile boolean running = true;` — one thread sets to false, others check in a loop.
*   **Double-Checked Locking**: `private volatile static Singleton instance;` — ensures the partially constructed object isn't visible to other threads.
*   **Publishing immutable objects**: After constructing an immutable object, writing its reference to a volatile field safely publishes all of its fields to other threads.

**What interviewers are really testing:** The distinction between visibility and atomicity. Understanding of the happens-before relationship. Knowledge of when volatile is sufficient vs when you need `synchronized` or atomics.

**Red flag answer:** "Volatile makes a variable thread-safe" — it provides visibility, not atomicity. Or not knowing about the happens-before relationship.

**Follow-up:**
1. "Is `volatile long` atomic on a 32-bit JVM? Why or why not?"
2. "Explain the Double-Checked Locking pattern. Why does removing `volatile` from the singleton field break it?"
3. "When would you use `volatile` vs `AtomicReference` vs `synchronized`? Give a scenario for each."

<Accordion title="21. Thread Lifecycle">
**Answer**:
Understanding thread states is essential for debugging hanging applications and interpreting thread dumps.

**The six states** (defined in `Thread.State` enum):
1. **NEW**: Thread object created (`new Thread(...)`) but `start()` not yet called.
2. **RUNNABLE**: Thread is eligible to run. This includes both "running on a CPU core" and "ready to run but waiting for CPU scheduling." Java doesn't distinguish between these — both are RUNNABLE.
3. **BLOCKED**: Thread is waiting to acquire an intrinsic lock (monitor) to enter a `synchronized` block/method. This happens when another thread already holds that lock.
4. **WAITING**: Thread is waiting indefinitely for another thread to perform a specific action. Enters this state via `Object.wait()`, `Thread.join()`, or `LockSupport.park()`.
5. **TIMED_WAITING**: Same as WAITING but with a timeout. Enters via `Thread.sleep(ms)`, `Object.wait(ms)`, `Thread.join(ms)`, `LockSupport.parkNanos()`.
6. **TERMINATED**: Thread's `run()` method has completed (normally or via exception). Cannot be restarted.

**Transitions that trip people up**:
*   RUNNABLE -> BLOCKED: Thread tries to enter `synchronized` block, lock is held by another thread.
*   WAITING -> RUNNABLE: Another thread calls `notify()`/`notifyAll()` on the same monitor, or `join()`-ed thread finishes.
*   There's no RUNNING state — it's part of RUNNABLE.

**Reading thread dumps**: In production, `jstack <pid>` or `kill -3 <pid>` (on Linux) produces a thread dump showing every thread's state. Patterns to look for:
*   Many threads BLOCKED on the same lock = contention bottleneck.
*   Threads WAITING on a `CountDownLatch`/`Condition` that's never signaled = potential hang.
*   Thread in RUNNABLE consuming 100% CPU = infinite loop or heavy computation.

**What interviewers are really testing:** The six official states (not the simplified 5-state textbook model), understanding of BLOCKED vs WAITING, and ability to read thread dumps.

**Red flag answer:** Using the simplified "New -> Runnable -> Running -> Blocked -> Dead" model without knowing about WAITING/TIMED_WAITING, or not knowing how to take a thread dump.

**Follow-up:**
1. "What's the difference between BLOCKED and WAITING? Give a code example for each."

<Accordion title="22. Synchronized and Intrinsic Locks">
**Answer**:
`synchronized` is Java's built-in mutual exclusion mechanism — but understanding its internals and limitations is what separates junior from senior answers.

**How it works**:
Every Java object has an associated **monitor** (intrinsic lock). `synchronized` acquires this monitor before executing the protected code and releases it afterward (even if an exception is thrown).

*   **Method-level**: `synchronized void transfer(...)` — locks on `this` (instance method) or the `Class` object (static method).
*   **Block-level**: `synchronized(lockObject) { ... }` — locks on the specified object. Always preferred because it's more granular — you can protect specific critical sections without locking the entire method.

**Internal mechanics (HotSpot)**:
*   **Biased Locking** (removed in Java 18): If only one thread ever acquires the lock, it's nearly free — just checks a thread ID in the object header.
*   **Thin Lock (CAS)**: When contention is low, uses Compare-And-Swap on the object's mark word. Very fast.
*   **Fat Lock (OS mutex)**: When contention is high, inflates to an OS-level mutex with wait queues. Expensive — involves OS kernel context switches.

**Reentrancy**: `synchronized` is reentrant — a thread can re-acquire a lock it already holds. The lock maintains a count. This prevents self-deadlock when a synchronized method calls another synchronized method on the same object.

**Limitations of `synchronized`**:
*   No way to try-and-fail: You can't attempt to acquire the lock with a timeout. The thread blocks indefinitely.
*   No fairness guarantee: There's no guaranteed order for waiting threads.
*   No read/write distinction: Even readers block each other.
*   No interruptibility: A thread BLOCKED on `synchronized` cannot be interrupted.

For these reasons, `java.util.concurrent.locks.ReentrantLock` is often preferred in production code — it provides `tryLock(timeout)`, fairness mode, interruptible locking, and `Condition` variables.

**What interviewers are really testing:** Intrinsic lock mechanics, the reentrancy property, limitations vs `ReentrantLock`, and understanding of lock contention and its performance impact.

**Red flag answer:** "Synchronized means only one thread can access it" — this is true but doesn't explain monitor mechanics, reentrancy, or limitations. Or using `synchronized(this)` when a more granular lock object would be better.

**Follow-up:**
1. "When would you choose `ReentrantLock` over `synchronized`? Give a concrete scenario."
2. "What's the performance difference between an uncontended vs contended synchronized block?"
3. "Your thread dump shows 200 threads BLOCKED on the same synchronized method. How do you fix the bottleneck?"

<Accordion title="23. Deadlock">
**Answer**:
Deadlock is when two or more threads are permanently blocked, each waiting for a lock held by the other. It's one of the most feared concurrency bugs because the application doesn't crash — it just silently hangs.

**The four necessary conditions** (Coffman conditions — all four must be true):
1. **Mutual Exclusion**: At least one resource is held in a non-sharable mode.
2. **Hold and Wait**: A thread holds at least one resource and is waiting for another.
3. **No Preemption**: Resources can't be forcibly taken from a thread.
4. **Circular Wait**: A circular chain of threads, each waiting for a resource held by the next.

**Classic example**:
```java
// Thread 1                    // Thread 2
synchronized(lockA) {          synchronized(lockB) {
    synchronized(lockB) {          synchronized(lockA) {
        // work                        // work
    }                              }
}                              }
```

**Prevention strategies**:
*   **Lock Ordering**: Always acquire locks in a consistent global order. If every thread acquires `lockA` before `lockB`, circular wait is impossible. In practice, define an ordering (e.g., by object ID or by a numeric rank) and enforce it in code review.
*   **`tryLock()` with timeout** (`ReentrantLock`): Attempt to acquire the lock for a bounded time. If it fails, release all held locks and retry. Breaks the "no preemption" condition.
*   **Single lock**: Sometimes the simplest fix is to use one coarser lock instead of multiple fine-grained locks.
*   **Lock-free algorithms**: Use `Atomic*` variables and CAS operations to avoid locks entirely.

**Detection in production**:
*   `jstack <pid>` — the JVM detects simple deadlocks and reports them at the bottom of the thread dump: "Found one Java-level deadlock."
*   `ThreadMXBean.findDeadlockedThreads()` — programmatic detection. Some monitoring tools (Datadog APM, New Relic) call this periodically.
*   Thread dumps showing the same threads BLOCKED for minutes — even if not detected as a formal deadlock, it indicates lock contention issues.

**What interviewers are really testing:** Can you explain the four conditions? Do you know prevention strategies? Can you detect deadlocks in production?

**Red flag answer:** Only describing the classic two-thread example without prevention strategies or detection tools.

**Follow-up:**
1. "You suspect a deadlock in production but `jstack` doesn't report one. What else could cause all threads to hang?"
2. "How would you implement lock ordering when locking on database rows with arbitrary IDs?"
3. "What's a livelock? How is it different from a deadlock, and how do you detect it?"
</Accordion>

<Accordion title="24. Executor Framework">
**Answer**:
The Executor Framework (`java.util.concurrent`) is the right way to manage threads in production Java — creating raw `new Thread(...)` is an anti-pattern in any non-trivial application.

**Why it exists**: Creating and destroying OS threads is expensive (~1ms each, ~1MB stack per thread). If you create a thread per request at 10,000 RPS, you're creating 10,000 OS threads — you'll exhaust memory or hit OS limits. Thread pools solve this by reusing a fixed number of threads.

**Key implementations**:

*   **`FixedThreadPool`** (`Executors.newFixedThreadPool(n)`): Fixed number of threads. Unbounded `LinkedBlockingQueue` for tasks. Use when you want predictable resource usage. **Gotcha**: The unbounded queue means if tasks pile up, memory grows without bound. In production, use `ThreadPoolExecutor` directly with a bounded queue and a rejection policy.

*   **`CachedThreadPool`** (`Executors.newCachedThreadPool()`): Creates threads on demand, reuses idle ones (after 60s timeout, idle threads are terminated). Uses `SynchronousQueue` (zero-capacity — task handoff must happen immediately). **Gotcha**: Under burst load, can create thousands of threads and crash the JVM. Never use for request handling with unpredictable load.

*   **`SingleThreadExecutor`**: One thread, unbounded queue. Guarantees sequential execution. Good for sequential background tasks (log writing, event dispatching).

*   **`ScheduledThreadPoolExecutor`**: For periodic/delayed tasks. `scheduleAtFixedRate()` vs `scheduleWithFixedDelay()` — know the difference. Replaces `Timer` which has problems (single thread, uncaught exceptions kill the timer).

**Best practice — use `ThreadPoolExecutor` directly**:
```java
new ThreadPoolExecutor(
    corePoolSize,       // minimum threads kept alive
    maxPoolSize,        // maximum threads
    keepAliveTime, unit,
    new ArrayBlockingQueue<>(queueCapacity), // BOUNDED queue
    new ThreadPoolExecutor.CallerRunsPolicy() // rejection policy
);
```
The `CallerRunsPolicy` is often ideal — when the pool is saturated, the submitting thread runs the task itself, providing natural back-pressure.

**What interviewers are really testing:** Knowledge of the gotchas with `Executors` factory methods (unbounded queues, unbounded thread creation). Ability to configure `ThreadPoolExecutor` with proper bounds and rejection policies. Understanding of back-pressure.

**Red flag answer:** "I use `Executors.newFixedThreadPool()`" without knowing about the unbounded queue problem. Or not knowing about rejection policies.

**Follow-up:**
1. "Your `FixedThreadPool` with 50 threads is causing `OutOfMemoryError` even though 50 threads shouldn't use much memory. What's happening?"
2. "Explain the four rejection policies. When would you use `CallerRunsPolicy` vs `AbortPolicy`?"
3. "How does Virtual Threads (Java 21) change the way you think about thread pools?"
</Accordion>

<Accordion title="25. Callable vs Runnable">
**Answer**:
Both represent tasks to be executed by threads, but `Callable` was introduced in Java 5 to address `Runnable`'s limitations.

*   **`Runnable`**: `void run()` — returns nothing and cannot throw checked exceptions. You can only communicate results via shared mutable state (error-prone) or by capturing a reference to a result holder. Been around since Java 1.0.

*   **`Callable<V>`**: `V call() throws Exception` — returns a typed result and can throw checked exceptions. Submitted to an `ExecutorService` via `submit()`, which returns a `Future<V>`. The `Future` is your handle to the result: `future.get()` blocks until the result is available (or throws `ExecutionException` wrapping any exception thrown by `call()`).

**The `Future` contract**:
*   `get()`: Blocks until result is ready. Can throw `InterruptedException`, `ExecutionException`, `CancellationException`.
*   `get(timeout, unit)`: Blocks with a timeout. Throws `TimeoutException` if not done.
*   `isDone()`: Non-blocking check.
*   `cancel(mayInterruptIfRunning)`: Attempts to cancel. If `mayInterruptIfRunning` is true and the task is running, the thread is interrupted (but the task must check `Thread.interrupted()` to actually stop).

**Limitations of `Future`**: No way to chain callbacks, compose multiple futures, or handle errors declaratively. You're forced to block on `get()`. This is why `CompletableFuture` (Java 8) was created — see next question.

**What interviewers are really testing:** Understanding of return values and exception propagation in concurrent tasks. Knowledge of `Future`'s blocking nature and why `CompletableFuture` was needed.

**Red flag answer:** "Callable returns a value and Runnable doesn't" — technically correct but doesn't mention `Future`, exception handling, or the blocking limitation.

**Follow-up:**
1. "If a `Callable` task throws a RuntimeException, where does that exception surface?"
2. "You have 10 Callable tasks and want to process results as they complete, not in submission order. How?"
3. "Why can't you submit a `Callable` to `Thread` directly? What's the design reason?"
</Accordion>

<Accordion title="26. CompletableFuture">
**Answer**:
`CompletableFuture` (Java 8) is Java's answer to callback hell and the limitations of `Future`. It enables non-blocking, composable, asynchronous programming.

**Core capabilities**:

*   **Async execution**: `CompletableFuture.supplyAsync(() -> fetchData(), executor)` — runs the supplier on a thread from the executor (defaults to `ForkJoinPool.commonPool()`).

*   **Chaining (pipeline)**:
    *   `thenApply(fn)`: Transform the result (like `map`). `fetchUser().thenApply(user -> user.getName())`.
    *   `thenAccept(consumer)`: Consume the result (no return). `fetchUser().thenAccept(user -> log.info("Got {}", user))`.
    *   `thenCompose(fn)`: Flat-map — chain another async operation. `fetchUser().thenCompose(user -> fetchOrders(user.getId()))`. This is the key one for avoiding nested futures.
    *   `thenRun(runnable)`: Run something after completion, ignoring the result.

*   **Combining multiple futures**:
    *   `allOf(cf1, cf2, cf3)`: Completes when ALL complete. Returns `CompletableFuture<Void>` — you need to get results individually.
    *   `anyOf(cf1, cf2, cf3)`: Completes when ANY one completes. Useful for racing (first-response-wins).
    *   `thenCombine(other, biFunction)`: Combine results of two independent futures.

*   **Error handling**:
    *   `exceptionally(fn)`: Recover from exceptions. Returns a fallback value.
    *   `handle(biFunction)`: Like `thenApply` but receives both result and exception — handle success and failure in one place.
    *   `whenComplete(biConsumer)`: Side-effect on completion (logging), doesn't transform the result.

**Thread pool gotcha**: By default, `supplyAsync` and all `*Async` variants use `ForkJoinPool.commonPool()`, which has `Runtime.getRuntime().availableProcessors() - 1` threads. If you have multiple subsystems all using the common pool, they compete for the same threads. In production, **always** pass a dedicated executor: `supplyAsync(task, myExecutor)`.

**Real-world pattern** — parallel independent calls:
```java
CompletableFuture<User> userFuture = supplyAsync(() -> userService.get(id), executor);
CompletableFuture<List<Order>> ordersFuture = supplyAsync(() -> orderService.list(id), executor);
CompletableFuture<Balance> balanceFuture = supplyAsync(() -> balanceService.get(id), executor);

CompletableFuture.allOf(userFuture, ordersFuture, balanceFuture).join();
// All three calls ran in parallel, total time = max(individual times)
```

**What interviewers are really testing:** Can you compose async operations? Do you know about the common pool trap? Can you handle errors in async pipelines?

**Red flag answer:** Using `future.get()` everywhere (blocking) instead of chaining. Or not specifying an executor (relying on the common pool).

**Follow-up:**
1. "What's the difference between `thenApply` and `thenCompose`? When do you get a `CompletableFuture<CompletableFuture<T>>`?"
2. "What happens if one of the three parallel calls in `allOf` throws an exception? How do you handle partial failures?"
3. "How do Virtual Threads (Java 21) reduce the need for CompletableFuture-style async programming?"
</Accordion>

<Accordion title="27. Atomic Variables and CAS">
**Answer**:
Atomic variables provide lock-free thread safety using hardware-level Compare-And-Swap (CAS) instructions — they're the foundation of most high-performance concurrent data structures.

**The problem**: `count++` on a shared `int` is read-modify-write (three steps). Two threads executing simultaneously can both read 5, both increment to 6, and write 6 — losing an increment. `synchronized` fixes this but blocks threads. Atomics fix it without blocking.

**Compare-And-Swap (CAS)**: A single CPU instruction (`CMPXCHG` on x86) that atomically does:
1. Read the current value at a memory location.
2. Compare it to an expected value.
3. If they match, write the new value. If not, fail.
4. Return whether the swap succeeded.

This is a single hardware operation — no lock needed. If the CAS fails (another thread modified the value), the code retries in a loop (spin). This is called an **optimistic lock** or **spin-wait**.

**Java's atomic classes** (`java.util.concurrent.atomic`):
*   `AtomicInteger`, `AtomicLong`: Atomic increment (`incrementAndGet()`), decrement, compareAndSet.
*   `AtomicReference<V>`: Atomic reference swap. Used for lock-free linked lists, stacks.
*   `AtomicStampedReference<V>`: Solves the ABA problem by including a version stamp.
*   `LongAdder` (Java 8): For high-contention counters. Instead of a single value, distributes across multiple cells (like `ConcurrentHashMap`'s segment approach). `sum()` aggregates. Much faster than `AtomicLong` under heavy contention — benchmarks show 5-10x throughput improvement with 16+ threads.

**ABA problem**: Thread reads value A, prepares to CAS(A, C). Meanwhile, another thread changes A -> B -> A. The CAS succeeds because the value is A, but the semantic meaning has changed. For pointers/references, this can cause data structure corruption. Fix: `AtomicStampedReference` includes a version number.

**When to use what**:
*   Low contention (few threads, infrequent writes): `AtomicInteger`/`AtomicLong` — simple, fast.
*   High contention (many threads, frequent writes): `LongAdder` — distributes contention.
*   Reference swapping: `AtomicReference` (simple), `AtomicStampedReference` (if ABA matters).
*   Multiple related fields must be updated atomically: `synchronized` or `ReentrantLock` — CAS can only atomically update ONE value.

**What interviewers are really testing:** Understanding CAS at the hardware level, the spin-loop retry pattern, ABA problem awareness, and knowing when CAS is insufficient (multi-variable updates).

**Red flag answer:** "AtomicInteger is thread-safe" without explaining CAS. Or not knowing about `LongAdder` for high-contention scenarios.

**Follow-up:**
1. "Explain the ABA problem with a real scenario. How does `AtomicStampedReference` fix it?"
2. "Under what conditions does CAS-based spinning become worse than a lock?"
3. "Why can't you atomically update two related variables with CAS? What's the alternative?"
</Accordion>

<Accordion title="28. Race Condition">
**Answer**:
A race condition occurs when the correctness of a program depends on the relative timing of thread execution — the output is non-deterministic and can differ between runs.

**Two main types**:
1. **Check-then-act**: Thread checks a condition, then acts on it, but the condition changes between the check and the act.
    ```java
    if (!map.containsKey(key)) {     // Check
        map.put(key, computeValue()); // Act — another thread might have put a value between check and act
    }
    ```
    Fix: Use `ConcurrentHashMap.computeIfAbsent(key, k -> computeValue())` — atomic check-and-act.

2. **Read-modify-write**: Thread reads a value, modifies it, writes it back. Another thread does the same — one update is lost.
    ```java
    balance = balance - amount; // read balance, subtract, write back — not atomic
    ```
    Fix: `synchronized`, `AtomicLong`, or database-level locking.

**Why race conditions are terrifying**: They're non-deterministic — they may pass all tests and only manifest under production load. They're timing-dependent — adding a log statement or running a debugger can change the timing enough to make them disappear (Heisenbug).

**Detection approaches**:
*   **Code review**: Look for shared mutable state accessed without synchronization.
*   **Thread sanitizers**: Java doesn't have a built-in one, but tools like FindBugs/SpotBugs can detect some patterns.
*   **Stress testing**: Run concurrent tests with high thread counts and `CountDownLatch` to maximize overlap.
*   **JCStress**: OpenJDK's concurrency stress testing framework — specifically designed to find subtle memory model violations.

**Real-world war story**: A common race condition in web applications — double-submit of a payment. User clicks "Pay" twice quickly. Two threads read the same order as "PENDING," both process payment, customer is charged twice. Fix: Use an idempotency key (UUID) and `INSERT ... ON CONFLICT DO NOTHING` at the database level, or use `SELECT ... FOR UPDATE` to lock the order row.

**What interviewers are really testing:** Can you identify race conditions in code? Do you know the patterns (check-then-act, read-modify-write)? Can you suggest the right fix for each?

**Red flag answer:** "Just add synchronized everywhere" — this kills performance and can introduce deadlocks. The fix should be targeted.

**Follow-up:**
1. "I show you this code: two threads incrementing a shared counter. Is `volatile int counter; counter++;` thread-safe? Why not?"
2. "How would you write a test that reliably reproduces a race condition?"
3. "What's the difference between a race condition and a data race in the Java Memory Model?"
</Accordion>

<Accordion title="29. ThreadLocal">
**Answer**:
`ThreadLocal` provides thread-scoped variables — each thread gets its own independent copy, eliminating the need for synchronization.

**How it works internally**: Each `Thread` object has a `ThreadLocal.ThreadLocalMap` — a hash map from `ThreadLocal` instances to values. When you call `threadLocal.get()`, it looks up the current thread's map. No synchronization needed because only the owning thread accesses its map.

**Common use cases**:
*   **Transaction context**: Spring's `@Transactional` stores the current `EntityManager`/`Connection` in a `ThreadLocal` so that all DAO calls within the same request use the same transaction. `TransactionSynchronizationManager` is built on `ThreadLocal`.
*   **User authentication context**: `SecurityContextHolder` (Spring Security) stores the authenticated user per thread. In a servlet container, each request is handled by one thread, so `ThreadLocal` naturally scopes to the request.
*   **SimpleDateFormat**: It's not thread-safe, so before Java 8, developers wrapped it in `ThreadLocal` to avoid creating a new instance per call. (Post-Java-8, just use `DateTimeFormatter` which is thread-safe.)
*   **Request ID / MDC**: SLF4J's `MDC` (Mapped Diagnostic Context) uses `ThreadLocal` to associate request IDs with log statements — essential for tracing in microservices.

**The memory leak trap**: In thread pools (Tomcat, Executors), threads are reused across requests. If you set a `ThreadLocal` and don't clear it, the value persists for the next request that uses that thread — causing:
1. **Memory leak**: Large objects retained indefinitely across pool-member threads.
2. **Data leakage**: User A's context leaking into User B's request. I've seen a production bug where user authentication contexts leaked between requests because `ThreadLocal.remove()` wasn't called in a `finally` block.

**Fix**: ALWAYS clean up in a `finally` block, a servlet filter, or Spring's `HandlerInterceptor.afterCompletion()`.

**Virtual Threads complication (Java 21)**: Virtual threads are cheap and not pooled the same way — but `ThreadLocal` still works with them (each virtual thread has its own ThreadLocalMap). However, if you create millions of virtual threads, each with a ThreadLocal, memory usage can explode. Java 21 introduced **Scoped Values** (`ScopedValue`) as a more efficient alternative for virtual threads.

**What interviewers are really testing:** Understanding of the memory leak in thread pools, proper cleanup, and the interaction with Virtual Threads.

**Red flag answer:** "ThreadLocal gives each thread its own variable" without mentioning the cleanup requirement or the thread pool leak scenario.

**Follow-up:**
1. "Walk me through exactly how a ThreadLocal memory leak happens in Tomcat. What retains what?"
2. "What are Scoped Values in Java 21 and why do they replace ThreadLocal for virtual threads?"
3. "How does Spring Security's `SecurityContextHolder` use ThreadLocal, and what happens in async (`@Async`) methods?"
</Accordion>

<Accordion title="30. Fork/Join Framework">
**Answer**:
The Fork/Join Framework (Java 7) is a specialized thread pool designed for recursive divide-and-conquer parallelism. It's the engine behind parallel streams.

**Core concept**:
1. **Fork**: Split a large task into smaller subtasks recursively (like merge sort splitting an array).
2. **Compute**: When subtasks are small enough (below a threshold), compute them directly.
3. **Join**: Combine results from subtasks.

**Key classes**:
*   `ForkJoinPool`: The thread pool. Default parallelism = number of CPU cores. The `commonPool()` is a shared instance used by parallel streams and `CompletableFuture` by default.
*   `RecursiveTask<V>`: Returns a result. Override `compute()`.
*   `RecursiveAction`: No result. Override `compute()`.

**Work Stealing algorithm** — this is the clever part:
Each worker thread has its own double-ended queue (deque) of tasks. When a worker finishes its tasks, it **steals** tasks from the tail of another worker's deque. Benefits:
*   Naturally load-balances across cores even with irregular task sizes.
*   Stealing from the tail (while the owner works from the head) minimizes contention.
*   Idle threads are put to productive use.

**When to use vs when to avoid**:
*   **Good for**: CPU-bound, recursively decomposable problems. Sorting, matrix multiplication, tree traversal, image processing.
*   **Bad for**: I/O-bound tasks. If a thread blocks on I/O, it wastes a pool slot. The Fork/Join pool tries to compensate by creating extra threads, but this is a band-aid. Use Virtual Threads or a separate I/O executor for I/O.
*   **Bad for**: Tasks with shared mutable state. Fork/Join assumes independent subtasks.

**Parallel Streams gotcha**: `list.parallelStream().map(...)` uses the common `ForkJoinPool`. If multiple parallel streams run concurrently, they share the same limited pool. A slow parallel stream in one subsystem can starve another. Mitigation: submit to a custom `ForkJoinPool`:
```java
ForkJoinPool custom = new ForkJoinPool(4);
custom.submit(() -> list.parallelStream().map(...).collect(...)).get();
```

**What interviewers are really testing:** Understanding of work stealing, the relationship to parallel streams, and knowing when Fork/Join is the wrong choice (I/O-bound tasks).

**Red flag answer:** "Fork/Join splits tasks into smaller tasks" without mentioning work stealing. Or not knowing about the common pool issue with parallel streams.

**Follow-up:**
1. "How does work stealing reduce contention compared to a shared task queue?"
2. "Your parallel stream processing is slower than sequential. What could be the causes?"
3. "What's the threshold heuristic for when to stop forking and compute directly? How do you tune it?"
</Accordion>

<Accordion title="31. Dependency Injection and IoC">
**Answer**:
Inversion of Control (IoC) is the principle; Dependency Injection (DI) is the mechanism. Together, they form the foundation of Spring — and understanding *why* they exist is more important than knowing the annotations.

**The problem DI solves**: Without DI, a class creates its own dependencies:
```java
class OrderService {
    private PaymentGateway gateway = new StripeGateway(); // tightly coupled
}
```
This is untestable (can't substitute a mock), inflexible (changing to PayPal requires code changes), and violates the Dependency Inversion Principle (depending on a concrete implementation, not an abstraction).

**With DI**:
```java
class OrderService {
    private final PaymentGateway gateway; // depends on interface
    OrderService(PaymentGateway gateway) { this.gateway = gateway; } // injected
}
```
The Spring container (ApplicationContext) creates the `OrderService`, sees it needs a `PaymentGateway`, finds a bean implementing that interface, and injects it.

**Three injection types** (and the right choice):
1. **Constructor injection** (recommended): All dependencies in the constructor. Ensures the object is fully initialized. Enables `final` fields (immutability). Spring auto-detects when there's only one constructor — no `@Autowired` needed.
2. **Setter injection**: Via `@Autowired` on setters. Allows optional dependencies. But the object can exist in a partially initialized state — dangerous.
3. **Field injection**: `@Autowired` on fields directly. Shortest code but terrible for testing (can't inject mocks without reflection or Spring context), hides dependencies (can't see them from the constructor), prevents `final` fields. Most teams' code review standards prohibit this.

**How Spring resolves beans**:
*   By type first. If there's one `PaymentGateway` implementation, it's injected.
*   If multiple implementations exist: Use `@Primary` (default choice), `@Qualifier("stripe")` (explicit choice), or `@Profile("prod")` (environment-based).
*   `@Lazy`: Don't create the bean until it's first accessed. Useful for breaking circular dependencies or expensive initialization.

**What interviewers are really testing:** Why DI matters (testability, flexibility, SOLID). That you prefer constructor injection and can articulate why. Understanding of how Spring resolves ambiguous beans.

**Red flag answer:** "I use `@Autowired` on fields because it's less code" — this shows prioritizing convenience over correctness. Or not knowing why DI matters beyond "Spring does it."

**Follow-up:**
1. "Why is constructor injection preferred over field injection? What specifically breaks when you test with field injection?"
2. "Two beans implement `PaymentGateway`. How does Spring decide which one to inject? Walk me through the resolution order."
3. "What's the difference between `BeanFactory` and `ApplicationContext`?"
</Accordion>

<Accordion title="32. Bean Lifecycle">
**Answer**:
Understanding the bean lifecycle is crucial for proper initialization, cleanup, and debugging "bean not found" or "bean not ready" issues.

**Full lifecycle (in order)**:
1. **Instantiation**: Spring creates the object (via constructor).
2. **Dependency injection**: Properties and dependencies are set (`@Autowired`, setter injection).
3. **Aware interfaces** (if implemented):
    *   `BeanNameAware.setBeanName()` — bean learns its own name.
    *   `BeanFactoryAware.setBeanFactory()` — gets reference to the factory.
    *   `ApplicationContextAware.setApplicationContext()` — gets the context.
4. **`BeanPostProcessor.postProcessBeforeInitialization()`**: Applied to ALL beans. Frameworks use this to modify beans before initialization. Example: `@PostConstruct` is processed here.
5. **Initialization**:
    *   `@PostConstruct` method runs.
    *   `InitializingBean.afterPropertiesSet()` (older approach).
    *   Custom `init-method`.
6. **`BeanPostProcessor.postProcessAfterInitialization()`**: Applied to ALL beans. This is where **AOP proxies are created** — critical detail. The `@Transactional` proxy wraps the bean here.
7. **Bean is ready** — placed in the application context, ready for use.
8. **Destruction** (on context shutdown):
    *   `@PreDestroy` method runs.
    *   `DisposableBean.destroy()` (older approach).
    *   Custom `destroy-method`.

**Why the ordering matters**: AOP proxies are created in step 6 (after initialization). This means inside `@PostConstruct`, `this` is the raw bean, not the proxy. If you call a `@Transactional` method from `@PostConstruct`, the transaction annotation is IGNORED because the proxy doesn't exist yet. This is a common and subtle bug.

**`BeanPostProcessor` is the magic behind Spring**: Almost all Spring features are implemented as BeanPostProcessors:
*   `@Autowired` -> `AutowiredAnnotationBeanPostProcessor`
*   `@Transactional`/AOP -> `AbstractAutoProxyCreator`
*   `@Scheduled` -> `ScheduledAnnotationBeanPostProcessor`

**What interviewers are really testing:** Do you know the ordering? Do you understand when AOP proxies are created and why `@PostConstruct` calling `@Transactional` methods doesn't work?

**Red flag answer:** Just listing steps without understanding the proxy-creation timing issue.

**Follow-up:**
1. "Why does calling a `@Transactional` method from `@PostConstruct` fail? What's the workaround?"

<Accordion title="33. Spring Bean Scopes">
**Answer**:
Bean scope determines the lifecycle and visibility of a bean instance. Getting it wrong causes subtle bugs — especially when mixing scopes.

*   **Singleton** (default): One instance per Spring IoC container (not per JVM — if you have two `ApplicationContext`s, you get two instances). Created eagerly at startup (unless `@Lazy`). 95% of your beans should be singletons. Must be thread-safe since multiple threads share the same instance.

*   **Prototype**: A new instance every time the bean is requested from the container. Spring creates it and hands it off — does NOT manage its lifecycle after creation (`@PreDestroy` is NOT called). Use for stateful, non-thread-safe objects (builders, form objects).

*   **Request** (`@RequestScope`): One instance per HTTP request. Only available in web-aware contexts. Useful for request-scoped data (current user's permissions, request correlation ID).

*   **Session** (`@SessionScope`): One instance per HTTP session. Lives as long as the session. Use for shopping cart, user preferences.

*   **Application**: One per `ServletContext`. Rarely used — similar to singleton but scoped to the web app.

*   **WebSocket**: One per WebSocket session.

**The scope mismatch trap**: Injecting a shorter-lived scoped bean (Request) into a longer-lived one (Singleton) doesn't work as expected:
```java
@Service // Singleton
class OrderService {
    @Autowired
    private RequestContext context; // Request-scoped — but this is a singleton!
}
```
The singleton is created once at startup and gets ONE instance of `RequestContext`. Every request uses the same stale instance. Fix: Inject a **scoped proxy** (`@Scope(value = "request", proxyMode = ScopedProxyMode.TARGET_CLASS)`) — Spring injects a CGLIB proxy that delegates to the current request's instance on each call. Or use `ObjectFactory<RequestContext>` / `Provider<RequestContext>`.

**What interviewers are really testing:** Singleton as default and thread-safety implications. The scope mismatch problem and how scoped proxies solve it. That Prototype beans aren't managed after creation.

**Red flag answer:** Just listing the scopes without explaining the injection-mismatch problem.

**Follow-up:**
1. "You inject a Prototype bean into a Singleton. How many Prototype instances are created? How do you fix this?"
2. "Is a Singleton bean the same as the Singleton design pattern? What's the difference?"
3. "Why doesn't Spring call `@PreDestroy` on Prototype beans? How do you clean up Prototype resources?"

<Accordion title="34. @SpringBootApplication and Auto-Configuration">
**Answer**:
`@SpringBootApplication` is the entry point annotation that makes Spring Boot "magic" work — but understanding what's behind it removes the magic and makes debugging possible.

**It's a composite of three annotations**:
1. **`@Configuration`**: Marks the class as a source of bean definitions. Equivalent to an XML config file. Uses CGLIB proxying to ensure `@Bean` methods called internally return the same singleton instance (not a new object every call).
2. **`@EnableAutoConfiguration`**: The heart of Spring Boot. Tells Spring to automatically configure beans based on what's on the classpath. If `spring-boot-starter-web` is in your dependencies, it auto-configures an embedded Tomcat, a `DispatcherServlet`, Jackson for JSON, etc. — all without you writing a single `@Bean` definition.
3. **`@ComponentScan`**: Scans the package (and sub-packages) of the annotated class for `@Component`, `@Service`, `@Repository`, `@Controller` annotations and registers them as beans.

**How auto-configuration works internally**:
*   Spring Boot ships hundreds of `*AutoConfiguration` classes (e.g., `DataSourceAutoConfiguration`, `JacksonAutoConfiguration`).
*   These are listed in `META-INF/spring/org.springframework.boot.autoconfigure.AutoConfiguration.imports` (Spring Boot 3) or the older `META-INF/spring.factories`.
*   Each auto-configuration class is annotated with conditions:
    *   `@ConditionalOnClass(DataSource.class)` — only apply if the class is on the classpath.
    *   `@ConditionalOnMissingBean(DataSource.class)` — only apply if the user hasn't defined their own.
    *   `@ConditionalOnProperty("spring.datasource.url")` — only apply if the property exists.

**Debugging auto-configuration**: Add `--debug` flag or `debug=true` in `application.properties`. Spring Boot prints a report showing which auto-configurations were applied (Positive matches) and which were skipped (Negative matches) and why.

**What interviewers are really testing:** Do you understand the conditional mechanism? Can you override auto-configured beans? Can you debug why a bean wasn't created?

**Red flag answer:** "It's a convenience annotation that sets everything up" — this shows no understanding of the mechanism.

<Accordion title="36. @PathVariable vs @RequestParam">
**Answer**:
This is really a REST API design question disguised as an annotation question — the interviewer wants to see if you understand RESTful URL conventions.

*   **`@PathVariable`**: Extracts values from the URL path. Used for **resource identification**.
    *   `GET /users/42` -> `@GetMapping("/users/{id}")` -> `@PathVariable Long id` = 42
    *   `GET /users/42/orders/7` -> nested resources
    *   Semantic meaning: "the resource IS the URL." The path is the identity.

*   **`@RequestParam`**: Extracts values from query string parameters. Used for **filtering, sorting, pagination, and optional modifiers**.
    *   `GET /users?status=active&sort=name&page=2` -> `@RequestParam String status`, etc.
    *   By default, required. Use `required = false` and `defaultValue` for optional params.
    *   `@RequestParam List<String> status` handles `?status=active&status=pending`.

**Design guidelines**:
*   Resources and their IDs go in the path: `/orders/123`, `/products/abc-123`.
*   Filtering, sorting, pagination, and format go in query params: `?category=electronics&sort=price&page=3`.
*   Never put verbs in the URL: `/users/42/activate` is debatable but `/activateUser?id=42` is definitely wrong REST.
*   Use `@RequestBody` (not either of these) for POST/PUT payloads.

**Other annotations to know**: `@RequestHeader` (extract headers), `@CookieValue` (extract cookies), `@MatrixVariable` (semicolon-separated params in path segments — rare).

**What interviewers are really testing:** REST API design sense. Do you know when each is appropriate? Can you design clean, consistent URLs?

**Red flag answer:** "PathVariable is in the URL and RequestParam is after the question mark" — technically correct but shows no understanding of RESTful design principles.

**Follow-up:**
1. "Design the URL structure for a blogging platform: users, posts, comments, tags. Which parts are path variables vs query params?"

<Accordion title="37. Spring Data JPA">
**Answer**:
Spring Data JPA eliminates boilerplate repository code by generating implementations from interface method signatures — but understanding what happens under the hood is critical for avoiding performance disasters.

**Core abstractions**:
*   `JpaRepository<Entity, ID>` extends `CrudRepository` extends `Repository`. Provides `save()`, `findById()`, `findAll()`, `delete()`, `count()`, pagination, sorting out of the box.
*   **Derived queries**: Spring parses method names and generates JPQL:

*   **Custom queries**: When derived queries aren't enough:

<Accordion title="38. Transaction Management (@Transactional)">
**Answer**:
`@Transactional` is deceptively simple to use and deceptively complex to get right. Most production transaction bugs come from misunderstanding how the proxy works.

**How it works**:
Spring creates a CGLIB or JDK dynamic **proxy** around your bean. When a `@Transactional` method is called:
1. The proxy intercepts the call.
2. Gets a database connection from the `DataSource` / `EntityManager` from the `EntityManagerFactory`.
3. Begins a transaction.
4. Delegates to the actual method.
5. If the method returns normally: **commit**.
6. If the method throws an unchecked exception (RuntimeException or Error): **rollback**.
7. If the method throws a checked exception: **commit** (by default! This surprises people).

**Critical gotchas**:

1. **Self-invocation bypass**: If method A (non-transactional) calls method B (transactional) **on the same object** (`this.methodB()`), the proxy is bypassed — B runs without a transaction. This is because `this` is the raw object, not the proxy. Fix: inject the bean into itself (`@Lazy` self-reference), use `AopContext.currentProxy()`, or restructure to call from another bean.

2. **Checked exception rollback**: `@Transactional` does NOT rollback on checked exceptions by default. You need `@Transactional(rollbackFor = Exception.class)` or `@Transactional(rollbackFor = MyCheckedException.class)`. I've seen production data inconsistencies caused by this.

3. **Propagation levels** (know at least these three):
    *   `REQUIRED` (default): Join existing transaction or create new one.
    *   `REQUIRES_NEW`: Always create a new transaction. Suspends the outer one. Use for audit logging that must persist even if the outer transaction rolls back.
    *   `NOT_SUPPORTED`: Runs without a transaction. Suspends any existing one.

4. **Isolation levels**: `READ_COMMITTED` (default for most databases), `REPEATABLE_READ`, `SERIALIZABLE`. Higher isolation = fewer anomalies but more locking/contention.

5. **Read-only optimization**: `@Transactional(readOnly = true)` — Hibernate skips dirty checking, some databases route to read replicas. Always use for read-only operations.

**What interviewers are really testing:** The self-invocation trap, checked exception rollback behavior, and propagation levels. These are the top three sources of production bugs with `@Transactional`.

**Red flag answer:** "I put `@Transactional` on the service method and it handles transactions" without knowing about self-invocation or checked exception rollback.

**Follow-up:**
1. "Method A calls method B, both in the same class. A is `@Transactional`, B is `@Transactional(propagation = REQUIRES_NEW)`. Does B get a new transaction? Why or why not?"
2. "Your method catches a `RuntimeException`, logs it, and re-throws a custom checked exception. Does the transaction roll back?"
3. "How would you test that `@Transactional` rollback is working correctly?"
</Accordion>

<Accordion title="39. Spring AOP (Aspect Oriented Programming)">
**Answer**:
AOP lets you extract cross-cutting concerns (logging, security, transactions, metrics) from business logic — Spring's `@Transactional`, `@Secured`, `@Cacheable`, and `@Async` all work through AOP.

**Core terminology**:
*   **Aspect**: The module containing cross-cutting logic. A class annotated with `@Aspect`.
*   **Advice**: The action to take and when. Types:
    *   `@Before`: Runs before the target method. Use for validation, logging entry.
    *   `@After`: Runs after (regardless of outcome). Use for cleanup.
    *   `@AfterReturning`: Runs after successful return. Can access the return value.
    *   `@AfterThrowing`: Runs after an exception. Can access the exception.
    *   `@Around`: Wraps the method entirely. Most powerful — controls whether the method executes at all, can modify arguments and return value. Use for timing, circuit breaking, retry logic.
*   **Pointcut**: The WHERE — which methods the advice applies to.
    *   `@Pointcut("execution(* com.myapp.service.*.*(..))")` — all methods in service package.
    *   `@Pointcut("@annotation(com.myapp.Timed)")` — methods with `@Timed` annotation.
    *   Can be combined: `@Pointcut("serviceLayer() && !loggingExcluded()")`.
*   **JoinPoint**: The actual method invocation point. `ProceedingJoinPoint` (in `@Around`) lets you call `proceed()` to execute the target method.

**How Spring AOP works (proxy-based)**:
Spring AOP uses **runtime proxies**, not bytecode weaving. Two proxy types:
*   **JDK Dynamic Proxy**: When the target implements an interface. Proxy implements the same interface.
*   **CGLIB Proxy**: When the target is a concrete class. Creates a subclass at runtime (via bytecode generation). Spring Boot uses CGLIB by default since Spring Boot 2.

**Limitation of proxy-based AOP**: Only works for public method calls from outside the object. Internal calls (`this.method()`) bypass the proxy — same limitation as `@Transactional`. If you need to advise private methods or field access, you need full AspectJ (compile-time or load-time weaving).

**Real-world example** — method timing:
```java
@Around("@annotation(Timed)")
public Object timeMethod(ProceedingJoinPoint pjp) throws Throwable {
    long start = System.nanoTime();
    try {
        return pjp.proceed();
    } finally {
        long duration = System.nanoTime() - start;
        metrics.timer(pjp.getSignature().getName()).record(duration, NANOSECONDS);
    }
}
```

**What interviewers are really testing:** Understanding of proxy-based AOP limitations (self-invocation), the difference between advice types, and when to use `@Around` vs simpler advice types.

**Red flag answer:** Describing AOP only theoretically ("separation of concerns") without mentioning the proxy mechanism or its limitations.

**Follow-up:**
1. "You apply `@Cacheable` and `@Transactional` to the same method. In what order do the aspects execute? Can you control it?"
2. "Why does Spring Boot default to CGLIB proxies instead of JDK dynamic proxies?"
3. "What's the performance overhead of AOP proxies? When does it matter?"
</Accordion>

<Accordion title="41. RestTemplate vs WebClient">
**Answer**:
This is really a question about blocking vs non-blocking I/O — one of the most important architectural decisions in modern microservices.

*   **RestTemplate**: Synchronous, blocking HTTP client. Each HTTP call blocks the calling thread until the response arrives. Thread is idle during network wait. Simple API: `restTemplate.getForObject(url, User.class)`. Officially in **maintenance mode** since Spring 5 (not deprecated yet, but no new features). Adequate for low-throughput services or batch jobs where simplicity matters more than throughput.

*   **WebClient** (Spring WebFlux): Non-blocking, reactive HTTP client. The calling thread is released during network wait and can handle other requests. Returns `Mono<T>` or `Flux<T>`. Can also be used in a blocking way (`block()`) as a modern replacement for RestTemplate even in non-reactive apps.
    ```java
    webClient.get()
        .uri("/users/{id}", id)
        .retrieve()
        .bodyToMono(User.class)
        .subscribe(user -> process(user));
    ```

**Why the difference matters at scale**: If your service makes 3 downstream HTTP calls per request (database, auth service, payment service) and each takes 100ms, with RestTemplate you need one thread blocked for 300ms per request. At 1000 RPS, you need 300 threads just waiting on I/O. With WebClient (non-blocking), a handful of threads can handle all 1000 requests — the threads aren't blocked, they're processing other requests while waiting for I/O.

**The new option — HTTP Interface Clients** (Spring 6 / Spring Boot 3):
```java
@HttpExchange("/users")
interface UserClient {
    @GetExchange("/{id}")
    User getUser(@PathVariable Long id);
}
```
Spring generates the implementation — similar to Feign but built into Spring. Can use either RestClient (blocking) or WebClient (reactive) as the underlying transport.

**RestClient** (Spring 6.1): New synchronous client with a modern, fluent API similar to WebClient. Intended as the true RestTemplate replacement for non-reactive applications.

**What interviewers are really testing:** Understanding of blocking vs non-blocking I/O and the throughput implications. Awareness of the current landscape (RestClient, HTTP Interfaces).

**Red flag answer:** "RestTemplate is deprecated so use WebClient" — RestTemplate is in maintenance mode, not deprecated. And for non-reactive apps, RestClient is the recommended replacement, not necessarily WebClient.

**Follow-up:**
1. "Your service makes 5 downstream HTTP calls per request. With RestTemplate, you're running out of threads at 500 RPS. What are your options?"
2. "Can you use WebClient in a non-reactive (Spring MVC) application? How?"
3. "What's the difference between WebClient and the new RestClient in Spring 6.1?"
</Accordion>

<Accordion title="43. Annotation vs XML Configuration">
**Answer**:
This question is partly historical context and partly about understanding how Spring configuration has evolved — and when XML might still appear.

**The evolution**:
1. **Spring 1-2 era (2004-2006)**: Everything was XML. `applicationContext.xml` with `<bean id="..." class="...">`. Verbose, error-prone (typos in class names caught only at runtime), no IDE refactoring support.
2. **Spring 2.5 (2007)**: Annotations introduced — `@Component`, `@Autowired`, `@Service`. Component scanning replaced manual XML bean declarations.
3. **Spring 3+ (2009)**: Java-based configuration — `@Configuration` + `@Bean`. Full type safety, IDE support, programmatic logic in configuration.
4. **Spring Boot (2014)**: Auto-configuration made most explicit configuration unnecessary.

**Why annotations/Java config won**:
*   **Type safety**: Refactoring a class name updates the configuration. XML breaks silently.
*   **IDE support**: Autocomplete, navigation, compile-time checks.
*   **Proximity**: Configuration lives near the code it configures.
*   **Conditional logic**: `@Conditional*` annotations enable dynamic configuration that XML can't express cleanly.

**When XML still appears**:
*   Legacy applications that haven't been migrated.
*   Some advanced Spring Integration or Spring Batch flows where XML DSL is more expressive.
*   Third-party library configuration that only supports XML.
*   XML is still valid and supported — Spring doesn't force migration.

**What interviewers are really testing:** Whether you understand the evolution and can articulate why annotations are preferred. If you've worked with legacy Spring, you should be comfortable with both.

**Red flag answer:** Not knowing that XML configuration exists, or claiming it's "completely obsolete."

<Accordion title="44. Handling Circular Dependencies">
**Answer**:
A circular dependency occurs when Bean A depends on Bean B and Bean B depends on Bean A — either directly or transitively. Spring Boot 2.6+ **disallows** this by default (it was allowed in earlier versions with a warning).

**Why it happens**: Often a design smell. `OrderService` needs `PaymentService` (to process payments), and `PaymentService` needs `OrderService` (to update order status). This tight coupling suggests the responsibilities aren't cleanly separated.

**Workarounds** (in order of preference):

1. **Refactor the design** (best solution): Extract the shared logic into a third class. Maybe `OrderStatusUpdater` that `PaymentService` calls, breaking the cycle. If two beans both need each other, they're probably doing too much.

2. **Use events**: Instead of `PaymentService` calling `OrderService.updateStatus()` directly, publish a `PaymentCompletedEvent`. `OrderService` listens for it. This decouples the two completely. Spring's `ApplicationEventPublisher` makes this easy.

3. **`@Lazy` on one dependency**: `@Lazy @Autowired PaymentService paymentService` — Spring injects a proxy that doesn't trigger creation of `PaymentService` until first use. By that time, `OrderService` is fully created, breaking the cycle. This is a quick fix but doesn't address the underlying design issue.

4. **Setter injection instead of constructor injection**: Constructor injection requires all dependencies at creation time (chicken-and-egg). Setter injection allows Spring to create both beans first, then inject dependencies. But this means your beans have a partially-initialized window — not ideal.

5. **`ObjectFactory<T>` / `Provider<T>`**: Wrap the dependency in a lazy provider. `private final ObjectFactory<PaymentService> paymentServiceFactory;` — resolved lazily on first `getObject()` call.

**Spring Boot 2.6+ behavior**: Circular dependencies throw `BeanCurrentlyInCreationException` at startup by default. You can allow them with `spring.main.allow-circular-references=true`, but this should be treated as tech debt, not a permanent solution.

**What interviewers are really testing:** Do you see circular dependencies as a design problem first? Can you suggest the refactoring approach before the workaround? Do you know about the Spring Boot 2.6+ change?

**Red flag answer:** Jumping straight to `@Lazy` without questioning the design.

**Follow-up:**
1. "Why did Spring Boot 2.6 change the default to disallow circular dependencies? What problems were they seeing?"
2. "How would you use Spring Events to break a circular dependency between OrderService and NotificationService?"
3. "How do you detect circular dependencies before they blow up at startup?"
</Accordion>

<Accordion title="45. Hibernate N+1 Problem">
**Answer**:
The N+1 problem is the single most common performance issue in JPA/Hibernate applications. It's subtle, doesn't show up in unit tests, and can bring a production database to its knees.

**The problem**: You load a list of N parent entities (1 query). Then, for each parent, Hibernate lazily loads the associated children (N queries). Total: N+1 queries.

```java
List<Author> authors = authorRepo.findAll(); // 1 query: SELECT * FROM authors
for (Author a : authors) {
    a.getBooks().size(); // N queries: SELECT * FROM books WHERE author_id = ?
}
```
If you have 100 authors, that's 101 queries. With 10,000 authors, it's 10,001 queries. I've seen this pattern generate 50,000+ queries for a single API call — response times went from 50ms to 30 seconds.

**Solutions**:

1. **`JOIN FETCH` in JPQL** (most common fix):
    ```java
    @Query("SELECT a FROM Author a JOIN FETCH a.books")
    List<Author> findAllWithBooks();
    ```
    Single query with a JOIN. Loads all data at once. **Gotcha**: Cannot use `JOIN FETCH` with pagination (`Pageable`) — Hibernate fetches ALL results into memory and paginates in-memory ("HHH000104: firstResult/maxResults specified with collection fetch; applying in memory!").

2. **`@EntityGraph`** (JPA 2.1+):
    ```java
    @EntityGraph(attributePaths = {"books"})
    List<Author> findAll();
    ```
    Declarative eager loading. Cleaner than JPQL for simple cases.

3. **`@BatchSize(size = 25)`** on the collection: Instead of loading one author's books at a time, loads 25 at a time with `IN` clause. Reduces N+1 to N/25+1. A good "blanket" optimization.

4. **`@Fetch(FetchMode.SUBSELECT)`**: Loads ALL children in one subselect query. Total: 2 queries regardless of N.

5. **DTO projection**: Skip entities entirely for read-only operations:
    ```java
    @Query("SELECT new com.myapp.AuthorBookDto(a.name, b.title) FROM Author a JOIN a.books b")
    List<AuthorBookDto> findAuthorBooks();
    ```
    No entity management overhead, no lazy loading surprises. Best performance.

**Detection**: Enable Hibernate SQL logging (`spring.jpa.show-sql=true` or `logging.level.org.hibernate.SQL=DEBUG`). Count queries per request. Use tools like `datasource-proxy` with query counting assertions in tests.

**What interviewers are really testing:** Can you explain the problem with specific query counts? Do you know multiple solutions and when each is appropriate? Have you detected and fixed this in production?

**Red flag answer:** Only mentioning `FetchType.EAGER` as a fix — this causes ALL associations to be eagerly loaded everywhere, leading to massive unnecessary data loading and potential Cartesian product explosions.

**Follow-up:**
1. "Why can't you use `JOIN FETCH` with Spring Data JPA pagination? What's the workaround?"
2. "You switch to `FetchType.EAGER` to fix N+1 and now your API returns 10MB responses. What went wrong?"
3. "How would you write an integration test that fails if a method triggers more than 3 SQL queries?"
</Accordion>

<Accordion title="46. DTO Pattern">
**Answer**:
The Data Transfer Object pattern separates what your database stores from what your API exposes — and it's not optional in any serious application.

**Why you need DTOs**:
1. **Security**: Your `User` entity has a `passwordHash` field. Without a DTO, Jackson serializes the entire entity — including the hash — to the API response. DTOs explicitly include only safe fields.
2. **Decoupling**: Your entity structure matches the database schema (normalized, with JPA annotations). Your API structure matches what the client needs (often denormalized, different naming). These evolve independently — you don't want a database migration to break your API contract.
3. **Performance**: Entities carry lazy-loaded proxies, Hibernate metadata, bidirectional references (infinite serialization loops). DTOs are plain POJOs — fast to serialize, no surprises.
4. **Validation**: Input DTOs can have `@Valid` annotations tailored to the API contract (`@NotBlank`, `@Size`), separate from entity constraints.

**DTO mapping approaches**:

*   **Manual mapping**: Write a static method or constructor. Full control, most verbose, most type-safe. Fine for a few entities.
*   **MapStruct** (recommended for most projects): Compile-time code generation. `@Mapper` interface, MapStruct generates the implementation class at build time. Zero runtime overhead, compile-time safety, good IDE support. Handles nested mappings, type conversions, custom mappings.
*   **ModelMapper**: Runtime reflection-based. Convention-over-configuration (matches by field name). Quick to set up but slower, harder to debug, can have surprising behavior with nested objects.
*   **Java Records** (Java 16+): Perfect for DTOs — immutable, compact, auto-generated `equals`/`hashCode`/`toString`. `record UserResponse(Long id, String name, String email) {}`

**Request vs Response DTOs**: Often you need different shapes:
*   `CreateUserRequest` — no `id`, includes `password`.
*   `UpdateUserRequest` — partial fields, maybe nullable.
*   `UserResponse` — has `id`, no `password`, maybe includes computed fields like `memberSince`.

**What interviewers are really testing:** Understanding of why DTOs exist (security, decoupling, performance), knowledge of mapping tools, and the discipline to use them consistently.

**Red flag answer:** "I return entities directly from my controller" — this is a security and maintenance nightmare.

**Follow-up:**
1. "Your team says 'DTOs are too much boilerplate, let's just use entities.' How do you convince them?"
2. "How does MapStruct compare to ModelMapper in terms of performance and debuggability?"
3. "How do you handle partial updates (PATCH) with DTOs? Not all fields should be updated."

**Answer**:

*   **`@SpringBootTest`** (Integration test): Loads the FULL application context. All beans, all auto-configuration. <Accordion title="48. Spring Security">
**Answer**:
Spring Security is a powerful but complex framework built around a **filter chain** architecture. Understanding the chain is key to customizing security correctly.

**Architecture**:
1. An HTTP request enters the servlet container (Tomcat).
2. It passes through Spring Security's `FilterChainProxy`, which delegates to a `SecurityFilterChain`.
3. The chain contains ordered filters — each responsible for one concern:
    *   `CorsFilter`: CORS handling.
    *   `CsrfFilter`: CSRF token validation.
    *   `UsernamePasswordAuthenticationFilter`: Form login processing.
    *   `BearerTokenAuthenticationFilter`: JWT/OAuth2 token extraction.
    *   `AuthorizationFilter`: Checks if the authenticated user has permission for the requested resource.
    *   `ExceptionTranslationFilter`: Converts security exceptions to HTTP responses (401/403).

**Authentication vs Authorization**:
*   **Authentication** (WHO are you?): Validates credentials. Returns an `Authentication` object with the principal, credentials, and granted authorities. Stored in `SecurityContextHolder` (ThreadLocal).
*   **Authorization** (WHAT can you access?): Checks if the authenticated user has the required role/permission. `@PreAuthorize("hasRole('ADMIN')")`, `@Secured("ROLE_USER")`, or URL-based rules in `SecurityFilterChain` configuration.

**JWT authentication flow** (most common in modern APIs):
1. Client sends `Authorization: Bearer <token>` header.
2. `BearerTokenAuthenticationFilter` extracts the token.
3. Token is validated (signature verification, expiry check).
4. On success, creates an `Authentication` object and sets it in `SecurityContext`.
5. Subsequent filters and the controller can access the authenticated user.

**OAuth2 / OpenID Connect**: Spring Security has first-class support for both OAuth2 client (your app needs to call a protected API) and OAuth2 resource server (your app IS the protected API). `spring-boot-starter-oauth2-resource-server` auto-configures JWT validation.

**Spring Security 6 / Spring Boot 3 changes**: Lambda DSL is the default. `SecurityFilterChain` bean replaces `WebSecurityConfigurerAdapter` (removed). Explicit `requestMatchers()` replaces `antMatchers()`.

**What interviewers are really testing:** Understanding of the filter chain architecture, the difference between authentication and authorization, and practical JWT/OAuth2 experience.

**Red flag answer:** "I use `@Secured` and it protects my endpoints" without understanding the filter chain or how authentication actually works.

<Accordion title="49. Reactive Programming (Spring WebFlux)">
**Answer**:
Reactive programming is about building non-blocking, event-driven applications that handle high concurrency with minimal threads. Spring WebFlux is the reactive alternative to Spring MVC.

**Core concepts** (Project Reactor):
*   **`Mono<T>`**: A reactive type that emits 0 or 1 item. Think of it as an async `Optional`. `Mono<User> user = userRepo.findById(id);`
*   **`Flux<T>`**: A reactive type that emits 0 to N items. Think of it as an async `List`/`Stream`. `Flux<Order> orders = orderRepo.findByUserId(id);`
*   Both are **lazy** — nothing happens until you subscribe. The pipeline is a blueprint, not an execution.

**Operators** (similar to Stream API but async):
*   Transform: `map()`, `flatMap()` (async), `filter()`
*   Combine: `zip()`, `merge()`, `concat()`
*   Error: `onErrorResume()`, `onErrorReturn()`, `retry()`
*   Side effects: `doOnNext()`, `doOnError()`, `doOnComplete()`

**Back-pressure**: The consumer tells the producer how many items it can handle. If the producer is faster than the consumer, Reactor buffers, drops, or signals the producer to slow down. Strategies: `onBackpressureBuffer()`, `onBackpressureDrop()`, `onBackpressureLatest()`. This prevents `OutOfMemoryError` from unbounded buffering.

**Threading model**: WebFlux uses a small number of event-loop threads (Netty's NIO — typically core count). A single thread handles many concurrent requests by never blocking. If you block on an event-loop thread (JDBC call, `Thread.sleep()`), you freeze all concurrent requests on that thread.

**When to use WebFlux vs MVC**:
*   **WebFlux**: High concurrency with I/O-bound workloads (API gateways, chat servers, streaming, SSE). When you need to handle 10,000+ concurrent connections.
*   **MVC**: Most applications. Simpler programming model, easier debugging (stack traces are readable), more library support (JDBC is blocking). When you have 100-1,000 concurrent connections, MVC with thread pools is simpler and sufficient.
*   **Virtual Threads (Java 21)** challenge the reactive premise: With virtual threads, you can write blocking code that doesn't waste OS threads — getting many benefits of reactive without the complexity.

**What interviewers are really testing:** Understanding of the non-blocking model, back-pressure, and honest judgment about when reactive is (and isn't) worth the complexity.

**Red flag answer:** "Reactive is faster than MVC" — it's not about speed per request, it's about concurrency and resource efficiency. Or not mentioning the debugging/complexity trade-off.

**Follow-up:**
1. "What happens if you make a blocking JDBC call inside a WebFlux handler? How do you avoid this?"
2. "How does back-pressure work end-to-end from a Reactor Flux to an HTTP client?"
3. "With Virtual Threads in Java 21, is there still a case for reactive programming? What does reactive do that virtual threads don't?"

<Accordion title="50. Logging Best Practices (SLF4J + Logback)">
**Answer**:
Logging seems simple but is one of the most impactful cross-cutting concerns in production. Bad logging is noise; good logging saves you at 3 AM.

**Architecture**:
*   **SLF4J** (Simple Logging Facade for Java): An API/interface. Your code depends on SLF4J. This decouples your code from the logging implementation — you can swap Logback for Log4j2 without changing a single line of application code.
*   **Logback**: The default implementation in Spring Boot. Created by the same author as SLF4J and Log4j.
*   **Log4j2**: Alternative implementation. Async logging is significantly faster (LMAX Disruptor-based). Better for ultra-high-throughput applications.

**Best practices**:

1. **Use parameterized logging**: `log.info("Processing order {} for user {}", orderId, userId)` — NOT `log.info("Processing order " + orderId)`. The parameterized version avoids string concatenation when the log level is disabled.

2. **Log the exception properly**: `log.error("Payment failed for order {}", orderId, exception)` — the exception as the LAST argument gives you the full stack trace. Common mistake: `log.error("Error: " + e.getMessage())` — loses the stack trace entirely.

3. **Structured logging**: Use JSON format in production for machine parsing. Logback's `LogstashEncoder` outputs JSON that Elasticsearch/Kibana can index. Include correlation ID, user ID, request path in every log line.

4. **MDC (Mapped Diagnostic Context)**: Thread-local context that's included in every log line. Set `MDC.put("requestId", uuid)` in a filter at request entry — every log line from that request includes the ID. Essential for tracing requests across microservices.

5. **Log levels discipline**:
    *   `ERROR`: Something failed that shouldn't have. Requires investigation. Triggers alerts.
    *   `WARN`: Something unexpected but handled. May need attention eventually.
    *   `INFO`: Normal, significant business events. "Order 123 placed," "Deployment started."
    *   `DEBUG`: Detailed flow for debugging. Disabled in production by default.
    *   `TRACE`: Very detailed. Almost never used in production.

6. **Async appender**: `AsyncAppender` wraps the real appender and queues log events. Prevents logging from blocking request threads. Set `queueSize`, `discardingThreshold`, `neverBlock=true` for production.

7. **Never log sensitive data**: PII (emails, SSNs), passwords, tokens, credit card numbers. Mask or redact.

**What interviewers are really testing:** Production logging hygiene. MDC usage. Structured logging awareness. Understanding the performance impact of synchronous logging.

**Red flag answer:** Using `System.out.println()` or `e.printStackTrace()` in production code. Or not knowing about MDC.

**Follow-up:**
1. "How do you trace a single request across 5 microservices? What's the role of MDC and distributed tracing?"
2. "Your application's throughput drops 30% when you enable DEBUG logging. Why, and how do you fix it?"
3. "How do you change log levels at runtime in production without restarting the application?"

<Accordion title="52. Stream API Internals and Performance">
**Answer**:
The Stream API (Java 8) is ubiquitous in modern Java, but most developers use it without understanding how it works internally — leading to subtle performance issues.

**How streams work internally**:
*   A stream pipeline consists of a **source** (collection, array, generator), zero or more **intermediate operations** (filter, map, flatMap, sorted, distinct), and a **terminal operation** (collect, forEach, reduce, count).
*   Intermediate operations are **lazy** — they just build a pipeline description. Nothing executes until a terminal operation is invoked.
*   Execution is **fused** — elements flow through the entire pipeline one at a time. `filter().map().collect()` does NOT create intermediate collections. Each element passes through filter, then map, then is collected. This is called "loop fusion."
*   **Short-circuit operations** (`findFirst()`, `limit()`, `anyMatch()`) stop processing early. `stream.filter(expensive).findFirst()` stops after finding the first match.

**Performance considerations**:

1. **Stream overhead**: Streams have object creation overhead (pipeline objects, lambdas, spliterators). For small collections (&lt;100 elements), a plain `for` loop is faster. For large collections or complex pipelines, streams are comparable or negligible overhead.

2. **Parallel streams — handle with care**:
    *   Uses `ForkJoinPool.commonPool()` (shared, limited threads).
    *   Only beneficial for CPU-intensive operations on large datasets. `parallelStream().map(x -> x * 2)` on 100 elements is SLOWER than sequential due to thread pool overhead.
    *   `ArrayList` and arrays split well (random access). `LinkedList` and `HashSet` split poorly.
    *   Stateful operations like `sorted()` and `distinct()` add synchronization overhead in parallel.
    *   Rule of thumb: N * Q > 10,000 (N = elements, Q = cost per element in nanoseconds) for parallel to help.

3. **Collecting into unmodifiable collections** (Java 10+): `stream.collect(Collectors.toUnmodifiableList())` or `stream.toList()` (Java 16+) — returns an unmodifiable list. More efficient than `Collectors.toList()` which creates a mutable `ArrayList`.

4. **`flatMap` gotcha**: `flatMap` creates a new stream per element. For millions of elements, the overhead is significant. Alternatives: use `mapMulti()` (Java 16+) which avoids intermediate stream creation.

**Common anti-patterns**:
*   `stream.peek(item -> item.setProcessed(true))` — using `peek()` for mutation. It's meant for debugging only and may not execute for all elements in optimized pipelines.
*   `list.stream().forEach(...)` instead of `list.forEach(...)` — unnecessary stream overhead.
*   Collecting into a list just to iterate: `stream.collect(toList()).forEach(...)` — just use `stream.forEach(...)`.

**What interviewers are really testing:** Understanding of lazy evaluation and loop fusion. Knowing when parallel streams help vs hurt. Awareness of stream vs for-loop performance trade-offs.

**Red flag answer:** "I always use parallel streams for performance" or not understanding lazy evaluation.

**Follow-up:**
1. "Explain the difference between `stream.map().filter().collect()` and writing the equivalent for-loop in terms of execution model."
2. "You have a `List<Order>` with 10 million entries. When is `parallelStream()` faster and when is it slower?"
3. "What's the difference between `Collectors.toList()` and `Stream.toList()` in Java 16+?"
</Accordion>

<Accordion title="53. Virtual Threads (Project Loom) — Java 21">
**Answer**:
Virtual Threads are the biggest change to Java's concurrency model since the language was created. They eliminate the trade-off between simple blocking code and scalable I/O.

**The problem they solve**:
Traditional Java threads (platform threads) are 1:1 mapped to OS threads. Each OS thread costs ~1MB of stack memory and is expensive to create/schedule. This limits practical concurrency to ~thousands of threads. When each thread blocks on I/O (HTTP call, DB query), the OS thread is wasted. This forces the choice between:
*   **Thread-per-request** (simple code, limited concurrency) — Tomcat's model.
*   **Reactive/async** (high concurrency, complex code) — WebFlux/CompletableFuture.

**Virtual Threads solution**:
*   Managed by the JVM, not the OS. Extremely lightweight (~1KB initial stack, grows as needed).
*   You can create **millions** of virtual threads. `Thread.startVirtualThread(() -> ...)` or `Executors.newVirtualThreadPerTaskExecutor()`.
*   When a virtual thread blocks on I/O, the JVM **unmounts** it from the carrier (platform) thread, freeing the carrier to run another virtual thread. When the I/O completes, the virtual thread is remounted on any available carrier.
*   You write simple, synchronous, blocking code — the JVM makes it non-blocking under the hood.

**Key behaviors and gotchas**:

1. **Carrier threads**: Virtual threads run on a small pool of platform (carrier) threads — typically `Runtime.getRuntime().availableProcessors()`. The mapping is M:N (many virtual threads to few carriers).

2. **Pinning**: A virtual thread gets "pinned" to its carrier (can't unmount) when:
    *   Inside a `synchronized` block — because monitor locks are tied to OS threads. Fix: use `ReentrantLock` instead.
    *   During native (JNI) calls.
    *   Pinning wastes carriers. `-Djdk.tracePinnedThreads=full` logs pinning events.

3. **Don't pool virtual threads**: Pooling defeats the purpose. Create a new virtual thread per task. `Executors.newVirtualThreadPerTaskExecutor()` is the right pattern.

4. **ThreadLocal caution**: Each virtual thread gets its own ThreadLocal map. With millions of virtual threads, this can consume significant memory. Java 21 introduces **Scoped Values** (`ScopedValue`) as a more efficient alternative.

5. **Not for CPU-bound work**: Virtual threads shine for I/O-bound workloads (web servers, API calls). For CPU-intensive work, you're still limited by the number of CPU cores — virtual threads add overhead without benefit.

**Spring Boot 3.2+ integration**: Set `spring.threads.virtual.enabled=true` and Spring Boot auto-configures Tomcat/Jetty to use virtual threads for request handling. Each request gets a new virtual thread. Combined with JDBC (blocking I/O), you get the concurrency of reactive with the simplicity of blocking code.

**What interviewers are really testing:** Understanding of the M:N threading model, pinning, and when virtual threads are (and aren't) appropriate. This is a must-know topic for Java interviews in 2025+.

**Red flag answer:** "Virtual threads are faster threads" — they're not faster per-task, they're lighter and better for I/O-bound concurrency. Or not knowing about pinning.

**Follow-up:**
1. "Why does `synchronized` cause pinning with virtual threads but `ReentrantLock` doesn't? What's the technical difference?"
2. "You enable virtual threads in Spring Boot and your database connection pool exhaustion alerts fire immediately. Why?"
3. "Does Spring WebFlux become obsolete with virtual threads? What does reactive still offer?"
</Accordion>

<Accordion title="54. Immutability and Records">
**Answer**:
Immutability is a core principle for writing safe, predictable Java code — especially in concurrent systems. Records (Java 16) make it dramatically easier.

**Why immutability matters**:
1. **Thread safety for free**: Immutable objects can be shared across threads without synchronization. No race conditions, no visibility issues. `String`, `Integer`, `LocalDate` are all immutable — and all thread-safe.
2. **Safe as Map keys/Set elements**: Once created, the hash code never changes. Mutable keys can "disappear" from a HashMap if mutated after insertion.
3. **Easier reasoning**: No defensive copying needed when passing objects around. The object you received can't change under you.

**Creating immutable classes (pre-Records)**:
1. Declare the class `final` (prevent subclassing that could add mutability).
2. All fields are `private final`.
3. No setters.
4. Constructor initializes all fields.
5. Deep copy any mutable objects in constructor and getters. `this.dates = new ArrayList<>(dates)` — not `this.dates = dates` which shares the reference.
6. Override `equals()`, `hashCode()`, `toString()`.

That's 50+ lines of boilerplate for a simple class.

**Records (Java 16+)** solve this:
```java
record Point(int x, int y) {}
```
This single line generates:
*   `private final` fields for `x` and `y`
*   A canonical constructor
*   Accessor methods `x()` and `y()` (not `getX()`)
*   Correct `equals()`, `hashCode()`, `toString()`
*   The class is implicitly `final`

**Records are NOT entities**: JPA/Hibernate requires a no-arg constructor, setters (or field access), and the ability to proxy the class. Records have none of these. Records are ideal for DTOs, value objects, query results, and API response objects — NOT for JPA entities.

**Sealed classes** complement Records:
```java
sealed interface Shape permits Circle, Rectangle, Triangle {}
record Circle(double radius) implements Shape {}
record Rectangle(double width, double height) implements Shape {}
record Triangle(double base, double height) implements Shape {}
```
The compiler knows all possible `Shape` subtypes — enabling exhaustive `switch` expressions with pattern matching.

**What interviewers are really testing:** Understanding of why immutability matters (thread safety, hashing), how Records work, and the limitations (no JPA entities).

**Red flag answer:** "Records are just less boilerplate for POJOs" without mentioning immutability guarantees or the JPA limitation.

**Follow-up:**
1. "Can you add custom validation to a Record's constructor? How?"
2. "Why can't Records be JPA entities? What specifically is incompatible?"
3. "Your Record contains a `List<String>`. Is it truly immutable? How do you fix it?"
</Accordion>

<Accordion title="55. Microservice Communication Patterns">
**Answer**:
In a microservices architecture, how services communicate is one of the most critical design decisions — it affects reliability, latency, coupling, and operational complexity.

**Synchronous communication** (request-response):
*   **REST/HTTP**: The default. Simple, well-understood, good tooling. Trade-off: temporal coupling — if the downstream service is down, the caller fails. Use circuit breakers (Resilience4j) to prevent cascading failures.
*   **gRPC**: Binary protocol (Protobuf), HTTP/2 multiplexing, bidirectional streaming. 2-10x faster than REST for internal service-to-service calls. Strong schema enforcement via `.proto` files. Trade-off: harder to debug (binary), requires Protobuf tooling, less browser support.

**Asynchronous communication** (event-driven):
*   **Message queues** (RabbitMQ, Amazon SQS): Point-to-point. One producer, one consumer (or competing consumers for load balancing). Good for task distribution, work queues.
*   **Event streaming** (Apache Kafka, Amazon Kinesis): Publish-subscribe. Events are persisted in an ordered log. Multiple consumer groups can read independently. Enables event sourcing, CQRS, real-time analytics. Trade-off: operational complexity, eventual consistency, harder debugging.

**Patterns**:

1. **API Gateway**: Single entry point for clients. Routes, rate-limits, authenticates. Spring Cloud Gateway, Kong, AWS API Gateway. Avoids clients needing to know individual service URLs.

2. **Circuit Breaker** (Resilience4j): If a downstream service fails repeatedly, the circuit "opens" and fails fast (returns a fallback) instead of wasting resources on doomed calls. States: Closed (normal) -> Open (failing fast) -> Half-Open (testing recovery).

3. **Saga pattern**: Distributed transaction across services without 2PC. Each service performs its local transaction and publishes an event. If a step fails, compensating transactions undo previous steps. Choreography (events) vs Orchestration (central coordinator).

4. **Service discovery**: Services register themselves (Eureka, Consul) and discover each other dynamically. In Kubernetes, DNS-based service discovery replaces dedicated tools.

5. **Outbox pattern**: To reliably publish events AND update the database atomically (without distributed transactions): write the event to an "outbox" table in the same database transaction, then a separate process (Debezium CDC) tails the outbox and publishes to Kafka. Prevents the "database updated but event not published" scenario.

**What interviewers are really testing:** Understanding of synchronous vs asynchronous trade-offs, experience with failure patterns (circuit breaker, saga), and awareness of consistency challenges.

**Red flag answer:** "We use REST for everything" without discussing async alternatives. Or not knowing about circuit breakers, which are table-stakes for microservices.

**Follow-up:**
1. "Service A calls Service B which calls Service C. Service C is slow. How does this cascade and how do you prevent it?"
2. "When would you choose Kafka over RabbitMQ? What's the fundamental architectural difference?"
3. "Explain the Outbox pattern. Why can't you just publish to Kafka and write to the database in the same method?"
</Accordion>

<Accordion title="56. Java Concurrency Utilities Deep Dive">
**Answer**:
Beyond `synchronized` and `Atomic*`, the `java.util.concurrent` package has powerful utilities that solve specific coordination problems. Senior engineers should know when to reach for each.

**Coordination utilities**:

*   **`CountDownLatch`**: One-time barrier. Initialize with count N. Threads call `await()` to block. Other threads call `countDown()`. When count reaches 0, all waiting threads proceed. Use case: "start processing only after all 5 services are initialized." Cannot be reset.

*   **`CyclicBarrier`**: Reusable barrier. N threads call `await()`. When all N arrive, the barrier trips and all proceed. Can be reused for multiple rounds. Use case: parallel algorithm phases — "all threads must finish phase 1 before any starts phase 2."

*   **`Semaphore`**: Controls access to N permits. `acquire()` takes a permit (blocks if none available), `release()` returns one. Use case: rate limiting (at most 10 concurrent DB connections), resource pooling.

*   **`Phaser`**: Advanced version of CountDownLatch + CyclicBarrier. Dynamic registration/deregistration of parties. Multiple phases. Used internally by parallel streams.

**Concurrent collections**:

*   **`ConcurrentHashMap`**: Lock-free reads, fine-grained locking for writes (per-bin in Java 8+). `computeIfAbsent()` is atomic. Replaces `Collections.synchronizedMap()` which locks the entire map.

*   **`CopyOnWriteArrayList`**: Every modification creates a new copy of the underlying array. Reads are lock-free and fast. Writes are expensive (O(n) copy). Use case: event listener lists — registered/unregistered rarely, iterated frequently. NOT suitable for high-write workloads.

*   **`BlockingQueue`** (`ArrayBlockingQueue`, `LinkedBlockingQueue`): Thread-safe queue with blocking `put()`/`take()`. Foundation of producer-consumer pattern. `ArrayBlockingQueue` is bounded (back-pressure built in). `LinkedBlockingQueue` is optionally bounded.

**`StampedLock`** (Java 8): Optimistic read locking. Three modes: write lock, pessimistic read lock, and optimistic read (no lock at all — just validates that no write happened). Significantly faster than `ReadWriteLock` for read-heavy workloads.

```java
long stamp = lock.tryOptimisticRead();
int x = this.x; int y = this.y;
if (!lock.validate(stamp)) { // was there a write?
    stamp = lock.readLock(); // fall back to pessimistic
    try { x = this.x; y = this.y; } finally { lock.unlockRead(stamp); }
}
```

**What interviewers are really testing:** Knowing which utility to use for which coordination problem. Understanding the performance characteristics of concurrent collections.

**Red flag answer:** Using `synchronized` or `wait()`/`notify()` for problems that have purpose-built solutions in `java.util.concurrent`.

3. "When is `CopyOnWriteArrayList` the right choice? What's the break-even point where it becomes worse than a synchronized list?"

<Accordion title="57. Spring Boot Error Handling and API Design">
**Answer**:
How you handle errors in a Spring Boot API reveals your maturity as a backend developer. Bad error handling leaks internals, confuses clients, and makes debugging in production nearly impossible.

**The error handling hierarchy**:

1. **`@ExceptionHandler`** (controller-level): Handles exceptions thrown by methods in that specific controller. Limited scope.

2. **`@ControllerAdvice` / `@RestControllerAdvice`** (global): Centralizes exception handling for all controllers. This is the standard approach:
```java
@RestControllerAdvice
public class GlobalExceptionHandler {

    @ExceptionHandler(ResourceNotFoundException.class)
    @ResponseStatus(HttpStatus.NOT_FOUND)
    public ErrorResponse handleNotFound(ResourceNotFoundException ex) {
        return new ErrorResponse("NOT_FOUND", ex.getMessage(), Instant.now());
    }

    @ExceptionHandler(MethodArgumentNotValidException.class)
    @ResponseStatus(HttpStatus.BAD_REQUEST)
    public ErrorResponse handleValidation(MethodArgumentNotValidException ex) {
        Map<String, String> errors = ex.getBindingResult().getFieldErrors()
            .stream().collect(toMap(FieldError::getField, FieldError::getDefaultMessage));
        return new ErrorResponse("VALIDATION_FAILED", "Invalid input", errors);
    }
}
```

3. **Problem Detail (RFC 7807)** — Spring Boot 3+ native support:
```json
{
    "type": "https://myapi.com/errors/insufficient-funds",
    "title": "Insufficient Funds",
    "status": 400,
    "detail": "Account 123 has $50.00 but withdrawal requires $100.00",
    "instance": "/accounts/123/withdraw"
}
```
Enable with `spring.mvc.problemdetail.enabled=true`. Standardized error format that clients can reliably parse.

**Best practices**:
*   **Never expose stack traces** to clients. Log them server-side, return a human-readable message and error code.
*   **Use appropriate HTTP status codes**: 400 (bad input), 401 (not authenticated), 403 (not authorized), 404 (not found), 409 (conflict), 422 (unprocessable), 429 (rate limited), 500 (server error).
*   **Include an error code**: `"code": "PAYMENT_DECLINED"` — more specific than HTTP status. Clients can programmatically handle different error types.
*   **Include a correlation ID**: `"traceId": "abc-123"` — allows clients to report issues with a reference that maps to your distributed tracing.
*   **Validation errors**: Return field-level details: `{"field": "email", "message": "must be a valid email"}`.

**What interviewers are really testing:** Can you design a consistent, client-friendly error format? Do you know about RFC 7807? Do you handle validation errors with field-level detail?

**Red flag answer:** Returning Spring's default `WhitelabelError` page or raw stack traces from the API. Or using 200 OK for everything with error details in the body.
