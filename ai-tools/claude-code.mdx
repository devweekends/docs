Claude Code is Anthropic's official CLI-based AI coding assistant. Unlike IDE-integrated tools (Cursor, Windsurf), Claude Code runs entirely in your terminal and operates directly on your filesystem. This makes it particularly strong for documentation workflows where you need to make changes across many files, enforce consistency, and run validation commands -- all without leaving your terminal.

**When to choose Claude Code over alternatives:**
- You prefer terminal workflows and already live in your shell.
- You need to make bulk edits across dozens of MDX files (Claude Code can read, edit, and validate files in a single session).
- You want full control over what the AI can and cannot do -- Claude Code's permission model lets you approve each file change.
- You need the AI to run shell commands (linting, build checks, git operations) as part of the workflow.

**When an IDE-based tool might be better:**
- You want inline autocomplete as you type (Claude Code does not provide this; Cursor and Windsurf do).
- You prefer a visual diff interface for reviewing changes.
- Your team is more comfortable with GUI-based tools.

## Prerequisites

- Active Claude subscription (Pro, Max, or API access)
- Node.js 18 or later installed

## Setup

<Steps>
<Step title="Install Claude Code">

  ```bash
  npm install -g @anthropic-ai/claude-code
  ```
  
  <Tip>
  If you encounter permission errors on macOS/Linux, use `sudo npm install -g @anthropic-ai/claude-code` or configure npm to use a local prefix (`npm config set prefix ~/.npm-global`).
  </Tip>
</Step>

<Step title="Navigate to your docs directory">
  Claude Code uses the current working directory as context, so always start from your docs root.
  
  ```bash
  cd /path/to/your/docs
  ```
</Step>

<Step title="Add a CLAUDE.md file (recommended)">
  This file trains Claude Code on your project's conventions. See the template below.
</Step>

<Step title="Start Claude Code">

  ```bash
  claude
  ```
  
  On first run, Claude Code will authenticate with your Anthropic account and index your project directory.
</Step>
</Steps>

## Real-World Usage Scenarios

Here are practical workflows where Claude Code excels for documentation projects:

**Bulk consistency fixes across all MDX files:**
```bash
# Ask Claude Code to find and fix inconsistent frontmatter
claude "Check all MDX files in this project for missing or incomplete frontmatter. 
Fix any files that are missing a title or description field."
```

**Generating new documentation pages from existing patterns:**
```bash
# Point Claude Code at an example and ask it to follow the pattern
claude "Look at the structure of case-studies/example-multi-vendor.mdx and create 
a similar case study template for a SaaS dashboard project."
```

**Running validation as part of the edit workflow:**
```bash
# Claude Code can run commands to verify its changes
claude "Update all internal links in the case-studies directory to use relative paths, 
then run 'npx mintlify dev' to verify nothing is broken."
```

<Tip>
Claude Code's ability to read files, make edits, and run shell commands in a single session makes it the strongest option for documentation maintenance tasks that span multiple files. For single-file writing tasks, an IDE-based tool with inline suggestions may feel faster.
</Tip>

## Create a CLAUDE.md File

Create a `CLAUDE.md` file at the root of your documentation repository to train Claude Code on your specific documentation standards. This file is loaded automatically every time Claude Code starts in your project directory, so it acts as persistent context that shapes every interaction:

````markdown
# Mintlify documentation

## Working relationship
- You can push back on ideas-this can lead to better documentation. Cite sources and explain your reasoning when you do so
- ALWAYS ask for clarification rather than making assumptions
- NEVER lie, guess, or make up information

## Project context
- Format: MDX files with YAML frontmatter
- Config: docs.json for navigation, theme, settings
- Components: Mintlify components

## Content strategy
- Document just enough for user success - not too much, not too little
- Prioritize accuracy and usability of information
- Make content evergreen when possible
- Search for existing information before adding new content. Avoid duplication unless it is done for a strategic reason
- Check existing patterns for consistency
- Start by making the smallest reasonable changes

## Frontmatter requirements for pages
- title: Clear, descriptive page title
- description: Concise summary for SEO/navigation

## Writing standards
- Second-person voice ("you")
- Prerequisites at start of procedural content
- Test all code examples before publishing
- Match style and formatting of existing pages
- Include both basic and advanced use cases
- Language tags on all code blocks
- Alt text on all images
- Relative paths for internal links

## Git workflow
- NEVER use --no-verify when committing
- Ask how to handle uncommitted changes before starting
- Create a new branch when no clear branch exists for changes
- Commit frequently throughout development
- NEVER skip or disable pre-commit hooks

## Do not
- Skip frontmatter on any MDX file
- Use absolute URLs for internal links
- Include untested code examples
- Make assumptions - always ask for clarification
````
