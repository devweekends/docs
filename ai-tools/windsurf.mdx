Configure Windsurf's Cascade AI assistant to help you write and maintain documentation. Windsurf is an AI-native code editor whose standout feature is **Cascade** -- a context-aware assistant that maintains a chain of reasoning across multiple files and actions. This guide shows how to set up Windsurf specifically for your Mintlify documentation workflow.

**When to choose Windsurf over alternatives:**
- You want an AI that **maintains context across a chain of related edits**. Cascade remembers what it did three steps ago and uses that context for the next step, which is valuable when restructuring documentation across multiple files.
- You need **codebase-aware suggestions** that understand how your MDX files reference each other (navigation, cross-links, shared components).
- You prefer a GUI-based editor but want deeper AI integration than what a VS Code extension provides.

**When another tool might be better:**
- For **inline autocomplete while typing**, Cursor's tab-completion is more refined and faster.
- For **bulk edits across 20+ files with shell command validation**, Claude Code's terminal workflow is more efficient because it reads, edits, and runs validation commands in a single session.
- If your team is heavily invested in the **VS Code extension ecosystem**, Cursor (a VS Code fork) provides better extension compatibility.

<Tip>
Windsurf's free tier is generous enough for evaluating whether it fits your workflow. If you are deciding between Windsurf and Cursor, try both for a week on actual documentation tasks. The difference often comes down to whether you value Cascade's multi-step context (Windsurf) or inline autocomplete speed (Cursor).
</Tip>

## Prerequisites

- Windsurf editor installed (download from [windsurf.com](https://windsurf.com))
- Access to your documentation repository

## Workspace Rules

Workspace rules are Windsurf's mechanism for persistent project context. They are loaded automatically when Windsurf opens your project, so they shape every AI interaction without you repeating instructions.

Create `.windsurf/rules.md` in your project root:

````markdown
# Mintlify technical writing rule

## Project context

- This is a documentation project on the Mintlify platform
- We use MDX files with YAML frontmatter  
- Navigation is configured in `docs.json`
- We follow technical writing best practices

## Writing standards

- Use second person ("you") for instructions
- Write in active voice and present tense
- Start procedures with prerequisites
- Include expected outcomes for major steps
- Use descriptive, keyword-rich headings
- Keep sentences concise but informative

## Required page structure

Every page must start with frontmatter:

```yaml
---
title: "Clear, specific title"
description: "Concise description for SEO and navigation"
---
```

## Mintlify components

### Callouts

- `<Note>` for helpful supplementary information
- `<Warning>` for important cautions and breaking changes
- `<Tip>` for best practices and expert advice  
- `<Info>` for neutral contextual information
- `<Check>` for success confirmations

### Code examples

- When appropriate, include complete, runnable examples
- Use `<CodeGroup>` for multiple language examples
- Specify language tags on all code blocks
- Include realistic data, not placeholders
- Use `<RequestExample>` and `<ResponseExample>` for API docs

### Procedures

- Use `<Steps>` component for sequential instructions
- Include verification steps with `<Check>` components when relevant
- Break complex procedures into smaller steps

### Content organization

- Use `<Tabs>` for platform-specific content
- Use `<Accordion>` for progressive disclosure
- Use `<Card>` and `<CardGroup>` for highlighting content
- Wrap images in `<Frame>` components with descriptive alt text

## API documentation requirements

- Document all parameters with `<ParamField>` 
- Show response structure with `<ResponseField>`
- Include both success and error examples
- Use `<Expandable>` for nested object properties
- Always include authentication examples

## Quality standards

- Test all code examples before publishing
- Use relative paths for internal links
- Include alt text for all images
- Ensure proper heading hierarchy (start with h2)
- Check existing patterns for consistency
````

## Real-World Usage Scenarios

Here are practical workflows where Windsurf excels for documentation projects:

**Multi-step documentation restructuring with Cascade:**
Open Cascade and describe a restructuring task: "I need to split the case-studies/guide.mdx page into two pages -- one for the rubric and one for the writing guide. Update the navigation in docs.json and fix all internal links that reference the original page." Cascade will plan the steps, execute them in sequence, and maintain context across each file change.

**Generating documentation from code:**
Point Cascade at a source file: "Read the API route handlers in `src/routes/orders.ts` and generate an API documentation page in MDX format following the patterns in our existing docs." Cascade's codebase awareness helps it match your existing documentation style.

**Consistency audits across your documentation:**
Ask Cascade: "Review all MDX files in the case-studies directory. For each file, check that the frontmatter has both a title and description, that all code blocks have language tags, and that Mintlify callout components are used correctly. List any issues." Cascade will traverse the files and report findings.

<Tip>
Cascade is at its strongest when the task involves **reasoning across multiple files in sequence**. If you need to rename a concept throughout your documentation, update cross-references, and verify the navigation config -- all in one flow -- Cascade handles this better than piecemeal autocomplete.
</Tip>

## Tool Selection Trade-offs -- Windsurf vs. Cursor vs. Claude Code

| Capability | Windsurf | Cursor | Claude Code |
|------------|----------|--------|-------------|
| **Multi-step context chain** | Strongest -- Cascade maintains reasoning across steps | Composer mode -- good but less persistent | Session-based -- maintains context within a conversation |
| **Inline autocomplete** | Available but less refined | Strongest -- real-time tab completion | Not available (CLI tool) |
| **Codebase awareness** | Strong -- indexes project structure | Good with `@` file references | Reads files on demand per session |
| **VS Code extensions** | Partial compatibility | Full compatibility (VS Code fork) | Not applicable |
| **Terminal integration** | Built-in terminal panel | Built-in terminal panel | Native -- it IS the terminal |
| **Team consistency** | `.windsurf/rules.md` checked into repo | `.cursor/rules.md` checked into repo | `CLAUDE.md` checked into repo |
| **Cost** | Free tier available; Pro for advanced features | Free tier limited; Pro for heavy use | Uses Claude subscription (Pro/Max/API) |
| **Best for** | Multi-file restructuring, codebase-aware generation | Writing new content with autocomplete | Bulk edits, validation, cross-file consistency |

<Note>
No single tool is best for every documentation task. Many teams use **Cursor for writing** (inline autocomplete makes drafting fast) and **Claude Code for maintenance** (bulk edits, link validation, frontmatter audits). Windsurf fits in when the task requires understanding how files relate to each other across a multi-step workflow.
</Note>
