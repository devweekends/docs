## What is Heap?

**Heap** is a complete binary tree that maintains the heap property: parent is always smaller (min-heap) or larger (max-heap) than children. It provides O(log n) insertion and O(1) access to min/max.

Think of a corporate hierarchy where the CEO (root) always has the highest salary. Every manager earns more than their direct reports. To find the highest-paid person, just look at the top -- O(1). To add someone or remove the CEO, you shuffle people up or down the hierarchy to maintain the property -- O(log n) because the tree height is log n.

Heaps are typically stored as **arrays**, not tree nodes. For a node at index `i`, its left child is at `2i + 1`, right child at `2i + 2`, and parent at `(i - 1) / 2`. This compact representation has zero pointer overhead and excellent cache locality.

**Why heaps instead of sorting?** When you need to repeatedly find or remove the min/max from a changing dataset, sorting gives you the answer once but costs O(n log n) to re-sort after each change. A heap maintains the invariant incrementally: O(log n) per insertion or removal. For problems like "k-th largest in a stream" where elements arrive one at a time, heaps are the natural choice.

**Key complexity**: Insert O(log n), Extract-min/max O(log n), Peek O(1), Build-heap from array O(n).

<Note>
**Quick Recognition**: If you need **k largest/smallest**, **merge sorted lists**, or **continuously find min/max**, think Heap!
</Note>

## Pattern Recognition Cheatsheet (LeetCode Triggers)

<Tip>
**If the problem says any of these, reach for a heap before you reach for sort:**

- **"kth largest / kth smallest"** -- min-heap of size k (for k largest), max-heap of size k (for k smallest). LC 215, 703, 973.
- **"top K"** -- frequency map plus min-heap of size k keyed by frequency. LC 347, 692, 1985.
- **"merge K sorted"** -- min-heap holding one entry per list, push successor when you pop. LC 23, 378, 632.
- **"median of stream" / "running median"** -- two-heap split (max-heap on the smaller half, min-heap on the larger half). LC 295, 480.
- **"schedule by priority" / "minimum number of CPUs/rooms/halls"** -- min-heap of end times to track active resources. LC 253, 621, 767, 1834.
- **"closest / farthest from origin"** -- max-heap of size k on distance squared. LC 973, 658.
- **"dijkstra" / "shortest path with weights"** -- min-heap keyed by distance. LC 743, 787, 1631.
- **"maximize / minimize cost while picking up to K items"** -- two-heap greedy (LC 502 IPO is the canonical example).

**Anti-patterns -- do NOT use a heap when:**
- You need elements in fully sorted order at the end -- just sort, O(n log n).
- You need to delete or update arbitrary elements by key -- use a balanced BST (Java TreeMap, C++ std::set, Python sortedcontainers.SortedList) or an indexed priority queue.
- The data is small (under 100 items) -- the overhead is not worth it.
- You need O(1) lookup by value -- use a hash map.
</Tip>

## Heap Templates You Should Have Memorized

Before LeetCode contest day, these four templates should be muscle memory. Type them from blank in under two minutes each.

### Template 1: Min-Heap Top-K (Kth Largest)

The single most reusable heap template on LeetCode. Memorize the shape and you can solve LC 215, 347, 692, 973, 1985 with the same skeleton.

<CodeGroup>
```python Python
import heapq

def kth_largest(nums, k):
    heap = []                          # min-heap (heapq is min-heap by default)
    for n in nums:
        heapq.heappush(heap, n)
        if len(heap) > k:              # gatekeeper: keep only k largest
            heapq.heappop(heap)        # pops the smallest of the k -- evicted
    return heap[0]                     # smallest of k largest = kth largest
```

```java Java
// PriorityQueue is a min-heap by default in Java.
public int kthLargest(int[] nums, int k) {
    PriorityQueue<Integer> heap = new PriorityQueue<>();
    for (int n : nums) {
        heap.offer(n);
        if (heap.size() > k) heap.poll();
    }
    return heap.peek();
}
```

```javascript JavaScript
// JS has no native heap. Use a MinHeap class or the npm "heap-js" library.
// Quick contest-style fallback: keep a sorted array of size k.

    const heap = new MinHeap();         // assume MinHeap with push/pop/peek/size
    for (const n of nums) {
        heap.push(n);
        if (heap.size() > k) heap.pop();
    }
    return heap.peek();
}
```

```cpp C++

    }
    return heap.top();
}
```

### Template 2: Custom Comparator via Tuple (with Tiebreaker)

`heapq` compares tuples lexicographically. If two priorities tie and the next element is non-comparable (like a `ListNode` or a custom object), Python crashes. The fix is a monotonically increasing **tiebreaker counter** as the second tuple element.

<CodeGroup>
```python Python
import heapq, itertools

class Task:
    def __init__(self, name): self.name = name

heap = []
counter = itertools.count()             # 0, 1, 2, ... unique forever

def push(priority, task):
    heapq.heappush(heap, (priority, next(counter), task))
    # If two tasks share priority, the counter breaks the tie deterministically
    # without ever comparing Task objects (which would crash).

def pop():
    priority, _, task = heapq.heappop(heap)
    return task
```

```java Java
// Use a Comparator. No tiebreaker needed because Java's PriorityQueue
// does not require Comparable on the underlying object if you supply a Comparator.
PriorityQueue<int[]> heap = new PriorityQueue<>((a, b) -> {
    if (a[0] != b[0]) return Integer.compare(a[0], b[0]);
    return Integer.compare(a[1], b[1]);   // explicit tiebreaker on insertion id
});
```

```cpp C++
// Lambda comparator; remember the inverted-sign convention for min-heaps.
auto cmp = [](const tuple<int,int,Task*>& a, const tuple<int,int,Task*>& b){
    return get<0>(a) > get<0>(b);          // > makes it a min-heap on priority
};
priority_queue<tuple<int,int,Task*>, vector<tuple<int,int,Task*>>, decltype(cmp)> heap(cmp);
```
</CodeGroup>

### Template 3: Two-Heap Median (Stream)

The canonical pattern for "running median," "running kth percentile," and any problem asking for an order statistic over a stream.

<CodeGroup>
```python Python
import heapq

class MedianFinder:
    def __init__(self):
        self.lo = []   # max-heap of the smaller half (negate values)
        self.hi = []   # min-heap of the larger half

    def addNum(self, num):
        heapq.heappush(self.lo, -num)                       # always push to lo first
        heapq.heappush(self.hi, -heapq.heappop(self.lo))    # move max of lo to hi
        if len(self.hi) > len(self.lo):                      # rebalance
            heapq.heappush(self.lo, -heapq.heappop(self.hi))

    def findMedian(self):
        if len(self.lo) > len(self.hi):
            return -self.lo[0]
        return (-self.lo[0] + self.hi[0]) / 2.0
```

```java Java
class MedianFinder {
    PriorityQueue<Integer> lo = new PriorityQueue<>(Collections.reverseOrder());
    PriorityQueue<Integer> hi = new PriorityQueue<>();
    public void addNum(int n) {
        lo.offer(n); hi.offer(lo.poll());
        if (hi.size() > lo.size()) lo.offer(hi.poll());
    }
    public double findMedian() {
        return lo.size() > hi.size() ? lo.peek() : (lo.peek() + hi.peek()) / 2.0;
    }
}
```

```cpp C++
class MedianFinder {
    priority_queue<int> lo;                                    // max-heap
    priority_queue<int, vector<int>, greater<int>> hi;         // min-heap
public:
    void addNum(int n) {
        lo.push(n); hi.push(lo.top()); lo.pop();
        if (hi.size() > lo.size()) { lo.push(hi.top()); hi.pop(); }
    }
    double findMedian() {
        return lo.size() > hi.size() ? lo.top() : (lo.top() + hi.top()) / 2.0;
    }
};
```

### Template 4: Lazy Deletion

Heaps do not support arbitrary delete in O(log n). The standard trick is **lazy deletion**: keep a hash map of "removed" entries and skip them when they surface at the top. Used in LC 480 (Sliding Window Median), LC 1438, and many sliding-window-with-priority problems.

<CodeGroup>
```python Python
import heapq
from collections import defaultdict

class LazyHeap:
    def __init__(self):
        self.heap = []
        self.removed = defaultdict(int)   # value -> count of pending removals

    def push(self, x):
        heapq.heappush(self.heap, x)

    def remove(self, x):
        self.removed[x] += 1               # mark for lazy removal; no heap op

    def _purge(self):
        # Drop any top entries that are tombstoned
        while self.heap and self.removed[self.heap[0]] > 0:
            self.removed[self.heap[0]] -= 1
            heapq.heappop(self.heap)

    def peek(self):
        self._purge()
        return self.heap[0] if self.heap else None

    def pop(self):
        self._purge()
        return heapq.heappop(self.heap)
```
</CodeGroup>

When tombstones grow to dominate the heap (say, more than half of entries are removed), call `heapq.heapify(self.heap)` after a full filter pass to rebuild from scratch in O(n).

## Pattern Variations

### 1. Kth Largest Element

The counter-intuitive trick that confuses beginners: to find the k **largest**, use a **min**-heap of size k. After processing all elements, the heap contains exactly the k largest, and the smallest of those (the heap's root) is the k-th largest overall.

**Why a min-heap and not a max-heap?** You want to keep the k largest elements and discard everything smaller. A min-heap lets you efficiently find and remove the smallest element in your collection. When a new number arrives, if it is larger than the current k-th largest (heap root), pop the root and push the new number. If it is smaller, ignore it.

**Time: O(n log k).** **Space: O(k).** This is better than sorting O(n log n) when k is small relative to n.

**Alternative**: Quickselect (average O(n), worst O(n^2)) is faster but less commonly asked. Mention it as a follow-up.

<CodeGroup>
```python Python
import heapq

def find_kth_largest(nums, k):
    """Find kth largest using min-heap of size k"""
    min_heap = []
    
    for num in nums:
        heapq.heappush(min_heap, num)  # Add element
        if len(min_heap) > k:          # If heap exceeds size k...
            heapq.heappop(min_heap)    # ...remove the smallest (not in top k)
    
    return min_heap[0]  # Root of min-heap = smallest of the k largest = kth largest

# Alternative: Python's built-in (less educational but concise)
def find_kth_largest_alt(nums, k):
    return heapq.nlargest(k, nums)[-1]
```

```java Java
public int findKthLargest(int[] nums, int k) {
    // Find kth largest using min-heap of size k
    PriorityQueue<Integer> minHeap = new PriorityQueue<>();
    
    for (int num : nums) {
        minHeap.offer(num);
        if (minHeap.size() > k) {
            minHeap.poll();
        }
    }
    
    return minHeap.peek();
}
```

```cpp C++
int findKthLargest(vector<int>& nums, int k) {
    // Find kth largest using min-heap of size k
    priority_queue<int, vector<int>, greater<int>> minHeap;
    
    for (int num : nums) {
        minHeap.push(num);
        if (minHeap.size() > k) {
            minHeap.pop();
        }
    }
    
    return minHeap.top();
}
```

### 2. Top K Frequent Elements

A two-step problem: first count frequencies, then find the top k. The heap approach uses a min-heap of size k keyed by frequency -- the same gatekeeper pattern as Kth Largest, but now we are comparing frequencies instead of values.

**Alternative approaches**: (1) Sort by frequency: O(n log n). (2) Bucket sort by frequency: O(n), since max frequency is at most n. The bucket sort approach is optimal but less commonly tested. (3) Quickselect on frequencies: O(n) average.

**Time: O(n log k).** **Space: O(n)** for the frequency map + O(k) for the heap.

<CodeGroup>
```python Python
import heapq
from collections import Counter

def top_k_frequent(nums, k):
    """Find k most frequent elements"""
    count = Counter(nums)
    
    # Min-heap of (frequency, element)
    min_heap = []
    for num, freq in count.items():
        heapq.heappush(min_heap, (freq, num))
        if len(min_heap) > k:
            heapq.heappop(min_heap)
    
    return [num for freq, num in min_heap]
```

```java Java
public int[] topKFrequent(int[] nums, int k) {
    // Find k most frequent elements
    Map<Integer, Integer> count = new HashMap<>();
    for (int num : nums) {
        count.put(num, count.getOrDefault(num, 0) + 1);
    }
    
    // Min-heap of (frequency, element)
    PriorityQueue<int[]> minHeap = new PriorityQueue<>((a, b) -> a[0] - b[0]);
    
    for (Map.Entry<Integer, Integer> entry : count.entrySet()) {
        minHeap.offer(new int[]{entry.getValue(), entry.getKey()});
        if (minHeap.size() > k) {
            minHeap.poll();
        }
    }
    
    int[] result = new int[k];
    for (int i = 0; i < k; i++) {
        result[i] = minHeap.poll()[1];
    }
    return result;
}
```

```cpp C++
vector<int> topKFrequent(vector<int>& nums, int k) {
    // Find k most frequent elements
    unordered_map<int, int> count;
    for (int num : nums) {
        count[num]++;
    }
    
    // Min-heap of (frequency, element)
    priority_queue<pair<int, int>, vector<pair<int, int>>, 
                   greater<pair<int, int>>> minHeap;
    
    for (auto& [num, freq] : count) {
        minHeap.push({freq, num});
        if (minHeap.size() > k) {
            minHeap.pop();
        }
    }
    
    vector<int> result;
    while (!minHeap.empty()) {
        result.push_back(minHeap.top().second);
        minHeap.pop();
    }
    return result;
}
```

### 3. Merge K Sorted Lists

A beautiful application of heaps. The brute-force approach merges lists pairwise, but a min-heap lets us always pick the globally smallest element across all k lists in O(log k) time.

**How it works**: Put the head of each list into a min-heap. Pop the smallest, add it to the result, and push its successor (if any) from the same list. Repeat until the heap is empty.

**Why the `i` index in the heap tuple?** Python's heapq compares tuples element by element. If two nodes have the same value, it tries to compare the next element. ListNode objects are not comparable, so including a unique index `i` as a tiebreaker prevents TypeError.

**Time: O(N log k)** where N is the total number of nodes across all lists and k is the number of lists. Each of the N nodes is pushed and popped from a size-k heap. **Space: O(k)** for the heap.

<CodeGroup>
```python Python
import heapq

def merge_k_lists(lists):
    """Merge k sorted linked lists"""
    min_heap = []
    
    # Seed heap with the head of each list
    # (value, list_index_for_tiebreaking, node_reference)
    for i, lst in enumerate(lists):
        if lst:
            heapq.heappush(min_heap, (lst.val, i, lst))
    
    dummy = ListNode(0)
    current = dummy
    
    while min_heap:
        val, i, node = heapq.heappop(min_heap)   # Pop globally smallest
        current.next = node
        current = current.next
        
        if node.next:                              # Push successor from same list
            heapq.heappush(min_heap, (node.next.val, i, node.next))
    
    return dummy.next
```

```java Java
public ListNode mergeKLists(ListNode[] lists) {
    // Merge k sorted linked lists
    PriorityQueue<ListNode> minHeap = new PriorityQueue<>(
        (a, b) -> a.val - b.val
    );
    
    // Add first element from each list
    for (ListNode list : lists) {
        if (list != null) {
            minHeap.offer(list);
        }
    }
    
    ListNode dummy = new ListNode(0);
    ListNode current = dummy;
    
    while (!minHeap.isEmpty()) {
        ListNode node = minHeap.poll();
        current.next = node;
        current = current.next;
        
        if (node.next != null) {
            minHeap.offer(node.next);
        }
    }
    
    return dummy.next;
}
```

```cpp C++
ListNode* mergeKLists(vector<ListNode*>& lists) {
    // Merge k sorted linked lists
    auto cmp = [](ListNode* a, ListNode* b) { return a->val > b->val; };
    priority_queue<ListNode*, vector<ListNode*>, decltype(cmp)> minHeap(cmp);
    
    // Add first element from each list
    for (ListNode* list : lists) {
        if (list != nullptr) {
            minHeap.push(list);
        }
    }
    
    ListNode dummy(0);
    ListNode* current = &dummy;
    
    while (!minHeap.empty()) {
        ListNode* node = minHeap.top();
        minHeap.pop();
        current->next = node;
        current = current->next;
        
        if (node->next != nullptr) {
            minHeap.push(node->next);
        }
    }
    
    return dummy.next;
}
```

### 4. Find Median from Data Stream

One of the most elegant heap applications. The idea: split all numbers into two halves. A **max-heap** holds the smaller half (so the largest of the small numbers is at the top), and a **min-heap** holds the larger half (so the smallest of the large numbers is at the top). The median is either the top of the max-heap (if odd count) or the average of both tops.

**Invariants to maintain**:
1. **Ordering**: Every element in `small` (max-heap) must be at most every element in `large` (min-heap).
2. **Balance**: The heaps differ in size by at most 1 (`small` can have one extra).

**Python max-heap trick**: Python's `heapq` is min-heap only. Simulate a max-heap by negating values on push and negating again on pop.

**Time: O(log n) per addNum, O(1) per findMedian.** **Space: O(n).**

<CodeGroup>
```python Python
import heapq

class MedianFinder:
    """Two heaps: max-heap for smaller half, min-heap for larger half"""
    
    def __init__(self):
        self.small = []  # Max-heap (store NEGATED values since Python has min-heap only)
        self.large = []  # Min-heap (store values as-is)
    
    def addNum(self, num):
        # Always add to max-heap first
        heapq.heappush(self.small, -num)
        
        # INVARIANT 1: max of small must be <= min of large
        if self.small and self.large and -self.small[0] > self.large[0]:
            val = -heapq.heappop(self.small)
            heapq.heappush(self.large, val)
        
        # INVARIANT 2: balance sizes (small can have at most 1 more)
        if len(self.small) > len(self.large) + 1:
            val = -heapq.heappop(self.small)
            heapq.heappush(self.large, val)
        if len(self.large) > len(self.small):
            val = heapq.heappop(self.large)
            heapq.heappush(self.small, -val)
    
    def findMedian(self):
        if len(self.small) > len(self.large):
            return -self.small[0]           # Odd count: median is max of smaller half
        return (-self.small[0] + self.large[0]) / 2  # Even count: average of two middles
```

```java Java
class MedianFinder {
    // Two heaps: max-heap for smaller half, min-heap for larger half
    private PriorityQueue<Integer> small;  // Max-heap
    private PriorityQueue<Integer> large;  // Min-heap
    
    public MedianFinder() {
        small = new PriorityQueue<>(Collections.reverseOrder());
        large = new PriorityQueue<>();
    }
    
    public void addNum(int num) {
        small.offer(num);
        
        // Ensure max of small <= min of large
        if (!small.isEmpty() && !large.isEmpty() && small.peek() > large.peek()) {
            large.offer(small.poll());
        }
        
        // Balance sizes
        if (small.size() > large.size() + 1) {
            large.offer(small.poll());
        }
        if (large.size() > small.size()) {
            small.offer(large.poll());
        }
    }
    
    public double findMedian() {
        if (small.size() > large.size()) {
            return small.peek();
        }
        return (small.peek() + large.peek()) / 2.0;
    }
}
```

### 5. K Closest Points to Origin

<CodeGroup>
```python Python
import heapq

def k_closest(points, k):
    """Find k closest points to origin"""
    # Max-heap of (-distance, point) to keep k smallest
    max_heap = []
    
    for x, y in points:
        dist = -(x * x + y * y)  # Negate for max-heap
        
        if len(max_heap) < k:
            heapq.heappush(max_heap, (dist, [x, y]))
        elif dist > max_heap[0][0]:
            heapq.heapreplace(max_heap, (dist, [x, y]))
    
    return [point for _, point in max_heap]
```

```java Java
public int[][] kClosest(int[][] points, int k) {
    // Find k closest points to origin
    // Max-heap to keep k smallest
    PriorityQueue<int[]> maxHeap = new PriorityQueue<>(
        (a, b) -> (b[0]*b[0] + b[1]*b[1]) - (a[0]*a[0] + a[1]*a[1])
    );
    
    for (int[] point : points) {
        maxHeap.offer(point);
        if (maxHeap.size() > k) {
            maxHeap.poll();
        }
    }
    
    int[][] result = new int[k][2];
    for (int i = 0; i < k; i++) {
        result[i] = maxHeap.poll();
    }
    return result;
}
```

```cpp C++
vector<vector<int>> kClosest(vector<vector<int>>& points, int k) {
    // Find k closest points to origin
    auto cmp = [](vector<int>& a, vector<int>& b) {
        return a[0]*a[0] + a[1]*a[1] < b[0]*b[0] + b[1]*b[1];
    };
    priority_queue<vector<int>, vector<vector<int>>, decltype(cmp)> maxHeap(cmp);
    
    for (auto& point : points) {
        maxHeap.push(point);
        if (maxHeap.size() > k) {
            maxHeap.pop();
        }
    }
    
    vector<vector<int>> result;
    while (!maxHeap.empty()) {
        result.push_back(maxHeap.top());
        maxHeap.pop();
    }
    return result;
}
```

## Common Mistakes

<Warning>
**Avoid These Pitfalls:**
1. **Using the wrong heap type**: To find k **largest**, use a **min**-heap of size k. To find k **smallest**, use a **max**-heap of size k. This is counter-intuitive. Think of it this way: the heap root is the "gatekeeper" -- it is the weakest element in your top-k collection. For k largest, the gatekeeper is the smallest of the top k (hence min-heap).
2. **Python max-heap pitfall**: Python only has min-heap (`heapq`). To simulate max-heap, negate values on push and negate again on pop. A common bug: forgetting to negate on pop, which returns negative values as your "maximum."
3. **Not managing heap size**: For top-k problems, always pop when heap size exceeds k. If you forget, the heap grows to size n and you have done nothing better than sorting.
4. **Comparing non-comparable objects in Python**: `heapq` compares tuples element by element. If two tuples have equal first elements, it compares the second element. If the second element is a ListNode (not comparable), you get a TypeError. Always include a unique tiebreaker (e.g., an index) as the second element: `(priority, index, item)`.
5. **Confusing heapify with repeated inserts**: Building a heap from an array using `heapq.heapify()` is O(n). Inserting n elements one by one is O(n log n). When you already have all elements, use heapify.
</Warning>

## Worked LeetCode Problems

Five high-frequency heap problems, each walked through end to end. The point is not just the answer but the **decision** -- why a heap, why this heap, what is in the heap, and what alternative you would mention to the interviewer.

### LC 215: Kth Largest Element in an Array (Medium)

**Decision:** Min-heap of size k. Streaming-friendly even though the input is an array, and O(n log k) is the cleanest. Mention quickselect as the O(n)-average alternative.

```python
import heapq

def findKthLargest(nums, k):
    heap = []
    for n in nums:
        heapq.heappush(heap, n)
        if len(heap) > k:
            heapq.heappop(heap)
    return heap[0]
```

**Trace** with `nums = [3,2,1,5,6,4], k = 2`:
- push 3 -> [3], push 2 -> [2,3], size now 2.
- push 1 -> [1,2,3], pop -> [2,3].
- push 5 -> [2,3,5], pop -> [3,5].
- push 6 -> [3,5,6], pop -> [5,6].
- push 4 -> [4,5,6], pop -> [5,6]. Return `heap[0] = 5`.

### LC 295: Find Median from Data Stream (Hard)

**Decision:** Two heaps. Max-heap for the lower half, min-heap for the upper half. The trick is the cross-heap invariant: `max(lo) <= min(hi)`.

See Template 3 above for the full code. Common interview trap: forgetting to push to `lo` first and rebalance after. The "push to lo, move top to hi, rebalance back if needed" sequence is the **canonical 3-step move** -- memorize that order; it is correct regardless of whether the new number belongs in the lower or upper half.

### LC 23: Merge K Sorted Lists (Hard)

**Decision:** Min-heap of size k holding `(value, tiebreaker_index, node)`. The tiebreaker is mandatory in Python because `ListNode` is not comparable.

```python
import heapq

def mergeKLists(lists):
    heap = []
    for i, head in enumerate(lists):
        if head:
            heapq.heappush(heap, (head.val, i, head))
    dummy = ListNode(0); cur = dummy
    while heap:
        val, i, node = heapq.heappop(heap)
        cur.next = node; cur = node
        if node.next:
            heapq.heappush(heap, (node.next.val, i, node.next))
    return dummy.next
```

**Why O(N log k) beats O(N log N):** the heap holds at most k items at any time. Each of the N total nodes is pushed and popped exactly once at O(log k). Mention divide-and-conquer pairwise merge as the equally good alternative -- same complexity, no heap, but harder to extend to a streaming setup.

### LC 347: Top K Frequent Elements (Medium)

**Decision:** Hash map for counts, then min-heap of size k keyed by frequency. A senior signal here is to mention the **bucket sort** alternative: indexing buckets from 0 to n by count gives an O(n) solution, which beats the heap when k is close to n.

```python
import heapq
from collections import Counter

def topKFrequent(nums, k):
    cnt = Counter(nums)
    heap = []
    for num, freq in cnt.items():
        heapq.heappush(heap, (freq, num))
        if len(heap) > k:
            heapq.heappop(heap)
    return [num for _, num in heap]
```

### LC 502: IPO (Hard)

**Decision:** Two heaps. A min-heap on `capital` to know which projects are unlocked at the current capital, plus a max-heap on `profit` to greedily pick the most profitable unlocked project. The interaction between them is the whole point of the problem.

```python
import heapq

def findMaximizedCapital(k, w, profits, capital):
    by_capital = sorted(zip(capital, profits))    # ascending by capital
    available = []                                  # max-heap on profit
    i = 0
    for _ in range(k):
        while i < len(by_capital) and by_capital[i][0] <= w:
            heapq.heappush(available, -by_capital[i][1])
            i += 1
        if not available:
            break                                    # no project affordable; stop
        w += -heapq.heappop(available)               # take the most profitable
    return w
```

The pattern -- **"unlock items as a budget grows, then greedily pick from the unlocked pool"** -- shows up in LC 1834, 1942, 2402, and many scheduling problems. Recognizing it is worth more than memorizing this specific code.

## Caveats and Traps

<Warning>
**Heap traps that bite even strong candidates:**

1. **`heapq` is min-heap only -- there is no `max_heappush`.** For a max-heap of integers, push `-x`. For a max-heap of objects, either wrap in a class with `__lt__` reversed, or store `(-priority, tiebreaker, obj)` tuples. A frequent bug is forgetting to negate on **pop**, which silently returns negative values as your "maximum."
2. **Tuple comparison crashes on non-comparable secondary keys.** `heapq.heappush(heap, (priority, my_obj))` fails the moment two priorities tie and Python tries to compare two `my_obj` instances. Always use a tiebreaker counter: `(priority, next(counter), my_obj)`. This is the bug behind most "TypeError: '<' not supported between instances" runtime errors in heap solutions.
3. **Heaps do not support arbitrary delete in O(log n).** Calling `heap.remove(x)` is O(n) because Python has to find the element first. If you need to delete by key, use **lazy deletion** (Template 4 above) or switch to a balanced BST (`sortedcontainers.SortedList`, Java `TreeMap`, C++ `std::set`).
4. **Building a heap with N pushes is O(N log N), not O(N).** If you have all elements upfront, call `heapq.heapify(arr)` -- that is O(N) using Floyd's algorithm. The N-pushes mistake is the most common subtle performance bug in heap code.
5. **Forgetting to bound the heap size in top-K problems.** If you push every element without popping, the heap grows to size n and you have done nothing better than sorting. The whole point of the gatekeeper pattern is the size cap.
6. **`heapq.heappush` then `heapq.heappop` is two operations; `heapq.heapreplace` and `heapq.heappushpop` are one.** Use them when you know you will both push and pop -- they are roughly twice as fast and avoid an unnecessary sift cycle.
7. **Java `PriorityQueue.contains(x)` is O(n).** Many candidates assume it is O(log n). For membership tests, maintain a parallel `HashSet`.
8. **C++ `priority_queue` is a max-heap by default.** Forgetting `greater<int>` flips your algorithm and you will not notice until you debug for an hour.
</Warning>

<Tip>
**Patterns that pair with the warnings above:**

- **Negation idiom for max-heap in Python:** push `-x`, pop `-heap[0]`, peek `-heap[0]`. Cleaner than wrapping in a class for primitive types.
- **Itertools counter for tiebreakers:** `counter = itertools.count()` then `next(counter)` is the cleanest way to produce a monotonically increasing tiebreaker. It avoids manually tracking an `idx` variable.
- **Lazy deletion threshold:** if more than half the heap is tombstones, call `heapq.heapify` after filtering. O(n) once is much cheaper than carrying tombstones forever.
- **Heap plus hash set for "exists" queries:** the heap gives you ordering, the set gives you O(1) membership. Together they mimic an indexed priority queue without writing one.
- **Use `heapreplace` over push-then-pop in top-K loops:** one sift instead of two. On large datasets this can shave 30 percent off your runtime.
</Tip>

## Interview Questions

Deep-dive questions that test real understanding of heap internals, trade-offs, and production judgment -- not just whether you can call `heapq.heappush`.

<AccordionGroup>
  <Accordion title="Q1: Why do you use a min-heap to find the K largest elements instead of a max-heap? Walk me through the reasoning." icon="brain">
    **Difficulty:** Foundational

    **What interviewers are really testing:** Whether you actually understand the counter-intuitive heap choice or just memorized "use min-heap for K largest." This is the single most common heap confusion, and a candidate who can clearly articulate WHY reveals they understand the data structure, not just the recipe.

    **Strong answer:**

    - The core insight is that we want the heap to act as a **gatekeeper**. A min-heap of size K keeps the smallest of the K largest elements at the top. Any incoming element smaller than the top cannot be in the K largest, so we skip it. Any element larger than the top kicks out the current smallest.
    - If we used a max-heap of all N elements instead, we would need to pop K times to get the K largest. That is O(N + K log N) -- building the heap is O(N), each pop is O(log N). With the min-heap of size K, we process each element in O(log K), giving O(N log K) total.
    - **When K is much smaller than N** (the common case -- "find the top 10 out of 10 million"), the min-heap approach is dramatically better: O(N log 10) versus O(N + 10 log N). Both are technically O(N) in this case, but the min-heap uses O(K) space versus O(N) space for the max-heap approach.
    - Think of it like a velvet rope at a club: the min-heap's top is the "shortest person allowed in." Anyone shorter gets rejected. Anyone taller kicks out the current shortest. After everyone has walked by, you have the K tallest people.

    ```python
    # Min-heap of size K acts as a gatekeeper
    import heapq

    def find_k_largest(nums, k):
        min_heap = []
        for num in nums:
            heapq.heappush(min_heap, num)
            if len(min_heap) > k:
                heapq.heappop(min_heap)  # Evict the smallest of our candidates
        return min_heap  # Contains k largest; min_heap[0] is the kth largest
    ```

    **Red flag answer:** "Because Python only has min-heap." This is an implementation detail, not the actual reason. The algorithm design choice is the same regardless of language. Another red flag: not being able to explain why a max-heap of size N is worse, or saying "I just memorized it."

    **Follow-ups:**
    1. What if K is close to N (say K = N - 5)? Is a min-heap of size K still the best approach? (Answer: no. When K is close to N, it is cheaper to find the N - K smallest elements using a max-heap of size N - K and return everything else. Or simply sort -- at K near N, O(N log N) sorting is competitive with O(N log K) heap.)
    2. What is the alternative approach using Quickselect, and when would you prefer it over a heap? (Answer: Quickselect is O(N) average case for finding the Kth largest in-place, but O(N^2) worst case. Prefer it when you do not need the elements sorted, can tolerate worst-case risk, and want average-case speed. Prefer the heap when data is streaming or you need a stable O(N log K) guarantee.)
  </Accordion>

  <Accordion title="Q2: Explain how a heap is represented as an array. Why not use an actual tree with node pointers?" icon="sitemap">
    **Difficulty:** Intermediate

    **What interviewers are really testing:** Whether you understand the internal representation of a heap and can reason about why the array layout matters for performance. This separates candidates who use heaps as a black box from those who understand the data structure deeply enough to implement or debug one.

    **Strong answer:**

    - A heap is a **complete binary tree**, meaning every level is fully filled except possibly the last, which is filled left to right. This completeness property means we can map every node to an array index with **zero wasted space**: for a node at index `i`, its left child is at `2i + 1`, right child at `2i + 2`, and parent at `(i - 1) // 2`.
    - **Why not pointers?** Three reasons, all performance-critical:
      1. **Cache locality:** Array elements are contiguous in memory. When you bubble up or bubble down, you access parent-child pairs that are close in the array. With pointer-based trees, nodes could be scattered across memory, causing cache misses on every traversal. On modern CPUs, a cache miss can cost 100+ cycles vs 4 cycles for an L1 hit.
      2. **Space overhead:** Each pointer-based node needs two child pointers (16 bytes on 64-bit systems) plus a parent pointer. An array node needs zero extra bytes -- position encodes the relationships.
      3. **Allocation cost:** Pointer-based trees require per-node memory allocation. Arrays can be allocated in a single block and grown with amortized O(1) doubling.
    - In practice, the array representation is so superior for heaps that I have never seen a production priority queue use pointer-based nodes. The cache locality advantage alone makes it 5-10x faster in benchmarks on large heaps.

    **Red flag answer:** "Because arrays are simpler to code." This is true but misses the fundamental performance argument. Another red flag: not knowing the index formulas for parent/child, or confusing a heap's array representation with a BST (which cannot use an array efficiently because BSTs are not necessarily complete).

    **Follow-ups:**
    1. What happens to the array representation when you delete an arbitrary element (not the root) from the heap? (Answer: you swap the target element with the last element, remove the last element, then bubble up or bubble down the swapped element to restore the heap property. This is O(log N) but finding the element is O(N) unless you maintain a separate index map -- which is exactly what indexed priority queues like those in Dijkstra implementations do.)
    2. Can you build a heap from an unsorted array in O(N) instead of O(N log N)? How? (Answer: yes, using Floyd's heapify algorithm. Start from the last non-leaf node and sift down each node. The key insight is that most nodes are near the leaves and sift down a short distance. The sum `N/4 * 1 + N/8 * 2 + N/16 * 3 + ...` converges to O(N), not O(N log N). This is the same algorithm used by `heapq.heapify` in Python.)
  </Accordion>

  <Accordion title="Q3: Design a data structure that finds the median from a continuous stream of numbers. Walk me through every design decision." icon="chart-simple">
    **Difficulty:** Senior

    **What interviewers are really testing:** This is the classic two-heap problem, but the interviewer wants to see whether you can derive the approach from first principles, handle edge cases precisely, and reason about why two heaps are the right choice over alternatives. It is also a test of your ability to explain a moderately complex invariant clearly under pressure.

    **Strong answer:**

    - **The core idea:** Split the stream into two halves. A **max-heap** stores the smaller half (so the largest of the small elements is at the top). A **min-heap** stores the larger half (so the smallest of the large elements is at the top). The median is either the top of the max-heap (odd count) or the average of both tops (even count).
    - **Invariant to maintain:** (1) Every element in the max-heap is less than or equal to every element in the min-heap. (2) The heaps are balanced: their sizes differ by at most 1, with the max-heap allowed to have the extra element.
    - **Insertion logic:** Push to the max-heap first. If the max-heap's top exceeds the min-heap's top, move it over. Then rebalance sizes.
    - **Why two heaps and not a sorted array?** A sorted array gives O(1) median access but O(N) insertion (shifting elements). Two heaps give O(1) median access and O(log N) insertion. For a stream of 10 million numbers, that is the difference between 10 million shifts per insert versus 23 comparisons.
    - **Why not a balanced BST (like a red-black tree)?** A balanced BST can find the median in O(log N) and insert in O(log N), so it is asymptotically equivalent. But the constant factor is worse due to pointer chasing and rebalancing overhead. Two heaps are simpler to implement and faster in practice. However, a BST (or an order-statistic tree) wins if you need to find arbitrary percentiles, not just the median.

    ```python
    import heapq

    class MedianFinder:
        def __init__(self):
            self.small = []  # max-heap (negated)
            self.large = []  # min-heap

        def addNum(self, num):
            heapq.heappush(self.small, -num)

            # Ensure max of small <= min of large
            if self.small and self.large and -self.small[0] > self.large[0]:
                heapq.heappush(self.large, -heapq.heappop(self.small))

            # Rebalance: small can have at most 1 extra
            if len(self.small) > len(self.large) + 1:
                heapq.heappush(self.large, -heapq.heappop(self.small))
            elif len(self.large) > len(self.small):
                heapq.heappush(self.small, -heapq.heappop(self.large))

        def findMedian(self):
            if len(self.small) > len(self.large):
                return -self.small[0]
            return (-self.small[0] + self.large[0]) / 2.0
    ```

    **Red flag answer:** "Use a sorted list and return the middle element." This is O(N) per insertion and shows the candidate does not understand why heaps exist for this problem. Another red flag: getting the balancing logic wrong, such as allowing the heaps to differ by more than 1, or not handling the cross-heap invariant (max of small must not exceed min of large). If a candidate cannot explain what goes wrong when that invariant is violated, they memorized the code but do not understand it.

    **Follow-ups:**
    1. How would you extend this to support `removeNum(val)` -- removing an arbitrary value from the stream? What is the complexity? (Answer: this is the "Sliding Window Median" variant (LC 480). You cannot efficiently delete from a heap. The standard trick is **lazy deletion**: mark elements as removed, and only actually pop them when they appear at the top during a normal operation. You need a HashMap tracking counts of "removed" elements. This makes the approach O(log N) amortized per operation.)
    2. What if instead of the median, you need the 95th percentile from a stream? Do two heaps still work? (Answer: yes, you can generalize to two heaps where the "small" heap holds 95% of the elements and the "large" heap holds 5%. The rebalancing condition changes from "differ by at most 1" to "maintain the 95/5 ratio." Alternatively, an order-statistic tree or a quantile sketch like t-digest is more appropriate at scale.)
  </Accordion>

  <Accordion title="Q4: You are given K sorted lists with a total of N elements. Merge them into one sorted list. What approach do you use, and why is it better than merging pairs iteratively?" icon="code-merge">
    **Difficulty:** Intermediate

    **What interviewers are really testing:** Whether you understand K-way merge as a heap application and can articulate the complexity advantage over the naive pairwise approach. This also reveals whether you understand why the heap size is K (not N), which is the key to the efficiency.

    **Strong answer:**

    - **Approach:** Use a min-heap of size K. Initialize it with the first element of each list (along with a list index to know which list it came from). Repeatedly pop the smallest element, append it to the result, and push the next element from that same list.
    - **Why the heap stays at size K:** At any moment, the heap contains exactly one element per list -- the current "head" of each list. We are always choosing the global minimum among K candidates. That is exactly what a min-heap of size K does in O(log K) per operation.
    - **Total complexity:** O(N log K) time, O(K) space for the heap. Each of the N elements is pushed and popped exactly once, each operation is O(log K).
    - **Why not pairwise merge?** If you merge two lists at a time (merge list 1 and 2, then result with 3, then with 4, etc.), each element from the first list gets touched in every subsequent merge. For K lists of size M each (N = K*M): pairwise is O(N*K) in the worst case. Heap merge is O(N log K). For K = 1000 lists, that is 1000x versus 10x overhead per element.
    - **The one nuance:** If you do pairwise merge using a **divide-and-conquer** strategy (merge adjacent pairs, then merge those results, etc.), you get O(N log K) as well -- same as the heap. But the heap approach is simpler to implement for linked lists and works naturally in a streaming context where lists arrive over time.

    ```python
    import heapq

    def merge_k_sorted(lists):
        min_heap = []
        for i, lst in enumerate(lists):
            if lst:
                heapq.heappush(min_heap, (lst[0], i, 0))  # (value, list_idx, elem_idx)

        result = []
        while min_heap:
            val, list_idx, elem_idx = heapq.heappop(min_heap)
            result.append(val)
            if elem_idx + 1 < len(lists[list_idx]):
                next_val = lists[list_idx][elem_idx + 1]
                heapq.heappush(min_heap, (next_val, list_idx, elem_idx + 1))
        return result
    ```

    **Red flag answer:** "Dump all elements into one list and sort." This is O(N log N) and ignores the fact that the lists are already sorted -- a major hint you are not leveraging. Another red flag: saying the heap has N elements instead of K, which means you do not understand the algorithm.

    **Follow-ups:**
    1. In Python's `heapq`, what happens when two elements have the same value? How does the tuple `(value, list_idx, elem_idx)` help? (Answer: Python compares tuples element by element. If values are equal, it compares `list_idx`, which breaks the tie deterministically. Without `list_idx`, comparing linked list nodes would crash because they are not comparable. This is a real Python gotcha that trips people up in interviews.)
    2. If the K sorted lists are being generated in real-time (e.g., K sensors each producing sorted readings), how does the heap approach adapt compared to the divide-and-conquer merge approach? (Answer: the heap approach adapts naturally -- just push each new reading into the heap as it arrives. Divide-and-conquer requires all data upfront. This is why K-way merge heaps are used in database merge operations and external sort.)
  </Accordion>

  <Accordion title="Q5: Implement a Task Scheduler. Given tasks with cooldown constraints, find the minimum time to complete all tasks. Why is a max-heap the right choice?" icon="list-check">
    **Difficulty:** Senior

    **What interviewers are really testing:** Whether you can apply heaps to a scheduling/greedy problem that is not obviously a "find K elements" pattern. This tests your ability to recognize that "always process the most frequent task first" is a greedy strategy that maps to a max-heap, and whether you can handle the cooldown queue correctly.

    **Strong answer:**

    - **Greedy insight:** To minimize total time, always execute the task with the **highest remaining frequency** first. Why? Idle slots happen when we cannot schedule any task during a cooldown period. High-frequency tasks create the most cooldown conflicts, so processing them first gives other tasks time to fill the gaps.
    - **Data structures needed:** A **max-heap** ordered by remaining frequency (so we always grab the most urgent task), plus a **cooldown queue** (a regular queue of `(available_time, frequency)` pairs) to hold tasks that are waiting out their cooldown.
    - **Algorithm:** At each time step: (1) move any task whose cooldown has expired from the queue back to the heap, (2) pop the most frequent task from the heap, decrement its count, (3) if it still has remaining executions, add it to the cooldown queue with `available_time = current_time + n + 1`, (4) if the heap is empty and the queue is not, increment time (idle slot).
    - **Complexity:** O(N * log 26) where N is total number of tasks. Since there are at most 26 distinct tasks, heap operations are O(log 26) = O(1). Total time: O(N). Space: O(26) = O(1).

    ```python
    import heapq
    from collections import Counter, deque

    def least_interval(tasks, n):
        freq = Counter(tasks)
        max_heap = [-count for count in freq.values()]
        heapq.heapify(max_heap)

        cooldown = deque()  # (available_time, neg_count)
        time = 0

        while max_heap or cooldown:
            time += 1
            if max_heap:
                count = heapq.heappop(max_heap) + 1  # +1 because negated
                if count != 0:
                    cooldown.append((time + n, count))
            if cooldown and cooldown[0][0] == time:
                heapq.heappush(max_heap, cooldown.popleft()[1])

        return time
    ```

    **Red flag answer:** Trying to simulate the schedule by iterating through task types in round-robin order without a heap. This works but is much harder to implement correctly and does not reflect understanding of the greedy strategy. Another red flag: not realizing there is a mathematical formula approach (`(max_freq - 1) * (n + 1) + count_of_max_freq_tasks`) that solves this in O(N) without simulation. If a candidate only knows the formula but cannot explain the simulation, or only knows the simulation but cannot derive the formula, both are partial understanding.

    **Follow-ups:**
    1. Can you derive the mathematical formula for the minimum intervals without simulating? (Answer: the most frequent task creates `max_freq - 1` gaps of size `n`. Each gap can be filled with other tasks. If there are enough other tasks, no idle slots. The formula is `max((max_freq - 1) * (n + 1) + count_of_tasks_with_max_freq, total_tasks)`.)
    2. What if tasks have different cooldowns -- task A needs 3 cooldown, task B needs 1 cooldown? Does the heap approach still work? (Answer: yes, the heap approach generalizes naturally because you just set each task's `available_time` based on its own cooldown. The formula approach breaks down completely because the structure of the gaps is no longer uniform.)
  </Accordion>

  <Accordion title="Q6: What is the time complexity of building a heap from an unsorted array? Most candidates say O(N log N). Why is it actually O(N)?" icon="clock">
    **Difficulty:** Intermediate

    **What interviewers are really testing:** Whether you understand Floyd's heapify algorithm and can reason about its non-obvious O(N) complexity. This is one of the most commonly mis-stated complexities in all of CS, and getting it right signals real algorithmic maturity.

    **Strong answer:**

    - The naive analysis says: we call sift-down on N/2 nodes, each sift-down is O(log N), so total is O(N log N). **This is a loose upper bound, not a tight one.**
    - The tight analysis: sift-down's cost depends on the **height of the node**, not the height of the tree. Most nodes are near the bottom of the tree where they have little to sift down. Specifically:
      - N/2 nodes are leaves (height 0) -- zero work.
      - N/4 nodes are at height 1 -- at most 1 swap each.
      - N/8 nodes are at height 2 -- at most 2 swaps each.
      - 1 node (root) is at height log N -- at most log N swaps.
    - The total work is: `sum from h=0 to log N of (N / 2^(h+1)) * h`. This simplifies to `N * sum from h=0 to infinity of h / 2^(h+1)` which converges to **N * 2 = O(N)**.
    - **Why sift-down from the bottom and not sift-up from the top?** If you insert elements one by one using sift-up, each insertion is O(log N) and you do N insertions, giving O(N log N). The sift-down approach is smarter because it does the expensive work (long sifts) on very few nodes (near the root) and cheap work (short sifts) on many nodes (near the leaves). Sift-up does the opposite -- expensive work on many nodes.

    **Red flag answer:** "It is O(N log N) because each insertion is O(log N)." This confuses the N-insertions approach with Floyd's heapify, which is a fundamentally different algorithm. Another red flag: knowing it is O(N) but being unable to explain why -- just saying "it is a known result" without the height-based argument.

    **Follow-ups:**
    1. Python's `heapq.heapify` is O(N). But what is the complexity of creating a heap by calling `heappush` N times? (Answer: O(N log N), because each `heappush` sifts up from the bottom. If you have all elements upfront, always use `heapify` rather than N pushes. This is a real performance trap in production Python code.)
    2. Does the O(N) build time still hold for a d-ary heap (each node has d children instead of 2)? (Answer: yes, but the constant factor changes. The height of a d-ary heap is O(log_d N), and sift-down at each level compares d children. The build is still O(N) by the same convergent series argument, but each sift-down step is O(d) instead of O(1), so the constant is larger.)
  </Accordion>

  <Accordion title="Q7: In Dijkstra's algorithm, why do we use a min-heap? What goes wrong if we use a regular queue (BFS) or a max-heap?" icon="route">
    **Difficulty:** Intermediate

    **What interviewers are really testing:** Whether you understand the connection between heaps and graph algorithms, specifically why priority ordering is essential for correctness in shortest-path algorithms. This bridges data structure knowledge with algorithm design and reveals whether the candidate sees heaps as a tool for greedy selection.

    **Strong answer:**

    - Dijkstra's algorithm processes nodes in order of **increasing distance from the source**. A min-heap guarantees we always extract the unvisited node with the smallest tentative distance. This is critical because once we process a node, we commit to its distance being final -- and that is only correct if no shorter path to it exists through an unprocessed node.
    - **With a regular queue (BFS):** BFS processes nodes in FIFO order, which is optimal only when all edge weights are equal (or 1). With weighted edges, BFS might process a node via a longer path before discovering the shorter path. For example: edges A->B (weight 10) and A->C->B (weights 1 + 1). BFS from A processes B via the direct edge first, committing to distance 10, missing the shorter path of distance 2.
    - **With a max-heap:** You would process the farthest node first, which is the worst possible greedy choice. You would constantly process nodes whose distances are not yet finalized, leading to incorrect results and potentially visiting every node multiple times.
    - **The heap makes Dijkstra a greedy algorithm:** at each step, we greedily finalize the node closest to the source. The min-heap is what enables this greedy selection efficiently. Without it, we would need O(V) linear scan per extraction, giving O(V^2 + E) total. With the heap, we get O((V + E) log V).

    ```python
    import heapq

    def dijkstra(graph, source):
        dist = {node: float('inf') for node in graph}
        dist[source] = 0
        min_heap = [(0, source)]

        while min_heap:
            d, u = heapq.heappop(min_heap)
            if d > dist[u]:
                continue  # Skip stale entries
            for v, weight in graph[u]:
                if dist[u] + weight < dist[v]:
                    dist[v] = dist[u] + weight
                    heapq.heappush(min_heap, (dist[v], v))
        return dist
    ```

    **Red flag answer:** "We use a min-heap because it is faster." This does not explain WHY the priority order matters for correctness. Speed is a consequence, not the reason. Another red flag: not knowing that Dijkstra fails with negative edge weights (the greedy assumption breaks), or claiming BFS works with weighted graphs.

    **Follow-ups:**
    1. The code above uses a lazy deletion approach (pushing duplicates and skipping stale entries). What is the alternative, and when is each approach preferred? (Answer: the alternative is a **decrease-key** operation on an indexed priority queue, which updates the priority of an existing entry. Decrease-key avoids duplicate entries but requires a more complex heap implementation. In practice, lazy deletion with a binary heap is simpler, and the overhead of stale entries is usually small. For dense graphs, the indexed approach with a Fibonacci heap gives O(V log V + E), but Fibonacci heaps are rarely used in practice due to large constant factors.)
    2. For a graph with only weights 0 and 1 on edges, is a heap still the best choice? (Answer: no. Use a **0-1 BFS** with a deque: push weight-0 edges to the front and weight-1 edges to the back. This gives O(V + E) time without any heap overhead. Generalizes to weights 0 through K with a K-bucket queue.)
  </Accordion>

  <Accordion title="Q8: You have an infinite stream of numbers and need to return the Kth largest at any point. How do you design this, and what are the trade-offs of different K values?" icon="infinity">
    **Difficulty:** Foundational

    **What interviewers are really testing:** Whether you can apply the min-heap-of-size-K pattern to a streaming context, and whether you understand how the choice of K affects space, time per insertion, and the overall design. This is a common system design adjacent question that bridges DSA with practical engineering.

    **Strong answer:**

    - **Approach:** Maintain a min-heap of size K. For each incoming number: if the heap has fewer than K elements, push it. Otherwise, if the number is larger than the heap's top (the current Kth largest), pop the top and push the new number. The Kth largest at any point is always `heap[0]`.
    - **Per-element cost:** O(log K) for the push/pop when the element qualifies; O(1) when it does not (just a comparison). In practice, as the stream progresses, fewer and fewer elements beat the top, so most elements are O(1) rejections.
    - **Space:** O(K), which is fixed regardless of stream length. After seeing 10 billion numbers, you still only store K.
    - **Trade-offs by K value:**
      - K = 1 (find the maximum): a single variable suffices. No heap needed.
      - K = 10: tiny heap, extremely fast. Perfect for "top 10 trending" features.
      - K = N/2 (median): the heap holds half the stream, which grows unbounded. Better to use the two-heap median approach.
      - K close to N: you are essentially storing the whole stream. At that point, a sorted structure or just sorting periodically is more practical.

    ```python
    import heapq

    class KthLargestStream:
        def __init__(self, k, initial_nums):
            self.k = k
            self.heap = []
            for num in initial_nums:
                self.add(num)

        def add(self, val):
            if len(self.heap) < self.k:
                heapq.heappush(self.heap, val)
            elif val > self.heap[0]:
                heapq.heapreplace(self.heap, val)  # More efficient than pop + push
            return self.heap[0]
    ```

    **Red flag answer:** "Sort the stream so far each time and return the Kth element." This is O(N log N) per query, catastrophic for a stream. Another red flag: using a max-heap instead of a min-heap and not realizing the error until prompted. Also watch for candidates who do not mention `heapreplace` -- it is a single operation (O(log K)) versus pop then push (two O(log K) operations with worse constant factor).

    **Follow-ups:**
    1. What if K changes dynamically -- sometimes the user asks for the 10th largest, sometimes the 100th? Can you adapt your data structure? (Answer: you would need to grow or shrink the heap. Growing from K to K' means you need access to previously rejected elements -- which you discarded. So you either need to store more data (an order-statistic tree, or a larger heap with lazy semantics), or accept that K changes are expensive. In practice, systems like Redis Sorted Sets handle this by storing all elements and using O(log N) rank queries.)
    2. In a distributed system with 100 servers each seeing a shard of the stream, how would you compute the global Kth largest? (Answer: each server maintains a local min-heap of size K. To find the global Kth largest, gather all 100 heaps (100*K elements), merge them, and find the Kth largest of that merged set. This is approximate if done continuously but exact if done as a periodic snapshot. For exact streaming, you need coordination -- e.g., each server streams its top-K updates to a coordinator that maintains a global heap.)
  </Accordion>

  <Accordion title="Q9: Compare using a heap versus a balanced BST versus a sorted array for a priority queue. When does each win?" icon="scale-balanced">
    **Difficulty:** Senior

    **What interviewers are really testing:** Whether you think in trade-offs rather than absolutes. Every data structure has a sweet spot, and an experienced engineer knows when a heap is NOT the right choice. This question reveals system design judgment and breadth of data structure knowledge.

    **Strong answer:**

    - **Binary heap (array-based):**
      - Insert: O(log N). Extract-min: O(log N). Peek: O(1). Build: O(N). Delete arbitrary: O(N) find + O(log N) remove.
      - **Wins when:** You only need insert + extract-min/max. This covers 90% of priority queue use cases. Excellent cache locality, tiny memory overhead, simple implementation.
      - **Loses when:** You need to search, delete, or update arbitrary elements efficiently.

    - **Balanced BST (Red-Black tree, AVL, or skip list):**
      - Insert: O(log N). Extract-min: O(log N). Peek: O(log N) or O(1) with a cached pointer. Delete arbitrary: O(log N). Find arbitrary: O(log N).
      - **Wins when:** You need **decrease-key**, **delete arbitrary**, or **search** operations. Examples: Dijkstra with decrease-key, sliding window problems where you remove elements that leave the window, or maintaining a leaderboard where players can leave.
      - **Loses when:** You only need basic priority queue operations. Worse cache locality, higher memory overhead (pointers), more complex implementation.

    - **Sorted array:**
      - Insert: O(N) due to shifting. Extract-min: O(1) from the end. Peek: O(1).
      - **Wins when:** Insertions are rare but extractions are frequent. Or the dataset is small (under ~1000 elements) where the O(N) shift is negligible and the sorted property gives you binary search for free.
      - **Loses when:** Insertions are frequent. The O(N) insert kills throughput at scale.

    - **Real-world decision framework:** Start with a heap. If you find yourself needing arbitrary delete or decrease-key, upgrade to a BST-based structure (like Java's `TreeMap` or C++'s `std::set`). If the dataset is tiny and you want simplicity, a sorted array is fine.

    **Red flag answer:** "Always use a heap for priority queues." This is the most common answer and it is incomplete. A candidate who cannot articulate when a BST or sorted array is better does not understand the trade-off space. Another red flag: not knowing that `std::priority_queue` in C++ does not support decrease-key or arbitrary deletion, which is a frequent source of bugs.

    **Follow-ups:**
    1. Java's `PriorityQueue` is a binary heap. But for the Sliding Window Median problem (LC 480), many solutions use two `TreeMap` structures instead. Why? (Answer: because the sliding window requires removing arbitrary elements as they leave the window. A binary heap has O(N) arbitrary deletion. A `TreeMap` (balanced BST) has O(log N) arbitrary deletion, which is critical when the window slides N times.)
    2. In what scenario would a Fibonacci heap be worth its implementation complexity? (Answer: theoretically, when you need O(1) amortized decrease-key, such as in Dijkstra's or Prim's algorithm on dense graphs. In practice, almost never -- the constant factors are so large that a binary heap with lazy deletion outperforms it until the graph has millions of nodes. The only real-world uses I know of are in specialized network optimization libraries.)
  </Accordion>

  <Accordion title="Q10: You are given a list of meeting intervals. Find the minimum number of conference rooms required. How does a heap help here, and what does the heap represent?" icon="calendar-check">
    **Difficulty:** Intermediate

    **What interviewers are really testing:** Whether you can apply a heap to a scheduling problem where the heap does not store "top K" elements but instead tracks **active resource usage**. This tests whether your understanding of heaps extends beyond the K-largest template to a fundamentally different use case: tracking the earliest-available resource.

    **Strong answer:**

    - **Key insight:** Sort meetings by start time. Use a min-heap to track the **end times of currently active meetings** (i.e., rooms in use). The heap represents rooms, and the value at the top is the room that becomes free the earliest.
    - **Algorithm:** For each meeting (in start-time order): if the earliest-ending meeting (heap top) ends before or at the current meeting's start, that room is reusable -- pop it. Then push the current meeting's end time. The heap's size at any point equals the number of rooms in use. Track the maximum size.
    - **Why a heap and not just a counter?** A counter only tells you how many rooms you need right now. But to decide whether to reuse a room, you need to know WHICH room frees up first. The min-heap gives you that in O(log N).
    - **Alternative approach:** The "event sweep" method -- create a sorted list of all start times (+1) and end times (-1), sweep through, and track the running count. This is also O(N log N) and arguably simpler, but the heap approach is more intuitive for many candidates and extends better to variants (like assigning specific rooms).

    ```python
    import heapq

    def min_meeting_rooms(intervals):
        if not intervals:
            return 0

        intervals.sort(key=lambda x: x[0])
        rooms = []  # min-heap of end times

        for start, end in intervals:
            if rooms and rooms[0] <= start:
                heapq.heappop(rooms)  # Reuse the room that freed up earliest
            heapq.heappush(rooms, end)

        return len(rooms)  # Peak concurrent rooms
    ```

    **Red flag answer:** "Sort by end time and use a greedy interval scheduling approach." That solves a different problem (maximum non-overlapping intervals). Confusing the two problems is a common mistake that shows the candidate is pattern-matching without reading the problem. Another red flag: not being able to explain what the heap elements represent (end times of active meetings, not the meetings themselves).

    **Follow-ups:**
    1. What if you also need to return WHICH meetings are assigned to which room? How does the data structure change? (Answer: instead of just storing end times, store `(end_time, room_id)` tuples in the heap. When reusing a room, pop the tuple and push a new one with the same `room_id`. When allocating a new room, assign a new `room_id`. This transforms the problem from counting rooms to actually assigning them.)
    2. The meeting intervals are arriving as a real-time stream (you do not know future meetings). Does the approach still work? How does your answer change? (Answer: the heap approach works perfectly in a streaming context -- you just process each meeting as it arrives. The event sweep approach requires all events upfront, so it breaks. This is a real advantage of the heap approach in production scheduling systems like Google Calendar's room booking engine.)
  </Accordion>

## Interview Deep-Dive

<AccordionGroup>
  <Accordion title="Why do you use a min-heap of size K for the K largest elements, not a max-heap of all N? What is the exact complexity difference?">
    **Strong Answer:**
    - The min-heap of size K is a **gatekeeper**. Its top is the smallest of the K largest seen so far. Elements smaller than the top are rejected in O(1). Larger elements kick out the top in O(log K).
    - **Max-heap of all N:** Build O(N), pop K times at O(log N). Total: O(N + K log N). Space: O(N).
    - **Min-heap of size K:** Process each element O(log K). Total: O(N log K). Space: O(K).
    - **When K is small (common case):** K=10, N=10M. Min-heap: O(10M * 3.3). Max-heap: O(10M + 10 * 23). Max-heap is theoretically faster but uses O(N) space.
    - **The real advantage:** O(K) space. For streaming data where N is unbounded, max-heap is impossible -- you cannot store all N elements.
    - Think of it as a velvet rope: the top is the "shortest person allowed in." Anyone shorter is rejected. Anyone taller kicks out the current shortest.

    **Follow-up: What if K is close to N (say K = N - 5)?**
    - Find the 5 smallest using a max-heap of size 5, return everything else. Or just sort -- O(N log N) is competitive and simpler when K is near N.
  </Accordion>

  <Accordion title="Building a heap from an unsorted array is O(N), not O(N log N). Explain why with the height-based argument.">
    **Strong Answer:**
    - **Naive bound:** Sift-down on N/2 nodes, each O(log N). Total: O(N log N). This is a **loose** bound.
    - **Tight bound:** Sift-down cost depends on the node's **height**, not the tree height. N/2 leaves: 0 work. N/4 at height 1: 1 swap each. N/8 at height 2: 2 swaps. 1 root: log N swaps.
    - Total = N * sum(h / 2^(h+1)) for h=0 to log N. This converges to **2N = O(N)**.
    - **Why sift-down beats sift-up:** Sift-down does heavy work on few nodes (root) and light work on many (leaves). Sift-up does the opposite. Most nodes are leaves with little to do.
    - **Practical implication:** `heapq.heapify()` is O(N). Calling `heappush()` N times is O(N log N). Always use heapify when all elements are available upfront.

    **Follow-up: Python's heapq has heapify. But what if you need a max-heap?**
    - Python only provides min-heap. For max-heap, negate values: push `-x`, pop gives `-max`. For custom comparisons, wrap in a class with `__lt__` or use `(priority, item)` tuples. At scale, consider `sortedcontainers.SortedList` for O(log N) operations with better flexibility.
  </Accordion>

  <Accordion title="Design a median-finding data structure for a continuous stream. Why two heaps, and what invariant must hold?">
    **Strong Answer:**
    - **Two heaps:** Max-heap for smaller half (top = largest of small), min-heap for larger half (top = smallest of large). Median is max-heap top (odd count) or average of both tops (even count).
    - **Invariant:** (1) All elements in max-heap are at most all elements in min-heap. (2) Sizes differ by at most 1 (max-heap gets the extra element).
    - **Why not sorted array:** O(1) median but O(N) insertion (shifting). Two heaps: O(1) median, O(log N) insertion.
    - **Why not balanced BST:** O(log N) for both operations, but worse constants due to pointer chasing. Two heaps are simpler and faster in practice. BST wins if you need arbitrary percentiles.
    - **Insertion:** Push to max-heap. If tops violate invariant, move max-heap top to min-heap. Rebalance sizes. At most 3 heap ops per insertion.

    **Follow-up: Extend to support removing elements (Sliding Window Median). What changes?**
    - Direct heap deletion is O(N). Use **lazy deletion**: HashMap tracks "pending removes," elements only removed when they surface at the top during normal operations. Each operation remains O(log N) amortized.
  </Accordion>

## Interview Q&A: Structured Whiteboard Drills

Three high-frequency heap interview questions, each rehearsed with the structure your interviewer wants to hear: framework first, then code, then crisp follow-ups.

<AccordionGroup>
  <Accordion title="Q1: Find Median from Data Stream -- design and code (LC 295)" icon="chart-simple">
    **Strong Answer Framework:**

    1. **Restate the problem in your own words.** "Numbers arrive one at a time. After each arrival, I need to return the median of everything seen so far. So insertion and `findMedian` happen interleaved."
    2. **Reject the obvious.** Sorted list -> O(n) insert (shifting). Sorting on every query -> O(n log n) per query. Both are too slow. State this out loud so the interviewer sees your tradeoff thinking.
    3. **Propose two heaps.** A max-heap `lo` for the smaller half, a min-heap `hi` for the larger half. The median is either the top of `lo` (odd count) or the mean of the two tops (even count).
    4. **State the invariants.** (a) Every element in `lo` is at most every element in `hi`. (b) `len(lo) == len(hi)` or `len(lo) == len(hi) + 1`.
    5. **Walk through the insertion dance.** Push to `lo`. Move `lo.top()` to `hi`. If `hi` is now larger than `lo`, push `hi.top()` back to `lo`. Three ops, always correct.
    6. **State complexity.** Insert O(log n), query O(1), space O(n). Mention that an order-statistic tree gives the same insert complexity but supports arbitrary percentiles.

    **Real LeetCode Example -- LC 295 full solution:**

    ```python
    import heapq

    class MedianFinder:
        def __init__(self):
            self.lo = []   # max-heap (negated values)
            self.hi = []   # min-heap

        def addNum(self, num: int) -> None:
            heapq.heappush(self.lo, -num)                     # 1. push to lo
            heapq.heappush(self.hi, -heapq.heappop(self.lo))  # 2. move lo.top -> hi
            if len(self.hi) > len(self.lo):                    # 3. rebalance
                heapq.heappush(self.lo, -heapq.heappop(self.hi))

        def findMedian(self) -> float:
            if len(self.lo) > len(self.hi):
                return float(-self.lo[0])
            return (-self.lo[0] + self.hi[0]) / 2.0
    ```

    Trace: add 1 -> lo=[-1], hi=[], median 1.0. add 2 -> lo=[-1], hi=[2], median 1.5. add 3 -> lo=[-2,-1], hi=[3], median 2.0.

    <Note>
    **Senior follow-up 1: How do you remove an arbitrary element (Sliding Window Median, LC 480)?**

    Use lazy deletion. Maintain a `removed` hash map. Whenever the top of either heap matches a tombstoned value, pop it. Each public operation calls a private `_purge` helper before reading the top. Amortized O(log n) per operation; worst case O(n) when the heap is dense in tombstones, but the amortized bound holds because each element is tombstoned at most once.
    </Note>

    <Note>
    **Senior follow-up 2: What if you need the 95th percentile instead of the median?**

    Generalize to two heaps where `lo` holds the bottom 95 percent and `hi` holds the top 5 percent. The rebalance condition becomes "maintain the 95/5 ratio." For arbitrary or shifting percentiles, switch to an order-statistic tree (Python `sortedcontainers.SortedList`, Java `TreeMap` with subtree-size augmentation) or a quantile sketch like t-digest for streaming at scale.
    </Note>

    <Note>
    **Senior follow-up 3: How does this scale to a distributed stream across 100 servers?**

    Each server maintains a local pair of heaps over its shard. To compute a global median, you need either (a) periodic snapshots merged at a coordinator (cheap but stale), or (b) a quantile sketch per server with mergeable summaries (t-digest is the production standard). Two heaps do not merge cleanly; sketches do. Mentioning t-digest signals you have seen this in production.
    </Note>

    **Common Wrong Answers:**

    1. **"Just keep a sorted list and index the middle."** Insertion is O(n) due to shifting. At 10 million numbers this is catastrophic; the interviewer is testing whether you recognize that.
    2. **"Sort every time `findMedian` is called."** O(n log n) per query. Even worse if queries are frequent.
    3. **"Two heaps but balance whenever sizes differ by anything."** Forgetting the cross-heap invariant (`max(lo) <= min(hi)`) -- you can keep balanced sizes while elements are on the wrong side. Tracing 1, 3, 2 in that order will reveal the bug.

    **Further Reading:**
    - LC 480 Sliding Window Median -- the lazy-deletion extension.
    - LC 1825 Finding MK Average -- harder generalization with three heaps and tombstones.
    - Dunning, "Computing Extremely Accurate Quantiles Using t-Digests" (2019).
  </Accordion>

  <Accordion title="Q2: Top K Frequent Elements -- heap versus bucket sort (LC 347)" icon="ranking-star">
    **Strong Answer Framework:**

    1. **Phase 1: count.** O(n) hash map. Trivial; do not dwell.
    2. **Phase 2: select top K by frequency.** This is where the interviewer wants to see range of thinking. Offer two solutions and weigh them.
    3. **Heap solution.** Min-heap of size k keyed on frequency. O(n log k). Space O(n + k). The same gatekeeper template as LC 215.
    4. **Bucket sort solution.** Frequency is at most n, so create n+1 buckets. Walk buckets from highest frequency to lowest, collect until you have k elements. O(n) time, O(n) space.
    5. **Recommend per context.** "If k is small relative to n, the heap is competitive and uses less memory. If k is close to n, bucket sort wins. The interviewer rule: lead with bucket sort if asked for optimal, lead with heap if asked for the most common pattern."

    **Real LeetCode Example -- LC 347 both solutions:**

    Heap (interview default):

    ```python
    import heapq
    from collections import Counter

    def topKFrequent(nums, k):
        cnt = Counter(nums)
        heap = []
        for num, freq in cnt.items():
            heapq.heappush(heap, (freq, num))
            if len(heap) > k:
                heapq.heappop(heap)
        return [num for _, num in heap]
    ```

    Bucket sort (optimal):

    ```python
    from collections import Counter

    def topKFrequent_bucket(nums, k):
        cnt = Counter(nums)
        buckets = [[] for _ in range(len(nums) + 1)]
        for num, freq in cnt.items():
            buckets[freq].append(num)
        result = []
        for f in range(len(buckets) - 1, 0, -1):
            for num in buckets[f]:
                result.append(num)
                if len(result) == k:
                    return result
        return result
    ```

    <Note>
    **Senior follow-up 1: Stream variant -- elements arrive forever and you need top K at any time.**

    Bucket sort cannot run on an unbounded stream because the bucket-array size is unbounded. The heap approach adapts: maintain a hash map of `value -> freq` and a min-heap of size k by current frequency. The wrinkle is that frequencies update in place -- a value already in the heap might need its frequency increased. Use **lazy deletion**: when you pop a `(freq, value)` pair, check whether the cached `value -> freq` matches; if not, the entry is stale and you continue popping.
    </Note>

    <Note>
    **Senior follow-up 2: Quickselect alternative -- when does it beat the heap?**

    Quickselect on the unique values by frequency is O(n) average, O(n^2) worst case. It is the "fastest in expectation" answer and the in-place answer, but the worst-case is real on adversarial inputs. In production code, a randomized quickselect with median-of-medians fallback gets you a guaranteed O(n) at the cost of larger constants. Most interviewers will accept "heap is the safe contract; quickselect is the throughput optimization" as the right framing.
    </Note>

    <Note>
    **Senior follow-up 3: What if the input is already sorted by frequency in a database query result?**

    Then there is no algorithm to run -- just take the first k. The senior signal here is recognizing that a lot of "top K" problems disappear when you control the upstream sort. In real systems this is a database `ORDER BY freq DESC LIMIT k` query, often backed by a precomputed index.
    </Note>

    **Common Wrong Answers:**

    1. **"Sort the frequency map by value."** O(n log n). Works but is suboptimal; the interviewer flagged "top K" specifically to test whether you know better.
    2. **"Max-heap of size n, then pop k times."** Builds a heap of all elements (O(n)), then pops k times (O(k log n)). Total O(n + k log n). Asymptotically fine when k is small but uses O(n) space versus the size-k heap's O(k). For streaming data the size-k version is the only option.
    3. **"Use Python's `Counter.most_common(k)` and call it done."** Functionally correct but says nothing about your algorithmic understanding. If the interviewer wanted a one-liner they would not have asked.

    **Further Reading:**
    - LC 692 Top K Frequent Words -- adds a lexicographic tiebreaker, exposes the custom comparator pattern.
    - LC 451 Sort Characters By Frequency -- bucket sort variant, cleaner statement.
    - LC 1985 Find the Kth Largest Integer in the Array -- string comparison plus heap, tests custom ordering.
  </Accordion>

  <Accordion title="Q3: Schedule meetings using a min-heap (LC 253)" icon="calendar-check">
    **Strong Answer Framework:**

    1. **Reframe as resource tracking.** "I am not finding K elements -- I am tracking how many rooms are simultaneously in use. The heap tracks active rooms by their end time."
    2. **Sort meetings by start.** O(n log n). Without this, you cannot reason about temporal overlap.
    3. **Iterate.** For each meeting, if the earliest-ending room frees up by my start time (heap top is at most my start), I reuse it -- pop. Then push my end time. Heap size at any moment is the number of rooms in use.
    4. **Answer.** The maximum heap size during the iteration -- which equals the heap size at the end after the if-pop step on each meeting -- is the minimum number of rooms required.
    5. **Mention the alternative.** Event sweep: create a sorted list of `(time, +1 or -1)` events, sweep through tracking a running counter. Same O(n log n), arguably simpler. Heap wins when you need to know which specific room each meeting was assigned to.

    **Real LeetCode Example -- LC 253 full solution:**

    ```python
    import heapq

    def minMeetingRooms(intervals):
        if not intervals:
            return 0
        intervals.sort(key=lambda x: x[0])      # sort by start time
        rooms = []                                # min-heap of end times
        for start, end in intervals:
            if rooms and rooms[0] <= start:
                heapq.heappop(rooms)              # earliest room is free; reuse
            heapq.heappush(rooms, end)
        return len(rooms)
    ```

    Trace `[[0,30],[5,10],[15,20]]`: sorted unchanged. Process `[0,30]` -- rooms=`[30]`. Process `[5,10]` -- top is 30, 30 is greater than 5, do not pop, push 10 -- rooms=`[10,30]`. Process `[15,20]` -- top is 10, 10 is at most 15, pop, push 20 -- rooms=`[20,30]`. Answer: 2.

    <Note>
    **Senior follow-up 1: Return the actual room assignment, not just the count.**

    Store `(end_time, room_id)` tuples in the heap. When you reuse, pop the tuple and push a new tuple with the same `room_id`. When you allocate a fresh room, increment a `next_room_id` counter. The output is a mapping from each meeting to a `room_id`. This is how a real calendar system actually does the scheduling -- not just counting rooms but assigning them.
    </Note>

    <Note>
    **Senior follow-up 2: Streaming meetings -- you do not know future meetings.**

    The heap approach works as-is; you process each meeting on arrival. Event sweep does not, because it requires all events upfront. This is the real-world advantage of the heap formulation. In a production booking service, your "current state" is exactly the heap of active reservations.
    </Note>

    <Note>
    **Senior follow-up 3: What if rooms have different sizes (capacity 4, 8, 16) and meetings have requested capacities?**

    Per-capacity heaps. One min-heap of end times per capacity tier. For a meeting requesting capacity c, search heaps starting at capacity c upward and pick the earliest-free slot. This generalizes to multi-resource scheduling. At very large scale you switch to interval trees because the per-capacity heap design starts thrashing when meetings have many possible capacity matches.
    </Note>

    **Common Wrong Answers:**

    1. **"Sort by end time and use greedy interval scheduling."** That solves a different problem -- maximum non-overlapping meetings. Confusing the two reveals you are pattern-matching keywords without reading.
    2. **"Just count concurrent intervals brute force, O(n^2)."** Correct but slow. The interviewer flagged "minimum rooms" because they expect O(n log n).
    3. **"Use a max-heap so the latest-ending room is on top."** The whole point is to find the earliest-freeing room to maximize reuse. Max-heap inverts the strategy.

    - LC 252 Meeting Rooms -- the easier "can attend all meetings" version, useful warmup.
    - LC 1834 Single-Threaded CPU -- two-heap variant of the same pattern.
    - LC 2402 Meeting Rooms III -- the assignment-tracking generalization.
