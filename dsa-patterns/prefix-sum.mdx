## What is Prefix Sum?

**Prefix Sum** (also called cumulative sum) is a preprocessing technique that stores running totals in an auxiliary array, enabling O(1) range sum queries after a one-time O(n) build step.

Think of it like a car's odometer. If you want to know the distance traveled between mile marker 30 and mile marker 75, you do not need to re-drive the route -- you simply subtract: 75 - 30 = 45 miles. A prefix sum array is the odometer for your data: `prefix[j+1] - prefix[i]` gives you the sum of any subarray `arr[i..j]` in constant time, no matter how large the range.

This is one of the highest-ROI patterns to learn: it is simple to implement, frequently tested, and forms the backbone of more advanced techniques like 2D prefix sums, difference arrays, and the HashMap + prefix sum combination for "subarray sum equals k."

<Note>
**Quick Recognition**: Problems involving range sums, subarray sums with a target, or cumulative calculations. Keywords: "sum of range", "subarray sum equals k", "cumulative", "contiguous array", "equilibrium index".
</Note>

## Canonical Template Code

Two templates cover almost every prefix sum problem you will see in interviews. Memorize both.

<CodeGroup>
```python Python
# Template A: BUILD a prefix sum array, then answer O(1) range queries.
# Use for: LC 303 (Range Sum Query - Immutable), LC 304 (2D version).
def build_prefix(arr):
    # Length n+1 with a leading 0. The leading 0 represents "sum of zero elements."
    # Why: lets us use the formula prefix[j+1] - prefix[i] for ANY range [i, j],
    #      including ranges starting at index 0, without special-casing.
    prefix = [0] * (len(arr) + 1)
    for i, v in enumerate(arr):
        prefix[i + 1] = prefix[i] + v        # prefix[i+1] = sum of arr[0..i]
    return prefix

def range_sum(prefix, i, j):
    # Returns sum of arr[i..j] inclusive in O(1).
    return prefix[j + 1] - prefix[i]


# Template B: PREFIX SUM + HASHMAP for "count / find subarrays with sum equals K".
# Use for: LC 560 (Subarray Sum Equals K), LC 974, LC 525, LC 1074 (after collapsing 2D->1D).
# Works WITH NEGATIVES (unlike sliding window).
def subarray_sum_equals_k(nums, k):
    # Insight: subarray nums[i+1..j] has sum K iff prefix[j+1] - prefix[i] == K
    # iff prefix[j+1] - K is a value we have already seen in the prefix array.
    count = 0
    prefix_sum = 0
    seen = {0: 1}                            # CRITICAL seed: empty prefix has sum 0, count 1.
                                              # Without this, you miss subarrays starting at index 0.
    for v in nums:
        prefix_sum += v
        # How many earlier prefix sums equal (prefix_sum - K)?
        # Each one corresponds to a subarray ending here that sums to K.
        if prefix_sum - k in seen:
            count += seen[prefix_sum - k]
        seen[prefix_sum] = seen.get(prefix_sum, 0) + 1
    return count
```

```java Java
// Template A: BUILD prefix sum array
public int[] buildPrefix(int[] arr) {
    int[] prefix = new int[arr.length + 1];   // leading 0 simplifies formula
    for (int i = 0; i < arr.length; i++) {
        prefix[i + 1] = prefix[i] + arr[i];
    }
    return prefix;
}
public int rangeSum(int[] prefix, int i, int j) {
    return prefix[j + 1] - prefix[i];         // O(1) range sum
}

// Template B: Prefix Sum + HashMap for subarray sum equals K
public int subarraySum(int[] nums, int k) {
    java.util.Map<Integer, Integer> seen = new java.util.HashMap<>();
    seen.put(0, 1);                           // CRITICAL: empty prefix
    int count = 0, prefixSum = 0;
    for (int v : nums) {
        prefixSum += v;
        count += seen.getOrDefault(prefixSum - k, 0);
        seen.merge(prefixSum, 1, Integer::sum);
    }
    return count;
}
```
</CodeGroup>

<Note>
**Why the leading 0 / `seen[0] = 1`?**
- In Template A, the leading 0 lets `prefix[j+1] - prefix[i]` work uniformly for ranges starting at index 0.
- In Template B, `seen[0] = 1` represents "the prefix of zero elements sums to 0, and we have seen it once." Without it, every subarray summing to K that starts at index 0 is missed -- because its required `prefix_sum - K = 0` would not be in the map.

This single line is the most commonly forgotten detail in interviews. Always seed the map.
</Note>

### 1. Basic Range Sum

```python Python
class NumArray:

    def __init__(self, nums):
        self.prefix = [0]
```

### 2. Subarray Sum Equals K

This is the most important prefix sum variation for interviews. The key insight: if `prefix_sum[j] - prefix_sum[i] == k`, then the subarray from index `i+1` to `j` sums to `k`. Instead of checking all pairs (O(n^2)), we use a HashMap to record how many times each prefix sum has occurred. For the current prefix sum, we ask: "How many earlier prefix sums equal `prefix_sum - k`?" Each one corresponds to a valid subarray.

**Complexity:** O(n) time, O(n) space.

**Critical detail:** Initialize `prefix_count = {0: 1}` to account for subarrays that start at index 0.

**Edge case:** This works with negative numbers, unlike the sliding window approach which requires all-positive values.

<CodeGroup>
```python Python
def subarray_sum(nums, k):
    """Count the number of contiguous subarrays that sum to exactly k.
    
    Example: nums = [1, 2, 3], k = 3
      prefix sums: 0, 1, 3, 6
      At prefix_sum=3: 3-3=0 is in map with count 1 -> found [1,2]
      At prefix_sum=6: 6-3=3 is in map with count 1 -> found [3]
    Answer: 2
    """
    count = 0
    prefix_sum = 0
    prefix_count = {0: 1}  # Base case: empty prefix has sum 0
    
    for num in nums:
        prefix_sum += num
        
        # How many earlier prefixes give us a subarray summing to k?
        if prefix_sum - k in prefix_count:
            count += prefix_count[prefix_sum - k]
        
        # Record this prefix sum for future lookups
        prefix_count[prefix_sum] = prefix_count.get(prefix_sum, 0) + 1
    
    return count
```

```java Java
public int subarraySum(int[] nums, int k) {
    // Count subarrays with sum exactly k
    int count = 0;
    int prefixSum = 0;
    Map<Integer, Integer> prefixCount = new HashMap<>();
    prefixCount.put(0, 1); // sum to frequency
    
    for (int num : nums) {
        prefixSum += num;
        
        // If (prefixSum - k) exists, we found subarrays
        if (prefixCount.containsKey(prefixSum - k)) {
            count += prefixCount.get(prefixSum - k);
        }
        
        prefixCount.put(prefixSum, 
            prefixCount.getOrDefault(prefixSum, 0) + 1);
    }
    
    return count;
}
```

```cpp C++
int subarraySum(vector<int>& nums, int k) {
    // Count subarrays with sum exactly k
    int count = 0;
    int prefixSum = 0;
    unordered_map<int, int> prefixCount;
    prefixCount[0] = 1; // sum to frequency
    
    for (int num : nums) {
        prefixSum += num;
        
        // If (prefixSum - k) exists, we found subarrays
        if (prefixCount.find(prefixSum - k) != prefixCount.end()) {
            count += prefixCount[prefixSum - k];
        }
        
        prefixCount[prefixSum]++;
    }
    
    return count;
}
```
</CodeGroup>

### 3. Product of Array Except Self

This is the "prefix product" variant. For each index, the answer is `(product of everything to the left) * (product of everything to the right)`. We build the left products in a forward pass and the right products in a backward pass, combining them in the result array.

**Complexity:** O(n) time, O(1) extra space (the output array does not count per the problem's convention).

**Why no division?** The problem explicitly forbids it to handle the zero case cleanly. With division you would need special logic for zeros; without it, the prefix approach handles zeros naturally.

<CodeGroup>
```python Python
def product_except_self(nums):
    """Compute product of all elements except self, without division.
    
    Example: [1, 2, 3, 4]
      Left pass:  result = [1, 1, 2, 6]   (product of elements before i)
      Right pass: result = [24, 12, 8, 6]  (multiply by product of elements after i)
    """
    n = len(nums)
    result = [1] * n
    
    # Forward pass: result[i] = product of all nums[0..i-1]
    left_product = 1
    for i in range(n):
        result[i] = left_product
        left_product *= nums[i]
    
    # Backward pass: multiply result[i] by product of all nums[i+1..n-1]
    right_product = 1
    for i in range(n - 1, -1, -1):
        result[i] *= right_product
        right_product *= nums[i]
    
    return result
```

```java Java
public int[] productExceptSelf(int[] nums) {
    // Without division, O(1) extra space (excluding output)
    int n = nums.length;
    int[] result = new int[n];
    Arrays.fill(result, 1);
    
    // Left products
    int leftProduct = 1;
    for (int i = 0; i < n; i++) {
        result[i] = leftProduct;
        leftProduct *= nums[i];
    }
    
    // Right products
    int rightProduct = 1;
    for (int i = n - 1; i >= 0; i--) {
        result[i] *= rightProduct;
        rightProduct *= nums[i];
    }
    
    return result;
}
```

```cpp C++

    for (int i = n - 1; i >= 0; i--) {
        result[i] *= rightProduct;
        rightProduct *= nums[i];
    }
    
    return result;
}
```

### 4. 2D Prefix Sum (Matrix Region Sum)

<CodeGroup>
```python Python
class NumMatrix:
    """2D range sum query"""
    
    def __init__(self, matrix):
        if not matrix:
            return
        
        m, n = len(matrix), len(matrix[0])
        self.prefix = [[0] * (n + 1) for _ in range(m + 1)]
        
        for i in range(m):
            for j in range(n):
                self.prefix[i+1][j+1] = (matrix[i][j] 
                    + self.prefix[i][j+1] 
                    + self.prefix[i+1][j] 
                    - self.prefix[i][j])
    
    def sumRegion(self, r1, c1, r2, c2):
        return (self.prefix[r2+1][c2+1] 
            - self.prefix[r1][c2+1] 
            - self.prefix[r2+1][c1] 
            + self.prefix[r1][c1])
```

Five canonical problems. For each: signal, brute force, optimized, common bugs.

### LC 303 -- Range Sum Query - Immutable

**Problem.** Implement `NumArray` with constructor and `sumRange(left, right)` returning the sum of `arr[left..right]` inclusive. Many queries on the same array.

**Pattern fit.** "Range sum query, multiple queries" -> build prefix sum once, answer each query in O(1).

**Brute force.** Loop through the range on every query. O(n) per query.

**Optimized.**

```python
class NumArray:
    def __init__(self, nums):
        self.prefix = [0] * (len(nums) + 1)
        for i, v in enumerate(nums):
            self.prefix[i + 1] = self.prefix[i] + v

    def sumRange(self, left, right):
        return self.prefix[right + 1] - self.prefix[left]
```

**Complexity.** O(n) build, O(1) per query, O(n) space.

**Common bugs.**
- **Returning `prefix[right] - prefix[left]`.** Off-by-one. Should be `prefix[right + 1] - prefix[left]`.
- **Building prefix without the leading 0.** Forces special-casing `left == 0`, leading to bugs.

### LC 560 -- Subarray Sum Equals K

**Problem.** Given an array of integers (can be negative) and a target K, count the number of contiguous subarrays summing to K.

**Pattern fit.** "Count subarrays with sum K, with negatives" -> Prefix Sum + HashMap. (Sliding window does not work due to negatives.)

**Brute force.** All pairs O(n squared) or all subarrays O(n cubed).

**Optimized.**

```python
def subarraySum(nums, k):
    count = 0
    prefix_sum = 0
    seen = {0: 1}                       # seed: prefix of zero elements sums to 0
    for v in nums:
        prefix_sum += v
        # Each earlier prefix equal to (prefix_sum - k) gives one subarray summing to k
        if prefix_sum - k in seen:
            count += seen[prefix_sum - k]
        seen[prefix_sum] = seen.get(prefix_sum, 0) + 1
    return count
```

**Why it works.** If `prefix[j] - prefix[i] == k`, then `nums[i..j-1]` sums to k. We do not store positions -- we store frequency, because multiple earlier prefixes can match.

**Complexity.** O(n) time, O(n) space.

**Common bugs.**
- **Forgetting `seen = {0: 1}`.** Misses every subarray starting at index 0. The single most common bug on LC 560.
- **Looking up `prefix_sum + k` instead of `prefix_sum - k`.** Wrong direction; counts subarrays summing to `-k`.
- **Storing the index instead of frequency.** This is the "longest" variant; for "count" you need frequencies because multiple earlier prefixes can satisfy the equation.

---

### LC 525 -- Contiguous Array

**Problem.** Given a binary array, return the maximum length of a contiguous subarray with equal numbers of 0s and 1s.

**Pattern fit.** "Equal counts of 0 and 1" -> transform 0 to -1, then find longest subarray with sum 0. Prefix Sum + HashMap, store first occurrence.

**Brute force.** All subarrays O(n squared).

**Optimized.**

```python
def findMaxLength(nums):
    seen = {0: -1}                      # prefix sum 0 first occurs at "index -1" (before array)
    prefix_sum = 0
    best = 0
    for i, v in enumerate(nums):
        prefix_sum += 1 if v == 1 else -1   # transform 0 -> -1
        if prefix_sum in seen:
            best = max(best, i - seen[prefix_sum])  # length = i - first_occurrence
        else:
            seen[prefix_sum] = i             # store ONLY the first occurrence
    return best
```

**Why store only first occurrence?** We want the *longest* subarray. The earliest occurrence of a prefix sum gives the longest possible subarray ending at any later index with the same prefix sum.

**Complexity.** O(n) time, O(n) space.

**Common bugs.**
- **Storing the latest occurrence instead of the first.** Returns shorter subarrays.
- **Forgetting `seen = {0: -1}`.** Misses subarrays starting at index 0. The `-1` makes `i - (-1) = i + 1` the correct length.
- **Not transforming 0 to -1.** You cannot detect "equal counts" with raw 0s and 1s.

### LC 238 -- Product of Array Except Self

**Problem.** Return an array where `output[i] = product of all elements except nums[i]`. Cannot use division. O(n) time required.

**Pattern fit.** Prefix product + suffix product, combined.

**Brute force.** For each i, multiply all others. O(n squared).

**Optimized (O(1) extra space).**

```python
def productExceptSelf(nums):
    n = len(nums)
    res = [1] * n

    # Forward pass: res[i] = product of nums[0..i-1]
    left = 1
    for i in range(n):
        res[i] = left
        left *= nums[i]

    # Backward pass: multiply res[i] by product of nums[i+1..n-1]
    right = 1
    for i in range(n - 1, -1, -1):
        res[i] *= right
        right *= nums[i]

    return res
```

**Why no division?** If `nums` contains a zero, division-based logic explodes. The prefix/suffix approach handles zeros naturally.

**Complexity.** O(n) time, O(1) extra space (output array does not count per problem convention).

**Common bugs.**
- **Using division anyway.** Fails when array contains zero(s).
- **Initializing `res` to nums or to 0.** Must initialize to 1, since we are accumulating products.
- **Forgetting the second pass.** Returns only left products, missing the suffix contribution.

### LC 304 -- Range Sum Query 2D - Immutable

**Problem.** Given a 2D matrix, answer many queries asking for the sum of elements within a sub-rectangle defined by `(row1, col1)` to `(row2, col2)`.

**Pattern fit.** "2D range sum, multiple queries" -> 2D prefix sum + inclusion-exclusion.

**Brute force.** Loop over the rectangle on every query. O(m * n) per query.

**Optimized.**

```python
class NumMatrix:
    def __init__(self, matrix):
        if not matrix or not matrix[0]:
            return
        m, n = len(matrix), len(matrix[0])
        self.p = [[0] * (n + 1) for _ in range(m + 1)]
        for i in range(m):
            for j in range(n):
                # inclusion-exclusion: top + left - top-left + current
                self.p[i+1][j+1] = (matrix[i][j]
                                    + self.p[i][j+1]
                                    + self.p[i+1][j]
                                    - self.p[i][j])

    def sumRegion(self, r1, c1, r2, c2):
        # query = bottom-right - top strip - left strip + top-left corner
        return (self.p[r2+1][c2+1]
                - self.p[r1][c2+1]
                - self.p[r2+1][c1]
                + self.p[r1][c1])
```

**Why inclusion-exclusion?** When you subtract the top strip and the left strip, the top-left rectangle is subtracted twice; add it back once.

**Complexity.** O(m * n) build, O(1) per query, O(m * n) space.

**Common bugs.**
- **Off-by-one in indices.** Mixing `r2` vs `r2 + 1` causes silent wrong answers. Stick to one convention.
- **Subtracting `p[r1][c1]` instead of `+ p[r1][c1]`.** Inclusion-exclusion sign flip.
- **Building without padding row/column.** Forces special-casing edge queries.

---

<Warning>
**Caveats and Traps -- Prefix Sum**

- **Forgetting `seen = {0: 1}` in count problems / `seen = {0: -1}` in longest problems.** This is THE classic prefix sum bug. The seed represents the empty prefix; without it, subarrays starting at index 0 are missed.
- **Storing index vs frequency in the wrong problem.** "Count subarrays" -> store frequency. "Longest subarray" -> store first index. "Shortest subarray" -> store latest index.
- **Using prefix sum on a problem that needs updates between queries.** Prefix sum is O(n) to rebuild after any update. Use Fenwick Tree / Segment Tree for mutable arrays.
- **Using prefix sum for min / max range queries.** Min and max have no inverse, so prefix arrays do not work. Use Sparse Table (O(1) query, O(n log n) build) or Segment Tree.
- **Off-by-one in `prefix[j+1] - prefix[i]` vs `prefix[j] - prefix[i-1]`.** Pick one convention. The leading-0 convention (`prefix[j+1] - prefix[i]`) is recommended -- it has fewer special cases.
- **2D inclusion-exclusion sign error.** Build: `+ left + top - top-left + cell`. Query: `- top - left + corner + bottom-right`. Memorize both formulas as a single block.
- **Prefix product with zeros.** If you build a single-direction prefix product and divide for queries, zeros break it. Always use the dual-direction (left product * right product) approach.
- **Confusing prefix sum with sliding window.** Sliding window is O(1) space and works on positive monotonic problems. Prefix Sum + HashMap is O(n) space and handles negatives. They are not interchangeable.
- **Modular arithmetic with prefix sums.** "Subarray sum divisible by K" requires storing remainders modulo K, not raw sums. Handle negative remainders carefully (`((sum % k) + k) % k` in languages where `%` can be negative).
- **Difference array / range update bugs.** For range update `[l, r] += val`: do `diff[l] += val` and `diff[r + 1] -= val`. Forgetting the `-=` after the range causes the addition to leak into later elements.
</Warning>

<Tip>
**Solutions and Patterns -- Prefix Sum Idioms**

- **Always seed the HashMap.** `{0: 1}` for counting subarrays summing to K. `{0: -1}` for longest subarray with property. This is the most important habit to build.
- **Build prefix array with length n+1 and leading 0.** Eliminates a class of off-by-one bugs at the cost of one extra array slot.
- **For "subarray sum divisible by K":** key by remainder, not sum. `(prefix_sum % K + K) % K` to normalize negatives.
- **For "longest" problems with prefix sums:** store FIRST occurrence; do not overwrite. For "shortest" problems, overwrite to keep latest.
- **For 2D prefix sums:** memorize the build and query formulas as a single mental block. Always pad with row 0 and column 0 of zeros.
- **For "subrectangle summing to target" (LC 1074):** fix two row boundaries, collapse to 1D column sums, run LC 560 on the 1D array. Total O(rows squared * cols).
- **For "product except self":** two passes, one accumulating from left, one from right. Avoid division to handle zeros gracefully.
- **For range updates with point queries:** use a difference array. `diff[l] += val; diff[r+1] -= val`. After all updates, prefix sum the diff array to get final values.
- **For range updates AND range queries:** Fenwick Tree with the difference-array trick, or a Segment Tree with lazy propagation.
- **For "minimum operations to reduce X to zero" (LC 1658):** reframe as "longest subarray summing to total - X". Total reframing is the prefix-sum power move.
</Tip>

## Interview Questions

### 1. Explain the prefix sum technique and why we add a leading zero to the prefix array.

**What interviewers are really testing:** Whether you understand the mathematical foundation, not just the code recipe. The leading zero is the most common source of off-by-one errors and reveals if you truly grok the indexing.

**Strong Answer:**
- Prefix sum is a preprocessing technique where `prefix[i]` stores the sum of all elements from index 0 to i-1 in the original array. This lets us compute any range sum `arr[i..j]` in O(1) using `prefix[j+1] - prefix[i]`, after a one-time O(n) build step.
- The leading zero (`prefix[0] = 0`) represents "the sum of zero elements." Without it, querying a range that starts at index 0 becomes a special case -- you would need an if-statement to check whether `i == 0`. With the leading zero, the formula `prefix[j+1] - prefix[i]` works uniformly for every range, including those starting at index 0.
- Think of it like a car odometer: `prefix[j+1]` is the reading at mile marker j+1, and `prefix[i]` is the reading at mile marker i. The difference gives you the distance between them. The odometer starts at 0, not at the first mile of driving.
- In practice, I always build prefix arrays with length `n+1` and initialize `prefix[0] = 0`. This eliminates an entire class of boundary bugs that show up in interviews under time pressure.

**Red flag answer:** "I just copy the elements into a new array and add them up." This shows no understanding of the constant-time query advantage. Another red flag is being unable to explain why the prefix array has `n+1` elements instead of `n`.

**Follow-ups:**
1. If the original array can be modified between queries, does prefix sum still work? What data structure would you switch to?
2. How would you extend prefix sums to handle 2D range queries on a matrix, and what is the inclusion-exclusion formula?

---

### 2. Walk me through how the HashMap + prefix sum technique solves "subarray sum equals k" in O(n).

**What interviewers are really testing:** This is the single most important prefix sum variant in interviews. They want to see if you can connect prefix sums to hash maps and understand the algebraic insight that transforms an O(n^2) scan into O(n).

**Strong Answer:**
- The core insight is algebraic: if `prefix[j] - prefix[i] == k`, then the subarray from index `i+1` to `j` sums to `k`. Instead of checking all O(n^2) pairs of (i, j), we flip the equation: for each `j`, we ask "how many earlier prefix sums equal `prefix[j] - k`?" A HashMap tracks the frequency of every prefix sum seen so far, giving us that count in O(1).
- The critical initialization is `prefix_count = {0: 1}`. This accounts for subarrays that start at index 0 -- without it, you miss every subarray whose sum equals `k` from the very beginning of the array. I have seen this bug in real interviews; it is the number one mistake on this problem.
- Unlike sliding window, this approach handles negative numbers. Sliding window relies on the property that extending the window increases the sum, which breaks with negatives. The HashMap approach has no such restriction.
- Time is O(n), space is O(n) for the HashMap. In the worst case, every prefix sum is unique, so the map holds n entries.

**Red flag answer:** "I would use a nested loop to check all subarrays." That is O(n^2). Another red flag is forgetting to initialize the map with `{0: 1}` and not being able to explain why it is needed.

**Follow-ups:**
1. What if the problem asks for the longest subarray with sum k instead of counting all of them? How does your HashMap strategy change?
2. Can this technique work with floating-point numbers? What practical problems arise?

### 3. Solve "Product of Array Except Self" without using division. What is the trick?

**What interviewers are really testing:** Whether you can think in terms of prefix/suffix decomposition and handle the division-free constraint gracefully. This also tests O(1) space optimization skills.

**Strong Answer:**
- The key idea is to decompose each answer into two parts: the product of everything to the left of index i, and the product of everything to the right. We compute these with two linear passes.
- Forward pass: build `result[i] = product of nums[0..i-1]` using a running `left_product`. After this pass, `result = [1, nums[0], nums[0]*nums[1], ...]`.
- Backward pass: walk right to left with a running `right_product`, multiplying `result[i] *= right_product` at each step. After this, each `result[i]` contains the full product excluding `nums[i]`.
- Space is O(1) extra because we reuse the output array for the left products and only need a single variable for the right product. The problem statement says the output array does not count toward space complexity.
- The reason we avoid division: division introduces special-case handling for zeros. If the array contains one zero, every position except the zero's position would be zero; if it contains two or more zeros, everything is zero. The prefix approach handles zeros naturally without any branching.

**Red flag answer:** "Just multiply everything and divide by each element." This fails when the array contains zeros, and the problem explicitly says no division. If a candidate immediately reaches for division, they have not read the constraints.

**Follow-ups:**
1. What if the array could contain zeros? Walk me through exactly what happens in the two-pass approach.
2. Could you extend this to a "sum of array except self" variant in O(n) time and O(1) extra space?

### 4. Explain the 2D prefix sum technique and the inclusion-exclusion formula for querying a rectangular region.

**What interviewers are really testing:** Your ability to extend a 1D technique to 2D and handle the double-counting problem. This is a direct test of mathematical reasoning and spatial thinking.

**Strong Answer:**
- A 2D prefix sum array stores `prefix[i][j]` = sum of all elements in the submatrix from (0,0) to (i-1, j-1). We build an `(m+1) x (n+1)` matrix with a padding row and column of zeros, mirroring the 1D leading-zero trick.
- Build formula: `prefix[i+1][j+1] = matrix[i][j] + prefix[i][j+1] + prefix[i+1][j] - prefix[i][j]`. We add the top portion and left portion, but the top-left corner gets counted twice, so we subtract it once. This is the inclusion-exclusion principle.
- Query formula for the rectangle from (r1, c1) to (r2, c2): `prefix[r2+1][c2+1] - prefix[r1][c2+1] - prefix[r2+1][c1] + prefix[r1][c1]`. Same inclusion-exclusion logic in reverse: take the big rectangle, subtract the top strip and left strip, add back the corner that was subtracted twice.
- Build is O(m*n), each query is O(1). This is used in problems like "number of submatrices that sum to target" where you need many range queries on a matrix.

**Red flag answer:** Getting the formula wrong or not being able to explain the "subtract and add back" logic. If someone memorizes the formula but cannot draw the four rectangles on a whiteboard, they will not survive follow-up pressure.

**Follow-ups:**
1. What is the time complexity if you need to answer Q queries on an m-by-n matrix? How does 2D prefix sum compare to brute force?
2. How would you handle updates to individual cells in the matrix? What data structure replaces the 2D prefix sum?

### 5. When would you NOT use a prefix sum array? Name situations where it is the wrong tool.

**What interviewers are really testing:** Engineering judgment and awareness of alternatives. Knowing when not to use a tool is just as important as knowing when to use it.

**Strong Answer:**
- When the array is modified frequently between queries. Prefix sums require O(n) to rebuild after any update, making them impractical. Use a Segment Tree or Binary Indexed Tree (Fenwick Tree) instead -- they handle both updates and queries in O(log n).
- When you need range minimum or range maximum instead of range sum. Prefix sums only apply to operations that have an inverse (subtraction undoes addition). Min/max have no inverse -- you cannot "subtract" a minimum. Use a Sparse Table (O(1) query, O(n log n) build, no updates) or Segment Tree.
- When there is only a single range query. Building the prefix array is O(n) and then one query is O(1), but just iterating over the range is also O(n). Prefix sum only pays off when you have multiple queries.
- When the operation is multiplication and the array contains zeros. Prefix products break because you cannot divide by zero to "undo" the product. You would need special-case handling or a different approach entirely.

**Red flag answer:** "Prefix sum always works for any range query." This shows a lack of understanding about the mathematical requirements (invertible operations) and the update limitation.

**Follow-ups:**
1. Explain the difference between a Segment Tree and a Fenwick Tree. When would you prefer one over the other?
2. What is a difference array, and how does it relate to prefix sums?

### 6. You have an array of 0s and 1s. Find the longest contiguous subarray with an equal number of 0s and 1s. Walk me through your approach.

**What interviewers are really testing:** Whether you can transform a problem into the prefix sum + HashMap pattern. This requires the creative insight of converting 0s to -1s.

**Strong Answer:**
- The trick is to transform the problem: replace every 0 with -1. Now "equal number of 0s and 1s" becomes "subarray with sum 0." This is a classic prefix sum + HashMap problem.
- After the transformation, compute running prefix sums. If `prefix[j] == prefix[i]`, then the subarray from `i+1` to `j` sums to zero, meaning it has equal 0s and 1s in the original array.
- Use a HashMap to store the first occurrence of each prefix sum value. For each new prefix sum, check if we have seen this value before. If yes, the distance `j - i` is a candidate for the longest subarray. We keep the first occurrence because we want the longest subarray, not the count.
- Edge case: initialize the map with `{0: -1}` to handle subarrays starting from index 0. If the prefix sum reaches 0 at index j, the entire subarray `[0..j]` is valid, and its length is `j - (-1) = j + 1`.
- Time O(n), space O(n). This is LeetCode 525 (Contiguous Array), frequently asked at Meta and Amazon.

**Red flag answer:** "I would check all subarrays and count zeros and ones." That is O(n^2). A worse red flag is not recognizing the 0-to-(-1) transformation and trying to force a sliding window, which does not work here because the transformed values include negative numbers.

**Follow-ups:**
1. What if instead of equal counts of 0 and 1, you need exactly k more 1s than 0s? How does the approach change?
2. How does this problem relate to "Subarray Sum Equals K"? What is the structural similarity?

### 7. An interviewer gives you an array and asks for the equilibrium index -- the index where the sum of elements to the left equals the sum to the right. How would you solve this?

**What interviewers are really testing:** Whether you can apply prefix sums cleanly to a simple problem without overcomplicating it. This also tests edge-case handling at the boundaries.

**Strong Answer:**
- Build a prefix sum array. For each index `i`, the left sum is `prefix[i]` and the right sum is `prefix[n] - prefix[i+1]`. If these are equal, `i` is an equilibrium index.
- You do not even need the full prefix array explicitly. Compute `total_sum` in one pass, then walk through the array maintaining a running `left_sum`. At each index, `right_sum = total_sum - left_sum - nums[i]`. If `left_sum == right_sum`, return the index. Then add `nums[i]` to `left_sum` and continue.
- This gives O(n) time and O(1) space -- no auxiliary array needed.
- Edge cases: the leftmost index (left sum is 0) and the rightmost index (right sum is 0) are both valid candidates. Negative numbers are fine; no special handling needed.

**Red flag answer:** Building two separate arrays for left sums and right sums with O(n) extra space when the O(1) approach is straightforward. This is not wrong, but it signals a lack of optimization awareness that interviewers notice at the senior level.

**Follow-ups:**
1. What if there are multiple equilibrium indices? Should you return the first, last, or all of them?
2. Can you adapt this approach to find a "pivot" in a 2D matrix where the top-left and bottom-right quadrant sums are equal?

    **What interviewers are really testing:** Whether you can transform a problem into the prefix sum + HashMap pattern, and whether you understand the fundamental limitation of sliding window when negative values are involved.

    **Strong Answer:**
    - The trick is to transform the array: replace every 0 with -1. Now "equal number of 0s and 1s" becomes "subarray with sum 0." A subarray with k ones and k zeros has sum `k * 1 + k * (-1) = 0` after the transformation.

After the 0-to-(-1) transformation, elements can be negative, so expanding the window might decrease the sum. You cannot reason about when to shrink. Use the same prefix sum + HashMap approach, but instead of looking for `prefix[j] == prefix[i]`, look for `prefix[j] - prefix[i] == K`, which means checking if `prefix[j] - K` exists in the map. Store the first occurrence of each prefix sum value (not the count, since we want the longest, not the count of subarrays). Time and space remain O(n).

  <Accordion title="Given a matrix of integers, count the number of submatrices that sum to a given target. How does the 2D prefix sum help, and what is the time complexity?">
    **What interviewers are really testing:** Whether you can combine 2D prefix sums with the HashMap technique from "Subarray Sum Equals K" to reduce a 4D brute force into something manageable. This is a hard problem that tests both spatial reasoning and pattern composition.

    **Strong Answer:**
    - Brute force would enumerate all O(n^2 * m^2) pairs of top-left and bottom-right corners and compute each sum. Even with 2D prefix sums reducing the sum query to O(1), the enumeration itself is O(n^2 * m^2).
    - The optimized approach fixes two row boundaries (top row `r1` and bottom row `r2`) and collapses the 2D problem into a 1D "Subarray Sum Equals K" problem. For a fixed pair of rows, compute the column-wise sum between those two rows for each column j: `col_sum[j] = sum of matrix[r1..r2][j]`. This is a 1D array, and we need to count subarrays of this 1D array that sum to the target. Use the HashMap + prefix sum technique in O(m) time.
    - We iterate over all O(n^2) pairs of row boundaries, and for each pair we do O(m) work. Total time: O(n^2 * m). If we fix column boundaries instead and iterate row-wise, the time is O(m^2 * n). Choose whichever dimension is smaller to fix as the outer loop.
    - The column-wise sums can be computed incrementally: when extending `r2` by one row, update each `col_sum[j]` by adding `matrix[r2][j]`. This avoids rebuilding the sums from scratch.

    **Follow-up:** What if the matrix is very large (say 10,000 x 10,000) and the target is zero? Are there any optimizations specific to the target being zero?

    **Follow-up Answer:** For target zero, the HashMap + prefix sum approach does not change algorithmically, but there is a practical consideration: when the target is zero, `prefix_sum - target == prefix_sum`, so you are counting duplicate prefix sums within each row-pair iteration. This means the HashMap will have many hits if the column sums frequently return to the same value. For a dense matrix with small integer values, this can happen often, and the count can be very large. The algorithm's time complexity does not change (still O(n^2 * m)), but the output size can be huge. For a 10,000 x 10,000 matrix, O(n^2 * m) = O(10^12) which is too slow. At that scale, you would need approximate methods, sampling, or problem-specific constraints (like sparsity) to make it tractable. This is a good moment in an interview to push back and ask: "What are the actual dimensions and value ranges? That determines whether an exact algorithm is feasible."
  </Accordion>

  <Accordion title="You have an array where elements are frequently updated, and you also need to answer range sum queries. Prefix sums rebuild in O(n) per update. What data structure would you use instead, and why?">
    **What interviewers are really testing:** Whether you know the limitations of prefix sums and can articulate when to graduate to more advanced data structures like Fenwick Trees or Segment Trees. This is a classic "when does your tool break?" question.

    **Strong Answer:**
    - Prefix sums are optimal for the static case: O(n) build, O(1) query, but O(n) per update (you must rebuild the entire prefix array). When updates are frequent, this becomes a bottleneck.
    - **Fenwick Tree (Binary Indexed Tree):** Supports both point updates and prefix sum queries in O(log n) each. The idea is to store partial sums in a tree structure indexed by the binary representation of indices. It is compact (uses an array of size n), has small constant factors, and is straightforward to implement in about 15 lines of code. Use this when you need single-element updates and range sum queries.
    - **Segment Tree:** Supports range updates and range queries in O(log n). More general than Fenwick Tree -- it can handle not just sums but min, max, GCD, and any associative operation. It also supports lazy propagation for range updates (like "add 5 to all elements from index 3 to index 7"). The downside: it uses 4n space and is more complex to implement.
    - **Decision rule:** If the problem only needs point updates and prefix/range sums, use a Fenwick Tree -- it is simpler, faster in practice, and uses less memory. If the problem needs range updates, range queries with non-invertible operations (like min/max), or more complex aggregation, use a Segment Tree.
    - In the real world, databases use B-trees (which are segment-tree-like) for indexed range queries, and systems like Apache Druid use pre-aggregated rollups (which are prefix-sum-like) for analytics workloads where data is append-only.

    **Follow-up:** Can you explain how a Fenwick Tree achieves O(log n) for both update and query? What is the role of the lowest set bit?

    **Follow-up Answer:** A Fenwick Tree stores partial sums where each index `i` is responsible for a range of elements determined by its lowest set bit. The value at index `i` stores the sum of `lowbit(i)` elements ending at index `i`, where `lowbit(i) = i & (-i)`. To query the prefix sum up to index `i`, you add `tree[i]` and then move to `i - lowbit(i)`, repeating until `i = 0`. Each step clears the lowest set bit, so there are at most `log n` steps. To update index `i`, you add the delta to `tree[i]` and then move to `i + lowbit(i)`, repeating until `i > n`. Each step sets the next higher bit, again at most `log n` steps. The beauty is that the entire structure is just an array with arithmetic index manipulation -- no pointers, no nodes, no allocations.
  </Accordion>

  <Accordion title="Explain the difference array technique. How does it relate to prefix sums, and when would you use it instead?">
    **What interviewers are really testing:** Whether you understand that prefix sums and difference arrays are inverses of each other, and whether you can apply the right one based on the access pattern. This is a nuanced optimization question.

    **Strong Answer:**
    - A difference array is the inverse of a prefix sum. If you have an array `arr`, its difference array `diff` is defined as `diff[0] = arr[0]` and `diff[i] = arr[i] - arr[i-1]` for `i > 0`. Applying a prefix sum to the difference array recovers the original array. They are inverse operations: `prefixSum(differenceArray(arr)) == arr`.
    - The difference array is powerful for range update problems. If you need to add a value `val` to every element from index `l` to index `r`, a brute-force approach takes O(r - l + 1). With a difference array, you do `diff[l] += val` and `diff[r+1] -= val` -- two O(1) operations. After all range updates are applied, compute the prefix sum of the difference array to get the final values.
    - **Use prefix sum** when: many range sum queries, few updates. O(n) build, O(1) query, O(n) update.
    - **Use difference array** when: many range updates, then one final read of all values. O(1) per range update, O(n) to reconstruct.
    - Real-world example: flight booking problem. You have n flights and a list of bookings like "book 10 seats on flights 2 through 5." Apply each booking as a range update on the difference array, then prefix sum to get the total seats booked per flight. This is O(n + m) where m is the number of bookings, versus O(n * m) for the brute force.

    **Follow-up:** Can you combine the difference array with a Fenwick Tree to support both range updates and range queries efficiently?

    **Follow-up Answer:** Yes, and this is a powerful technique. A standard Fenwick Tree supports point updates and range queries. If you apply the difference array concept, a range update `add(l, r, val)` becomes two point updates on the Fenwick Tree: `update(l, val)` and `update(r+1, -val)`. A point query `query(i)` becomes a prefix sum query on the Fenwick Tree. This gives O(log n) range updates and O(log n) point queries. For range updates AND range queries simultaneously, you need two Fenwick Trees with a more involved formula, but the idea extends naturally. This combination appears in competitive programming and in systems that need to efficiently apply batch updates and then answer aggregate queries.
  </Accordion>
</AccordionGroup>

## Interview Q&A: Senior-Level Format

The questions below mirror real senior interviews: a primary question, a real LeetCode example with full walkthrough, three senior follow-ups in `<Note>` blocks, and the wrong answers that lose points.

<AccordionGroup>
  <Accordion title="Subarray Sum Equals K (LC 560): walk me through your approach, including why you initialize the HashMap with {0: 1}.">
    **Strong Answer Framework:**

    1. **Restate the problem.** "Count contiguous subarrays summing to K, with values that can be negative."
    2. **Eliminate sliding window.** "Negatives mean sliding window's monotonicity breaks. Sliding window only works for positive-valued sum problems."
    3. **State the algebraic insight.** "If `prefix[j] - prefix[i] == k`, then `nums[i..j-1]` sums to k. So for each j, I want to know how many earlier prefix sums equal `prefix[j] - k`."
    4. **Use a HashMap of prefix sums to frequencies.** "Each new prefix_sum, lookup `prefix_sum - k` in the map, add the frequency to the count."
    5. **Seed the map with `{0: 1}`.** "This represents the empty prefix. Without it, every subarray starting at index 0 with sum k is missed."
    6. **State complexity.** "O(n) time, O(n) space."

    **Real LeetCode Example:**

    `nums = [3, 4, 7, 2, -3, 1, 4, 2]`, `k = 7`.

    | i | nums[i] | prefix_sum | prefix_sum - k | seen before? | count | seen[prefix_sum] after |
    |---|---|---|---|---|---|---|
    | start | - | 0 | - | seed | 0 | {0:1} |
    | 0 | 3 | 3 | -4 | no | 0 | {0:1, 3:1} |
    | 1 | 4 | 7 | 0 | yes (1) | 1 | {0:1, 3:1, 7:1} |
    | 2 | 7 | 14 | 7 | yes (1) | 2 | {..., 14:1} |
    | 3 | 2 | 16 | 9 | no | 2 | {..., 16:1} |
    | 4 | -3 | 13 | 6 | no | 2 | {..., 13:1} |
    | 5 | 1 | 14 | 7 | yes (1) | 3 | {..., 14:2} |
    | 6 | 4 | 18 | 11 | no | 3 | {..., 18:1} |
    | 7 | 2 | 20 | 13 | yes (1) | 4 | {..., 20:1} |

    Final count = 4. Verify: subarrays summing to 7 are `[3,4]`, `[7]`, `[7,2,-3,1]`, `[1,4,2]` -- four total.

    <Note>
    **Senior Follow-up 1:** What if the problem asked for the *longest* subarray with sum k instead of the count? (Store the *first occurrence* of each prefix sum in the map, not the frequency. For each new prefix_sum, if `prefix_sum - k` is in the map, the candidate length is `i - first_index[prefix_sum - k]`. Track the max.)
    </Note>

    <Note>
    **Senior Follow-up 2:** What if values were modular -- find subarrays whose sum is divisible by k? (Key by remainder, not raw sum. `(prefix_sum % k + k) % k` to normalize negatives. Same HashMap approach, but the lookup is for matching remainders, not `prefix_sum - k`.)
    </Note>

    <Note>
    **Senior Follow-up 3:** How does this scale if the array is a stream, and you only need an approximate count? (You cannot store every prefix sum forever -- memory grows linearly. Use Count-Min Sketch or similar probabilistic structures for approximate frequency. Real-world analytics systems like Apache Druid use exactly this trick.)
    </Note>

    **Common Wrong Answers:**

    - "I would use sliding window." Wrong because of negatives. Strong red flag for senior-level.
    - "I would store indices, not frequencies." Misses multiple subarrays ending at the same point. Returns wrong count.
    - "I do not need the `{0: 1}` seed." Misses every subarray that starts at index 0 with sum k. Off by exactly the count of such subarrays.

    **Further Reading:**
    - LC 560 (Subarray Sum Equals K), LC 974 (Subarray Sums Divisible by K), LC 525 (Contiguous Array).
    - "The Algorithms Book" -- chapter on prefix sums and their HashMap composition.
  </Accordion>

  <Accordion title="Range Sum Query Immutable (LC 303): why do we add a leading zero, and what bug does it prevent?">
    **Strong Answer Framework:**

    1. **State the formula.** "For a range `[l, r]`, the sum is `prefix[r+1] - prefix[l]`. The leading zero makes this formula uniform."
    2. **Show the alternative without the seed.** "Without it, querying a range starting at l=0 becomes a special case requiring an `if l == 0` branch."
    3. **Mathematical interpretation.** "The leading 0 represents the sum of the empty prefix -- zero elements summed equals zero."
    4. **Practical benefit.** "It eliminates an entire class of off-by-one bugs that bite under interview pressure."
    5. **State complexity.** "O(n) build, O(1) per query, O(n) space."

    **Real LeetCode Example:**

    `nums = [1, 2, 3, 4, 5]`. Build prefix `[0, 1, 3, 6, 10, 15]` (length n+1).

    Query `sumRange(0, 2)` = sum of `nums[0..2]` = 1 + 2 + 3 = 6.
    Formula: `prefix[3] - prefix[0]` = 6 - 0 = 6. Works.

    Without the leading 0, prefix would be `[1, 3, 6, 10, 15]`. Query `sumRange(0, 2)`: `prefix[2] - prefix[-1]`? Index error or special case. The leading 0 just sidesteps this entirely.

    <Note>
    **Senior Follow-up 1:** What if the array can be updated between queries? (Prefix sum requires O(n) rebuild per update. Switch to Fenwick Tree (O(log n) update + O(log n) query) or Segment Tree.)
    </Note>

    <Note>
    **Senior Follow-up 2:** What if you need range minimum instead of range sum? (Min has no inverse, so prefix arrays cannot work. Use a Sparse Table for O(1) query if static, or a Segment Tree if mutable.)
    </Note>

    <Note>
    **Senior Follow-up 3:** What if you need to answer Q queries of a million on n = 10 to the 5? (Q * O(1) per query = 10 to the 6 ops -- trivial. The build cost O(n) dominates. This is exactly the case prefix sums were designed for: many queries on a static array.)
    </Note>

    **Common Wrong Answers:**

    - "I just sum the range each time." O(n) per query, fails when Q is large.
    - "I would use a Segment Tree." Overkill for the static case. Senior interviewers want minimum tool for the job.
    - "I do not need the leading 0." Then you have a special case for `l = 0` and risk off-by-one bugs.

    **Further Reading:**
    - LC 303, LC 304 (2D version), LC 307 (mutable -> Fenwick Tree).
    - "Competitive Programmer's Handbook" by Antti Laaksonen, prefix sum chapter.
  </Accordion>

  <Accordion title="Submatrices that sum to target (LC 1074): how do you compose 2D Prefix Sum with the Subarray Sum Equals K trick?">
    **Strong Answer Framework:**

    1. **Recognize the composition.** "This is LC 560 in 2D. I cannot enumerate all O(m squared * n squared) submatrices, but I can collapse one dimension."
    2. **Fix two row boundaries.** "For each pair of rows (r1, r2), compute the sum of each column between those rows. This produces a 1D array of column sums."
    3. **Run LC 560 on the 1D array.** "Count subarrays of the 1D column-sum array summing to target. Each such subarray corresponds to a submatrix with rows in [r1, r2]."
    4. **Sum over all row pairs.** "O(m squared) row pairs * O(n) inner = O(m squared * n) total."
    5. **Pick the smaller dimension as outer.** "If n is less than m, fix column pairs instead and iterate row-wise. Always O(min(m, n) squared * max(m, n))."

    **Real LeetCode Example:**

    `matrix = [[0,1,0],[1,1,1],[0,1,0]]`, `target = 0`.

    For row pair (0, 0): column sums = `[0, 1, 0]`. Subarrays summing to 0: `[0]` at index 0, `[0]` at index 2 -- 2 submatrices.

    For row pair (0, 1): column sums = `[1, 2, 1]`. Subarrays summing to 0: none.

    For row pair (0, 2): column sums = `[1, 3, 1]`. Subarrays summing to 0: none.

    For row pair (1, 1): column sums = `[1, 1, 1]`. Subarrays summing to 0: none.

    For row pair (1, 2): column sums = `[1, 2, 1]`. Subarrays summing to 0: none.

    For row pair (2, 2): column sums = `[0, 1, 0]`. Subarrays summing to 0: 2.

    Total = 4.

    <Note>
    **Senior Follow-up 1:** What if target = 0 and matrix is large (10000 by 10000)? (O(m squared * n) approaches 10 to the 12, infeasible. You would need approximate methods, sparsity exploitation, or different problem framing. Always ask the dimensions.)
    </Note>

    <Note>
    **Senior Follow-up 2:** Could you avoid recomputing column sums for each row pair? (Yes -- maintain incremental column sums. When extending r2 by one, add `matrix[r2]` to each column sum. Saves a factor of m on the inner loop.)
    </Note>

    <Note>
    **Senior Follow-up 3:** What if you needed the *area* (count of cells) of submatrices summing to target, not the count? (Trivially derivable: each submatrix's area is `(r2 - r1 + 1) * (j - i + 1)` where (i, j) are the column-sum subarray boundaries. Sum these.)
    </Note>

    **Common Wrong Answers:**

    - "I would use 4D enumeration with 2D prefix sum for O(1) per submatrix." Still O(m squared * n squared) total -- too slow.
    - "I would use 2D segment tree." Overkill and slower in practice. The collapse trick is the canonical solution.
    - "I do not see the LC 560 connection." Pattern-matching failure. The whole point is recognizing 1D primitives in 2D problems.

    **Further Reading:**
    - LC 1074 (Number of Submatrices That Sum to Target), LC 363 (Max Sum Rectangle No Larger Than K).
    - "Programming Competitive: A Quick Reference" -- 2D prefix sum techniques.
  </Accordion>

  <Accordion title="Difference array vs Prefix Sum: when would you use a difference array, and how do they relate?">
    **Strong Answer Framework:**

    1. **State the inverse relationship.** "Prefix sum and difference array are inverses. `prefixSum(differenceArray(arr)) == arr`. Pick based on access pattern."
    2. **Prefix sum: many queries, few updates.** "O(n) build, O(1) query, O(n) update -- bad for frequent updates."
    3. **Difference array: many range updates, one final read.** "O(1) per range update via `diff[l] += val; diff[r+1] -= val`, then O(n) to reconstruct."
    4. **Combined with Fenwick Tree.** "For range updates + point queries, build Fenwick Tree on the difference array. Each range update is two Fenwick point updates; each query is a Fenwick prefix sum -- both O(log n)."
    5. **Real-world example.** "Flight bookings (LC 1109): m bookings, each updates a range of seats. Difference array gives O(n + m); naive iteration gives O(n * m)."

    **Real LeetCode Example:**

    LC 1109 -- Corporate Flight Bookings. `bookings = [[1,2,10], [2,3,20], [2,5,25]]`, `n = 5`.

    Difference array: start with `[0, 0, 0, 0, 0]`.
    - Booking (1, 2, 10): diff[0] += 10, diff[2] -= 10 -> `[10, 0, -10, 0, 0]`.
    - Booking (2, 3, 20): diff[1] += 20, diff[3] -= 20 -> `[10, 20, -10, -20, 0]`.
    - Booking (2, 5, 25): diff[1] += 25, diff[5] -= 25 -- indices 0..4 are valid, so just `[10, 45, -10, -20, 0]`.

    Prefix sum the diff array: `[10, 55, 45, 25, 25]`. That is the answer.

    <Note>
    **Senior Follow-up 1:** What if the bookings come as a stream and you need running totals after each? (You cannot batch updates anymore. Switch to Fenwick Tree on the diff array. Range update is O(log n); query at any time is O(log n).)
    </Note>

    <Note>
    **Senior Follow-up 2:** Can you support range updates AND range queries simultaneously? (Yes, using two Fenwick Trees with a clever formula. The formal name is "BIT with range updates and range queries" -- standard competitive programming technique.)
    </Note>

    <Note>
    **Senior Follow-up 3:** What if updates are not additive (e.g., set, multiply)? (Difference array tricks rely on additive composition. For non-additive operations like "set range to value", you need a Segment Tree with lazy propagation, not a difference array.)
    </Note>

    **Common Wrong Answers:**

    - "I would loop over each booking and update each cell." O(n * m), fails at scale.
    - "Just use prefix sum." Prefix sum works for queries, not for updates. Update on prefix sum is O(n) per update.
    - "Use Segment Tree." Not wrong, but overkill for this offline batch case. Difference array is simpler and equally fast.

    **Further Reading:**
    - LC 1109 (Corporate Flight Bookings), LC 1854 (Maximum Population Year).
    - "Competitive Programming 4" by Halim, chapter on range update / range query data structures.
  </Accordion>
