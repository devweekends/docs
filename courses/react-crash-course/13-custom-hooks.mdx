# Custom Hooks & Advanced Patterns

Custom hooks are the most powerful abstraction in React. They let you extract and reuse stateful logic across components while keeping code clean and testable.

**Real-world analogy**: If components are like recipes, custom hooks are like cooking techniques. "Saute" is a technique you can use in any recipe -- you do not reinvent it each time. Similarly, a custom hook like `useLocalStorage` encapsulates the *technique* of persisting state to localStorage so any component can use it without duplicating the logic.

## What is a Custom Hook?

A custom hook is a JavaScript function that:
1. Starts with `use` (e.g., `useLocalStorage`) -- this naming convention tells React's linter to enforce the Rules of Hooks
2. Can call other hooks inside it (`useState`, `useEffect`, other custom hooks)
3. Returns values (state, functions, etc.) for components to use

Custom hooks do not share state between components that call them -- each call gets its own independent copy. They share *logic*, not *data*.

```javascript
// Custom hook
function useCounter(initialValue = 0) {
  const [count, setCount] = useState(initialValue);
  
  const increment = () => setCount(c => c + 1);
  const decrement = () => setCount(c => c - 1);
  const reset = () => setCount(initialValue);
  
  return { count, increment, decrement, reset };
}

// Usage in components
function Counter() {
  const { count, increment, decrement, reset } = useCounter(10);
  
  return (
    <div>
      <p>Count: {count}</p>
      <button onClick={increment}>+</button>
      <button onClick={decrement}>-</button>
      <button onClick={reset}>Reset</button>
    </div>
  );
}
```

## useRef Hook

`useRef` provides a mutable reference that persists across renders without causing re-renders.

### DOM Element References

```javascript
function FocusInput() {
  const inputRef = useRef(null);

  const focusInput = () => {
    inputRef.current.focus();
  };

  return (
    <div>
      <input ref={inputRef} type="text" />
      <button onClick={focusInput}>Focus Input</button>
    </div>
  );
}
```

### Storing Previous Values

This is one of the most elegant uses of `useRef`. The trick: `useEffect` runs *after* the render completes, so during the current render `ref.current` still holds the value from the *previous* render. After this render commits to the DOM, the effect updates `ref.current` to the new value -- ready for the next render.

```javascript
function usePrevious(value) {
  const ref = useRef();
  
  // This runs AFTER render, so during the current render,
  // ref.current still holds the old value. After the render
  // commits, it updates to the current value.
  useEffect(() => {
    ref.current = value;
  }, [value]);
  
  // Returns the value from the previous render
  return ref.current;
}

// Usage
function Counter() {
  const [count, setCount] = useState(0);
  const previousCount = usePrevious(count);

  return (
    <div>
      <p>Current: {count}</p>
      <p>Previous: {previousCount}</p>
    </div>
  );
}
```

### Storing Instance Variables

`useRef` is the React equivalent of a class's instance variable. It persists between renders but does not trigger re-renders when you change it. This makes it perfect for storing timer IDs, WebSocket connections, previous values, and anything you need to "remember" without re-rendering.

```javascript
function Timer() {
  const [seconds, setSeconds] = useState(0);
  // intervalRef.current holds the interval ID between renders.
  // We cannot use a regular variable because it would be reset
  // to null on every render. We cannot use useState because
  // changing a timer ID should not trigger a re-render.
  const intervalRef = useRef(null);

  const start = () => {
    // Guard: prevent starting multiple intervals
    if (intervalRef.current) return;
    
    intervalRef.current = setInterval(() => {
      // Functional update avoids a stale closure over `seconds`
      setSeconds(s => s + 1);
    }, 1000);
  };

  const stop = () => {
    clearInterval(intervalRef.current);
    intervalRef.current = null;
  };

  const reset = () => {
    stop();
    setSeconds(0);
  };

  // Cleanup on unmount
  useEffect(() => {
    return () => clearInterval(intervalRef.current);
  }, []);

  return (
    <div>
      <p>Seconds: {seconds}</p>
      <button onClick={start}>Start</button>
      <button onClick={stop}>Stop</button>
      <button onClick={reset}>Reset</button>
    </div>
  );
}
```

<Note>
**useRef vs useState**:
- `useState`: Changes trigger re-renders
- `useRef`: Changes do NOT trigger re-renders

Use `useRef` when you need to track something but don't want to re-render when it changes.
</Note>

## useReducer Hook

`useReducer` is an alternative to `useState` for complex state logic. Use it when your state has multiple sub-values, or when the next state depends on the previous one in non-trivial ways.

**When to reach for useReducer**: If you find yourself with three or more `useState` calls that always change together, or if your state update logic has `if/else` branches that are getting tangled, `useReducer` gives you a centralized place to reason about all state transitions. It also works beautifully with Context to build a mini-Redux for a specific feature.

### Basic Syntax

```javascript
const [state, dispatch] = useReducer(reducer, initialState);
```

### Counter Example

```javascript
const initialState = { count: 0 };

function reducer(state, action) {
  switch (action.type) {
    case 'increment':
      return { count: state.count + 1 };
    case 'decrement':
      return { count: state.count - 1 };
    case 'reset':
      return initialState;
    case 'set':
      return { count: action.payload };
    default:
      throw new Error(`Unknown action: ${action.type}`);
  }
}

function Counter() {
  const [state, dispatch] = useReducer(reducer, initialState);

  return (
    <div>
      <p>Count: {state.count}</p>
      <button onClick={() => dispatch({ type: 'increment' })}>+</button>
      <button onClick={() => dispatch({ type: 'decrement' })}>-</button>
      <button onClick={() => dispatch({ type: 'reset' })}>Reset</button>
      <button onClick={() => dispatch({ type: 'set', payload: 100 })}>Set to 100</button>
    </div>
  );
}
```

### Complex State Example

```javascript
const initialState = {
  items: [],
  loading: false,
  error: null,
  filter: 'all'
};

function todoReducer(state, action) {
  switch (action.type) {
    case 'FETCH_START':
      return { ...state, loading: true, error: null };
    
    case 'FETCH_SUCCESS':
      return { ...state, loading: false, items: action.payload };
    
    case 'FETCH_ERROR':
      return { ...state, loading: false, error: action.payload };
    
    case 'ADD_ITEM':
      return {
        ...state,
        items: [...state.items, { id: Date.now(), ...action.payload, done: false }]
      };
    
    case 'TOGGLE_ITEM':
      return {
        ...state,
        items: state.items.map(item =>
          item.id === action.payload
            ? { ...item, done: !item.done }
            : item
        )
      };
    
    case 'DELETE_ITEM':
      return {
        ...state,
        items: state.items.filter(item => item.id !== action.payload)
      };
    
    case 'SET_FILTER':
      return { ...state, filter: action.payload };
    
    default:
      return state;
  }
}
```

### When to Use useReducer

| useState | useReducer |
|----------|------------|
| Simple state (primitives) | Complex state (objects) |
| Few update patterns | Many update patterns |
| Updates in one place | Updates from many places |
| Related state pieces | Independent state logic |

## Building Custom Hooks

### useLocalStorage

```javascript
function useLocalStorage(key, initialValue) {
  // Lazy initialization
  const [storedValue, setStoredValue] = useState(() => {
    try {
      const item = window.localStorage.getItem(key);
      return item ? JSON.parse(item) : initialValue;
    } catch (error) {
      console.error(error);
      return initialValue;
    }
  });

  const setValue = (value) => {
    try {
      // Allow value to be a function
      const valueToStore = value instanceof Function ? value(storedValue) : value;
      setStoredValue(valueToStore);
      window.localStorage.setItem(key, JSON.stringify(valueToStore));
    } catch (error) {
      console.error(error);
    }
  };

  return [storedValue, setValue];
}

// Usage
function Settings() {
  const [theme, setTheme] = useLocalStorage('theme', 'light');
  const [language, setLanguage] = useLocalStorage('language', 'en');
  
  return (/* ... */);
}
```

### useFetch

```javascript
function useFetch(url) {
  const [data, setData] = useState(null);
  const [loading, setLoading] = useState(true);
  const [error, setError] = useState(null);

  useEffect(() => {
    const controller = new AbortController();

    const fetchData = async () => {
      try {
        setLoading(true);
        setError(null);
        
        const response = await fetch(url, { signal: controller.signal });
        
        if (!response.ok) {
          throw new Error(`HTTP error! status: ${response.status}`);
        }
        
        const json = await response.json();
        setData(json);
      } catch (err) {
        if (err.name !== 'AbortError') {
          setError(err.message);
        }
      } finally {
        setLoading(false);
      }
    };

    fetchData();

    return () => controller.abort();
  }, [url]);

  return { data, loading, error };
}

// Usage
function UserProfile({ userId }) {
  const { data: user, loading, error } = useFetch(`/api/users/${userId}`);

  if (loading) return <Spinner />;
  if (error) return <Error message={error} />;
  
  return <div>{user.name}</div>;
}
```

### useToggle

```javascript
function useToggle(initialValue = false) {
  const [value, setValue] = useState(initialValue);

  const toggle = useCallback(() => setValue(v => !v), []);
  const setTrue = useCallback(() => setValue(true), []);
  const setFalse = useCallback(() => setValue(false), []);

  return { value, toggle, setTrue, setFalse };
}

// Usage
function Modal() {
  const { value: isOpen, toggle, setFalse: close } = useToggle();

  return (
    <>
      <button onClick={toggle}>Toggle Modal</button>
      {isOpen && (
        <div className="modal">
          <button onClick={close}>Close</button>
        </div>
      )}
    </>
  );
}
```

### useDebounce

```javascript
function useDebounce(value, delay) {
  const [debouncedValue, setDebouncedValue] = useState(value);

  useEffect(() => {
    const timer = setTimeout(() => {
      setDebouncedValue(value);
    }, delay);

    return () => clearTimeout(timer);
  }, [value, delay]);

  return debouncedValue;
}

// Usage
function SearchInput() {
  const [query, setQuery] = useState('');
  const debouncedQuery = useDebounce(query, 300);
  const { data } = useFetch(`/api/search?q=${debouncedQuery}`);
  
  return (/* ... */);
}
```

### useOnClickOutside

```javascript
function useOnClickOutside(ref, handler) {
  useEffect(() => {
    const listener = (event) => {
      if (!ref.current || ref.current.contains(event.target)) {
        return;
      }
      handler(event);
    };

    document.addEventListener('mousedown', listener);
    document.addEventListener('touchstart', listener);

    return () => {
      document.removeEventListener('mousedown', listener);
      document.removeEventListener('touchstart', listener);
    };
  }, [ref, handler]);
}

// Usage
function Dropdown() {
  const [isOpen, setIsOpen] = useState(false);
  const dropdownRef = useRef(null);

  useOnClickOutside(dropdownRef, () => setIsOpen(false));

  return (
    <div ref={dropdownRef}>
      <button onClick={() => setIsOpen(!isOpen)}>Toggle</button>
      {isOpen && <div className="dropdown-menu">...</div>}
    </div>
  );
}
```

### useMediaQuery

```javascript
function useMediaQuery(query) {
  const [matches, setMatches] = useState(() => 
    window.matchMedia(query).matches
  );

  useEffect(() => {
    const mediaQuery = window.matchMedia(query);
    
    const handler = (event) => setMatches(event.matches);
    
    mediaQuery.addEventListener('change', handler);
    return () => mediaQuery.removeEventListener('change', handler);
  }, [query]);

  return matches;
}

// Usage
function ResponsiveComponent() {
  const isMobile = useMediaQuery('(max-width: 768px)');
  const isTablet = useMediaQuery('(min-width: 769px) and (max-width: 1024px)');
  const prefersDark = useMediaQuery('(prefers-color-scheme: dark)');

  return (
    <div className={isMobile ? 'mobile-layout' : 'desktop-layout'}>
      {isMobile ? <MobileNav /> : <DesktopNav />}
    </div>
  );
}
```

### useAsync

```javascript
function useAsync(asyncFunction, immediate = true) {
  const [status, setStatus] = useState('idle');
  const [data, setData] = useState(null);
  const [error, setError] = useState(null);

  const execute = useCallback(async (...args) => {
    setStatus('pending');
    setData(null);
    setError(null);

    try {
      const result = await asyncFunction(...args);
      setData(result);
      setStatus('success');
      return result;
    } catch (err) {
      setError(err);
      setStatus('error');
      throw err;
    }
  }, [asyncFunction]);

  useEffect(() => {
    if (immediate) {
      execute();
    }
  }, [execute, immediate]);

  return { execute, status, data, error, isLoading: status === 'pending' };
}

// Usage
function UserList() {
  const { data, error, isLoading, execute: refetch } = useAsync(
    () => fetch('/api/users').then(r => r.json())
  );

  return (
    <div>
      <button onClick={refetch}>Refresh</button>
      {isLoading && <Spinner />}
      {error && <Error message={error.message} />}
      {data && data.map(user => <UserCard key={user.id} user={user} />)}
    </div>
  );
}
```

## Custom Hook Pitfalls

<Warning>
**Pitfall 1 -- Returning a new object/array reference every render**: If your custom hook returns an object literal, every component that calls the hook gets a new reference on every render. This defeats `React.memo` on child components.

```javascript
// BAD: new object on every render
function useUser() {
  const [name, setName] = useState('');
  return { name, setName }; // New object every time
}

// BETTER: use useMemo to stabilize the reference (only when needed)
function useUser() {
  const [name, setName] = useState('');
  return useMemo(() => ({ name, setName }), [name]);
}
```

**Pitfall 2 -- Stale closures in hook callbacks**: If your custom hook creates a callback that references state, make sure the callback captures the latest value. The `useRef` "latest value" pattern is your best friend here:

```javascript
function useInterval(callback, delay) {
  // Store the latest callback in a ref so the interval
  // always calls the current version, not a stale closure.
  const savedCallback = useRef(callback);
  useEffect(() => { savedCallback.current = callback; }, [callback]);

  useEffect(() => {
    if (delay === null) return;
    const id = setInterval(() => savedCallback.current(), delay);
    return () => clearInterval(id);
  }, [delay]);
}
```

**Pitfall 3 -- Breaking the Rules of Hooks inside custom hooks**: Custom hooks follow the same rules as built-in hooks -- they cannot be called conditionally or inside loops. A common mistake is trying to call `useState` inside an `if` block within a custom hook. The linter will catch this if you name your hook with the `use` prefix.

**Pitfall 4 -- Forgetting that each call gets its own state**: If two components call `useCounter()`, they get independent counters. Custom hooks share *logic*, not *data*. If you need shared data, combine the hook with Context.
</Warning>

## Advanced Patterns

### Compound Hooks

Combine multiple hooks for complex functionality:

```javascript
function useForm(initialValues, validationRules) {
  const [values, setValues] = useState(initialValues);
  const [errors, setErrors] = useState({});
  const [touched, setTouched] = useState({});
  const [isSubmitting, setIsSubmitting] = useState(false);

  const validate = useCallback((fieldName, value) => {
    const rules = validationRules[fieldName];
    if (!rules) return '';

    for (const rule of rules) {
      const error = rule(value, values);
      if (error) return error;
    }
    return '';
  }, [validationRules, values]);

  const handleChange = useCallback((e) => {
    const { name, value } = e.target;
    setValues(prev => ({ ...prev, [name]: value }));
    
    if (touched[name]) {
      setErrors(prev => ({ ...prev, [name]: validate(name, value) }));
    }
  }, [touched, validate]);

  const handleBlur = useCallback((e) => {
    const { name, value } = e.target;
    setTouched(prev => ({ ...prev, [name]: true }));
    setErrors(prev => ({ ...prev, [name]: validate(name, value) }));
  }, [validate]);

  const handleSubmit = useCallback((onSubmit) => async (e) => {
    e.preventDefault();
    
    // Validate all fields
    const newErrors = {};
    let hasErrors = false;
    
    for (const [name, value] of Object.entries(values)) {
      const error = validate(name, value);
      if (error) {
        newErrors[name] = error;
        hasErrors = true;
      }
    }
    
    setErrors(newErrors);
    setTouched(Object.keys(values).reduce((acc, key) => ({ ...acc, [key]: true }), {}));
    
    if (!hasErrors) {
      setIsSubmitting(true);
      try {
        await onSubmit(values);
      } finally {
        setIsSubmitting(false);
      }
    }
  }, [values, validate]);

  const reset = useCallback(() => {
    setValues(initialValues);
    setErrors({});
    setTouched({});
  }, [initialValues]);

  return {
    values,
    errors,
    touched,
    isSubmitting,
    handleChange,
    handleBlur,
    handleSubmit,
    reset,
    setValues
  };
}

// Validation rules
const required = (value) => !value ? 'This field is required' : '';
const email = (value) => !/\S+@\S+\.\S+/.test(value) ? 'Invalid email' : '';
const minLength = (min) => (value) => 
  value.length < min ? `Must be at least ${min} characters` : '';

// Usage
function SignupForm() {
  const form = useForm(
    { email: '', password: '' },
    {
      email: [required, email],
      password: [required, minLength(8)]
    }
  );

  const onSubmit = async (values) => {
    await api.signup(values);
  };

  return (
    <form onSubmit={form.handleSubmit(onSubmit)}>
      <input
        name="email"
        value={form.values.email}
        onChange={form.handleChange}
        onBlur={form.handleBlur}
      />
      {form.touched.email && form.errors.email && (
        <span>{form.errors.email}</span>
      )}
      {/* ... */}
    </form>
  );
}
```

```javascript
function useInterval(callback, delay) {
  const savedCallback = useRef(callback);

  // Remember the latest callback
  useEffect(() => {
    savedCallback.current = callback;
  }, [callback]);

  // Set up the interval
  useEffect(() => {
    if (delay === null) return;
    
    const tick = () => savedCallback.current();
    const id = setInterval(tick, delay);
    
    return () => clearInterval(id);
  }, [delay]);
}

// Usage
function Timer() {

  );

  return (
    <div>
      <p>Count: {count}</p>
      <button onClick={() => setIsRunning(!isRunning)}>
        {isRunning ? 'Pause' : 'Resume'}
```

```javascript
function useCopyToClipboard() {
  const [copiedText, setCopiedText] = useState(null);
  const [error, setError] = useState(null);

  const copy = useCallback(async (text) => {
    if (!navigator?.clipboard) {
      setError('Clipboard not supported');
      return false;
    }

    try {
      await navigator.clipboard.writeText(text);
      setCopiedText(text);
      setError(null);
      return true;
    } catch (err) {
      setError(err.message);
      setCopiedText(null);
      return false;
    }
  }, []);

  const reset = useCallback(() => {
    setCopiedText(null);
    setError(null);
  }, []);

  return { copiedText, error, copy, reset };
}

// Usage
function CopyButton({ text }) {
  const { copiedText, copy } = useCopyToClipboard();
  const isCopied = copiedText === text;

  return (
    <button onClick={() => copy(text)}>
      {isCopied ? '✓ Copied!' : 'Copy'}
    </button>
  );
}
```

<AccordionGroup>
<Accordion title="What are the Rules of Hooks, why do they exist, and what would break if you called a hook inside a conditional?">
**Strong Answer:**
The two Rules of Hooks are: only call hooks at the top level (not inside loops, conditions, or nested functions), and only call hooks from React function components or custom hooks.

These rules exist because of how React tracks hook state internally. React does not identify hooks by name -- it identifies them by call order. On every render, React maintains an array of hook states. The first `useState` call reads from index 0, the second from index 1, the third `useEffect` from index 2, and so on. React relies on this order being identical across renders to match each hook call with its stored state.

If you wrap a `useState` in a conditional (`if (loggedIn) { const [name, setName] = useState('') }`), the first render might call three hooks (indices 0, 1, 2), but the second render might call only two (indices 0, 1) because the condition changed. React tries to read index 2 for a hook that no longer exists, or worse, matches the wrong hook state to the wrong hook call. This causes corrupted state, cryptic errors, and unpredictable behavior.

The `eslint-plugin-react-hooks` enforces these rules statically. In practice, the constraint is rarely limiting: if you need conditional logic, put the condition inside the hook (like `useEffect(() => { if (condition) { ... } }, [condition])`) rather than around it.

The deeper reason this design was chosen: it avoids the need for a unique string key per hook (which would be verbose and error-prone) and enables hooks to be fully composable. A custom hook can call any number of hooks internally, and React tracks them all by position, making hook composition zero-overhead.

**Follow-up: How does the `use` hook in React 19 change these rules, and why is it special?**

The `use` hook is the first hook that CAN be called conditionally. It is designed for reading resources (Promises, Contexts) and can be called inside conditionals, loops, and even after early returns. This is possible because `use` does not store its own state in React's hook array -- it reads from external resources (a Promise's resolved value or a Context's current value) rather than maintaining internal state that needs positional tracking.

This makes patterns like `if (shouldFetch) { const data = use(fetchPromise) }` valid, which was previously impossible without restructuring the component. It integrates with Suspense: if the Promise is not yet resolved, `use` suspends the component. It also replaces `useContext` with a more flexible API that works inside conditionals.
</Accordion>

<Accordion title="You are asked to build a useDebounce hook. Walk through your implementation, edge cases, and how you would test it.">
**Strong Answer:**
The core implementation: the hook accepts a value and a delay, stores the debounced value in state, and uses `useEffect` to set a timeout that updates the debounced value after the delay. The cleanup function clears the timeout when the value changes before the delay expires.

```javascript
function useDebounce(value, delay) {
  const [debouncedValue, setDebouncedValue] = useState(value);
  useEffect(() => {
    const timer = setTimeout(() => setDebouncedValue(value), delay);
    return () => clearTimeout(timer);
  }, [value, delay]);
  return debouncedValue;
}
```

Edge cases: the delay could be 0 (should still debounce by one tick, useful for batching synchronous updates). The delay could change between renders (the effect re-runs with the new delay, which is correct). The value could be an object (reference changes trigger the effect even if contents are identical -- the consumer should memoize the input if this matters). The component could unmount while the timer is pending (the cleanup clears the timer, preventing a state update on an unmounted component).

Testing strategy: use `vi.useFakeTimers()` to control time. Render a component using the hook, change the input value, assert the debounced value has not changed, advance timers by the delay, assert the debounced value updated. Test rapid value changes: change value 5 times in quick succession, advance by the delay, assert only the final value appears. Test unmounting: change value, unmount before the delay, assert no errors.

A production enhancement: add a `leading` option that fires immediately on the first change and then debounces subsequent changes. This is useful for search inputs where you want instant results for the first keystroke but debounce the rest.

**Follow-up: What is the difference between debouncing and throttling, and when would you use each in a React app?**

Debouncing delays execution until a pause in activity. If events keep firing, the execution keeps being postponed. Use it for search inputs (wait until the user stops typing), form validation (validate after the user finishes a field), and window resize handlers (recalculate layout after resize ends).

Throttling ensures execution happens at most once per time interval, regardless of how many events fire. Use it for scroll handlers (update position indicator at most every 100ms), mousemove handlers (track cursor position at a controlled rate), and API polling rate limits.

The key distinction: debounce waits for silence, throttle limits frequency. For a search input, debounce is correct because you want the final query. For a scroll progress bar, throttle is correct because you want regular updates during scrolling, not just one update when scrolling stops.

<Accordion title="When should you use useRef instead of useState? Describe a scenario where using useState would cause a bug that useRef avoids.">
**Strong Answer:**
The fundamental difference: `useState` triggers a re-render when the value changes; `useRef` does not. Use `useRef` when you need to persist a value across renders without causing re-renders -- interval IDs, previous values, DOM element references, and any mutable value that the UI does not directly depend on.

The classic bug scenario: a stopwatch component that tracks elapsed time. If you store the interval ID in state (`const [intervalId, setIntervalId] = useState(null)`), setting it triggers a re-render. The re-render creates new closures, and if the interval callback reads state from the closure, it captures the value from the render when the interval was created (stale closure). Worse, if the component re-renders frequently, each render might see a different `intervalId` in state, making cleanup unreliable.

With `useRef`: `const intervalRef = useRef(null)`. Setting `intervalRef.current = setInterval(...)` does not trigger a re-render. The ref persists the same object across renders, and any code that reads `intervalRef.current` always gets the latest value because refs are mutable -- they do not capture in closures the way state does.

Another scenario: tracking whether a component is mounted. Some codebases use `const isMounted = useRef(true)` and set it to `false` in a cleanup effect, then check `if (isMounted.current)` before calling setState in async callbacks. Using useState for this would cause an infinite loop: setting the state triggers a re-render, which runs the effect, which sets the state again.

The mental model: if the value is used in JSX or needs to trigger UI updates, use state. If the value is used only in callbacks, effects, or as an implementation detail, use ref.

**Follow-up: Can you use useRef to persist state across renders AND trigger updates? What pattern enables this?**

Yes, but you need to manually trigger the re-render. The pattern is combining a ref (for the mutable value) with a forceUpdate mechanism: `const [, forceUpdate] = useReducer(x => x + 1, 0)`. Update the ref for the data, call `forceUpdate()` when you want the UI to reflect the change.

This is an advanced pattern used internally by libraries like Zustand and React Query. The external store holds the real data (like a ref), and `useSyncExternalStore` (React 18) notifies React to re-render when the data changes. The advantage over useState is that you have fine-grained control over when re-renders happen, avoiding the batching and timing constraints of setState.

For application code, I almost never use this pattern. If you need mutable data plus re-renders, useState is the right tool. The ref-plus-forceUpdate pattern is for library authors building custom subscription systems.
