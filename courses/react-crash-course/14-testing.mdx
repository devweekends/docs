# Testing React Components

Testing ensures your React application works correctly and prevents regressions. Modern React testing focuses on testing components from a user's perspective -- not how the component is implemented internally, but what the user sees and interacts with.

**Key mindset**: The more your tests resemble the way your software is used, the more confidence they give you. If your test knows that a component uses `useState` internally or checks class names on divs, it is testing implementation details. When you refactor the component, the test breaks even though the feature still works. Instead, query elements the way a user would: by visible text, form labels, and ARIA roles.

**Analogy**: Think of testing like being a restaurant health inspector. A bad inspector opens every pot, reads every recipe, and checks the brand of every spice (implementation details). A good inspector orders food from the menu, checks if it arrives hot, tastes correct, and is served on time (user-facing behavior). The restaurant can change suppliers, reorganize the kitchen, or hire new chefs -- the inspector's checklist still works because it tests *outcomes*, not *process*.

## Testing Philosophy

```
┌─────────────────────────────────────────────────────────────┐
│                    Testing Pyramid                          │
│                                                             │
│                         /\                                  │
│                        /  \                                 │
│                       / E2E \  (Cypress, Playwright)        │
│                      /──────\                               │
│                     /        \                              │
│                    /Integration\  (React Testing Library)   │
│                   /────────────\                            │
│                  /              \                           │
│                 /     Unit       \  (Jest)                  │
│                /──────────────────\                         │
│                                                             │
│          More tests at bottom, fewer at top                 │
│          Bottom = faster, Top = more realistic              │
└─────────────────────────────────────────────────────────────┘
```

<Tip>
**Testing Library Philosophy**: Test your components the way users use them. Query by roles, labels, and text — not by implementation details like class names or test IDs.
</Tip>

## Basic Component Testing

### Your First Test

```javascript
// src/components/Button.jsx
function Button({ children, onClick, disabled }) {
  return (
    <button onClick={onClick} disabled={disabled}>
      {children}
    </button>
  );
}

export default Button;
```

```javascript
// src/components/Button.test.jsx
import { render, screen } from '@testing-library/react';
import userEvent from '@testing-library/user-event';
import Button from './Button';

describe('Button', () => {
  it('renders with text', () => {
    render(<Button>Click me</Button>);
    
    expect(screen.getByRole('button', { name: /click me/i })).toBeInTheDocument();
  });

  it('calls onClick when clicked', async () => {
    const handleClick = vi.fn(); // or jest.fn() for Jest
    const user = userEvent.setup();
    
    render(<Button onClick={handleClick}>Click me</Button>);
    
    await user.click(screen.getByRole('button'));
    
    expect(handleClick).toHaveBeenCalledTimes(1);
  });

  it('is disabled when disabled prop is true', () => {
    render(<Button disabled>Click me</Button>);
    
    expect(screen.getByRole('button')).toBeDisabled();
  });

  it('does not call onClick when disabled', async () => {
    const handleClick = vi.fn();
    const user = userEvent.setup();
    
    render(<Button onClick={handleClick} disabled>Click me</Button>);
    
    await user.click(screen.getByRole('button'));
    
    expect(handleClick).not.toHaveBeenCalled();
  });
});
```

## Query Methods

React Testing Library provides several ways to query elements:

### Query Priority (Best to Worst)

| Priority | Query | When to Use |
|----------|-------|-------------|
| 1 | `getByRole` | Most accessible queries |
| 2 | `getByLabelText` | Form fields |
| 3 | `getByPlaceholderText` | Inputs without labels |
| 4 | `getByText` | Non-interactive elements |
| 5 | `getByDisplayValue` | Current form field values |
| 6 | `getByAltText` | Images |
| 7 | `getByTitle` | Title attributes |
| 8 | `getByTestId` | Last resort |

### Query Types

| Query | No Match | 1 Match | >1 Match | Async |
|-------|----------|---------|----------|-------|
| `getBy*` | Error | Return | Error | No |
| `queryBy*` | null | Return | Error | No |
| `findBy*` | Error | Return | Error | Yes |
| `getAllBy*` | Error | Array | Array | No |
| `queryAllBy*` | [] | Array | Array | No |
| `findAllBy*` | Error | Array | Array | Yes |

### Query Examples

```javascript
// ✅ Good: Queries by role (accessible)
screen.getByRole('button', { name: /submit/i });
screen.getByRole('heading', { level: 1 });
screen.getByRole('link', { name: /home/i });
screen.getByRole('checkbox', { checked: true });
screen.getByRole('textbox', { name: /email/i });

// ✅ Good: Form elements
screen.getByLabelText(/email address/i);
screen.getByPlaceholderText(/search.../i);
screen.getByDisplayValue('current value');

// ✅ Good: Text content
screen.getByText(/welcome back/i);
screen.getByText((content, element) => {
  return element.tagName === 'P' && content.includes('Hello');
});

// ⚠️ Last resort: Test ID
screen.getByTestId('custom-element');
```

<Warning>
Avoid using `getByTestId` unless absolutely necessary. It tests implementation details rather than what users see.
</Warning>

## Testing User Interactions

### Using userEvent

`userEvent` is preferred over `fireEvent` because it simulates real user behavior:

```javascript
import userEvent from '@testing-library/user-event';

describe('Form interactions', () => {
  it('handles text input', async () => {
    const user = userEvent.setup();
    render(<TextInput />);
    
    const input = screen.getByRole('textbox');
    
    await user.type(input, 'Hello World');
    
    expect(input).toHaveValue('Hello World');
  });

  it('handles keyboard navigation', async () => {
    const user = userEvent.setup();
    render(<Form />);
    
    await user.tab(); // Focus first element
    await user.keyboard('{Enter}'); // Press Enter
    await user.keyboard('{Escape}'); // Press Escape
  });

  it('handles selections', async () => {
    const user = userEvent.setup();
    render(<Select options={['A', 'B', 'C']} />);
    
    await user.selectOptions(screen.getByRole('combobox'), 'B');
    
    expect(screen.getByRole('option', { name: 'B' })).toBeSelected();
  });

  it('handles file uploads', async () => {
    const user = userEvent.setup();
    render(<FileUpload />);
    
    const file = new File(['hello'], 'hello.txt', { type: 'text/plain' });
    const input = screen.getByLabelText(/upload file/i);
    
    await user.upload(input, file);
    
    expect(input.files[0]).toBe(file);
  });

  it('handles double click', async () => {
    const user = userEvent.setup();
    const onDoubleClick = vi.fn();
    
    render(<div onDoubleClick={onDoubleClick}>Double click me</div>);
    
    await user.dblClick(screen.getByText(/double click me/i));
    
    expect(onDoubleClick).toHaveBeenCalled();
  });

  it('handles hover', async () => {
    const user = userEvent.setup();
    render(<Tooltip text="Hello" content="World" />);
    
    await user.hover(screen.getByText('Hello'));
    
    expect(screen.getByText('World')).toBeVisible();
    
    await user.unhover(screen.getByText('Hello'));
    
    expect(screen.queryByText('World')).not.toBeInTheDocument();
  });
});
```

## Testing Async Behavior

### Waiting for Elements

```javascript
describe('Async operations', () => {
  it('loads data on mount', async () => {
    render(<UserProfile userId="123" />);
    
    // Wait for loading state to disappear
    expect(screen.getByText(/loading/i)).toBeInTheDocument();
    
    // Wait for content to appear
    const userName = await screen.findByText(/john doe/i);
    expect(userName).toBeInTheDocument();
  });

  it('handles async form submission', async () => {
    const user = userEvent.setup();
    render(<ContactForm />);
    
    await user.type(screen.getByLabelText(/email/i), 'test@example.com');
    await user.type(screen.getByLabelText(/message/i), 'Hello');
    await user.click(screen.getByRole('button', { name: /send/i }));
    
    // Wait for success message
    await screen.findByText(/message sent/i);
  });
});
```

### Using waitFor

```javascript
import { waitFor } from '@testing-library/react';

it('updates after async action', async () => {
  const user = userEvent.setup();
  render(<Counter />);
  
  await user.click(screen.getByRole('button', { name: /increment/i }));
  
  // Wait for assertion to pass
  await waitFor(() => {
    expect(screen.getByText(/count: 1/i)).toBeInTheDocument();
  });
});

it('removes item after delete', async () => {
  const user = userEvent.setup();
  render(<TodoList />);
  
  await user.click(screen.getByRole('button', { name: /delete/i }));
  
  // Wait for element to disappear
  await waitFor(() => {
    expect(screen.queryByText(/buy milk/i)).not.toBeInTheDocument();
  });
});
```

### Using waitForElementToBeRemoved

```javascript
import { waitForElementToBeRemoved } from '@testing-library/react';

it('shows then hides loading indicator', async () => {
  render(<DataLoader />);
  
  await waitForElementToBeRemoved(() => screen.queryByText(/loading/i));
  
  expect(screen.getByText(/data loaded/i)).toBeInTheDocument();
});
```

## Testing Pitfalls

<Warning>
**Pitfall 1 -- Testing implementation details**: If your test checks internal state, CSS class names, or component structure, it will break every time you refactor -- even if the feature still works. Test what the user sees: text content, form behavior, navigation.

```javascript
// BAD: testing implementation detail
expect(wrapper.state('isOpen')).toBe(true);
expect(container.querySelector('.modal--active')).toBeTruthy();

// GOOD: testing user-visible behavior
expect(screen.getByRole('dialog')).toBeVisible();
```

**Pitfall 2 -- Using `fireEvent` instead of `userEvent`**: `fireEvent` dispatches a single DOM event directly. `userEvent` simulates full user interactions (focus, keydown, keypress, keyup, input, change). For example, `userEvent.type()` fires events in the same order a real keyboard does, which catches bugs that `fireEvent.change()` misses (like components that listen to `onKeyDown`).

**Pitfall 3 -- Not awaiting async operations**: Forgetting `await` before `userEvent` actions or `findBy` queries causes tests to complete before the UI updates. The test passes today but fails intermittently later. Always `await` any interaction or query that involves async behavior.

**Pitfall 4 -- Asserting too soon on disappearing elements**: When testing that an element disappears, `queryBy` returns `null` synchronously. If the removal is async (e.g., after an API call), use `waitForElementToBeRemoved` instead of immediately checking `queryByText`.

**Pitfall 5 -- Forgetting to clean up mocks**: If you mock `global.fetch` or a module in one test and forget to restore it, subsequent tests inherit the mock. Always use `afterEach(() => vi.restoreAllMocks())` or the equivalent Jest cleanup.
</Warning>

## Snapshot Testing

```javascript
import { render } from '@testing-library/react';

describe('Card', () => {
  it('matches snapshot', () => {
    const { container } = render(
      <Card title="Hello" description="World" />
    );
    
    expect(container).toMatchSnapshot();
  });

  it('matches snapshot with inline snapshot', () => {
    const { container } = render(<Badge>New</Badge>);
    
    expect(container.innerHTML).toMatchInlineSnapshot(`
      "<span class="badge badge-primary">New</span>"
    `);
  });
});
```

<Warning>
Use snapshots sparingly. They can become a maintenance burden and may encourage "update all" without reviewing changes.

## Common Testing Patterns

### Testing Conditional Rendering

```javascript
describe('Alert', () => {
  it('renders nothing when closed', () => {
    const { container } = render(<Alert isOpen={false} message="Hello" />);
    
    expect(container).toBeEmptyDOMElement();
  });

  it('renders message when open', () => {
    render(<Alert isOpen={true} message="Hello World" />);
    
    expect(screen.getByText(/hello world/i)).toBeInTheDocument();
  });
});
```

### Testing Lists

```javascript
describe('TodoList', () => {
  const todos = [
    { id: 1, text: 'Buy milk', completed: false },
    { id: 2, text: 'Walk dog', completed: true }
  ];

  it('renders all items', () => {
    render(<TodoList items={todos} />);
    
    expect(screen.getAllByRole('listitem')).toHaveLength(2);
  });

  it('shows completed state', () => {
    render(<TodoList items={todos} />);
    
    expect(screen.getByText(/walk dog/i)).toHaveClass('completed');
  });

  it('shows empty message when no items', () => {
    render(<TodoList items={[]} />);
    
    expect(screen.getByText(/no todos yet/i)).toBeInTheDocument();
  });
});
```

### Testing Error Boundaries

```javascript
// Suppress console.error for error boundary tests
beforeEach(() => {
  vi.spyOn(console, 'error').mockImplementation(() => {});
});

afterEach(() => {
  console.error.mockRestore();
});

        <ThrowError />
      </ErrorBoundary>
    );
    
    expect(screen.getByText(/something went wrong/i)).toBeInTheDocument();
  });
});
```

## Interview Deep-Dive

<AccordionGroup>
<Accordion title="What does 'testing implementation details' mean in React testing, and why does React Testing Library discourage it?">
**Strong Answer:**
Testing implementation details means your test knows HOW a component works internally rather than WHAT it does for the user. Examples: querying by CSS class names, checking internal state variables, asserting that a specific hook was called, or verifying that a component renders a specific child component by name.

The problem: when you refactor the component's internals (rename a CSS class, restructure the JSX, extract a sub-component, switch from useState to useReducer), the test breaks even though the feature still works perfectly from the user's perspective. This creates a maintenance burden where developers spend more time updating tests than catching bugs. It also creates false confidence: tests pass because the implementation matches the assertion, not because the feature works.

React Testing Library enforces user-centric testing through its query priority: `getByRole`, `getByLabelText`, `getByText`. These query elements the way a user or screen reader would find them -- by visible text, form labels, and ARIA roles. If you change a `div` to a `section`, the role query still works. If you rename a CSS class, the text query still works.

The practical test I apply: "Could a non-developer QA person describe this test case?" If yes ("verify that clicking Submit shows a success message"), it tests behavior. If no ("verify that the submitHandler state is set to true and the FormWrapper component receives isSubmitting prop"), it tests implementation.

The one exception: snapshot tests are inherently implementation-detail tests because they capture the entire rendered structure. Use them sparingly for components where visual regression matters (design system components) and review snapshot diffs carefully instead of blindly updating them.

**Follow-up: How do you test a component that uses useEffect for data fetching without testing implementation details?**

Test the observable behavior: the loading state, the loaded data, and the error state. Mock the API at the network level (using `msw` or mocking `fetch`), render the component, assert the loading indicator appears, wait for the data to load using `findByText` or `waitFor`, then assert the data is displayed.

You should NOT: mock `useEffect` to prevent it from running, spy on the internal `fetchData` function, or assert that `useState` was called with specific values. These are all implementation details.

With Mock Service Worker (msw), you define server handlers that intercept network requests and return mock responses. The component makes real `fetch` calls, msw intercepts them, and the component processes the response normally. Your test does not know or care whether the component uses `useEffect`, React Query, or SWR to fetch data. It only cares that the user sees the right content.
</Accordion>

<Accordion title="How do you test components that depend on Context, React Router, or Redux? What patterns keep tests maintainable?">
**Strong Answer:**
The pattern is creating a custom render wrapper that provides all necessary providers with sensible defaults, allowing individual tests to override only what they need.

```javascript
function renderWithProviders(ui, options = {}) {
  const { user = null, route = '/', theme = 'light' } = options;
  return render(
    <MemoryRouter initialEntries={[route]}>
      <AuthContext.Provider value={{ user, login: vi.fn(), logout: vi.fn() }}>
        <ThemeContext.Provider value={{ theme }}>
          {ui}
        </ThemeContext.Provider>
      </AuthContext.Provider>
    </MemoryRouter>,
  );
}
```

For Redux, use the real store with `configureStore` and a preloaded state rather than mocking `useSelector`. This tests the actual integration between the component and the store. Redux's own testing docs recommend this approach.

For React Router, use `MemoryRouter` with `initialEntries` to set the starting route. This avoids needing a real browser history and lets you test route-dependent behavior (showing different content based on URL params).

The maintenance principle: test utilities should be in a shared `test-utils.js` file imported by all tests. When you add a new global provider (say, a NotificationProvider), you update one file, not 200 tests.

For unit tests where you want to isolate a component from its context dependencies, mock the custom hook directly: `vi.mock('../hooks/useAuth')`. This is faster and lets you test edge cases (like `user = null` without setting up the full auth flow) but sacrifices integration confidence. I use hook mocking for unit tests and the full wrapper for integration tests.

**Follow-up: How do you choose between unit tests, integration tests, and end-to-end tests for React components?**

My testing strategy follows the trophy model (popularized by Kent C. Dodds): mostly integration tests, some unit tests, fewer E2E tests, and a thin layer of static analysis (TypeScript, ESLint).

Unit tests: pure utility functions (formatDate, calculateTotal), custom hooks with complex logic (useForm validation), and isolated components with many edge cases (a DatePicker with locale handling).

Integration tests: most components, tested with their hooks, context, and router but with mocked API calls. These catch real bugs (a form that does not submit because the handler reads the wrong field) without the slowness of E2E. This is where React Testing Library excels.

E2E tests (Cypress, Playwright): critical user flows. Login, checkout, onboarding. These run in a real browser against a real (or test) backend. They are slow, flaky, and expensive to maintain, so I limit them to flows where a bug costs real money or users.

The anti-pattern: 100% unit test coverage with every component tested in isolation, all dependencies mocked. This gives false confidence because the mocks may not match reality. A component that works in a unit test may break when connected to the real context because the mock's API diverged.
</Accordion>

<Accordion title="What is the difference between getBy, queryBy, and findBy queries in React Testing Library? When do you use each?">
**Strong Answer:**
The three query types differ in how they handle missing elements and asynchronous rendering.

`getBy` throws an error immediately if the element is not found, and throws if more than one match exists. Use it when the element MUST be present at query time. If the element is missing, the test fails with a clear error message showing the available DOM. This is your default query for synchronous assertions.

`queryBy` returns `null` if the element is not found (instead of throwing). Use it to assert that something does NOT exist: `expect(screen.queryByText('Error')).not.toBeInTheDocument()`. Using `getBy` for absence assertions would throw before the expect runs.

`findBy` returns a Promise that resolves when the element appears (with a default timeout of 1000ms). It is equivalent to `waitFor(() => getBy(...))`. Use it for elements that appear asynchronously -- after a data fetch, after an animation, or after a state update triggered by a timer.

The common mistake: using `getBy` for elements that appear after an async operation. The test fails immediately because the element does not exist yet. Use `findBy` and await it: `const heading = await screen.findByText('Welcome')`.

Another mistake: using `findBy` when `getBy` would work. `findBy` adds unnecessary async overhead and can mask timing bugs (the element should be present immediately but only appears after a tick). If the element is synchronously rendered, use `getBy` for clarity and speed.

The `All` variants (`getAllBy`, `queryAllBy`, `findAllBy`) return arrays when multiple matches are expected. `getAllBy` throws if no matches exist; `queryAllBy` returns an empty array.

**Follow-up: Your test passes locally but fails in CI. The element exists in the DOM but findBy times out. What do you investigate?**

Several causes, in order of likelihood. First, CI is slower than your local machine. The default 1000ms timeout for `findBy` may not be enough. Increase it: `await screen.findByText('data', {}, { timeout: 3000 })`. But this is a band-aid -- the real question is why the async operation is slow.

Second, a timer or animation in the component uses real timers that run slowly in CI. Use `vi.useFakeTimers()` and advance time programmatically so tests do not depend on real clock speed.

Third, the API mock is not set up correctly in the CI environment. Environment-specific configuration (like a different mock server URL) may cause the mock to miss the request, so the fetch hangs indefinitely. Check that mock setup runs before the test.

Fourth, React batching differences. In React 18, some updates are batched differently in test mode vs browser mode. Using `act()` properly (React Testing Library does this automatically for most cases) resolves this, but custom async flows may need explicit `act` wrapping.

Debug approach: add `screen.debug()` before the failing assertion to print the current DOM state. If the element is not in the DOM at all, the async operation did not complete. If it IS in the DOM, the query might be wrong (case sensitivity, partial text matching).
</Accordion>
