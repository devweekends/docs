# TypeScript with React

TypeScript adds static type checking to JavaScript, catching errors at compile time and providing better IDE support. It's now the standard for professional React development.

**Why it matters for React specifically**: In plain JavaScript, if you pass `userId` as a string to a component that expects a number, you will not discover the bug until runtime -- maybe in production, maybe in an obscure edge case. TypeScript catches this the moment you type it, right in your editor with a red underline. For React components with many props, context values, and hook return types, TypeScript acts as living documentation that stays in sync with your code automatically.

## Basic TypeScript Concepts

### Type Annotations

```typescript
// Primitives
let name: string = 'John';
let age: number = 25;
let isActive: boolean = true;

// Arrays
let numbers: number[] = [1, 2, 3];
let names: Array<string> = ['a', 'b', 'c'];

// Objects
let user: { name: string; age: number } = {
  name: 'John',
  age: 25
};

// Functions
function greet(name: string): string {
  return `Hello, ${name}!`;
}

// Arrow functions
const add = (a: number, b: number): number => a + b;
```

### Interfaces vs Types

```typescript
// Interface - preferred for objects
interface User {
  id: number;
  name: string;
  email: string;
}

// Type - preferred for unions, primitives, tuples
type Status = 'idle' | 'loading' | 'success' | 'error';
type ID = string | number;
type Point = [number, number];

// Both can be extended
interface Admin extends User {
  role: 'admin';
  permissions: string[];
}

type SuperAdmin = Admin & {
  superPowers: string[];
};
```

### Generics

Generics are like function parameters, but for types. Just as a function parameter (`name: string`) lets the caller decide which value to pass, a generic parameter (`<T>`) lets the caller decide which *type* to pass. This is how you write reusable code that works with any type while still maintaining type safety.

```typescript
// Generic function -- T is a placeholder for "whatever type the caller uses"
function first<T>(arr: T[]): T | undefined {
  return arr[0];
}

// TypeScript infers T from the argument:
const num = first([1, 2, 3]);     // T = number, return type = number | undefined
const str = first(['a', 'b']);    // T = string, return type = string | undefined

// Generic interface
interface ApiResponse<T> {
  data: T;
  status: number;
  message: string;
}

const userResponse: ApiResponse<User> = {
  data: { id: 1, name: 'John', email: 'john@example.com' },
  status: 200,
  message: 'Success'
};
```

### useReducer

```tsx
interface State {
  count: number;
  error: string | null;
}

type Action =
  | { type: 'increment' }
  | { type: 'decrement' }
  | { type: 'reset' }
  | { type: 'setCount'; payload: number }
  | { type: 'setError'; payload: string };

function reducer(state: State, action: Action): State {
  switch (action.type) {
    case 'increment':
      return { ...state, count: state.count + 1, error: null };
    case 'decrement':
      return { ...state, count: state.count - 1, error: null };
    case 'reset':
      return { count: 0, error: null };
    case 'setCount':
      return { ...state, count: action.payload };
    case 'setError':
      return { ...state, error: action.payload };
    default:
      return state;
  }
}

function Counter() {
  const [state, dispatch] = useReducer(reducer, { count: 0, error: null });
  
  return (
    <button onClick={() => dispatch({ type: 'increment' })}>
      Count: {state.count}
    </button>
  );
}
```

## Typing Custom Hooks

```tsx
// Simple hook with return type inferred
function useToggle(initialValue = false) {
  const [value, setValue] = useState(initialValue);
  
  const toggle = useCallback(() => setValue(v => !v), []);
  const setTrue = useCallback(() => setValue(true), []);
  const setFalse = useCallback(() => setValue(false), []);

  return { value, toggle, setTrue, setFalse };
}

// Hook with explicit types
interface UseFetchResult<T> {
  data: T | null;
  loading: boolean;
  error: Error | null;
  refetch: () => void;
}

function useFetch<T>(url: string): UseFetchResult<T> {
  const [data, setData] = useState<T | null>(null);
  const [loading, setLoading] = useState(true);
  const [error, setError] = useState<Error | null>(null);

  const fetchData = useCallback(async () => {
    try {
      setLoading(true);
      const response = await fetch(url);
      const json = await response.json();
      setData(json);
    } catch (e) {
      setError(e as Error);
    } finally {
      setLoading(false);
    }
  }, [url]);

  useEffect(() => {
    fetchData();
  }, [fetchData]);

  return { data, loading, error, refetch: fetchData };
}

// Usage
interface User {
  id: number;
  name: string;
}

function UserProfile() {
  const { data: user, loading, error } = useFetch<User>('/api/user');
  
  if (loading) return <Spinner />;
  if (error) return <Error message={error.message} />;
  
  return <div>{user?.name}</div>;
}
```

// State type
interface CounterState {

  status: 'idle' | 'loading' | 'failed';
}

const initialState: CounterState = {
  value: 0,
  status: 'idle'

// Slice
const counterSlice = createSlice({
  name: 'counter',
  initialState,
  reducers: {
    increment: (state) => {
      state.value += 1;
    },
    decrement: (state) => {
      state.value -= 1;
    },
    incrementByAmount: (state, action: PayloadAction<number>) => {

    }
  }

// Store
const store = configureStore({
  reducer: {
    counter: counterSlice.reducer
  }

export type AppDispatch = typeof store.dispatch;

import { useDispatch, useSelector, TypedUseSelectorHook } from 'react-redux';

export const useAppSelector: TypedUseSelectorHook<RootState> = useSelector;

## TypeScript + React Pitfalls

<Warning>
**Pitfall 1 -- Typing state as the wrong union**: A common mistake is typing `useState<User>(null)` instead of `useState<User | null>(null)`. The first gives a type error because `null` is not assignable to `User`. Always include `null` in the union when the initial value can be null.

**Pitfall 2 -- Overusing `any` to silence errors**: Sprinkling `any` everywhere defeats the purpose of TypeScript. If you find yourself reaching for `any`, first try `unknown` (which forces you to narrow the type before using it), or define a proper interface. Every `any` is a hole in your type safety.

```typescript
// BAD: any spreads and infects everything it touches
const handleResponse = (data: any) => {
  setUser(data); // No error even if data is wrong shape
};

// GOOD: type the expected shape
interface UserResponse {
  id: number;
  name: string;
  email: string;
}
const handleResponse = (data: UserResponse) => {
  setUser(data); // TypeScript validates the shape
};
```

**Pitfall 3 -- Forgetting that event types differ by element**: `React.ChangeEvent<HTMLInputElement>` and `React.ChangeEvent<HTMLSelectElement>` are different types. If you write a shared `handleChange` for inputs and selects, type the event parameter as `React.ChangeEvent<HTMLInputElement | HTMLSelectElement>`.

**Pitfall 4 -- Using `React.FC` with default props**: `React.FC` wraps your component and can interfere with how TypeScript infers default prop values. If you use destructured defaults (`{ size = 'md' }`), prefer typing props directly on the function parameter rather than using `React.FC`.

**Pitfall 5 -- Not narrowing `useRef` correctly**: `useRef<HTMLInputElement>(null)` creates a `RefObject` where `current` is readonly. If you need a mutable ref (to store a timer ID, for example), use `useRef<number | null>(null)` -- note the `null` in the generic, not as the initial value to a non-null type.
</Warning>

## Common Patterns & Utility Types

```tsx
// Partial - all properties optional
type PartialUser = Partial<User>;
// { id?: number; name?: string; email?: string }

// Required - all properties required
type RequiredUser = Required<PartialUser>;

// Pick - select specific properties
type UserPreview = Pick<User, 'id' | 'name'>;
// { id: number; name: string }

// Omit - exclude specific properties
type UserWithoutEmail = Omit<User, 'email'>;
// { id: number; name: string }

// Record - create object type
type UserRoles = Record<string, 'admin' | 'user' | 'guest'>;
// { [key: string]: 'admin' | 'user' | 'guest' }

// Extract - extract union members
type Status = 'active' | 'inactive' | 'pending';
type ActiveStatus = Extract<Status, 'active' | 'pending'>;
// 'active' | 'pending'

// Exclude - exclude union members
type InactiveStatus = Exclude<Status, 'active'>;
// 'inactive' | 'pending'

// NonNullable - remove null and undefined
type MaybeString = string | null | undefined;
type DefiniteString = NonNullable<MaybeString>;
// string

// ReturnType - infer function return type
function createUser() {
  return { id: 1, name: 'John' };
}
type CreateUserReturn = ReturnType<typeof createUser>;
// { id: number; name: string }

// Parameters - infer function parameters
type CreateUserParams = Parameters<typeof createUser>;
// []
```

  | { type: 'ADD'; payload: Omit<Todo, 'id' | 'createdAt'> }
  | { type: 'TOGGLE'; payload: string }

  switch (action.type) {
    case 'ADD':
      return [
        ...state,
        {
          ...action.payload,
          id: crypto.randomUUID(),
          createdAt: new Date()
        }
      ];
    case 'TOGGLE':
      return state.map(todo =>
        todo.id === action.payload
          ? { ...todo, completed: !todo.completed }

    case 'DELETE':
      return state.filter(todo => todo.id !== action.payload);
    case 'EDIT':
      return state.map(todo =>
        todo.id === action.payload.id
          ? { ...todo, text: action.payload.text }
          : todo
      );
    case 'CLEAR_COMPLETED':
      return state.filter(todo => !todo.completed);
    default:
      return state;
  }
}

## Interview Deep-Dive

<AccordionGroup>
<Accordion title="What is the difference between 'interface' and 'type' in TypeScript for React props? When do you choose one over the other?">
**Strong Answer:**
Both `interface` and `type` can define the shape of React props, and for most component prop definitions they are interchangeable. The differences are subtle but matter in specific scenarios.

Interfaces support declaration merging: if you declare `interface User` twice in the same scope, TypeScript merges them into one. This is useful for extending third-party types (augmenting a library's type definitions) but dangerous for application code because accidental merges can add unexpected properties silently. Types do not merge -- redeclaring a type is a compile error.

Types support union types, intersection types, mapped types, and conditional types. You cannot express `type Status = 'idle' | 'loading' | 'error'` with an interface. You cannot express `type Props = BaseProps & { extra: string }` as cleanly (interfaces use `extends`, which is more verbose for simple intersections).

For React props specifically: I use `interface` when defining component props because it reads cleanly (`interface ButtonProps { ... }`), supports `extends` for composition (`interface SubmitButtonProps extends ButtonProps { ... }`), and is the convention in most React codebases. I use `type` for unions (`type Variant = 'primary' | 'secondary'`), function signatures (`type Handler = (e: Event) => void`), and complex derived types using utility types.

The practical rule: if your type is an object shape (props, state, API response), use interface. If your type involves unions, intersections, or mapped/conditional types, use type. Do not mix them randomly within the same codebase -- pick a convention and enforce it with ESLint.

**Follow-up: How do discriminated unions in TypeScript help you build type-safe React components with conditional props?**

Discriminated unions let you define multiple valid prop combinations where certain props are required or forbidden based on a discriminant field. The classic example is a Button component that can be either a link or a button:

```typescript
type ButtonProps =
  | { variant: 'link'; href: string; onClick?: never }
  | { variant: 'button'; onClick: () => void; href?: never };
```

The `never` type means "this prop cannot be provided." If you pass `variant="link"`, TypeScript requires `href` and forbids `onClick`. If you pass `variant="button"`, TypeScript requires `onClick` and forbids `href`. This catches invalid combinations at compile time rather than runtime.

Inside the component, TypeScript narrows the type when you check the discriminant: `if (props.variant === 'link') { /* props.href is string here */ }`. This eliminates the need for runtime type checking and makes the component's behavior predictable from its type signature alone.

This pattern is extremely valuable for design system components where prop combinations need to be constrained (a Select that is either single-value or multi-value, a Modal that is either controlled or uncontrolled).
</Accordion>

<Accordion title="How do you type a generic custom hook like useFetch<T>? Walk through the type design and explain why generics are necessary here.">
**Strong Answer:**
Without generics, `useFetch` would return `data: any`, which defeats the purpose of TypeScript. Every consumer would need to cast the result, and the compiler could not catch type mismatches between the API response and how the component uses the data.

With generics, the consumer specifies the expected response type: `const { data } = useFetch<User[]>('/api/users')`. Inside the hook, `data` is typed as `User[] | null`, and the compiler enforces that the component accesses only valid properties (`data[0].name` is valid, `data[0].foo` is an error).

The implementation:

```typescript
interface UseFetchResult<T> {
  data: T | null;
  loading: boolean;
  error: Error | null;
}

function useFetch<T>(url: string): UseFetchResult<T> {
  const [data, setData] = useState<T | null>(null);
  const [loading, setLoading] = useState(true);
  const [error, setError] = useState<Error | null>(null);
  // ... fetch logic
  return { data, loading, error };
}
```

The generic `T` flows from the call site through the hook's internals to the return type. `useState<T | null>(null)` tells TypeScript that `data` will eventually be `T` or null. The `UseFetchResult<T>` interface propagates the type to the consumer.

Why `T | null` instead of just `T`: before the fetch resolves, there is no data. Using `T` alone would require a default value of type `T`, which is impossible for a generic (what is the default User?). The `null` represents the "not yet loaded" state.

**Follow-up: How do you handle TypeScript when the fetch response does not match the expected type T at runtime?**

TypeScript's type system is erased at runtime -- there are no runtime checks that the API response actually matches `T`. If the server returns `{ userName: 'Alice' }` but your type says `{ name: string }`, TypeScript is happy at compile time but the code fails at runtime when accessing `data.name` (which is undefined).

Three defenses. First, use a runtime validation library like Zod: define a schema `const UserSchema = z.object({ name: z.string() })`, parse the response `UserSchema.parse(json)`, and infer the TypeScript type from the schema `type User = z.infer<typeof UserSchema>`. This gives you both compile-time and runtime safety from a single source of truth.

Second, for less critical code, add a type guard function: `function isUser(data: unknown): data is User { return typeof data === 'object' && data !== null && 'name' in data }`. Call it after fetching and throw an error if it fails.

Third, for internal APIs where you control both client and server, generate types from the API schema (OpenAPI, GraphQL codegen) so the types are always in sync with the actual response shape. This eliminates the drift problem at its source.
</Accordion>

<Accordion title="What TypeScript utility types do you use most often in React codebases, and why?">
**Strong Answer:**
Five utility types I use constantly:

`Partial<T>` makes all properties optional. Essential for update functions: `function updateUser(id: string, changes: Partial<User>)`. The caller can pass any subset of User fields without providing them all.

`Pick<T, K>` selects specific properties. Perfect for component props that only need a subset of a larger type: `type UserCardProps = Pick<User, 'name' | 'avatar'>`. This keeps props minimal and avoids passing unnecessary data.

`Omit<T, K>` excludes specific properties. Common for "everything except" patterns: `type CreateUserInput = Omit<User, 'id' | 'createdAt'>` -- the server generates these fields, so the client should not provide them.

`Record<K, V>` creates an object type with known key types. Useful for lookup maps: `Record<string, React.ComponentType>` for dynamic component rendering, or `Record<Status, string>` for mapping enum values to display labels.

`React.ComponentProps<typeof Component>` extracts the props type from an existing component. Invaluable when wrapping third-party components: you can extend their props without manually retyping them. `interface MyButtonProps extends React.ComponentProps<typeof ThirdPartyButton> { customProp: string }`.

A bonus pattern: `ReturnType<typeof useCustomHook>` infers the return type of a custom hook. This is useful when you need to pass hook results as props to a child component and want the types to stay in sync automatically.

**Follow-up: How do you handle the `children` prop in TypeScript? What are the differences between ReactNode, ReactElement, and PropsWithChildren?**

`React.ReactNode` is the most permissive: it accepts strings, numbers, booleans, null, undefined, JSX elements, arrays, and fragments. Use it for components that accept any renderable content, which is most components with children.

`React.ReactElement` is stricter: only JSX elements (the result of `<Component />` or `React.createElement()`). Use it when you need to clone or inspect children with `React.cloneElement` or `React.Children.map`, since those APIs require actual elements, not strings or numbers.

`PropsWithChildren<P>` is a shorthand: `type PropsWithChildren<P> = P & { children?: ReactNode }`. It adds an optional `children` prop to any props type. Useful but declining in popularity because explicitly declaring `children: React.ReactNode` in your interface is clearer and lets you make it required rather than optional.

The gotcha: in React 18's TypeScript types, `React.FC` no longer includes `children` implicitly. You must declare it in your props interface. This was an intentional change to make the children prop explicit and catch cases where a component should not accept children.
</Accordion>
