## How React Native Works

React Native allows you to build mobile apps using JavaScript and React. But unlike web apps that render to a browser's DOM, React Native renders to actual native views -- the same `UIView` and `android.view.View` components that Swift and Kotlin apps use. Your code is not running inside a hidden web browser; it is producing genuinely native UI elements.

```
┌─────────────────────────────────────────────────────────────────────────┐
│                    React Native Rendering Pipeline                       │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                          │
│   Your React Code                                                        │
│         │                                                                │
│         ▼                                                                │
│   ┌─────────────┐      ┌─────────────┐      ┌─────────────┐             │
│   │   React     │      │   Bridge/   │      │   Native    │             │
│   │ Components  │ ───► │    JSI      │ ───► │   Views     │             │
│   │   (JSX)     │      │             │      │             │             │
│   └─────────────┘      └─────────────┘      └─────────────┘             │
│                                                                          │
│   <View>         →     Serialization  →     UIView (iOS)                │
│   <Text>         →     / JSI calls    →     TextView (Android)          │
│   <Image>        →                    →     UIImageView / ImageView     │
│                                                                          │
└─────────────────────────────────────────────────────────────────────────┘
```

### The JavaScript Thread

Your React code runs in a JavaScript engine (Hermes or JavaScriptCore):

```javascript
// This runs on the JavaScript thread
const App = () => {
  const [count, setCount] = useState(0);
  
  return (
    <View style={styles.container}>
      <Text>Count: {count}</Text>
      <Button 
        title="Increment" 
        onPress={() => setCount(count + 1)} 
      />
    </View>
  );
};
```

### The Native Thread

Native modules and UI rendering run on the native thread:

- **iOS**: Objective-C/Swift code using UIKit
- **Android**: Java/Kotlin code using Android SDK

### The Bridge (Legacy Architecture)

The old architecture used a "bridge" for JS-Native communication:

```
┌────────────────────────────────────────────────────────────────────┐
│                     Legacy Bridge Architecture                      │
├────────────────────────────────────────────────────────────────────┤
│                                                                     │
│   JavaScript Thread          Bridge          Native Thread          │
│   ─────────────────          ──────          ─────────────          │
│                                                                     │
│   setCount(5)        →    JSON.stringify()   →    Update UI        │
│                           { type: "update",                        │
│                             data: { count: 5 }}                    │
│                                                                     │
│   Problems:                                                         │
│   • Async by default (can't be synchronous)                        │
│   • Serialization overhead for large data                          │
│   • Single-threaded bottleneck                                     │
│                                                                     │
└────────────────────────────────────────────────────────────────────┘
```

## The New Architecture

React Native's new architecture (Fabric + TurboModules + JSI) solves legacy bridge limitations:

```
┌────────────────────────────────────────────────────────────────────────┐
│                     New Architecture (RN 0.74+)                         │
├────────────────────────────────────────────────────────────────────────┤
│                                                                         │
│   ┌───────────────────────────────────────────────────────────────┐    │
│   │                  JavaScript Interface (JSI)                    │    │
│   │  ─────────────────────────────────────────────────────────────│    │
│   │  • Direct C++ bindings - no serialization needed              │    │
│   │  • Synchronous calls possible                                 │    │
│   │  • Enables Fabric and TurboModules                            │    │
│   └───────────────────────────────────────────────────────────────┘    │
│                                                                         │
│   ┌─────────────────────┐    ┌─────────────────────────────────────┐   │
│   │   Fabric Renderer    │    │         TurboModules                │   │
│   │   ─────────────────  │    │         ────────────                │   │
│   │   • New UI rendering │    │   • Lazy-loaded native modules     │   │
│   │   • Concurrent mode  │    │   • Type-safe (CodeGen)            │   │
│   │   • Improved perf    │    │   • Direct JS-Native calls         │   │
│   └─────────────────────┘    └─────────────────────────────────────┘   │
│                                                                         │
│   Benefits:                                                             │
│   ✓ Up to 10x faster startup                                           │
│   ✓ Synchronous native calls when needed                               │
│   ✓ Better memory usage                                                │
│   ✓ Concurrent React features                                          │
│                                                                         │
└────────────────────────────────────────────────────────────────────────┘
```

### Hermes JavaScript Engine

Hermes is a JavaScript engine optimized for React Native:

```javascript
// app.json or app.config.js - Hermes is now enabled by default
{
  "expo": {
    "jsEngine": "hermes"  // Default in Expo SDK 48+
  }
}

// Benefits of Hermes:
// - Faster app startup: JS is compiled to bytecode at BUILD time, not at runtime.
//   Think of it like pre-cooking a meal vs cooking from scratch when the customer orders.
// - Lower memory usage: bytecode is more compact in memory than raw source + AST
// - Smaller app size: bytecode is smaller than minified JavaScript source
// - Better garbage collection: Hermes uses a generational GC tuned for mobile,
//   which reduces the "jank" pauses that old-gen GC sweeps can cause
```

<Note>
**Hermes is now the default** in both Expo and React Native CLI projects. It provides significant performance improvements, especially for app startup time.
</Note>

## Expo vs React Native CLI

<Frame>
  <img src="/images/courses/react-native-crash-course/expo-vs-cli.svg" alt="Expo vs React Native CLI" />
</Frame>

### Expo (Recommended for Most Projects)

```bash
# Create a new Expo project
npx create-expo-app@latest my-app
cd my-app
npx expo start
```

**Advantages:**
- Zero configuration to get started
- Expo Go app for instant testing
- Managed build service (EAS)
- Over-the-air updates
- Pre-built native modules
- Web support included

**When to use:**
- Starting a new project
- Rapid prototyping
- Teams without native experience
- Most production apps

### React Native CLI

```bash
# Create a new React Native CLI project
npx react-native init MyApp
cd MyApp
npx react-native run-ios # or run-android
```

**Advantages:**
- Full native code access
- Custom native modules
- Integration with existing native apps
- More control over build process

**When to use:**
- Need custom native code
- Integrating into existing native app
- Specific native SDK requirements

<Warning>
**Recommendation**: Start with Expo. You can always "eject" or use development builds for native code access. Most apps never need to leave Expo.
</Warning>

## Hot Reloading & Fast Refresh

React Native supports **Fast Refresh** - your changes appear instantly without losing component state:

```tsx
// Edit this and watch it update instantly!
export default function App() {
  const [count, setCount] = useState(0);
  
  return (
    <View style={styles.container}>
      {/* Change this text - state (count) is preserved! */}
      <Text style={styles.title}>Fast Refresh Demo</Text>
      <Text>Count: {count}</Text>
      <Button title="+" onPress={() => setCount(c => c + 1)} />
    </View>
  );
}
```

<Tip>
**Fast Refresh preserves state** for function components with hooks. If you need to reset state, add `// @refresh reset` comment at the top of the file. Be aware that Fast Refresh has some limitations: it cannot preserve state in class components, and if a file exports anything other than a React component (such as a constant or utility function), it will trigger a full reload instead of a hot update. Keep your component files focused on exporting components only.
</Tip>

## Interview Deep-Dive

<AccordionGroup>
  <Accordion title="Compare Hermes and JavaScriptCore for React Native. When might you choose one over the other?">
    **Strong Answer:**
    - Hermes uses ahead-of-time compilation: JS is compiled to bytecode at build time, so at runtime there is no parsing or compilation phase. This gives 53% faster startup, 26% less memory, and 46% smaller app size compared to JSC on Meta's benchmarks. Hermes is the default and right choice for virtually all production React Native apps.
    - JavaScriptCore uses JIT compilation at runtime, which means it can optimize hot code paths to near-native speed. For apps with heavy client-side computation (complex data transformations, client-side encryption), JSC or V8 may execute specific algorithms faster than Hermes's interpreter.
    - In practice, the scenarios where JSC wins are exactly the scenarios where you should move that computation to a native module anyway. Mobile apps should not be running heavy algorithms in JavaScript. Hermes's startup and memory advantages matter more for real-world mobile UX than JIT speed on micro-benchmarks.
    - One edge case: if you are debugging with Chrome DevTools (remote debugging), your code runs in Chrome's V8 engine regardless of your configured JS engine. This means performance characteristics differ between debugging and production, which is a common source of confusion.

    **Follow-up:** You enable Hermes on an existing project and a third-party library crashes at runtime. What is the likely cause?

    **Follow-up Answer:**
    - Hermes does not support all JavaScript features that JSC does. The most common culprit is the library using `Proxy` objects (Hermes added Proxy support in later versions but older Hermes builds lack it), or using `eval()` or `new Function()` which Hermes restricts for security reasons.
    - Check the Hermes compatibility table for the specific JS feature. If the library uses an unsupported feature, options are: update Hermes to the latest version, find an alternative library, or polyfill the missing feature.
    - Run `npx react-native doctor` to verify Hermes is enabled correctly and check for known compatibility issues with your dependency tree.
  </Accordion>

  <Accordion title="What is the difference between the JavaScript thread, UI thread, and shadow thread in React Native?">
    **Strong Answer:**
    - The JavaScript thread runs your React code, business logic, state management, and event handlers. It is single-threaded, so any blocking operation (heavy computation, synchronous file I/O) prevents all other JS work from executing. This is the most common source of UI jank in React Native apps.
    - The UI/main thread is the platform's main thread (UIKit on iOS, Android Main Thread). It handles rendering native views, processing touch events, and running native animations. This thread must complete each frame in 16.67ms to maintain 60fps. If your native view hierarchy is too complex (deeply nested Views with shadows), this thread becomes the bottleneck.
    - The shadow thread (legacy) or C++ thread (new architecture) runs Yoga, the Flexbox layout engine. It calculates positions and dimensions for every view in the tree. With the new architecture, this computation happens in shared C++ code accessible from both JS and native threads, enabling synchronous layout measurements.
    - The practical implication: when users report "the app feels slow," you need to determine which thread is the bottleneck. JS thread: React DevTools Profiler. UI thread: Xcode Instruments Time Profiler or Android Profiler. Shadow thread: look for deeply nested Flexbox layouts with complex constraints.

    **Follow-up:** Your app has an animation that moves a view based on a gesture. The animation stutters when the user also triggers an API call. Which thread is the problem and how do you fix it?

    **Follow-up Answer:**
    - The API call triggers `fetch()` on the JS thread. While the Promise resolves, the JS thread processes the response (JSON parsing, state updates, re-renders). If the animation is also driven by the JS thread (using `Animated` without `useNativeDriver`), the animation and API processing compete for the same thread.
    - Fix: use Reanimated with worklets for the gesture-driven animation. Worklets run on the UI thread, completely independent of the JS thread. The gesture handler fires directly into the worklet, which updates the animated values and applies them to the native view without any JS involvement.
    - For the API call side, ensure JSON parsing happens incrementally or that the response is small enough to parse in a single frame budget. If the response is large, use `InteractionManager.runAfterInteractions()` to defer processing until the animation completes.
  </Accordion>

  <Accordion title="Walk me through what happens when you run npx expo start and scan the QR code with Expo Go on a physical device.">
    **Strong Answer:**
    - `npx expo start` launches Metro Bundler, which starts an HTTP server (default port 8081) and a WebSocket server for live communication. Metro scans your project, builds a dependency graph of all modules starting from your entry point, and waits for bundle requests.
    - The terminal displays a QR code encoding the Metro server URL (typically `exp://192.168.x.x:8081`). When you scan this with the Expo Go app, Expo Go makes an HTTP request to that URL to download a manifest describing the app (name, entry point, SDK version).
    - Expo Go then requests the JavaScript bundle from Metro (`/index.bundle?platform=ios`). Metro transforms all source files (JSX to JS, TypeScript to JS via Babel), resolves all imports into a single bundle, and streams it back to the device.
    - Expo Go loads the bundle into its embedded Hermes engine, evaluates the code, and renders your root component using the native modules pre-bundled into Expo Go. The WebSocket connection stays open for Fast Refresh -- when you save a file, Metro pushes the updated module over the WebSocket, and Expo Go hot-swaps it without a full reload.
    - The key limitation: Expo Go includes a fixed set of native modules. If your app uses a module not in Expo Go (like a custom camera library), the JS code loads but crashes when calling the native function. This is when you switch to a development build.

    **Follow-up:** The QR code scan works on Android but the iOS device shows "Network response timed out." What is wrong?

    **Follow-up Answer:**
    - The most common cause is a firewall or network isolation. iOS and the development machine must be on the same WiFi network, and the network must allow device-to-device communication (some corporate WiFi networks block this). Test by pinging the dev machine's IP from another device.
    - Second: macOS firewall may be blocking incoming connections on port 8081. Check System Preferences, Security and Privacy, Firewall, and either disable it temporarily or add Node.js to the allowed list.
    - Third: if you are on a VPN, the VPN may route traffic through a different interface than the WiFi. Try disconnecting the VPN or running `npx expo start --tunnel` which uses Expo's cloud relay (ngrok) to tunnel traffic through the internet, bypassing local network restrictions entirely. This is slower but universally reliable.
  </Accordion>
</AccordionGroup>
