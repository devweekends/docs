# Introduction to React Native

React Native is an open-source framework for building mobile applications using JavaScript and React. Created by Facebook in 2015, it allows developers to build truly native mobile apps for iOS and Android using a single codebase.

## What is React Native?

React Native lets you build mobile apps using only JavaScript. Unlike hybrid frameworks that use WebViews, **React Native renders actual native UI components**, giving you the performance and look-and-feel of a native app.

## How React Native Works

```
┌─────────────────────────────────────────────────────────────────────┐
│                   React Native Architecture                          │
├─────────────────────────────────────────────────────────────────────┤
│                                                                      │
│   JavaScript Thread                   Native Thread                 │
│   ─────────────────                   ─────────────                 │
│                                                                      │
│   ┌──────────────────┐               ┌──────────────────┐           │
│   │   Your React     │               │   Native UI      │           │
│   │   Components     │               │   Components     │           │
│   └────────┬─────────┘               └─────────▲────────┘           │
│            │                                   │                    │
│            │        ┌──────────────┐           │                    │
│            ├───────►│   Bridge     │───────────┤                    │
│            │        │  (Async)     │           │                    │
│            │        └──────────────┘           │                    │
│            │                                   │                    │
│   ┌────────▼─────────┐               ┌─────────┴────────┐           │
│   │  Metro Bundler   │               │  Native Modules  │           │
│   │  (JS Bundle)     │               │  (Camera, GPS)   │           │
│   └──────────────────┘               └──────────────────┘           │
│                                                                      │
│   JavaScriptCore (iOS)              iOS: UIKit, SwiftUI             │
│   or Hermes (Opt-in)                Android: Views, Kotlin          │
│                                                                      │
└─────────────────────────────────────────────────────────────────────┘
```

### The Bridge

The **Bridge** is the heart of React Native. It is an asynchronous communication layer between JavaScript and native code. Think of it like a postal service between two offices that speak different languages: every request gets written down, translated, and physically delivered. It works reliably, but the round-trip adds latency for time-sensitive operations like 60 FPS animations.

1. **JavaScript** writes what UI should look like (declarative descriptions)
2. **Bridge** serializes messages (JSON) and batches them for efficiency
3. **Native** renders actual UI components on the platform's main thread
4. **Native** sends events (taps, scrolls) back through bridge to JavaScript

<Note>
**Performance Consideration**: The bridge is asynchronous and serializes data. For high-frequency updates (animations, gestures), React Native provides ways to bypass it using the **Reanimated** library or **JSI** (JavaScript Interface).
</Note>

## Metro Bundler

Metro is the JavaScript bundler for React Native -- think of it as Webpack's mobile-specialized cousin. While Webpack is a general-purpose bundler for the web, Metro is purpose-built for React Native's needs: it needs to serve bundles quickly to a running app on a simulator or device, and it needs to support fast incremental rebuilds during development.

Metro does four things:

1. **Transforms** your code (JSX to JS, TypeScript to JS) using Babel
2. **Bundles** all files into a single JavaScript file that the native app can execute
3. **Serves** the bundle over HTTP to your app during development (default port 8081)
4. **Watches** for file changes and triggers Fast Refresh updates without a full rebuild

```
┌──────────────────────────────────────────────────────────────┐
│                    Metro Bundler Flow                         │
├──────────────────────────────────────────────────────────────┤
│                                                               │
│   Your Code (JSX/TS)                                          │
│         │                                                     │
│         ▼                                                     │
│   ┌──────────────┐                                            │
│   │  Transform   │  → Babel (JSX → JS)                       │
│   │              │  → TypeScript (TS → JS)                   │
│   └──────┬───────┘                                            │
│         │                                                     │
│         ▼                                                     │
│   ┌──────────────┐                                            │
│   │  Resolution  │  → Find all imports/requires              │
│   │              │  → Build dependency graph                 │
│   └──────┬───────┘                                            │
│         │                                                     │
│         ▼                                                     │
│   ┌──────────────┐                                            │
│   │   Bundle     │  → Combine all modules                    │
│   │              │  → Minify (production)                    │
│   └──────┬───────┘                                            │
│         │                                                     │
│         ▼                                                     │
│   JavaScript Bundle → Sent to app                            │
│                                                               │
└──────────────────────────────────────────────────────────────┘
```

## Hot Reloading vs Fast Refresh

<Tabs>
  <Tab title="Fast Refresh (Recommended)">
    **Enabled by default in React Native 0.61+**
    
    - Preserves component state during edits
    - Works with function components and hooks
    - Automatically enabled
    
    ```tsx
    // Edit this and see changes instantly
    const Counter = () => {
      const [count, setCount] = useState(0);
      
      return (
        <View>
          <Text>Count: {count}</Text>
          <Button title="Increment" onPress={() => setCount(count + 1)} />
        </View>
      );
    };
    ```
    
    After editing, **count state is preserved**! ✨
  </Tab>
  
  <Tab title="Hot Reloading (Legacy)">
    **Old approach** (before Fast Refresh)
    
    - Reloads entire component tree
    - Loses component state
    - Press `Cmd+R` (iOS) or `R+R` (Android)
  </Tab>
  
  <Tab title="Live Reloading">
    **Full app reload**
    
    - Reloads entire app from scratch
    - Loses all state
    - Slower than Fast Refresh
    
    Enable: Shake device → "Enable Live Reload"
  </Tab>
</Tabs>

## Your First App: Hello World++

Let's build a simple but complete app:

```tsx
import React, { useState } from 'react';
import {
  SafeAreaView,
  StyleSheet,
  Text,
  TextInput,
  TouchableOpacity,
  View,
} from 'react-native';

function App(): React.JSX.Element {
  const [name, setName] = useState('');
  const [greeting, setGreeting] = useState('');

  const handlePress = () => {
    setGreeting(`Hello, ${name || 'Stranger'}! 👋`);
  };

  return (
    <SafeAreaView style={styles.container}>
      <View style={styles.content}>
        <Text style={styles.title}>Welcome to React Native!</Text>
        
        <TextInput
          style={styles.input}
          placeholder="Enter your name"
          value={name}
          onChangeText={setName}
        />
        
        <TouchableOpacity style={styles.button} onPress={handlePress}>
          <Text style={styles.buttonText}>Say Hello</Text>
        </TouchableOpacity>
        
        {greeting ? (
          <Text style={styles.greeting}>{greeting}</Text>
        ) : null}
      </View>
    </SafeAreaView>
  );
}

const styles = StyleSheet.create({
  container: {
    flex: 1,
    backgroundColor: '#f5f5f5',
  },
  content: {
    flex: 1,
    justifyContent: 'center',
    alignItems: 'center',
```

## Interview Deep-Dive

<AccordionGroup>
  <Accordion title="Explain what Metro Bundler does and how it differs from Webpack. Why does React Native need its own bundler?">
    **Strong Answer:**
    - Metro is a JavaScript bundler purpose-built for React Native's unique constraints. Unlike Webpack, which produces bundles for browsers that support multiple entry points, code splitting, and HTML generation, Metro produces a single JavaScript file that the native app loads into its JS engine (Hermes or JSC).
    - Metro's three-phase pipeline (Transform, Resolution, Bundle) is optimized for speed during development. It maintains an in-memory dependency graph and uses incremental builds -- when you save a file, only that file is re-transformed, and only affected modules are re-bundled. This enables Fast Refresh to update a running app in under 200ms, which is critical for mobile development where full rebuilds take minutes.
    - Metro also handles platform-specific file resolution automatically. If you import `./Button`, Metro checks for `Button.ios.tsx`, `Button.android.tsx`, and `Button.tsx` and serves the correct one based on the build target. Webpack can do this with plugins, but Metro has it built in because it is a core React Native pattern.
    - The trade-off: Metro has fewer features than Webpack (no native CSS modules, no HTML generation, limited plugin ecosystem). But for React Native's use case -- serving a single JS bundle to a native app container with fast incremental updates -- it is significantly faster and simpler.

    **Follow-up:** A developer on your team says Metro is slow and wants to switch to Webpack for the React Native project. How do you respond?

    **Follow-up Answer:**
    - Metro's perceived slowness is usually the initial cold start (building the full dependency graph for the first time). Subsequent incremental builds are fast. The fix is better caching, not replacing the bundler.
    - Switching to Webpack for React Native is technically possible (with `@callstack/repack`) but introduces significant complexity: you lose native platform file resolution, Fast Refresh requires custom configuration, and EAS Build and Expo tooling assume Metro. The migration cost is high and the debugging surface area increases.
    - Instead, optimize Metro: increase the `maxWorkers` config for parallel transformation, ensure watchman is installed (it is 10x faster than Metro's built-in file watcher), and use `--reset-cache` only when necessary (not on every start).
  </Accordion>

  <Accordion title="You are debugging a React Native app where buttons feel 'unresponsive' -- there is a noticeable delay between tapping and the visual feedback. The JS thread shows heavy computation. What is happening and how do you fix it?">
    **Strong Answer:**
    - This is a classic JS thread bottleneck. Here is the timeline: the user taps the button, the native UI thread detects the touch event immediately and dispatches it to the JS thread. But the JS thread is busy with heavy computation (maybe parsing a large API response, or running an expensive filter/sort on a list). The touch event sits in the JS message queue waiting for the current computation to finish. Only when the JS thread is free does the `onPress` callback fire and trigger the visual state change.
    - The fix depends on what is blocking the JS thread. If it is a large data transformation: move it to a `useEffect` with `InteractionManager.runAfterInteractions()` so it runs after the current frame and touch interactions are complete. If it is JSON parsing of a large response: parse it in chunks using a streaming JSON parser, or better yet, have the server paginate the response.
    - For the button's visual feedback specifically, use `Pressable` with the `style` callback function that receives `{ pressed }`. The pressed state is managed partially on the native side, so the opacity/scale feedback can appear before the JS thread processes the event. With `Animated` and `useNativeDriver: true`, the feedback animation runs on the UI thread entirely.
    - Long-term solution: audit what runs on the JS thread at app startup and on screen mount. Every `useEffect` that computes derived data, every context provider that re-evaluates, every large `JSON.parse` -- these all compete for the same single-threaded JS runtime. Profile with React DevTools Profiler to find the worst offenders.

    **Follow-up:** The remote debugging works fine in Chrome DevTools but the production app is janky. Why the discrepancy?

    **Follow-up Answer:**
    - Remote debugging runs your JavaScript code in Chrome's V8 engine on your development machine, not on the device's Hermes or JSC engine. Chrome V8 has a JIT compiler that optimizes hot code paths to near-native speed, plus your development machine has 10-100x more CPU and RAM than a mobile device. This masks performance problems that are real on-device.
    - Additionally, remote debugging introduces network latency for every bridge communication (messages travel over WebSocket from device to Chrome and back), which paradoxically can make some timing bugs disappear while creating others. Race conditions that depend on execution speed behave differently.
    - The correct approach: use console logging or Flipper for debugging instead of Chrome remote debugging when performance is a concern. For profiling specifically, always test release builds on physical devices -- development mode adds significant overhead (extra checks, verbose error messages, the React DevTools integration).
  </Accordion>

  <Accordion title="What is the difference between Fast Refresh and a full reload in React Native? When does Fast Refresh fail and fall back to a full reload?">
    **Strong Answer:**
    - Fast Refresh preserves component state during code edits. When you change a file, Metro re-transforms only that file, the runtime hot-swaps the module, and React re-renders the affected components without unmounting them. Your useState values, scroll positions, and navigation state survive. A full reload destroys the entire React tree, re-evaluates all modules from scratch, and starts the app from the root component.
    - Fast Refresh falls back to a full reload in several situations: (1) the file exports something other than React components (like a utility function or constant alongside a component), because the runtime cannot determine which exports are components, (2) the file uses class components, because their prototype-based structure cannot be hot-swapped safely, (3) the file contains syntax errors that break module evaluation, and (4) the file is an entry point or a module that many others depend on.
    - There is also a subtle failure mode: if a `useEffect` cleanup function depends on the old state and the new code changes the state shape, Fast Refresh can leave the component in an inconsistent state. Adding `// @refresh reset` at the top of the file forces a full remount for that module, which is sometimes necessary during large refactors.
    - In practice, keep component files focused on exporting a single component. Move constants, types, and utility functions to separate files. This maximizes Fast Refresh's ability to hot-swap individual components without affecting the rest of the app.

    **Follow-up:** Fast Refresh works for UI changes but does not pick up changes to your app's navigation structure. Why, and what is your workaround during development?

    **Follow-up Answer:**
    - Navigation structure is typically defined at the app root level (in your `NavigationContainer` or Expo Router layout files) and evaluated once during app startup. Fast Refresh can update individual screen components, but adding/removing/renaming screens in the navigator requires the navigation container to re-initialize with the new configuration, which effectively requires a full reload.
    - The workaround: use navigation state persistence during development. Save the navigation state to AsyncStorage on every state change, and restore it on app launch. This way, even after a full reload, you land back on the exact screen you were editing. React Navigation provides this out of the box via the `initialState` and `onStateChange` props on `NavigationContainer`. Just remember to disable this in production -- persisted navigation state from an old app version can crash on launch if screen names changed.
  </Accordion>
