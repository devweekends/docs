## Welcome to React Native Mastery

This is a **comprehensive, production-ready course** designed to take you from React Native fundamentals to building and deploying real-world mobile applications. Whether you are a web developer looking to break into mobile development or building your startup's mobile app, this course has you covered.

Mobile development is fundamentally different from web development in ways that surprise most people. You are dealing with two operating systems, each with their own design language and user expectations. You have constrained memory, battery budgets, unreliable network connections, and app store review processes that add days to your release cycle. This course prepares you for all of it.

## Interview Deep-Dive

<AccordionGroup>
  <Accordion title="Why would a company choose React Native over Flutter or fully native development for a new mobile product?">
    **Strong Answer:**
    - The decision hinges on team composition, timeline, and product requirements. If your engineering team already has deep JavaScript/React expertise (common at web-first companies), React Native lets them ship a mobile app in weeks instead of months, with 90%+ code reuse across iOS and Android. Flutter requires learning Dart, which is a non-trivial ramp-up.
    - React Native renders to actual native UI components (UIView, TextView), so the end result looks and feels native to each platform. Flutter uses its own Skia rendering engine, which gives pixel-perfect control but can feel slightly "off" to users accustomed to platform conventions.
    - The npm ecosystem is a massive advantage -- need a payment SDK, analytics, or charting library? There is almost certainly a React Native wrapper. Flutter's pub.dev ecosystem is growing but still smaller.
    - Over-the-air updates via EAS Update or CodePush let you push bug fixes to users without going through app store review, which is a significant operational advantage that native apps and Flutter do not have out of the box.
    - The trade-off: for graphics-intensive apps (games, AR/VR, complex custom animations), fully native or Flutter may be better choices because they provide lower-level rendering control.

    **Follow-up:** Your React Native app has a screen that requires a complex custom camera overlay with real-time object detection. How would you handle this when the JS thread cannot keep up with 30fps frame processing?

    **Follow-up Answer:**
    - This is a classic case where you drop into native code. You would write a native module (Swift on iOS, Kotlin on Android) that handles the camera feed and ML inference directly on the native thread, then surfaces only the results (bounding boxes, labels) back to the JS layer for rendering.
    - With the new architecture (JSI/TurboModules), you can communicate synchronously between JS and native, which reduces the latency compared to the old async bridge. For the overlay rendering, use Reanimated worklets running on the UI thread so the bounding box animations do not depend on the JS thread at all.
    - In practice, companies like Shopify use this pattern: React Native for 90% of the app, native modules for the 10% that needs raw performance. The key is designing clear interfaces between the JS and native layers so they can evolve independently.
  </Accordion>

  <Accordion title="How does the React Native rendering pipeline differ from a web React application, and what are the performance implications?">
    **Strong Answer:**
    - In a web React app, the reconciler diffs the virtual DOM and produces DOM mutations that the browser renders. In React Native, the reconciler produces a tree of native view commands that get sent across the bridge (or via JSI in the new architecture) to the native rendering system. The native side creates actual UIView/android.view.View instances.
    - The critical performance implication is the thread model. Web React runs everything on a single main thread (with web workers as an opt-in escape hatch). React Native has at least three threads: the JS thread (your React code), the main/UI thread (native rendering and touch events), and the shadow thread (layout calculations via Yoga). If the JS thread is blocked (parsing large JSON, heavy computation), the UI thread still handles touch events, but your JS callbacks will not fire -- making the app feel unresponsive.
    - This is why `useNativeDriver: true` on animations is critical: it sends the entire animation description to the native side once, so all 60 frames per second are calculated on the UI thread without crossing the bridge. Without it, each frame requires a JS-to-native round trip that often cannot complete within the 16ms budget.
    - The new architecture (Fabric + JSI) collapses the bridge into direct C++ bindings, enabling synchronous layout measurements and concurrent React features. This is a fundamental shift -- the old bridge was like sending letters between offices, JSI is like having both people in the same room speaking the same language.

    **Follow-up:** A user reports that scrolling a FlatList with 500 items is janky only on a budget Android phone but smooth on an iPhone 14. Walk me through your debugging approach.

    **Follow-up Answer:**
    - Start by connecting the Android Profiler in Android Studio to see if the bottleneck is the JS thread (slow renders) or the UI thread (complex view hierarchy). Use the React DevTools Profiler to identify which list items are re-rendering unnecessarily.
    - Common culprits on low-end Android: not using `React.memo` on list items, inline arrow functions in `renderItem` creating new closures every render, missing `getItemLayout` forcing the list to measure every item dynamically, and heavy images without caching.
    - Switch from FlatList to FlashList (by Shopify), which uses a RecyclerView-based approach and consistently delivers better performance on Android. Set `estimatedItemSize`, use `keyExtractor`, and ensure `removeClippedSubviews` is enabled.
    - On the image side, use `expo-image` with blurhash placeholders and `cachePolicy: 'memory-disk'`. Budget Android devices often have limited GPU memory, so large unoptimized images cause frame drops during scroll.
  </Accordion>

  <Accordion title="Explain the difference between Expo managed workflow and bare React Native CLI. When would you eject from Expo?">
    **Strong Answer:**
    - Expo managed workflow provides a curated set of native modules, cloud build services (EAS Build), and over-the-air updates. You never touch Xcode or Android Studio directly. The bare CLI gives you the `ios/` and `android/` directories with full native code access, but you are responsible for native configuration, signing, and builds.
    - The historical advice was "start with Expo, eject when you need custom native code." That advice is outdated. Modern Expo supports "development builds" (custom Expo Go clients with your native modules baked in) and config plugins that modify native code without ejecting. Companies like Shopify, Discord, and Coinbase use Expo in production.
    - You would use bare CLI when: integrating React Native into an existing native app (brownfield), needing a native SDK that has no Expo wrapper and cannot be added via a config plugin, or when your team has strong native expertise and wants full control over the build pipeline.
    - The key trade-off is operational complexity. Expo handles code signing, provisioning profiles, and build environments in the cloud. With bare CLI, your team must maintain those processes, which is a significant ongoing cost -- especially for iOS, where certificate management is notoriously painful.

    **Follow-up:** You are mid-project with Expo and discover you need a native SDK that requires modifying the Android `MainApplication.java`. What are your options without fully ejecting?

    **Follow-up Answer:**
    - First, check if an Expo config plugin already exists for the SDK. Config plugins let you modify native files (AndroidManifest.xml, MainApplication, Info.plist) at build time without maintaining native code in your repository.
    - If no plugin exists, write a custom config plugin. It is a JavaScript function that receives the Expo config and returns modified native configuration. This is the modern replacement for ejecting.
    - If the SDK requires runtime native code (not just configuration), create a local Expo module using `expo-modules-core`. This lets you write Swift/Kotlin code that integrates with the Expo module system while keeping the managed workflow.
    - As a last resort, switch to a development build (`npx expo run:android`) which gives you the `android/` directory while still using Expo's module system. This is not ejecting -- you can still use EAS Build and EAS Update.
  </Accordion>
</AccordionGroup>
