# Binary Search

<img src="/images/courses/cp/binary-search-visualization.svg" alt="Binary Search Visualization" />

## The Mental Model

Binary search is like playing "guess the number" optimally. Instead of guessing randomly, you always guess the middle. With each guess, you eliminate half the remaining possibilities. This is why binary search is O(log n)---even with a billion elements, you need at most 30 guesses (since 2^30 > 10^9).

Think of it like searching a physical dictionary. You would never start at page 1 and flip forward. You open roughly to the middle, check whether your word is before or after, and repeat. Each step cuts the remaining pages in half: 1000 pages becomes 500, then 250, then 125... After just 10 splits you are down to a single page.

But here is the deeper insight that separates beginners from experts: binary search is not really about "sorted arrays." It is about **monotonic predicates**. Any time you have a yes/no question where the answer flips from "no" to "yes" (or vice versa) at a single threshold, binary search can find that threshold. The sorted array is just the most obvious instance of this. Binary search on the answer---where you search over possible output values rather than array indices---is the more powerful and more frequently tested pattern on Codeforces.

<Note>
**Pattern Recognition Signals**:
- "Minimum/Maximum value that satisfies condition"
- "Find first/last occurrence"
- Sorted array or **monotonic property**
- Constraints allow O(log n) or O(n log n)
- "What is the largest X such that..." or "What is the smallest Y such that..."
</Note>

## The Two Types of Binary Search

### Type 1: Binary Search on Index

Find an element or position in a sorted array.

### Type 2: Binary Search on Answer

The answer is a number in a range. Check if each candidate answer is valid. The validity forms a monotonic sequence (all NO, then all YES, or vice versa).

<Tip>
**The Insight**: Binary search works whenever you have a **monotonic predicate**—a condition that's false for all values up to some point, then true for all values after (or vice versa).
</Tip>

## Template: Binary Search on Index

### Finding Exact Element

```cpp
int binarySearch(vector<int>& arr, int target) {
    int lo = 0, hi = arr.size() - 1;
    
    while (lo <= hi) {
        // Use lo + (hi - lo) / 2 instead of (lo + hi) / 2
        // to prevent integer overflow when lo + hi > INT_MAX
        int mid = lo + (hi - lo) / 2;
        
        if (arr[mid] == target) {
            return mid;           // Found: return index
        } else if (arr[mid] < target) {
            lo = mid + 1;         // Target is in right half
        } else {
            hi = mid - 1;         // Target is in left half
        }
    }
    
    return -1;  // Not found
}
```

### Finding Lower Bound (First >= Target)

```cpp
int lowerBound(vector<int>& arr, int target) {
    int lo = 0, hi = arr.size();
    
    while (lo < hi) {
        int mid = lo + (hi - lo) / 2;
        
        if (arr[mid] < target) {
            lo = mid + 1;
        } else {
            hi = mid;  // Don't exclude mid, it might be the answer
        }
    }
    
    return lo;  // First position where arr[i] >= target
}
```

### Finding Upper Bound (First > Target)

```cpp
int upperBound(vector<int>& arr, int target) {
    int lo = 0, hi = arr.size();
    
    while (lo < hi) {
        int mid = lo + (hi - lo) / 2;
        
        if (arr[mid] <= target) {
            lo = mid + 1;
        } else {
            hi = mid;
        }
    }
    
    return lo;  // First position where arr[i] > target
}
```

## Template: Binary Search on Answer

This is the **game-changer** pattern. Many problems that seem impossible become easy with this technique.

The core idea: instead of searching for an element in a sorted array, you search for the best value in a range of possible answers. You define a function `canAchieve(x)` that returns true/false, and because the predicate is monotonic (all false then all true, or vice versa), binary search finds the boundary.

**Analogy**: Imagine you are testing how much weight a shelf can hold. You know it can hold at least 0 kg and at most 100 kg. You test 50 kg--it holds. Test 75 kg--it breaks. Test 62 kg--it holds. You are binary searching on the answer: "what is the maximum weight the shelf supports?"

```cpp
// General template
bool canAchieve(int x) {
    // Return true if 'x' is achievable/valid
    // This function must be monotonic!
}

int binarySearchOnAnswer() {
    int lo = MIN_POSSIBLE_ANSWER;
    int hi = MAX_POSSIBLE_ANSWER;
    int answer = -1;
    
    while (lo <= hi) {
        int mid = lo + (hi - lo) / 2;
        
        if (canAchieve(mid)) {
            answer = mid;       // mid is valid, remember it
            hi = mid - 1;       // Try to find smaller valid (for minimum)
            // lo = mid + 1;    // Try to find larger valid (for maximum)
        } else {
            lo = mid + 1;       // Need larger value
            // hi = mid - 1;    // Need smaller value (for maximum)
        }
    }
    
    return answer;
}
```

## Classic Pattern: Minimum Maximum

**Problem Type**: "Minimize the maximum" or "Maximize the minimum"

**Example**: Split array into K subarrays to minimize the maximum subarray sum.

**Approach**: Binary search on the answer (the maximum sum). For each candidate, check if we can split into ≤ K subarrays.

```cpp
// Greedy check: can we split arr into <= k subarrays 
// where each subarray sum is at most maxSum?
bool canSplit(vector<int>& arr, int maxSum, int k) {
    int count = 1;        // Start with one subarray
    int currentSum = 0;
    
    for (int x : arr) {
        if (x > maxSum) return false;  // Single element exceeds limit
        
        if (currentSum + x > maxSum) {
            count++;              // Start a new subarray
            currentSum = x;       // This element starts the new subarray
        } else {
            currentSum += x;      // Add to current subarray
        }
    }
    
    return count <= k;
}

int splitArray(vector<int>& nums, int k) {
    // lo = at minimum, the answer must fit the largest single element
    int lo = *max_element(nums.begin(), nums.end());
    // hi = at maximum, one subarray holds everything
    int hi = accumulate(nums.begin(), nums.end(), 0);
    
    // Binary search on the answer: minimize the maximum subarray sum
    // Monotonic property: if canSplit(mid) is true, canSplit(mid+1) is also true
    while (lo < hi) {
        int mid = lo + (hi - lo) / 2;
        
        if (canSplit(nums, mid, k)) {
            hi = mid;      // mid works, try smaller maximum
        } else {
            lo = mid + 1;  // mid too small, need larger
        }
    }
    
    return lo;
}
// Time: O(n log(sum - max)), Space: O(1)
```

**Worked Example**: nums = [7, 2, 5, 10, 8], k = 2
- lo = 10 (max element), hi = 32 (total sum)
- mid = 21: canSplit? [7,2,5] sum=14, [10,8] sum=18. count=2 <= 2. YES. hi=21
- mid = 15: canSplit? [7,2,5] sum=14, [10] sum=10, oops count=2 at [10,8]? 10+8=18>15, so [10],[8]. count=3 > 2. NO. lo=16
- mid = 18: canSplit? [7,2,5] sum=14, [10,8] sum=18. count=2 <= 2. YES. hi=18
- mid = 17: canSplit? [7,2,5] sum=14, [10] then 10+8=18>17, [8]. count=3 > 2. NO. lo=18
- lo == hi == 18. Answer: 18.

<Tip>
**Contest Tip**: The "minimize the maximum" pattern is one of the most common binary search on answer applications on Codeforces (rating 1200-1600). The moment you see "split/distribute/assign into groups and minimize the worst group," think binary search. The check function is almost always greedy.
</Tip>

## Classic Pattern: Kth Element

**Problem**: Find the Kth smallest element in sorted matrix / merged sorted arrays.

**Approach**: Binary search on the value. Count how many elements are ≤ mid.

```cpp
int kthSmallest(vector<vector<int>>& matrix, int k) {
    int n = matrix.size();
    int lo = matrix[0][0];
    int hi = matrix[n-1][n-1];
    
    while (lo < hi) {
        int mid = lo + (hi - lo) / 2;
        
        // Count elements <= mid
        int count = 0;
        int col = n - 1;
        for (int row = 0; row < n; row++) {
            while (col >= 0 && matrix[row][col] > mid) {
                col--;
            }
            count += col + 1;
        }
        
        if (count < k) {
            lo = mid + 1;
        } else {
            hi = mid;
        }
    }
    
    return lo;
}
```

## Common Mistakes

<Warning>
**Mistake 1: Wrong Loop Condition**
```cpp
// For exact search
while (lo <= hi)  // Use <= 

// For lower/upper bound
while (lo < hi)   // Use <
```
</Warning>

<Warning>
**Mistake 2: Integer Overflow in Mid Calculation**
```cpp
// WRONG: Can overflow if lo + hi > INT_MAX
int mid = (lo + hi) / 2;

// CORRECT
int mid = lo + (hi - lo) / 2;
```
</Warning>

<Warning>
**Mistake 3: Wrong Bounds Update**
When searching for minimum valid:
- If `canAchieve(mid)` is true → `hi = mid` (not `mid - 1`)
- If false → `lo = mid + 1`

When searching for maximum valid:
- If `canAchieve(mid)` is true → `lo = mid + 1`
- If false → `hi = mid - 1`
</Warning>

<Warning>
**Mistake 4: Non-Monotonic Predicate**
Binary search only works if the predicate is monotonic. Always verify:
- If canAchieve(x) is true, is canAchieve(x+1) also true? (or vice versa)
</Warning>

## The Binary Search Decision Tree

```
Need to find something in sorted array?
├── Exact element → Standard binary search (lo <= hi)
├── First >= target → Lower bound
├── First > target → Upper bound
└── Last <= target → Upper bound - 1

Need to optimize (min/max) something?
├── Can you define canAchieve(x)?
│   ├── Is it monotonic? → Binary search on answer
│   └── Not monotonic → Different technique
└── No clear predicate → Not binary search
```

## Advanced: Ternary Search

For **unimodal functions** (one peak/valley), use ternary search. A unimodal function is one that strictly increases up to some point, then strictly decreases (or vice versa). Binary search on the answer does not work here because there is no monotonic predicate--both sides of the peak satisfy "function is increasing." Ternary search handles this by comparing two probe points to determine which third of the range to eliminate.

<Tip>
**Contest tip**: For floating-point ternary search, use a fixed iteration count (100 iterations gives precision ~10^-30) instead of `while (hi - lo > EPS)`. This avoids edge cases with floating-point comparison and is simpler to get right under pressure.
</Tip>

```cpp
// Find maximum of unimodal function f
double ternarySearch(double lo, double hi) {
    for (int i = 0; i < 100; i++) {  // 100 iterations = sufficient precision
        double m1 = lo + (hi - lo) / 3;   // First third point
        double m2 = hi - (hi - lo) / 3;   // Second third point
        
        if (f(m1) < f(m2)) {
            lo = m1;   // Peak is in the right 2/3
        } else {
            hi = m2;   // Peak is in the left 2/3
        }
    }
    
    return (lo + hi) / 2;
}
```

## Interview Deep-Dive

<AccordionGroup>
  <Accordion title="You are given N cows and M stalls along a number line. Place N cows in M stalls to maximize the minimum distance between any two cows. Describe your approach.">
    **Strong Answer:**

    - This is the classic "maximize the minimum" binary search on answer problem. The answer (minimum distance) is some value D. I binary search on D, and for each candidate, check whether I can place all N cows with at least distance D between consecutive ones.
    - Search space: lo = 0 (cows at the same stall), hi = (max_stall - min_stall) / (N-1) (evenly spaced, the theoretical maximum). Actually, hi = max_stall - min_stall is safe.
    - Check function (greedy): sort the stalls. Place the first cow in the first stall. For each subsequent cow, find the next stall that is at least D away from the last placed cow. If I can place all N cows, return true.
    - Monotonicity proof: if I can place cows with minimum distance D, I can certainly place them with minimum distance D-1 (less restrictive). So canPlace(D) is monotonic: true for small D, false for large D. Binary search finds the transition point.
    - Complexity: O(M log M) to sort stalls once. Binary search makes O(log(max_distance)) iterations, each running the O(M) greedy check. Total: O(M log M + M log(max_distance)).

    **Follow-up:** What is the key insight that makes the check function greedy rather than requiring DP or backtracking?

    The greedy works because we are maximizing the minimum. Once we commit to placing a cow at the earliest possible valid stall, there is no benefit to delaying -- placing it later cannot help and might prevent placing future cows. This is the greedy choice property: the locally optimal decision (place as early as possible) is globally optimal. The proof is by exchange argument: if the optimal solution skips a valid stall, we can swap to use it instead without reducing any pairwise distance.
  </Accordion>

  <Accordion title="Explain the difference between lower_bound and upper_bound in C++ STL. Then show how to use them to count elements in a sorted array that fall within a range [a, b].">
    **Strong Answer:**

    - `lower_bound(begin, end, x)` returns an iterator to the first element >= x. `upper_bound(begin, end, x)` returns an iterator to the first element > x. The difference is whether the target value itself is included.
    - To count elements in range [a, b]: `upper_bound(all(arr), b) - lower_bound(all(arr), a)`. This gives the number of elements where a <= element <= b. `lower_bound` points to the first element >= a (start of range), and `upper_bound` points past the last element <= b (end of range). The difference is the count.
    - Common mistakes: (1) Using lower_bound on an unsorted array -- undefined behavior. (2) Confusing the two: lower_bound finds first >= x, upper_bound finds first > x. To find the last element <= x, use `upper_bound(x) - 1` (check it is not begin first). (3) Forgetting that both return iterators, not indices -- subtract begin to get the index.
    - Related pattern: counting occurrences of value x = `upper_bound(all(arr), x) - lower_bound(all(arr), x)`. Or equivalently, `equal_range(all(arr), x)` returns both at once.

    **Follow-up:** How do lower_bound and upper_bound relate to the manual binary search templates you would write?

    lower_bound IS the "find first >= target" binary search: `while (lo < hi) { mid = lo + (hi-lo)/2; if (arr[mid] < target) lo = mid+1; else hi = mid; }`. upper_bound IS the "find first > target" binary search: same but with `arr[mid] <= target` in the condition. The STL functions are just well-tested implementations of these. In contests, I use the STL functions for simple lookups and write manual binary search only for binary-search-on-answer or complex predicates that do not fit the STL interface.
  </Accordion>

  <Accordion title="When would you use ternary search instead of binary search? Explain the difference and give a concrete example.">
    **Strong Answer:**

    - Binary search requires a monotonic predicate (all false then all true, or vice versa). Ternary search requires a unimodal function (strictly increases to a peak, then strictly decreases -- or the reverse for a valley).
    - Ternary search evaluates two probe points (at 1/3 and 2/3 of the range), compares their function values, and eliminates the 1/3 of the range that cannot contain the peak. Each iteration removes 1/3 of the search space, so it converges in O(log_{3/2}(range)) = O(log(range)) iterations (with a slightly larger constant than binary search).
    - Concrete example: "find the point on the x-axis that minimizes the maximum distance to a set of given points." The function f(x) = max distance to any point from x is V-shaped (convex), with a minimum at some point. Binary search on the answer does not directly work because f is not monotonic. Ternary search on x finds the minimum.
    - For floating-point ternary search, I use a fixed iteration count (100 iterations) rather than an epsilon-based termination condition, because floating-point comparison with epsilon is error-prone. 100 iterations give precision around 10^-30, which is far beyond any problem's requirement.
    - Important nuance: ternary search on integers is trickier because the function might be flat at the peak. Ensure the termination condition handles lo + 1 >= hi correctly.

    **Follow-up:** Can you always replace ternary search with binary search on the derivative?

    For differentiable functions, yes. If f is unimodal with a maximum, then f'(x) changes sign from positive to negative at the peak. Binary search on "is f'(x) > 0?" finds the sign change, which is the peak. This is often simpler and faster (each iteration eliminates half, not 1/3). For discrete functions where the derivative is not well-defined, ternary search on the function values is more natural.
  </Accordion>
</AccordionGroup>
