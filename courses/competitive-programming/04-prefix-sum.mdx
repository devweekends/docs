# Prefix Sum & Difference Arrays

## The Mental Model

Imagine you are driving on a highway with mile markers. If you want to know the distance between mile marker 35 and mile marker 87, you do not need to drive the whole stretch---you just subtract: 87 - 35 = 52 miles. A prefix sum array is exactly those mile markers for your data. You precompute the cumulative sum at every position, and then any range query becomes a single subtraction.

Another way to think about it: instead of re-adding grocery items from scratch every time a customer asks "what's my total so far?", you keep a running tally. Prefix sums precompute that tally so **range queries become O(1) after O(n) preprocessing**.

<Note>
**Pattern Recognition Signal**: Whenever you see "sum of elements from index L to R" or "multiple range queries on static array", think **Prefix Sum**. If you see the word "subarray" combined with "sum" and the constraints are n up to 10^5 or more, prefix sums (often paired with a hashmap) are almost certainly part of the solution.
</Note>

## The Core Idea

### Building Prefix Sum Array

```
Original:  [3, 1, 4, 1, 5, 9, 2, 6]
Prefix:    [0, 3, 4, 8, 9, 14, 23, 25, 31]
            ^-- We add a 0 at the start for easier math
```

**Key Insight**: `sum(L, R) = prefix[R+1] - prefix[L]`

Why? Because `prefix[R+1]` contains sum of elements 0 to R, and `prefix[L]` contains sum of elements 0 to L-1. Subtracting gives us exactly L to R.

---

## Implementation Template

```cpp
// Build prefix sum array
vector<long long> buildPrefix(const vector<int>& arr) {
    int n = arr.size();
    vector<long long> prefix(n + 1, 0);
    for (int i = 0; i < n; i++) {
        prefix[i + 1] = prefix[i] + arr[i];
    }
    return prefix;
}

// Query sum from index L to R (0-indexed, inclusive)
long long rangeSum(const vector<long long>& prefix, int L, int R) {
    return prefix[R + 1] - prefix[L];
}

// Usage
vector<int> arr = {3, 1, 4, 1, 5, 9, 2, 6};
auto prefix = buildPrefix(arr);
cout << rangeSum(prefix, 2, 5);  // sum of arr[2..5] = 4+1+5+9 = 19
```

## Pattern 1: Subarray Sum Equals K

**Problem**: Count subarrays with sum equal to K.

**Identification Signals**:
- "Count subarrays" or "find subarray"
- "Sum equals K"
- Constraints allow O(n) solution

**The Insight**: If `prefix[j] - prefix[i] = K`, then subarray `(i, j]` has sum K. Rearranging: `prefix[i] = prefix[j] - K`. So for each position j, we need to count how many earlier positions i had `prefix[i] = currentSum - K`. A hashmap does this in O(1) per lookup.

**Complexity**: O(n) time (one pass, O(1) hashmap operations per element), O(n) space for the hashmap.

```cpp
int subarraySum(vector<int>& nums, int k) {
    unordered_map<long long, int> prefixCount;
    prefixCount[0] = 1;  // Empty prefix has sum 0 (handles subarrays starting at index 0)
    
    long long currentSum = 0;
    int count = 0;
    
    for (int num : nums) {
        currentSum += num;
        
        // How many earlier prefix sums equal currentSum - k?
        // Each one represents a subarray ending here with sum exactly k.
        if (prefixCount.count(currentSum - k)) {
            count += prefixCount[currentSum - k];
        }
        
        prefixCount[currentSum]++;  // Record this prefix sum for future positions
    }
    
    return count;
}
```

**Worked Example**: nums = [1, 2, 3, -2, 5], k = 5
- i=0: sum=1, need 1-5=-4. Not found. Record {0:1, 1:1}
- i=1: sum=3, need 3-5=-2. Not found. Record {0:1, 1:1, 3:1}
- i=2: sum=6, need 6-5=1. Found once (prefix sum 1 at i=0). count=1. Subarray: [2,3]
- i=3: sum=4, need 4-5=-1. Not found. Record {0:1, 1:1, 3:1, 6:1, 4:1}
- i=4: sum=9, need 9-5=4. Found once. count=2. Subarray: [-2,5] wait, 3+(-2)+5=6 not 5. Actually the subarray is (i=3 to i=4] meaning sum(4..4)=9-4=5. That is [5]. Hmm let me recalculate...

Actually: prefix sums are [0, 1, 3, 6, 4, 9]. At i=4 (sum=9), we look for 9-5=4, which exists (at position 3). So the subarray is indices 4..4 = [5]. And at i=2 (sum=6), we look for 6-5=1, found at position 0. Subarray is indices 1..2 = [2, 3]. count=2.

## Pattern 2: 2D Prefix Sum

**Problem**: Answer multiple rectangle sum queries on a 2D grid.

**The Insight**: Build a 2D prefix sum where `prefix[i][j]` = sum of all elements in the rectangle from (0,0) to (i-1, j-1).

```cpp
// Build 2D prefix sum
vector<vector<long long>> build2DPrefix(const vector<vector<int>>& grid) {
    int n = grid.size(), m = grid[0].size();
    vector<vector<long long>> prefix(n + 1, vector<long long>(m + 1, 0));
    
    for (int i = 1; i <= n; i++) {
        for (int j = 1; j <= m; j++) {
            prefix[i][j] = grid[i-1][j-1] 
                         + prefix[i-1][j] 
                         + prefix[i][j-1] 
                         - prefix[i-1][j-1];
        }
    }
    return prefix;
}

// Query sum of rectangle from (r1,c1) to (r2,c2) inclusive
long long rectSum(const vector<vector<long long>>& prefix, 
                  int r1, int c1, int r2, int c2) {
    return prefix[r2+1][c2+1] 
         - prefix[r1][c2+1] 
         - prefix[r2+1][c1] 
         + prefix[r1][c1];
}
```

**Visual**: To get the sum of the green rectangle, we use inclusion-exclusion. Think of it like a Venn diagram with overlapping rectangles---if you subtract two overlapping areas, you have removed their intersection twice, so you add it back once:

```
+---+---+---+
| A | B |   |    prefix[r2+1][c2+1] = everything from (0,0) to (r2,c2)
+---+---+---+    prefix[r1][c2+1]   = everything above the target
| C | G |   |    prefix[r2+1][c1]   = everything left of the target
+---+---+---+    prefix[r1][c1]     = the overlap we subtracted twice
                  G = prefix[r2+1][c2+1] - prefix[r1][c2+1] - prefix[r2+1][c1] + prefix[r1][c1]
```

**Complexity**: Building the 2D prefix table takes O(n * m). Each rectangle sum query is O(1). This makes it ideal when you have many queries on a static grid.

<Tip>
**Contest Tip**: Drawing the 2D prefix sum formula on paper before coding saves you from sign errors. The inclusion-exclusion pattern (add total, subtract two sides, add back the corner) is easy to mess up under pressure if you try to remember it purely from memory.
</Tip>

## Pattern 3: Difference Array (Range Updates)

**Problem**: Apply multiple range updates, then query final array.

**Identification Signals**:
- "Add value to range [L, R]" multiple times
- Query final values after all updates
- No queries between updates

**The Insight**: Instead of updating each element in the range (which is O(R - L + 1) per update), you mark only the start and end of the change. Think of it like placing a "start adding 5" sign at position L and a "stop adding 5" sign at position R+1. When you later sweep left to right and accumulate these signs, you reconstruct the final array in a single pass.

Each range update is O(1). Reconstruction at the end is O(n). So Q updates on an array of size n costs O(Q + n) instead of the naive O(Q * n).

```cpp
// Apply updates: add 'val' to range [L, R]
// O(1) per update -- we only touch two positions
void rangeUpdate(vector<long long>& diff, int L, int R, long long val) {
    diff[L] += val;          // "Start adding val here"
    if (R + 1 < (int)diff.size()) {
        diff[R + 1] -= val;  // "Stop adding val here"
    }
}

// Reconstruct array from difference array
// O(n) -- one pass, accumulating the difference markers
vector<long long> reconstruct(const vector<long long>& diff) {
    vector<long long> arr(diff.size());
    arr[0] = diff[0];
    for (int i = 1; i < (int)diff.size(); i++) {
        arr[i] = arr[i-1] + diff[i];
    }
    return arr;
}

// Usage -- walkthrough:
int n = 5;
vector<long long> diff(n, 0);          // diff = [0, 0, 0, 0, 0]
rangeUpdate(diff, 1, 3, 5);            // diff = [0, 5, 0, 0, -5]
rangeUpdate(diff, 2, 4, 3);            // diff = [0, 5, 3, 0, -5]
auto result = reconstruct(diff);
// Accumulate: 0, 0+5=5, 5+3=8, 8+0=8, 8+(-5)=3
// result = [0, 5, 8, 8, 3]
```

<Tip>
**Contest Tip**: Difference arrays and prefix sums are inverses of each other. A prefix sum turns a difference array into the original array, and differencing turns a prefix sum array back. If you need range updates followed by range queries (no interleaving), apply difference arrays for the updates, reconstruct with prefix sum, then build a second prefix sum for the queries. This two-layer technique solves problems like [CF 295A - Greg and Array](https://codeforces.com/problemset/problem/295/A).
</Tip>

## Common Mistakes

<Warning>
**Mistake 1: Integer Overflow**
Prefix sums can grow large. Use `long long`:
```cpp
// WRONG
vector<int> prefix(n + 1, 0);

// CORRECT
vector<long long> prefix(n + 1, 0);
```
</Warning>

<Warning>
**Mistake 2: Off-by-One Errors**
Remember: `sum(L, R) = prefix[R+1] - prefix[L]` (not `prefix[R] - prefix[L-1]` with 1-indexed arrays, unless you adjust).
</Warning>

<Warning>
**Mistake 3: Forgetting to Handle Empty Prefix**
Initialize `prefixCount[0] = 1` for "subarray sum equals K" problems.
</Warning>

## Interview Deep-Dive

<AccordionGroup>
  <Accordion title="Given an array of integers and a target K, count the number of subarrays with sum exactly equal to K. The array may contain negative numbers. What is your approach and complexity?">
    **Strong Answer:**

    - This is the classic prefix-sum-plus-hashmap pattern. The key insight: if prefix[j] - prefix[i] = K, then the subarray from i+1 to j has sum K. Rearranging: prefix[i] = prefix[j] - K. So for each position j, I need to count how many earlier positions i had prefix[i] = currentSum - K.
    - Implementation: maintain a hashmap of prefix sum frequencies. Initialize with prefixCount[0] = 1 (the empty prefix). For each element, accumulate the running sum, look up (currentSum - K) in the hashmap, add that count to the answer, then increment prefixCount[currentSum].
    - Complexity: O(n) time with O(n) space for the hashmap. Each element is processed once with O(1) hashmap operations.
    - Why two pointers or sliding window fails here: those techniques require monotonicity (expanding the window always increases the sum). With negative numbers, expanding the window can decrease the sum, so the monotonic property breaks. The prefix-sum-hashmap approach handles negative numbers naturally.
    - Edge case: K = 0. The hashmap correctly counts subarrays where the prefix sum equals itself -- this happens when two positions have the same prefix sum.

    **Follow-up:** How would you modify this to find the longest subarray with sum K instead of counting all such subarrays?

    Instead of counting, store the first occurrence of each prefix sum. For each position j with currentSum, check if (currentSum - K) exists in the hashmap. If yes, the subarray length is j - firstOccurrence[currentSum - K]. Track the maximum length seen. Key difference: use a hashmap that stores the earliest index (do not overwrite), because we want the longest subarray, which means the earliest start.
  </Accordion>

  <Accordion title="Explain the difference array technique. You have 10^5 range updates of the form 'add 5 to all elements from index L to R.' How do you apply all updates and reconstruct the final array?">
    **Strong Answer:**

    - The naive approach applies each update by iterating from L to R and adding the value -- O(R-L+1) per update, O(n * Q) total, which for 10^5 updates on an array of 10^5 is 10^10 operations. Far too slow.
    - The difference array technique: instead of updating every element in the range, mark only the endpoints. For "add val to [L, R]": diff[L] += val and diff[R+1] -= val. Each update is O(1). After all updates, reconstruct the final array by computing the prefix sum of the difference array.
    - Why this works: the prefix sum accumulates the "starts" and "stops." When you encounter diff[L] = +val, the prefix sum increases by val and stays elevated. When you later encounter diff[R+1] = -val, the prefix sum drops back down. The net effect is that positions L through R are increased by val, and positions outside this range are unaffected.
    - Total complexity: O(Q) for all updates (O(1) each) plus O(n) for reconstruction. Total: O(Q + n), which for 10^5 each is 2 * 10^5 operations.
    - Connection to prefix sums: the difference array and prefix sum are inverse operations. Taking the prefix sum of a difference array gives the original array. Taking the difference of a prefix sum array gives the original array. This duality is fundamental to range update/query problems.

    **Follow-up:** What if you need to answer range sum queries BETWEEN updates, not just after all updates?

    Then you need a more powerful data structure. Options: (1) Fenwick Tree with range updates and point queries -- supports O(log n) per update and O(log n) per query. (2) Segment Tree with lazy propagation -- supports O(log n) range updates, O(log n) range queries, and handles interleaved updates and queries. The difference array technique only works when all updates come before all queries.
  </Accordion>

  <Accordion title="How does the 2D prefix sum work? Derive the formula for querying a rectangle sum, and explain why inclusion-exclusion is needed.">
    **Strong Answer:**

    - Build a 2D prefix table where prefix[i][j] = sum of all elements in the rectangle from (0,0) to (i-1, j-1). Building uses the recurrence: prefix[i][j] = grid[i-1][j-1] + prefix[i-1][j] + prefix[i][j-1] - prefix[i-1][j-1]. The subtraction removes the double-counted overlap.
    - To query the sum of rectangle (r1,c1) to (r2,c2): result = prefix[r2+1][c2+1] - prefix[r1][c2+1] - prefix[r2+1][c1] + prefix[r1][c1].
    - Why inclusion-exclusion: prefix[r2+1][c2+1] includes everything from (0,0) to (r2,c2). Subtracting prefix[r1][c2+1] removes the rows above the target rectangle. Subtracting prefix[r2+1][c1] removes the columns left of the target. But the top-left corner (above AND left) was subtracted twice, so we add it back with prefix[r1][c1].
    - This is exactly the inclusion-exclusion principle from combinatorics: |A union B| = |A| + |B| - |A intersect B|, applied to rectangular regions.
    - Complexity: O(nm) to build, O(1) per query. Essential for problems with many rectangle sum queries on a static grid.

    **Follow-up:** Can you extend this to 3D? What is the inclusion-exclusion formula for a 3D box query?

    Yes. For 3D prefix sums, the query for a box from (x1,y1,z1) to (x2,y2,z2) involves 8 terms: add the total box, subtract 3 faces, add back 3 edges, subtract the corner. The formula has alternating signs following the inclusion-exclusion pattern for 3 sets. Building takes O(n*m*k) and each query is O(1). In practice, 3D prefix sums appear in problems involving 3D grids or when you are doing prefix sums over two dimensions plus a value dimension.
  </Accordion>
</AccordionGroup>
