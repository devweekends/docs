# Fast I/O & Debugging

## Why I/O Speed Matters

In competitive programming, I/O can be the bottleneck. A problem with 10^6 integers using `cin/cout` without optimization will TLE, while the same code with fast I/O passes easily.

<Note>
**The I/O Bottleneck**: Standard `cin/cout` is synchronized with C's `scanf/printf` by default, making it significantly slower. Disable this sync to gain 5-10x speedup.
</Note>

## Fast I/O Techniques

### Pattern 1: Sync Disable (Essential)

```cpp
int main() {
    // MUST be at the start of main
    ios::sync_with_stdio(false);  // Disable sync with C I/O
    cin.tie(nullptr);             // Untie cin from cout
    
    // Now cin/cout are as fast as scanf/printf
}
```

**Why this works**: By default, C++ keeps its I/O streams (`cin`/`cout`) synchronized with C's streams (`stdin`/`stdout`). This synchronization means every `cin` call checks whether C's `scanf` also has pending input---a safety feature that costs significant overhead. `sync_with_stdio(false)` tells C++ "I promise I will not mix C and C++ I/O," allowing it to use its own faster buffering. `cin.tie(nullptr)` prevents `cout` from being flushed before every `cin` read, which removes another hidden cost. Together, these two lines can make I/O 5-10x faster.

<Warning>
**Important**: After `ios::sync_with_stdio(false)`, do NOT mix `cin/cout` with `scanf/printf` in the same program. Breaking this rule causes undefined behavior---input may be read in the wrong order or lost entirely.
</Warning>

### Pattern 2: Reading Entire Lines

```cpp
// Reading a line with spaces
string line;
getline(cin, line);

// After reading a number, clear the newline before getline
int n;
cin >> n;
cin.ignore();  // or getline(cin, line); to consume newline
getline(cin, line);
```

### Pattern 3: Reading Unknown Number of Inputs

```cpp
// Read until EOF
int x;
while (cin >> x) {
    // Process x
}

// Read n integers into vector
int n;
cin >> n;
vector<int> a(n);
for (int& x : a) cin >> x;

// Or using istream_iterator (fancy but slower)
vector<int> a(istream_iterator<int>(cin), istream_iterator<int>());
```

### Pattern 4: Fast Output

```cpp
// Use '\n' instead of endl (endl flushes buffer)
cout << ans << '\n';  // Fast - just adds a character to the buffer
cout << ans << endl;  // Slow - writes newline AND forces the buffer to disk

// For many outputs, build string then print once
string result;
for (int i = 0; i < n; i++) {
    result += to_string(arr[i]) + " ";
}
cout << result << '\n';
```

## Debugging Techniques

### Pattern 1: Debug Macro

```cpp
#ifdef LOCAL
#define debug(x) cerr << #x << " = " << (x) << '\n'
#define debugv(v) cerr << #v << " = "; for (auto& x : v) cerr << x << ' '; cerr << '\n'
#else
#define debug(x)
#define debugv(v)
#endif

// Usage
int x = 5;
debug(x);  // Prints: x = 5 (only when compiled with -DLOCAL)

vector<int> v = {1, 2, 3};
debugv(v);  // Prints: v = 1 2 3
```

Compile locally with: `g++ -DLOCAL solution.cpp`

### Pattern 2: Visualizing 2D Arrays

```cpp
void print2D(vector<vector<int>>& grid) {
    cerr << "Grid:\n";
    for (auto& row : grid) {
        for (int x : row) {
            cerr << setw(4) << x;
        }
        cerr << '\n';
    }
    cerr << '\n';
}
```

### Pattern 3: Assert for Assumptions

```cpp
// Crash immediately if assumption is wrong
assert(n > 0 && "n must be positive");
assert(arr.size() == n);

// Custom assertion with message
#define ASSERT(cond, msg) \
    if (!(cond)) { cerr << "Assertion failed: " << msg << '\n'; exit(1); }

ASSERT(left <= right, "Invalid range");
```

## Common I/O Patterns in CP

### Pattern: Multi-Test Case

```cpp
void solve() {
    int n;
    cin >> n;
    // Solve single test case
    cout << answer << '\n';
}

int main() {
    ios::sync_with_stdio(false);
    cin.tie(nullptr);
    
    int t;
    cin >> t;
    while (t--) {
        solve();
    }
}
```

### Pattern: Graph Input

```cpp
// n nodes, m edges
int n, m;
cin >> n >> m;

// Adjacency list
vector<vector<int>> adj(n + 1);  // 1-indexed
for (int i = 0; i < m; i++) {
    int u, v;
    cin >> u >> v;
    adj[u].push_back(v);
    adj[v].push_back(u);  // For undirected graph
}

// Weighted edges
vector<vector<pair<int, int>>> adj(n + 1);
for (int i = 0; i < m; i++) {
    int u, v, w;
    cin >> u >> v >> w;
    adj[u].push_back({v, w});
    adj[v].push_back({u, w});
}
```

### Pattern: Matrix Input

```cpp
int n, m;
cin >> n >> m;

// 2D vector
vector<vector<int>> grid(n, vector<int>(m));
for (int i = 0; i < n; i++) {
    for (int j = 0; j < m; j++) {
        cin >> grid[i][j];
    }
}

// Character grid
vector<string> grid(n);
for (int i = 0; i < n; i++) {
    cin >> grid[i];
}
// Access: grid[i][j]
```

## Avoiding Common Mistakes

### Mistake 1: Integer Overflow

```cpp
// WRONG: Overflow when n > 46340
int n = 100000;
int result = n * n;  // Overflow!

// CORRECT: Cast to long long
long long result = (long long)n * n;
// Or use ll from the start
ll n = 100000;
ll result = n * n;
```

### Mistake 2: Array Index Out of Bounds

```cpp
// WRONG: Accessing arr[n] when array has n elements
vector<int> arr(n);
cout << arr[n];  // Undefined behavior!

// CORRECT: Valid indices are 0 to n-1
cout << arr[n-1];
```

### Mistake 3: Uninitialized Variables

```cpp
// WRONG: Uninitialized
int sum;
for (int x : arr) sum += x;

// CORRECT: Initialize
int sum = 0;
for (int x : arr) sum += x;
```

### Mistake 4: Forgetting to Reset Global Variables

```cpp
// WRONG: Global variables not reset between test cases
int visited[100001];  // Global

void solve() {
    // visited still has values from previous test case!
}

// CORRECT: Reset at start of solve()
void solve() {
    memset(visited, 0, sizeof(visited));
    // or use local variables
    vector<int> visited(n + 1, 0);
}
```

## Output Format Gotchas

### Precision for Floating Point

```cpp
// Default precision may not be enough
double ans = 1.0 / 3.0;
cout << ans << '\n';  // 0.333333

// Set precision
cout << fixed << setprecision(9) << ans << '\n';  // 0.333333333
```

### Yes/No Output

```cpp
// Check problem statement for exact format!
// "YES"/"NO" vs "Yes"/"No" vs "yes"/"no"
cout << (condition ? "YES" : "NO") << '\n';
```

### Multiple Values on Same Line

```cpp
// Space-separated
for (int i = 0; i < n; i++) {
    cout << arr[i] << " \n"[i == n - 1];  // Space, then newline at end
}

// Or simpler
for (int i = 0; i < n; i++) {
    if (i) cout << ' ';
    cout << arr[i];
}
cout << '\n';
```

## Stress Testing

When your solution gets WA and you can't find the bug:

```cpp
// generate.cpp - Generates random test cases
#include <bits/stdc++.h>
using namespace std;

int main(int argc, char* argv[]) {
    srand(atoi(argv[1]));  // Seed from command line
    
    int n = rand() % 10 + 1;  // Small n for testing
    cout << n << '\n';
    for (int i = 0; i < n; i++) {
        cout << rand() % 100 << ' ';
    }
    cout << '\n';
}
```

```bash
# stress.sh - Run until mismatch
for ((i = 1; ; i++)); do
    ./generate $i > input.txt
    ./solution < input.txt > output1.txt
    ./brute < input.txt > output2.txt
    if ! diff -q output1.txt output2.txt > /dev/null; then
        echo "Mismatch found on test $i!"
        cat input.txt
        break
    fi
    echo "Test $i passed"
done
```

## Interview Deep-Dive

<AccordionGroup>
  <Accordion title="Why does ios::sync_with_stdio(false) speed up C++ I/O by 5-10x? What is actually happening at the system level, and what constraint does it impose on your code?">
    **Strong Answer:**

    - By default, C++ synchronizes its iostream buffers with C's stdio buffers. This means every cin/cout operation checks whether scanf/printf has pending data, maintaining a single shared buffer. This synchronization involves locking, checking, and potentially flushing between two independent I/O systems -- significant overhead per operation.
    - Calling `sync_with_stdio(false)` tells C++ to stop synchronizing. Now cin/cout use their own independent buffer, which is much faster because it eliminates the cross-system checks. `cin.tie(nullptr)` further decouples cin from cout, preventing automatic cout flushes before each cin read.
    - The constraint: after disabling sync, you must NOT mix C I/O (scanf/printf) with C++ I/O (cin/cout) in the same program. The two systems now use separate buffers, so mixing them causes reads to happen in the wrong order or data to be lost. In competitive programming this is rarely a problem since we use one style consistently.
    - Practical impact: for a problem reading 10^6 integers, cin without sync_with_stdio takes about 3-4 seconds, while cin with sync_with_stdio(false) takes about 0.3-0.5 seconds. This difference alone can be the gap between AC and TLE.

    **Follow-up:** When would you still use printf/scanf instead of cin/cout even with fast I/O enabled?

    For extremely tight time limits or very large I/O (10^7+ values), printf/scanf can still be marginally faster because they bypass C++'s stream formatting layer entirely. Some competitive programmers use a custom read function with getchar_unlocked() for the absolute fastest I/O -- reading characters directly and parsing integers manually. But for 99% of problems, cin/cout with sync disabled is sufficient and much more readable.
  </Accordion>

  <Accordion title="You get TLE on a problem and your algorithm is the correct complexity. What non-algorithmic optimizations would you check, and in what order?">
    **Strong Answer:**

    - First: I/O speed. Is `sync_with_stdio(false)` and `cin.tie(nullptr)` present? Am I using `endl` instead of `'\n'`? Each `endl` forces a buffer flush which is dramatically slower -- switching to `'\n'` can save seconds on problems with 10^5+ output lines.
    - Second: hidden O(n) operations inside loops. Am I passing vectors by value instead of by reference? Each pass-by-value copies the entire vector -- O(n) per call. Am I concatenating strings with `+=` in a loop? Each concatenation copies the growing string -- O(n^2) total. Am I using `substr()` inside a loop? Each call creates a new string.
    - Third: data structure constant factors. Am I using `map` (O(log n) with large constant) where an `unordered_map` (O(1) average) or direct array would work? Am I using `set::count()` instead of a boolean array for existence checks?
    - Fourth: compiler optimizations. Am I compiling with `-O2`? The optimization flag can speed up execution by 2-3x by enabling loop unrolling, inlining, and other optimizations. On online judges, this is usually enabled by default, but locally it might not be.
    - Fifth: cache efficiency. Processing a 2D array row-by-row is cache-friendly; column-by-column is cache-hostile. For large grids, the access pattern matters. Swapping loop order from `for(j) for(i)` to `for(i) for(j)` can halve execution time on cache-sensitive problems.

    **Follow-up:** How much does `endl` vs `'\n'` actually matter in practice?

    On a problem that outputs 10^6 lines, using `endl` flushes the buffer 10^6 times. Each flush is a system call. On a typical system, 10^6 system calls take 1-2 seconds. Using `'\n'` accumulates output in a buffer and flushes once at program termination -- essentially free. I have seen solutions go from 3.5 seconds (TLE) to 0.8 seconds (AC) just by switching from `endl` to `'\n'`. It is the single easiest optimization and should always be the default.
  </Accordion>

  <Accordion title="Explain the stress testing workflow from start to finish. A candidate says they 'never need stress testing because they check edge cases manually.' Convince them otherwise.">
    **Strong Answer:**

    - Manual edge case checking catches predictable bugs: n=0, n=1, all-same elements, sorted input. Stress testing catches unpredictable bugs: specific element orderings that trigger incorrect tie-breaking, boundary conditions in binary search for particular values, off-by-one errors that only manifest for certain array lengths.
    - The workflow: (1) write a brute force solution -- deliberately simple, O(n^2) or O(n^3), so simple it cannot be wrong. (2) Write a generator that produces random valid inputs with small n (1 to 10). (3) Run a loop: generate input, run both solutions, compare outputs, stop on first mismatch. Typically finds bugs within 100-500 iterations.
    - Why manual checking fails: for a problem where the answer depends on the relative order of elements, there are n! possible orderings. For n=8, that is 40320 orderings. You cannot manually check them all, but stress testing explores them randomly and efficiently.
    - Real example: I had a greedy solution for an interval problem that passed all 5 sample cases and 10 manual edge cases. Stress testing found a failing case at n=7 where three intervals shared a common endpoint -- a configuration I never would have thought to test manually. The bug was in my tie-breaking logic for intervals with equal end times.
    - The time investment: 5 minutes to set up (with pre-written templates), then it runs in the background while I think about the next problem. The expected value is enormous because a single caught bug saves 10-30 minutes of debugging.

    **Follow-up:** How do you write a correct brute force for a problem you cannot solve optimally?

    The brute force does not need to solve the problem efficiently -- it just needs to be obviously correct. For optimization problems, enumerate all possibilities: try all subsets (2^n), all permutations (n!), all pairs (n^2). For counting problems, enumerate and count manually. The key is keeping n tiny (n <= 10) so that even O(n!) runs in milliseconds. If you cannot write a correct brute force, you likely do not understand the problem statement itself, which is diagnostic information.
  </Accordion>
</AccordionGroup>
