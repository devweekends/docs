## <Icon icon="search" /> Question Type Patterns

### Pattern 1: "Count the number of..."

<CardGroup cols={2}>
  <Card title="Count subarrays with property" icon="calculator">
    **Technique:** Prefix sum + hashmap
    ```cpp
    // "Count subarrays with sum = k"
    map<int, int> prefix_count;
    prefix_count[0] = 1;
    int sum = 0, count = 0;
    for (int x : arr) {
        sum += x;
        count += prefix_count[sum - k];
        prefix_count[sum]++;
    }
    ```
  </Card>
  
  <Card title="Count subsequences with property" icon="layer-group">
    **Technique:** DP (usually 1D or 2D)
    ```cpp
    // "Count increasing subsequences of length k"
    // dp[i][j] = count ending at i, length j
    for (int i = 0; i < n; i++) {
        dp[i][1] = 1;
        for (int j = 0; j < i; j++) {
            if (a[j] < a[i]) {
                for (int k = 2; k <= K; k++) {
                    dp[i][k] += dp[j][k-1];
                }
            }
        }
    }
    ```
  </Card>
</CardGroup>

### Pattern 2: "Find the maximum/minimum..."

<img src="/images/courses/cp/optimize-pattern.svg" alt="Optimization Patterns" />

| What to Optimize | First Try | If TLE, Try |
|------------------|-----------|-------------|
| Max sum subarray | Kadane's O(n) | Already optimal |
| Min cost path | Dijkstra O(E log V) | 0-1 BFS if weights 0/1 |
| Max flow | Ford-Fulkerson | Dinic's algorithm |
| Max matching | Hungarian O(n³) | Hopcroft-Karp O(E√V) |
| Min operations | BFS | Binary search on answer |

### Pattern 3: "Check if possible..."

<CardGroup cols={2}>
  <Card title="Can we achieve X?" icon="check">
    **Technique:** Binary search on answer
    ```cpp
    // "Can we split array into k parts 
    // with max sum ≤ mid?"
    int lo = max_element, hi = total_sum;
    while (lo < hi) {
        int mid = (lo + hi) / 2;
        if (canSplit(arr, k, mid)) {
            hi = mid;
        } else {
            lo = mid + 1;
        }
    }
    return lo;
    ```
  </Card>
  
  <Card title="Does path/assignment exist?" icon="route">
    **Technique:** DFS/BFS or DP
    ```cpp
    // "Can we reach target?"
    bool dfs(int node, int target, 
             vector<bool>& visited) {
        if (node == target) return true;
        visited[node] = true;
        for (int next : adj[node]) {
            if (!visited[next] && 
                dfs(next, target, visited)) {
                return true;
            }
        }
        return false;
    }
    ```
  </Card>
</CardGroup>

### Pattern 4: "Find the kth..."

| Problem Type | Algorithm | Complexity |
|--------------|-----------|------------|
| Kth smallest in array | nth_element / quickselect | O(n) |
| Kth smallest in sorted arrays | Binary search | O(log(min(m,n))) |
| Kth smallest in BST | In-order traversal | O(k) |
| Kth smallest sum | Priority queue | O(k log k) |

<img src="/images/courses/cp/array-decision-tree.svg" alt="Array Problem Decision Tree" />

```
Q: What are you asked to find?
├── Sum/XOR of subarray?
│   ├── Single query → Prefix sum/XOR
│   └── Multiple queries → Prefix sum array
│
├── Maximum/Minimum in range?
│   ├── Static array → Sparse table
│   └── Updates allowed → Segment tree
│
├── Longest subarray with property?
│   ├── Two-pointer property → Two pointers
│   └── Complex property → DP
│

│   ├── Sum = k → Hashmap + prefix sum
│   ├── Fixed length → Sliding window
│   └── Other → DP or combinatorics
│
└── Reorder/partition array?
    ├── Minimize max → Binary search on answer
    └── Specific order → Greedy or DP
```

## <Icon icon="dumbbell" /> Pattern Recognition Drills

### Drill 1: Speed Classification

Read each problem description and identify the pattern in under 10 seconds:

<Accordion title="Problem Set (Click to expand)">
**P1:** "Given an array of n integers, find the number of pairs (i,j) where i < j and a[i] + a[j] = k."
- **Pattern:** Two Sum → Hashmap

**P2:** "Given a binary tree, find the maximum path sum from any node to any node."
- **Pattern:** Tree DP → DFS with return value

**P3:** "Given n intervals, find the minimum number of points such that each interval contains at least one point."
- **Pattern:** Interval covering → Sort by end + greedy

**P4:** "Given a string, find the longest palindromic substring."
- **Pattern:** Palindrome → DP or expand from center

**P5:** "Given a weighted graph, find if negative cycle exists."
- **Pattern:** Negative cycle → Bellman-Ford

**P6:** "Given array, answer queries for minimum in range [l,r]."
- **Pattern:** Range min query → Sparse table or segment tree

**P7:** "Given n items with weight and value, maximize value with capacity W."
- **Pattern:** 0/1 Knapsack → 2D DP

**P8:** "Given a sequence, count inversions (pairs where i < j but a[i] > a[j])."
- **Pattern:** Inversions → Merge sort or BIT
</Accordion>

## <Icon icon="bolt" /> Quick Reference Cheat Sheet

### One-Liner Pattern Recognition

```
Array + Sum → Prefix Sum
Array + Sorted + Target → Two Pointers / Binary Search
Array + Window Property → Sliding Window
Array + Subsequence → DP
Array + "Split into K groups" → Binary Search on Answer + Greedy check

String + Match → KMP / Hashing
String + Palindrome → DP / Manacher
String + Lexicographic → Greedy / Trie

Graph + Shortest → BFS (unweighted) / Dijkstra (weighted)
Graph + Connectivity → DFS / DSU
Graph + Minimum cost → MST (Kruskal/Prim)
Graph + Flow → Ford-Fulkerson / Dinic
Graph + Ordering → Topological Sort

Tree + Path → DFS
Tree + LCA → Binary Lifting
Tree + Subtree → Euler Tour
Tree + Diameter → Two BFS

"Number of ways" → DP + mod
"Minimum steps" → BFS
"Is it possible" → Binary Search on Answer / Greedy
"Maximum subset" → DP / Greedy
"Minimum/Maximum of ..." → Often Binary Search on Answer
"For each query" + static data → Precompute (prefix sum, sparse table)
"For each query" + updates → Segment Tree / Fenwick Tree
```

<Warning>
**The constraint is often more reliable than the problem statement for pattern recognition.** A problem about "arranging flowers" with n <= 20 is a bitmask problem, period---regardless of how botanical the story sounds. Train yourself to read constraints before the story.
</Warning>

## Interview Deep-Dive

<AccordionGroup>
  <Accordion title="Given a problem: 'Find the maximum number of non-overlapping intervals you can select from a list.' Walk me through your pattern recognition process from reading the problem to selecting the algorithm.">
    **Strong Answer:**

    - Keyword trigger: "non-overlapping intervals" plus "maximum number" immediately maps to the interval scheduling pattern. This is one of the most classic greedy problems in computer science.
    - Constraint check: if n <= 10^5 intervals, I need O(n log n). Sorting is O(n log n) and the greedy pass is O(n), so total is O(n log n). This fits.
    - Algorithm: sort intervals by end time, then greedily select each interval whose start time is >= the last selected interval's end time. This works because finishing early leaves maximum room for future intervals.
    - Why sort by end time and not start time or length? Sorting by start time fails: an early-starting but very long interval blocks many short ones. Sorting by length fails: a short interval might overlap many others. End time is the correct criterion because the interval that finishes earliest cannot be worse than any alternative.
    - Proof sketch (exchange argument): suppose the optimal solution O does not include the earliest-finishing interval E. Replace the first interval in O with E. Since E finishes no later, all subsequent intervals in O are still compatible. So the modified solution has the same count, proving E is safe to include.

    **Follow-up:** What if the problem changes to "select the minimum number of points such that every interval contains at least one point"?

    This is the dual problem -- interval piercing. Interestingly, the same greedy works: sort by end time, and whenever the current interval is not already pierced, place a point at its right endpoint. The point at the right end is optimal because it is most likely to also pierce subsequent overlapping intervals. The count of points placed equals the count of non-overlapping intervals, which is a beautiful duality.
  </Accordion>

  <Accordion title="You see a problem with constraint n <= 20. What algorithms should immediately come to mind, and how do you decide between them?">
    **Strong Answer:**

    - n <= 20 is the bitmask signal. 2^20 = 1,048,576, which is about 10^6 -- well within the 10^8 operations-per-second budget. This means I can enumerate all subsets of n elements.
    - Primary algorithms: bitmask DP (dp[mask] where mask represents which elements are included), meet-in-the-middle (split into two halves of 10, enumerate 2^10 states for each, combine), and backtracking with pruning.
    - Decision framework: if the problem asks "find the optimal way to partition/assign n items," bitmask DP is the go-to. If the problem asks "find a subset with sum closest to T" and n is exactly 20, meet-in-the-middle splits it into two groups of 10, enumerates all 1024 subset sums for each, and uses binary search to find the best combination -- O(2^(n/2) * n) which is about 10^4.
    - If n <= 15, I might also consider O(3^n) techniques like subset-sum DP over subsets and their complements. 3^15 = 14 million, which is borderline but usually passes.
    - The trap: sometimes n <= 20 is a red herring and there is a polynomial solution. If the problem can be solved in O(n^2) or O(n^3), do not over-engineer with bitmasks. Always check if a simpler approach exists first.

    **Follow-up:** Explain the meet-in-the-middle technique and when it outperforms bitmask DP.

    Meet-in-the-middle splits the problem into two halves, solves each exhaustively, and combines results. For "subset sum closest to T" with n=40, bitmask DP over all 2^40 subsets is impossible (10^12 states). But splitting into two groups of 20, each has 2^20 = 10^6 subsets. Enumerate all subset sums for each half, sort one half, and for each sum in the other half, binary search for the complement closest to T. Total: O(2^(n/2) * n) = about 2 * 10^7. Meet-in-the-middle outperforms bitmask DP whenever n is in the 30-40 range where 2^n is infeasible but 2^(n/2) is not.
  </Accordion>

  <Accordion title="How do you build a 'pattern library' in your head? What is the difference between someone who recognizes patterns instantly and someone who has to derive the approach from scratch every time?">
    **Strong Answer:**

    - The pattern library is built through deliberate categorization during upsolving. After solving (or reading the editorial for) every problem, I tag it with 1-2 pattern labels: "prefix sum + hashmap," "binary search on answer with greedy check," "BFS on implicit graph." Over time, these tags form clusters, and when I see a new problem, my brain pattern-matches against the cluster labels rather than re-deriving from first principles.
    - The instant recognizer has solved 10+ problems of each major pattern and can recall the pattern from structural cues in the problem statement. They see "minimum number of operations to make all elements equal" and instantly think "sort, pick median" because they have seen that exact reduction 5 times before. The derivation approach would require them to prove that the median minimizes absolute deviations every time -- correct but slow.
    - The practice that builds recognition fastest: after solving a problem, find 2-3 similar problems (same tag on Codeforces) and solve them the same day. This builds the cluster. A single exposure creates a fragile memory; 3-4 exposures in quick succession create a robust pattern.
    - Recognition has limits. Problems above 1800 rating often combine two patterns or require a novel twist on a known pattern. Pure recognition fails there -- you need the ability to decompose the problem into known sub-patterns and combine them. This is the "expert" level: recognizing the components and assembling them creatively.

    **Follow-up:** Can you give an example of a problem that combines two patterns, and how you would decompose it?

    "Find the minimum cost to split an array into K contiguous segments where the cost of a segment is the number of distinct elements in it." This combines two patterns: (1) DP over segments (dp[i][j] = min cost to split first i elements into j segments), and (2) sliding window or two pointers to efficiently compute the number of distinct elements as segment boundaries change. Neither pattern alone solves the problem. The decomposition insight is that the DP framework handles the "split into K parts" structure, while the sliding window efficiently computes the cost function that the DP needs. Total complexity: O(nK) with the right data structure for distinct counting.
  </Accordion>
</AccordionGroup>
