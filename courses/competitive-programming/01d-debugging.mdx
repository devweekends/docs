# Debugging & Stress Testing

## The Cost of Debugging in CP

Every minute spent debugging is a minute not solving the next problem. Learn to debug fast.

```
┌─────────────────────────────────────────────────────┐
│         TIME COST OF BUGS IN CONTEST                │
├─────────────────────────────────────────────────────┤
│  Wrong Answer (WA)                                  │
│    - Finding bug: 3-10 minutes                      │
│    - Penalty: varies by scoring system (see below)  │
│    - Total: significant time lost                   │
│                                                     │
│  Time Limit Exceeded (TLE)                          │
│    - Rethinking approach: 5-15 minutes              │
│    - Rewriting: 5-10 minutes                        │
│    - Total: 10-25 minutes lost                      │
└─────────────────────────────────────────────────────┘
```

<Note>
**Contest Scoring Systems on Codeforces**:

**ICPC Rules** (most Div 2/3/4 contests):
- Each wrong submission adds **+10 minute penalty**
- Final score = time of AC + (10 × wrong attempts)
- Earlier solve = better rank

**IOI Rules** (Educational rounds, some special contests):
- Partial scoring based on test cases passed
- No time penalty for wrong submissions
- Score = points earned (not time-based)
- Maximum points per problem (e.g., 100, 250, 500, etc.)

**Codeforces Scoring** (standard Div 1/2 rounds):
- Points decrease over time (max points at start)
- Each wrong submission costs **-50 points** (before AC)
- No penalty after AC
</Note>

## Prevention > Debugging

### Before You Code

<Steps>
  <Step title="Understand the Problem Completely">
    Re-read after you think you understand. Check examples manually.
  </Step>
  <Step title="Plan Your Approach">
    Think through edge cases before coding.
  </Step>
  <Step title="Choose Data Types Carefully">
    Will values overflow `int`? Use `long long` proactively.
  </Step>
  <Step title="Write Clean Code">
    Use meaningful variable names. Keep functions small.
  </Step>
</Steps>

### Common Bug Prevention

```cpp
// Use these by default
#define int long long  // Prevent overflow by making ALL ints 64-bit
// NOTE: When using this trick, you MUST change main() to:
//   int32_t main() { ... }
// because main() must return a 32-bit int per the C++ standard.
// Some compilers accept "signed main()" as well.

// Check array bounds explicitly
if (i >= 0 && i < n && j >= 0 && j < m) {
    // Safe to access grid[i][j]
}

// Initialize variables
int sum = 0;  // Not just: int sum;

// Use const for array sizes
const int N = 2e5 + 5;
int arr[N];
```

## Systematic Debugging Process

### Step 1: Read the Verdict

| Verdict | First Action |
|---------|-------------|
| **WA** | Check edge cases, trace smallest example |
| **TLE** | Check complexity, look for infinite loops |
| **RE** | Check bounds, division by zero |
| **MLE** | Check array sizes, recursive depth |

### Step 2: Test Edge Cases

Always test these inputs:

```cpp
// Minimum cases
n = 1
n = 0 (if allowed)
empty input

// Boundary cases
All same values
All different values
Maximum constraints
Minimum values (0, 1, -1)

// Special patterns
Sorted ascending
Sorted descending
Alternating pattern
```

### Step 3: Trace Through by Hand

For small inputs, trace your code step by step:

```cpp
// For input: n=3, arr=[1, 2, 3]
// Trace each line:
// i=0: sum=0+1=1
// i=1: sum=1+2=3
// i=2: sum=3+3=6
// Output: 6 ✓
```

### Step 4: Add Debug Output

```cpp
#ifdef LOCAL
#define debug(x) cerr << #x << " = " << x << endl
#else
#define debug(x)
#endif

// Usage
int ans = calculate(n);
debug(ans);  // Only prints when compiled with -DLOCAL
```

## Common Bug Patterns

### Integer Overflow

```cpp
// BUG: Overflow in multiplication
int a = 100000, b = 100000;
int product = a * b;  // Overflow!

// FIX: Use long long
long long product = (long long)a * b;

// Or use #define
#define int long long
```

### Off-by-One Errors

```cpp
// BUG: Accessing arr[n] (out of bounds)
for (int i = 0; i <= n; i++) {
    arr[i] = 0;  // When i=n, this is out of bounds!
}

// FIX: Use < instead of <=
for (int i = 0; i < n; i++) {
    arr[i] = 0;
}
```

### Uninitialized Variables

```cpp
// BUG: Using uninitialized variable
int count;
for (int x : arr) {
    if (x > 0) count++;  // count is garbage!
}

// FIX: Initialize
int count = 0;
```

### Array Index Errors

```cpp
// BUG: 1-indexed vs 0-indexed confusion
for (int i = 1; i <= n; i++) {
    cin >> arr[i];  // Works if arr has size n+1
}
// Later...
for (int i = 0; i < n; i++) {
    // arr[i] has garbage for i=0!
}

// FIX: Be consistent, prefer 0-indexed
```

### Multiple Test Case Issues

```cpp
// BUG: Not resetting between test cases
int globalCounter = 0;
vector<int> seen;

void solve() {
    // globalCounter and seen retain old values!
}

// FIX: Reset at start of each test case
void solve() {
    int counter = 0;  // Local variable
    vector<int> seen; // Fresh vector
    // Or explicitly clear: seen.clear();
}
```

### Floating Point Comparison

```cpp
// BUG: Direct floating point comparison
double a = 0.1 + 0.2;
if (a == 0.3) {  // This is FALSE!
    // ...
}

// FIX: Use epsilon comparison
const double EPS = 1e-9;
if (abs(a - 0.3) < EPS) {
    // ...
}
```

## Debug Macros Template

Add this to your template:

```cpp
#ifdef LOCAL
    #define debug(x) cerr << "[" << #x << " = " << (x) << "]" << endl
    #define debugv(v) { cerr << "[" << #v << ": "; for(auto& x : v) cerr << x << " "; cerr << "]" << endl; }
    #define debugm(m) { cerr << "[" << #m << "]:" << endl; for(auto& row : m) { for(auto& x : row) cerr << x << " "; cerr << endl; } }
#else
    #define debug(x)
    #define debugv(v)
    #define debugm(m)
#endif

// Usage:
int n = 5;
debug(n);  // Prints: [n = 5]

vector<int> arr = {1, 2, 3};
debugv(arr);  // Prints: [arr: 1 2 3]

vector<vector<int>> grid = {{1,2},{3,4}};
debugm(grid);  // Prints 2D grid
```

Compile with `-DLOCAL` to enable debug output:
```bash
g++ -DLOCAL -o solution solution.cpp
```

## Stress Testing

### What is Stress Testing?

Stress testing = Running your solution against a brute force on random inputs until they disagree.

```
┌─────────────────────────────────────────────────────┐
│              STRESS TEST FLOW                       │
├─────────────────────────────────────────────────────┤
│  1. Write brute force solution (slow but correct)   │
│  2. Write optimized solution (fast but buggy?)      │
│  3. Generate random test case                        │
│  4. Run both solutions                               │
│  5. Compare outputs                                  │
│  6. If different → found the bug!                   │
│  7. Repeat until confident                           │
└─────────────────────────────────────────────────────┘
```

### Complete Stress Test Setup

**File 1: solution.cpp** (your optimized code)
```cpp
#include <bits/stdc++.h>
using namespace std;

int main() {
    int n;
    cin >> n;
    vector<int> arr(n);
    for (int& x : arr) cin >> x;
    
    // Your optimized solution
    int ans = 0;
    // ...
    
    cout << ans << endl;
    return 0;
}
```

**File 2: brute.cpp** (simple correct solution)
```cpp
#include <bits/stdc++.h>
using namespace std;

int main() {
    int n;
    cin >> n;
    vector<int> arr(n);
    for (int& x : arr) cin >> x;
    
    // Brute force O(n²) or O(n³) - slow but correct
    int ans = 0;
    for (int i = 0; i < n; i++) {
        for (int j = i; j < n; j++) {
            // Check every possibility
        }
    }
    
    cout << ans << endl;
    return 0;
}
```

**File 3: gen.cpp** (random test generator)
```cpp
#include <bits/stdc++.h>
using namespace std;

int main(int argc, char* argv[]) {
    mt19937 rng(atoi(argv[1]));  // Seed from command line
    
    int n = rng() % 10 + 1;  // n from 1 to 10 (small for stress)
    cout << n << endl;
    
    for (int i = 0; i < n; i++) {
        cout << (rng() % 100 + 1);  // values from 1 to 100
        if (i < n-1) cout << " ";
    }
    cout << endl;
    
    return 0;
}
```

**File 4: stress.bat** (Windows)
```batch
@echo off
g++ -O2 solution.cpp -o solution.exe
g++ -O2 brute.cpp -o brute.exe
g++ -O2 gen.cpp -o gen.exe

for /L %%i in (1,1,1000) do (
    gen.exe %%i > input.txt
    solution.exe < input.txt > output1.txt
    brute.exe < input.txt > output2.txt
    fc output1.txt output2.txt > nul
    if errorlevel 1 (
        echo Test %%i FAILED!
        type input.txt
        echo Solution output:
        type output1.txt
        echo Brute output:
        type output2.txt
        exit /b 1
    )
    echo Test %%i passed
)
echo All tests passed!
```

**File 4: stress.sh** (Linux/Mac)
```bash
#!/bin/bash
g++ -O2 solution.cpp -o solution
g++ -O2 brute.cpp -o brute
g++ -O2 gen.cpp -o gen

for i in $(seq 1 1000); do
    ./gen $i > input.txt
    ./solution < input.txt > output1.txt
    ./brute < input.txt > output2.txt
    if ! diff -q output1.txt output2.txt > /dev/null; then
        echo "Test $i FAILED!"
        echo "Input:"
        cat input.txt
        echo "Solution output:"
        cat output1.txt
        echo "Brute output:"
        cat output2.txt
        exit 1
    fi
    echo "Test $i passed"
done
echo "All tests passed!"
```

## Quick Debug Checklist

When you get WA, check in this order:

<Steps>
  <Step title="Did I read the problem correctly?">
    Re-read problem statement. Check constraints again.
  </Step>
  <Step title="Integer overflow?">
    Use `long long` for any multiplication or large sums.
  </Step>
  <Step title="Edge cases?">
    Test n=1, n=0, all same, all different.
  </Step>
  <Step title="Off-by-one?">
    Check loop bounds, array indexing.
  </Step>
  <Step title="Uninitialized variables?">
    Check all variables are initialized.
  </Step>
  <Step title="Multiple test cases?">
    Are you resetting everything between cases?
  </Step>
  <Step title="Output format?">
    Spaces, newlines, YES vs Yes vs yes?
  </Step>
</Steps>

## Debugging Different Verdicts

### Wrong Answer (WA)

1. Test all examples manually
2. Create small custom test
3. Check edge cases
4. Add debug output
5. Stress test if needed

### Time Limit Exceeded (TLE)

1. Check complexity analysis
2. Look for accidental O(n²) operations
3. Check for infinite loops
4. Use faster I/O if not already
5. Consider algorithm change

```cpp
// Common hidden O(n²):
for (int i = 0; i < n; i++) {
    string s = str.substr(i);  // O(n) each time!
}

// Fix: Avoid creating substrings
```

### Runtime Error (RE)

1. Check array bounds
2. Check division by zero
3. Check stack overflow (deep recursion)
4. Check null pointer / empty container

```cpp
// Common RE causes:
arr[n];     // Out of bounds (0-indexed)
x / y;      // y might be 0
vec.back(); // vec might be empty
*ptr;       // ptr might be null
```

### Memory Limit Exceeded (MLE)

1. Check array sizes
2. Avoid unnecessary copies
3. Use iterative instead of recursive
4. Use memory-efficient data structures

```cpp
// MLE causes:
int arr[1000000000];  // Too big!
vector<vector<int>> adj(n, vector<int>(n));  // O(n²) memory

// Fixes:
vector<int> arr(n);  // Dynamic allocation
vector<vector<int>> adj(n);  // Adjacency list
```

## Pro Debugging Tips

### Print and Exit

When stuck on one test case:

```cpp
// Find which test case fails
int caseNum = 0;
while (t--) {
    caseNum++;
    // If answer should be X for case 5:
    if (caseNum == 5) {
        cerr << "Debug case 5" << endl;
        debug(n);
        debugv(arr);
    }
    solve();
}
```

### Binary Search on Test Cases

If failing on test case 50 out of 100:

```cpp
// Skip first 49 cases
int skip = 49;
while (t--) {
    if (skip > 0) {
        skip--;
        // Read input but don't process
        int n; cin >> n;
        // ... discard ...
        continue;
    }
    solve();  // Only solve case 50
}
```

### Assert for Assumptions

```cpp
#include <cassert>

void solve(int n) {
    assert(n >= 1 && n <= 100000);  // Crash if invalid
    // ...
}
```

## Interview Deep-Dive

<AccordionGroup>
  <Accordion title="Your solution passes all sample cases but gets Runtime Error on a large test case. You cannot see the input. Walk me through your systematic debugging process.">
    **Strong Answer:**

    - RE on a large test case almost always means one of four things: array out of bounds, stack overflow from deep recursion, division by zero, or accessing an empty container.
    - First, I compile with `-fsanitize=address,undefined` and run against the sample inputs. Even if samples pass, the sanitizer can flag potential issues like signed integer overflow that silently produce wrong results on some platforms but crash on others.
    - For array bounds: I audit every array access. If I declared `int arr[200001]` but access with `i <= n` instead of `i < n`, the out-of-bounds write corrupts adjacent memory. For recursive DFS on a path graph of 200000 nodes, the recursion depth exceeds the default stack size -- I convert to iterative DFS with an explicit stack.
    - For division by zero: I check every `/` and `%` operation. Is there a code path where the denominator could be zero? This is common in GCD computations or when dividing by a computed value that happens to be zero for specific inputs.
    - If I still cannot reproduce, I generate random large inputs (n = 200000) and run with sanitizers. Even without a brute force, the sanitizer will pinpoint the exact line where the violation occurs.

    **Follow-up:** What is the difference between `-fsanitize=address` and `-fsanitize=undefined`?

    AddressSanitizer catches memory errors (out-of-bounds, use-after-free, stack overflow). UndefinedBehaviorSanitizer catches undefined behavior (signed overflow, division by zero, null dereference, invalid shifts). For contest debugging, use both together. ASan finds "where did I access bad memory" and UBSan finds "what operation produced an invalid result." Together they catch about 90% of RE-causing bugs with exact line numbers, at a 2-3x runtime cost.
  </Accordion>

  <Accordion title="Explain the stress testing technique. When is it worth the time investment during a contest versus manually checking edge cases?">
    **Strong Answer:**

    - Stress testing runs your optimized solution against a brute force on random small inputs until they disagree. You need three files: optimized solution, brute force, and random test generator. A script loops: generate, run both, compare outputs.
    - Worth it during a contest when: you have gotten WA 2+ times and inspection fails, the problem has tricky combinatorial edge cases, and you have 30+ minutes remaining. Not worth it with less than 15 minutes left or when the bug is clearly an overflow or format issue.
    - The key to speed is having pre-written generator templates for arrays, trees, and graphs. With templates ready, stress test setup takes 3-5 minutes instead of 15.
    - The brute force should be trivially simple -- enumerate all possibilities on tiny inputs (n <= 10). O(n!) is fine because the inputs are small. If you cannot write a correct brute force, you likely do not fully understand the problem.
    - **Example:** A greedy solution passed 48 of 50 pretests. A 5-minute stress test with n from 1 to 8 found a failing case at n=6 with a specific ordering -- the bug was an incorrect tie-breaking condition.

    **Follow-up:** How do you structure a test generator for graph problems specifically?

    For graph generators, I parameterize by graph type. For a tree: generate n-1 random edges where each new node connects to a random existing node (produces a random tree). For a general graph: generate a random tree first (ensures connectivity), then add k random extra edges. For testing shortest path algorithms, assign random weights. Always include degenerate cases in your generator: star graphs (one node connected to all others), path graphs (a single chain), and complete graphs. These hit different code paths -- star graphs stress adjacency list iteration, path graphs stress recursion depth.
  </Accordion>

  <Accordion title="What is the most common category of bugs in competitive programming, and how would you build a systematic prevention system for it?">
    **Strong Answer:**

    - Integer overflow is the most common WA-causing bug in competitive programming by a wide margin. Multiplying two ints that each reach 10^5 produces 10^10, which overflows a 32-bit int (max roughly 2 times 10^9). The result wraps around to a garbage value, and your logic silently produces wrong output.
    - Prevention system: a pre-submit checklist. Before every submission, I scan for: (1) any multiplication -- does the product exceed 2 times 10^9? Cast to long long. (2) any accumulation -- does the running sum exceed 2 times 10^9? Use long long for the accumulator. (3) modular arithmetic -- am I taking mod at every step, not just at the end?
    - Second most common: off-by-one errors in loops and binary search. Prevention: use tested templates for binary search (always `lo + (hi - lo) / 2`, always `lo < hi` for bound search). For loops, default to `i < n` and only use `i <= n` with explicit justification.
    - Third most common: not resetting state between test cases. Prevention: declare all variables inside the `solve()` function, never globally. If globals are necessary for performance, add an explicit `reset()` function called at the top of `solve()`.
    - The meta-prevention system: maintain a mistake journal. After every WA, log the category. Review weekly. If any category appears 3+ times, create a code snippet or checklist item to prevent it.

    **Follow-up:** How does the `#define int long long` macro help, and what are its drawbacks?

    It replaces every `int` with `long long`, eliminating overflow bugs entirely. Downside: doubles memory usage for all integer arrays. A segment tree with 4 million nodes goes from 16 MB to 32 MB, which can cause MLE on tight memory limits. It also requires using `signed main()` instead of `int main()`. In contests, the time saved from never debugging overflow far outweighs the occasional MLE. In interviews, use explicit `long long` where needed to show type awareness.
  </Accordion>
</AccordionGroup>
