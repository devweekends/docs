## The Strategic Approach

### The "See the Pattern" Framework

When you read a problem, run through this mental checklist:

<Steps>
  <Step title="Constraints Analysis (30 seconds)">
    - n ≤ 20 → Bitmask DP / Brute force
    - n ≤ 10³ → O(n²) solutions
    - n ≤ 10⁵ → O(n log n) or O(n)
    - n ≤ 10⁶ → O(n) only
  </Step>
  <Step title="Problem Type Recognition (1 minute)">
    - "Minimum/Maximum" → DP, Greedy, or Binary Search on Answer
    - "Count ways" → DP or Combinatorics
    - "Yes/No possible" → Greedy, Graph connectivity, or Binary Search
    - "Shortest path" → BFS/Dijkstra
    - "Subarray/Substring" → Two Pointers, Sliding Window, or Prefix Sum
  </Step>
  <Step title="Pattern Matching (2-3 minutes)">
    Match to known patterns. If stuck, think: "What's the simplest version of this problem I can solve?" Reduce the problem to its core---strip away the story and ask: "Is this really just a two-sum? A shortest path? A sliding window?" Most CF problems below 1600 rating are variations of 15-20 classical patterns.
  </Step>
  <Step title="Edge Cases (Before Coding)">
    n=1, n=2, all same elements, all different, sorted, reverse sorted, negative numbers. Also consider: is the answer 0? Is the answer -1 (impossible)? Can the answer overflow a 32-bit integer?
  </Step>
</Steps>

## Interview Deep-Dive

<AccordionGroup>
  <Accordion title="You're given a problem with constraint n <= 10^5. Walk me through how you decide which algorithmic approach to use before writing any code.">
    **Strong Answer:**

    - The constraint n <= 10^5 immediately tells me my solution must be O(n log n) or better. An O(n^2) solution would require 10^10 operations, which at roughly 10^8 operations per second would take about 100 seconds -- guaranteed TLE on any judge with a 1-2 second time limit.
    - My mental framework is constraint-first: I look at the input bounds before even understanding the full problem statement. The constraint narrows the algorithm space dramatically. For n <= 10^5, I am typically thinking sorting, binary search, segment trees, or linear-time techniques like two pointers and prefix sums.
    - After the constraint check, I read the problem type. "Find minimum/maximum" suggests DP, greedy, or binary search on answer. "Count ways" suggests DP or combinatorics. "Shortest path" points to BFS or Dijkstra. I cross-reference the constraint ceiling with the problem type to converge on 1-2 candidate algorithms.
    - I also check for hidden constant factors. If my O(n log n) solution uses a map (log n per operation) inside a loop, I am effectively at O(n log^2 n) which might be tight. Switching to unordered_map or a direct array can save the day.
    - **Example:** A problem says "find the longest subarray with sum <= K, n <= 10^5." Constraint says O(n log n) or O(n). The keyword "subarray" plus "longest" screams sliding window or two pointers -- O(n). I would not even consider an O(n^2) approach.

    **Follow-up:** What changes in your thinking when the constraint is "sum of all n across test cases <= 10^5" versus "n <= 10^5 per test case"?

    The "sum of n" constraint means I can afford O(n) or O(n log n) per test case because the total work across all test cases is bounded. This is actually more permissive in some ways -- each individual test case might have a small n, so even an O(n^2) approach might work if individual test cases are small. But I still need to be careful about resetting data structures between test cases. The key is that my total work is bounded by the sum, not multiplied by the number of test cases.
  </Accordion>

  <Accordion title="An interviewer shows you two solutions to the same problem: one uses a greedy approach at O(n log n) and the other uses DP at O(n^2). The greedy passes but the DP gets TLE. How would you explain to a junior engineer why the greedy approach is preferred here, and when DP might actually be necessary instead?">
    **Strong Answer:**

    - The greedy solution wins here because the problem has the greedy choice property: making a locally optimal decision at each step leads to a globally optimal result. Greedy algorithms avoid the overhead of tracking all subproblem states, so they typically run faster with simpler code.
    - The O(n log n) comes from sorting (the greedy preprocessing step), after which a single linear pass makes the greedy choices. The DP solution, by contrast, fills an n-by-n table or an O(n) table with O(n) transitions each, resulting in O(n^2) work.
    - However, greedy fails when local optima do not compose into a global optimum. Classic example: the coin change problem with arbitrary denominations. Greedy (take the largest coin first) fails for denominations like {1, 3, 4} with target 6 -- greedy gives 4+1+1 (3 coins) but optimal is 3+3 (2 coins). DP is required because it considers all possible ways to reach each sub-target.
    - The rule of thumb: if you can prove a greedy exchange argument (swapping any two adjacent decisions in the optimal solution does not improve it), greedy works. If you cannot prove it, or if the problem involves counting all valid solutions rather than finding just the optimal one, DP is the safer choice.
    - **Example:** Interval scheduling (select maximum non-overlapping intervals) is greedy -- sort by end time, always pick the earliest-finishing interval. Knapsack (maximize value within weight limit) requires DP because taking a lighter item now might unlock more valuable combinations later.

    **Follow-up:** Can you describe the "exchange argument" proof technique for greedy in more detail?

    The exchange argument works by assuming you have some optimal solution O that differs from the greedy solution G. You then show that you can modify O by swapping one decision to match G without making the solution worse. Repeating this process transforms O into G, proving G is also optimal. For interval scheduling, if O skips the earliest-ending interval in favor of a later-ending one, you can swap them -- the earlier-ending interval leaves at least as much room for future intervals, so the solution cannot get worse.
  </Accordion>

  <Accordion title="You are mentoring someone preparing for competitive programming. They have solved 200 LeetCode problems but keep getting WA and TLE on Codeforces. What are the most likely gaps, and how would you diagnose them?">
    **Strong Answer:**

    - The most common gap is I/O handling. LeetCode provides function signatures; Codeforces requires reading from stdin and printing to stdout. Missing `ios::sync_with_stdio(false)` and `cin.tie(nullptr)` alone can cause TLE on problems with 10^5+ integers of input.
    - Second gap: multiple test cases. Most CF problems have T test cases in a single run. Forgetting to reset global variables between test cases is the number one source of mysterious WA on Codeforces for LeetCode migrants.
    - Third gap: problem reading skills. CF problems embed the actual problem inside a narrative story. LeetCode problems are clearly stated. A LeetCode solver often misreads constraints, misses edge cases mentioned in the "Note" section, or confuses 0-indexed vs 1-indexed.
    - Fourth gap: time complexity awareness. LeetCode time limits are generous. CF judges are strict -- 10^8 operations per second is the safe budget. A solution that passes LeetCode's 2-second limit at O(n^2) with n=10^4 will TLE on CF with n=10^5 and a 1-second limit.
    - To diagnose: I would ask them to show me their last 5 WA submissions and categorize each: was it a logic error, an I/O issue, a variable reset bug, an overflow, or an edge case? Patterns emerge fast. If 3 of 5 are overflow bugs, we focus on disciplined use of long long. If 3 of 5 are WA on test 2 (edge case), we build a habit of testing n=1, n=0, all-same-elements.

    **Follow-up:** What specific debugging technique would you teach them first?

    Stress testing. Write a brute force solution you are confident is correct, write a random test generator with small constraints (n from 1 to 10), and compare outputs in a loop. The moment outputs diverge, you have a minimal failing test case. This finds edge cases that manual testing misses, and it builds the habit of validating solutions before submitting.
  </Accordion>
</AccordionGroup>
