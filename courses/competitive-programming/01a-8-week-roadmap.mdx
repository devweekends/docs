# 8-Week Roadmap: LeetCode to CF Specialist

<img src="/images/courses/cp/8-week-roadmap.svg" alt="8-Week Roadmap to Specialist" />

## Who This Is For

You've solved LeetCode problems. You understand basic DSA. But Codeforces feels like a different universe—weird I/O, tight time limits, cryptic problem statements, and ratings that make no sense.

**This roadmap bridges that gap.**

<Note>
**Goal**: Reach **Specialist (1400+)** in 8 weeks, then **Expert (1600+)** in 3 more weeks.

**Prerequisites**: 
- Solved 100+ LeetCode problems (Easy/Medium)
- Know basic data structures (arrays, stacks, queues, trees)
- Understand Big O notation
- Can code in C++ (or willing to switch)
</Note>

---

## The LeetCode → Codeforces Gap

| Aspect | LeetCode | Codeforces |
|--------|----------|------------|
| **I/O** | Function signature given | Read from stdin, print to stdout |
| **Time Limit** | Generous (often 1-2s) | Tight (1-2s for 10^6 operations) |
| **Problem Style** | Clear, structured | Story-based, need to extract problem |
| **Test Cases** | Run before submit | Pretests only, system tests after |
| **Edge Cases** | Often handled | Will fail you brutally |
| **Speed** | Accuracy matters | Speed + Accuracy both matter |
| **Multiple Tests** | Usually single | Often T test cases in one run |

## Common Pitfalls to Avoid

<Warning>
**Week 1-2 Mistakes**:
- Not reading problems fully (missing constraints)
- Forgetting `long long` for large numbers
- Not handling multiple test cases properly
- Submitting without testing edge cases
</Warning>

<Warning>
**Week 3-4 Mistakes**:
- Overcomplicating greedy problems
- Not proving greedy approach works
- Integer overflow in math problems
- Off-by-one errors in loops
</Warning>

<Warning>
**Week 5-8 Mistakes**:
- Using wrong binary search variant
- Infinite loops in binary search
- Not initializing graph arrays
- DP state design too complex
</Warning>

## Interview Deep-Dive

<AccordionGroup>
  <Accordion title="If you had exactly 8 weeks to prepare a candidate who knows basic DSA for a coding interview that tests competitive-programming-style problems, how would you structure the preparation? What would you prioritize and what would you cut?">
    **Strong Answer:**

    - Weeks 1-2: Implementation speed and correctness. Most interview failures are not algorithm failures -- they are bugs, off-by-one errors, and misread problems. I would drill fast I/O patterns, edge case habits (always test n=0, n=1, all-same), and clean code structure. This is the highest-ROI investment early on.
    - Weeks 3-4: Sorting, greedy, and binary search. These three techniques cover roughly 40% of problems rated 800-1400 on Codeforces. Sorting enables two pointers and greedy; binary search on answer is a pattern that appears in interviews constantly ("minimize the maximum," "find the smallest valid threshold").
    - Weeks 5-6: Graphs (BFS/DFS) and prefix sums/two pointers. Graph problems are common in interviews and contests alike. BFS for shortest path on unweighted graphs, DFS for connectivity and cycle detection. Prefix sums unlock O(1) range queries, and two pointers convert O(n^2) into O(n).
    - Weeks 7-8: Dynamic programming fundamentals and contest simulation. DP is the technique most feared by candidates but also the most testable. Focus on 1D DP (climbing stairs, coin change) and basic 2D DP (grid paths, LCS). Run 2-3 timed virtual contests per week to build time pressure tolerance.
    - What I would cut: segment trees, advanced graph algorithms (SCC, bridges), string algorithms (KMP, Z-function), and bitmask DP. These are rarely tested in standard coding interviews and require significant time investment for diminishing returns in an 8-week window.

    **Follow-up:** How do you measure whether the candidate is actually improving, versus just accumulating solved problems?

    Three metrics matter more than problem count: (1) Time-to-solve for problems at their current level -- if they solved 800-rated problems in 20 minutes in week 1 and now solve them in 8 minutes, that is real improvement. (2) First-submission acceptance rate -- are they submitting correct solutions more often, or still getting WA on first attempt? (3) Upsolve depth -- can they solve problems one rating tier above their comfort zone after reading the editorial and then re-solving from scratch a week later? If they can, the pattern is sticking. If they cannot, they are memorizing solutions rather than learning techniques.
  </Accordion>

  <Accordion title="A candidate you are mentoring has been stuck at Codeforces rating 1200 for two months despite solving 5 problems per day. What is likely going wrong and how would you diagnose it?">
    **Strong Answer:**

    - The most common plateau cause at 1200 is solving in the comfort zone. If all 5 daily problems are rated 800-1100, the candidate is reinforcing skills they already have rather than building new ones. Rating growth requires struggling with problems 100-200 rating points above your level, not breezing through easy ones.
    - Second diagnosis: no upsolving. If the candidate participates in contests but does not solve the problems they failed during the contest, they are wasting the highest-value learning opportunities. The contest reveals gaps; upsolving fills them.
    - Third diagnosis: pattern gaps. At 1200, the typical blocker is one of: (a) cannot recognize when to use binary search on answer, (b) weak at greedy proofs so they second-guess correct greedy approaches, (c) DP state design is shaky so they avoid DP problems entirely. I would look at their CF profile, filter by problem tags, and find which tags have zero or very few solves.
    - To fix: (1) Shift to solving 3 problems per day at rating 1200-1400 instead of 5 at 800-1100. (2) Do two virtual contests per week with strict upsolving within 24 hours. (3) Identify the weakest topic from their solve history and spend a focused week on that topic only.

    **Follow-up:** How do you decide whether a candidate should spend more time on breadth (covering more topics) versus depth (mastering one topic)?

    At the 1200 level, breadth almost always wins. The candidate needs basic competence across all core topics -- greedy, binary search, two pointers, BFS/DFS, simple DP -- to handle the variety of problems in a Div 2 contest. Depth matters more at 1600+, where a contest might require deep knowledge of segment trees or advanced DP. My rule: if you have more than 2 topics where you cannot solve a single 1200-rated problem with that tag, focus on breadth. If you consistently solve 1200-1400 across all tags but cannot break into 1500+, pick your weakest 2-3 topics and go deep.
  </Accordion>

  <Accordion title="Why do competitive programmers almost universally use C++ rather than Python or Java? What are the concrete performance differences, and when might a different language actually be the better choice?">
    **Strong Answer:**

    - C++ is 10-50x faster than Python for raw computation. A problem with n = 10^6 requiring O(n log n) operations produces about 2 times 10^7 operations. In C++ this takes about 0.2 seconds. In Python, the same loop takes 5-10 seconds due to interpreter overhead, dynamic typing, and object allocation for every integer.
    - C++ also provides the STL, which is extremely well-suited for CP: vectors, sets, maps, priority queues, sorting, binary search -- all with predictable O() performance and minimal overhead. Python has equivalent structures, but the constant factors are much larger.
    - Java is closer to C++ in speed (2-5x slower) and Codeforces compensates with higher time limits for Java. But Java has heavier I/O (must use BufferedReader, not Scanner), more verbose syntax, and auto-boxing overhead with collections. Still viable but slower to code in a contest.
    - When Python wins: math-heavy problems where BigInteger is needed (Python handles arbitrary-precision integers natively), quick prototyping for brute-force solutions in stress testing, and problems with very generous time limits or very small constraints (n <= 1000).
    - **Example:** A problem requiring 10^18 modular exponentiation is trivially handled in Python with `pow(base, exp, mod)` -- a single built-in call. In C++ you write your own binary exponentiation function.

    **Follow-up:** What about Rust or Go for competitive programming?

    Rust has C++-level performance and memory safety, but the borrow checker adds friction during timed contests -- fighting the compiler on mutable references in graph algorithms wastes precious minutes. Go is simpler but lacks generics (until recently), has a less mature competitive programming ecosystem, and its garbage collector can cause unpredictable pauses. Neither has the community support, problem-setter tooling, or library ecosystem that C++ has for CP. For interviews specifically, use whatever language you are fastest in -- correctness and clarity matter more than raw speed.
  </Accordion>
</AccordionGroup>
