# Time Complexity Analysis

<img src="/images/courses/cp/time-complexity-chart.svg" alt="Time Complexity Growth Comparison" />

## Why This Matters in Competitive Programming

In competitive programming, you have **2-3 seconds** to process up to **10^6** or even **10^8** operations. Before writing a single line of code, you must estimate: **"Will my solution pass?"**

Think of it like a budget. You walk into a contest with roughly **10^8 "operations dollars"** per second. Your algorithm is a spending plan. If your plan costs more than the budget, the judge rejects it with TLE---no negotiation, no partial credit. Learning to estimate your spending before you write a single line of code is the single most important skill in competitive programming.

<Note>
**The Golden Rule**: A modern computer performs about **10^8 simple operations per second**. If your algorithm exceeds this for the given constraints, you'll get **TLE (Time Limit Exceeded)**.
</Note>

## Pattern Recognition: Constraint Analysis

<img src="/images/courses/cp/constraint-analysis.svg" alt="Constraint Analysis Flowchart" />

### The Constraint-to-Complexity Table

Memorize this table—it's your first tool for problem-solving:

| Constraint (n) | Max Complexity | Typical Algorithms |
|----------------|----------------|-------------------|
| n ≤ 10 | O(n!) | Brute force permutations |
| n ≤ 20 | O(2^n) | Bitmask DP, subset enumeration |
| n ≤ 100 | O(n^3) | Floyd-Warshall, 3 nested loops |
| n ≤ 1,000 | O(n^2) | Simple DP, nested loops |
| n ≤ 10^5 | O(n log n) | Sorting, binary search, segment trees |
| n ≤ 10^6 | O(n) | Two pointers, prefix sums, linear DP |
| n ≤ 10^8 | O(log n) or O(1) | Binary search on answer, math |

<Tip>
**Quick Mental Check**: Look at constraints FIRST. If n = 10^5 and you're thinking of O(n^2), stop---that is 10^10 operations, which would take roughly 100 seconds. You need a better approach.

**Worked Example**: A problem says n <= 2 * 10^5 and time limit is 2 seconds. Your budget: roughly 2 * 10^8 operations.
- O(n) = 2 * 10^5 operations. Way under budget.
- O(n log n) = 2 * 10^5 * 17 = ~3.4 * 10^6. Comfortable.
- O(n * sqrt(n)) = 2 * 10^5 * 447 = ~9 * 10^7. Tight but usually passes.
- O(n^2) = 4 * 10^10. Fifty times over budget. Guaranteed TLE.
</Tip>

## Step-by-Step Complexity Analysis

### Step 1: Count the Loops

```cpp
// O(n) - Single loop
for (int i = 0; i < n; i++) {
    // O(1) work
}

// O(n^2) - Nested loops
for (int i = 0; i < n; i++) {
    for (int j = 0; j < n; j++) {
        // O(1) work
    }
}

// O(n * m) - Different bounds
for (int i = 0; i < n; i++) {
    for (int j = 0; j < m; j++) {
        // O(1) work
    }
}
```

### Step 2: Recognize Logarithmic Patterns

```cpp
// O(log n) - Halving each iteration
while (n > 0) {
    n /= 2;  // or n >>= 1
}

// O(n log n) - Loop with halving
for (int i = 0; i < n; i++) {
    int x = n;
    while (x > 0) {
        x /= 2;
    }
}

// O(log n) - Binary search
int lo = 0, hi = n;
while (lo < hi) {
    int mid = (lo + hi) / 2;
    if (check(mid)) hi = mid;
    else lo = mid + 1;
}
```

### Step 3: Identify Amortized Complexity

Some algorithms look slow but are actually fast. The key idea behind **amortized analysis** is to look at the total work across all iterations, not the per-iteration worst case. Think of it like a subscription service: any single month might seem expensive, but averaged over a year the cost-per-day is low.

```cpp
// Looks like O(n^2), actually O(n)
// Two pointers - each pointer moves at most n times total
// Why O(n)? The variable j never resets to 0. Across ALL iterations
// of the outer loop, j increments at most n times total.
// Total work = n (outer loop) + n (total j increments) = O(n).
int j = 0;
for (int i = 0; i < n; i++) {
    while (j < n && arr[j] < arr[i]) {
        j++;  // j increases, never resets
    }
}

// Looks like O(n^2), actually O(n log n)
// Merge sort - each level does O(n), log n levels
void mergeSort(vector<int>& arr, int l, int r) {
    if (l >= r) return;
    int mid = (l + r) / 2;
    mergeSort(arr, l, mid);
    mergeSort(arr, mid + 1, r);
    merge(arr, l, mid, r);  // O(n)
}
```

## Common Complexity Patterns

### Pattern 1: Nested Loops with Dependency

```cpp
// What's the complexity?
for (int i = 0; i < n; i++) {
    for (int j = i; j < n; j++) {  // j starts from i
        // O(1) work
    }
}
```

<Accordion title="Answer">
O(n^2). The inner loop runs n + (n-1) + (n-2) + ... + 1 = n(n+1)/2 times.
</Accordion>

### Pattern 2: Logarithmic Inner Loop

```cpp
// What's the complexity?
for (int i = 1; i <= n; i++) {
    for (int j = 1; j <= n; j *= 2) {
        // O(1) work
    }
}
```

<Accordion title="Answer">
O(n log n). Outer loop: O(n). Inner loop: O(log n) since j doubles each time.
</Accordion>

### Pattern 3: Divisor Loop

```cpp
// What's the complexity?
for (int i = 1; i <= n; i++) {
    for (int j = i; j <= n; j += i) {
        // O(1) work
    }
}
```

<Accordion title="Answer">
O(n log n). The inner loop for a given i runs n/i times (from i to n in steps of i). Summing over all i: n/1 + n/2 + n/3 + ... + n/n = n * (1 + 1/2 + 1/3 + ... + 1/n) = n * H_n where H_n is the harmonic series, approximately ln(n). So the total is O(n log n). This pattern appears in the Sieve of Eratosthenes and in problems that iterate over multiples/divisors.
</Accordion>

## The CP Complexity Checklist

Before submitting, ask yourself:

<Steps>
  <Step title="Read Constraints">
    What are the input limits? (n, m, q, etc.)
  </Step>
  <Step title="Estimate Operations">
    Multiply your complexity by the largest input. Example: O(n^2) with n = 10^5 → 10^10 operations → TLE.
  </Step>
  <Step title="Consider Hidden Factors">
    Are you using `map` (log n per operation) or `unordered_map` (O(1) amortized)?
    Are you copying vectors/strings in loops?
  </Step>
  <Step title="Check Recursive Depth">
    Stack overflow can occur with recursion depth > 10^6. Use iterative or increase stack size.
  </Step>
</Steps>

## Common Pitfalls

### Pitfall 1: String Concatenation

```cpp
// O(n^2) - Copying entire string each time!
string result = "";
for (int i = 0; i < n; i++) {
    result += s[i];  // Each += is O(length)
}

// O(n) - Use a vector or stringstream
vector<char> result;
for (int i = 0; i < n; i++) {
    result.push_back(s[i]);
}
```

### Pitfall 2: Passing by Value

```cpp
// O(n) per call - Copies the entire vector
void process(vector<int> arr) { ... }

// O(1) - Pass by reference
void process(vector<int>& arr) { ... }
void process(const vector<int>& arr) { ... }  // If not modifying
```

### Pitfall 3: Map vs Unordered_Map

```cpp
// O(log n) per operation
map<int, int> mp;
mp[x]++;  // O(log n)

// O(1) amortized per operation
unordered_map<int, int> mp;
mp[x]++;  // O(1) average
```

<Warning>
**Codeforces Gotcha**: `unordered_map` can be hacked with adversarial inputs causing O(n) per operation. The default hash function is known, so other contestants can craft inputs that force all keys into the same bucket, turning your O(1) lookups into O(n). Defenses: use a custom hash function (see the STL chapter) or stick with `map` (O(log n) but safe) in problems where hacking is enabled.
</Warning>

## Master Theorem (Quick Reference)

For recursive algorithms: T(n) = aT(n/b) + f(n)

Think of a recursive algorithm as a tree of work. You split the problem into `a` subproblems, each of size `n/b`, and do `f(n)` work at each level to combine results. The Master Theorem tells you which part dominates---the splitting (leaves) or the combining (internal work).

| Condition | Complexity | Intuition |
|-----------|------------|-----------|
| f(n) = O(n^c) where c < log_b(a) | O(n^(log_b(a))) | Leaves dominate---recursion does the heavy lifting |
| f(n) = O(n^c) where c = log_b(a) | O(n^c log n) | Balanced---each level contributes equally |
| f(n) = O(n^c) where c > log_b(a) | O(f(n)) | Root dominates---combining is the bottleneck |

**Examples**:
- Merge Sort: T(n) = 2T(n/2) + O(n) --- a=2, b=2, f(n)=n, c=1=log_2(2) --- Case 2 --- O(n log n)
- Binary Search: T(n) = T(n/2) + O(1) --- a=1, b=2, f(n)=1, c=0=log_2(1) --- Case 2 --- O(log n)

<Tip>
**Contest Shortcut**: You rarely need the Master Theorem during a contest. For most recursive algorithms, just ask: "How many total subproblems exist, and how much work does each do?" Multiply and you are close enough. The Master Theorem is most useful for verifying your intuition during practice.
</Tip>

## Interview Deep-Dive

<AccordionGroup>
  <Accordion title="You see a nested loop where the inner loop starts at j = i and goes to n. Your teammate says it is O(n^2). Your other teammate says it is O(n). Who is right, and can you prove it mathematically?">
    **Strong Answer:**

    - The first teammate is right: it is O(n^2). The inner loop runs (n - i) iterations for each value of i. Summing from i=0 to n-1: n + (n-1) + (n-2) + ... + 1 = n(n+1)/2, which is O(n^2).
    - The common confusion arises because people see the inner loop "starts later each time" and think it must be less than O(n^2). It is less than n^2 by exactly half, but n(n+1)/2 is still O(n^2) because we drop constant factors in Big O notation.
    - The scenario where a similar-looking pattern is O(n) is when the inner variable never resets. For example, if j is declared outside the outer loop and only increments (never resets to 0), then j moves at most n times total across all iterations of the outer loop. Total work = O(n) for the outer loop plus O(n) for all j increments = O(n). This is the two-pointer / amortized analysis pattern.
    - **Example:** In the two-pointer technique for "longest subarray with sum <= K," the right pointer advances at most n times and the left pointer advances at most n times. Even though they are in nested loops, the total work is O(2n) = O(n).

    **Follow-up:** What is amortized analysis, and how does it differ from average-case analysis?

    Amortized analysis gives a worst-case guarantee on the average cost per operation over a sequence of operations. It is not about probability -- it is a strict upper bound. Average-case analysis, by contrast, assumes a probability distribution over inputs. For example, a dynamic array's push_back is O(1) amortized because although resizing is O(n), it happens only every n insertions, so the cost per insertion averages to O(1) in the worst case over any sequence. Average-case analysis of quicksort assumes random pivots to claim O(n log n) expected time, but the worst case is still O(n^2).
  </Accordion>

  <Accordion title="A problem has n <= 10^5 and your solution uses an unordered_map inside a loop of size n. You claim O(n) total, but the solution gets TLE. What could be happening, and how would you fix it?">
    **Strong Answer:**

    - The most likely cause is hash collision attacks. On Codeforces, the default hash function for unordered_map is known, and adversarial test cases (hacks) can force all keys into the same bucket. This degrades every lookup from O(1) to O(n), making the total complexity O(n^2).
    - Even without adversarial inputs, unordered_map has a significant constant factor compared to direct array access. Each operation involves hashing, bucket lookup, and potential chain traversal. For n = 10^5 with 10^5 lookups, the overhead can push a theoretically O(n) solution past a tight 1-second time limit.
    - Fix options, ranked by preference: (1) If keys are integers in a bounded range, use a plain array or vector instead -- O(1) with negligible constant. (2) Use a custom hash function that randomizes the hash to prevent adversarial collisions. A common one XORs the key with a random value derived from chrono. (3) Switch to a regular map (O(log n) per operation) which gives O(n log n) total -- slower in theory but immune to hash attacks and more predictable.
    - **Example:** If the problem is counting frequencies of values from 0 to 10^6, I would use `int freq[1000001] = {};` instead of unordered_map. If values can be up to 10^9, I would use coordinate compression to map them into a smaller range first, then use an array.

    **Follow-up:** Can you write the custom hash function you mentioned?

    A standard defense used in competitive programming is to XOR the input with a random seed based on the current time, then apply a mixing function. Something like: `struct custom_hash { size_t operator()(long long x) const { x = (x ^ (x >> 30)) * 0xbf58476d1ce4e5b9LL; x = (x ^ (x >> 27)) * 0x94d049bb133111ebLL; return x ^ (x >> 31); } };` Then declare `unordered_map<long long, int, custom_hash> mp;`. The magic constants are from a well-known integer hash mixing function (splitmix64). The key insight is that because the mixing is deterministic but the input is transformed, an adversary cannot predict which keys will collide.
  </Accordion>

  <Accordion title="Explain the Master Theorem in plain language. Then tell me: for a recursive function that splits the problem into 3 subproblems of size n/2 and does O(n) work at each level, what is the overall complexity?">
    **Strong Answer:**

    - The Master Theorem answers: "In a recursive algorithm that creates a subproblems of size n/b and does f(n) work to split/combine, which part dominates -- the recursive branching (leaves) or the combining (root)?" There are three cases depending on whether the leaf work or the combine work grows faster.
    - For the specific example: T(n) = 3T(n/2) + O(n). Here a=3, b=2, f(n)=O(n), so c=1. We compute log_b(a) = log_2(3) which is approximately 1.585. Since c=1 is less than log_2(3) approximately 1.585, this is Case 1: the leaves dominate. The overall complexity is O(n^(log_2(3))) which is approximately O(n^1.585).
    - Intuitively, at each level of recursion the total work across all subproblems is growing. Level 0 does n work, level 1 does 3 times n/2 = 1.5n work, level 2 does 9 times n/4 = 2.25n work. The work per level is increasing by a factor of 3/2 each time, so the deepest levels dominate.
    - In contrast, merge sort has T(n) = 2T(n/2) + O(n), where a=2, b=2, log_2(2)=1, and c=1. Since c equals log_b(a), this is Case 2: balanced. Each level does O(n) work and there are O(log n) levels, giving O(n log n).

    **Follow-up:** When does the Master Theorem NOT apply?

    The Master Theorem does not apply when: (1) the subproblems are not of equal size (e.g., T(n) = T(n/3) + T(2n/3) + O(n) -- the Akra-Bazzi method handles this), (2) the number of subproblems depends on n (e.g., T(n) = nT(n/2)), (3) f(n) is not a polynomial (e.g., f(n) = n/log(n)), or (4) the subproblem sizes are not a constant fraction of n (e.g., T(n) = T(n-1) + O(n), which is the recurrence for selection sort and gives O(n^2) by telescoping, not the Master Theorem).
  </Accordion>

  <Accordion title="What is the time complexity of the Sieve of Eratosthenes, and why is the analysis non-obvious?">
    **Strong Answer:**

    - The Sieve of Eratosthenes runs in O(n log log n), which is nearly linear. The analysis is non-obvious because it involves the harmonic series restricted to primes.
    - The outer loop iterates over each prime p up to sqrt(n). For each prime p, the inner loop marks multiples of p starting from p^2, doing approximately n/p work. The total work is the sum over all primes p of n/p: n times (1/2 + 1/3 + 1/5 + 1/7 + 1/11 + ...).
    - The sum of reciprocals of primes up to n is known to be approximately log(log(n)) by Mertens' theorem. This is a deep result from analytic number theory. So the total is O(n log log n).
    - The reason this is practically O(n) is that log(log(n)) grows incredibly slowly. For n = 10^9, log(log(10^9)) is roughly log(30) which is approximately 3.4. So the sieve does only about 3-4 times more work than a single pass through the array.
    - This same pattern (summing n/i for certain values of i) appears in other contexts: the divisor-loop pattern where the inner loop goes j = i, 2i, 3i, ..., n. Summing over all i from 1 to n gives n times the full harmonic series H_n = O(n log n). Restricting to primes gives the log log n factor instead.

    **Follow-up:** What is the space complexity of the sieve, and how can you reduce it?

    The basic sieve uses O(n) space for the boolean array. For very large n (say 10^12), this is impractical. The segmented sieve reduces space to O(sqrt(n)): first sieve primes up to sqrt(n), then process the range [1, n] in blocks of size sqrt(n), using the small primes to mark composites within each block. Each block requires only O(sqrt(n)) memory. This is essential for problems where you need to find primes in a range [L, R] where R can be up to 10^12 but R-L is at most 10^6.
  </Accordion>
</AccordionGroup>
