# The Art of Reading Problems

## Why Most People Fail Before They Start

**The problem isn't your algorithm knowledge. It's how you read.**

Most competitive programmers rush through problem statements, miss critical details, and waste 20+ minutes debugging something that was never the right approach.

<Note>
**The best CP players spend 40% of their time understanding the problem and 60% coding. Beginners do the opposite—and that's why they fail.**
</Note>

---

## The 5-Layer Reading System

<img src="/images/courses/cp/reading-layers.svg" alt="5-Layer Reading System" />

### Layer 1: The 30-Second Scan

Before reading anything in detail, scan for:

```
┌─────────────────────────────────────────────────────┐
│                30-SECOND SCAN                       │
├─────────────────────────────────────────────────────┤
│  1. Constraints (n, m values)                       │
│  2. Time limit (1s, 2s, 3s?)                        │
│  3. Number of test cases (T)                        │
│  4. Output format (YES/NO, number, array?)          │
│  5. Problem rating (if visible)                     │
└─────────────────────────────────────────────────────┘
```

**Why this matters:**

| Constraint | Maximum Complexity | Likely Approach |
|-----------|-------------------|-----------------|
| n ≤ 10 | O(n!) | Brute force, all permutations |
| n ≤ 20 | O(2ⁿ) | Bitmask DP, meet in middle |
| n ≤ 100 | O(n³) | Floyd-Warshall, triple loop |
| n ≤ 1,000 | O(n²) | DP, nested loops |
| n ≤ 100,000 | O(n log n) | Sorting, binary search, segment tree |
| n ≤ 1,000,000 | O(n) | Linear scan, two pointers, prefix sum |
| n ≤ 10⁸ | O(log n) or O(1) | Math formula, binary search |

### Layer 2: Story Extraction

CF problems often have elaborate stories. Learn to extract the **core problem** from the narrative.

<img src="/images/courses/cp/story-extraction.svg" alt="Story Extraction Process" />

**Example:**

> "Princess Twilight has a magic garden with n flowers arranged in a row. Each flower has a beauty value. The evil Discord wants to steal flowers, but he can only take consecutive flowers. Princess Twilight will be sad if Discord takes flowers with total beauty more than k. Find the maximum number of consecutive flowers Discord can take without making Twilight too sad."

**Translation:**
- Array of n integers (beauty values)
- Find longest subarray with sum ≤ k
- **Pattern: Sliding Window / Two Pointers**

### The Translation Table

| Story Element | Technical Translation |
|--------------|----------------------|
| "Choose k items" | Subset selection |
| "Arrange in order" | Permutation |
| "Consecutive elements" | Subarray / Sliding window |
| "Reach from A to B" | Graph path / BFS |
| "Minimum operations" | BFS / Greedy / DP |
| "Count ways" | DP / Combinatorics |
| "Is it possible" | Yes/No → Often greedy or graph |
| "Maximum/Minimum" | DP / Greedy / Binary search |
| "For all pairs" | Often O(n²) or clever O(n) |

### Layer 3: Constraint Implications

Constraints tell you the algorithm before you even understand the problem.

<img src="/images/courses/cp/constraint-analysis.svg" alt="Constraint Analysis Flowchart" />

**Read constraints as a programming hint:**

```cpp
// n ≤ 20 → Think bitmask
for (int mask = 0; mask < (1 << n); mask++) {
    // Process subset
}

// n ≤ 10^5, queries ≤ 10^5 → Think segment tree or prefix sum
// sum of all n ≤ 10^5 → Each test case can be O(n)
// 1 ≤ a[i] ≤ 10^9 → Coordinate compression might help
// n is even → Parity trick possible
// |a[i]| ≤ 10^6 → Can use array instead of map
```

### Layer 4: Example Tracing

**Never skip the examples. Trace them by hand.**

<Steps>
  <Step title="Read Input Format">
    Understand what each line/number represents.
  </Step>
  <Step title="Trace Example 1 Manually">
    Work through it step by step, writing on paper.
  </Step>
  <Step title="Verify Against Output">
    Does your manual trace match the expected output?
  </Step>
  <Step title="Read the Note Section">
    This often explains the solution approach!
  </Step>
  <Step title="Create Your Own Test">
    Make a simpler example to verify understanding.
  </Step>
</Steps>

### Layer 5: Edge Case Hunting

Before coding, think about edge cases:

```
┌─────────────────────────────────────────────────────┐
│              EDGE CASE CHECKLIST                    │
├─────────────────────────────────────────────────────┤
│  □ n = 1 (single element)                           │
│  □ n = 2 (minimum for "pair" problems)              │
│  □ All elements are the same                        │
│  □ All elements are different                       │
│  □ Already sorted (ascending)                       │
│  □ Reverse sorted (descending)                      │
│  □ Maximum constraint values                        │
│  □ Minimum values (0, 1, -1)                        │
│  □ Empty input (if allowed)                         │
│  □ Answer is 0 or -1                                │
└─────────────────────────────────────────────────────┘
```

## Visualization Techniques

### Technique 1: Draw the Data Structure

<img src="/images/courses/cp/draw-data-structure.svg" alt="Drawing Data Structures" />

For every problem, draw what you're working with:
- **Array?** Draw boxes with indices
- **Graph?** Draw nodes and edges
- **Tree?** Draw the hierarchy
- **Grid?** Draw the 2D layout
- **String?** Write it out with indices

### Technique 2: Trace State Changes

For problems involving operations or transitions:

<img src="/images/courses/cp/state-transitions.svg" alt="State Transitions" />

```
Initial: [3, 1, 4, 1, 5]
After op 1: [3, 4, 1, 5] (removed index 1)
After op 2: [4, 1, 5] (removed index 0)
...
```

### Technique 3: The Two-Example Method

If you don't understand the problem:
1. Create the **simplest possible** valid input
2. Create a **slightly more complex** input
3. Solve both by hand
4. Look for patterns in your solutions

## Mental Models for Common Patterns

### "Choosing Subset" Problems

<img src="/images/courses/cp/subset-mental-model.svg" alt="Subset Mental Model" />

When you see "choose k elements" or "select some elements":
- Can each element be taken independently? → **Greedy**
- Does order matter? → **Permutation**
- Does taking one affect others? → **DP**

### "Range Query" Problems

<img src="/images/courses/cp/range-query-mental-model.svg" alt="Range Query Mental Model" />

When you see "from index l to r":
- Static array, sum/min/max query? → **Prefix Sum / Sparse Table**
- Array with updates? → **Segment Tree / Fenwick Tree**
- Counting in range? → **Merge Sort Tree / Wavelet Tree**

### "Shortest Path" Problems

<img src="/images/courses/cp/shortest-path-mental-model.svg" alt="Shortest Path Mental Model" />

When you see "minimum steps/moves/cost":
- All edges weight 1? → **BFS**
- Edges have weights? → **Dijkstra**
- Negative weights? → **Bellman-Ford**
- All pairs? → **Floyd-Warshall**

## The "What If" Framework

When stuck, systematically ask "What If" questions. This is not random brainstorming---it is a deliberate search through transformation space. Each question maps to a specific algorithmic technique:

```
What if n was only 10?
  → What brute force would work? (reveals the "correct" answer you need to match)
  
What if I sorted the array?
  → Would that make the problem easier? (sorting enables two pointers, binary search, greedy)
  
What if I processed elements in a different order?
  → Largest first? Smallest first? (greedy ordering is often optimal)
  
What if I thought about this from the answer's perspective?
  → Binary search on answer: "Can I achieve X? What is the minimum X?"
  
What if I reversed the problem?
  → Instead of building up, what if I broke down? (sometimes removing is easier than adding)
  
What if I looked at the complement?
  → Instead of counting valid things, count total minus invalid? (inclusion-exclusion)

What if I fixed one variable?
  → If you fix element i, does the problem for the rest become easy? (leads to O(n * f(n)) solutions)
```

## Common Reading Mistakes

<Warning>
**Mistake 1: Skipping the Note Section**

The Note section in CF problems often contains the solution approach. Always read it.
</Warning>

<Warning>
**Mistake 2: Assuming Based on Problem Title**

"Binary Search" might not be a binary search problem. Read the actual constraints.
</Warning>

<Warning>
**Mistake 3: Not Verifying Examples**

If your understanding doesn't match Example 1, you don't understand the problem. Stop and re-read.
</Warning>

<Warning>
**Mistake 4: Ignoring "It is guaranteed..."**

These guarantees simplify the problem. "It is guaranteed a solution exists" means no impossible case.
</Warning>

<Warning>
**Mistake 5: Misreading 0-indexed vs 1-indexed**

CF uses 1-indexed. Your code probably uses 0-indexed. Mind the gap.
</Warning>

## Practice Exercise: Read Like a Pro

Try reading these problem descriptions and identify:
1. What is the core problem?
2. What pattern does it match?
3. What's the likely complexity?

**Problem A:**
> Given an array of n integers, find the minimum number of operations to make all elements equal. In one operation, you can increase or decrease any element by 1.

<Accordion title="Analysis">
**Core problem:** Find target value that minimizes total distance  
**Pattern:** Median minimizes sum of absolute differences  
**Complexity:** O(n log n) for sorting, O(n) for sum  
**Approach:** Sort and pick median
</Accordion>

**Problem B:**
> You have n items with weights w[i] and values v[i]. Find maximum value of items you can take such that total weight doesn't exceed W.

<Accordion title="Analysis">
**Core problem:** 0/1 Knapsack  
**Pattern:** DP with state (index, remaining capacity)  
**Complexity:** O(n × W)  
**Approach:** DP table or recursive with memoization
</Accordion>

**Problem C:**
> Given a string s of length n, find the longest palindromic substring.

<Accordion title="Analysis">
**Core problem:** Longest palindromic substring  
**Pattern:** Expand from center / DP / Manacher  
**Complexity:** O(n²) naive, O(n) with Manacher  
**Approach:** For CP, usually O(n²) is fine if n ≤ 5000
</Accordion>

## The 3-Minute Rule

Before coding anything:

1. **Minute 1**: Scan constraints, identify complexity bound
2. **Minute 2**: Extract core problem from story
3. **Minute 3**: Trace smallest example, verify understanding

If you can't explain the problem to an imaginary friend in simple terms, you don't understand it yet.

## Interview Deep-Dive

<AccordionGroup>
  <Accordion title="You are given a problem with an elaborate story about castles and dragons. The constraint is n <= 10^5. How do you extract the actual algorithmic problem, and what is your mental translation process?">
    **Strong Answer:**

    - I use a keyword-to-pattern mapping. I scan the problem for structural keywords: "consecutive" maps to subarray/sliding window, "reach from A to B" maps to graph path/BFS, "minimum operations" maps to BFS or greedy, "count ways" maps to DP or combinatorics, "maximum/minimum" maps to DP, greedy, or binary search on answer.
    - I ignore the story entirely on first read and focus on: what is the input structure (array, graph, string, grid)? What am I asked to compute (count, optimize, check feasibility, construct)? What are the constraints?
    - The constraint n <= 10^5 tells me I need O(n log n) or better. I cross-reference: "find the longest subarray such that..." with n <= 10^5 strongly suggests two pointers or sliding window at O(n).
    - I then trace the first example by hand to verify my translation. If my understanding does not reproduce the expected output, I re-read. The Note section often explains the example in plain language and confirms the translation.
    - Common trap: the story implies a specific approach that is suboptimal. A problem about "choosing items sequentially" might sound greedy but actually require DP. I always verify by checking if the greedy choice property holds.

    **Follow-up:** What do you do when the problem constraints seem to allow multiple approaches?

    When n <= 5000, both O(n^2) and O(n log n) work. I choose whichever is simpler to implement correctly. In a contest, implementation speed and bug-free code matter more than theoretical optimality. I would use the O(n^2) DP if I am confident in it, rather than risk an O(n log n) approach with a subtle binary search bug. The exception: if the problem has multiple test cases with "sum of n <= 10^5," then even though individual n might be small, the total work must be efficient. In that case, I go for the faster algorithm.
  </Accordion>

  <Accordion title="You have 2 minutes left in a contest and you just read a new problem. How do you decide whether to attempt it or spend the time debugging an existing submission?">
    **Strong Answer:**

    - With 2 minutes left, I should NOT start implementing a new solution. Even a simple problem takes 3-5 minutes to code and debug. Instead, I evaluate two options: (1) can I fix my existing WA in 2 minutes (is it a simple output format or overflow fix?), or (2) can I submit a heuristic or partial solution for the new problem that might pass pretests?
    - If my existing WA is likely an integer overflow (I used int where long long was needed), that is a 30-second fix. I change the type, resubmit, and spend remaining time checking output format.
    - If the bug is not obvious, spending 2 minutes randomly trying fixes has very low expected value. Better to re-read the new problem and see if I can spot an O(1) or O(n) solution that takes 60 seconds to code.
    - The meta-strategy: I should never be in this position. Good time management means solving problems A and B in the first 30 minutes, spending the middle hour on C and D, and reserving the last 15-20 minutes for debugging and attempting E. If I am rushing in the last 2 minutes, my time allocation earlier was wrong.

    **Follow-up:** How do you develop better time management instincts for contests?

    Virtual contests with strict self-imposed time limits. After each virtual, I log: how long did each problem take, where did I lose time (re-reading, debugging, wrong approach), and what would the optimal time allocation have been? After 10 virtuals, patterns emerge -- maybe I consistently spend too long on problem A by over-engineering, or I waste 15 minutes on C before realizing I should skip to D. The fix is explicit time budgets: "If I do not have a clear approach for C after 10 minutes, I move to D."
  </Accordion>

  <Accordion title="What is the '3-minute rule' for reading problems, and why does spending more time reading actually save time overall?">
    **Strong Answer:**

    - The 3-minute rule: spend the first minute on constraints (determines algorithm class), the second minute on extracting the core problem from the story, and the third minute on tracing the smallest example to verify understanding. Only after these 3 minutes do I start thinking about implementation.
    - This saves time because the most expensive contest mistake is implementing the wrong approach. If I spend 3 minutes reading and identify the correct pattern, I code a clean solution in 10 minutes. If I skip reading and guess the approach, I might code for 15 minutes, get WA, spend 10 minutes debugging, realize the approach is fundamentally wrong, and start over -- 25 minutes wasted.
    - Empirically, I find that problems I read carefully have a 70%+ first-submission acceptance rate, while problems I rush through have a 30% first-submission acceptance rate. The time invested in reading pays for itself 3x over through reduced debugging.
    - The Note section is criminally underread. In roughly 40% of CF problems I have seen, the Note section either directly hints at the approach or clarifies an ambiguity in the problem statement that would otherwise cause WA.

    **Follow-up:** How do you handle problems where you understand the statement but cannot see any approach after 5 minutes?

    I apply the "What If" framework: What if I sorted the array? What if n was only 5 and I solved it by hand? What if I binary searched on the answer? What if I reversed the problem (instead of building up, what if I broke down)? Each "what if" takes 30-60 seconds to evaluate. If none yields progress after 3-4 attempts, I skip the problem and return later. Fresh eyes after solving another problem often unlock the insight. The worst thing I can do is stare at one problem for 20 minutes with zero progress.
  </Accordion>
