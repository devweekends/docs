# Sorting & Custom Comparators

## The Mental Model

Sorting is often the **first step** in solving a problem, not the solution itself. A sorted array enables binary search, two pointers, and greedy approaches. The key skill is knowing **when to sort** and **how to define the sort order**.

<Note>
**Pattern Recognition Signals**:
- Problem mentions "minimum" or "maximum"
- Need to pair elements optimally
- Need to process elements in a specific order
- Two pointers would work "if only the array were sorted"
</Note>

## Sorting Fundamentals

**Analogy**: Think of sorting like organizing a bookshelf. `sort` arranges every book alphabetically. `partial_sort` is like only arranging the first shelf -- you just need the top k items in order, the rest can stay messy. `nth_element` is even lazier: you only care which book is in the kth position, and you do not even need the rest sorted. Each level of laziness buys you speed.

### Basic Sorting

```cpp
vector<int> arr = {5, 2, 8, 1, 9};

// Ascending (default) -- O(n log n)
sort(arr.begin(), arr.end());
// arr = {1, 2, 5, 8, 9}

// Descending -- pass greater<int>() as the comparator
sort(arr.begin(), arr.end(), greater<int>());
// arr = {9, 8, 5, 2, 1}

// Partial sort: only the first k elements end up sorted -- O(n log k)
// Useful when you only need the top-k smallest values
partial_sort(arr.begin(), arr.begin() + 3, arr.end());
// First 3 elements are the 3 smallest, in sorted order; rest are unspecified

// Nth element: find kth smallest in O(n) average (quickselect under the hood)
// After this call, arr[k] holds the value that would be at index k in a sorted array
// Elements before k are all <= arr[k], elements after are all >= arr[k] (unsorted)
nth_element(arr.begin(), arr.begin() + k, arr.end());
// arr[k] is the (k+1)th smallest element
```

<Tip>
**Contest tip**: Use `nth_element` when you only need the median or kth-smallest value without fully sorting. It runs in O(n) average time versus O(n log n) for a full sort -- a meaningful constant factor improvement when n is large and the time limit is tight.
</Tip>

### Sorting Pairs and Tuples

```cpp
vector<pair<int, int>> pairs = {{3, 1}, {1, 2}, {1, 1}};

// Default: sort by first, then by second
sort(pairs.begin(), pairs.end());
// pairs = {{1, 1}, {1, 2}, {3, 1}}

// Sort by second element
sort(pairs.begin(), pairs.end(), [](auto& a, auto& b) {
    return a.second < b.second;
});
// pairs = {{3, 1}, {1, 1}, {1, 2}}

// Tuples sort lexicographically by default
vector<tuple<int, int, int>> v = {{1, 2, 3}, {1, 1, 4}, {0, 5, 5}};
sort(v.begin(), v.end());
// v = {{0, 5, 5}, {1, 1, 4}, {1, 2, 3}}
```

## Custom Comparators: The Power Tool

Custom comparators let you redefine what "less than" means. The comparator is a function that returns true if a should come before b. This is the single most important sorting skill in CP -- once you master custom comparators, you can sort by any criteria the problem demands.

### Lambda Comparators

```cpp
// Sort by absolute value -- the comparator compares |a| vs |b|
vector<int> arr = {-5, 2, -8, 1, 9};
sort(arr.begin(), arr.end(), [](int a, int b) {
    return abs(a) < abs(b);
});
// arr = {1, 2, -5, -8, 9}

// Sort strings by length, then lexicographically
vector<string> words = {"ab", "a", "abc", "b"};
sort(words.begin(), words.end(), [](const string& a, const string& b) {
    if (a.size() != b.size()) return a.size() < b.size();
    return a < b;
});
// words = {"a", "b", "ab", "abc"}
```

### Struct with Custom Comparator

```cpp
struct Event {
    int start, end, priority;
};

// Option 1: Lambda
vector<Event> events;
sort(events.begin(), events.end(), [](const Event& a, const Event& b) {
    return a.end < b.end;  // Sort by end time
});

// Option 2: Overload operator<
struct Event {
    int start, end, priority;
    
    bool operator<(const Event& other) const {
        return end < other.end;
    }
};
sort(events.begin(), events.end());  // Uses operator<

// Option 3: Separate comparator function
bool compareByStart(const Event& a, const Event& b) {
    return a.start < b.start;
}
sort(events.begin(), events.end(), compareByStart);
```

## Pattern: Coordinate Compression

**Problem**: Values are large (up to 10^9) but count is small (up to 10^5).

**Solution**: Map values to smaller range while preserving order.

**Analogy**: Imagine ranking students. You do not care that their scores were 98,432 and 97,211--you only care that one ranked 1st and the other 2nd. Coordinate compression replaces actual values with their ranks, shrinking a huge value space into a compact range you can index into.

**When you need it**: A segment tree or BIT requires indexing by value. If values go up to 10^9, you cannot create an array of that size. But if there are only 10^5 distinct values, compress them to the range [0, 10^5) and index by rank.

```cpp
vector<int> compress(vector<int>& arr) {
    // Step 1: Copy and sort to get unique values in order
    vector<int> sorted = arr;
    sort(sorted.begin(), sorted.end());
    sorted.erase(unique(sorted.begin(), sorted.end()), sorted.end());
    
    // Step 2: Map each value to its rank via binary search
    vector<int> result(arr.size());
    for (int i = 0; i < arr.size(); i++) {
        result[i] = lower_bound(sorted.begin(), sorted.end(), arr[i]) 
                    - sorted.begin();
    }
    
    return result;
    // sorted[] serves as the reverse mapping: sorted[rank] = original value
}

// Example: arr = {100, 5, 100, 3, 1000}
// Sorted unique: {3, 5, 100, 1000}
// Compressed:    {2, 1, 2, 0, 3}  (0-indexed ranks)
```

**Complexity**: O(n log n) for the sort and binary searches.

**Use Cases**: Segment trees, BIT when values are too large, 2D sweep-line problems.

## Pattern: Sorting by Multiple Criteria

**Problem**: Sort students by grade (descending), then by name (ascending).

```cpp
struct Student {
    string name;
    int grade;
};

vector<Student> students;

sort(students.begin(), students.end(), [](const Student& a, const Student& b) {
    if (a.grade != b.grade) {
        return a.grade > b.grade;  // Higher grade first
    }
    return a.name < b.name;  // Alphabetically for same grade
});
```

**Alternative using tuples** (for complex sorting):

```cpp
// Trick: negate numeric fields to reverse their order, then sort 
// lexicographically. The tuple comparison handles multi-criteria 
// sorting in a single clean expression.
sort(students.begin(), students.end(), [](const Student& a, const Student& b) {
    return make_tuple(-a.grade, a.name) < make_tuple(-b.grade, b.name);
});
```

<Tip>
**Contest tip**: The tuple trick is the fastest way to sort by multiple criteria during contests. Negate fields you want descending. This avoids error-prone multi-level if-else comparators.
</Tip>

## Pattern: Interval Scheduling

**Problem**: Select maximum number of non-overlapping intervals.

**Key Insight**: Sort by end time. Greedy works because finishing early leaves room for more.

**Why end time and not start time or duration?** Sorting by start time fails because an early-starting but very long interval blocks many short ones. Sorting by duration fails because a short interval might overlap many others. End time is the right criterion: an interval that finishes earliest can never be worse than any alternative--it leaves at least as much room for future intervals.

```cpp
int maxNonOverlapping(vector<pair<int, int>>& intervals) {
    // Sort by end time -- this is the greedy key
    sort(intervals.begin(), intervals.end(), [](auto& a, auto& b) {
        return a.second < b.second;
    });
    
    int count = 0;
    int lastEnd = INT_MIN;  // End time of the last selected interval
    
    for (auto& [start, end] : intervals) {
        if (start >= lastEnd) {   // No overlap with last selected
            count++;
            lastEnd = end;        // Select this interval
        }
    }
    
    return count;
}
// Time: O(n log n) for sorting, O(n) for the greedy pass
```

## Pattern: Custom Order for Greedy

**Problem**: Arrange elements to maximize/minimize some combined result.

**Example**: Concatenate numbers to form largest number.

**Why this comparator works**: For two numbers, there are only two orderings: "ab" or "ba" (where we concatenate their string forms). If "ab" > "ba" lexicographically, then a should come first. This pairwise comparison defines a valid total ordering because string concatenation comparison is transitive.

**Worked example**: nums = [3, 30, 34]
- Compare 3 vs 30: "330" > "303" so 3 before 30
- Compare 3 vs 34: "334" < "343" so 34 before 3
- Compare 34 vs 30: "3430" > "3034" so 34 before 30
- Order: [34, 3, 30] and result = "34330"

```cpp
string largestNumber(vector<int>& nums) {
    vector<string> strs;
    for (int n : nums) strs.push_back(to_string(n));
    
    // Key insight: compare concatenations "ab" vs "ba"
    // If a+b > b+a, then a should come first to maximize the result
    sort(strs.begin(), strs.end(), [](const string& a, const string& b) {
        return a + b > b + a;
    });
    
    if (strs[0] == "0") return "0";  // Edge case: all zeros (e.g., [0, 0])
    
    string result;
    for (const string& s : strs) result += s;
    return result;
}

// Example: [3, 30, 34] -> "34330"
```

<Tip>
**Contest tip**: This "compare by concatenation" trick appears in many CF problems. The same approach works for minimizing the result -- just flip the comparator to `a + b < b + a`. Also watch for the all-zeros edge case, which trips many contestants.
</Tip>

## Stable Sort

**When**: You need to preserve relative order of equal elements.

**Analogy**: Imagine sorting students by grade. With `sort`, two students with the same grade might end up in any order. With `stable_sort`, students who had the same grade keep their original relative order -- like a "tie-breaker by arrival time." This matters when you sort by one criterion first, then by another: a stable sort on the second criterion preserves the first sort's ordering within ties.

```cpp
vector<pair<int, string>> data = {{3, "a"}, {1, "b"}, {3, "c"}, {1, "d"}};

stable_sort(data.begin(), data.end(), [](auto& a, auto& b) {
    return a.first < b.first;
});
// data = {{1, "b"}, {1, "d"}, {3, "a"}, {3, "c"}}
// Notice: "b" before "d" and "a" before "c" (original order preserved)
```

<Warning>
**Performance note**: `stable_sort` uses O(n) extra memory and can be slightly slower than `sort` (O(n log^2 n) when memory allocation is constrained). In contests, prefer `sort` unless stability is explicitly required by the problem.
</Warning>

## Common Mistakes

<Warning>
**Mistake 1: Inconsistent Comparator**
Comparator must define a strict weak ordering:
- `a < a` must be false (irreflexivity)
- If `a < b` then `b < a` must be false (antisymmetry)
- If `a < b` and `b < c` then `a < c` (transitivity)

```cpp
// WRONG: Can cause undefined behavior
sort(arr.begin(), arr.end(), [](int a, int b) {
    return a <= b;  // Using <= instead of <
});

// CORRECT
sort(arr.begin(), arr.end(), [](int a, int b) {
    return a < b;
});
```
</Warning>

<Warning>
**Mistake 2: Sorting Changes Index Mapping**
If you need original indices, create pairs:
```cpp
vector<pair<int, int>> indexed(n);
for (int i = 0; i < n; i++) {
    indexed[i] = {arr[i], i};
}
sort(indexed.begin(), indexed.end());
```
</Warning>

<Warning>
**Mistake 3: Forgetting That Sort Modifies Array**
If you need the original, copy first:
```cpp
vector<int> original = arr;
sort(arr.begin(), arr.end());
```
</Warning>
