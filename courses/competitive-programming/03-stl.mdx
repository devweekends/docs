# STL for Competitive Programming

## Why STL is Essential

The C++ STL provides battle-tested, optimized implementations of common data structures and algorithms. Using STL effectively can be the difference between solving a problem in 5 minutes vs 30 minutes.

Think of the STL as your toolbox at a construction site. You would never hand-forge a hammer before driving a nail. In the same way, writing your own hash map or sorting algorithm during a contest is almost always a mistake. The STL implementations are faster, better tested, and already sitting in your `#include`.

<Note>
**The CP Mindset**: Don't implement what STL provides. Know what's available, know its complexity, and use it without hesitation.
</Note>

---

## Containers Cheat Sheet

### When to Use What

| Need | Container | Operations |
|------|-----------|------------|
| Dynamic array | `vector<T>` | O(1) push_back, O(1) access |
| LIFO stack | `stack<T>` | O(1) push, pop, top |
| FIFO queue | `queue<T>` | O(1) push, pop, front |
| Sorted unique keys | `set<T>` | O(log n) insert, find, erase |
| Sorted with duplicates | `multiset<T>` | O(log n) operations |
| Key-value (sorted) | `map<K,V>` | O(log n) operations |
| Key-value (fast) | `unordered_map<K,V>` | O(1) average |
| Min/max element | `priority_queue<T>` | O(log n) push, O(1) top |
| Double-ended | `deque<T>` | O(1) front/back operations |

## Vector: Your Default Container

### Essential Operations

```cpp
vector<int> v;                    // Empty vector
vector<int> v(n);                 // n elements, default initialized (0)
vector<int> v(n, val);            // n elements, all = val
vector<int> v = {1, 2, 3};        // Initializer list
vector<vector<int>> v(n, vector<int>(m, 0));  // n x m 2D vector

// Adding elements
v.push_back(x);                   // Add to end: O(1) amortized
v.emplace_back(x);                // More efficient for objects
v.pop_back();                     // Remove last: O(1)

// Access
v[i];                             // Access by index: O(1) (no bounds check)
v.at(i);                          // Access with bounds check
v.front();                        // First element
v.back();                         // Last element

// Size
v.size();                         // Number of elements
v.empty();                        // Check if empty
v.clear();                        // Remove all elements

// Iteration
for (int x : v) { }               // Range-based for
for (int& x : v) { x++; }         // Modify elements
for (int i = 0; i < v.size(); i++) { }  // Index-based
```

### Vector Patterns

```cpp
// Read n integers
int n; cin >> n;
vector<int> a(n);
for (int& x : a) cin >> x;

// Reverse
reverse(a.begin(), a.end());
// or
reverse(all(a));  // Using macro

// Sort
sort(all(a));                     // Ascending
sort(all(a), greater<int>());     // Descending

// Remove duplicates from sorted vector
a.erase(unique(all(a)), a.end());

// Find element
auto it = find(all(a), x);
if (it != a.end()) {
    int idx = it - a.begin();
}

// Binary search (on sorted vector)
bool found = binary_search(all(a), x);
auto it = lower_bound(all(a), x);  // First >= x
auto it = upper_bound(all(a), x);  // First > x
```

## Set and Map: Sorted Containers

### Set Operations

```cpp
set<int> s;

// Insert: O(log n)
s.insert(5);
s.insert(3);
s.insert(5);  // Ignored, already exists
// s = {3, 5}

// Check existence: O(log n)
if (s.count(5)) { }       // Returns 0 or 1
if (s.find(5) != s.end()) { }

// Erase: O(log n)
s.erase(5);               // By value
s.erase(s.begin());       // By iterator

// Bounds (powerful!)
auto it = s.lower_bound(4);  // First >= 4, returns iterator to 5
auto it = s.upper_bound(4);  // First > 4, returns iterator to 5

// Min/Max
*s.begin();               // Minimum
*s.rbegin();              // Maximum (or *prev(s.end()))

// Iterate (sorted order)
for (int x : s) { }
```

### Map Operations

```cpp
map<string, int> m;

// Insert/Update: O(log n)
m["apple"] = 5;
m["banana"]++;            // Inserts with value 0, then increments

// Access: O(log n)
cout << m["apple"];       // 5
cout << m["cherry"];      // Creates entry with value 0!

// Check existence first
if (m.count("apple")) { }
if (m.find("apple") != m.end()) { }

// Safe access
auto it = m.find("apple");
if (it != m.end()) {
    cout << it->second;  // Value
}

// Iterate (sorted by key)
for (auto& [key, value] : m) {
    cout << key << ": " << value << '\n';
}

// Erase
m.erase("apple");
```

### Multiset (Allows Duplicates)

```cpp
multiset<int> ms;
ms.insert(5);
ms.insert(5);
ms.insert(3);
// ms = {3, 5, 5}

ms.count(5);              // Returns 2
ms.erase(5);              // Erases ALL 5s! (ms = {3})
ms.erase(ms.find(5));     // Erases ONE 5 (ms = {3, 5})
```

<Warning>
**Multiset Trap**: The single most common multiset bug in contests is calling `ms.erase(value)` when you meant to erase only one copy. This erases ALL copies. Always use `ms.erase(ms.find(value))` to remove exactly one. Check that `ms.find(value) != ms.end()` first if the element might not exist.
</Warning>

## Priority Queue: Heap

### Basic Usage

```cpp
// Max heap (default)
priority_queue<int> pq;
pq.push(3);
pq.push(1);
pq.push(4);
pq.top();   // 4 (maximum)
pq.pop();   // Removes 4

// Min heap
priority_queue<int, vector<int>, greater<int>> minPq;
minPq.push(3);
minPq.push(1);
minPq.push(4);
minPq.top();  // 1 (minimum)

// With pairs (compares by first, then second)
priority_queue<pair<int, int>> pq;
pq.push({5, 1});
pq.push({3, 2});
pq.top();  // {5, 1}
```

### Pattern: Dijkstra's Algorithm

```cpp
// Min heap for shortest paths
// greater<pii> makes it a MIN-heap (smallest distance on top)
// Pairs compare by first element, so store {distance, node}
priority_queue<pii, vector<pii>, greater<pii>> pq;
pq.push({0, start});  // {distance, node}

while (!pq.empty()) {
    auto [d, u] = pq.top();
    pq.pop();
    
    if (d > dist[u]) continue;  // Already found a shorter path; skip
    
    for (auto [v, w] : adj[u]) {
        if (dist[u] + w < dist[v]) {
            dist[v] = dist[u] + w;
            pq.push({dist[v], v});  // Lazy deletion: old entries stay in heap
        }
    }
}
```

<Tip>
**Why `greater<>`?** The default `priority_queue` is a MAX-heap. For Dijkstra you need the node with the smallest distance on top. Adding `greater<pii>` as the third template parameter flips it to a MIN-heap. This is one of the most common CP patterns to memorize.
</Tip>

## Algorithms Header

### Sorting

```cpp
#include <algorithm>

vector<int> a = {3, 1, 4, 1, 5};

// Sort ascending
sort(all(a));  // {1, 1, 3, 4, 5}

// Sort descending
sort(all(a), greater<int>());  // {5, 4, 3, 1, 1}

// Custom comparator
sort(all(a), [](int x, int y) {
    return x > y;  // Descending
});

// Sort pairs by second element
vector<pair<int, int>> v = {{1, 3}, {2, 1}, {3, 2}};
sort(all(v), [](auto& a, auto& b) {
    return a.second < b.second;
});
// v = {{2, 1}, {3, 2}, {1, 3}}

// Partial sort (first k elements)
partial_sort(a.begin(), a.begin() + k, a.end());

// Nth element (find kth smallest, O(n) average)
nth_element(a.begin(), a.begin() + k, a.end());
// a[k] is now the (k+1)th smallest
```

### Binary Search

```cpp
vector<int> a = {1, 2, 3, 4, 5};  // Must be sorted!

// Check if exists
binary_search(all(a), 3);  // true

// Lower bound: first >= target
auto it = lower_bound(all(a), 3);  // Points to 3
int idx = it - a.begin();  // Index 2

// Upper bound: first > target
auto it = upper_bound(all(a), 3);  // Points to 4
int idx = it - a.begin();  // Index 3

// Count occurrences
int count = upper_bound(all(a), x) - lower_bound(all(a), x);

// Find range of equal elements
auto [lo, hi] = equal_range(all(a), 3);
```

### Other Useful Algorithms

```cpp
// Min/Max
*min_element(all(a));           // Minimum value
*max_element(all(a));           // Maximum value
auto [mn, mx] = minmax_element(all(a));

// Sum
accumulate(all(a), 0);          // Sum (start from 0)
accumulate(all(a), 0LL);        // Use long long for large sums

// Count
count(all(a), x);               // Count occurrences of x
count_if(all(a), [](int x) { return x > 0; });

// Reverse
reverse(all(a));

// Rotate
rotate(a.begin(), a.begin() + k, a.end());  // Left rotate by k

// Unique (remove consecutive duplicates)
a.erase(unique(all(a)), a.end());  // Works on sorted array

// Next/Prev permutation
do {
    // Process current permutation
} while (next_permutation(all(a)));

// Fill
fill(all(a), 0);
fill(arr, arr + n, 0);  // For C-style arrays

// GCD/LCM (C++17)
__gcd(a, b);           // GCD
gcd(a, b);             // C++17
lcm(a, b);             // C++17
```

## String Operations

```cpp
string s = "hello";

// Basics
s.size();                // or s.length()
s.empty();
s[0];                    // 'h'
s.front();               // 'h'
s.back();                // 'o'

// Modification
s.push_back('!');        // "hello!"
s.pop_back();            // "hello"
s += " world";           // "hello world"

// Substring
s.substr(0, 5);          // "hello" (pos, len)
s.substr(6);             // "world" (pos to end)

// Find
s.find("world");         // 6 (position, or string::npos if not found)
s.find('o');             // 4

// Compare
s < "hi";                // true (lexicographic)
s == "hello world";      // true

// Convert
to_string(123);          // "123"
stoi("123");             // 123
stoll("123456789012");   // 123456789012LL

// Reverse
reverse(all(s));
```

## Pattern Recognition: Choosing the Right Container

<Warning>
**The `unordered_map` Hacking Risk**: On Codeforces, other participants can "hack" your solution by crafting inputs that cause hash collisions in `unordered_map`, degrading it from O(1) to O(n) per operation. This can turn an O(n) solution into O(n^2) and cause TLE. Two defenses:

1. **Use `map` instead** (guaranteed O(log n), safe from hacks)
2. **Use a custom hash** to prevent adversarial collisions:
```cpp
struct custom_hash {
    size_t operator()(long long x) const {
        x = (x ^ (x >> 30)) * 0xbf58476d1ce4e5b9LL;
        x = (x ^ (x >> 27)) * 0x94d049bb133111ebLL;
        return x ^ (x >> 31);
    }
};
unordered_map<long long, int, custom_hash> mp;
```
</Warning>

    - Check existence → `set` or `unordered_set`
  </Card>
  <Card title="Need Ordering?" icon="sort">

## Common Pitfalls

### Pitfall 1: Iterator Invalidation

```cpp
// WRONG: Erasing while iterating
for (auto it = s.begin(); it != s.end(); it++) {
    if (*it == target) {
        s.erase(it);  // Iterator invalidated!
    }
}

// CORRECT
for (auto it = s.begin(); it != s.end(); ) {
    if (*it == target) {
        it = s.erase(it);  // erase returns next valid iterator
    } else {
        it++;
    }
}
```

### Pitfall 2: Modifying Map While Iterating

```cpp
// WRONG
for (auto& [k, v] : m) {
    if (v == 0) m.erase(k);  // Undefined behavior
}

// CORRECT: Collect keys first
vector<int> toRemove;
for (auto& [k, v] : m) {
    if (v == 0) toRemove.push_back(k);
}
for (int k : toRemove) m.erase(k);
```

### Pitfall 3: Using [] on Non-Existent Key

```cpp
map<int, int> m;
if (m[5] > 0) { }  // Creates m[5] = 0!

// CORRECT
if (m.count(5) && m[5] > 0) { }
```

## Interview Deep-Dive

<AccordionGroup>
  <Accordion title="You need to maintain a collection where you frequently insert elements, delete specific elements, and query for the minimum. Which STL container do you choose and why?">
    **Strong Answer:**

    - `multiset` is the ideal choice. It provides O(log n) insert, O(log n) erase by iterator, and O(1) access to the minimum via `*ms.begin()`. It handles duplicates, maintains sorted order, and supports lower_bound/upper_bound for range queries.
    - Why not `priority_queue`? A priority queue gives O(1) access to the min/max and O(log n) insert, but does NOT support O(log n) deletion of arbitrary elements. Deleting a specific element from a priority queue requires lazy deletion (mark as deleted, skip when popping), which adds complexity and can degrade performance if many elements are "deleted" but still in the heap.
    - Why not `set`? A set discards duplicates. If my data can have repeated values and I need to track each occurrence independently (e.g., delete only one copy of value 5), set fails silently by ignoring duplicate inserts. Multiset handles this correctly.
    - Critical gotcha: `ms.erase(value)` removes ALL copies of that value. To remove exactly one copy, use `ms.erase(ms.find(value))`. This is the most common multiset bug in contests.

    **Follow-up:** What if you also need O(1) access to both the minimum AND maximum simultaneously?

    Multiset gives both: `*ms.begin()` for minimum and `*ms.rbegin()` for maximum, both O(1). For a data structure that also needs frequent "remove minimum" and "remove maximum" operations (a double-ended priority queue), multiset is still the best STL choice. The alternative is maintaining two priority queues (a min-heap and max-heap) with lazy deletion, which is more complex and error-prone.
  </Accordion>

  <Accordion title="Explain why unordered_map can be attacked on Codeforces, and write the custom hash function that prevents this.">
    **Strong Answer:**

    - The default hash function for integers in most STL implementations is the identity function or a simple modular hash. Since the implementation is open-source, an adversary can craft inputs where all keys hash to the same bucket, degrading every operation from O(1) to O(n). On Codeforces, during the "hacking" phase, other contestants can submit such adversarial inputs against your solution.
    - The fix is a custom hash that mixes bits unpredictably. The splitmix64 hash applies two multiply-XOR-shift operations that spread input bits across the output space, making collisions extremely unlikely even for adversarial inputs.
    - The hash function: take the input, XOR with a random seed (prevents the adversary from predicting the hash), then apply two rounds of multiply-and-shift with carefully chosen constants. The constants come from splitmix64, a well-known PRNG.
    - Using a `chrono`-based seed adds per-run randomization, so even if the adversary knows the hash function, they cannot predict the seed and therefore cannot craft collisions.
    - Performance impact: the custom hash adds a few nanoseconds per operation -- negligible compared to the 100+ nanosecond cost of a hash table lookup. The protection is essentially free.

    **Follow-up:** When should you just use a regular `map` instead of fighting with unordered_map?

    Use `map` when: (1) the problem has hacking enabled and you do not want to risk it, (2) you need ordered iteration (e.g., iterating keys in sorted order), (3) the number of operations is small enough that O(n log n) fits comfortably in the time limit (n <= 10^5 with 2-second limit). Use unordered_map (with custom hash) when: n >= 10^6 and the O(log n) factor of map would push you over the time limit.
  </Accordion>

  <Accordion title="What is iterator invalidation, and can you give a concrete example of how it causes bugs in competitive programming code?">
    **Strong Answer:**

    - Iterator invalidation occurs when an operation on a container (insert, erase, reallocation) makes existing iterators point to invalid memory. Accessing an invalidated iterator is undefined behavior -- it might crash, produce garbage, or appear to work (the worst case).
    - Concrete example: iterating over a set and erasing elements. If you write `for (auto it = s.begin(); it != s.end(); it++) { if (*it == target) s.erase(it); }`, the erase invalidates `it`, and the subsequent `it++` is undefined behavior. Fix: `it = s.erase(it)` returns the next valid iterator.
    - For vectors, inserting or erasing elements invalidates all iterators from the point of insertion/deletion onward. More critically, `push_back` can trigger reallocation, invalidating ALL iterators and references to elements. If you store a reference to `v[0]` and then push_back causes reallocation, that reference is now dangling.
    - For maps and sets, only iterators to the erased element are invalidated -- other iterators remain valid. This is a key advantage of node-based containers over contiguous-memory containers.
    - Prevention in contests: avoid modifying containers while iterating. Collect elements to remove in a separate vector, then remove them after the iteration loop. Or use the erase-with-return-iterator pattern for sets and maps.

    **Follow-up:** How does this apply to the common pattern of "process and remove elements from a priority queue"?

    Priority queues do not expose iterators at all -- you can only access `top()` and call `pop()`. So iterator invalidation is not directly an issue. But the analogous problem is "lazy deletion": you mark elements as deleted in a separate set, and when you pop the top, check if it is marked and skip it. The bug is forgetting to check, which processes stale data. This is common in Dijkstra implementations where a node gets re-pushed with a shorter distance -- the old entry with longer distance remains in the heap and must be skipped when popped.
  </Accordion>
</AccordionGroup>
