# Hands-on Projects

Theory is essential, but hands-on practice is what separates good candidates from great ones. These projects will help you build practical skills that interviewers look for.

Think of these projects the way the kernel community thinks about drivers: nobody truly understands a subsystem until they have written code that talks to it. Reading about namespaces is one thing. Writing `clone(CLONE_NEWPID)` and watching your process become PID 1 in a new universe -- that is when the mental model clicks into place.

<Info>
**Skill Level**: Intermediate to Advanced  
**Time Investment**: 20-40 hours total  
**Outcome**: Portfolio pieces for interviews + deep understanding
</Info>

<Warning>
**A senior engineer would say**: "When I interview someone who has built a container from scratch, even a toy one, the quality of their answers about namespaces and cgroups is fundamentally different. They do not describe what namespaces do -- they describe what happened when they forgot to call `pivot_root` and the container could still see the host filesystem. That is the kind of understanding you cannot get from reading documentation."
</Warning>

## Project 1: Build a Container from Scratch

**Difficulty**: ⭐⭐⭐  
**Time**: 4-6 hours  
**Skills**: Namespaces, cgroups, syscalls

### Goal

Build a minimal container runtime in C or Go that demonstrates your understanding of Linux isolation primitives.

### Requirements

<Steps>
  <Step title="Create isolated namespaces">
    ```c
    // clone() is the kernel's Swiss Army knife for process creation.
    // Each CLONE_NEW* flag creates a NEW instance of that namespace
    // for the child process. Without these flags, the child shares
    // the parent's namespaces (which is what a normal fork() does).
    //
    // CLONE_NEWPID:  Child becomes PID 1 in a new PID namespace.
    //                It cannot see or signal processes outside.
    // CLONE_NEWNS:   Child gets its own mount table. Mounts inside
    //                the container are invisible to the host.
    // CLONE_NEWNET:  Child gets an empty network stack -- no interfaces,
    //                no routes, no iptables rules. You must set up a
    //                veth pair to give it connectivity.
    // CLONE_NEWUTS:  Child can set its own hostname without affecting
    //                the host. This is how containers get unique hostnames.
    // CLONE_NEWIPC:  Separate IPC namespace -- message queues, semaphores,
    //                and shared memory are isolated.
    int flags = CLONE_NEWPID | CLONE_NEWNS | CLONE_NEWNET | 
                CLONE_NEWUTS | CLONE_NEWIPC;
    pid_t pid = clone(child_func, stack_top, flags | SIGCHLD, NULL);
    // SIGCHLD tells the kernel to send SIGCHLD to the parent when the
    // child exits -- this is how wait() works. Without it, the parent
    // cannot reap the child and it becomes a zombie.
    ```
  </Step>
  
  <Step title="Set up cgroup resource limits">
    ```c
    // Create a new cgroup by making a directory under the cgroup v2
    // hierarchy. The kernel automatically populates it with control
    // files (memory.max, cpu.max, etc.). This is how Docker and
    // containerd create per-container cgroups.
    mkdir("/sys/fs/cgroup/mycontainer", 0755);
    
    // Set memory limit -- "100M" means 100 megabytes. The kernel parses
    // human-readable suffixes (K, M, G). This maps to memory.max in
    // cgroup v2. When the cgroup's total memory usage (RSS + page cache
    // + slab) exceeds this limit, the OOM killer is invoked.
    write_file("/sys/fs/cgroup/mycontainer/memory.max", "100M");
    
    // Set CPU limit (50% of 1 CPU) -- "50000 100000" means "50,000
    // microseconds of CPU time per 100,000 microsecond period."
    // The CFS bandwidth controller enforces this. If the container
    // uses its 50ms quota within the first 30ms of a period, it is
    // throttled (descheduled) for the remaining 70ms.
    write_file("/sys/fs/cgroup/mycontainer/cpu.max", "50000 100000");
    
    // Add process to cgroup -- writing the PID to cgroup.procs moves
    // the process (and all its threads) into this cgroup. All future
    // children of this process will also be in this cgroup (they inherit
    // it via fork/clone). This is the mechanism that keeps all container
    // processes under the same resource limits.
    write_file("/sys/fs/cgroup/mycontainer/cgroup.procs", pid_str);
    ```
    
    <Tip>
    **Debugging tip**: If your cgroup creation fails with EBUSY or EINVAL, check that you are working with cgroup v2 (unified hierarchy). Run `mount | grep cgroup` to verify. On cgroup v1 systems, you need separate directories under each controller (`/sys/fs/cgroup/memory/`, `/sys/fs/cgroup/cpu/`, etc.).
    </Tip>
  </Step>
  
  <Step title="Set up root filesystem with pivot_root">
    ```c
    // Bind-mount the new rootfs onto itself. This seems redundant but
    // is REQUIRED by pivot_root -- both new_root and put_old must be
    // mount points (not just directories). The MS_BIND flag creates a
    // bind mount, and MS_REC makes it recursive (so submounts within
    // the rootfs are also visible).
    mount(rootfs, rootfs, NULL, MS_BIND | MS_REC, NULL);
    
    // Create a directory inside the new root to "store" the old root.
    // pivot_root needs somewhere to move the current root to.
    mkdir(old_root, 0755);
    
    // pivot_root atomically swaps the root mount. After this call:
    // - The new rootfs becomes "/"
    // - The old host root is moved to "/old_root"
    // This is different from chroot! chroot only changes the path
    // resolution root but the process can still access the real root
    // via /proc/1/root or by walking up with "..". pivot_root actually
    // changes the mount namespace, making escape much harder.
    syscall(SYS_pivot_root, rootfs, old_root);
    
    // Now we are inside the new root. Unmount the old root so the
    // container cannot access host files. MNT_DETACH is a lazy unmount
    // that succeeds even if something inside old_root is still in use
    // (the kernel cleans it up when the last reference is dropped).
    chdir("/");
    umount2("/old_root", MNT_DETACH);
    rmdir("/old_root");
    ```
    
    <Warning>
    **Common mistake**: Using `chroot` instead of `pivot_root`. `chroot` only changes the process's path resolution root -- it does NOT modify the mount namespace. A process inside a `chroot` jail can trivially escape by opening a file descriptor to "/" before the `chroot` call, or by using `chdir("../..")` relative to an unconfined working directory. `pivot_root` changes the actual mount topology, which is much harder to escape. This is why real container runtimes use `pivot_root`.
    </Warning>
  </Step>
  
  <Step title="Execute user command">
    ```c
    // Set hostname -- this only works because we created a new UTS
    // namespace (CLONE_NEWUTS). Without it, sethostname would change
    // the host's hostname, which is definitely not what you want.
    // The "9" is the length of "container" (not including null terminator).
    sethostname("container", 9);
    
    // execve replaces the current process image with the new program.
    // After this call, the current code is gone -- execve never returns
    // on success. The new program inherits all the namespace isolation,
    // cgroup limits, and filesystem setup we configured above.
    // If execve fails (binary not found, permission denied), it returns
    // -1 and you should handle the error -- otherwise the child process
    // silently continues executing the parent's code, which is a
    // security nightmare.
    execve("/bin/sh", argv, envp);
    perror("execve failed");  // Only reached if execve fails
    exit(1);
    ```
  </Step>
</Steps>

### Bonus Features

- [ ] Network namespace with veth pair
- [ ] User namespace for rootless containers
- [ ] Seccomp filtering
- [ ] Capability dropping

### What You'll Learn

- How Docker/containerd actually work
- Practical namespace manipulation
- Cgroup setup and limits
- Filesystem isolation with pivot_root

## Project 2: Syscall Tracer with eBPF

**Difficulty**: ⭐⭐⭐⭐  
**Time**: 6-8 hours  
**Skills**: eBPF, kernel tracing, data structures

### Goal

Build a strace-like tool using eBPF that can trace syscalls with minimal overhead.

### Requirements

<Steps>
  <Step title="Set up BPF program to trace syscalls">
    ```c
    // syscall_tracer.bpf.c
    //
    // This program attaches to the raw_syscalls/sys_enter tracepoint,
    // which fires at the very start of every system call. The tracepoint
    // sits inside do_syscall_64() on x86_64, before any syscall-specific
    // handler runs. This means we see EVERY syscall, including ones
    // that will later be rejected by seccomp or capabilities checks.
    
    SEC("tracepoint/raw_syscalls/sys_enter")
    int trace_enter(struct trace_event_raw_sys_enter *ctx)
    {
        // bpf_get_current_pid_tgid() returns a 64-bit value:
        // upper 32 bits = tgid (what user space calls PID)
        // lower 32 bits = tid (the thread ID, equal to gettid())
        u64 id = bpf_get_current_pid_tgid();
        u32 pid = id >> 32;
        
        // Early exit for non-target PIDs. This check runs on EVERY
        // syscall system-wide, so it must be fast. Comparing two u32
        // values is ~1ns -- negligible overhead.
        if (target_pid && pid != target_pid)
            return 0;
        
        // Reserve space in the ring buffer. Ring buffers are the modern
        // replacement for perf buffers in BPF. They offer better
        // performance (no per-CPU allocation, better cache locality)
        // and a cleaner API. If the ring buffer is full (user space
        // is not reading fast enough), this returns NULL and we drop
        // the event -- a deliberate design choice to avoid slowing
        // down the traced process.
        struct syscall_event *event;
        event = bpf_ringbuf_reserve(&events, sizeof(*event), 0);
        if (!event)
            return 0;
        
        event->pid = pid;
        event->tid = id;
        event->syscall_nr = ctx->id;  // Syscall number (e.g., 0=read, 1=write)
        event->timestamp = bpf_ktime_get_ns();  // Monotonic clock, not wall time
        bpf_get_current_comm(&event->comm, sizeof(event->comm));
        
        // Capture all 6 syscall arguments. On x86_64, syscall arguments
        // are passed in registers: rdi, rsi, rdx, r10, r8, r9.
        // The tracepoint framework copies them into the ctx->args array
        // so we do not need to read registers directly.
        event->args[0] = ctx->args[0];
        event->args[1] = ctx->args[1];
        event->args[2] = ctx->args[2];
        event->args[3] = ctx->args[3];
        event->args[4] = ctx->args[4];
        event->args[5] = ctx->args[5];
        
        // Submit the event to the ring buffer. User space will read it
        // via ring_buffer__poll() or ring_buffer__consume().
        bpf_ringbuf_submit(event, 0);
        return 0;
    }
    ```
  </Step>
  
  <Step title="Capture return values">
    ```c
    // sys_exit fires after the syscall handler returns but before control
    // returns to user space. This gives us the return value, which we
    // need to correlate with the entry event to show complete syscall
    // traces (e.g., "read(3, buf, 4096) = 1234").
    SEC("tracepoint/raw_syscalls/sys_exit")
    int trace_exit(struct trace_event_raw_sys_exit *ctx)
    {
        u64 id = bpf_get_current_pid_tgid();
        
        struct exit_event *event;
        event = bpf_ringbuf_reserve(&exits, sizeof(*event), 0);
        if (!event)
            return 0;
        
        // We use tid (not pid) as the correlation key because within
        // a multi-threaded process, different threads may be executing
        // different syscalls concurrently. The tid uniquely identifies
        // which entry event this exit corresponds to.
        event->tid = id;
        event->ret = ctx->ret;  // Return value: >= 0 for success, < 0 for error
        event->timestamp = bpf_ktime_get_ns();
        
        bpf_ringbuf_submit(event, 0);
        return 0;
    }
    ```
    
    <Tip>
    **Debugging tip**: If your entry and exit events are getting mismatched in user space (wrong return values paired with wrong syscalls), check that you are correlating by `(pid, tid)` pair AND timestamp ordering. In highly threaded applications, events from different threads can interleave in the ring buffer. Using a BPF hash map keyed by tid to store the entry event and looking it up on exit is more reliable than trying to match events in user space.
    </Tip>
  </Step>
  
  <Step title="Build user-space consumer">
    ```c
    // User-space: Read from ring buffer
    static int handle_event(void *ctx, void *data, size_t len)
    {
        struct syscall_event *e = data;
        
        printf("[%s:%d] %s(",
               e->comm, e->pid, syscall_name(e->syscall_nr));
        
        // Format arguments based on syscall type
        format_syscall_args(e->syscall_nr, e->args);
        
        printf(")\n");
        return 0;
    }
    ```
  </Step>
  
  <Step title="Add filtering capabilities">
    - Filter by PID
    - Filter by syscall type (file, network, memory)
    - Show only slow syscalls (> threshold)
  </Step>
</Steps>

### Expected Output

[myapp:1234] fstat(3, {...}) = 0

## Project 3: Memory Leak Detector

**Difficulty**: ⭐⭐⭐⭐  
**Time**: 6-8 hours  
**Skills**: eBPF, memory management, stack traces

### Goal

Build a tool that tracks memory allocations and identifies leaks in running processes.

### Approach

The core idea is simple: track every `malloc` and `free` call, and whatever is in the "allocated but not freed" set after a period of time is a potential leak. The implementation uses uprobes -- BPF programs that attach to user-space function entry/exit points.

```c
// Track allocations -- uprobe fires when ANY process calls malloc in
// libc.so. We use uprobes (not tracepoints) because malloc is a
// user-space function, not a kernel function. Uprobes work by patching
// a breakpoint instruction (INT3 on x86) at the function entry point.
// When the CPU hits the breakpoint, the kernel's uprobe handler runs
// our BPF program, then resumes the original function.
SEC("uprobe/libc.so:malloc")
int trace_malloc(struct pt_regs *ctx)
{
    // PT_REGS_PARM1 extracts the first function argument from registers.
    // For malloc, this is the requested allocation size.
    // On x86_64: PARM1 = rdi, PARM2 = rsi, etc.
    u64 size = PT_REGS_PARM1(ctx);
    u64 id = bpf_get_current_pid_tgid();
    
    // We cannot get the return value (the pointer) in the entry probe.
    // Store the size temporarily, keyed by tid, so we can retrieve it
    // in the return probe. This two-phase approach is necessary because
    // BPF does not have a way to "pause" execution and wait for the
    // function to return.
    bpf_map_update_elem(&alloc_sizes, &id, &size, BPF_ANY);
    return 0;
}

// uretprobe fires when malloc() RETURNS. Now we have both the size
// (from the entry probe) and the returned pointer address.
SEC("uretprobe/libc.so:malloc")
int trace_malloc_ret(struct pt_regs *ctx)
{
    // PT_REGS_RC extracts the return value register (rax on x86_64).
    // For malloc, this is the pointer to the allocated memory.
    u64 addr = PT_REGS_RC(ctx);
    u64 id = bpf_get_current_pid_tgid();
    
    u64 *size = bpf_map_lookup_elem(&alloc_sizes, &id);
    if (!size)
        return 0;
    
    struct alloc_info info = {
        .size = *size,
        .timestamp = bpf_ktime_get_ns(),
    };
    // Capture the user-space call stack at the point of allocation.
    // BPF_F_USER_STACK tells the kernel to walk the user-space stack
    // frames (not kernel frames). This gives us the function call chain
    // that led to this malloc -- essential for identifying WHERE in the
    // code the leak originates.
    bpf_get_stack(ctx, &info.stack, sizeof(info.stack), BPF_F_USER_STACK);
    
    // Store the allocation record, keyed by the pointer address.
    // When free(addr) is called, we remove this entry.
    // Whatever remains in this map after the tracing period = leaked memory.
    bpf_map_update_elem(&allocations, &addr, &info, BPF_ANY);
    bpf_map_delete_elem(&alloc_sizes, &id);
    
    return 0;
}

// Track deallocations -- when free() is called with a pointer, we
// remove the corresponding entry from the allocations map. If the
// pointer was not in our map (because the map was full when it was
// allocated, or it was allocated before tracing started), the delete
// is a harmless no-op.
SEC("uprobe/libc.so:free")
int trace_free(struct pt_regs *ctx)
{
    u64 addr = PT_REGS_PARM1(ctx);
    
    bpf_map_delete_elem(&allocations, &addr);
    
    return 0;
}
```

<Warning>
**Production gotcha**: This approach tracks `malloc`/`free` but NOT `mmap`/`munmap`. Many allocators (jemalloc, tcmalloc) use `mmap` for large allocations (typically > 128 KB). A memory leak via `mmap` will be invisible to this tool. For a production-grade leak detector, also attach uprobes to `mmap` and `munmap`, and consider tracking `brk` for programs that use the glibc heap directly. Also note that uprobes have higher overhead than tracepoints (~5us per hit vs ~100ns) because they involve a breakpoint trap and signal delivery. On an application doing millions of mallocs per second, this overhead is significant.
</Warning>

## Project 4: Production CPU Profiler

**Difficulty**: ⭐⭐⭐⭐⭐  
**Time**: 8-10 hours  
**Skills**: Perf events, stack unwinding, visualization

### Goal

Build a sampling CPU profiler that generates flame graphs, suitable for production use.

### Components

1. **Sampler**: Use perf_event_open() for low-overhead sampling
2. **Stack Walker**: Capture kernel and user stacks
3. **Aggregator**: Collapse and count stacks
4. **Visualizer**: Generate flame graph SVG

### Key Implementation

```c
// perf_event_open is the kernel's interface for hardware and software
// performance counters. It creates a file descriptor that produces
// periodic samples -- essentially telling the kernel "interrupt me
// every N events and record what the CPU is doing."
struct perf_event_attr attr = {
    .type = PERF_TYPE_SOFTWARE,
    // PERF_COUNT_SW_CPU_CLOCK counts CPU clock cycles consumed by
    // the target process. This means samples are proportional to
    // actual CPU time, not wall time. If the process is sleeping,
    // no samples are generated -- which is exactly what we want for
    // an on-CPU profiler.
    .config = PERF_COUNT_SW_CPU_CLOCK,
    // sample_period = 10,000,000 ns = 10ms -> ~100 samples/second.
    // This is a good balance between accuracy and overhead. Higher
    // rates (1000 Hz) give better resolution but add ~1% CPU overhead.
    // Lower rates (10 Hz) are nearly free but miss short-lived functions.
    .sample_period = 10000000,
    // PERF_SAMPLE_CALLCHAIN tells the kernel to record the full call
    // stack (both user and kernel frames) with each sample. This is
    // what makes flame graphs possible -- each sample is a snapshot
    // of "where were we in the code at this instant."
    .sample_type = PERF_SAMPLE_CALLCHAIN | PERF_SAMPLE_TID,
    .exclude_kernel = 0,  // Include kernel stack frames
    .exclude_user = 0,    // Include user-space stack frames
};

// perf_event_open returns a file descriptor. The kernel creates a
// ring buffer associated with this fd for delivering samples.
// pid = target process, cpu = -1 means any CPU, group_fd = -1 (no group).
int fd = perf_event_open(&attr, pid, -1, -1, 0);

// Memory-map the ring buffer for zero-copy sample reading. The size
// must be 1 + 2^n pages (one metadata page plus a power-of-two data
// buffer). The kernel writes samples into this buffer, and we read
// them without any syscall overhead -- pure shared-memory communication.
void *mmap_base = mmap(NULL, page_size * (1 + 2^n),
                       PROT_READ | PROT_WRITE, MAP_SHARED, fd, 0);

// Process samples by reading from the ring buffer. Each sample has
// a header indicating its type and size. We only care about
// PERF_RECORD_SAMPLE -- the actual stack trace samples.
struct perf_event_header *header;
while ((header = get_next_sample(mmap_base)) != NULL) {
    if (header->type == PERF_RECORD_SAMPLE) {
        process_sample((struct sample_event *)header);
    }
}
```

<Tip>
**Debugging tip**: If your profiler produces "unknown" or "[unknown]" frames in the call stack, the problem is usually missing debug symbols or frame pointer information. Compile the target application with `-fno-omit-frame-pointer` to enable frame-pointer-based stack walking (the simplest and fastest method). For production binaries without frame pointers, you will need DWARF-based unwinding (more complex, higher overhead) or ORC unwinding (Linux kernel's custom format, kernel-only).
</Tip>

### Flame Graph Generation

```python
# Collapse stacks
stacks = {}
for sample in samples:
    stack_str = ";".join(reversed(sample.callchain))
    stacks[stack_str] = stacks.get(stack_str, 0) + 1

# Output folded format
for stack, count in stacks.items():
    print(f"{stack} {count}")
```

## Project 5: Network Connection Tracker

**Difficulty**: ⭐⭐⭐  
**Time**: 4-6 hours  
**Skills**: eBPF, networking, state machines

### Goal

Build a tool that tracks all TCP connections with latency metrics.

### Features

- Track connection establishment latency
- Track connection duration
- Group by remote IP/port
- Show retransmission rates

### BPF Program

```c
// kprobe on tcp_v4_connect -- this kernel function is called when a
// process initiates an outbound TCP connection (the connect() syscall
// for IPv4 TCP). We record the timestamp so we can later calculate
// how long the TCP handshake took. The struct sock pointer is the
// kernel's representation of a TCP connection -- it contains all
// connection state (IP addresses, ports, sequence numbers, etc.).
SEC("kprobe/tcp_v4_connect")
int trace_connect(struct pt_regs *ctx)
{
    // The first parameter to tcp_v4_connect is the struct sock.
    // We use it as the key in our map because it uniquely identifies
    // this connection attempt throughout the TCP handshake lifecycle.
    struct sock *sk = (struct sock *)PT_REGS_PARM1(ctx);
    u64 ts = bpf_ktime_get_ns();
    
    bpf_map_update_elem(&connect_start, &sk, &ts, BPF_ANY);
    return 0;
}

// tcp_rcv_state_process handles incoming TCP segments that cause
// state transitions. We are interested in the moment the connection
// enters TCP_ESTABLISHED -- that means the 3-way handshake (SYN,
// SYN-ACK, ACK) completed successfully. The latency between our
// connect() probe and this point is the TCP connection establishment
// latency, which includes network round-trip time + server processing.
SEC("kprobe/tcp_rcv_state_process")
int trace_state_change(struct pt_regs *ctx)
{
    struct sock *sk = (struct sock *)PT_REGS_PARM1(ctx);
    // BPF_CORE_READ is a CO-RE (Compile Once, Run Everywhere) macro
    // that reads struct fields in a way that is portable across kernel
    // versions. The kernel's struct sock layout changes between versions,
    // and CO-RE handles the offset relocation at load time.
    int state = BPF_CORE_READ(sk, __sk_common.skc_state);
    
    if (state == TCP_ESTABLISHED) {
        u64 *start_ts = bpf_map_lookup_elem(&connect_start, &sk);
        if (start_ts) {
            u64 latency = bpf_ktime_get_ns() - *start_ts;
            emit_event(sk, latency);
            // Clean up the map entry to prevent unbounded growth.
            // If we did not delete here, failed connections (RST,
            // timeout) would leak entries in the map.
            bpf_map_delete_elem(&connect_start, &sk);
        }
    }
    return 0;
}
```

<Warning>
**Production gotcha**: kprobes attach to specific kernel function names, which can change across kernel versions (they are not a stable API). For a production tool, prefer tracepoints (e.g., `tracepoint/tcp/tcp_probe`) where available, or use BTF-based kprobes with CO-RE to handle struct layout changes automatically. Also remember to handle connection failures -- if `tcp_v4_connect` fires but the connection is refused or times out, `tcp_rcv_state_process` may never fire for TCP_ESTABLISHED. Add a cleanup probe on `tcp_done` or `tcp_close` to prevent map entry leaks for failed connections.
</Warning>

## Interview Discussion Points

When discussing these projects in interviews, think about what separates a "I followed a tutorial" answer from a "I built this and learned from the pain" answer. The first describes what the code does. The second describes what went wrong, what surprised you, and what you would change in version 2.

1. **Design decisions**: Why did you choose this approach? What alternatives did you consider? For example: "I chose ring buffers over perf buffers because ring buffers share a single buffer across all CPUs, which simplifies the user-space consumer. Perf buffers give per-CPU isolation, which matters more for high-throughput tracing, but my use case was 10K events/sec where ring buffers are simpler and fast enough."
2. **Trade-offs**: What are the limitations? Every tool has them. Be honest: "This tool cannot trace short-lived processes because the uprobe attachment happens after the process starts. By the time we attach, the first 50ms of allocations are missed."
3. **Production readiness**: How would you make it production-safe? Key areas: bounded memory usage (map sizes), graceful degradation when maps are full, error handling for BPF load failures, and a kill switch (ability to detach quickly if overhead is too high).
4. **Extensions**: How would you add feature X? Think about this before the interview. Common requests: "How would you add filtering by container?", "How would you export to Prometheus?", "How would you handle PID reuse?"
5. **Debugging**: How did you debug issues while building it? This is the gold. "The BPF verifier rejected my program because I forgot to check the return value of `bpf_map_lookup_elem` before dereferencing. The verifier error pointed to the exact instruction, and I learned that BPF enforces null-pointer safety at load time."

<Warning>
**Important**: Don't just copy code -- understand every line. Interviewers will ask follow-up questions to verify your understanding. A common technique is to change one detail in the scenario and ask how your implementation would need to change. If you truly built it, you can reason about the change. If you copied it, you cannot.
</Warning>

## Interview Deep-Dive

<AccordionGroup>
  <Accordion title="You built a container from scratch. How does your implementation differ from what Docker actually does, and what would break in production?">
    **What the interviewer is testing**: Whether you understand the gap between a toy implementation and a production container runtime, and whether you can reason about what is missing.

    **Strong answer framework**:

    The toy container from Project 1 creates namespaces, sets up cgroups, pivots the root filesystem, and executes a command. What it is missing for production use:

    **Networking**: The toy container creates a network namespace but does not set up any networking. A real container runtime creates a veth pair (virtual ethernet link), moves one end into the container's network namespace, assigns an IP address, sets up routing, and configures iptables rules for NAT (so the container can reach the internet through the host). Docker also manages a bridge network (`docker0`) that connects all containers on the host.

    **Image management**: The toy container expects a pre-existing rootfs directory. Docker uses overlay filesystems (OverlayFS) to layer images -- a read-only base layer (the image), plus a writable upper layer for container changes. This is why multiple containers from the same image share the base layer's disk space. The container runtime also handles image pulling, layer caching, and content-addressable storage.

    **Security hardening**: The toy container runs with full root capabilities. Production containers drop all capabilities except a minimal set (~14 for Docker's default), install a seccomp filter (blocking ~50 dangerous syscalls), and optionally apply AppArmor or SELinux profiles. Without these, a container escape is trivial.

    **Process management**: The toy container runs a single process. Production containers need an init process (PID 1) that handles signal forwarding and zombie reaping. Without this, zombie processes accumulate because orphaned child processes are reparented to PID 1, but our shell may not call `wait()` on them.

    **What impresses interviewers**: Mention that understanding these gaps is actually the *point* of building the toy version. "I built it to understand which parts are the kernel's job (namespaces, cgroups) and which parts are the runtime's job (networking, image management, security). That distinction is blurry until you build it yourself."
  </Accordion>

  <Accordion title="Your syscall tracer uses ring buffers. What happens when user space cannot keep up with the event rate? How would you handle backpressure?">
    **What the interviewer is testing**: Understanding of producer-consumer dynamics in high-throughput tracing systems, and practical experience with data loss vs. overhead trade-offs.

    **Strong answer framework**:

    When the BPF ring buffer fills up because user space is not consuming events fast enough, `bpf_ringbuf_reserve()` returns NULL. Our BPF program checks for this and simply drops the event. This is a deliberate design choice -- we prioritize keeping the traced application running at full speed over capturing every single event.

    **The trade-off spectrum**:
    - **Drop events** (what we do): The traced application is never slowed down, but we lose visibility during high-event periods -- exactly when we most want data.
    - **Block the producer**: The BPF program could spin-wait for buffer space, but this would slow down the traced application's syscall path -- unacceptable in production.
    - **Increase buffer size**: Larger ring buffers (e.g., 64 MB instead of 4 MB) give more headroom, but at the cost of kernel memory. On a system with hundreds of BPF programs, this adds up.

    **Production backpressure strategies**:
    1. **Adaptive sampling**: When the drop rate exceeds a threshold, increase the sampling interval in the BPF program. Sample 1-in-10 events instead of every event. This is what Datadog and other production tracing systems do.
    2. **Aggregation in BPF**: Instead of emitting individual events, aggregate in a BPF hash map (count per syscall, histogram of latencies) and periodically read the aggregated data from user space. This reduces the event rate from millions/sec to one read per collection interval.
    3. **Per-CPU ring buffers**: Use `BPF_MAP_TYPE_PERF_EVENT_ARRAY` with per-CPU buffers so that a slow consumer on one CPU does not block events from other CPUs.

    **What impresses interviewers**: Mention that you can monitor the drop rate by checking the ring buffer's consumer/producer positions from user space. "In my tool, I log a warning when the drop rate exceeds 1% so the operator knows they need to either reduce the event rate (add filters) or increase the buffer size."
  </Accordion>

  <Accordion title="Your memory leak detector tracks malloc/free. How would it handle a Go or Java application where memory is managed by a garbage collector?">
    **What the interviewer is testing**: Understanding that the uprobe-on-malloc approach is C/C++-specific, and awareness of how runtime-managed memory complicates leak detection.

    **Strong answer framework**:

    The malloc/free uprobe approach fundamentally does not work for garbage-collected languages because these runtimes manage their own memory. Here is why and what to do instead:

    **Go**: The Go runtime allocates memory from the OS in large chunks using `mmap` (arenas), then sub-allocates from those arenas using its own allocator (`mallocgc`). A "memory leak" in Go is usually a goroutine leak (goroutines holding references that prevent GC from collecting objects) or a reference held in a global map that is never deleted. To detect leaks: (1) use Go's built-in `pprof` heap profiler (`net/http/pprof`), which tracks allocations at the Go level, (2) use `bpftrace` to trace `runtime.mallocgc` for allocation patterns, or (3) monitor the process's RSS over time and correlate with `runtime.MemStats`.

    **Java**: The JVM allocates a large heap upfront (`-Xmx`) via `mmap` and manages all objects within that heap. A "memory leak" in Java means objects are reachable (so GC cannot collect them) but are no longer useful to the application. The malloc/free probes would only show the JVM's initial heap allocation, not individual object allocations. To detect leaks: use `jmap -histo` for a quick object count, `jmap -dump:live` for a full heap dump, or attach with VisualVM/YourKit for real-time analysis.

    **A general approach that works across runtimes**: Instead of tracking allocator calls, monitor the process's RSS growth over time using the kernel's `memory.stat` in cgroups. If RSS grows monotonically despite stable traffic, there is a leak. Then use runtime-specific tools to identify the source. The BPF-based approach can still help by tracking `mmap`/`brk` syscalls to see if the runtime is requesting more memory from the kernel over time.

    **What impresses interviewers**: "My tool is designed for C/C++ native applications. For managed runtimes, I would take a different approach because the abstraction layer is different -- I need to trace at the runtime level, not the syscall level. The kernel sees a Go program as one big `mmap` region; the interesting allocation behavior is happening inside the Go runtime, above the syscall boundary."
  </Accordion>
</AccordionGroup>
