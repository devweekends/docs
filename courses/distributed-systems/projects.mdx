# Capstone Projects

The best way to master distributed systems is to build them. Reading about Raft consensus is valuable, but the real understanding comes when your implementation drops messages during a leader election and you have to reason about why two nodes disagree on the committed log.

This is the difference between reading a cookbook and actually running a kitchen during dinner rush. The cookbook tells you that souffles are delicate; the kitchen teaches you exactly how the oven door's draft collapses them. These projects are your kitchen. They are designed to challenge your understanding of consensus, replication, partitioning, and fault tolerance -- and more importantly, to give you the kind of battle-tested intuition that interviewers at Staff+ level are looking for.

<Info>
**Staff+ Level Expectation**: For these projects, don't just "make it work." Focus on **observability**, **determinism**, **performance benchmarking**, and **rigorous testing** (e.g., Jepsen-style failure injection).
</Info>

## Project 1: Distributed Key-Value Store (The "Mini-Etcd")

Build a strongly consistent, sharded key-value store that survives node failures and network partitions.

### Technical Requirements
- **Consensus**: Implement the **Raft protocol** from scratch (Leader Election, Log Replication, Safety).
- **Storage**: Use an **LSM-tree** or **B-Tree** for local storage on each node.
- **Partitioning**: Implement **Multi-Raft** or **Sharding** to scale beyond a single Raft group.
- **API**: gRPC or Thrift interface for clients.
- **Client Library**: Implement a client with transparent retries and leader discovery.

### Milestones
1.  **Single-Node Store**: Basic GET/PUT with persistence.
2.  **Raft Leader Election**: Nodes can elect a leader and handle heartbeats.
3.  **Log Replication**: Leader replicates entries to followers; commits only after quorum.
4.  **Log Compaction**: Implement snapshots to prevent logs from growing infinitely.
5.  **Sharding**: Partition keys across multiple Raft groups using consistent hashing.

### Challenge: Jepsen Testing
Use a tool like **Maelstrom** or a custom Docker-based orchestrator to inject:
- **Network Partitions**: Can your system maintain consistency during a split-brain? Test both symmetric partitions (A cannot reach B, and B cannot reach A) and asymmetric partitions (A can reach B, but B cannot reach A).
- **Process Crashes**: Does the system recover state after a reboot? The critical test: crash a node during an in-flight commit. When it restarts, does it correctly determine what was committed vs. what was only proposed?
- **Clock Skew**: Does your implementation rely too heavily on synchronized clocks? Inject 5-second clock jumps and verify that leader election timeouts still behave correctly. A common bug: using `time.Now()` for timeouts instead of monotonic clocks, causing elections to fire (or not fire) when the wall clock is adjusted.

<Warning>
**Pitfall**: The most insidious bugs in consensus implementations only surface under specific failure timing. A leader crash at the exact moment between writing to its own log and sending AppendEntries to followers creates a state that is perfectly valid but deeply confusing to debug. Run your tests thousands of times with random failure injection -- once is never enough.
</Warning>

## Project 2: Distributed Message Queue (The "Mini-Kafka")

Design and build a high-throughput, partitioned message queue.

### Technical Requirements
- **Storage**: Use a **segmented append-only log** with zero-copy I/O (sendfile).
- **Replication**: Implement **In-Sync Replicas (ISR)** model for high availability.
- **Coordination**: Use Zookeeper or a custom Raft-based controller for metadata (partitions, leader election).
- **Consumption**: Implement **Consumer Groups** with server-side partition rebalancing.
- **Durability**: Configurable `acks` (0, 1, all).

### Milestones
1.  **Append-Only Log**: Efficient disk storage with index files.
2.  **Producer/Consumer API**: Basic pub-sub functionality.
3.  **Partitioning**: Distribute data across multiple brokers.
4.  **Replication**: High-watermark tracking and follower synchronization.
5.  **Consumer Rebalancing**: Automatically reassign partitions when a consumer joins or leaves.

### Challenge: Exactly-Once Semantics
Implement **Transactional Producing** and **Idempotent Consumers**. Ensure that even if a producer retries, a message is only written once, and even if a consumer crashes, it processes each message exactly once.

<Warning>
**Pitfall -- the ISR shrink trap**: When a follower falls behind the leader's high-watermark, it gets removed from the ISR. If `min.insync.replicas = 2` and you only have 2 replicas, losing one ISR member means the partition becomes read-only (producers with `acks=all` get errors). This is the correct safety behavior, but it surprises teams that sized their replication factor too tightly. A senior engineer would say: "Replication factor 3 with `min.insync.replicas = 2` gives you durability **and** availability for single-replica failures. RF=2 with min.isr=2 gives you durability but zero headroom."
</Warning>

<Warning>
**Pitfall -- consumer offset commit ordering**: If your consumer processes messages in parallel (multiple threads per partition), you can accidentally commit offset 10 before offset 7 finishes processing. If the consumer then crashes, it restarts from offset 11 and skips message 7 entirely. The fix: only commit the **lowest uncommitted offset**, or use per-message tracking with a bitmap.
</Warning>

## Project 3: Distributed Rate Limiter (The "Edge-Limiter")

Build a globally distributed rate limiting service with sub-millisecond latency.

### Technical Requirements
- **Algorithm**: Implement **Generic Cell Rate Algorithm (GCRA)** or **Sliding Window**.
- **State Management**: Hybrid approach (Local cache + Centralized sync).
- **Communication**: Gossip protocol (SWIM) for cluster membership and health checking.
- **Performance**: Must handle 100k+ RPS with < 1ms overhead.

### Milestones
1.  **Local Limiter**: Memory-efficient bucket implementation. The core data structure is a token bucket or sliding window counter per rate-limit key. Think of each bucket as a leaky jar: tokens drip in at a fixed rate, and each request removes one token. When the jar is empty, requests are rejected until more tokens drip in.
2.  **Cluster Discovery**: Nodes find each other via Gossip. This is where you learn why gossip protocols converge in $O(\log N)$ rounds but can temporarily disagree about membership -- and why that is acceptable for a rate limiter (slightly inconsistent counts are fine, but a split-brain central counter is not).
3.  **State Synchronization**: Efficiently sync counters across nodes without a central bottleneck. The key trade-off: sync more frequently for accuracy (but consume more bandwidth), or sync less frequently for performance (but allow temporary over-admission). A CRDT-based counter (PN-Counter) is an elegant solution here.
4.  **Observability**: Real-time metrics on dropped vs. allowed requests. Expose per-key counters, sync lag, and gossip convergence time.

<Warning>
**Pitfall -- the thundering herd on counter reset**: If your rate limiter uses fixed windows (e.g., "100 requests per minute"), all counters reset at the minute boundary. Clients that were throttled at :59 will all retry at :00, creating a massive spike. Sliding windows or GCRA avoid this by distributing the reset across time. This is not a theoretical concern -- Cloudflare documented this exact issue in their early rate limiting infrastructure.
</Warning>

## How to Approach These Projects

1.  **Start with the Spec**: Write down your safety and liveness guarantees before coding. This is not bureaucratic overhead -- it is the single most important step. If you cannot state "this system guarantees that no two nodes will ever disagree on the committed value at index N" in plain English, you cannot implement it correctly. The spec is your contract with yourself.
2.  **Instrument Early**: Add tracing and metrics from Day 1. You can't debug a distributed system with `print` statements -- by the time you read the logs, the state has changed. Structured logging with vector timestamps (or at minimum, Lamport timestamps) lets you reconstruct the causal ordering of events across nodes.
3.  **Simulate Failures**: Don't wait for production to see if your Raft implementation works. Use **Deterministic Simulation Testing** (the approach pioneered by FoundationDB). The idea: replace all non-deterministic operations (network, disk, time) with simulated versions that you control. This lets you replay the exact same sequence of events when you find a bug.
4.  **Code in Go, Rust, or Java**: These languages have the best ecosystems for distributed systems (gRPC, Raft libraries for reference, etc.). Go is the most common choice in the industry (etcd, CockroachDB, TiKV's PD). Rust is excellent if you care about memory safety without GC pauses (TiKV itself). Java works well for Kafka-style projects where the JVM ecosystem is strongest.

## Recommended Resources
- **Maelstrom**: A workbench for learning distributed systems by implementation. Start here -- it provides a network simulator and correctness checker so you can focus on the algorithm, not the infrastructure plumbing.
- **Distributed Systems Labs (MIT 6.5840, formerly 6.824)**: The gold standard for academic DistSys projects. The Raft lab in particular is an excellent complement to Project 1 above -- it gives you a test harness that finds subtle safety violations you would never catch manually.
- **FoundationDB Papers**: Excellent insights into simulation testing. Their "Testing Distributed Systems w/ Deterministic Simulation" talk explains how they achieved 1 million simulated hours of testing per day, finding bugs that would take years to surface in production.
- **TLA+ Toolbox**: Use TLA+ to model your protocol before implementing it. Even a simple 50-line spec can catch design-level bugs that no amount of unit testing will find.
