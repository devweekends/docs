# Multimodal Models

## Connecting Vision and Language

**Multimodal models** understand multiple types of data -- images, text, audio -- in a shared representation space. Think of it like a universal translator: just as a bilingual person can hear a sentence in French and picture the scene it describes, multimodal models learn a common "language" where images and text live side by side in the same mathematical space.

The core insight is deceptively simple: if you can map both a photo of a cat and the sentence "a photo of a cat" to nearby points in a shared vector space, you've built a bridge between vision and language. Everything else -- search, classification, captioning, generation -- follows from that bridge.

Key applications:
- **Image-text search** -- find images matching a text query, or text describing an image
- **Visual question answering** -- answer natural language questions about image content
- **Image captioning** -- generate natural language descriptions of images
- **Text-to-image generation** -- create images from text descriptions (DALL-E, Stable Diffusion)

## CLIP: Contrastive Language-Image Pretraining

<Frame>
  <img src="/images/courses/deep-learning-mastery/clip-architecture.svg" alt="CLIP Architecture" />
</Frame>

### Core Idea

Learn a shared embedding space where matching image-text pairs are close together. The analogy here is a dating app for data: CLIP learns to match images with their "perfect partner" text descriptions, while pushing non-matching pairs apart. During training, it sees 400 million image-text pairs scraped from the internet and learns which images go with which captions.

<Tip>
**Why CLIP was revolutionary**: Before CLIP, you needed labeled training data for every new classification task. CLIP can classify images it has never been explicitly trained on -- just describe the categories in plain English and CLIP matches images to descriptions. This is called **zero-shot classification**, and it changed how practitioners think about building vision systems.
</Tip>

$$
\mathcal{L} = -\frac{1}{N}\sum_{i=1}^{N} \log \frac{\exp(\text{sim}(I_i, T_i)/\tau)}{\sum_{j=1}^{N} \exp(\text{sim}(I_i, T_j)/\tau)}
$$

### CLIP Implementation

```python
import torch
import torch.nn as nn
import torch.nn.functional as F

class CLIP(nn.Module):
    """Simplified CLIP model.
    
    Architecture overview:
    - Two separate encoders (one for images, one for text)
    - Projection heads that map both modalities to the same dimensionality
    - Learnable temperature parameter that controls the sharpness of similarity scores
    """
    
    def __init__(self, image_encoder, text_encoder, embed_dim=512, temperature=0.07):
        super().__init__()
        self.image_encoder = image_encoder
        self.text_encoder = text_encoder
        
        # Projection heads map encoder outputs to a shared embedding space
        # This is where the "bridge" between modalities is built
        self.image_proj = nn.Linear(image_encoder.embed_dim, embed_dim)
        self.text_proj = nn.Linear(text_encoder.embed_dim, embed_dim)
        
        # Temperature controls how peaked the softmax distribution is:
        # lower temp = sharper distinctions, higher temp = softer matching
        self.temperature = nn.Parameter(torch.ones([]) * temperature)
    
    def encode_image(self, images):
        features = self.image_encoder(images)
        # L2 normalize so embeddings live on a unit hypersphere --
        # this makes cosine similarity equivalent to dot product
        return F.normalize(self.image_proj(features), dim=-1)
    
    def encode_text(self, text):
        features = self.text_encoder(text)
        return F.normalize(self.text_proj(features), dim=-1)
    
    def forward(self, images, text):
        image_embeds = self.encode_image(images)
        text_embeds = self.encode_text(text)
        
        # Compute NxN similarity matrix: each image against every text
        # Diagonal entries are the matching pairs (positives)
        logits = (image_embeds @ text_embeds.T) / self.temperature
        
        return logits


def clip_loss(logits):
    """Symmetric contrastive loss.
    
    The key insight: treat it as a classification problem in BOTH directions.
    For image-to-text: each image should "select" its matching text from all texts.
    For text-to-image: each text should "select" its matching image from all images.
    The labels are simply the diagonal (index i matches index i).
    """
    labels = torch.arange(logits.shape[0], device=logits.device)
    
    # Image-to-text loss: row-wise softmax (which text does this image match?)
    loss_i2t = F.cross_entropy(logits, labels)
    # Text-to-image loss: column-wise softmax (which image does this text match?)
    loss_t2i = F.cross_entropy(logits.T, labels)
    
    return (loss_i2t + loss_t2i) / 2
```

## Using Pretrained CLIP

```python
from transformers import CLIPProcessor, CLIPModel
import torch
from PIL import Image

# Load pretrained CLIP
model = CLIPModel.from_pretrained("openai/clip-vit-base-patch32")
processor = CLIPProcessor.from_pretrained("openai/clip-vit-base-patch32")

# Zero-shot classification
image = Image.open("cat.jpg")
texts = ["a photo of a cat", "a photo of a dog", "a photo of a bird"]

inputs = processor(
    text=texts,
    images=image,
    return_tensors="pt",
    padding=True
)

with torch.no_grad():
    outputs = model(**inputs)
    logits_per_image = outputs.logits_per_image
    probs = logits_per_image.softmax(dim=1)

print(f"Predictions: {dict(zip(texts, probs[0].tolist()))}")
```

### Zero-Shot Classification

```python
def zero_shot_classify(model, processor, image, class_names):
    """Classify image using text descriptions."""
    # Create text prompts
    prompts = [f"a photo of a {cls}" for cls in class_names]
    
    inputs = processor(
        text=prompts,
        images=image,
        return_tensors="pt",
        padding=True
    )
    
    with torch.no_grad():
        outputs = model(**inputs)
        probs = outputs.logits_per_image.softmax(dim=1)
    
    predicted_class = class_names[probs.argmax().item()]
    confidence = probs.max().item()
    
    return predicted_class, confidence
```

## Visual Question Answering (VQA)

```python
from transformers import BlipProcessor, BlipForQuestionAnswering

# Load BLIP for VQA
processor = BlipProcessor.from_pretrained("Salesforce/blip-vqa-base")
model = BlipForQuestionAnswering.from_pretrained("Salesforce/blip-vqa-base")

def answer_question(image, question):
    inputs = processor(image, question, return_tensors="pt")
    
    with torch.no_grad():
        outputs = model.generate(**inputs)
    
    answer = processor.decode(outputs[0], skip_special_tokens=True)
    return answer

# Usage
image = Image.open("scene.jpg")
question = "What color is the car?"
answer = answer_question(image, question)
print(f"Q: {question}\nA: {answer}")
```

## Image Captioning

```python
from transformers import BlipProcessor, BlipForConditionalGeneration

processor = BlipProcessor.from_pretrained("Salesforce/blip-image-captioning-base")
model = BlipForConditionalGeneration.from_pretrained("Salesforce/blip-image-captioning-base")

def generate_caption(image, conditional_text=None):
    if conditional_text:
        inputs = processor(image, conditional_text, return_tensors="pt")
    else:
        inputs = processor(image, return_tensors="pt")
    
    with torch.no_grad():
        outputs = model.generate(**inputs, max_length=50)
    
    caption = processor.decode(outputs[0], skip_special_tokens=True)
    return caption

# Unconditional captioning
caption = generate_caption(image)
print(f"Caption: {caption}")

# Conditional captioning
caption = generate_caption(image, "a photograph of")
print(f"Caption: {caption}")
```

## LLaVA: Large Language-and-Vision Assistant

LLaVA represents the next evolution: instead of just matching images to text, it enables genuine multi-turn conversations about images. The architecture is elegant -- take a pretrained vision encoder (CLIP), a pretrained language model (LLaMA), and train a small "projection layer" that translates visual tokens into a format the language model can understand. Think of it as giving an LLM a pair of eyes by plugging visual features directly into its token stream.

```python
from transformers import AutoProcessor, LlavaForConditionalGeneration

model = LlavaForConditionalGeneration.from_pretrained(
    "llava-hf/llava-1.5-7b-hf",
    torch_dtype=torch.float16,
    device_map="auto"
)
processor = AutoProcessor.from_pretrained("llava-hf/llava-1.5-7b-hf")

def chat_with_image(image, prompt):
    conversation = [
        {
            "role": "user",
            "content": [
                {"type": "image"},
                {"type": "text", "text": prompt},
            ],
        },
    ]
    
    text_prompt = processor.apply_chat_template(conversation, add_generation_prompt=True)
    inputs = processor(images=image, text=text_prompt, return_tensors="pt").to(model.device)
    
    outputs = model.generate(**inputs, max_new_tokens=200)
    return processor.decode(outputs[0], skip_special_tokens=True)

# Usage
response = chat_with_image(image, "Describe this image in detail.")
```

## Building a Multimodal Model

### Vision-Language Projector

The projector is the critical bridge component. It takes high-dimensional vision features and compresses them into a fixed number of "visual tokens" that the language model can process alongside text tokens. This is similar to how a translator might summarize a long visual scene into a few key sentences.

```python
class VisionLanguageProjector(nn.Module):
    """Project vision features to language model space.
    
    This Q-Former style projector uses learnable query tokens that
    "ask questions" of the visual features via cross-attention,
    extracting the most relevant visual information for language tasks.
    """
    
    def __init__(self, vision_dim, language_dim, num_query_tokens=32):
        super().__init__()
        
        # Learnable query tokens act as "questions" about the image
        # The model learns WHAT to extract from visual features
        self.query_tokens = nn.Parameter(torch.randn(1, num_query_tokens, language_dim))
        
        # Cross-attention: queries attend to vision features to extract relevant info
        self.cross_attention = nn.MultiheadAttention(
            language_dim, num_heads=8, batch_first=True
        )
        
        # Project vision features from vision encoder dim to language model dim
        self.vision_proj = nn.Linear(vision_dim, language_dim)
        
        # MLP refines the extracted representations
        self.mlp = nn.Sequential(
            nn.Linear(language_dim, language_dim * 4),
            nn.GELU(),
            nn.Linear(language_dim * 4, language_dim),
        )
    
    def forward(self, vision_features):
        B = vision_features.shape[0]
        
        # Step 1: Map vision features to language model's dimensionality
        vision_features = self.vision_proj(vision_features)
        
        # Step 2: Expand query tokens for the batch
        queries = self.query_tokens.expand(B, -1, -1)
        
        # Step 3: Cross-attention -- queries "look at" the image and extract
        # the most relevant information (e.g., objects, spatial layout, attributes)
        attended, _ = self.cross_attention(queries, vision_features, vision_features)
        
        # Step 4: Refine through MLP
        output = self.mlp(attended)
        
        return output  # (B, num_query_tokens, language_dim)
```

## Image-Text Retrieval

```python
def build_image_text_index(model, images, texts, processor):
    """Build index for image-text retrieval."""
    model.eval()
    
    # Encode all images
    image_embeds = []
    for image in images:
        inputs = processor(images=image, return_tensors="pt")
        with torch.no_grad():
            embed = model.get_image_features(**inputs)
            image_embeds.append(F.normalize(embed, dim=-1))
    image_embeds = torch.cat(image_embeds, dim=0)
    
    # Encode all texts
    text_embeds = []
    for text in texts:
        inputs = processor(text=text, return_tensors="pt")
        with torch.no_grad():
            embed = model.get_text_features(**inputs)
            text_embeds.append(F.normalize(embed, dim=-1))
    text_embeds = torch.cat(text_embeds, dim=0)
    
    return image_embeds, text_embeds


def retrieve_images(query_text, image_embeds, images, model, processor, top_k=5):
    """Retrieve images matching text query."""
    inputs = processor(text=query_text, return_tensors="pt")
    with torch.no_grad():
        text_embed = model.get_text_features(**inputs)
        text_embed = F.normalize(text_embed, dim=-1)
    
    # Compute similarities
    similarities = (text_embed @ image_embeds.T).squeeze(0)
    top_indices = similarities.argsort(descending=True)[:top_k]
    
    return [images[i] for i in top_indices], similarities[top_indices]
```

## Multimodal Model Comparison

| Model | Vision | Language | Training Data | Use Case |
|-------|--------|----------|---------------|----------|
| **CLIP** | ViT/ResNet | Transformer | 400M pairs | Retrieval, zero-shot |
| **BLIP** | ViT | BERT | 129M pairs | Captioning, VQA |
| **LLaVA** | CLIP | LLaMA/Vicuna | 600K | Conversation |
| **GPT-4V** | Proprietary | GPT-4 | Unknown | General |
| **Flamingo** | NFNet | Chinchilla | Interleaved | Few-shot |

<Warning>
**Practical pitfall**: Model selection is about more than benchmark scores. CLIP is excellent for retrieval and zero-shot tasks but cannot generate text. BLIP handles captioning and VQA but struggles with open-ended conversation. LLaVA enables rich conversations but requires significant GPU memory (7B+ parameters). Always match the model to your actual deployment constraints -- latency budget, memory limits, and the specific task you need to solve.
</Warning>

## Interview Deep-Dive

<AccordionGroup>
  <Accordion title="How does CLIP's contrastive learning objective differ from a standard classification loss, and why does that matter for zero-shot transfer?">
    **Strong Answer:**

    The key difference is that CLIP never sees fixed class labels during training. Instead of learning "this image belongs to class 47," CLIP learns to align images and free-form text descriptions in a shared embedding space using a symmetric contrastive loss (InfoNCE). For a batch of N image-text pairs, CLIP treats every other pair in the batch as a negative, turning training into an N-way matching problem in both directions -- image-to-text and text-to-image.

    This matters for zero-shot transfer because the model's learned representation is not anchored to a fixed label set. At inference time, you can compose any text description -- "a photo of a Shiba Inu wearing sunglasses" -- and the model will score its similarity against the image without any additional training. A standard classification head is structurally locked to the classes it was trained on; adding a new class requires retraining. CLIP's embedding space is open-vocabulary by construction.

    The practical trade-off: CLIP's per-class accuracy on any specific dataset is often lower than a fine-tuned classifier, but its generality across thousands of possible classification tasks is far superior. In production, teams at companies like Canva use CLIP embeddings as a first-pass retrieval layer and then apply a lightweight task-specific head for the final ranking.

    **Follow-up: What failure modes have you seen with CLIP in production, and how do you mitigate them?**

    CLIP inherits biases from its internet-scraped training data. It performs poorly on fine-grained distinctions that are underrepresented in web text -- for example, distinguishing bird species or medical imaging subtypes. It can also exhibit social biases because the alt-text it was trained on reflects the biases of web content authors.

    Mitigation strategies include: domain-specific fine-tuning with a contrastive objective on curated pairs, prompt engineering (using "a photograph of a {class}, a type of bird" rather than just "{class}"), and adding a lightweight linear probe trained on labeled domain data. For bias, filtering training data and adding fairness constraints to the contrastive objective help but do not fully solve the problem.
  </Accordion>

  <Accordion title="Walk me through how a vision-language projector like Q-Former bridges the modality gap between a frozen vision encoder and a frozen language model.">
    **Strong Answer:**

    The core challenge is dimensional and semantic: a vision encoder like CLIP ViT produces a grid of spatial feature tokens (say, 197 tokens of dimension 768), while a language model like LLaMA expects a sequence of tokens in its own embedding space (dimension 4096). Simply projecting with a linear layer loses spatial information and floods the language model with too many visual tokens.

    The Q-Former introduces learnable query tokens -- typically 32 to 64 -- that act as "questions" the model learns to ask about the image. These queries attend to the vision encoder's output via cross-attention, selectively extracting the most relevant visual information. The mechanism works in three steps: (1) the vision encoder produces spatial features, (2) query tokens attend to those features and compress them into a fixed number of visual tokens, (3) an MLP refines those tokens into the language model's embedding space.

    The critical design choice is the number of query tokens. Too few and you lose visual detail. Too many and you increase the language model's context length and slow down generation. BLIP-2 uses 32 query tokens, which works well for most captioning and VQA tasks but can miss fine spatial details needed for tasks like visual grounding.

    In production, the projector is the only component that gets trained from scratch -- both the vision encoder and language model stay frozen. This is what makes the approach practical: you train maybe 100M parameters instead of 7B+, which takes hours on a single node instead of weeks on a cluster.

    **Follow-up: Why not just concatenate flattened image patch embeddings directly into the language model's token sequence?**

    Some architectures do exactly this -- LLaVA 1.0 uses a simple linear projection from CLIP features into the LLM's embedding space, one visual token per patch. It works, but has drawbacks at scale. A ViT-L/14 at 336x336 resolution produces 576 visual tokens. That consumes a large fraction of the language model's context window, leaving less room for the text prompt and generated output. It also means every forward pass of the language model processes hundreds of visual tokens through self-attention, which scales quadratically. The Q-Former approach compresses this to a fixed budget (32 tokens regardless of image resolution), making inference cost predictable and controllable.
  </Accordion>

  <Accordion title="You are building a multimodal content moderation system that needs to flag harmful images based on both visual content and accompanying text. How would you architect this, and what are the deployment trade-offs?">
    **Strong Answer:**

    I would start with a dual-signal pipeline rather than a single multimodal model, because content moderation has extreme precision/recall requirements and you need to understand which signal triggered the flag.

    Architecture: Run a CLIP-based similarity scorer against a curated set of harmful content descriptions ("graphic violence," "explicit content," etc.) for a fast first pass. Simultaneously, run a fine-tuned image classifier on known harmful visual patterns. The text pathway processes the accompanying caption or post text through a fine-tuned text classifier. A fusion layer combines the three scores -- CLIP similarity, visual classifier confidence, and text classifier output -- into a final moderation decision.

    Why not a single end-to-end model? Three reasons. First, explainability: regulators and trust-and-safety teams need to know whether an image was flagged for its visual content, its text, or the combination. A single fused model makes this attribution difficult. Second, latency: you can run the CLIP scorer and text classifier in parallel since they share no dependencies, and short-circuit if either exceeds a high-confidence threshold. Third, updateability: new harmful content categories emerge frequently, and you can add new text descriptions to the CLIP scorer without retraining.

    The main deployment trade-off is latency versus accuracy. CLIP zero-shot catches maybe 80% of harmful content at 50ms per image. The fine-tuned visual classifier adds another 30ms but catches edge cases CLIP misses. Running both in parallel with early termination gives you the best of both worlds but doubles GPU utilization. At scale (say, 100M images/day), that GPU cost is significant -- about $15K-$25K/month on cloud infrastructure.

    **Follow-up: How do you handle adversarial attacks where users add noise or overlays specifically to evade CLIP-based moderation?**

    This is a real problem. Adversarial perturbations that are imperceptible to humans can flip CLIP's similarity score significantly. The defense is defense-in-depth: the fine-tuned visual classifier trained with adversarial training (PGD-AT) is more robust to pixel-level perturbations than CLIP's zero-shot approach. Additionally, hashing-based systems (like PhotoDNA) catch exact or near-exact copies of known harmful images regardless of adversarial noise. The text pathway provides another independent signal that is harder to attack simultaneously. Finally, user behavior signals (new account, rapid posting, suspicious metadata) provide non-content signals that adversarial image perturbations cannot defeat.
  </Accordion>

  <Accordion title="Explain the role of the temperature parameter in CLIP's contrastive loss. What happens if you set it too high or too low?">
    **Strong Answer:**

    The temperature parameter controls the sharpness of the softmax distribution over similarity scores. In CLIP's loss, you compute cosine similarities between image and text embeddings, divide by temperature, and apply softmax. Mathematically, it scales the logits: logits = (image_embeds @ text_embeds.T) / tau.

    When temperature is too low (say 0.01), the softmax becomes extremely peaked -- it assigns nearly all probability mass to the single most similar pair and treats everything else as equally dissimilar. This makes training unstable because the gradients become very sparse, and the model gets overconfident about small differences in similarity. It is analogous to a teacher who only gives A+ or F with nothing in between.

    When temperature is too high (say 10.0), the softmax becomes nearly uniform -- all pairs look roughly equally similar or dissimilar. The model receives very weak gradient signals because the loss surface is flat. Learning stalls because the model cannot distinguish good matches from bad ones. This is like grading where everyone gets a B regardless of performance.

    CLIP initializes temperature as a learnable parameter starting at 0.07 and lets the model find its own optimal value during training. In practice, the learned temperature typically converges to around 0.01-0.05. Making it learnable is important because the optimal temperature depends on the embedding dimensionality and the batch size -- larger batches need slightly different temperatures because the difficulty of the N-way classification changes.

    One production gotcha: when using CLIP embeddings for retrieval, you often want to re-calibrate the temperature at inference time. The training temperature optimizes for contrastive loss, not for your specific retrieval precision/recall trade-off. Teams often find that adjusting the temperature post-hoc improves retrieval metrics significantly.

    **Follow-up: How does batch size interact with contrastive learning performance in CLIP, and what is the practical implication?**

    Batch size is critical because each batch defines the set of negatives. With a batch of 32, each positive image-text pair is contrasted against only 31 negatives -- many of which may be semantically very different and thus "easy" negatives. With a batch of 32,768 (what OpenAI used), the probability of encountering hard negatives -- pairs that are similar but not matching -- increases dramatically. This forces the model to learn finer-grained distinctions.

    The practical implication is that CLIP-style training benefits enormously from large batches, far more than standard classification. OpenAI used 32K batch sizes across many GPUs. If you are training with smaller compute, you can partially compensate with memory banks (storing embeddings from previous batches as additional negatives) or hard-negative mining. But in my experience, going below batch size 1024 for contrastive learning leads to notably worse representations.
  </Accordion>
