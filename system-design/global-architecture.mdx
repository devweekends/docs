Think of global architecture like running a chain of hospitals across different countries. Each hospital (region) needs to be self-sufficient for emergencies (local availability), but they also need to share patient records (data replication). The hard question is never "should we have multiple hospitals?" -- it is "what happens when two hospitals update the same patient record at the same time?" That conflict resolution question is the heart of global system design.

## Multi-Region Architecture Patterns

### Pattern 1: Active-Passive (Disaster Recovery)

```
┌─────────────────────────────────────────────────────────────────┐
│                    ACTIVE-PASSIVE                               │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│         Primary Region              Secondary Region            │
│         (Active)                    (Passive)                   │
│                                                                 │
│         ┌──────────┐                ┌──────────┐               │
│         │ Traffic  │                │  Standby │               │
│         │    ▼     │   Replication  │          │               │
│  Users ─► Services ├───────────────►│ Services │               │
│         │    ▼     │                │          │               │
│         │    DB    │───────────────►│    DB    │               │
│         └──────────┘    (async)     └──────────┘               │
│                                                                 │
│  RPO (Recovery Point Objective): Minutes (data loss)           │
│  RTO (Recovery Time Objective): Minutes to hours (downtime)    │
│                                                                 │
│  ✓ Pros: Simple, cost-effective                                │
│  ✗ Cons: Wasted passive capacity, failover complexity          │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### Pattern 2: Active-Active (Multi-Region)

```
┌─────────────────────────────────────────────────────────────────┐
│                    ACTIVE-ACTIVE                                │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│        US-EAST                         EU-WEST                  │
│        ┌──────────┐                   ┌──────────┐             │
│  US ───► Services │◄─────────────────►│ Services │◄─── EU      │
│  Users │    ▼     │  Bi-directional  │    ▼     │    Users    │
│        │    DB    │◄─────────────────►│    DB    │             │
│        └──────────┘   Replication    └──────────┘             │
│                                                                 │
│  RPO: Near-zero (bi-directional sync)                          │
│  RTO: Seconds (automatic failover)                             │
│                                                                 │
│  ✓ Pros: Better latency, no wasted capacity                   │
│  ✗ Cons: Conflict resolution, data consistency complexity     │
│                                                                 │
│  Challenge: What if US and EU both update the same record?    │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

```python
from enum import Enum
from dataclasses import dataclass
from typing import List, Dict, Optional
import asyncio
import time

class ReplicationMode(Enum):
    SYNC = "synchronous"       # Wait for all replicas
    SEMI_SYNC = "semi-sync"    # Wait for N replicas
    ASYNC = "asynchronous"     # Fire and forget

@dataclass
class WriteResult:
    success: bool
    replicas_acked: int
    latency_ms: float

class MultiRegionDatabase:
    """
    Database with configurable replication strategies.
    """
    
    def __init__(
        self,
        local_region: str,
        regions: Dict[str, 'RegionConnection'],
        mode: ReplicationMode = ReplicationMode.ASYNC,
        min_replicas: int = 1
    ):
        self.local_region = local_region
        self.regions = regions
        self.mode = mode
        self.min_replicas = min_replicas  # For semi-sync
    
    async def write(self, key: str, value: str) -> WriteResult:
        """Write with configured replication strategy"""
        start = time.time()
        
        # Always write to local first
        await self.regions[self.local_region].write(key, value)
        
        remote_regions = [r for r in self.regions.values() 
                         if r.name != self.local_region]
        
        if self.mode == ReplicationMode.SYNC:
            # Wait for ALL replicas
            results = await asyncio.gather(
                *[r.write(key, value) for r in remote_regions],
                return_exceptions=True
            )
            acked = sum(1 for r in results if r is True)
            success = acked == len(remote_regions)
            
        elif self.mode == ReplicationMode.SEMI_SYNC:
            # Wait for N replicas
            acked = 0
            pending = [r.write(key, value) for r in remote_regions]
            
            for coro in asyncio.as_completed(pending):
                try:
                    await coro
                    acked += 1
                    if acked >= self.min_replicas:
                        # Don't wait for remaining
                        break
                except Exception:
                    pass
            
            success = acked >= self.min_replicas
            
            # Fire and forget remaining replications
            asyncio.create_task(self._complete_async_replication(pending))
            
        else:  # ASYNC
            # Fire and forget
            for region in remote_regions:
                asyncio.create_task(region.write(key, value))
            acked = 0
            success = True
        
        latency = (time.time() - start) * 1000
        return WriteResult(success, acked + 1, latency)
    
    async def read(
        self, 
        key: str, 
        consistency: str = "local"
    ) -> Optional[str]:
        """
        Read with configurable consistency.
        
        - local: Read from local region (fast, possibly stale)
        - leader: Read from leader region (consistent, slower)
        - quorum: Read from majority (consistent, medium latency)
        """
        if consistency == "local":
            return await self.regions[self.local_region].read(key)
        
        elif consistency == "leader":
            leader = self._get_leader_region()
            return await self.regions[leader].read(key)
        
        elif consistency == "quorum":
            # Read from majority, take latest value
            results = await asyncio.gather(
                *[r.read(key) for r in self.regions.values()],
                return_exceptions=True
            )
            
            valid_results = [r for r in results if not isinstance(r, Exception)]
            if len(valid_results) <= len(self.regions) // 2:
                raise Exception("Quorum not reached")
            
            # Return most recent value (assuming versioned)
            return max(valid_results, key=lambda x: x.version if x else 0)
    
    async def _complete_async_replication(self, pending):
        """Complete remaining async replications"""
        try:
            await asyncio.gather(*pending, return_exceptions=True)
        except Exception:
            pass  # Log and handle retry in production


class CRDTCounter:
    """
    Conflict-free Replicated Data Type for multi-region counters.
    Never loses increments, even with concurrent updates.
    """
    
    def __init__(self, region_id: str):
        self.region_id = region_id
        self.counts: Dict[str, int] = {}  # region_id -> count
    
    def increment(self, amount: int = 1):
        """Local increment - conflict-free"""
        self.counts[self.region_id] = self.counts.get(self.region_id, 0) + amount
    
    def value(self) -> int:
        """Total value across all regions"""
        return sum(self.counts.values())
    
    def merge(self, other: 'CRDTCounter'):
        """Merge with another counter - takes max of each region's count"""
        for region, count in other.counts.items():
            self.counts[region] = max(self.counts.get(region, 0), count)
    
    def to_dict(self) -> Dict[str, int]:
        return self.counts.copy()
    
    @classmethod
    def from_dict(cls, region_id: str, data: Dict[str, int]) -> 'CRDTCounter':
        counter = cls(region_id)
        counter.counts = data.copy()
        return counter


# Example: Global like counter
class GlobalLikeCounter:
    """
    Like counter that works across regions without conflicts.
    """
    
    def __init__(self, region: str, redis_client):
        self.region = region
        self.redis = redis_client
    
    async def like(self, post_id: str, user_id: str):
        """Increment like count and track user like"""
        # Track that user liked (for unlike functionality)
        await self.redis.sadd(f"likes:{post_id}:users", user_id)
        
        # Increment region counter
        await self.redis.hincrby(f"likes:{post_id}:counts", self.region, 1)
    
    async def unlike(self, post_id: str, user_id: str):
        """Decrement if user had liked"""
        removed = await self.redis.srem(f"likes:{post_id}:users", user_id)
        if removed:
            await self.redis.hincrby(f"likes:{post_id}:counts", self.region, -1)
    
    async def get_count(self, post_id: str) -> int:
        """Get total likes across all regions"""
        counts = await self.redis.hgetall(f"likes:{post_id}:counts")
        return sum(int(c) for c in counts.values())
    
    async def sync_with_region(self, post_id: str, remote_counts: Dict[str, int]):
        """Merge counts from another region"""
        local_counts = await self.redis.hgetall(f"likes:{post_id}:counts")
        
        # Take max for each region (CRDT merge)
        for region, count in remote_counts.items():
            local_count = int(local_counts.get(region, 0))
            if count > local_count:
                await self.redis.hset(
                    f"likes:{post_id}:counts", 
                    region, 
                    count
                )
```

## Conflict Resolution Strategies

Conflict resolution is the "who wins when two people edit the same thing at the same time?" problem. Think of it like two airline agents booking the last seat on a flight simultaneously from different cities. The system needs a deterministic way to resolve who gets the seat. Each strategy below trades off simplicity against data preservation -- and choosing the wrong one is a common source of production bugs in multi-region systems.

```
┌─────────────────────────────────────────────────────────────────┐
│                  CONFLICT RESOLUTION                            │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  1. LAST-WRITE-WINS (LWW)                                      │
│     • Use timestamp to pick winner                             │
│     • Simple but can lose data                                 │
│     • Good for: User preferences, settings                     │
│                                                                 │
│  2. FIRST-WRITE-WINS                                           │
│     • First value is kept, others discarded                    │
│     • Good for: Unique constraints, reservations               │
│                                                                 │
│  3. MERGE/CRDT                                                  │
│     • Mathematically combine values                            │
│     • Counters: Add together                                   │
│     • Sets: Union                                              │
│     • Good for: Counters, shopping carts                       │
│                                                                 │
│  4. APPLICATION RESOLUTION                                      │
│     • Store all versions                                       │
│     • Let application/user decide                              │
│     • Good for: Documents, complex merges                      │
│                                                                 │
│  5. REGION PRIORITY                                             │
│     • Designate "primary" region for conflicts                 │
│     • Simple, deterministic                                    │
│     • Good for: When one region is authoritative               │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### Failover Implementation

```python
from enum import Enum
from dataclasses import dataclass
from typing import List, Dict, Optional, Callable
import asyncio
import time

class RegionStatus(Enum):
    HEALTHY = "healthy"
    DEGRADED = "degraded"
    UNHEALTHY = "unhealthy"

@dataclass
class HealthCheck:
    region: str
    status: RegionStatus
    latency_ms: float
    last_check: float
    consecutive_failures: int

class GlobalFailoverManager:
    """
    Manages failover between regions based on health checks.
    """
    
    def __init__(
        self,
        regions: List[str],
        primary_region: str,
        health_check_fn: Callable,
        failure_threshold: int = 3,
        check_interval: float = 5.0
    ):
        self.regions = regions
        self.primary_region = primary_region
        self.current_primary = primary_region
        self.health_check_fn = health_check_fn
        self.failure_threshold = failure_threshold
        self.check_interval = check_interval
        
        self.health_status: Dict[str, HealthCheck] = {}
        self.failover_history: List[Dict] = []
        
        # Callbacks
        self.on_failover: Optional[Callable] = None
        self.on_failback: Optional[Callable] = None
    
    async def start_monitoring(self):
        """Start continuous health monitoring"""
        while True:
            await self._check_all_regions()
            await self._evaluate_failover()
            await asyncio.sleep(self.check_interval)
    
    async def _check_all_regions(self):
        """Check health of all regions"""
        tasks = [self._check_region(region) for region in self.regions]
        await asyncio.gather(*tasks, return_exceptions=True)
    
    async def _check_region(self, region: str):
        """Check health of a single region"""
        start = time.time()
        
        try:
            is_healthy = await self.health_check_fn(region)
            latency = (time.time() - start) * 1000
            
            if is_healthy:
                status = RegionStatus.HEALTHY if latency < 100 else RegionStatus.DEGRADED
                consecutive_failures = 0
            else:
                status = RegionStatus.UNHEALTHY
                prev = self.health_status.get(region)
                consecutive_failures = (prev.consecutive_failures + 1) if prev else 1
                
        except Exception as e:
            status = RegionStatus.UNHEALTHY
            latency = -1
            prev = self.health_status.get(region)
            consecutive_failures = (prev.consecutive_failures + 1) if prev else 1
        
        self.health_status[region] = HealthCheck(
            region=region,
            status=status,
            latency_ms=latency,
            last_check=time.time(),
            consecutive_failures=consecutive_failures
        )
    
    async def _evaluate_failover(self):
        """Decide if failover is needed"""
        primary_health = self.health_status.get(self.current_primary)
        
        if not primary_health:
            return
        
        # Check if primary has exceeded failure threshold
        if primary_health.consecutive_failures >= self.failure_threshold:
            # Find best healthy region
            new_primary = self._select_new_primary()
            
            if new_primary and new_primary != self.current_primary:
                await self._perform_failover(new_primary)
        
        # Check if original primary recovered (failback)
        elif (self.current_primary != self.primary_region and
              self.health_status.get(self.primary_region) and
              self.health_status[self.primary_region].status == RegionStatus.HEALTHY):
            
            # Original primary is healthy, consider failback
            await self._perform_failback()
    
    def _select_new_primary(self) -> Optional[str]:
        """Select best available region as new primary"""
        candidates = []
        
        for region, health in self.health_status.items():
            if region == self.current_primary:
                continue
            if health.status in [RegionStatus.HEALTHY, RegionStatus.DEGRADED]:
                candidates.append((region, health.latency_ms))
        
        if not candidates:
            return None
        
        # Select region with lowest latency
        candidates.sort(key=lambda x: x[1])
        return candidates[0][0]
    
    async def _perform_failover(self, new_primary: str):
        """Execute failover to new region"""
        old_primary = self.current_primary
        
        self.failover_history.append({
            "timestamp": time.time(),
            "type": "failover",
            "from": old_primary,
            "to": new_primary
        })
        
        self.current_primary = new_primary
        
        if self.on_failover:
            await self.on_failover(old_primary, new_primary)
        
        print(f"FAILOVER: {old_primary} → {new_primary}")
    
    async def _perform_failback(self):
        """Return to original primary region"""
        current = self.current_primary
        
        self.failover_history.append({
            "timestamp": time.time(),
            "type": "failback",
            "from": current,
            "to": self.primary_region
        })
        
        self.current_primary = self.primary_region
        
        if self.on_failback:
            await self.on_failback(current, self.primary_region)
        
        print(f"FAILBACK: {current} → {self.primary_region}")
    
    def get_current_primary(self) -> str:
        """Get current active primary region"""
        return self.current_primary
    
    def get_health_summary(self) -> Dict:
        """Get health status of all regions"""
        return {
            region: {
                "status": health.status.value,
                "latency_ms": health.latency_ms,
                "failures": health.consecutive_failures
            }
            for region, health in self.health_status.items()
        }
```

### User Home Region

```python
class UserHomeRegion:
    """
    Route users to their 'home' region based on their account location.
    Data stays in region for compliance.
    """
    
    def __init__(self, db):
        self.db = db
        self.region_mapping = {
            "US": ["us-east-1", "us-west-2"],
            "EU": ["eu-west-1", "eu-central-1"],
            "APAC": ["ap-northeast-1", "ap-southeast-1"],
        }
    
    async def get_user_region(self, user_id: str) -> str:
        """Get user's home region"""
        user = await self.db.get_user(user_id)
        
        # Determine region from user's country
        country = user.get("country", "US")
        
        if country in ["US", "CA", "MX"]:
            return self.region_mapping["US"][0]
        elif country in ["GB", "DE", "FR", "IT", "ES"]:
            return self.region_mapping["EU"][0]
        else:
            return self.region_mapping["APAC"][0]
    
    async def route_request(self, user_id: str, request):
        """Route request to user's home region"""
        home_region = await self.get_user_region(user_id)
        
        if self._is_local_region(home_region):
            # Process locally
            return await self._handle_locally(request)
        else:
            # Forward to home region
            return await self._forward_to_region(home_region, request)
```

## Choosing the Right Pattern: Trade-off Summary

| Pattern | Consistency | Latency | Ops Complexity | Cost | Best Fit |
|---------|-------------|---------|----------------|------|----------|
| Active-Passive | Strong | High (writes go to one region) | Low | $$ (standby wasted) | DR-only, small teams |
| Active-Active | Eventual | Low (local reads+writes) | Very High | $$$$ | Global consumer apps |
| Read Local, Write Global | Strong (writes), Stale (reads) | Low reads, High writes | Medium | $$$ | Read-heavy (social, content) |
| Geo-Partitioning | Strong (within partition) | Low (within partition) | Medium | $$$ | Compliance-heavy (healthcare, finance) |

<Tip>
**Practical tip -- a senior engineer would say:** "The cheapest multi-region strategy is active-passive with async replication and a well-tested failover runbook. Most teams that jump to active-active underestimate the conflict resolution complexity and end up with subtle data corruption bugs that take months to surface. Start with active-passive, automate the failover, and only move to active-active when you have concrete latency requirements that demand it."
</Tip>

### Common Questions

**Q: How do you handle a user who travels between regions?**
> Use sticky sessions or user home region. Route all their data operations to their home region regardless of current location.

**Q: What consistency model would you use for a global e-commerce cart?**
> CRDT-based cart that merges additions. "Add item" operations are commutative. For checkout, route to single region for consistency.

> Chaos engineering: regularly kill regions in production. GameDay exercises quarterly. Automated failover tests in staging.

  <Accordion title="Q1: You are designing a global e-commerce platform. A user in Frankfurt adds items to their cart, then boards a flight and lands in New York where they want to check out. How do you ensure the cart is available and consistent across regions?" icon="cart-shopping">
    **What the interviewer is really testing:** Whether you understand the tension between data locality, user mobility, and transactional consistency in a multi-region system -- and can pick the right pattern for different phases of a workflow.

    **Strong Answer:**
    - The cart is a read-heavy, write-light data structure during browsing, but it becomes a critical transactional object at checkout. These two phases have fundamentally different consistency requirements, so the architecture should treat them differently.
    - During browsing and cart modification, use a CRDT-based cart replicated across regions with eventual consistency. "Add item" and "remove item" operations are commutative and idempotent when modeled correctly (an OR-Set CRDT, for example). When the user lands in New York, the US-East replica merges with EU-West within seconds. The user sees their cart with negligible delay.
    - At checkout time, route the entire checkout flow to a single authoritative region -- the user's "home region" (EU-West, since they are a Frankfurt user). This means checkout latency for the user in New York includes a cross-Atlantic round trip (~100ms), but you get strong consistency for inventory reservation, payment, and order creation. This is an acceptable trade-off because checkout is a one-time action, not a repeated interaction.
    - The alternative -- running checkout in US-East against replicated data -- risks inventory oversell. If two users in different regions both try to buy the last item, eventual consistency means both regions might think the item is available. A single-region checkout with a distributed lock or serialized writes eliminates this.
    - Edge case: if the home region (EU-West) is down when the user tries to check out from New York, you need a fallback. Option one: queue the checkout and process it when EU-West recovers (bad UX). Option two: promote US-East as temporary checkout authority for this user, then reconcile later (complex but better UX). In practice, most companies accept the queuing approach for the rare scenario of both user travel and region failure.
    - **Example:** Amazon handles this by assigning each customer a "home marketplace" (amazon.de for the Frankfurt user). Cart data syncs globally, but the order is always placed against the home marketplace's inventory and payment systems. If the user explicitly switches to amazon.com, that is treated as a different marketplace with its own cart.

    **Follow-up: What if the user modifies their cart on the plane (offline) and the modifications conflict with a price change that happened while they were airborne?**

    You need a conflict resolution policy for price changes. The standard approach: cart items store a "price snapshot" at the time of addition, but checkout always re-validates against current prices. Show the user a "prices have changed" notification at checkout if any item's price differs from the snapshot. This is a better UX than silently charging a different amount. For CRDT merges, price is not part of the CRDT state -- only item IDs and quantities are. Price is resolved at read time from the catalog service.

  <Accordion title="Q2: Your company is expanding from US-only to a global presence. You need to comply with GDPR for EU users and data localization laws in India. How do you architect the data layer to satisfy these requirements without building entirely separate systems per region?" icon="globe">
    **What the interviewer is really testing:** Whether you can balance regulatory constraints against engineering pragmatism -- building compliant systems without the cost and complexity explosion of fully isolated regional deployments.

    **Strong Answer:**
    - The core principle is geo-partitioning at the data layer with a unified application layer. User data is stored in the region mandated by regulation, but the application code is the same everywhere. You are not building separate systems -- you are building one system with data residency awareness baked in.
    - Assign each user a "data home region" at account creation based on their country of residence. Store this mapping in a lightweight global directory (replicated everywhere, very small). All subsequent reads and writes for that user are routed to their home region. The application layer in any region can serve any user -- it just proxies data requests to the correct region.
    - For GDPR specifically: EU user data (PII, behavioral data, anything covered by the regulation) lives in EU-West. Non-PII data like aggregated analytics can be processed globally, but you must anonymize or pseudonymize before it leaves the EU region. Implement a data classification system: tag every field as PII, sensitive, or public. The routing layer uses these tags to enforce residency.
    - For India's data localization: payment data and certain categories of personal data must be stored on servers in India. This means you need an India region (e.g., ap-south-1) with at least a database for Indian user data. The application servers can be shared or regional depending on latency requirements.
    - Cross-region queries are the hard part. A global admin dashboard that shows "all users who signed up last week" cannot simply query all regions -- that would move PII across borders. Instead, pre-compute anonymized aggregates in each region and merge the aggregates centrally. The central dashboard sees counts and trends, never raw PII.
    - The trade-off: latency increases for users whose home region is far from their current location, and engineering complexity increases because every data access path must be region-aware. But you avoid the much worse alternative of maintaining entirely separate codebases per region.
    - **Example:** Shopify handles this with a "data residency" feature that pins merchant data to specific regions. Their application layer is global, but their storage layer respects residency. They invested heavily in making their ORM region-aware so that individual engineers do not need to think about data routing -- it happens automatically based on the merchant's configured region.

    **Follow-up: A product manager wants to build a global leaderboard showing the top 100 sellers across all regions. How do you build this without violating data residency rules?**

    Each region computes its own local top-N sellers using anonymized seller IDs (or pseudonymized IDs that cannot be reversed without the regional key). These local top-N lists are sent to a central aggregation service that merges them into a global top 100. The central service only sees pseudonymized IDs and revenue figures -- no PII crosses borders. If the leaderboard needs to display seller names, the frontend makes a separate request to the seller's home region to fetch the display name, which is returned directly to the user's browser (not stored centrally).

  <Accordion title="Q3: You are running an active-active multi-region setup and both US-East and EU-West receive a write to the same user's profile at the same time -- US-East changes the email, EU-West changes the display name. Walk me through exactly what happens and how you resolve it." icon="code-merge">
    **What the interviewer is really testing:** Whether you understand conflict resolution at a mechanical level -- not just "use CRDTs" as a hand-wave, but the actual data structures, merge semantics, and edge cases.

    **Strong Answer:**
    - With active-active replication and concurrent writes to different fields of the same record, the outcome depends entirely on your conflict resolution strategy. There are several options, and each has real consequences.
    - **Field-level last-write-wins (LWW):** The most practical approach for user profiles. Each field carries its own timestamp. US-East writes email with timestamp T1, EU-West writes display_name with timestamp T2. When replication delivers both writes to both regions, the merge logic compares per-field: email takes the US-East value (latest write to that field), display_name takes the EU-West value (latest write to that field). Both changes are preserved. This works because the writes target different fields.
    - The dangerous case is when both regions write the same field. If US-East sets email to alice@new.com at T1 and EU-West sets email to alice@work.com at T2 (T2 > T1), LWW picks alice@work.com and silently discards alice@new.com. The US-East user sees their change "disappear." This is why LWW requires careful clock synchronization -- if clocks are skewed, the "wrong" write can win.
    - **Vector clocks / version vectors:** More sophisticated. Each region maintains a vector of logical clocks. When a conflict is detected (concurrent writes, neither "happens before" the other), you know it is a true conflict rather than a sequential update that arrived out of order. You can then apply a deterministic resolution rule (LWW, region priority) or store both versions and let the application resolve.
    - **CRDTs for specific fields:** For fields like "tags" or "interests" (set-valued), use an OR-Set CRDT where additions from both regions are merged via union. For counters (like "login count"), use a G-Counter where each region's increments are independent and the total is the sum.
    - For a user profile specifically, I would use field-level LWW with vector clocks for conflict detection. Most profile updates are to different fields (name, email, avatar, bio), so conflicts are rare. When a true conflict occurs on the same field, LWW with a deterministic tiebreaker (region ID as secondary sort) ensures convergence. Log all conflicts for analysis -- if conflict rates are high, it signals a design problem.
    - **Example:** DynamoDB's global tables use LWW at the item level (not field level), which means a profile update in US-East that changes email will overwrite a concurrent EU-West update that changed display_name, because DynamoDB treats the entire item as the conflict unit. This is a known limitation. CockroachDB, by contrast, uses serializable isolation and will reject one of the concurrent writes, requiring the application to retry.

    **Follow-up: Your conflict rate is spiking because a background job in US-East updates the "last_login" field every time the user makes a request, while the user is actively editing their profile from EU-West. How do you fix this?**

    Separate hot metadata (last_login, request_count, session info) from cold profile data (name, email, bio). Hot metadata goes into a dedicated record or table that uses a CRDT counter/LWW register and is expected to have frequent concurrent writes. Cold profile data uses single-leader writes (route profile edits to the user's home region) so conflicts cannot occur. This eliminates the cross-contamination where a high-frequency background update causes conflicts with low-frequency user edits.
  </Accordion>

  <Accordion title="Q4: Your active-passive disaster recovery setup has an RPO of 5 minutes (async replication lag). An outage hits the primary region. How do you decide whether to failover, and what data loss scenarios do you communicate to stakeholders?" icon="triangle-exclamation">
    **What the interviewer is really testing:** Whether you understand that failover is a business decision, not just a technical one -- and whether you can reason about the gap between "last replicated state" and "actual state at failure time."

    **Strong Answer:**
    - The first question is not "how do we failover" but "should we failover." Failover to a secondary region with 5 minutes of replication lag means accepting up to 5 minutes of data loss. Depending on the business, that could mean 5 minutes of lost orders, lost financial transactions, or lost user registrations. The decision depends on comparing the cost of data loss against the cost of extended downtime.
    - Decision framework: (1) Is the outage confirmed and likely to last longer than your RTO tolerance? A 2-minute blip that auto-recovers is worse to failover for than to ride out -- failover itself introduces risk. (2) What is the current replication lag? If the async replication was caught up to within 10 seconds before the outage, your actual data loss is 10 seconds, not 5 minutes. Check the replication lag metrics. (3) What is the business impact of continued downtime vs. data loss? For a social media platform, 5 minutes of lost posts is annoying but recoverable. For a financial trading platform, 5 minutes of lost trades is catastrophic and you might prefer to wait for the primary to recover.
    - When you decide to failover: (1) Promote the secondary to primary. (2) Update DNS or load balancer to point to the new primary. (3) Bring up application servers in the secondary region. (4) Monitor closely for issues -- the secondary has not been handling production traffic and may have cold caches, unwarmed connection pools, etc. Expect degraded performance for the first few minutes.
    - The data loss conversation with stakeholders: "We are failing over to US-West. Based on our replication monitoring, the last confirmed replication checkpoint was 3 minutes and 12 seconds ago. This means any data written in the last 3 minutes and 12 seconds in US-East may not be in US-West. For our platform, that is approximately 45,000 API writes, 12,000 new user sessions, and 3,200 orders. We will reconcile these when US-East comes back online using our transaction log comparison."
    - After the original primary recovers: do NOT immediately failback. First, compare the transaction logs between both regions to identify the gap. Replay or reconcile the missing transactions. Only after reconciliation is complete should you consider failing back, and even then, validate with a parallel-read test before switching traffic.
    - **Example:** GitHub experienced a 24-hour incident in 2018 caused by a 43-second network partition. Their MySQL primary in US-East failed over to US-West, but the replication lag meant some writes were lost. They spent days reconciling data and ultimately had to surface the inconsistencies to affected users. The lesson: failover is the easy part; reconciliation is where the real complexity lives.

    **Follow-up: After failover, the original primary comes back online with 5 minutes of data that the new primary does not have. Some of that data conflicts with writes that users made against the new primary during the outage. How do you reconcile?**

    This is a split-brain reconciliation problem. Export the unreconciled writes from the old primary (everything after the last confirmed replication checkpoint). Compare each write against the new primary's state. For non-conflicting writes (new records, updates to records that were not touched on the new primary), replay them. For conflicts (same record updated on both sides), apply your conflict resolution policy -- typically, the new primary wins because those writes reflect the user's most recent intent. Log all discarded writes and notify affected users or systems. For financial data, every discarded write must go through manual review.
