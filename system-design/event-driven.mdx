## Introduction

Event-driven architecture (EDA) is the backbone of modern distributed systems. From real-time notifications to complex business workflows, understanding EDA is crucial for system design interviews. The fundamental shift from request-driven to event-driven thinking is this: instead of Service A calling Service B directly ("Hey B, process this order"), Service A announces a fact about the world ("An order was placed") and anyone who cares can react to it. This feels like a small syntactic difference, but it changes everything -- services become decoupled, new consumers can be added without modifying the producer, and the system naturally handles asynchronous workflows.

The real-world analogy: request-driven architecture is like a manager assigning tasks to specific employees by name ("Sarah, file this report"). Event-driven architecture is like posting a task on a shared board ("Report needs filing") -- whoever is available and capable picks it up. The board approach scales better, survives individual employees being unavailable, and does not require the poster to know who will do the work.

<Note>
**Why This Matters**: Companies like Netflix, Uber, and LinkedIn process billions of events daily. Understanding event-driven patterns shows you can design scalable, decoupled systems.
</Note>

## Event-Driven Patterns

### 1. Event Notification

```
                    Event Published
                          │
      ┌───────────────────┼───────────────────┐
      │                   │                   │
      ▼                   ▼                   ▼
┌──────────┐       ┌──────────┐       ┌──────────┐
│Service A │       │Service B │       │Service C │
│          │       │          │       │          │
│ Reacts   │       │ Reacts   │       │ Ignores  │
└──────────┘       └──────────┘       └──────────┘

Use Case: Loose coupling, simple notifications
Example: "UserRegistered" → Email service sends welcome email
```

### 2. Event-Carried State Transfer

```
Event with full state:
{
  "type": "order.updated",
  "data": {
    "order_id": "123",
    "status": "shipped",
    "items": [...],          ← Full order data
    "shipping_address": {...},
    "customer": {...}
  }
}

Benefits:
• Consumers don't need to query source
• Reduces coupling
• Works offline

Drawbacks:
• Larger message size
• Data might be stale
```

### 3. Event Sourcing

Event Sourcing is arguably the most powerful (and most misunderstood) pattern in this chapter. Instead of storing current state ("balance = $525"), you store the sequence of events that led to that state. Think of it like an accounting ledger: you do not erase and rewrite the balance; you record every credit and debit, and the balance is always derivable by replaying the ledger. This gives you a complete audit trail for free, the ability to reconstruct state at any point in time, and the ability to build new read models by replaying history against new logic.

The trade-off: event-sourced systems are more complex to query (you cannot simply SELECT current state; you must replay or maintain materialized views), and the event store grows forever. You mitigate this with snapshots (periodically saving the materialized state so you do not replay from the beginning of time) and projections (maintaining read-optimized views that are updated as new events arrive).

```
┌─────────────────────────────────────────────────────────────┐
│                     Event Store                              │
├─────────────────────────────────────────────────────────────┤
│ 1. AccountOpened(id=A1, owner="John", balance=0)            │
│ 2. MoneyDeposited(id=A1, amount=1000)                       │
│ 3. MoneyWithdrawn(id=A1, amount=200)                        │
│ 4. MoneyTransferred(from=A1, to=A2, amount=300)             │
│ 5. InterestApplied(id=A1, rate=0.05)                        │
├─────────────────────────────────────────────────────────────┤
│                         │                                    │
│                         ▼                                    │
│              Replay Events to Get State                      │
│                         │                                    │
│                         ▼                                    │
│              Current Balance: $525                           │
└─────────────────────────────────────────────────────────────┘
```

from abc import ABC, abstractmethod
from dataclasses import dataclass, field
from datetime import datetime
from typing import List, Dict, Any, Optional
import json

class DomainEvent(ABC):
    """Base class for domain events."""
    
    @abstractmethod
    def apply(self, aggregate):
        """Apply event to aggregate."""
        pass


@dataclass
class AccountOpened(DomainEvent):
    account_id: str
    owner: str
    timestamp: datetime = field(default_factory=datetime.utcnow)
    
    def apply(self, account):
        account.account_id = self.account_id
        account.owner = self.owner
        account.balance = 0
        account.is_open = True


@dataclass
class MoneyDeposited(DomainEvent):
    account_id: str
    amount: float
    timestamp: datetime = field(default_factory=datetime.utcnow)
    
    def apply(self, account):
        account.balance += self.amount


@dataclass
class MoneyWithdrawn(DomainEvent):
    account_id: str
    amount: float
    timestamp: datetime = field(default_factory=datetime.utcnow)
    
    def apply(self, account):
        account.balance -= self.amount


class BankAccount:
    """Event-sourced bank account aggregate."""
    
    def __init__(self):
        self.account_id: Optional[str] = None
        self.owner: Optional[str] = None
        self.balance: float = 0
        self.is_open: bool = False
        self._events: List[DomainEvent] = []
        self._version: int = 0
    
    def open_account(self, account_id: str, owner: str):
        """Command: Open a new account."""
        if self.is_open:
            raise ValueError("Account already open")
        
        event = AccountOpened(account_id, owner)
        self._apply_event(event)
        self._events.append(event)
    
    def deposit(self, amount: float):
        """Command: Deposit money."""
        if amount <= 0:
            raise ValueError("Amount must be positive")
        if not self.is_open:
            raise ValueError("Account not open")
        
        event = MoneyDeposited(self.account_id, amount)
        self._apply_event(event)
        self._events.append(event)
    
    def withdraw(self, amount: float):
        """Command: Withdraw money."""
        if amount <= 0:
            raise ValueError("Amount must be positive")
        if not self.is_open:
            raise ValueError("Account not open")
        if self.balance < amount:
            raise ValueError("Insufficient funds")
        
        event = MoneyWithdrawn(self.account_id, amount)
        self._apply_event(event)
        self._events.append(event)
    
    def _apply_event(self, event: DomainEvent):
        """Apply event to current state."""
        event.apply(self)
        self._version += 1
    
    def get_uncommitted_events(self) -> List[DomainEvent]:
        """Get events not yet persisted."""
        return self._events.copy()
    
    def clear_uncommitted_events(self):
        """Clear events after persistence."""
        self._events.clear()
    
    @classmethod
    def from_events(cls, events: List[DomainEvent]) -> 'BankAccount':
        """Reconstruct account from event history."""
        account = cls()
        for event in events:
            account._apply_event(event)
        return account


class EventStore:
    """Simple in-memory event store."""
    
    def __init__(self):
        self._events: Dict[str, List[Dict]] = {}
    
    def save_events(
        self, 
        aggregate_id: str, 
        events: List[DomainEvent],
        expected_version: int
    ):
        """Save events with optimistic concurrency."""
        stream = self._events.get(aggregate_id, [])
        
        if len(stream) != expected_version:
            raise ConcurrencyError(
                f"Expected version {expected_version}, "
                f"but stream has {len(stream)} events"
            )
        
        for event in events:
            stream.append({
                "type": type(event).__name__,
                "data": event.__dict__,
                "version": len(stream) + 1
            })
        
        self._events[aggregate_id] = stream
    
    def get_events(self, aggregate_id: str) -> List[DomainEvent]:
        """Load all events for an aggregate."""
        stream = self._events.get(aggregate_id, [])
        
        events = []
        for record in stream:
            event_class = globals()[record["type"]]
            events.append(event_class(**record["data"]))
        
        return events


class ConcurrencyError(Exception):
    pass


# Usage example
def example():
    store = EventStore()
    
    # Create new account
    account = BankAccount()
    account.open_account("ACC-001", "John Doe")
    account.deposit(1000)
    account.withdraw(200)
    
    # Save to event store
    store.save_events(
        "ACC-001",
        account.get_uncommitted_events(),
        0
    )
    account.clear_uncommitted_events()
    
    # Reconstruct from events
    events = store.get_events("ACC-001")
    reconstructed = BankAccount.from_events(events)
    
    print(f"Balance: ${reconstructed.balance}")  # $800

class DomainEvent {
    constructor() {
        this.timestamp = new Date().toISOString();
    }
    
    apply(aggregate) {
        throw new Error('Must implement apply method');
    }
}


class AccountOpened extends DomainEvent {
    constructor(accountId, owner) {
        super();
        this.accountId = accountId;
        this.owner = owner;
    }
    
    apply(account) {
        account.accountId = this.accountId;
        account.owner = this.owner;
        account.balance = 0;
        account.isOpen = true;
    }
}


class MoneyDeposited extends DomainEvent {
    constructor(accountId, amount) {
        super();
        this.accountId = accountId;
        this.amount = amount;
    }
    
    apply(account) {
        account.balance += this.amount;
    }
}


class MoneyWithdrawn extends DomainEvent {
    constructor(accountId, amount) {
        super();
        this.accountId = accountId;
        this.amount = amount;
    }
    
    apply(account) {
        account.balance -= this.amount;
    }
}


class BankAccount {
    constructor() {
        this.accountId = null;
        this.owner = null;
        this.balance = 0;
        this.isOpen = false;
        this._events = [];
        this._version = 0;
    }
    
    openAccount(accountId, owner) {
        if (this.isOpen) {
            throw new Error('Account already open');
        }
        
        const event = new AccountOpened(accountId, owner);
        this._applyEvent(event);
        this._events.push(event);
    }
    
    deposit(amount) {
        if (amount <= 0) throw new Error('Amount must be positive');
        if (!this.isOpen) throw new Error('Account not open');
        
        const event = new MoneyDeposited(this.accountId, amount);
        this._applyEvent(event);
        this._events.push(event);
    }
    
    withdraw(amount) {
        if (amount <= 0) throw new Error('Amount must be positive');
        if (!this.isOpen) throw new Error('Account not open');
        if (this.balance < amount) throw new Error('Insufficient funds');
        
        const event = new MoneyWithdrawn(this.accountId, amount);
        this._applyEvent(event);
        this._events.push(event);
    }
    
    _applyEvent(event) {
        event.apply(this);
        this._version++;
    }
    
    getUncommittedEvents() {
        return [...this._events];
    }
    
    clearUncommittedEvents() {
        this._events = [];
    }
    
    static fromEvents(events) {
        const account = new BankAccount();
        events.forEach(event => account._applyEvent(event));
        return account;
    }
}


class EventStore {
    constructor() {
        this._streams = new Map();
    }
    
    saveEvents(aggregateId, events, expectedVersion) {
        const stream = this._streams.get(aggregateId) || [];
        
        if (stream.length !== expectedVersion) {
            throw new ConcurrencyError(
                `Expected version ${expectedVersion}, ` +
                `but stream has ${stream.length} events`
            );
        }
        
        events.forEach(event => {
            stream.push({
                type: event.constructor.name,
                data: { ...event },
                version: stream.length + 1
            });
        });
        
        this._streams.set(aggregateId, stream);
    }
    
    getEvents(aggregateId) {
        const stream = this._streams.get(aggregateId) || [];
        
        const eventClasses = {
            AccountOpened,
            MoneyDeposited,
            MoneyWithdrawn
        };
        
        return stream.map(record => {
            const EventClass = eventClasses[record.type];
            return Object.assign(new EventClass(), record.data);
        });
    }
}


class ConcurrencyError extends Error {
    constructor(message) {
        super(message);
        this.name = 'ConcurrencyError';
    }
}

module.exports = {
    BankAccount,
    EventStore,
    AccountOpened,
    MoneyDeposited,
    MoneyWithdrawn
};

## Message Broker Patterns

### Topic Design

from typing import Callable, Dict, Any

class DeadLetterHandler:

        self, 
        dlq_store, 
        original_producer,
        max_retries: int = 3,
        retry_delay: timedelta = timedelta(minutes=5)

        self.producer = original_producer
        self.max_retries = max_retries
        self.retry_delay = retry_delay
        self.logger = logging.getLogger(__name__)
    
    async def handle_failure(
        self,
        message: Dict[str, Any],
        error: Exception,
        topic: str,
        retry_count: int = 0
    ):
        """Send failed message to DLQ."""
        entry = DeadLetterEntry(
            original_message=message,
            error=str(error),
            retry_count=retry_count + 1,
            first_failure=datetime.utcnow(),
            last_failure=datetime.utcnow(),
            topic=topic
        )
        
        await self.dlq_store.save(entry)
        
        self.logger.warning(
            f"Message sent to DLQ: {message.get('id')} "
            f"(retry {retry_count + 1})"
        )
        
        if retry_count + 1 >= self.max_retries:
            await self.alert_operations(entry)
    
    async def process_dlq(self):
        """Background job to retry DLQ messages."""
        while True:
            entries = await self.dlq_store.get_retryable(
                max_retries=self.max_retries,
                min_age=self.retry_delay
            )
            
            for entry in entries:
                try:
                    # Retry sending to original topic
                    await self.producer.send(
                        entry.topic,
                        entry.original_message
                    )
                    await self.dlq_store.mark_success(entry)
                    
                except Exception as e:
                    # Update retry count
                    entry.retry_count += 1
                    entry.last_failure = datetime.utcnow()
                    entry.error = str(e)

                action=lambda: self.shipping_service.schedule(
                    order_data["order_id"],
                    order_data["shipping_address"]
                ),
                compensation=lambda: self.shipping_service.cancel(
                    order_data["order_id"]
                )
            )
        ]
        
        try:
            for step in steps:
                result = await step.action()
                step.completed = True
                self.completed_steps.append(step)
                
                # Store result for later steps
                if step.name == "process_payment":
                    order_data["payment_id"] = result["payment_id"]
            
            self.state = SagaState.COMPLETED
            return self.saga_id
            
        except Exception as e:
            # Compensate in reverse order
            await self.compensate()
            raise SagaFailedException(f"Saga failed: {e}")
    
    async def compensate(self):
        """Execute compensation in reverse order."""
        self.state = SagaState.COMPENSATING
        
        for step in reversed(self.completed_steps):
            try:
                await step.compensation()
            except Exception as e:
                # Log and continue - compensation must be idempotent
                print(f"Compensation failed for {step.name}: {e}")
        
        self.state = SagaState.FAILED

## Best Practices

### Event Design Principles

```
✓ Events are immutable - never modify past events
✓ Events are self-describing - include all needed context
✓ Use past tense - "OrderPlaced", not "PlaceOrder"
✓ Version your events - for schema evolution
✓ Include correlation IDs - for tracing
✓ Keep events small - but with enough context
```

### Idempotency

<Warning>
**Critical**: Event handlers MUST be idempotent. Events may be delivered more than once.
</Warning>

```
// ❌ Not idempotent
async function handleOrderPlaced(event) {
    await incrementOrderCount();  // Will double-count on retry
}

// ✅ Idempotent
async function handleOrderPlaced(event) {
    const exists = await checkProcessed(event.id);
    if (exists) return;
    
    await incrementOrderCount();
    await markProcessed(event.id);
}
```

## Interview Questions

<AccordionGroup>
  <Accordion title="What is the difference between an Event, a Command, and a Query? Why does the distinction matter in event-driven systems?">
    **Strong Answer:**
    - An Event is an immutable fact about something that already happened, named in past tense: `OrderPlaced`, `PaymentProcessed`. It is a broadcast -- the producer does not know or care who consumes it. A Command is an intent to do something, named in imperative: `PlaceOrder`, `ChargePayment`. It is point-to-point -- it targets a specific handler and expects it to act. A Query is a request for data: `GetOrder`. It expects a synchronous response and has no side effects.
    - The distinction matters because it drives coupling decisions. Events decouple producers from consumers -- the Order Service publishes `OrderPlaced` without knowing that the Email Service, Analytics Service, and Inventory Service all react to it. You can add a new consumer tomorrow without modifying the producer. Commands create direct coupling -- the sender must know who handles the command. This is intentional when you need guaranteed handling (exactly one service should process `ChargePayment`).
    - A common mistake is using events as disguised commands. If `OrderPlaced` is published with the expectation that the Payment Service must process it, that is a command in event clothing. True events allow consumers to ignore them. If no consumer processes the event, that should be acceptable. If it is not acceptable, you actually have a command.
    - In practice, most systems use a mix: events for fan-out notifications (loose coupling) and commands for directed actions where you need acknowledgment. CQRS formalizes the separation -- commands modify state, queries read state, and events propagate changes.

    **Red flag answer:** "They are all just messages on a queue" -- this collapses a critical architectural distinction. Or "events are asynchronous and commands are synchronous" -- this confuses transport mechanism with semantic intent. Commands can be asynchronous (a command on a queue), and queries are synchronous, but the real difference is about intent, ownership, and coupling.

    **Follow-ups:**
    1. You discover that your team has been publishing "events" that are really commands -- for example, `SendEmailToCustomer` is published to a topic and only the Email Service consumes it. What problems does this cause and how do you refactor?
    2. A new Product Catalog Service needs to react to `OrderPlaced` events. Who is responsible for the contract of the event schema -- the Order Service (producer) or the Product Catalog Service (consumer)?
  </Accordion>

  <Accordion title="Explain Event Sourcing. Why would you choose it over traditional CRUD, and what are the operational costs you need to be prepared for?">
    **Strong Answer:**
    - Event Sourcing stores every state change as an immutable event in an append-only log, rather than overwriting the current state in a database row. To get the current state of an entity, you replay all its events from the beginning. Think of it like a bank ledger: instead of storing "balance = $525," you store every deposit and withdrawal, and the balance is derived by replaying the ledger.
    - The benefits are compelling for the right use cases: (1) Full audit trail for free -- you know exactly what happened, when, and in what order. Essential for financial systems, healthcare, and compliance-heavy domains. (2) Temporal queries -- you can reconstruct the state at any past point in time by replaying events up to that timestamp. "What was the inventory level at 3pm yesterday?" is trivial. (3) Debugging -- when a bug produces wrong state, you can replay events through fixed logic to get correct state without a risky data migration. (4) New read models -- if you later need a new view of the data (e.g., a report that groups orders by region), you replay existing events through a new projection without any backfill job.
    - The operational costs: (1) The event store grows forever. A bank account opened 10 years ago with daily transactions has tens of thousands of events. Replaying all of them to load current state is slow. You mitigate with snapshots -- periodically save the materialized state and only replay events after the snapshot. (2) Querying is painful without projections. You cannot `SELECT * FROM accounts WHERE balance > 1000` against an event store. You must build read models (projections) that materialize the views you need, kept in sync by consuming events. This is where CQRS comes in. (3) Schema evolution -- event version 1 had `amount` as an integer (cents). Version 2 uses a decimal. Now your replay logic must handle both formats. You need upcasting (transforming old events to the new schema during replay). (4) Eventual consistency -- projections lag behind writes, so users may not see their own changes immediately without careful handling.
    - When to choose it: financial systems, audit-heavy domains, systems where business wants to answer "what if" questions about historical data. When to avoid: simple CRUD applications, prototypes, domains where history is not valuable, and teams without experience in eventual consistency patterns.

    **Red flag answer:** "Event Sourcing gives you better performance because you only append" -- this confuses write performance (which is good) with read performance (which is worse without projections). Or "we use Event Sourcing for everything" -- this is a massive over-engineering red flag for most domains.

    **Follow-ups:**
    1. Your event-sourced aggregate has 50,000 events. Loading it takes 3 seconds. Walk me through how you implement snapshotting, and how do you decide the snapshot frequency?
    2. You need to delete a customer's data for GDPR compliance, but your event store is append-only and immutable. How do you reconcile "right to be forgotten" with event sourcing?
  </Accordion>

  <Accordion title="What is CQRS and how does it relate to Event Sourcing? Can you use one without the other?">
    **Strong Answer:**
    - CQRS -- Command Query Responsibility Segregation -- means using a different model for writing data (commands) and reading data (queries). Instead of one database table that handles both inserts/updates and SELECT queries, you have a write model optimized for business rule enforcement and a read model optimized for queries. The write model might be a normalized relational schema or an event store; the read model might be a denormalized document, a search index, or a materialized view.
    - CQRS and Event Sourcing are independent patterns that are frequently combined but do not require each other. You can do CQRS without Event Sourcing: write to a normalized PostgreSQL schema, publish change events, and build denormalized read models in Elasticsearch or Redis. You can do Event Sourcing without CQRS: store events and replay them for both writes and reads (though this is painful for complex queries, which is why the two are usually combined).
    - Why they pair well: Event Sourcing gives you a natural stream of events. CQRS gives you a reason to consume those events -- building read-optimized projections. The write side stores events; the read side subscribes to those events and maintains query-optimized views. The Event Sourcing side handles "make this change correctly" and the CQRS read side handles "answer this query fast."
    - When CQRS alone is valuable: when your read and write patterns have very different characteristics. An e-commerce product catalog with 100x more reads than writes is a classic case -- the write side enforces business rules (price validation, category constraints), while the read side serves product listings from a denormalized store optimized for search and filtering. You do not need Event Sourcing for this; a simple change data capture pipeline from the write database to the read database works.
    - The cost of CQRS: eventual consistency between write and read models. A user creates an order (write), then immediately views "My Orders" (read from projection). If the projection has not caught up, the new order is missing. Solutions include: write-through caching (update the read model synchronously for the current user), optimistic UI (show the item immediately on the client side), or versioned reads (pass the write version to the read side and wait for the projection to catch up).

    **Red flag answer:** "CQRS is the same as Event Sourcing" -- conflating two distinct patterns. Or "you always need CQRS in microservices" -- CQRS adds significant complexity and is only warranted for specific read/write asymmetries.

    **Follow-ups:**
    1. Your CQRS read projection crashes and you need to rebuild it from scratch. If you are using Event Sourcing, you can replay events. But what if you are doing CQRS without Event Sourcing -- how do you rebuild the read model?
    2. You have 5 different read models (projections) consuming from the same event stream. One of them is lagging 2 minutes behind. How do you diagnose and fix this without affecting the other 4 projections?
  </Accordion>

  <Accordion title="How do you guarantee exactly-once processing in an event-driven system? Is it truly possible?">
    **Strong Answer:**
    - True exactly-once processing across distributed systems is theoretically impossible in the general case (this is related to the Two Generals problem and the impossibility results around distributed consensus with unreliable networks). What we actually achieve is "effectively exactly-once" through a combination of at-least-once delivery and idempotent consumers.
    - At-least-once delivery means the broker guarantees every message is delivered, but may deliver it more than once (due to retries, consumer crashes before acking, network issues). Most production message brokers (Kafka, RabbitMQ, SQS) provide this by default. At-most-once delivery (fire and forget, no retries) loses messages. Exactly-once delivery from the broker is either not possible or extremely expensive (Kafka's exactly-once semantics use transactional producers and are limited to within-Kafka processing).
    - The practical solution: make your consumers idempotent. An idempotent consumer produces the same result whether it processes a message once or ten times. There are several strategies: (1) Idempotency key: store a processed-event-ID set. Before processing, check if the event ID has been seen. If yes, skip. This requires an atomic check-and-process operation -- use a database transaction that inserts the event ID and performs the business logic together. (2) Natural idempotency: design operations to be inherently idempotent. `SET balance = 500` is idempotent; `SET balance = balance + 100` is not. Use absolute values instead of deltas when possible. (3) Conditional writes: use version numbers or ETags. "Update order status to shipped WHERE version = 5" -- if the event was already processed and the version is now 6, the update is a no-op.
    - Kafka's transactional exactly-once (EOS) works for stream processing within Kafka (consume from topic A, process, produce to topic B atomically). But the moment your consumer writes to an external database, you are back to at-least-once plus idempotency, because there is no transaction spanning Kafka and your database.
    - The real-world gotcha: idempotency checks have their own failure modes. If you store processed event IDs in a separate table and the process crashes between committing the business logic and storing the event ID, you will process the event again. The fix: store the event ID in the same transaction as the business state change.

    **Red flag answer:** "Kafka has exactly-once semantics so we do not need to worry about duplicates" -- this shows a shallow understanding of where Kafka EOS applies. Or "we just use acknowledge mode and that guarantees exactly-once" -- conflating delivery acknowledgment with processing guarantees.

    **Follow-ups:**
    1. Your idempotency table has 500 million entries after a year of processing. Lookups are getting slow. How do you manage the size of the deduplication store without risking reprocessing old events?
    2. Your consumer processes an event and writes to the database, but crashes before sending the acknowledgment to the broker. The broker redelivers the event. Your idempotency check catches the duplicate because you stored the event ID in the same transaction. But what if your consumer writes to two databases (e.g., PostgreSQL and Elasticsearch)? How do you ensure both are consistent?
  </Accordion>

  <Accordion title="What is a Dead Letter Queue (DLQ) and how do you design one for a production event-driven system?">
    **Strong Answer:**
    - A DLQ is where messages go to die -- or more precisely, where messages go when they have failed processing after all retry attempts. Instead of discarding failed messages (losing data) or retrying forever (blocking the queue), you move them to a separate queue for investigation and manual or automated recovery.
    - The design: (1) The main consumer attempts processing. On failure, it retries with exponential backoff (1s, 2s, 4s, 8s). After N retries (typically 3-5), it moves the message to the DLQ with metadata: the original topic, the error message, the retry count, the first and last failure timestamps, and the stack trace. (2) The DLQ has its own consumer -- either a dashboard for manual investigation or an automated re-processor that retries messages after a longer delay (e.g., 1 hour). (3) Messages in the DLQ have an expiration policy -- if not reprocessed within 14 days, archive them to cold storage or discard them.
    - The critical detail most people miss: you must understand why a message failed before retrying it. There are two categories of failures. Transient failures (network timeout, service temporarily unavailable, rate limit hit) are worth retrying -- the same message will likely succeed later. Permanent failures (invalid message format, business rule violation, missing required field) will never succeed no matter how many times you retry. Retrying a malformed message forever is a waste of resources. Your DLQ should categorize messages by failure type and only auto-retry transient failures.
    - In production, the DLQ needs monitoring: alert when the DLQ depth exceeds a threshold (e.g., more than 100 messages, or growing faster than 10/minute). A growing DLQ is a leading indicator of a systemic issue -- maybe a downstream service is down, or a recent deployment introduced a bug that rejects a class of messages.
    - Kafka-specific note: Kafka does not have a built-in DLQ. You implement it by producing failed messages to a separate `*.dlq` topic. Spring Kafka and other frameworks provide this out of the box. SQS has native DLQ support with configurable `maxReceiveCount`.

    **Red flag answer:** "We just retry failed messages forever" -- this blocks the queue and causes cascading backpressure. Or "if a message fails, we log the error and move on" -- this silently drops data with no recovery path.

    **Follow-ups:**
    1. Your DLQ has 50,000 messages that all failed because of a bug in the consumer code. You fix the bug. How do you replay 50,000 DLQ messages back to the main topic without overwhelming the system?
    2. A poison pill message (syntactically valid but causes the consumer to crash) enters the main queue. It gets retried 3 times, crashing the consumer each time, then goes to the DLQ. But before reaching the DLQ, it has consumed all 3 retries and caused 3 consumer restarts. How do you detect and isolate poison pill messages faster?
  </Accordion>

  <Accordion title="Explain the difference between choreography and orchestration in the context of event-driven sagas. How do you debug a failing choreography-based saga in production?">
    **Strong Answer:**
    - In choreography, each service independently reacts to events and publishes its own events. There is no central coordinator -- the saga's flow is an emergent property of the event chain. Order Service publishes `OrderCreated`, Payment Service subscribes and publishes `PaymentCharged`, Inventory Service subscribes and publishes `StockReserved`, and so on. Each service only knows about its own immediate triggers and outputs.
    - In orchestration, a dedicated Saga Orchestrator coordinates the entire flow. It explicitly says: "First call Order Service, then call Payment Service, then call Inventory Service." The orchestrator tracks which step the saga is on, handles failures, and initiates compensations. The flow logic lives in one place.
    - Debugging a choreography saga in production is notoriously difficult because the flow is distributed. Here is a systematic approach: (1) Correlation IDs are non-negotiable. Every event in a saga must carry a `correlation_id` that links all events in the same business transaction. When investigating a failure, search all topics for events with the matching correlation ID. This shows you the exact event chain: which events were published, which services consumed them, and where the chain broke. (2) Build a saga state view: a dedicated "saga tracker" service that consumes all events, groups them by correlation ID, and materializes the current state of each saga in a queryable store. This gives you a dashboard showing "Order 12345: OrderCreated (done), PaymentCharged (done), StockReserved (failed), ShippingScheduled (not started)." (3) Distributed tracing: if you instrument event publishing and consumption with OpenTelemetry spans, the trace shows the full saga flow as a single trace with spans in each service. (4) For the specific failure: look at the DLQ of the service that failed. The error message and stack trace, combined with the correlation ID, tell you what went wrong.
    - The reason orchestration is often preferred for complex sagas: all of this debugging infrastructure is built into the orchestrator for free. The orchestrator already tracks which step the saga is on, what failed, and what needs compensation. You query one table instead of searching across 8 services.

    **Red flag answer:** "Choreography does not need debugging because each service is independent" -- this ignores that business transactions span services regardless of how they are coordinated. Or "we just check the logs" without mentioning correlation IDs -- this means the candidate has never debugged a real distributed saga.

    **Follow-ups:**
    1. In your choreography saga, Service C publishes an event that both Service D and Service E should react to. Service D succeeds but Service E fails. How does compensation work -- does Service D need to undo its work even though it succeeded?
    2. You are converting a choreography saga to orchestration. During the migration, some sagas are running under the old choreography model and some under the new orchestrator. How do you ensure no sagas are lost or duplicated during the cutover?
  </Accordion>

  <Accordion title="How do you handle schema evolution in an event-driven system? A field in your event schema needs to change, but you have 3 years of historical events in the store.">
    **Strong Answer:**
    - Schema evolution is one of the hardest long-term challenges in event-driven systems, especially with Event Sourcing where events are immutable and historical events are replayed. You cannot just change the schema and deploy -- old consumers will break on new events, and new consumers will break on old events.
    - The strategies, from simplest to most robust: (1) Always additive changes: add new fields with defaults, never remove or rename fields. This is backward and forward compatible. Old consumers ignore the new field; new consumers use the default if the old field is absent. This works for 80% of schema changes but gets messy over time as events accumulate optional fields. (2) Schema versioning: include a `schema_version` field in every event. Consumers check the version and use the appropriate deserialization logic. Version 1 has `amount` as cents (integer); version 2 has `amount` as a decimal string. The consumer handles both. This works but pushes complexity into every consumer. (3) Upcasting: when replaying events, transform old events to the latest schema version before processing. An upcaster converts a version 1 event to version 2 format in-flight. This centralizes the migration logic and keeps consumer code clean. The event store retains the original events; the transformation happens at read time. (4) Schema registry (like Confluent Schema Registry for Kafka): enforce compatibility rules at the broker level. Configure "backward compatible" -- new schemas must be able to read old data. The registry rejects incompatible schema changes before they reach production.
    - For the 3 years of historical events: do not rewrite them. Use upcasting to transform them during replay, or build a new projection that reads old events and applies transformation logic. Rewriting historical events violates the immutability contract and risks data corruption.
    - For breaking changes that cannot be made additive (e.g., splitting one event into two, or fundamentally changing the event's meaning): publish a new event type (`OrderPlacedV2`) alongside the old one. Run consumers of both types during a migration period. Once all consumers have migrated, stop publishing the old type. Do not delete old events -- they are historical records.

    **Red flag answer:** "We just update the schema and redeploy all services at the same time" -- this requires coordinated deployments, which defeats the purpose of microservices. Or "we delete the old events and repopulate" -- this destroys the audit trail and is often impossible.

    **Follow-ups:**
    1. You are using Avro with a Confluent Schema Registry. A developer submits a schema change that removes a required field. The registry rejects it as backward-incompatible. The developer argues the field is no longer needed. How do you proceed?
    2. Your upcaster logic has a bug -- it incorrectly transforms version 1 events during replay, causing 10,000 orders to have wrong totals in the read model. How do you detect and recover from this?
  </Accordion>

  <Accordion title="You are designing a notification system that sends emails, push notifications, and SMS based on business events. The system must handle 50,000 events per second at peak. Walk me through the architecture.">
    **Strong Answer:**
    - The architecture is a classic fan-out pattern with event-driven decoupling. Business services publish events (`OrderPlaced`, `PaymentFailed`, `ShipmentDelivered`) to a high-throughput message broker. A Notification Router service consumes these events, determines which notifications to send based on user preferences and business rules, and publishes delivery commands to channel-specific topics (`email.send`, `push.send`, `sms.send`). Channel-specific workers consume from their topic and call the actual delivery APIs (SendGrid, Firebase Cloud Messaging, Twilio).
    - For 50,000 events/second: Kafka is the natural choice for the message broker. A single Kafka cluster handles millions of events per second. Partition the event topics by user ID or event type for parallelism. The Notification Router can run as a Kafka Streams application or a consumer group with 20-50 instances, each processing a subset of partitions.
    - The Notification Router is the brain: it holds notification rules ("send email on OrderPlaced only if user has email_notifications enabled"), templates ("use the OrderConfirmation template for OrderPlaced"), and rate limiting ("never send more than 5 push notifications per hour to the same user"). This logic is separate from the business services -- they just publish facts about the world; the router decides what to communicate.
    - The channel workers need different scaling profiles. Email (SendGrid API) handles high throughput but has rate limits -- use a token bucket rate limiter. Push notifications (FCM) are fast but have payload limits -- batch multiple notifications to the same device. SMS (Twilio) is slow and expensive -- use aggressive deduplication and only send SMS for critical alerts.
    - Failure handling per channel: if SendGrid returns a 429 (rate limit), back off and retry. If it returns a 400 (invalid email), do not retry -- mark the email as invalid and update the user profile. If the push token is expired, remove it. Each channel worker publishes delivery results back to a `notification.results` topic for analytics and retry coordination.
    - At 50K events/second, not every event generates a notification. The Notification Router filters aggressively. Maybe 10% of events (5,000/s) produce notifications, and those fan out to ~8,000/s across channels (some events trigger both email and push). The channel workers need to handle this sustained throughput with headroom for spikes.

    **Red flag answer:** "We call SendGrid directly from each business service when an event happens" -- this tightly couples business services to notification delivery and makes it impossible to manage user preferences centrally. Or "we use a single queue for all notification types" -- this means a slow SMS API blocks email delivery.

    **Follow-ups:**
    1. A spike event causes 500,000 users to receive a push notification simultaneously. Your push notification provider (FCM) starts rate-limiting you. How do you handle the backpressure without dropping notifications?
    2. A user receives the same "Your order has shipped" email three times because the `ShipmentDelivered` event was published three times (producer retry). How do you prevent duplicate notifications at this scale?
  </Accordion>
