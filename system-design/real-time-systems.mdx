## Real-Time Communication Overview

Real-time communication is the difference between a walkie-talkie and a mailbox. HTTP polling is like checking your mailbox every 5 minutes -- you will eventually get the letter, but you waste trips and there is always a delay. Long polling is like standing at the mailbox waiting for the postal worker -- better, but you still restart the wait after each letter. SSE is like a loudspeaker that broadcasts announcements one way. WebSockets are a phone call -- both sides talk whenever they want over a persistent connection. The choice depends on whether you need one-way or two-way communication, how much latency you can tolerate, and how much operational complexity you are willing to take on.

```
┌─────────────────────────────────────────────────────────────────┐
│                REAL-TIME COMMUNICATION METHODS                  │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  1. HTTP Polling                                                │
│     Client → Server (every N seconds)                          │
│     Simple but wasteful                                         │
│                                                                 │
│  2. Long Polling                                                │
│     Client → Server (waits for response)                       │
│     Better, still overhead per request                         │
│                                                                 │
│  3. Server-Sent Events (SSE)                                   │
│     Server → Client (one-way stream)                           │
│     Simple, HTTP-based, auto-reconnect                         │
│                                                                 │
│  4. WebSockets                                                  │
│     Client ↔ Server (bidirectional)                            │
│     Full duplex, low latency, complex                          │
│                                                                 │
│  5. WebRTC                                                      │
│     Client ↔ Client (peer-to-peer)                             │
│     Audio/video, lowest latency                                │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

## Comparison Matrix

| Method | Direction | Latency | Complexity | Use Case |
|--------|-----------|---------|------------|----------|
| Polling | Client→Server | High | Low | Legacy, simple updates |
| Long Polling | Client→Server | Medium | Medium | Chat (fallback) |
| SSE | Server→Client | Low | Low | Notifications, feeds |
| WebSocket | Bidirectional | Very Low | High | Chat, gaming, trading |
| WebRTC | P2P | Lowest | Very High | Video calls, streaming |

```python
from fastapi import FastAPI, WebSocket, WebSocketDisconnect
from fastapi.middleware.cors import CORSMiddleware
from typing import Dict, Set, Optional
from dataclasses import dataclass, field
from datetime import datetime
import asyncio
import json
import redis.asyncio as redis
import logging

logger = logging.getLogger(__name__)

app = FastAPI()

@dataclass
class Connection:
    websocket: WebSocket
    user_id: str
    connected_at: datetime = field(default_factory=datetime.utcnow)
    rooms: Set[str] = field(default_factory=set)

class ConnectionManager:
    """
    Manages WebSocket connections with Redis Pub/Sub for scaling.
    """
    
    def __init__(self):
        self.connections: Dict[str, Connection] = {}  # user_id -> Connection
        self.rooms: Dict[str, Set[str]] = {}  # room_id -> {user_ids}
        self.redis: Optional[redis.Redis] = None
        self.pubsub = None
    
    async def init_redis(self, redis_url: str):
        """Initialize Redis for cross-server messaging"""
        self.redis = await redis.from_url(redis_url)
        self.pubsub = self.redis.pubsub()
        asyncio.create_task(self._listen_to_redis())
    
    async def _listen_to_redis(self):
        """Listen for messages from other servers"""
        await self.pubsub.psubscribe("ws:*")
        
        async for message in self.pubsub.listen():
            if message["type"] != "pmessage":
                continue
            
            channel = message["channel"].decode()
            data = json.loads(message["data"])
            
            if channel.startswith("ws:room:"):
                room_id = channel.split(":")[2]
                await self._broadcast_to_local_room(room_id, data)
            elif channel.startswith("ws:user:"):
                user_id = channel.split(":")[2]
                await self._send_to_local_user(user_id, data)
    
    async def connect(self, websocket: WebSocket, user_id: str) -> Connection:
        """Accept WebSocket connection"""
        await websocket.accept()
        
        conn = Connection(websocket=websocket, user_id=user_id)
        self.connections[user_id] = conn
        
        # Track connection in Redis for presence
        if self.redis:
            await self.redis.sadd("online_users", user_id)
            await self.redis.hset(
                f"user:{user_id}:presence",
                mapping={
                    "server_id": "server-1",  # Your server ID
                    "connected_at": conn.connected_at.isoformat()
                }
            )
        
        logger.info(f"User {user_id} connected. Total: {len(self.connections)}")
        return conn
    
    async def disconnect(self, user_id: str):
        """Handle WebSocket disconnect"""
        if user_id not in self.connections:
            return
        
        conn = self.connections.pop(user_id)
        
        # Leave all rooms
        for room_id in conn.rooms.copy():
            await self.leave_room(user_id, room_id)
        
        # Update Redis presence
        if self.redis:
            await self.redis.srem("online_users", user_id)
            await self.redis.delete(f"user:{user_id}:presence")
        
        logger.info(f"User {user_id} disconnected. Total: {len(self.connections)}")
    
    async def join_room(self, user_id: str, room_id: str):
        """Add user to a room"""
        if user_id not in self.connections:
            return
        
        if room_id not in self.rooms:
            self.rooms[room_id] = set()
        
        self.rooms[room_id].add(user_id)
        self.connections[user_id].rooms.add(room_id)
        
        # Notify room
        await self.broadcast_to_room(room_id, {
            "type": "user_joined",
            "user_id": user_id,
            "room_id": room_id
        })
    
    async def leave_room(self, user_id: str, room_id: str):
        """Remove user from a room"""
        if room_id in self.rooms:
            self.rooms[room_id].discard(user_id)
            if not self.rooms[room_id]:
                del self.rooms[room_id]
        
        if user_id in self.connections:
            self.connections[user_id].rooms.discard(room_id)
        
        await self.broadcast_to_room(room_id, {
            "type": "user_left",
            "user_id": user_id,
            "room_id": room_id
        })
    
    async def send_to_user(self, user_id: str, message: dict):
        """Send message to a specific user (cross-server)"""
        # Try local first
        if user_id in self.connections:
            await self._send_to_local_user(user_id, message)
        
        # Also publish to Redis for other servers
        if self.redis:
            await self.redis.publish(
                f"ws:user:{user_id}",
                json.dumps(message)
            )
    
    async def _send_to_local_user(self, user_id: str, message: dict):
        """Send to user on this server only"""
        if user_id in self.connections:
            try:
                await self.connections[user_id].websocket.send_json(message)
            except Exception as e:
                logger.error(f"Failed to send to {user_id}: {e}")
                await self.disconnect(user_id)
    
    async def broadcast_to_room(self, room_id: str, message: dict):
        """Broadcast to all users in a room (cross-server)"""
        # Local broadcast
        await self._broadcast_to_local_room(room_id, message)
        
        # Publish to Redis for other servers
        if self.redis:
            await self.redis.publish(
                f"ws:room:{room_id}",
                json.dumps(message)
            )
    
    async def _broadcast_to_local_room(self, room_id: str, message: dict):
        """Broadcast to room users on this server only"""
        if room_id not in self.rooms:
            return
        
        tasks = []
        for user_id in self.rooms[room_id]:
            if user_id in self.connections:
                tasks.append(
                    self.connections[user_id].websocket.send_json(message)
                )
        
        await asyncio.gather(*tasks, return_exceptions=True)
    
    async def get_online_users(self) -> Set[str]:
        """Get all online users across all servers"""
        if self.redis:
            return await self.redis.smembers("online_users")
        return set(self.connections.keys())

    except Exception as e:

    message_type = data.get("type")
    
    if message_type == "join_room":
        await manager.join_room(user_id, data["room_id"])
    
    elif message_type == "leave_room":
        await manager.leave_room(user_id, data["room_id"])
    
    elif message_type == "room_message":
        await manager.broadcast_to_room(data["room_id"], {
            "type": "message",
            "from": user_id,
            "content": data["content"],
            "timestamp": datetime.utcnow().isoformat()

    elif message_type == "direct_message":

            "timestamp": datetime.utcnow().isoformat()
```

```javascript
const WebSocket = require('ws');
const Redis = require('ioredis');
const http = require('http');

class WebSocketServer {
  constructor(options = {}) {
    this.port = options.port || 8080;
    this.redisUrl = options.redisUrl || 'redis://localhost:6379';
    
    // Connection tracking
    this.connections = new Map();  // userId -> ws
    this.rooms = new Map();        // roomId -> Set<userId>
    
    // Redis for scaling
    this.redisPub = new Redis(this.redisUrl);
    this.redisSub = new Redis(this.redisUrl);
    this.redisClient = new Redis(this.redisUrl);
  }

  async start() {
    // HTTP server for WebSocket upgrade
    this.server = http.createServer();
    this.wss = new WebSocket.Server({ server: this.server });

    // Handle WebSocket connections
    this.wss.on('connection', (ws, req) => {
      this.handleConnection(ws, req);
    });

    // Subscribe to Redis channels
    await this.setupRedisSubscription();

    // Start server
    this.server.listen(this.port, () => {
      console.log(`WebSocket server running on port ${this.port}`);
    });
  }

  async handleConnection(ws, req) {
    // Extract user ID from query params
    const url = new URL(req.url, `http://localhost`);
    const userId = url.searchParams.get('userId');

    if (!userId) {
      ws.close(4001, 'User ID required');
      return;
    }

    // Store connection
    this.connections.set(userId, ws);
    await this.updatePresence(userId, 'online');

    console.log(`User ${userId} connected. Total: ${this.connections.size}`);

    // Handle messages
    ws.on('message', async (data) => {
      try {
        const message = JSON.parse(data);
        await this.handleMessage(userId, message);
      } catch (error) {
        console.error('Invalid message:', error);
      }
    });

    // Handle disconnect
    ws.on('close', async () => {
      this.connections.delete(userId);
      await this.updatePresence(userId, 'offline');
      await this.leaveAllRooms(userId);
      console.log(`User ${userId} disconnected. Total: ${this.connections.size}`);
    });

    // Heartbeat
    ws.isAlive = true;
    ws.on('pong', () => { ws.isAlive = true; });
  }

  async handleMessage(userId, message) {
    const { type, ...payload } = message;

    switch (type) {
      case 'join_room':
        await this.joinRoom(userId, payload.roomId);
        break;

      case 'leave_room':
        await this.leaveRoom(userId, payload.roomId);
        break;

      case 'room_message':
        await this.broadcastToRoom(payload.roomId, {
          type: 'message',
          from: userId,
          content: payload.content,
          timestamp: new Date().toISOString()
        });
        break;

      case 'direct_message':
        await this.sendToUser(payload.toUser, {
          type: 'direct_message',
          from: userId,
          content: payload.content,
          timestamp: new Date().toISOString()
        });
        break;

      case 'ping':
        await this.sendToUser(userId, { type: 'pong' });
        break;
    }
  }

  async joinRoom(userId, roomId) {
    if (!this.rooms.has(roomId)) {
      this.rooms.set(roomId, new Set());
    }
    this.rooms.get(roomId).add(userId);

    // Track in Redis
    await this.redisClient.sadd(`room:${roomId}:users`, userId);

    await this.broadcastToRoom(roomId, {
      type: 'user_joined',
      userId,
      roomId
    });
  }

  async leaveRoom(userId, roomId) {
    if (this.rooms.has(roomId)) {
      this.rooms.get(roomId).delete(userId);
      if (this.rooms.get(roomId).size === 0) {
        this.rooms.delete(roomId);
      }
    }

    await this.redisClient.srem(`room:${roomId}:users`, userId);

    await this.broadcastToRoom(roomId, {
      type: 'user_left',
      userId,
      roomId
    });
  }

  async leaveAllRooms(userId) {
    for (const [roomId, users] of this.rooms) {
      if (users.has(userId)) {
        await this.leaveRoom(userId, roomId);
      }
    }
  }

  async sendToUser(userId, message) {
    // Try local first
    const ws = this.connections.get(userId);
    if (ws && ws.readyState === WebSocket.OPEN) {
      ws.send(JSON.stringify(message));
      return;
    }

    // Publish to Redis for other servers
    await this.redisPub.publish(
      `ws:user:${userId}`,
      JSON.stringify(message)
    );
  }

  async broadcastToRoom(roomId, message) {
    // Local broadcast
    const localUsers = this.rooms.get(roomId) || new Set();
    for (const userId of localUsers) {
      const ws = this.connections.get(userId);
      if (ws && ws.readyState === WebSocket.OPEN) {
        ws.send(JSON.stringify(message));
      }
    }

    // Publish to Redis for other servers
    await this.redisPub.publish(
      `ws:room:${roomId}`,
      JSON.stringify(message)
    );
  }

  async setupRedisSubscription() {
    await this.redisSub.psubscribe('ws:*');

    this.redisSub.on('pmessage', async (pattern, channel, data) => {
      const message = JSON.parse(data);

      if (channel.startsWith('ws:room:')) {
        const roomId = channel.split(':')[2];
        // Only send to local users (to avoid loops)
        this.localBroadcastToRoom(roomId, message);
      } else if (channel.startsWith('ws:user:')) {
        const userId = channel.split(':')[2];
        this.localSendToUser(userId, message);
      }
    });
  }

  localSendToUser(userId, message) {
    const ws = this.connections.get(userId);
    if (ws && ws.readyState === WebSocket.OPEN) {
      ws.send(JSON.stringify(message));
    }
  }

  localBroadcastToRoom(roomId, message) {
    const users = this.rooms.get(roomId) || new Set();
    for (const userId of users) {
      this.localSendToUser(userId, message);
    }
  }

  async updatePresence(userId, status) {
    if (status === 'online') {
      await this.redisClient.sadd('online_users', userId);
      await this.redisClient.hset(`user:${userId}:presence`, {
        status: 'online',
        server: process.env.SERVER_ID || 'server-1',
        timestamp: Date.now()
      });
    } else {
      await this.redisClient.srem('online_users', userId);
      await this.redisClient.del(`user:${userId}:presence`);
    }
  }

  // Heartbeat to detect dead connections
  startHeartbeat() {
    setInterval(() => {
      this.wss.clients.forEach((ws) => {
        if (ws.isAlive === false) {
          return ws.terminate();
        }
        ws.isAlive = false;
        ws.ping();
      });
    }, 30000);
  }
}

// Usage
const server = new WebSocketServer({ port: 8080 });
server.start();
server.startHeartbeat();
```

## Server-Sent Events (SSE)

Simpler than WebSockets for server-to-client streaming.

```python
from fastapi import FastAPI
from fastapi.responses import StreamingResponse
from typing import AsyncGenerator
import asyncio
import json

app = FastAPI()

class EventBroker:
    """Simple in-memory event broker for SSE"""
    
    def __init__(self):
        self.subscribers: Dict[str, asyncio.Queue] = {}
    
    async def subscribe(self, user_id: str) -> asyncio.Queue:
        queue = asyncio.Queue()
        self.subscribers[user_id] = queue
        return queue
    
    def unsubscribe(self, user_id: str):
        self.subscribers.pop(user_id, None)
    
    async def publish(self, user_id: str, event: dict):
        if user_id in self.subscribers:
            await self.subscribers[user_id].put(event)
    
    async def broadcast(self, event: dict):
        for queue in self.subscribers.values():
            await queue.put(event)

broker = EventBroker()

async def event_stream(user_id: str) -> AsyncGenerator[str, None]:
    """Generate SSE stream for a user"""
    queue = await broker.subscribe(user_id)
    
    try:
        # Send initial connection event
        yield f"event: connected\ndata: {json.dumps({'user_id': user_id})}\n\n"
        
        while True:
            try:
                # Wait for event with timeout (for keepalive)
                event = await asyncio.wait_for(queue.get(), timeout=30)
                
                event_type = event.get("type", "message")
                data = json.dumps(event)
                
                yield f"event: {event_type}\ndata: {data}\n\n"
                
            except asyncio.TimeoutError:
                # Send keepalive comment
                yield ": keepalive\n\n"
    
    finally:
        broker.unsubscribe(user_id)

@app.get("/events/{user_id}")
async def sse_endpoint(user_id: str):
    """SSE endpoint for real-time notifications"""
    return StreamingResponse(
        event_stream(user_id),
        media_type="text/event-stream",
        headers={
            "Cache-Control": "no-cache",
            "Connection": "keep-alive",
            "X-Accel-Buffering": "no"  # Disable nginx buffering
        }
    )

@app.post("/notify/{user_id}")
async def send_notification(user_id: str, message: dict):
    """Send notification to a user"""
    await broker.publish(user_id, {
        "type": "notification",
        **message
    })
    return {"status": "sent"}
```

## Long Polling

Fallback for when WebSockets/SSE aren't available.

```python
from fastapi import FastAPI
from typing import Optional
import asyncio

app = FastAPI()

class LongPollManager:
    def __init__(self):
        self.pending: Dict[str, asyncio.Future] = {}
        self.messages: Dict[str, list] = {}
    
    async def wait_for_messages(
        self, 
        user_id: str, 
        timeout: float = 30
    ) -> list:
        """Wait for new messages with timeout"""
        
        # Check for pending messages first
        if user_id in self.messages and self.messages[user_id]:
            messages = self.messages.pop(user_id)
            return messages
        
        # Create a future to wait on
        future = asyncio.get_event_loop().create_future()
        self.pending[user_id] = future
        
        try:
            messages = await asyncio.wait_for(future, timeout=timeout)
            return messages
        except asyncio.TimeoutError:
            return []
        finally:
            self.pending.pop(user_id, None)
    
    async def send_message(self, user_id: str, message: dict):
        """Send message to user, either immediately or queue"""
        if user_id in self.pending:
            # User is waiting, resolve immediately
            future = self.pending.pop(user_id)
            if not future.done():
                future.set_result([message])
        else:
            # Queue for next poll
            if user_id not in self.messages:
                self.messages[user_id] = []
            self.messages[user_id].append(message)

poll_manager = LongPollManager()

@app.get("/poll/{user_id}")
async def long_poll(user_id: str, timeout: float = 30):
    """Long polling endpoint"""
    messages = await poll_manager.wait_for_messages(user_id, timeout)
    return {"messages": messages}

@app.post("/send/{user_id}")
async def send_message(user_id: str, message: dict):
    """Send message to user"""
    await poll_manager.send_message(user_id, message)
    return {"status": "sent"}
```

```python
class PresenceSystem:
    """
    Track online/offline status across servers.
    Uses Redis with TTL for automatic cleanup.
    """
    
    HEARTBEAT_INTERVAL = 30  # seconds
    PRESENCE_TTL = 60  # seconds (2x heartbeat)
    
    def __init__(self, redis_client: redis.Redis, server_id: str):
        self.redis = redis_client
        self.server_id = server_id
    
    async def set_online(self, user_id: str):
        """Mark user as online"""
        key = f"presence:{user_id}"
        await self.redis.hset(key, mapping={
            "server_id": self.server_id,
            "status": "online",
            "last_seen": datetime.utcnow().isoformat()
        })
        await self.redis.expire(key, self.PRESENCE_TTL)
        
        # Add to online set
        await self.redis.sadd("online_users", user_id)
    
    async def heartbeat(self, user_id: str):
        """Refresh presence TTL"""
        key = f"presence:{user_id}"
        
        # Update last_seen and extend TTL
        await self.redis.hset(key, "last_seen", datetime.utcnow().isoformat())
        await self.redis.expire(key, self.PRESENCE_TTL)
    
    async def set_offline(self, user_id: str):
        """Mark user as offline"""
        await self.redis.delete(f"presence:{user_id}")
        await self.redis.srem("online_users", user_id)
    
    async def is_online(self, user_id: str) -> bool:
        """Check if user is online"""
        return await self.redis.exists(f"presence:{user_id}")
    
    async def get_status(self, user_id: str) -> Dict:
        """Get user's presence status"""
        data = await self.redis.hgetall(f"presence:{user_id}")
        
        if not data:
            return {"status": "offline"}
        
        return {
            "status": data.get("status", "offline"),
            "server_id": data.get("server_id"),
            "last_seen": data.get("last_seen")
        }
    
    async def get_online_users(self) -> Set[str]:
        """Get all online users"""
        return await self.redis.smembers("online_users")
    
    async def get_online_count(self) -> int:
        """Get count of online users"""
        return await self.redis.scard("online_users")
    
    async def get_friends_online(self, user_id: str, friend_ids: List[str]) -> List[str]:
        """Get which friends are online"""
        online_friends = []
        
        # Use pipeline for efficiency
        async with self.redis.pipeline() as pipe:
            for friend_id in friend_ids:
                pipe.exists(f"presence:{friend_id}")
            results = await pipe.execute()
        
        for friend_id, is_online in zip(friend_ids, results):
            if is_online:
                online_friends.append(friend_id)
        
        return online_friends
    
    async def cleanup_stale(self):
        """Remove stale presence entries (run periodically)"""
        online_users = await self.redis.smembers("online_users")
        
        for user_id in online_users:
            if not await self.redis.exists(f"presence:{user_id}"):
                await self.redis.srem("online_users", user_id)
```

## Typing Indicators

Show when someone is typing.

```python
class TypingIndicator:
    """
    Track and broadcast typing indicators.
    Uses short TTLs to auto-expire.
    """
    
    TYPING_TTL = 3  # seconds
    
    def __init__(self, redis_client: redis.Redis, broadcast_fn):
        self.redis = redis_client
        self.broadcast = broadcast_fn
    
    async def set_typing(self, user_id: str, room_id: str):
        """Mark user as typing in a room"""
        key = f"typing:{room_id}:{user_id}"
        
        # Only broadcast if not already typing
        is_new = not await self.redis.exists(key)
        
        await self.redis.setex(key, self.TYPING_TTL, "1")
        
        if is_new:
            await self.broadcast(room_id, {
                "type": "typing_start",
                "user_id": user_id,
                "room_id": room_id
            })
    
    async def stop_typing(self, user_id: str, room_id: str):
        """Mark user as stopped typing"""
        key = f"typing:{room_id}:{user_id}"
        
        if await self.redis.delete(key):
            await self.broadcast(room_id, {
                "type": "typing_stop",
                "user_id": user_id,
                "room_id": room_id
            })
    
    async def get_typing_users(self, room_id: str) -> List[str]:
        """Get all users currently typing in a room"""
        pattern = f"typing:{room_id}:*"
        keys = await self.redis.keys(pattern)
        
        return [key.split(":")[-1] for key in keys]
```

## Trade-off Analysis: Choosing the Right Protocol

| Criterion | HTTP Polling | Long Polling | SSE | WebSocket | WebRTC |
|-----------|-------------|--------------|-----|-----------|--------|
| Implementation cost | Very low | Low | Low | Medium | High |
| Server resource usage | High (wasted requests) | Medium | Low | Low (per-connection memory) | Low (P2P) |
| Latency | Polling interval | Near-real-time | Real-time | Real-time | Real-time |
| Proxy/firewall friendly | Yes | Yes | Yes | Sometimes blocked | Often blocked |
| Auto-reconnect | N/A | Manual | Built-in | Manual | Manual |
| Binary data | No | No | No | Yes | Yes |

<Tip>
**Practical tip -- a senior engineer would say:** "Start with SSE unless you have a hard requirement for bidirectional communication. SSE works over standard HTTP, auto-reconnects, passes through every proxy and CDN, and is dramatically simpler to operate. Most 'real-time' features (notifications, live dashboards, activity feeds) are server-to-client only. Reserve WebSockets for chat, collaborative editing, or gaming where the client genuinely needs to send data on the persistent connection."
</Tip>

<Warning>
**Common production pitfall**: WebSocket connections consume server memory (~10-50KB each). At 100K concurrent connections, that is 1-5GB just for connection state. If you do not plan for this, your servers will OOM during traffic spikes. Always set connection limits per server, use sticky sessions at the load balancer, and have a strategy for graceful connection migration during deploys.
</Warning>

### Common Questions

**Q: How do you scale WebSocket servers?**
> Use sticky sessions at load balancer level (same client → same server). Use Redis Pub/Sub to broadcast messages across servers.

**Q: How do you handle disconnects?**
> Implement heartbeat (ping/pong every 30s). Client-side reconnection with exponential backoff. Server detects dead connections via ping timeout.

**Q: When would you use SSE over WebSockets?**
> SSE for server-to-client only (notifications, live updates, streaming). Simpler, HTTP-based, automatic reconnection. WebSockets when you need bidirectional or lower latency.

  <Accordion title="Q1: You are designing a real-time collaborative document editor (like Google Docs) where multiple users edit the same document simultaneously. What real-time communication pattern would you use, and how do you handle the case where two users edit the same paragraph at the same time?" icon="file-pen">
    **What the interviewer is really testing:** Whether you understand the difference between real-time transport (WebSockets) and real-time conflict resolution (OT or CRDT), and can reason about both layers.

    **Strong Answer:**
    - This problem has two distinct layers: the transport layer (how edits get from client to server and back) and the consistency layer (how concurrent edits are merged without losing anyone's changes).
    - For transport, WebSockets are the clear choice. Edits need to flow bidirectionally with low latency. SSE is server-to-client only, so the client would need a separate HTTP POST channel for sending edits, which adds complexity and latency. Long polling adds unnecessary round-trip overhead for something that requires sub-100ms propagation. WebSockets give you a persistent, full-duplex connection where edits flow as small messages in both directions.
    - For conflict resolution, there are two mainstream approaches: Operational Transformation (OT) and Conflict-free Replicated Data Types (CRDTs).
    - OT (what Google Docs uses): each edit is represented as an operation (insert character 'A' at position 5, delete character at position 10). When two operations are created concurrently, the server transforms one against the other so both can be applied in either order with the same result. If User A inserts "Hello" at position 0 and User B inserts "World" at position 0 simultaneously, the server transforms B's operation to insert at position 5 (after A's text). OT requires a central server to sequence operations, which makes it simpler to reason about but creates a single point of coordination.
    - CRDTs (what Figma and some newer editors use): the document is represented as a CRDT data structure where operations are commutative and associative -- they can be applied in any order and still converge to the same state. CRDTs work in peer-to-peer scenarios without a central server. The trade-off: CRDTs are more complex to implement, use more memory (they track causal metadata), and some document operations (like "move paragraph") are hard to express as CRDTs.
    - For a Google Docs-like system, I would use OT with a centralized server for simplicity. Each document has a single "owner server" that sequences all operations. Clients send operations over WebSockets, the server transforms and applies them, then broadcasts the transformed operations to all other connected clients. This gives real-time collaboration with a consistency model that is well-understood and battle-tested.
    - Presence and cursor tracking: in addition to document edits, show each user's cursor position and selection. This is simpler than OT -- just broadcast cursor positions via the same WebSocket connection. No conflict resolution needed because each user owns their own cursor state.
    - **Example:** Google Docs uses OT with a central coordination server per document. When they hit scaling limits (too many concurrent editors), they shard documents into sections and coordinate each section independently. Figma chose CRDTs because their vector graphics operations are harder to express in OT's sequential model, and they wanted to support offline editing where a central server is not available.

    **Follow-up: A user is on a flaky mobile connection and goes offline for 30 seconds while other users continue editing. When they reconnect, they have 30 seconds of local edits that the server has never seen, and the server has 30 seconds of other users' edits. How do you reconcile this?**

    With OT: the client tracks its local operations and the last server-acknowledged version number. On reconnect, the client sends all unacknowledged operations to the server. The server transforms these operations against all operations that occurred during the disconnect (using the version gap to identify which ones). The transformed operations are applied and broadcast. The key: the client must also apply the server's missed operations (transformed against the client's local ops) to update its local state. This is called the "Jupiter protocol" and is what Google Docs implements. With CRDTs: reconciliation is simpler because operations are commutative by design -- the client just sends all its local operations and the CRDT merge function handles the rest, regardless of ordering. This is why CRDTs are attractive for mobile and offline-first applications.
  </Accordion>

  <Accordion title="Q2: Your chat application has 500K concurrent WebSocket connections. The load balancer distributes connections across 25 servers. User A is connected to Server 3 and sends a message to User B on Server 17. Walk me through exactly how the message is delivered." icon="comments">
    **What the interviewer is really testing:** Whether you understand the cross-server message routing problem in horizontally scaled WebSocket architectures and can design a solution using pub/sub.

    **Strong Answer:**
    - The core challenge: WebSocket connections are stateful. User A's connection lives on Server 3, User B's connection lives on Server 17. Server 3 cannot directly push a message to a connection it does not own. You need a message bus that all servers subscribe to.
    - The delivery flow: (1) User A sends a message over their WebSocket connection to Server 3. (2) Server 3 receives the message, persists it to the message store (for durability and offline delivery), and publishes it to a Redis Pub/Sub channel. The channel can be per-user (e.g., `ws:user:userB`) or per-conversation (e.g., `ws:room:conv123`). (3) All 25 servers are subscribed to the relevant channels. Server 17, which holds User B's connection, receives the published message. (4) Server 17 looks up User B's local WebSocket connection and pushes the message through it.
    - Why Redis Pub/Sub specifically: it is fire-and-forget with no persistence, which is fine here because the message is already persisted in the database. Redis Pub/Sub adds minimal latency (sub-millisecond within a datacenter) and handles the fan-out efficiently. The message reaches Server 17 in under 1ms after publishing.
    - Alternative: Kafka instead of Redis Pub/Sub. Kafka provides durability and replay, which is useful if you need guaranteed delivery to the WebSocket server (e.g., if Server 17 briefly disconnects from Redis, it loses messages). With Kafka, each server is a consumer in a consumer group, and missed messages can be replayed. The trade-off: Kafka adds 5-20ms of latency compared to Redis's sub-millisecond, which matters for chat but might be acceptable.
    - Connection tracking: each server maintains a local in-memory map of user_id to WebSocket connection. When User B connects to Server 17, Server 17 registers User B in a global presence store (Redis hash: `user:userB:server = server-17`). When Server 3 needs to send a message to User B, it can either (a) publish to the per-user channel and let the right server pick it up (simple, slight waste for servers that do not have User B), or (b) look up User B's server in the presence store and publish to a server-specific channel (more efficient, more complexity).
    - At 500K connections across 25 servers, that is 20K connections per server. Each server needs to handle its own connection management plus the pub/sub message flow. The bottleneck is usually not the WebSocket handling but the pub/sub fan-out if you have high-volume group chats (a group with 1000 members means each message is published once and delivered to potentially many servers).
    - **Example:** Discord handles millions of concurrent WebSocket connections using this exact pattern. They use a combination of Erlang (for connection handling) and a custom pub/sub system backed by their "Guild" model -- each Discord server is a pub/sub topic, and only the WebSocket servers with connected members of that guild subscribe to its messages. This avoids the problem of every server receiving every message.

    **Follow-up: User B is offline (no active WebSocket connection). How do you ensure they receive the message when they come back online?**

    When Server 3 publishes the message, it also writes it to a persistent message store (Cassandra, PostgreSQL, or DynamoDB) with a "delivered" flag set to false for User B. When User B reconnects to any server, the reconnection handler queries the message store for all undelivered messages since the user's last active timestamp. These messages are sent over the new WebSocket connection and marked as delivered. For push notifications (mobile), a separate consumer listens for messages to offline users and triggers a push notification via APNs/FCM. The message store, not the pub/sub layer, is the source of truth for delivery status.
  </Accordion>

  <Accordion title="Q3: Your stock trading platform needs to push price updates to 100K connected clients with a maximum end-to-end latency of 50ms from when the exchange publishes the price to when the client receives it. How do you architect this?" icon="chart-line">

Alternatively, use a shared-memory IPC mechanism (like Aeron or Unix domain sockets) if the fan-out servers are on the same machine or rack. Kafka is too slow for this use case -- its batching and replication add 5-20ms.
    - Server-side delivery (5ms budget): Each WebSocket server receives the price update from the pub/sub layer. It looks up which local connections are subscribed to that particular stock symbol and writes the update to each WebSocket. With 100K connections across 10 servers (10K per server), and assuming 30% are subscribed to any given active stock, that is 3K writes per price update per server. Async I/O (epoll on Linux) handles this efficiently.

Use binary protocols (protobuf or MessagePack) instead of JSON to minimize parse time. Pre-allocate DOM elements for price tickers so updates are in-place mutations, not re-renders.

    - **Example:** IEX (the exchange) provides their market data feed with sub-50ms latency using a similar architecture. (3) Implement admission control: if connection count exceeds capacity, reject new connections with a "retry in 30 seconds" response rather than accepting them and degrading latency for existing connections. (4) Use connection draining on the load balancer so that existing connections are not disrupted if a server needs to be replaced. The key principle: for predictable load patterns, pre-scale rather than auto-scale.

  <Accordion title="2. Explain the WebSocket handshake process. What happens at the HTTP level, and why does it start as an HTTP request?">
    **Strong Answer:**
    - A WebSocket connection begins as a standard HTTP/1.1 GET request with special upgrade headers. The client sends `Upgrade: websocket` and `Connection: Upgrade`, along with a `Sec-WebSocket-Key` header containing a base64-encoded random value. The server responds with HTTP 101 Switching Protocols, echoes back a `Sec-WebSocket-Accept` header (a SHA-1 hash of the client's key concatenated with a magic GUID), and the connection transitions from HTTP to the WebSocket protocol. From this point, both sides communicate using WebSocket frames, not HTTP.
    - It starts as HTTP for a practical reason: compatibility. WebSocket connections need to traverse the same network infrastructure as HTTP -- firewalls, proxies, load balancers. By starting as HTTP, the initial handshake passes through infrastructure that understands HTTP. The `Upgrade` mechanism is part of the HTTP/1.1 spec (RFC 2616), so compliant proxies know how to handle it (though some older proxies do not, which is a real-world pain point).
    - The `Sec-WebSocket-Key` and `Sec-WebSocket-Accept` exchange is not for security -- it prevents accidental connections. It ensures that the server actually understands WebSocket and is not just a regular HTTP server blindly accepting the upgrade request. The key is not encrypted or secret. Real security comes from using `wss://` (WebSocket over TLS), which should always be used in production.
    - After the handshake, the connection uses a binary framing protocol. Each message is wrapped in a frame with an opcode (text, binary, ping, pong, close), a length, and optionally a masking key (client-to-server frames must be masked to prevent cache poisoning attacks on intermediary proxies). This framing is lightweight -- a small text message has only 2-6 bytes of overhead compared to HTTP's headers-per-request overhead.
    - **Example:** A developer deployed a WebSocket server behind AWS ALB and found that connections were dropping after 60 seconds. The root cause: ALB has a default idle timeout of 60 seconds. If no data flows on the WebSocket for 60 seconds, the ALB closes the connection. The fix was implementing ping/pong heartbeats every 30 seconds to keep the connection alive, and increasing the ALB idle timeout to 300 seconds. This is a classic gotcha that only surfaces when you understand the handshake and how proxies interact with WebSocket connections.

    **Red flag answer:** "WebSocket is its own protocol, it has nothing to do with HTTP." This is wrong -- the connection literally begins as HTTP, and understanding this is essential for debugging proxy, load balancer, and firewall issues.

    **Follow-ups:**
    1. Your WebSocket connections work locally but fail in production behind a corporate proxy. What are the most likely causes, and how would you debug this?
    2. HTTP/2 has multiplexed streams over a single connection. Does this eliminate the need for WebSockets? Why or why not?
  </Accordion>

  <Accordion title="3. You have 10 WebSocket servers behind a load balancer. User A on Server 1 sends a message to a chat room where User B is on Server 5 and User C is on Server 8. How do you deliver the message to all room members?">
    **Strong Answer:**
    - This is the cross-server fan-out problem, and it is the core architectural challenge of scaling WebSockets horizontally. Each server only knows about its own local connections. Server 1 has no direct access to the WebSocket connections on Server 5 or Server 8. You need a shared message bus.
    - The standard solution is Redis Pub/Sub. Each WebSocket server subscribes to channels corresponding to the rooms its connected users belong to. When User A sends a message to room "engineering," Server 1 publishes the message to the Redis channel `ws:room:engineering`. Servers 5 and 8 (and any other server with users in that room) receive the message from Redis and push it to their local WebSocket connections.
    - The message flow: (1) User A sends a WebSocket frame to Server 1. (2) Server 1 parses the message and identifies it as a room message for "engineering." (3) Server 1 broadcasts to its own local room members (if any). (4) Server 1 publishes the message to Redis `ws:room:engineering`. (5) All servers subscribed to that channel receive the message. (6) Each server checks its local room membership map and delivers the message to connected local users. Servers with no members in "engineering" receive and ignore the message.
    - The load balancer must use sticky sessions (session affinity) -- typically based on a cookie or the source IP. Without sticky sessions, a client's WebSocket connection could be routed to a different server on reconnect, and the server that held the connection state would not know about the new connection. Sticky sessions ensure that a client always reconnects to the same server.
    - Trade-offs of Redis Pub/Sub: it is fire-and-forget with no persistence. If Server 5 is briefly disconnected from Redis, it misses any messages published during that window. For a chat application, this might be acceptable if messages are also persisted to a database and the client fetches missed messages on reconnect. For a system that requires guaranteed delivery, Kafka is a better choice -- each server is a consumer group member and can replay missed messages, but at the cost of higher latency (5-20ms vs. sub-1ms for Redis).
    - **Example:** Slack uses a similar pattern at massive scale. Each Slack workspace is effectively a pub/sub topic. Their "message server" fleet subscribes to the workspace channels for their connected users. When you send a message in a Slack channel, it propagates through the message bus to every connected client's server in single-digit milliseconds. They also persist every message to their database, so clients that were offline can fetch the history via API on reconnect.

    **Red flag answer:** "Just have each server connect to every other server directly" or "Store all connections in one central server." Direct mesh does not scale (N-squared connections) and central connection storage is a single point of failure.

    **Follow-ups:**
    1. Your chat room has 50,000 members, and each message triggers delivery to potentially all 10 servers. At high message volume (100 messages per second in the room), Redis Pub/Sub becomes a bottleneck. How do you optimize this?
    2. A server crashes and 5,000 users reconnect to other servers within 10 seconds. What happens to the room membership state, and how do you handle this gracefully?
  </Accordion>

  <Accordion title="4. What is the purpose of a heartbeat (ping/pong) mechanism in WebSockets, and what happens if you do not implement one?">
    **Strong Answer:**
    - A heartbeat serves two purposes: detecting dead connections and keeping the connection alive through network intermediaries. Without a heartbeat, you accumulate zombie connections -- connections that the server believes are alive but the client has silently disappeared (phone lost signal, laptop lid closed, network cable unplugged). These zombies consume server memory, pollute your room membership counts, and cause messages to be sent into a void.
    - Dead connection detection: TCP has its own keepalive mechanism, but it is designed for general-purpose networking and has long default timeouts (often 2 hours on Linux). For a real-time application, discovering that a client is gone after 2 hours is useless. Application-level ping/pong gives you control: send a ping every 30 seconds, if the pong does not arrive within 10 seconds, terminate the connection and trigger cleanup (remove from rooms, update presence, notify peers).
    - Keepalive through intermediaries: load balancers, proxies, and NAT gateways all have idle connection timeouts. AWS ALB defaults to 60 seconds, Nginx defaults to 60 seconds, many NAT gateways use 5 minutes. If no data flows on the connection during the idle timeout, the intermediary closes it (often silently -- neither the client nor the server receives a close notification). Regular heartbeat traffic resets these idle timers and prevents the intermediary from killing a connection that is idle but still wanted.
    - Implementation details: the WebSocket protocol defines opcode `0x9` (ping) and `0xA` (pong) at the protocol level. Most libraries handle pong responses automatically -- if you send a protocol-level ping, the client's WebSocket implementation responds with a pong without application code involvement. Some teams implement application-level heartbeats instead (sending a JSON `{"type": "ping"}` message), which gives more flexibility (you can include data like server timestamp) but requires explicit handling on both sides.
    - The interval matters: too frequent (every 1 second) wastes bandwidth and CPU, especially at scale (100K connections means 100K pings per second). Too infrequent (every 5 minutes) means you might not detect a dead connection for 5+ minutes, during which you are wasting resources and delivering messages into the void. 30 seconds is the common sweet spot -- frequent enough to detect failures quickly, infrequent enough to be negligible in overhead.
    - **Example:** A gaming company discovered that 15% of their "online" users at any given time were actually zombie connections -- phones that had lost signal or users who had force-quit the app. They were sending game state updates to these 15K dead connections, wasting bandwidth and distorting their "concurrent user" metrics. Adding a 30-second ping/pong with 10-second timeout and cleanup reduced their "online" count by 15% but made it accurate, and reduced their outbound bandwidth by 12%.

    **Red flag answer:** "TCP already handles keepalive, so you do not need application-level heartbeats." This shows a misunderstanding of TCP keepalive defaults and the role of network intermediaries.

    **Follow-ups:**
    1. Your server sends pings every 30 seconds, but clients on mobile networks report frequent disconnections because the pong does not arrive within your 10-second timeout. How do you handle this without making the timeout so long that zombie detection becomes useless?
    2. At 500K concurrent connections with 30-second ping intervals, you are generating roughly 16K pings per second. Is this a concern, and how would you optimize if it became one?

  <Accordion title="5. Explain how a presence system works (showing who is online/offline). Why is it harder than it sounds?">
    **Strong Answer:**
    - At the surface, presence seems trivial: when a user connects, mark them as online; when they disconnect, mark them as offline. The difficulty comes from unreliable disconnect detection, multi-device users, distributed servers, and the thundering herd problem on state changes.
    - The core mechanism uses TTL-based expiry with heartbeats. When a user connects, write a key to Redis with a TTL (e.g., `presence:user123` with a 60-second TTL). The client sends a heartbeat every 30 seconds, which refreshes the TTL. If the client disconnects cleanly, delete the key immediately. If the client vanishes without a clean disconnect (crash, network loss), the key automatically expires after 60 seconds. This means presence is eventually consistent with a worst-case staleness of 60 seconds.
    - Multi-device complexity: a user might be logged in on their phone and laptop simultaneously. If the laptop closes, the user should still show as "online" because the phone is connected. This means presence is not a simple boolean -- it is a count or set of active sessions. Store `presence:user123` as a Redis set of session IDs, each with its own TTL-managed key. The user is "online" if any session key exists. This adds complexity to the cleanup logic because you need to evaluate "any session alive?" rather than just checking one key.
    - Distributed server challenge: with 10 servers, each server manages its own local connections. Server 3 knows that User A connected, but Server 7 has no idea unless there is a shared store. Redis solves this -- all servers read and write presence to the same Redis instance. But publishing presence changes (user came online / went offline) to other connected clients requires fan-out. In a system with 100K users where each user has 500 friends, a single user going offline could trigger 500 notifications to other users' WebSocket connections across multiple servers. This is the thundering herd problem.
    - Optimization for fan-out: (1) Batch presence updates -- instead of pushing each online/offline event immediately, buffer changes for 5 seconds and send a single update with all changes. (2) Only send updates for users the client is currently viewing. If User B is not looking at their friends list, do not push presence updates to them. (3) Use "presence channels" -- clients subscribe to specific users' presence rather than receiving updates for all users.
    - The `KEYS` anti-pattern: never use Redis `KEYS` command to find all typing/presence keys in production. `KEYS` scans the entire keyspace and blocks Redis. Use `SCAN` with a cursor, or maintain a separate `SADD`-based set of online users alongside the TTL keys.
    - **Example:** Discord handles presence for 150 million+ monthly active users. They do not push real-time presence for all friends -- they lazy-load presence when you open a DM or channel, and only maintain real-time presence updates for the servers (guilds) you are currently viewing. This dramatically reduces the fan-out burden. They also compress presence data by batching updates: instead of pushing 200 individual "user went offline" events when a server goes down, they send a single "bulk presence update" event.

    **Red flag answer:** "Just set a flag in the database when the user logs in and clear it when they log out." This ignores ungraceful disconnects entirely (the flag is never cleared when the browser crashes), leading to permanently "online" ghost users.

    **Follow-ups:**
    1. Your presence system uses 60-second TTLs, but users complain that their friends show as "online" for up to a minute after they close the app. How do you reduce this staleness without making the heartbeat interval unreasonably short?
    2. Your Redis instance holding presence data goes down. How does this affect the application, and what is your recovery strategy?
  </Accordion>

  <Accordion title="6. Compare Redis Pub/Sub and Kafka for the message bus in a WebSocket architecture. When would you pick each?">
    **Strong Answer:**
    - They solve the same broadcast problem (getting a message from the publishing server to all subscribing servers) but with fundamentally different trade-offs around durability, latency, and operational complexity.
    - Redis Pub/Sub: fire-and-forget. When a message is published, it is delivered to all currently connected subscribers and then gone. If a subscriber is temporarily disconnected, it misses the message permanently. Latency is sub-millisecond within a datacenter. No disk I/O, no replication overhead. It is effectively a network broadcast. Operationally simple -- Redis is a single process (or a cluster for HA), no topics to manage, no consumer groups to configure.
    - Kafka: durable, persistent, replayable. Messages are written to disk in ordered partitions and retained for a configurable period (hours, days, or forever). Consumers track their offset and can replay messages from any point. If a WebSocket server crashes and restarts, it resumes from its last committed offset and processes all missed messages. Latency is 5-20ms due to batching and disk writes (tunable with `linger.ms` and `acks` configuration, but you trade durability for latency).
    - When to pick Redis Pub/Sub: (1) Chat and notification systems where messages are also persisted to a database. If a WebSocket server misses a pub/sub message, the client fetches missed messages from the database on reconnect. The pub/sub layer is just a real-time delivery optimization, not the source of truth. (2) Presence updates and typing indicators -- ephemeral data where missing an event is acceptable (the next heartbeat or typing event corrects the state). (3) Systems where sub-millisecond latency between servers is critical (live gaming, trading).
    - When to pick Kafka: (1) Event-driven architectures where the message bus is the source of truth and messages must not be lost. (2) Systems with consumers that process at different speeds or might be temporarily offline. (3) When you need fan-out to multiple consumer types (the WebSocket layer is one consumer, but analytics and audit logging are others consuming the same stream). (4) When you need ordering guarantees per partition (e.g., all messages for a given conversation are processed in order).
    - A hybrid approach is common at scale: use Kafka as the durable event backbone (all messages flow through Kafka for persistence and multi-consumer access) and use Redis Pub/Sub as a "last-mile" real-time delivery layer (WebSocket servers subscribe to Redis for instant delivery, but also consume from Kafka for missed messages). This gives you both sub-millisecond delivery and durability.
    - **Example:** LinkedIn uses this hybrid approach for their messaging system. Messages are written to Kafka (durable, ordered, replayable) and simultaneously published to a real-time delivery layer for instant WebSocket push. If the real-time path fails, the client fetches from the Kafka-backed API on its next sync. This separation of "real-time best-effort" and "durable source of truth" is a pattern that appears in nearly every large-scale messaging system.

    **Red flag answer:** "Kafka is better because it is more powerful" or "Redis is better because it is faster." Both miss the point -- it is not about which is "better" in the abstract, it is about which fits the durability, latency, and operational requirements of your specific use case.

    **Follow-ups:**
    1. Your team uses Redis Pub/Sub for a chat system. During a Redis failover (primary goes down, replica promotes), all subscribers lose their connections for 2-3 seconds and miss all messages published during that window. How do you handle this?
    2. You switch from Redis Pub/Sub to Kafka for the WebSocket message bus and notice that chat message delivery latency increased from `<`1ms to 15ms. Users complain that messages feel "sluggish." How do you tune Kafka to reduce this latency, and what are the trade-offs?
  </Accordion>

  <Accordion title="7. What is sticky session (session affinity) in the context of WebSockets, and what happens if you do not use it?">
    **Strong Answer:**
    - Sticky sessions mean the load balancer routes all requests from a given client to the same backend server. For WebSockets, this is critical because a WebSocket connection is stateful -- once established, it lives on a specific server that holds the connection object, the user's room memberships, and any in-memory state associated with that session.
    - Without sticky sessions, the initial HTTP upgrade request might go to Server 1 (which creates the WebSocket connection), but if the client briefly disconnects and reconnects, the new connection might land on Server 5. Server 1 still has stale state for that user (thinking they are connected), and Server 5 has no context about what rooms they were in. You end up with ghost connections on Server 1, missing room memberships on Server 5, and broken message delivery.
    - How sticky sessions work at the load balancer: typically via a cookie (the LB sets a `SERVERID` cookie on the first response, and routes subsequent requests with that cookie to the same backend) or via source IP hash (hash the client's IP to determine the backend). Cookie-based is more reliable because IP can change (mobile networks, VPN reconnects), but it requires the LB to inspect HTTP headers during the upgrade. Most modern load balancers (AWS ALB, Nginx, HAProxy) support WebSocket-aware sticky sessions.
    - The downside of sticky sessions: they create uneven load distribution. If one server gets assigned a disproportionate number of heavy users (e.g., users in large chat rooms), that server is overloaded while others are idle. Mitigation: use "least connections" as the primary routing algorithm with sticky sessions as a secondary rule -- new connections go to the least-loaded server, but reconnections go to the original server.
    - Sticky sessions also complicate deploys. When you deploy a new version, you cannot just restart servers because that drops all their WebSocket connections. You need graceful connection draining: the load balancer stops sending new connections to the server being updated, existing connections are given time to complete or are explicitly migrated (the server sends a "reconnect" frame to clients, which then establish new connections that the LB routes to healthy servers).
    - **Example:** A team deployed their WebSocket application behind AWS ALB without enabling sticky sessions. Users in chat rooms would intermittently stop receiving messages. The root cause: when a client's connection was briefly interrupted (common on mobile), the reconnection hit a different server. The new server did not know the user was in any rooms, so it did not deliver messages. The old server still had the stale room membership and was trying to deliver messages to a dead connection. Enabling ALB sticky sessions (using the `AWSALB` cookie) fixed the issue immediately.

    **Red flag answer:** "You do not need sticky sessions if you store all state in Redis." While externalizing state to Redis reduces the need for sticky sessions, the WebSocket connection object itself is inherently local to a server -- you cannot store a TCP socket in Redis. Sticky sessions or a reconnection-to-same-server mechanism is still needed for the connection lifecycle.

    **Follow-ups:**
    1. During a deployment, you need to restart a WebSocket server that has 20K active connections. How do you do this without dropping all 20K connections at once?
    2. A mobile user's IP address changes as they switch from WiFi to cellular, breaking the IP-hash-based sticky session. Their WebSocket reconnects to a different server. How do you handle state continuity?
  </Accordion>

  <Accordion title="8. You are building a typing indicator feature for a chat application. It seems simple, but what are the edge cases and how do you prevent it from becoming a performance problem?">
    **Strong Answer:**
    - The basic idea: when a user starts typing, broadcast `{"type": "typing_start", "user_id": "alice"}` to the room. When they stop, broadcast `typing_stop`. But this seemingly simple feature has several edge cases that can degrade performance and user experience.
    - Edge case 1: the user stops typing without sending a message (they got distracted, switched tabs, or just paused). You never receive a clean "stop typing" signal. Solution: use TTL-based expiry. Set a short TTL (3 seconds) on the "typing" state in Redis. Every keystroke resets the TTL. If no keystroke comes within 3 seconds, the typing indicator automatically disappears. The client-side should also set a local timer that removes the indicator after 3 seconds of inactivity.
    - Edge case 2: broadcast storm. A fast typist in a room with 500 members triggers a `typing_start` event on every keystroke. At 5 keystrokes per second, that is 2,500 messages per second just for one person typing. Solution: debounce on the client side -- send `typing_start` at most once every 2-3 seconds. On the server side, only broadcast if the user was not already marked as typing (check Redis before broadcasting). This reduces the message volume from thousands per second to at most one every 2-3 seconds per typing user.
    - Edge case 3: user disconnects while typing. The server detects the WebSocket disconnect but may not clean up the typing state. Other users see "Alice is typing..." forever. Solution: the TTL-based approach handles this automatically -- the typing state expires in 3 seconds without a heartbeat to refresh it. Also, the disconnect handler should explicitly delete the typing key.
    - Edge case 4: showing "several people are typing" instead of individual names. When 5+ people are typing simultaneously, displaying all names is noisy and impractical. Most chat apps show individual names for 1-3 typists and switch to "several people are typing..." for 4+. This is a client-side display decision, but the server should still send individual typing events.
    - Performance at scale: typing indicators should not use the same delivery path as chat messages. They are ephemeral, lossy (missing one is fine), and high-frequency. In a large system, consider (1) not persisting typing events at all -- they are fire-and-forget via Redis Pub/Sub only, (2) rate-limiting per user (max one typing event per 2 seconds), (3) only sending typing events to users who are actively viewing the room (if a user has the room minimized, do not bother delivering typing indicators).
    - **Example:** Slack handles typing indicators by debouncing to once per 3 seconds on the client, using a 5-second TTL on the server, and only broadcasting to users who have the channel in focus. They estimated that without these optimizations, typing indicators alone would generate more network traffic than actual messages in busy workspaces.

    **Red flag answer:** "Just broadcast every keystroke to the room." This shows no awareness of the performance implications and would create a broadcast storm in any room with more than a handful of users.
