## Why Consensus Matters

In distributed systems, multiple nodes must agree on:
- Who is the leader?
- What is the correct order of operations?
- What is the current state?

Without consensus, you get split-brain, data inconsistency, or unavailability.

The real-world analogy: imagine three friends trying to decide on a restaurant over group text, but messages arrive out of order and some get dropped. Alice texts "Let's go to Sushi Place," Bob texts "Let's go to Pizza Place," and Carol's phone dies. When Carol's phone comes back, which restaurant did they agree on? This is the consensus problem -- and it gets dramatically harder when you add the constraint that the friends must *never* show up at different restaurants (safety) while also not waiting forever to decide (liveness).

Understanding consensus at depth is what separates senior from staff-level engineers. Most candidates can say "Raft uses leader election" -- far fewer can explain *why* the leader's log must be at least as up-to-date as any voter's log, or what happens when a leader is elected but has uncommitted entries from a prior term.

## FLP Impossibility Theorem

> "In an asynchronous network, it's impossible to guarantee consensus if even one node can fail."

This is one of the most important theoretical results in distributed computing, proven by Fischer, Lynch, and Paterson in 1985. It does not mean consensus is impossible in practice -- it means there is no algorithm that *always* terminates in all scenarios. Real systems work around this by using timeouts (which technically assume partial synchrony -- you assume messages arrive *eventually* within some bound, even if you do not know the exact bound).

This means we must make trade-offs:
- **Paxos/Raft**: Sacrifice liveness for safety (may get stuck during a partition, but never produce incorrect results -- the system pauses rather than diverges)
- **Eventual Consistency**: Sacrifice strong consistency for availability (every node responds immediately, but responses may be stale or conflicting, converging over time)

## Raft Consensus Algorithm

Raft was created by Diego Ongaro and John Ousterhout at Stanford specifically because Paxos was too hard for most engineers to understand and implement correctly. Their 2014 paper explicitly optimized for understandability as a primary design goal -- and it worked. Raft is now the consensus algorithm that powers much of modern infrastructure:

- **etcd** (the coordination layer for Kubernetes -- every K8s cluster runs Raft under the hood)
- **CockroachDB** (distributed SQL database using Raft for range-level replication)
- **TiKV** (the storage layer for TiDB, handling millions of Raft groups)
- **Consul** (HashiCorp's service mesh and service discovery platform)

```python
from enum import Enum
from dataclasses import dataclass, field
from typing import List, Dict, Optional, Callable
import asyncio
import random
import time
import logging

logger = logging.getLogger(__name__)

class NodeState(Enum):
    FOLLOWER = "follower"
    CANDIDATE = "candidate"
    LEADER = "leader"

@dataclass
class LogEntry:
    term: int
    index: int
    command: str
    
@dataclass
class RequestVoteArgs:
    term: int
    candidate_id: str
    last_log_index: int
    last_log_term: int

@dataclass
class RequestVoteReply:
    term: int
    vote_granted: bool

@dataclass
class AppendEntriesArgs:
    term: int
    leader_id: str
    prev_log_index: int
    prev_log_term: int
    entries: List[LogEntry]
    leader_commit: int

@dataclass
class AppendEntriesReply:
    term: int
    success: bool
    match_index: int = 0


class RaftNode:
    """
    Simplified Raft implementation for educational purposes.
    Production implementations need more error handling.
    """
    
    def __init__(self, node_id: str, peers: List[str], rpc_client):
        self.node_id = node_id
        self.peers = peers
        self.rpc = rpc_client
        
        # Persistent state (saved to disk)
        self.current_term = 0
        self.voted_for: Optional[str] = None
        self.log: List[LogEntry] = []
        
        # Volatile state
        self.state = NodeState.FOLLOWER
        self.commit_index = 0
        self.last_applied = 0
        
        # Leader state (reinitialized after election)
        self.next_index: Dict[str, int] = {}
        self.match_index: Dict[str, int] = {}
        
        # Timing
        self.election_timeout = self._random_timeout()
        self.last_heartbeat = time.time()
        
        # State machine
        self.state_machine: Dict[str, str] = {}
        
        # Callbacks
        self.on_become_leader: Optional[Callable] = None
        self.on_commit: Optional[Callable] = None
    
    def _random_timeout(self) -> float:
        """Random election timeout to prevent split votes"""
        return random.uniform(0.15, 0.3)  # 150-300ms
    
    def _last_log_index(self) -> int:
        return len(self.log)
    
    def _last_log_term(self) -> int:
        if not self.log:
            return 0
        return self.log[-1].term
    
    def _get_log_term(self, index: int) -> int:
        if index <= 0 or index > len(self.log):
            return 0
        return self.log[index - 1].term
    
    # ==================== Leader Election ====================
    
    async def run(self):
        """Main loop - runs forever"""
        while True:
            if self.state == NodeState.FOLLOWER:
                await self._run_follower()
            elif self.state == NodeState.CANDIDATE:
                await self._run_candidate()
            elif self.state == NodeState.LEADER:
                await self._run_leader()
    
    async def _run_follower(self):
        """Wait for heartbeat or start election"""
        while self.state == NodeState.FOLLOWER:
            if time.time() - self.last_heartbeat > self.election_timeout:
                logger.info(f"{self.node_id}: Election timeout, becoming candidate")
                self.state = NodeState.CANDIDATE
                return
            await asyncio.sleep(0.01)
    
    async def _run_candidate(self):
        """Request votes from all peers"""
        self.current_term += 1
        self.voted_for = self.node_id
        votes_received = 1  # Vote for self
        
        logger.info(f"{self.node_id}: Starting election for term {self.current_term}")
        
        # Reset election timeout
        self.election_timeout = self._random_timeout()
        election_start = time.time()
        
        # Request votes in parallel
        vote_requests = [
            self._request_vote(peer) for peer in self.peers
        ]
        
        responses = await asyncio.gather(*vote_requests, return_exceptions=True)
        
        for response in responses:
            if isinstance(response, Exception):
                continue
            
            if response.term > self.current_term:
                # Discovered higher term, step down
                self.current_term = response.term
                self.state = NodeState.FOLLOWER
                self.voted_for = None
                return
            
            if response.vote_granted:
                votes_received += 1
        
        # Check if we won
        majority = (len(self.peers) + 1) // 2 + 1
        if votes_received >= majority:
            logger.info(f"{self.node_id}: Won election with {votes_received} votes")
            self._become_leader()
        else:
            # Election failed, go back to follower
            logger.info(f"{self.node_id}: Lost election, got {votes_received}/{majority} votes")
            self.state = NodeState.FOLLOWER
    
    async def _request_vote(self, peer: str) -> RequestVoteReply:
        """Send RequestVote RPC to a peer"""
        args = RequestVoteArgs(
            term=self.current_term,
            candidate_id=self.node_id,
            last_log_index=self._last_log_index(),
            last_log_term=self._last_log_term()
        )
        return await self.rpc.request_vote(peer, args)
    
    def handle_request_vote(self, args: RequestVoteArgs) -> RequestVoteReply:
        """Handle incoming RequestVote RPC"""
        
        # If request term is old, reject
        if args.term < self.current_term:
            return RequestVoteReply(term=self.current_term, vote_granted=False)
        
        # If we see a higher term, update and become follower
        if args.term > self.current_term:
            self.current_term = args.term
            self.state = NodeState.FOLLOWER
            self.voted_for = None
        
        # Check if we can vote for this candidate
        can_vote = (
            (self.voted_for is None or self.voted_for == args.candidate_id) and
            self._is_log_up_to_date(args.last_log_index, args.last_log_term)
        )
        
        if can_vote:
            self.voted_for = args.candidate_id
            self.last_heartbeat = time.time()  # Reset timeout
            return RequestVoteReply(term=self.current_term, vote_granted=True)
        
        return RequestVoteReply(term=self.current_term, vote_granted=False)
    
    def _is_log_up_to_date(self, last_index: int, last_term: int) -> bool:
        """Check if candidate's log is at least as up-to-date as ours"""
        our_last_term = self._last_log_term()
        our_last_index = self._last_log_index()
        
        if last_term != our_last_term:
            return last_term > our_last_term
        return last_index >= our_last_index
    
    # ==================== Leader Operations ====================
    
    def _become_leader(self):
        """Initialize leader state"""
        self.state = NodeState.LEADER
        
        # Initialize next_index and match_index
        next_idx = self._last_log_index() + 1
        for peer in self.peers:
            self.next_index[peer] = next_idx
            self.match_index[peer] = 0
        
        if self.on_become_leader:
            self.on_become_leader()
    
    async def _run_leader(self):
        """Send heartbeats and replicate log"""
        while self.state == NodeState.LEADER:
            await self._send_heartbeats()
            await asyncio.sleep(0.05)  # Heartbeat interval: 50ms
    
    async def _send_heartbeats(self):
        """Send AppendEntries to all peers"""
        tasks = [
            self._send_append_entries(peer) for peer in self.peers
        ]
        await asyncio.gather(*tasks, return_exceptions=True)
    
    async def _send_append_entries(self, peer: str):
        """Send AppendEntries RPC to a specific peer"""
        next_idx = self.next_index[peer]
        prev_log_index = next_idx - 1
        prev_log_term = self._get_log_term(prev_log_index)
        
        # Get entries to send
        entries = self.log[next_idx - 1:] if next_idx <= len(self.log) else []
        
        args = AppendEntriesArgs(
            term=self.current_term,
            leader_id=self.node_id,
            prev_log_index=prev_log_index,
            prev_log_term=prev_log_term,
            entries=entries,
            leader_commit=self.commit_index
        )
        
        try:
            reply = await self.rpc.append_entries(peer, args)
            
            if reply.term > self.current_term:
                # Step down
                self.current_term = reply.term
                self.state = NodeState.FOLLOWER
                self.voted_for = None
                return
            
            if reply.success:
                # Update next_index and match_index
                self.next_index[peer] = next_idx + len(entries)
                self.match_index[peer] = prev_log_index + len(entries)
                
                # Check if we can advance commit index
                self._advance_commit_index()
            else:
                # Log inconsistency, decrement next_index and retry
                self.next_index[peer] = max(1, self.next_index[peer] - 1)
                
        except Exception as e:
            logger.warning(f"Failed to send AppendEntries to {peer}: {e}")
    
    def _advance_commit_index(self):
        """Advance commit index if majority have replicated"""
        for n in range(self.commit_index + 1, self._last_log_index() + 1):
            if self._get_log_term(n) != self.current_term:
                continue
            
            # Count replicas
            count = 1  # Self
            for peer in self.peers:
                if self.match_index.get(peer, 0) >= n:
                    count += 1
            
            majority = (len(self.peers) + 1) // 2 + 1
            if count >= majority:
                self.commit_index = n
                self._apply_committed()
    
    def handle_append_entries(self, args: AppendEntriesArgs) -> AppendEntriesReply:
        """Handle incoming AppendEntries RPC"""
        
        # Old term, reject
        if args.term < self.current_term:
            return AppendEntriesReply(term=self.current_term, success=False)
        
        # Update term and become follower
        if args.term > self.current_term:
            self.current_term = args.term
            self.voted_for = None
        
        self.state = NodeState.FOLLOWER
        self.last_heartbeat = time.time()
        
        # Check log consistency
        if args.prev_log_index > 0:
            if args.prev_log_index > len(self.log):
                return AppendEntriesReply(term=self.current_term, success=False)
            
            if self._get_log_term(args.prev_log_index) != args.prev_log_term:
                # Conflict, delete this and all following entries
                self.log = self.log[:args.prev_log_index - 1]
                return AppendEntriesReply(term=self.current_term, success=False)
        
        # Append new entries
        for i, entry in enumerate(args.entries):
            index = args.prev_log_index + 1 + i
            if index <= len(self.log):
                if self.log[index - 1].term != entry.term:
                    self.log = self.log[:index - 1]
                    self.log.append(entry)
            else:
                self.log.append(entry)
        
        # Update commit index
        if args.leader_commit > self.commit_index:
            self.commit_index = min(args.leader_commit, self._last_log_index())
            self._apply_committed()
        
        return AppendEntriesReply(
            term=self.current_term, 
            success=True,
            match_index=self._last_log_index()
        )
    
    # ==================== Client Operations ====================
    
    async def propose(self, command: str) -> bool:
        """Propose a new command (only works on leader)"""
        if self.state != NodeState.LEADER:
            return False
        
        entry = LogEntry(
            term=self.current_term,
            index=self._last_log_index() + 1,
            command=command
        )
        self.log.append(entry)
        
        # Replicate to followers
        await self._send_heartbeats()
        
        return True
    
    def _apply_committed(self):
        """Apply committed entries to state machine"""
        while self.last_applied < self.commit_index:
            self.last_applied += 1
            entry = self.log[self.last_applied - 1]
            
            # Apply to state machine
            self._apply_command(entry.command)
            
            if self.on_commit:
                self.on_commit(entry)
    
    def _apply_command(self, command: str):
        """Apply command to state machine (simple key-value store)"""
        parts = command.split()
        if parts[0] == "SET" and len(parts) == 3:
            self.state_machine[parts[1]] = parts[2]
        elif parts[0] == "DELETE" and len(parts) == 2:
            self.state_machine.pop(parts[1], None)
```

## Paxos

Paxos is the original consensus algorithm, but notoriously difficult to understand. Raft was designed as a more understandable alternative.

```
┌─────────────────────────────────────────────────────────────────┐
│                      BASIC PAXOS                                │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  Roles:                                                         │
│  • Proposer: Proposes values                                   │
│  • Acceptor: Votes on proposals                                │
│  • Learner: Learns the chosen value                            │
│                                                                 │
│  Phase 1: Prepare                                               │
│  ────────────────                                               │
│  Proposer ──────Prepare(n)──────► Acceptors                    │
│           ◄─────Promise(n, v)─────                             │
│                                                                 │
│  "I want to use proposal number n"                             │
│  "OK, I promise not to accept anything lower"                  │
│                                                                 │
│  Phase 2: Accept                                                │
│  ───────────────                                                │
│  Proposer ──────Accept(n, v)────► Acceptors                    │
│           ◄─────Accepted(n, v)────                             │
│                                                                 │
│  "Accept this value with proposal n"                           │
│  "OK, I accepted it"                                           │
│                                                                 │
│  Value is chosen when majority of acceptors accept             │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### Multi-Paxos

```
Single Paxos = Choose ONE value
Multi-Paxos = Choose a SEQUENCE of values (log)

Optimization: Elect a stable leader to skip Phase 1
This is essentially what Raft does with its leader election!
```

## Practical Leader Election

For many systems, you don't need full consensus—just leader election:

```python
import asyncio
import time
from typing import Optional, Callable
import redis

class RedisLeaderElection:
    """
    Leader election using Redis with TTL-based leases.
    Simpler than Raft, good for many use cases.
    """
    
    def __init__(
        self,
        redis_client: redis.Redis,
        node_id: str,
        key: str = "leader",
        ttl: int = 10
    ):
        self.redis = redis_client
        self.node_id = node_id
        self.key = key
        self.ttl = ttl
        self.is_leader = False
        self.on_become_leader: Optional[Callable] = None
        self.on_lose_leadership: Optional[Callable] = None
    
    async def run(self):
        """Main election loop"""
        while True:
            try:
                if self.is_leader:
                    # Try to renew lease
                    renewed = await self._renew_lease()
                    if not renewed:
                        self._step_down()
                else:
                    # Try to become leader
                    acquired = await self._acquire_lease()
                    if acquired:
                        self._become_leader()
                
                await asyncio.sleep(self.ttl / 3)  # Renew at 1/3 TTL
                
            except Exception as e:
                if self.is_leader:
                    self._step_down()
                await asyncio.sleep(1)
    
    async def _acquire_lease(self) -> bool:
        """Try to acquire leadership using SET NX"""
        result = await self.redis.set(
            self.key,
            self.node_id,
            nx=True,  # Only set if not exists
            ex=self.ttl
        )
        return result is True
    
    async def _renew_lease(self) -> bool:
        """Renew lease only if we're still leader"""
        # Lua script for atomic check-and-renew
        script = """
        if redis.call('GET', KEYS[1]) == ARGV[1] then
            redis.call('EXPIRE', KEYS[1], ARGV[2])
            return 1
        else
            return 0
        end
        """
        result = await self.redis.eval(script, 1, self.key, self.node_id, self.ttl)
        return result == 1
    
    def _become_leader(self):
        self.is_leader = True
        if self.on_become_leader:
            self.on_become_leader()
    
    def _step_down(self):
        self.is_leader = False
        if self.on_lose_leadership:
            self.on_lose_leadership()
    
    def get_leader(self) -> Optional[str]:
        """Get current leader (for followers to find leader)"""
        return self.redis.get(self.key)


# Fencing tokens for preventing split-brain
class FencedLeaderElection(RedisLeaderElection):
    """
    Leader election with fencing tokens.
    Prevents split-brain when old leader doesn't know it's no longer leader.
    """
    
    def __init__(self, *args, **kwargs):
        super().__init__(*args, **kwargs)
        self.fencing_token = 0
    
    async def _acquire_lease(self) -> bool:
        # Lua script for atomic acquire with incrementing token
        script = """
        if redis.call('SET', KEYS[1], ARGV[1], 'NX', 'EX', ARGV[2]) then
            local token = redis.call('INCR', KEYS[2])
            return token
        else
            return 0
        end
        """
        result = await self.redis.eval(
            script, 2, 
            self.key, f"{self.key}:token",
            self.node_id, self.ttl
        )
        
        if result > 0:
            self.fencing_token = result
            return True
        return False
    
    def get_fencing_token(self) -> int:
        """
        Return current fencing token.
        Use this token when writing to shared resources.
        Resource should reject writes with older tokens.
        """
        return self.fencing_token
```

### Common Questions

**Q: Why do we need consensus at all?**
> Without it, distributed nodes can disagree on state, leading to data loss or corruption. Consensus ensures all nodes agree on the order of operations.

**Q: What happens during a network partition?**
> The partition with majority continues operating. The minority partition cannot make progress (availability sacrificed for consistency).

**Q: Why is Raft easier than Paxos?**
> Raft separates concerns clearly (leader election, log replication, safety) and uses a strong leader model that makes the algorithm more intuitive.

**Q: When would you NOT use consensus?**
> When eventual consistency is acceptable (social feeds, analytics), or when you can tolerate some data loss. Consensus has latency and availability costs.

<Accordion title="You are running a 5-node Raft cluster for your metadata store. During a rolling deployment, 2 nodes are briefly down simultaneously. A client reports that writes are failing. Walk me through exactly what is happening and what you would do differently.">
**Strong Answer:**

A 5-node Raft cluster requires a majority of 3 nodes (N/2 + 1 = 3) to commit any write. With 2 nodes down, you have exactly 3 nodes remaining -- this should still work. But the devil is in the details.

- **If one of the 2 down nodes is the leader**: The remaining 3 nodes will trigger an election after the election timeout (150-300ms). During the election, no writes are accepted. If the rolling deployment bounces nodes quickly (say, 10 seconds each), and the new leader is one of the nodes about to be restarted, you could get a cascade: elect a leader, immediately restart that leader, elect again, restart again. The cluster spends most of its time in election mode, and clients see write failures.
- **The fix: deployment ordering matters**. Never deploy the current leader first. Check who the leader is (`etcdctl endpoint status`), deploy followers first, wait for each to rejoin and catch up, then finally deploy the leader. The leadership transfer happens once, cleanly.
- **The deeper mistake**: Taking 2 nodes down simultaneously in a 5-node cluster is operating at the absolute minimum quorum. One more failure and the cluster is unavailable. The right approach for rolling deployments: take down 1 node, wait for it to rejoin and confirm its log is caught up (`match_index` equals leader's `last_log_index`), then proceed to the next. This ensures you are never below N-1 healthy nodes.

**Back-of-envelope for unavailability window**: If each node restart takes 30 seconds and you have 5 nodes, a serial rolling deployment takes 5 * 30 = 150 seconds. During each 30-second window, the cluster operates at 4/5 nodes (safe). With 2 simultaneous, you cut the deployment to 75 seconds but risk availability. The 75-second savings is not worth the risk.

**Follow-up: Your Raft cluster is in steady state, but you notice that write latency is 50ms when it should be 2ms. All 5 nodes are healthy. What do you investigate?**

This is almost always a slow follower. Raft commits are gated by the second-fastest quorum response (for a 5-node cluster, that is the 3rd fastest). If one follower has a slow disk (SSD degradation, noisy neighbor in cloud), it does not matter -- the leader only waits for 2 followers. But if 2 followers are slow, the leader must wait for the slowest of those 2. Check: `raft_apply_latency` metrics per node, disk I/O latency (`iowait`), and network RTT between nodes. Another common cause: the leader's WAL write is slow (it must fsync its own log before sending AppendEntries). If the leader is on a degraded disk, every write is slow. Solution: transfer leadership to a healthy node using `raft.LeaderTransfer()` -- takes effect in one election round, zero downtime.

<Accordion title="Your company is evaluating whether to use Raft consensus (via etcd) or a simpler Redis-based leader election with TTL leases for a distributed task scheduler. The scheduler runs 10,000 tasks/day. Make the case for each approach.">
**Strong Answer:**

This is fundamentally a question about how catastrophic a split-brain scenario would be.

**Redis-based leader election (TTL lease)**:
- **How it works**: SET with NX and EX (set-if-not-exists with TTL). Leader renews the lease at 1/3 of the TTL interval. If it misses a renewal, another node acquires leadership.
- **The failure mode**: If the leader has a long GC pause, a network blip, or a slow Redis response, its lease expires, a new leader is elected, but the old leader does not know it has been deposed. For a brief window (until the old leader's next renewal attempt fails), both think they are leader. Two task schedulers might assign the same task to two workers.
- **Mitigation**: Fencing tokens. Each lease acquisition increments a counter. Workers reject task assignments from leaders with an older fencing token. This prevents duplicate execution but adds complexity.
- **Cost**: One Redis instance, ~$50/month. Latency: ~1ms for lease operations.

**Raft-based consensus (etcd)**:
- **How it works**: True distributed consensus. The leader is authoritative because a majority of nodes agree. Split-brain is impossible by construction -- the minority partition cannot elect a leader.
- **The failure mode**: During a network partition, the minority side is completely unavailable (no reads, no writes). If your scheduler is on the minority side, it stops scheduling tasks entirely until the partition heals. This is safe but has availability cost.
- **Cost**: Minimum 3 etcd nodes, ~$300-500/month. Latency: ~2-5ms for consensus operations.

**My recommendation for 10,000 tasks/day**: Redis with fencing tokens. Here is why: 10,000 tasks/day = ~7 tasks/minute. The probability of a split-brain during the exact moment a task is being assigned is extremely low. And even if it happens, a duplicate task execution for a scheduler (which should be idempotent anyway) is a minor inconvenience, not a data corruption event. The operational simplicity of Redis (one instance, no cluster management) wins over Raft for this use case.

**I would choose Raft if**: The scheduler manages financial transactions, database migrations, or any operation where duplicate execution causes data corruption. The safety guarantee of "never two leaders" is worth the operational cost of running an etcd cluster.

**Follow-up: Six months later, the scheduler now manages 1M tasks/day and some tasks trigger bank transfers. Does your recommendation change?**

Absolutely. At 1M tasks/day (~12 tasks/second) with financial implications, the Redis approach's split-brain window is no longer an acceptable risk. A fencing token protects against duplicate execution, but the complexity of ensuring every downstream system correctly validates fencing tokens across a growing number of worker types becomes a maintenance burden. Switch to etcd. The $500/month cost is negligible compared to the risk of a single duplicate bank transfer. More importantly, at 12 tasks/second, you probably need multiple scheduler instances for availability anyway -- and etcd's leader election trivially handles "which scheduler instance is active" with zero application-level fencing logic.
</Accordion>

<Accordion title="Estimate the network overhead of running Raft consensus in a 5-node cluster that processes 10,000 writes per second. How much bandwidth does Raft itself consume?">
**Strong Answer:**

Let me trace the network traffic for each Raft operation.

**Per write operation**:
1. Client sends write to leader: 1 message (~200 bytes for a typical key-value write).
2. Leader sends AppendEntries to 4 followers: 4 messages, each containing the log entry (~200 bytes payload + ~50 bytes RPC overhead = ~250 bytes each). Total: 1,000 bytes outbound from leader.
3. Followers send AppendEntries response: 4 messages, each ~50 bytes. Total: 200 bytes inbound to leader.
4. Leader commits and responds to client: 1 message (~100 bytes).

**Total per write**: ~1,500 bytes of network traffic. At 10,000 writes/sec: 15 MB/sec = 120 Mbps. This is well within the capacity of even a 1 Gbps network link, and trivial on a 10 Gbps datacenter network.

**Heartbeats (no writes in-flight)**:
- Leader sends empty AppendEntries to 4 followers every 50-100ms (configurable). Each heartbeat is ~50 bytes. At 4 followers * 10 heartbeats/sec = 40 messages/sec = 2 KB/sec. Negligible.

**The real bandwidth concern**: Log replication when a follower falls behind. If a follower was down for 10 minutes during which 6 million writes accumulated (10K/sec * 600sec), and each entry is 200 bytes, the catch-up transfer is 1.2 GB. At 1 Gbps, that takes ~10 seconds. During catch-up, the follower is not counted toward the quorum for new commits, so you are effectively running with 4 nodes (still above majority for a 5-node cluster).

**When bandwidth becomes a problem**: When your write payloads are large. If each write is 10KB instead of 200 bytes, the per-write traffic jumps to 50KB * 4 followers = 200KB. At 10K writes/sec: 2 GB/sec = 16 Gbps. Now you need 10 Gbps links and careful NIC tuning. This is why systems like CockroachDB (which uses Raft per range) use a separate data path for large values -- the Raft log contains only a reference, and the actual data is transferred out-of-band.

**Follow-up: You need to add a 6th node to your cluster for additional read capacity. What happens during the membership change, and why is it dangerous?**

Adding a node is a cluster membership change, and it is one of the trickiest operations in Raft. The danger is the "joint consensus" problem: during the transition from 5 to 6 nodes, there is a window where the old configuration (majority = 3) and the new configuration (majority = 4) could each independently elect a leader, violating the single-leader guarantee. Raft solves this with a two-phase configuration change. First, the leader commits a "joint" configuration entry (old + new) to the log. During this phase, commits require a majority from BOTH the old and new configurations. Once that is committed, the leader commits the final "new" configuration. The etcd implementation handles this automatically with `member add`, but you should monitor the cluster during the transition and avoid doing other risky operations (like deploying) simultaneously.
