## How to Use This Reference

This is a quick-lookup guide for system design patterns. Use it to:
- Refresh your memory before interviews
- Find the right pattern for a problem
- Understand trade-offs at a glance

Think of this page like a toolbox index. A carpenter does not use a hammer for every job -- they pick the right tool for the material. Similarly, system design patterns are tools for specific problems. The most common interview mistake is reaching for a pattern because you know it well rather than because it fits the problem. For each pattern below, pay attention to the "Use" line -- that tells you *when* to reach for it.

## 🏗️ Architecture Patterns

### Layered Architecture

```
┌─────────────────────────┐
│   Presentation Layer    │  UI, API endpoints
├─────────────────────────┤
│    Business Layer       │  Business logic
├─────────────────────────┤
│   Persistence Layer     │  Data access
├─────────────────────────┤
│     Database Layer      │  Storage
└─────────────────────────┘

Use: Traditional applications
Pros: Simple, separation of concerns
Cons: Tight coupling, hard to scale parts
```

### Microservices

```
┌─────────┐ ┌─────────┐ ┌─────────┐
│ Service │ │ Service │ │ Service │
│    A    │ │    B    │ │    C    │
└────┬────┘ └────┬────┘ └────┬────┘
     │           │           │
     └───────────┼───────────┘
                 │
         Message Bus / API Gateway

Use: Large systems, multiple teams
Pros: Independent scaling, deployment
Cons: Distributed complexity, latency
```

### Event-Driven

```
┌──────────┐     Event      ┌──────────┐
│ Producer │───────────────►│  Event   │
└──────────┘                │   Bus    │
                            └────┬─────┘
           ┌─────────────────────┼─────────────────────┐
           │                     │                     │
      ┌────▼────┐          ┌────▼────┐          ┌────▼────┐
      │Consumer │          │Consumer │          │Consumer │
      │    A    │          │    B    │          │    C    │
      └─────────┘          └─────────┘          └─────────┘

Use: Decoupled systems, real-time
Pros: Loose coupling, scalable
Cons: Eventual consistency, debugging
```

### CQRS

```
┌─────────────────────────────────────────┐
│                 Client                  │
└──────────┬─────────────────┬────────────┘
           │                 │
      ┌────▼────┐       ┌────▼────┐
      │ Command │       │  Query  │
      │   API   │       │   API   │
      └────┬────┘       └────┬────┘
           │                 │
      ┌────▼────┐       ┌────▼────┐
      │  Write  │──────►│  Read   │
      │  Model  │ sync  │  Model  │
      └─────────┘       └─────────┘

Use: Complex domains, high read/write ratio
Pros: Optimized models, scalable
Cons: Complexity, eventual consistency
```

### Circuit Breaker

```
         ┌─────────────────────────────┐
         │                             │
         │   CLOSED ◄──────────────────┤
         │     │                       │
         │     │ failures > threshold  │
         │     ▼                       │
         │   OPEN ─────────────────────┤
         │     │                       │
         │     │ timeout expires       │
         │     ▼                       │
         │  HALF-OPEN                  │
         │     │                       │
         │     ├── success → CLOSED    │
         │     └── failure → OPEN      │
         │                             │
         └─────────────────────────────┘

Use: Prevent cascade failures
Pros: Fast fails, recovery time
Cons: Needs tuning
```

### Retry with Backoff

```
Attempt 1: Immediate
Attempt 2: Wait 1s
Attempt 3: Wait 2s
Attempt 4: Wait 4s  (exponential)
Attempt 5: Wait 8s + random jitter

With jitter to prevent thundering herd:
delay = min(cap, base * 2^attempt) + random(0, 1)

Use: Transient failures
Pros: Handles temporary issues
Cons: Increased latency on failure
```

## 💨 Caching Patterns

### Cache-Aside

```
1. App checks cache
2. Cache miss → Read from DB
3. App writes to cache
4. Return data

┌─────┐      ┌─────┐      ┌────┐
│ App │─────►│Cache│      │ DB │
└──┬──┘      └─────┘      └──▲─┘
   │                         │
   └─────────────────────────┘
         Cache miss

Use: General purpose
Pros: Simple, lazy loading
Cons: Cache misses slow
```

### Write-Through

```
1. App writes to cache
2. Cache writes to DB
3. Both updated synchronously

┌─────┐      ┌─────┐      ┌────┐
│ App │─────►│Cache│─────►│ DB │
└─────┘      └─────┘      └────┘

Use: Strong consistency needed
Pros: Data always in sync
Cons: Write latency
```

### Write-Behind

```
1. App writes to cache
2. Cache ACKs immediately
3. Cache writes to DB async

┌─────┐      ┌─────┐  async  ┌────┐
│ App │─────►│Cache│ ─ ─ ─ ─►│ DB │
└─────┘      └─────┘         └────┘

Use: Write-heavy workloads
Pros: Fast writes
Cons: Data loss risk
```

## Quick Pattern Selection Guide

| Problem | Pattern | Trade-off to Mention in Interviews |
|---------|---------|--------------------------------------|
| API entry point | API Gateway | Single point of failure vs. centralized concerns |
| Database bottleneck | Caching, Read Replicas | Stale data risk vs. read performance |
| Service failures cascade | Circuit Breaker | Temporary unavailability vs. total cascade failure |
| Distributed transactions | Saga | Eventual consistency vs. simpler distributed coordination |
| High read/write ratio | CQRS | Operational complexity vs. optimized read/write models |
| Audit requirements | Event Sourcing | Storage growth vs. complete history |
| Variable load | Auto-scaling | Cold start latency vs. cost efficiency |
| Independent teams | Microservices | Distributed system complexity vs. team autonomy |
| Real-time updates | Event-Driven, WebSockets | Connection state management vs. low latency |
| Search functionality | Elasticsearch | Index staleness vs. search performance |
| Session management | Redis | Memory cost vs. sub-millisecond reads |
| File storage | Object Storage (S3) | Higher latency vs. infinite scale and durability |
| Time-series data | InfluxDB, TimescaleDB | Specialized ops knowledge vs. purpose-built performance |
| Graph relationships | Neo4j | Limited ecosystem vs. efficient relationship traversal |

<Tip>
**Practical tip**: In an interview, never just name a pattern. Always pair it with the trade-off. "I'd use a circuit breaker here" is a B-level answer. "I'd use a circuit breaker to prevent cascade failures -- the trade-off is that during the open state, users get degraded responses, so we need a fallback like cached data or a default response" is an A-level answer.
</Tip>

## Interview Questions

### Q1: You're designing an order processing system where a single order involves charging a credit card, reserving inventory, and scheduling delivery -- all across different services. If the payment succeeds but inventory reservation fails, you need to refund. Which pattern do you use and how do you implement it?

<Accordion title="Strong Answer">

This is a textbook Saga pattern problem, and the real interview signal is whether you understand the two flavors and which one fits here.

- **Pattern choice: Orchestration-based Saga.** An Order Saga Orchestrator (a dedicated service or state machine) drives the sequence: `CreateOrder -> ChargePayment -> ReserveInventory -> ScheduleDelivery`. If any step fails, the orchestrator runs compensating transactions in reverse: `CancelDelivery -> ReleaseInventory -> RefundPayment -> CancelOrder`.
- **Why orchestration over choreography here:** with 4 services in a strict sequence, choreography (each service emits an event for the next) creates an invisible workflow spread across services. Debugging "why did this order get stuck" becomes a distributed grep exercise. An orchestrator gives you a single place to see the saga state -- invaluable for customer support when someone says "I was charged but got no delivery."
- **Implementation detail that matters: idempotent compensations.** The refund must be idempotent -- if the orchestrator retries due to a network timeout, you can't double-refund. Stripe handles this with idempotency keys on refund calls. Your inventory release must also be idempotent -- releasing an already-released reservation should be a no-op, not an error.
- **State persistence:** the saga's current step and status must be durably stored (a `sagas` table in PostgreSQL, or a workflow engine like Temporal). If the orchestrator crashes between "payment charged" and "inventory reserved," it must resume from the correct step on restart -- not start over, not skip ahead.
- **What you sacrifice vs. distributed transactions (2PC):** Sagas don't give you isolation. During execution, the payment is charged but inventory isn't yet reserved -- other queries can see this intermediate state. For orders, this is acceptable (the order shows as "Processing"). For financial ledgers, it might not be.

</Accordion>

**Red flag answer:** "Use a distributed transaction with two-phase commit across all services" -- this shows the candidate doesn't understand that 2PC requires all participants to hold locks, which kills availability and throughput in a microservices architecture. Also bad: describing choreography but having no answer for how to trace a failed saga across 4 services.

**Follow-ups:**
1. Your saga orchestrator itself goes down for 30 seconds. What happens to the 500 in-flight sagas, and how do you ensure they resume correctly?
2. The payment provider occasionally takes 45 seconds to respond. Your saga has a 30-second timeout. How do you handle a charge that may or may not have succeeded?

### Q2: Your API gateway is handling 20K requests/second. Downstream, your recommendation service starts timing out intermittently. Within 2 minutes, your entire platform is unresponsive. What pattern should have prevented this, and how would you implement it?

<Accordion title="Strong Answer">

This is a cascading failure caused by a missing circuit breaker -- and likely missing bulkheads too.

- **What happened:** the recommendation service started timing out. The API gateway kept sending requests, each tying up a connection/thread for the full timeout duration (probably 30 seconds). At 20K req/sec, even if only 10% hit recommendations, that's 2K threads stuck waiting. The gateway's thread pool exhausts, and now ALL endpoints are dead -- including checkout, search, and account -- even though those downstream services are perfectly healthy.
- **Circuit breaker implementation:** wrap the recommendation service call in a circuit breaker (Hystrix pattern, now typically Resilience4j or a service mesh like Istio). Configuration: trip to OPEN after 5 failures in a 10-second window. When OPEN, immediately return a fallback (empty recommendations, a cached "popular items" list) -- don't even attempt the call. After 30 seconds, move to HALF-OPEN and try one request. If it succeeds, close the circuit.
- **Bulkhead pattern as the second layer:** even with a circuit breaker, you need resource isolation. Dedicate separate thread pools (or connection pools) per downstream service. The recommendation service gets a pool of 200 threads max. When that pool is full, new recommendation requests fail fast -- but the checkout thread pool (separate 500 threads) is unaffected. Netflix pioneered this: a failure in their recommendation engine never takes down the play button.
- **The fallback hierarchy matters for UX:** circuit open doesn't mean "show an error." It means "degrade gracefully." For recommendations: show cached popular items -> show trending items from a static list -> show nothing and hide the recommendation widget entirely. The user never sees a 500 error.
- **What most people miss:** the timeout value is part of the pattern. If your timeout is 30 seconds but the circuit breaker trips after 5 failures, you're still blocking 5 x 30s = 150 seconds of thread time before the circuit opens. Set aggressive timeouts (500ms for a recommendation call that normally takes 100ms) so failures are detected quickly.

</Accordion>

**Red flag answer:** "Add more servers to the API gateway" -- this treats the symptom (resource exhaustion) instead of the cause (cascading failure). Scaling the gateway just means more threads stuck waiting on a broken downstream service. Also bad: implementing a circuit breaker but having no fallback strategy -- returning a 503 to the user for recommendations is almost as bad as the cascade.

**Follow-ups:**
1. Your circuit breaker is tripping every Monday morning at 9 AM for exactly 3 minutes, then recovers. The recommendation service team says their service is healthy. What's going on?
2. How do you set circuit breaker thresholds differently for a payment service (where false-opens cost revenue) versus a recommendation service (where false-opens are barely noticed)?

### Q3: Your social media platform has a read-to-write ratio of 1000:1. Users write posts occasionally but scroll their feeds constantly. The database is at 90% CPU on reads. How do you redesign this using appropriate patterns?

<Accordion title="Strong Answer">

This is a classic CQRS + read replica + caching problem, and the right answer layers multiple patterns rather than reaching for one silver bullet.

- **Layer 1 -- Read replicas (immediate relief):** add 3-5 PostgreSQL read replicas. Route all feed queries to replicas. This drops primary CPU from 90% to ~20% within a day. At a 1000:1 ratio, you're moving 99.9% of load off the primary. This is the "stop the bleeding" step.
- **Layer 2 -- CQRS (structural fix):** separate your write model (normalized tables: `users`, `posts`, `follows`) from your read model (a denormalized `feed_entries` table or a materialized view optimized for "give me the latest 50 posts from people I follow"). When a user writes a post, a change-data-capture stream (Debezium on PostgreSQL WAL) updates the read model asynchronously. The read model is optimized purely for the feed query -- no joins, single table scan.
- **Layer 3 -- Cache (performance ceiling):** put Redis in front of the read model. Each user's feed is a sorted set in Redis keyed by `feed:`+`userId`, scored by timestamp. Cache the latest 200 entries. On feed request: read from Redis. On cache miss: fall back to the read model DB. Cache invalidation happens via the same CDC stream that updates the read model.
- **Fan-out strategy matters at scale:** when a user with 10M followers posts, you can't write 10M Redis entries synchronously. Use fan-out on read for high-follower accounts (pull their posts at read time and merge with the pre-computed feed). For normal users (<10K followers), fan-out on write (push to follower feeds) is fine. This is exactly how Twitter/X redesigned their timeline after the "whale" era.
- **The trade-off stack:** each layer adds complexity and eventual consistency. Layer 1 has seconds of replication lag. Layer 2 has CDC processing delay. Layer 3 has cache staleness. For a social feed, users tolerate 5-10 seconds of delay in seeing a new post -- this is acceptable.

</Accordion>

**Red flag answer:** "Just add a cache in front of the database" -- this might help but doesn't address the fundamental problem of read/write model mismatch. Also: "Shard the database" -- sharding helps with data volume, not with the read amplification pattern caused by social graph fan-out. The problem isn't data size, it's query pattern.

**Follow-ups:**
1. Your CDC pipeline has a 2-minute lag during peak hours, meaning new posts don't appear in feeds for 2 minutes. Users are complaining. How do you fix this without abandoning the async architecture?
2. A celebrity with 50M followers just posted. Walk me through exactly what happens in your system in the next 10 seconds.

### Q4: Your team is building a new checkout service. Someone suggests using event sourcing to track all order state changes. Is this the right pattern here? When would you say yes, and when would you push back?

<Accordion title="Strong Answer">

The answer depends entirely on what business problems you're trying to solve. Event sourcing is powerful but has a high complexity tax -- I'd say yes only if specific requirements demand it.

- **When I'd say YES to event sourcing for checkout:** (1) You need a complete audit trail for compliance -- PCI-DSS or SOX requires knowing exactly what happened to an order and when. (2) You need temporal queries -- "what was the order state at 3:47 PM when the customer called support?" (3) You need to rebuild read models retroactively -- "we just launched a new analytics dashboard and need to backfill 6 months of data." If any of these are real requirements, event sourcing is the right tool.
- **When I'd push back:** if the team just wants "a history of changes," a simple `order_events` audit log table with an append-only insert on every state change gives you 80% of the benefit at 20% of the complexity. You don't need full event sourcing (event store + projection rebuilds + snapshot management) for an audit log.
- **The complexity costs people underestimate:** (1) **Schema evolution** -- what happens when you add a `discountCode` field to the `OrderCreated` event 6 months in? Now you have events with and without that field, and every projection must handle both versions. (2) **Projection rebuilds** -- rebuilding a read model from 100M events takes hours. You need snapshotting, and snapshot invalidation is its own problem. (3) **Debugging** -- "why does this order show the wrong total?" means replaying events step by step instead of just querying a row.
- **The hybrid I'd actually recommend for most checkout systems:** standard mutable state in PostgreSQL for the order (current state is always queryable), plus an append-only `order_events` table for audit. This gives you debuggability and compliance without the full event sourcing machinery. Reserve true event sourcing for domains where replaying events is a core feature (financial ledgers, collaborative editing, gaming).

</Accordion>

**Red flag answer:** "Yes, always use event sourcing -- it's the best practice" without any analysis of whether the business requirements justify the complexity. Equally bad: "Event sourcing is over-engineered, never use it" -- this misses domains where it's genuinely the best fit. The interview signal is whether the candidate can articulate specific criteria for the decision.

**Follow-ups:**
1. You went with event sourcing. Six months later, a developer accidentally deployed a bug that emitted malformed `PaymentProcessed` events for 8 hours. How do you fix the corrupted event stream?
2. Your event store has 2 billion events and projection rebuilds now take 14 hours. Walk me through your strategy to bring this under 30 minutes.

### Q5: Your application needs to handle 100K WebSocket connections for a live sports scores feature. The current architecture is a single Node.js server. How would you scale this using appropriate patterns?

<Accordion title="Strong Answer">

100K concurrent WebSocket connections is beyond what a single server handles well. The key challenge isn't CPU -- it's memory (each connection holds state) and the broadcast problem (a score update must reach all 100K clients quickly).

- **Horizontal scaling with sticky sessions:** run N Node.js instances behind a load balancer. WebSocket connections are long-lived and stateful, so you need sticky sessions (route by connection ID or IP hash) to ensure a client's messages always reach the same server. AWS ALB supports WebSocket sticky sessions natively.
- **The pub/sub backbone:** when a score updates, every server instance needs to know about it. Use Redis Pub/Sub (or Kafka for durability) as the fan-out layer. The score ingestion service publishes to a `scores` channel. Every Node.js instance subscribes and broadcasts to its local WebSocket connections. This means a score update reaches all 100K clients in ~50ms regardless of how many server instances you have.
- **Connection management matters:** 100K connections at ~10KB memory each = ~1GB baseline. But if clients subscribe to different games, you need subscription management -- a `Map` of `gameId -> Set<WebSocket>` per instance. When a score updates for Game A, you only broadcast to the subset of connections watching Game A, not all 100K.
- **Scaling math:** a single Node.js process comfortably handles ~10K-20K idle WebSocket connections. So 5-10 instances gets you to 100K. If connections are chatty (live commentary, play-by-play), reduce to 5K-8K per instance. Use the OS-level tuning: `ulimit -n 65536`, TCP keep-alive settings, and disable Nagle's algorithm for low-latency pushes.
- **Graceful degradation pattern:** if a server instance dies, its 10K-20K connections drop. Clients must auto-reconnect with exponential backoff + jitter (to prevent a thundering herd of 20K simultaneous reconnects). On reconnect, send the client the latest state snapshot so they don't miss scores during the outage.

</Accordion>

**Red flag answer:** "Just use a bigger server" (vertical scaling has hard limits for WebSocket connections due to file descriptor and memory constraints). Also: "Use HTTP polling instead" -- at 100K clients polling every second, that's 100K req/sec of redundant traffic, far more expensive than persistent connections for a real-time use case.

**Follow-ups:**
1. During the Super Bowl, your connection count spikes to 2M in 10 minutes. Your auto-scaling can't provision instances fast enough. What's your pre-game scaling strategy?
2. You notice that 30% of your WebSocket connections are idle (users left the tab open but aren't watching). How do you reclaim those resources without degrading the experience for active users?

### Q6: Your microservices architecture has 15 services, and debugging production issues takes hours because logs are scattered everywhere. A request from the user hits 6 different services. How do you implement observability?

<Accordion title="Strong Answer">

The three pillars -- logs, metrics, traces -- work together, but distributed tracing is the highest-leverage investment for your specific problem of "debugging takes hours across 6 services."

- **Distributed tracing (top priority):** instrument every service with OpenTelemetry. When a request enters the API gateway, generate a trace ID (e.g., `trace-abc123`) and propagate it in headers (`traceparent` per W3C spec) through every downstream call. Each service creates spans with timing, status, and metadata. Ship spans to Jaeger or Grafana Tempo. Now when a user reports "my checkout failed," you search by their request ID and see: API Gateway (10ms) -> Auth (5ms) -> Order (200ms) -> Payment (TIMEOUT at 30s). The bottleneck is immediately visible.
- **Structured logging (not just tracing):** every log line must include `traceId`, `spanId`, `service`, `userId`, and be JSON-formatted. This lets you correlate logs across services: search Elasticsearch for `traceId: abc123` and get the full story in chronological order. The common mistake is unstructured logs with `console.log("payment failed")` -- useless in a distributed system because you can't correlate or filter.
- **Metrics with the RED method:** for each service, track Rate (requests/sec), Errors (error rate %), and Duration (latency histogram). Alert on error rate > 1% or p99 latency > 2x the baseline. Use Prometheus + Grafana. Metrics tell you "something is wrong" fast; traces tell you "what and where."
- **The pattern that ties it all together: correlation IDs at the edge.** The API gateway assigns both a `traceId` (for technical debugging) and a `correlationId` (business-level, e.g., `orderId`). Customer support searches by `orderId` and sees the full lifecycle. Engineers search by `traceId` and see the technical path. Both lead to the same request.
- **Practical rollout order:** (1) Add trace propagation headers and structured logging to all services -- 2-week effort. (2) Deploy a trace collector (Jaeger) and connect to existing log aggregation -- 1 week. (3) Build service dashboards with RED metrics -- 1 week. (4) Set up alerting rules. You get 80% of the value in month one.

</Accordion>

**Red flag answer:** "Add more logging" without any mention of correlation/trace IDs -- more unstructured logs just means more noise to sift through. Also: proposing a custom-built tracing solution instead of using OpenTelemetry, which is the industry standard and already has libraries for every major language.

**Follow-ups:**
1. Your tracing shows that the payment service takes 500ms, but the payment team says their service responds in 50ms. Where's the other 450ms, and how do you find it?
2. You've instrumented everything, but your Jaeger instance is receiving 500K spans/second and falling behind. How do you scale observability without losing critical debugging data?

### Q7: Your write-heavy IoT platform ingests sensor data from 1M devices, each sending a reading every 5 seconds. That's 200K writes/second. Your PostgreSQL database is maxed out. Which data pattern do you apply?

<Accordion title="Strong Answer">

At 200K writes/second of time-series data, you've outgrown PostgreSQL's architecture for this workload. The solution combines a purpose-built time-series storage pattern with write buffering.

- **Switch the storage engine:** time-series databases (TimescaleDB, InfluxDB, or QuestDB) are built for exactly this pattern: high-volume append-only writes with time-range queries. TimescaleDB is a pragmatic choice because it's a PostgreSQL extension -- your team keeps their SQL skills and existing tooling. It uses hypertable partitioning (automatic chunking by time) to maintain write performance as data grows.
- **Write buffering with Kafka:** don't let 1M devices write directly to the database. Place Kafka in front as a write buffer. Devices send readings to Kafka topics partitioned by `deviceId`. A consumer fleet batch-inserts into TimescaleDB in chunks of 1000-5000 rows. Batch inserts at 200K rows/sec are dramatically more efficient than 200K individual inserts -- the difference is 10-50x throughput because you amortize transaction overhead.
- **Sharding strategy: time-based partitioning.** TimescaleDB does this automatically with hypertables. Data older than 7 days moves to compressed chunks (10x storage reduction). Data older than 90 days moves to S3 in Parquet format for cold analytics. This keeps your hot storage bounded regardless of how long you retain data.
- **The aggregation pattern for reads:** nobody queries raw 5-second readings for a 30-day trend. Build continuous aggregates (TimescaleDB feature) that pre-compute 1-minute, 1-hour, and 1-day rollups. Dashboard queries hit the rollups, not raw data. This reduces query scan volume by 60-720x depending on the time range.
- **What I'd explicitly avoid:** sharding PostgreSQL across multiple instances to handle the write load. You can make it work, but you're fighting the tool. PostgreSQL isn't designed for append-heavy time-series workloads -- its MVCC model creates vacuum pressure at 200K writes/sec that will eventually degrade performance. Use the right tool.

</Accordion>

**Red flag answer:** "Just add more PostgreSQL replicas" -- replicas help reads, not writes. The write bottleneck is on the primary. Also: "Use MongoDB because it's good at writes" -- MongoDB handles high write throughput but lacks the time-series-specific features (continuous aggregation, automatic time partitioning, compression by age) that make this workload manageable long-term.

**Follow-ups:**
1. A device firmware bug causes 10K devices to send readings every 100ms instead of every 5 seconds, suddenly spiking your write load to 300K/sec. How does your architecture handle this without manual intervention?
2. The business wants to query "what was the average temperature across all 1M devices for the last hour" in under 2 seconds. How do you serve this query?

### Q8: You're designing a multi-tenant SaaS platform. A large enterprise customer (5% of your users) generates 60% of your traffic and their heavy queries are degrading performance for all other tenants. Which patterns do you apply to isolate tenants without running separate infrastructure for each?

<Accordion title="Strong Answer">

This is a noisy neighbor problem, and the solution is a layered isolation strategy using multiple patterns together.

- **Layer 1 -- Rate limiting per tenant (immediate fix):** implement a token bucket rate limiter keyed by `tenantId`. The enterprise customer gets a generous but bounded allocation (say, 5K req/sec). Other tenants share remaining capacity. This stops the bleeding within hours. Use Redis with a Lua script for atomic token bucket operations -- `MULTI`/`EXEC` is too slow at high throughput.
- **Layer 2 -- Bulkhead pattern at the compute layer:** create separate worker pools or even separate auto-scaling groups per tenant tier. "Enterprise" tenants route to a dedicated pool of instances; "standard" tenants share a different pool. If the enterprise customer's heavy queries consume CPU, it exhausts their pool only. AWS ALB listener rules can route by a `X-Tenant-Id` header to different target groups. Cost-efficient because you're sharing infrastructure within tiers, not per-tenant.
- **Layer 3 -- Database-level isolation:** the heavy queries are likely the real problem. Options in order of increasing isolation: (a) Query-level resource limits (PostgreSQL `statement_timeout` per role, set the enterprise tenant's role to a 5s timeout), (b) Separate read replicas per tenant tier (enterprise queries go to replica-A, everyone else to replica-B), (c) Separate database instances for the top 3 tenants (only if revenue justifies the cost -- it often does for enterprise customers paying $500K+/year).
- **Layer 4 -- Queue-based throttling for heavy operations:** if the enterprise customer runs expensive analytics queries or bulk exports, route those through a queue with per-tenant concurrency limits (max 3 concurrent heavy queries). This prevents one tenant from monopolizing database connections. The enterprise customer sees "Your export is queued, position 2 of 3" instead of everyone seeing 30-second response times.
- **The pricing/architecture alignment:** this is where engineering meets business. Tier your pricing by usage limits that align with your isolation boundaries. The enterprise customer pays for their dedicated compute pool and database replica. This turns a cost center (isolation infrastructure) into a revenue line. Slack, Datadog, and every mature SaaS company does this.

</Accordion>

**Red flag answer:** "Give each tenant their own database and infrastructure" -- this doesn't scale beyond 10-20 tenants and turns you into a managed hosting company instead of a SaaS company. Also: "Just add more servers" without any tenant-level isolation -- more servers with no fairness mechanism means the noisy neighbor just consumes the new capacity too.

**Follow-ups:**
1. The enterprise customer complains that their rate limit is too low and threatens to churn (they represent 30% of your ARR). How do you balance their needs against platform stability?
2. You implement per-tenant database replicas, but now schema migrations need to be applied across all replicas. One migration fails on the enterprise replica but succeeds everywhere else. How do you handle this?

<AccordionGroup>
  <Accordion title="Q1: You need to implement a distributed transaction across three services: Order, Payment, and Inventory. Compare the Saga pattern with two-phase commit. Which would you choose and why?" icon="arrows-rotate">
    **What the interviewer is really testing:** Whether you understand the fundamental trade-off between strong consistency (2PC) and availability/performance (Saga), and can pick the right tool based on real constraints.

    **Strong Answer:**
    - Two-phase commit (2PC) provides atomicity: all three services either commit or all abort. The coordinator sends a "prepare" message to each participant, waits for all to vote "yes," then sends "commit." If any participant votes "no" or times out, the coordinator sends "abort" to all. The result is a true distributed transaction -- no partial state is ever visible.
    - The Saga pattern breaks the transaction into a sequence of local transactions, each with a compensating action. If Payment succeeds but Inventory fails, the Saga runs "Refund Payment" to undo the completed step. Unlike 2PC, intermediate states are visible -- the payment is charged before inventory is reserved.
    - I would choose the Saga pattern for this scenario, and here is why: (1) 2PC requires all participants to hold locks during the prepare phase. If the Inventory Service is slow, the Order and Payment services are blocked -- they cannot process other requests until the transaction resolves. In a microservices architecture where each service handles thousands of requests per second, this blocking kills throughput. (2) 2PC has a single point of failure in the coordinator. If the coordinator crashes between "prepare" and "commit," all participants are stuck in a prepared state, unable to commit or abort, potentially holding locks for minutes until the coordinator recovers. (3) 2PC requires all participants to be available simultaneously. If Inventory is down, no orders can be processed, even though Order and Payment are healthy.
    - The cost of choosing Saga: you lose isolation. A query at the wrong moment sees "payment charged, inventory not reserved." For an e-commerce order, this is acceptable -- the order shows as "Processing." For a financial ledger where balances must always be consistent, you might need 2PC or a different approach (like serializing through a single database).
    - Implementation choice: use an Orchestration Saga with a workflow engine (Temporal, AWS Step Functions). The orchestrator explicitly defines the sequence and compensations. This is easier to debug than Choreography, where the workflow is implicit in event handlers across services.

    **Follow-up: What if the compensation itself fails -- the refund call to the payment provider times out. How do you handle this?**

    Compensations must be idempotent and retryable. If the refund call times out, the Saga orchestrator retries with the same idempotency key. After N retries (say 5, with exponential backoff), if the compensation still fails, escalate to a dead-letter queue for manual intervention. The order is flagged with a state like "COMPENSATION_FAILED" which triggers an alert to the operations team. You must never silently ignore a failed compensation -- the customer would be charged without receiving their product. Additionally, before retrying the refund, query the payment provider to check if the original refund actually went through (the timeout might have been on the response, not the request).
  </Accordion>

  <Accordion title="Q2: Your system uses Cache-Aside (lazy loading) and you are seeing a high rate of stale reads after writes. Users update their profile and immediately see the old data. What is going wrong and how do you fix it?" icon="database">
    **What the interviewer is really testing:** Whether you understand cache invalidation race conditions and can design a consistency strategy that balances performance with user expectations.

    **Strong Answer:**
    - The classic Cache-Aside race condition: (1) User updates their profile -- the application writes to the database and then invalidates the cache. (2) Between the database write and the cache invalidation, another request reads the stale cache entry. Or worse: (3) The invalidation happens, but a concurrent read triggers a cache-miss refill from a read replica that has not yet received the write (replication lag). The cache is now repopulated with stale data and will remain stale until the TTL expires.
    - Scenario (3) is the tricky one. The timeline is: T1: write hits the primary DB. T2: cache is invalidated. T3: a read request cache-misses, reads from the replica (which has not received T1's write yet), and populates the cache with stale data. T4: the replica catches up, but the cache now has stale data with a fresh TTL.
    - Fix 1 -- Read-your-own-writes: After a write, the writing user's subsequent reads bypass the cache and read directly from the primary database for a short window (e.g., 5 seconds, long enough for replication to catch up). Implement this by setting a per-user "cache-bypass" flag in Redis with a short TTL after any write. Other users continue reading from the cache.
    - Fix 2 -- Write-through for the writing user: When the user updates their profile, write to both the database AND the cache atomically (or near-atomically). The cache now has the latest data immediately. Other users who read from the cache get the fresh data too. The risk is that the database write succeeds but the cache write fails -- use the database as the source of truth and fall back to invalidation in that case.
    - Fix 3 -- Cache invalidation with a delay: After writing to the database, do not invalidate the cache immediately. Instead, invalidate it after a delay slightly longer than the replication lag (e.g., 500ms). This gives the replica time to catch up. The downside: stale reads for 500ms, but this prevents the race condition where an immediate invalidation causes a refill from a stale replica.
    - Fix 4 -- Use a version stamp: Include a version number in both the cache entry and the database row. When writing, increment the version. When populating the cache, only cache if the version is greater than or equal to the current cache entry's version. This prevents a stale replica read from overwriting a fresh cache entry.
    - In practice, I would combine Fix 1 (read-your-own-writes for the updating user) with Fix 3 (delayed double-delete for everyone else). This gives the best UX: the writer sees their change immediately, and other users see it within a second.

    **Follow-up: Your cache is Redis and your database is PostgreSQL with 3 read replicas. Replication lag is usually under 100ms but occasionally spikes to 5 seconds during peak load. How does this affect your cache invalidation strategy?**

    The variable replication lag makes the fixed-delay invalidation (Fix 3) unreliable -- 500ms delay works at 100ms lag but not at 5-second lag. Options: (1) Monitor replication lag in real-time and dynamically adjust the invalidation delay. Expose the replica's LSN (log sequence number) and compare it to the primary's LSN after each write. Only invalidate the cache (and allow cache refills) once the relevant replica has caught up past the write's LSN. (2) For the writing user, always read from the primary for their next few reads (Fix 1 is unaffected by replica lag). (3) During high-lag periods, automatically fall back to reading from the primary for all users, accepting the higher load on the primary as a temporary cost. Set an alerting threshold on replication lag (e.g., alert if lag exceeds 2 seconds for more than 1 minute) to address the underlying cause.
  </Accordion>

  <Accordion title="Q3: Compare the Event-Driven architecture pattern with a synchronous request-response architecture. A new team member says events make everything more complex. How do you explain when the complexity is justified?" icon="bolt">
    **What the interviewer is really testing:** Whether you can articulate the specific benefits of event-driven architecture beyond buzzwords and identify when it is NOT the right choice.

    **Strong Answer:**
    - The team member is right that events add complexity. The question is whether that complexity buys you something valuable. Event-driven architecture is justified when one or more of these conditions hold: (1) You need loose coupling between producers and consumers. The service that creates an order should not need to know about the 7 things that happen downstream (send email, update analytics, reserve inventory, notify warehouse, etc.). With request-response, the Order Service calls each downstream service, creating tight coupling. With events, it publishes "OrderCreated" and does not care who consumes it. (2) You need to add new consumers without modifying the producer. In a request-response world, adding a "fraud check" step means modifying the Order Service to call the Fraud Service. In an event-driven world, the Fraud Service subscribes to "OrderCreated" events -- zero changes to Order Service. (3) You need to handle bursty workloads. A flash sale creates 10x the normal order volume. In request-response, the Order Service must handle the burst in real-time or shed load. With events and a durable queue (Kafka), the Order Service publishes events at the burst rate, and downstream consumers process them at their own pace. The queue absorbs the burst. (4) You need eventual consistency and can tolerate it. Events naturally lead to eventually consistent systems, which is fine for notifications, analytics, and search indexing but not for inventory reservation during checkout.
    - When event-driven is NOT justified: (1) When you need an immediate response. "Is this credit card valid?" must be synchronous -- you cannot publish an event and wait for a consumer to eventually validate it. (2) When the workflow is simple and linear. If Service A always calls Service B and nothing else, an event bus between them is unnecessary indirection. (3) When your team is small and debugging distributed event flows is a bigger cost than the coupling they eliminate.
    - The specific complexity costs: (1) Debugging is harder -- a request-response failure shows you a stack trace. An event-driven failure shows you "the event was published but the consumer never processed it," and you need distributed tracing to figure out why. (2) Ordering guarantees are tricky. If "PaymentCharged" arrives before "OrderCreated" due to a race condition, the payment handler does not know what order to associate it with. You need idempotency, ordering guarantees (Kafka partitions), or explicit state machines. (3) Testing is harder -- integration tests need to set up event infrastructure, publish events, and wait for asynchronous processing to complete.
    - **Example:** Shopify uses a hybrid approach. Their checkout flow (cart to payment to confirmation) is synchronous request-response because it must be fast and consistent. Everything after the order is placed (send confirmation email, update analytics, notify shipping partner, trigger loyalty points) is event-driven because it is loosely coupled, can tolerate delays, and needs to support dozens of downstream consumers without the checkout service knowing about them.

    **Follow-up: You adopt event-driven architecture and six months later, you discover that three consumers of the "OrderCreated" event each interpret the event slightly differently, causing subtle data inconsistencies. How do you prevent this?**

    This is the schema evolution problem for events. Solutions: (1) Establish a schema registry (Confluent Schema Registry or similar) where every event has a versioned schema (Avro, Protobuf, or JSON Schema). Producers and consumers validate against the registry, catching incompatibilities before deployment. (2) Define clear event contracts owned by the producing team. The contract specifies exactly what each field means, which fields are required, and what the semantics are. Treat events like an API -- with documentation, versioning, and backward compatibility rules. (3) Use consumer-driven contract testing: each consumer team writes tests that define what they expect from the event. The producer's CI pipeline runs all consumer contract tests before merging changes. If a producer change breaks a consumer's expectations, the build fails.
  </Accordion>
