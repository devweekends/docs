## How to Use This Guide

This guide provides realistic mock interview scenarios with:
- Typical interviewer questions and follow-ups
- Expected discussion points at each stage
- Common mistakes to avoid
- Evaluation criteria

<Note>
**Interview Format**: Most system design interviews are 45-60 minutes. Aim to spend:
- 5 min: Requirements clarification
- 5 min: High-level design
- 25-30 min: Deep dive
- 5-10 min: Wrap-up and questions
</Note>

## Mock Interview 1: Design TinyURL

### Setup

```
Interviewer: "Let's design a URL shortening service like TinyURL 
or bit.ly. Users should be able to create short URLs and be 
redirected to the original URL."
```

### Phase 1: Requirements (5 min)

**Your questions should include:**

```
✓ "What's the expected scale? How many URLs per day?"
  → "Let's say 100M new URLs per day, 10B redirects"

✓ "What's the expected URL length?"
  → "As short as possible, let's say 7 characters"

✓ "Should URLs expire?"
  → "Yes, default 1 year, customizable"

✓ "Do we need analytics?"
  → "Yes, basic click counts and geographic data"

✓ "Custom short URLs?"
  → "Nice to have, but focus on auto-generated first"
```

### Phase 2: Capacity Estimation (3 min)

```
Show your math:

Write: 100M URLs/day
     = 100M / 86,400 ≈ 1,200 URLs/sec

Read: 10B redirects/day
    = 10B / 86,400 ≈ 115,000 redirects/sec
    → 100:1 read-heavy

Storage (5 years):
    100M × 365 × 5 = 182.5B URLs
    Each URL: ~500 bytes average
    Total: 182.5B × 500B = ~90 TB

Bandwidth:
    Write: 1,200 × 500B = 600 KB/s
    Read: 115,000 × 500B = 58 MB/s
```

### Phase 3: High-Level Design (5 min)

```
┌─────────────────────────────────────────────────────────┐
│                                                         │
│   ┌────────┐      ┌────────────┐      ┌────────────┐  │
│   │ Client │─────►│    API     │─────►│   Cache    │  │
│   └────────┘      │  Gateway   │      │  (Redis)   │  │
│                   └─────┬──────┘      └────────────┘  │
│                         │                    │         │
│           ┌─────────────┼─────────────┐     │         │
│           │             │             │     │         │
│      ┌────▼────┐  ┌────▼────┐  ┌────▼────┐ │         │
│      │  URL    │  │ Redirect│  │Analytics│  │         │
│      │ Creator │  │ Service │  │ Service │  │         │
│      └────┬────┘  └────┬────┘  └────┬────┘  │         │
│           │            │            │        │         │
│           └────────────┼────────────┘        │         │
│                        │                     │         │
│                  ┌─────▼─────┐               │         │
│                  │  Database │◄──────────────┘         │
│                  │  (Sharded)│                         │
│                  └───────────┘                         │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

### Phase 4: Deep Dive Questions

**Q: "How do you generate the short URL?"**

```
Approach 1: Base62 encoding of auto-increment ID
- IDs: 1, 2, 3, ...
- Encode: 62^7 = 3.5 trillion combinations
- Pros: Simple, no collisions
- Cons: Predictable, single point (ID generator)

Approach 2: Random generation + collision check
- Generate random 7 chars
- Check database
- Retry on collision
- Cons: Extra DB call, collision probability increases

Approach 3: Pre-generated key service
- Background service generates unused keys
- Store in key pool
- URL service fetches from pool
- Pros: No collision check at write time
- Cons: Extra service, key management

My recommendation: Approach 3 for this scale
- Handle 1,200 writes/sec easily
- No collision overhead
- Keys distributed across instances
```

**Q: "How do you handle the 115K reads/second?"**

```
1. Caching Strategy:
   - Redis/Memcached cluster
   - LRU eviction
   - Cache size: 20% of daily active = 20M entries
   - Expected hit rate: 90%+

2. Cache key: short_url
   Cache value: {long_url, created_at, expires_at}
   TTL: 24 hours (refresh on access)

3. Read path:
   cache_hit → redirect
   cache_miss → DB lookup → cache write → redirect
```

**Q: "How do you shard the database?"**

```
Shard by short URL hash:
- Consistent hashing with virtual nodes
- shard = hash(short_url) % num_shards
- Each shard handles ~1/N of traffic

Why not by user_id?
- Redirect requests don't have user context
- Need to lookup by short URL

Replication:
- 3 replicas per shard
- Async replication (eventual consistency OK)
- Promotes replica on primary failure
```

**Q: "How do you handle expiration?"**

```
Options:
1. TTL in database (native support)
2. Background cleanup job
3. Lazy deletion (check on read)

Hybrid approach:
- Check expiration on read (immediate)
- Background job for cleanup (storage)
- Tombstone records for analytics
```

### Evaluation Criteria

| Aspect | Junior | Senior | Staff |
|--------|--------|--------|-------|
| Requirements | Asks basic questions ("how many users?") | Identifies edge cases ("what about URL expiration?") | Challenges assumptions ("is 100M writes/day realistic?") |
| Scale | Basic math (gets QPS right) | Accurate estimates with peak factors and multi-year projections | Cost optimization ("Cassandra on 12 nodes vs DynamoDB on-demand at this scale") |
| Design | Working solution for the happy path | Handles failures, caching layers, read/write separation | Extensible architecture anticipating future requirements |
| Trade-offs | Mentions one when prompted | Proactively compares options with reasoning | Quantifies impact ("302 redirects cost 40% more server traffic but give 100% analytics accuracy") |
| Communication | Explains what they are doing | Explains why they chose each option | Checks alignment, adapts depth to interviewer signals |

## Mock Interview 2: Design Twitter Timeline

### Setup

```
Interviewer: "Let's design the home timeline feature for Twitter.
Users should see tweets from people they follow, in chronological
order or ranked by relevance."
```

### Requirements Clarification

```
✓ "How many users? Active users?"
  → "300M monthly active, 150M daily active"

✓ "Average follows per user?"
  → "Average 200, some celebrities with millions"

✓ "Tweets per user per day?"
  → "Average 2 tweets/day"

✓ "Timeline page size?"
  → "20 tweets per page"

✓ "Real-time updates needed?"
  → "Yes, new tweets should appear within seconds"

✓ "Chronological or ranked?"
  → "Support both, ranked is default"
```

### Capacity Estimation

```
Users: 150M DAU
Follows: 200 average
Tweets: 2/day/user

Write load:
    150M × 2 = 300M tweets/day
    = 3,500 tweets/second

Timeline reads:
    150M users × 10 timeline refreshes/day
    = 1.5B timeline reads/day
    = 17,000 timeline reads/second

Fan-out consideration:
    If user has 1M followers:
    1 tweet → 1M timeline updates
    100 tweets from celebs/day → 100M updates
```

### Key Design Decisions

**Q: "Fan-out on write vs fan-out on read?"**

```
Fan-out on Write:
- When user tweets, push to all followers' timelines
- Pre-compute timelines
- Fast reads (just fetch from cache)
- Problem: Celebrities with 50M followers

Fan-out on Read:
- When user opens timeline, fetch from followees
- Compute on demand
- Slow for users following many people
- Works well for celebrities

Hybrid Approach (Twitter's solution):
- Regular users: Fan-out on write
- Celebrities (>10K followers): Fan-out on read
- At read time: Merge cached timeline + celebrity tweets
- Best of both worlds
```

**Q: "How do you rank the timeline?"**

```
Ranking Signals:
├── Recency (newer = higher)
├── Engagement (likes, retweets, replies)
├── User affinity (interaction history)
├── Content type (media, links, text)
├── Author factors (verified, follower count)
└── Negative signals (muted words, reported)

Implementation:
1. Feature extraction at write time
2. Store features with tweet
3. ML model scores at read time
4. Sort by score
5. A/B test ranking changes
```

**Q: "How do you handle celebrities?"**

```
Celebrity Detection:
- Follower count > 10,000 (configurable)
- Flag in user table

Write Path for Celebrities:
- Don't fan out at write time
- Tweet goes to celebrity tweet store only

Read Path Merge:
1. Fetch user's cached timeline (fan-out on write)
2. Fetch celebrity tweets (users they follow)
3. Merge and rank
4. Return top N

Cache Strategy:
- Cache merged timeline briefly (1 minute)
- Invalidate on new celebrity tweet
```

### Deep Dive: Geospatial Indexing

```
Option 1: Geohash
- Encode lat/lng to string: "9q8yy"
- Prefix search for nearby: "9q8y*"
- Simple, works with regular indexes
- Edge case: nearby but different prefix

Option 2: QuadTree
- Recursively divide space into quadrants
- Dynamic depth based on density
- Efficient range queries
- Better for non-uniform distribution

Option 3: H3 (Uber's solution)
- Hexagonal hierarchical grid
- Multiple resolutions
- Consistent neighbor relationships
- Handles polar regions well

Implementation:
- H3 resolution 9: ~100m² hexagons
- Store driver_id → h3_cell mapping
- For pickup at H3 cell X:
  - Find drivers in cell X
  - Expand to ring-1 neighbors if needed
  - Sort by ETA, filter by availability
```

### Matching Algorithm

```python
def match_rider_to_driver(rider_location, ride_type):
    """
    Find best driver for rider request.
    """
    # 1. Get nearby drivers
    rider_cell = h3.geo_to_h3(
        rider_location.lat, 
        rider_location.lng, 
        resolution=9
    )
    
    search_cells = [rider_cell] + h3.k_ring(rider_cell, k=3)
    
    candidates = []
    for cell in search_cells:
        drivers = driver_index.get_drivers(cell)
        candidates.extend(drivers)
    
    # 2. Filter by availability and vehicle type
    available = [d for d in candidates 
                 if d.status == 'available' 
                 and d.vehicle_type == ride_type]
    
    # 3. Calculate ETA for each
    for driver in available:
        driver.eta = routing_service.get_eta(
            driver.location, 
            rider_location
        )
    
    # 4. Score and rank
    scored = []
    for driver in available:
        score = calculate_match_score(
            driver, 
            rider_location,
            eta=driver.eta,
            driver_rating=driver.rating,
            acceptance_rate=driver.acceptance_rate
        )
        scored.append((driver, score))
    
    scored.sort(key=lambda x: x[1], reverse=True)
    
    # 5. Dispatch to top candidate
    best_driver = scored[0][0]
    dispatch_ride(best_driver, rider_location)
    
    return best_driver
```

## Common Mistakes to Avoid

<Warning>
**Red Flags in System Design Interviews:**

1. **Jumping to solution** without understanding requirements -- this is the #1 reason candidates fail. Even if you know the "standard" answer for Twitter, ask clarifying questions. The interviewer may have a specific angle they want you to explore.

2. **Single point of failure** in design -- every box you draw on the whiteboard should have a redundancy plan. If you have a single Redis instance, the interviewer will ask "what happens when it goes down?" Have the answer ready before they ask.

3. **Ignoring scale** implications -- "we'll use a database" is not a design decision. At 100 QPS, PostgreSQL on a single machine is fine. At 100K QPS, you need sharding, read replicas, and caching. Know where the thresholds are.

4. **Over-engineering** for simple problems -- proposing Kafka, Kubernetes, and microservices for a URL shortener is a red flag. It signals that you cannot calibrate solutions to problems. Start with the simplest thing that works, then add complexity only when you can articulate the specific bottleneck it solves.

5. **Not discussing trade-offs** between options -- saying "I'll use Cassandra" is weak. Saying "I'll use Cassandra for its write throughput and partition tolerance, accepting that we lose JOIN capability and strong consistency across partitions" is strong. The trade-off discussion IS the interview.

6. **Forgetting about failures** and edge cases -- systems fail. Networks partition. Disks fill up. Clocks drift. If your design only works on a sunny day, it's not production-ready.

7. **Vague hand-waving** instead of concrete solutions -- "we'll just cache it" is not a caching strategy. What's the cache key? What's the TTL? What happens on cache miss? What's the invalidation strategy? Be specific.

8. **Not asking clarifying questions** -- an ambiguous requirement you assumed incorrectly costs you 10 minutes of wasted design. A 30-second clarifying question prevents this entirely.
</Warning>

## Interview Checklist

```
Before the interview:
□ Review system design patterns (fan-out, CQRS, saga,
  event sourcing, consistent hashing, circuit breaker)
□ Practice estimation out loud -- not in your head.
  "100M DAU x 10 actions / 100K seconds = 10K QPS"
  should roll off your tongue.
□ Know CAP theorem and consistency models -- and more
  importantly, know when each matters. Payment systems
  need CP. Social feeds can be AP.
□ Understand caching, databases, message queues -- not
  just what they are, but when to use each and why.
□ Practice with a timer. 45 minutes goes fast.

During the interview:
□ Clarify requirements (functional + non-functional)
  Aim for 3-5 sharp questions in the first 3 minutes.
□ Estimate scale and capacity -- say your assumptions
  out loud: "I'll round 86,400 to 100K for easy math"
□ Draw high-level design -- label every box and arrow.
  Use nouns for services, verbs for data flows.
□ Pick 2-3 components to go deep on -- you cannot cover
  everything in 45 minutes, so depth beats breadth.
□ Identify and address bottlenecks proactively -- don't
  wait for the interviewer to find them.
□ Discuss trade-offs for every major decision -- "I chose
  X over Y because Z, accepting the cost of W."
□ Consider failure scenarios -- "If this cache goes down,
  we fall back to the database with degraded latency."
□ Mention monitoring and alerting -- "We'd track P99
  latency, error rate, and cache hit ratio."

Communication:
□ Think out loud -- narrate your reasoning process.
□ Explain your reasoning -- the "why" matters more than
  the "what."
□ Acknowledge uncertainty -- "I'm not sure about the
  exact threshold, but my intuition says..." is better
  than guessing with false confidence.
□ Ask for feedback -- "Does this direction make sense,
  or would you like me to explore a different aspect?"
□ Be open to hints -- interviewers guide you toward what
  they want to hear. If they ask "what about failures?"
  they are hinting that you missed something. Take the
  hint gracefully.
□ Pause and check alignment -- every 10 minutes, say
  "Let me make sure we're on the same page" and
  summarize your design so far.
```

## What Separates Good From Great Answers

```
Junior answer:
  "I'll use Redis for caching."

Mid-level answer:
  "I'll use Redis for caching with a 5-minute TTL,
  using cache-aside pattern. On cache miss, we read
  from the database and backfill the cache."

Senior answer:
  "I'll use Redis for caching with a 5-minute TTL.
  Cache-aside pattern. On miss, we read from the
  read replica to avoid loading the primary. Cache
  key is the short_url, value is the original_url.

  For invalidation, I'll use TTL-based expiry rather
  than active invalidation because our data is
  write-once and essentially immutable.

  The 80/20 rule applies here -- 20% of URLs get 80%
  of traffic. So our cache needs to hold about 20% of
  active URLs. At 100M active URLs, that's 20M entries
  x 600 bytes = 12GB, which fits in a single Redis
  instance but I'd use a 3-node cluster for
  availability.

  If Redis goes down, we still serve from the database
  -- latency increases from 1ms to 10ms but the service
  stays up. We'd alert on cache hit rate dropping below
  90% as an early warning."

Notice the progression: the senior answer includes the
sizing math, the failure mode, the monitoring trigger,
and the justification for every choice. This is what
interviewers mean by "shows depth."
```

### Scenario 1: "Design a Chat Application Like WhatsApp"

**Interviewer:** "Design a messaging application that supports one-on-one and group chats, with message delivery guarantees and read receipts. Focus on the real-time messaging infrastructure."

<Accordion title="Phase 1: Requirements Clarification (5 min)">

**Candidate should ask:**

"Before I dive in, let me clarify the scope and constraints."

- "How many users are we designing for?" -- "2 billion registered, 500 million DAU."
- "What's the average message volume?" -- "50 messages sent per user per day."
- "Do we need to support media -- images, videos, voice messages?" -- "Yes, but focus on text first, then we can discuss media."
- "What are the delivery guarantees? Do messages need to arrive in order?" -- "Yes. Messages within a conversation must be ordered. Delivery should be at-least-once."
- "Do we need end-to-end encryption?" -- "Acknowledge it exists but don't design the crypto. Focus on the messaging infra."
- "Group chat size limit?" -- "Up to 1,024 members."
- "Offline message delivery?" -- "Yes, messages must be delivered when the user comes back online."

**What the interviewer is scoring here:** Did the candidate identify the hardest constraints (ordering, offline delivery, group fan-out) or just ask surface questions about features?

</Accordion>

<Accordion title="Phase 2: Capacity Estimation (3 min)">

**Strong candidate's math:**

```
Message volume:
  500M DAU x 50 messages = 25 billion messages/day
  QPS = 25B / 86,400 ≈ 25B / 100K = 250,000 messages/sec
  Peak QPS (3x) = 750,000 messages/sec

Storage:
  Average message: 200 bytes (text + metadata)
  Daily: 25B x 200 bytes = 5 TB/day
  Yearly: 5 TB x 365 = 1.8 PB/year
  With 3x replication: 5.4 PB/year

Connections:
  500M DAU, ~20% online concurrently = 100M WebSocket connections
  At 50K connections per server = 2,000 WebSocket servers
```

"The key insight here is this is a connection-heavy system, not a compute-heavy one. The bottleneck is maintaining 100 million persistent connections, not processing the messages themselves."

</Accordion>

<Accordion title="Phase 3: High-Level Design (5 min)">

**Strong answer with diagram:**

```
Client (WebSocket) --> Gateway Servers (connection management)
                          |
                    Message Service
                     /          \
              Message Queue      Presence Service
              (Kafka)            (Redis)
                  |
            Message Store        Group Service
            (Cassandra)          (metadata)
```

"Let me walk through the write path. User A sends a message to User B:
1. A's device sends the message over its WebSocket connection to a Gateway server.
2. The Gateway forwards it to the Message Service.
3. The Message Service writes the message to Cassandra (durable storage) and publishes it to Kafka.
4. A consumer checks if User B is online via the Presence Service (Redis).
5. If online: route the message to User B's Gateway server, which pushes it down B's WebSocket.
6. If offline: the message stays in Cassandra. When B reconnects, a sync process delivers all pending messages.
7. Once B's device acknowledges receipt, we update the delivery status."

</Accordion>

<Accordion title="Phase 4: Deep Dive -- Interviewer Probes">

**Interviewer:** "How do you guarantee message ordering within a conversation?"

**Strong answer:**
"Each message gets a monotonically increasing sequence number per conversation, not a global sequence. When User A sends messages 1, 2, 3 in a conversation with User B, those sequence numbers are assigned by the Message Service before writing to Cassandra.

I would use Cassandra with the partition key as `conversation_id` and clustering key as `sequence_number`. This means all messages in a conversation are stored on the same partition in order. Reads are a simple range query: `SELECT * FROM messages WHERE conversation_id = X AND sequence_number > last_seen ORDER BY sequence_number ASC`.

For conflict resolution -- if two users send a message at the exact same millisecond -- the Message Service serializes writes per conversation. I would use a consistent hash of `conversation_id` to route all messages for a conversation to the same Kafka partition, guaranteeing ordering at the consumer."

**Red flag answer:** "I would use timestamps for ordering." Timestamps are unreliable across distributed systems. Clock skew between servers means you can receive messages out of order. This is a fundamental distributed systems mistake.

---

**Interviewer:** "How do you handle group messages to a group of 1,000 members?"

**Strong answer:**
"This is essentially a fan-out problem. When a message is sent to a group:
1. Write the message once to the group's message store (partition key = `group_id`).
2. For delivery, I do NOT copy the message 1,000 times. Instead, I maintain a per-member read pointer (last sequence number read) for each group.
3. When a member opens the group chat, they fetch messages from the group's store with `sequence_number > their_read_pointer`.
4. For real-time push: iterate through the group's member list, check which members are online, and push a notification to their Gateway servers.

At 1,000 members, the fan-out for push notifications is bounded and fast. The key optimization is: do not store per-member copies, only store one copy per group and track read pointers."

**Red flag answer:** "I would send the message to all 1,000 members individually, like 1,000 separate one-on-one messages." This is a storage and write amplification disaster. A group with 1,000 members and 100 messages/day would generate 100,000 message copies per day.

---

**Interviewer:** "What happens when a user has been offline for 3 days and comes back online with 5,000 unread messages across 200 conversations?"

**Strong answer:**
"This is the sync problem. I would NOT push all 5,000 messages immediately -- that would overwhelm the client and the WebSocket connection.

Instead:
1. On reconnect, the client sends its `last_sync_timestamp` to the server.
2. The server returns a lightweight manifest: a list of conversations with unread counts and the latest message preview.
3. The client displays this immediately (fast initial load).
4. As the user taps into each conversation, we lazy-load the full messages for that conversation using cursor-based pagination.
5. Priority: conversations with recent activity sync first.

This is exactly how WhatsApp and Signal handle it. The initial sync is fast (just metadata), and full message content is loaded on demand."

</Accordion>

### Scenario 2: "Design a Distributed Task Scheduler Like Cron at Scale"

**Interviewer:** "Design a distributed task scheduler that can handle millions of scheduled jobs -- think of it as cron but for a cloud platform. Jobs need to execute at their scheduled time with at-least-once guarantee."

<Accordion title="Phase 1: Requirements Clarification (5 min)">

**Candidate should ask:**

- "How many jobs are we scheduling?" -- "100 million active scheduled jobs."
- "What is the timing precision required?" -- "Within 5 seconds of the scheduled time."
- "What types of schedules? One-time, recurring, cron expressions?" -- "All three."
- "What does 'execute' mean -- do we run the code, or do we trigger an HTTP callback?" -- "HTTP callback to user-defined endpoints."
- "What happens if a job execution fails?" -- "Retry up to 3 times with exponential backoff. Then mark as failed and notify."
- "Do we need job prioritization?" -- "Yes, premium customers get priority."
- "Consistency requirement -- is it acceptable to execute a job twice but never skip one?" -- "Yes, at-least-once is fine. Users should make their handlers idempotent."

</Accordion>

<Accordion title="Phase 2: High-Level Design">

**Strong answer:**

"I would split this into three subsystems:

**1. Job Store (the source of truth):**
- A sharded database (Postgres or Cassandra) storing all job definitions: schedule, callback URL, retry policy, status.
- Indexed by `next_execution_time` so we can efficiently query 'what needs to run in the next minute.'

**2. Scheduler (the brain):**
- Partitions the timeline into slots (each slot = 1 second or 1 minute).
- A set of Scheduler workers, each responsible for a range of time slots (assigned via consistent hashing or a coordinator).
- Every second, each Scheduler worker queries: 'give me all jobs where `next_execution_time` is in my assigned slot and `status = PENDING`.'
- Moves those jobs to the Execution Queue with `status = QUEUED`.

**3. Executor (the muscle):**
- Worker pool consuming from the Execution Queue (Kafka or SQS).
- Each worker: picks a job, makes the HTTP callback, handles success/failure.
- On success: update job status, compute `next_execution_time` for recurring jobs.
- On failure: retry with backoff, eventually move to dead-letter queue.

```
Job Store (Sharded Postgres)
       |
  Scheduler Workers (partition time slots)
       |
  Execution Queue (Kafka)
       |
  Executor Workers (HTTP callbacks)
```

The critical design choice is separating scheduling from execution. The Scheduler only decides WHAT runs WHEN. The Executor only runs jobs. This means you can scale execution independently of scheduling."

</Accordion>

<Accordion title="Phase 3: Deep Dive -- The Hard Problems">

**Interviewer:** "How do you guarantee that a job is not skipped if a Scheduler worker crashes?"

**Strong answer:**
"This is the core reliability challenge. I use a lease-based approach:

1. Each Scheduler worker acquires a lease on its time partition via a distributed lock (ZooKeeper or DynamoDB lease table).
2. The lease has a TTL of 30 seconds. The worker must heartbeat every 10 seconds to renew.
3. If a worker crashes, its lease expires after 30 seconds.
4. A standby worker detects the expired lease and takes over the partition.
5. The new worker queries all jobs in that time partition with `status = PENDING` and `next_execution_time < now` -- these are the jobs the crashed worker missed.

The worst-case delay is 30 seconds (lease TTL). Since our precision requirement is 5 seconds, I would set the lease TTL to 10 seconds with a 3-second heartbeat, accepting slightly higher ZooKeeper load for tighter failover."

**Red flag answer:** "I would use a single scheduler with a hot standby." This is a single point of failure masquerading as redundancy. A single scheduler cannot handle 100M jobs, and failover latency is unpredictable.

---

**Interviewer:** "100 million jobs, and you are querying by `next_execution_time` every second. How is that not destroying your database?"

**Strong answer:**
"Great question -- a naive approach would absolutely kill the database. Here is how I handle it:

1. **Two-tier scheduling:** The database is the cold store. Every minute, the Scheduler pulls all jobs due in the NEXT minute into a hot in-memory priority queue (a min-heap keyed by execution time). Within that minute, the in-memory queue fires jobs at the right second.
2. **Partition by time bucket:** The Job Store is partitioned by `next_execution_time` rounded to the hour. So querying 'what runs in the next minute' only scans one partition, not the entire table.
3. **Batch reads:** Pull jobs in batches of 10,000, not one at a time.

At 100M jobs, if they are uniformly distributed across the day, that is ~70K jobs per minute. Loading 70K rows per minute from a partitioned database is trivial -- about 1,200 rows/sec."

</Accordion>

### Scenario 3: "Design Google Docs -- Real-Time Collaborative Editing"

**Interviewer:** "Design a system where multiple users can edit the same document simultaneously, seeing each other's changes in real-time. Focus on the conflict resolution and consistency model."

<Accordion title="Phase 1: Requirements and Key Challenges">

**Candidate should identify:**

- "How many concurrent editors per document?" -- "Up to 100."
- "What's the acceptable latency for seeing another user's edit?" -- "Under 500ms."
- "Do we need offline editing with sync when reconnected?" -- "Yes, that is important."
- "What types of content -- plain text, rich text, images, tables?" -- "Start with rich text, we can discuss structured content later."
- "What is the scale? How many documents, how many concurrent editing sessions?" -- "1 billion documents, 10 million concurrent editing sessions."

**The core technical challenge (state this explicitly):**
"The fundamental problem in collaborative editing is conflict resolution. When User A and User B both edit the same paragraph simultaneously, we need a deterministic way to merge their changes that produces the same result regardless of the order changes arrive. This is where OT or CRDTs come in."

</Accordion>

<Accordion title="Phase 2: Conflict Resolution Deep Dive">

**Strong answer:**

"There are two established approaches: Operational Transformation (OT) and Conflict-free Replicated Data Types (CRDTs).

**Operational Transformation (what Google Docs uses):**
- Each edit is an 'operation': `insert(position, char)` or `delete(position)`.
- When two concurrent operations conflict, a transformation function adjusts one operation relative to the other.
- Example: User A inserts 'X' at position 5, User B inserts 'Y' at position 3. When A's operation arrives at B's client, position 5 must be adjusted to 6 because B's insert shifted everything after position 3.
- Requires a central server to establish a canonical operation order. The server is the source of truth for transformation.
- Pro: Well-understood, proven at Google scale. Con: Central server is a bottleneck and single point of failure per document.

**CRDTs (what Figma and some newer systems use):**
- Each character in the document gets a unique, globally ordered ID (e.g., a Lamport timestamp + client ID).
- Insertions create new IDs between existing IDs using fractional indexing or tree-based approaches.
- Merging is automatic -- just union the character sets and sort by ID.
- Pro: No central server needed, works offline natively. Con: Metadata overhead per character (each character carries its ID), tombstone accumulation for deletes.

**My recommendation for this design:** OT with a per-document coordination server because:
1. We have at most 100 concurrent editors per document -- a single server per document handles this easily.
2. Google has proven OT works at billion-document scale.
3. The central server simplifies cursor tracking, presence, and undo/redo.

**Architecture:**
```
Clients (WebSocket) --> Document Session Service (1 per active doc)
                              |
                        Operation Log (append-only)
                              |
                        Document Snapshot Store (periodic)
```

Each active document gets assigned a Session Server. All edits for that document go through this server, which applies OT transformations and broadcasts the resolved operations to all connected clients."

</Accordion>

<Accordion title="Phase 3: Interviewer Probes">

**Interviewer:** "What happens when the Document Session Server crashes mid-edit?"

**Strong answer:**
"Every operation is persisted to the Operation Log (Kafka or a durable append-only store) before being acknowledged to the client. The Operation Log is the source of truth, not the Session Server's in-memory state.

When a Session Server crashes:
1. The client detects the WebSocket disconnect.
2. The client reconnects to a load balancer, which assigns a new Session Server for that document.
3. The new Session Server rebuilds state by: loading the last snapshot from the Snapshot Store, then replaying all operations from the Operation Log since that snapshot.
4. Clients re-sync by sending their local operation counter -- the server sends any operations they missed.

To keep recovery fast, we take snapshots every 100 operations. So the worst-case replay is 100 operations, which takes milliseconds."

---

**Interviewer:** "How do you handle a user who has been editing offline for 2 hours and then reconnects?"

**Strong answer:**
"This is the hardest case for OT because we potentially need to transform 2 hours' worth of offline edits against 2 hours' worth of server edits.

My approach:
1. The offline client queues all local operations with their base revision number.
2. On reconnect, the client sends its base revision and all queued operations.
3. The server computes the diff: all server operations between the client's base revision and the current revision.
4. The server transforms the client's operations against all server operations (this can be expensive with thousands of ops on each side).
5. The transformed operations are applied and broadcast.

**Optimization:** If the offline edits and server edits affected completely different parts of the document (different paragraphs), the transformation is trivial -- no conflicts. We can short-circuit by checking affected ranges first.

**Fallback:** If the transformation fails (theoretically possible with very complex concurrent edits), we show the user a diff view: 'Your changes conflict with changes made while you were offline. Please review.' This is the same approach Google Docs uses for severe conflicts -- graceful degradation, not silent data loss."

**Red flag answer:** "We would just merge the changes." This hand-waves over the entire hard problem. WHAT merge strategy? HOW do you handle position shifts? This answer gives the interviewer nothing to score.

</Accordion>

### Scenario 4: "Design a Content Delivery Network (CDN)"

**Interviewer:** "Design a CDN that serves static assets -- images, JavaScript, CSS, video segments -- to users globally with low latency. Focus on the caching and routing layers."

<Accordion title="Phase 1: Requirements and Estimation">

**Candidate's clarifying questions:**

- "What is the total content volume?" -- "50 PB of content across all customers."
- "How many requests per second globally?" -- "10 million requests/second at peak."
- "What is the target cache hit ratio?" -- "Above 95% for popular content."
- "Geographic coverage?" -- "Global, with emphasis on North America, Europe, and Asia."
- "Do we need to support purge/invalidation?" -- "Yes, within 30 seconds globally."

**Estimation:**
```
10M requests/sec, average response 100 KB
Bandwidth: 10M x 100 KB = 1 TB/s = 8 Tbps
Across 200 edge locations: 50K req/sec per location, 40 Gbps per location
```

"At 8 Tbps total, we need approximately 200 edge Points of Presence (PoPs), each serving around 40 Gbps. This aligns with what Cloudflare and AWS CloudFront operate."

</Accordion>

<Accordion title="Phase 2: Architecture">

**Strong answer:**

"A CDN has three tiers:

**Tier 1 -- Edge PoPs (200+ locations globally):**
- Closest to users. Handles the majority of requests from cache.
- Each PoP has 10-50 servers with NVMe SSDs (high IOPS for random reads) and significant RAM for hot-object caching.
- Cache hierarchy within each PoP: RAM (hot objects, ~1TB) --> SSD (warm objects, ~50TB) --> origin fetch (cache miss).
- Uses consistent hashing within the PoP so each server is responsible for a subset of URLs. This maximizes per-server cache hit ratio.

**Tier 2 -- Regional Shields (10-20 locations):**
- Aggregation layer between edges and origin.
- When an edge PoP has a cache miss, it does NOT go directly to the customer's origin server. It goes to the regional shield first.
- The shield absorbs the cache miss storm that would otherwise hammer the origin during a cold start or after a purge.
- This is critical: without a shield tier, a popular asset purge causes 200 PoPs to simultaneously request the asset from origin. With a shield, only 10-20 shield servers request it.

**Tier 3 -- Origin:**
- The customer's actual servers or their cloud storage (S3, GCS).
- The CDN only contacts origin on a complete cache miss through all tiers.

**Request routing (how a user reaches the nearest PoP):**
1. **DNS-based routing:** The CDN's authoritative DNS server returns the IP of the nearest PoP based on the resolver's IP (imprecise but simple). This is what most CDNs use.
2. **Anycast:** Advertise the same IP from all PoPs via BGP. The internet's routing protocols send the user to the nearest PoP. Cloudflare uses this.
3. **HTTP redirect:** Less common, adds a round-trip.

I would use Anycast for the primary routing with DNS-based failover. Anycast handles PoP failures automatically -- if a PoP goes down, BGP withdraws the route and traffic shifts to the next-nearest PoP within seconds."

</Accordion>

<Accordion title="Phase 3: Deep Dive -- Cache Invalidation">

**Interviewer:** "A customer deploys a new version of their website and needs all cached assets purged globally within 30 seconds. How?"

**Strong answer:**
"Cache invalidation is the hard problem. Here is my approach:

**Fast-path purge (for urgent invalidations):**
1. Customer calls the Purge API with a URL or URL pattern.
2. The Purge Service publishes the invalidation to a global message bus (Kafka with replication across regions, or a purpose-built gossip protocol).
3. Every edge PoP subscribes to the invalidation stream. On receiving a purge message, it immediately marks the object as stale in its cache.
4. The next request for that URL triggers a fetch from the shield/origin. Meanwhile, we can optionally serve the stale object with a background refresh (stale-while-revalidate).

**Why 30 seconds is achievable:**
- Kafka cross-region replication with 3 regions: ~200ms.
- Processing the purge message at each PoP: ~10ms.
- The bottleneck is fan-out to 200 PoPs, but with a pub/sub model, this is parallel.
- Total: under 5 seconds for propagation, with monitoring alerting if any PoP has not acknowledged within 30 seconds.

**Cache versioning (alternative approach):**
- Instead of purging, the customer changes the URL (e.g., `app.v2.js` or `app.js?v=abc123`). The old URL naturally expires via TTL. The new URL is a cache miss that fetches fresh content.
- This avoids the purge mechanism entirely and is what most production systems prefer for deployable assets.
- Purge is reserved for 'oops, we published the wrong content' emergency scenarios."

**Red flag answer:** "I would set a short TTL on all cached objects." Short TTLs destroy the cache hit ratio. A 60-second TTL means every object is refetched from origin 1,440 times a day even if it never changes. The whole point of a CDN is to avoid that.

</Accordion>

### Scenario 5: "Design a Search Autocomplete System"

**Interviewer:** "Design the autocomplete/typeahead feature for a search engine. As the user types, suggestions should appear within 100ms."

<Accordion title="Phase 1: Requirements and Scale">

**Candidate's questions:**

- "How many unique queries do we need to support?" -- "5 billion unique query strings."
- "How many autocomplete requests per second?" -- "100K requests/second at peak (every keystroke triggers a request)."
- "How are suggestions ranked?" -- "By popularity (search frequency), with personalization as a stretch goal."
- "Do we need to handle typos?" -- "Basic fuzzy matching, yes."
- "How fresh do the suggestions need to be?" -- "Trending queries should appear within 15 minutes."

</Accordion>

<Accordion title="Phase 2: Core Data Structure">

**Strong answer:**

"The classic data structure for prefix-based autocomplete is a **Trie (prefix tree)**, but at 5 billion strings, a naive trie would consume hundreds of GB of RAM. Let me walk through the practical approach.

**Compressed Trie with top-K at each node:**
- Each node stores its character and a pre-computed list of the top 10-15 suggestions for that prefix.
- When the user types 'app', we traverse to the 'a' -> 'p' -> 'p' node and return its pre-computed top-K list. No further traversal needed.
- This makes read latency O(length of query), which at an average of 15 characters is ~15 pointer dereferences -- sub-microsecond.

**Why not just use a database with `LIKE 'app%'`?**
- At 5 billion strings and 100K QPS, a SQL `LIKE` query would require a range scan even with a B-tree index. That is 10-50ms per query, too slow for the 100ms end-to-end budget (which includes network round-trip).
- A trie serves this in microseconds from memory.

**Memory optimization:**
- 5 billion strings at ~20 bytes average = 100 GB raw. With a trie, many strings share prefixes, so actual trie size is ~30-50 GB.
- Still too large for a single server. Partition by first 2 characters: 'aa' through 'zz' gives 676 partitions across ~50 servers (~1 GB per partition).
- Each server holds its partition fully in memory.

**Ranking updates (keeping suggestions fresh):**
- A MapReduce job or streaming pipeline (Flink) processes search logs every 15 minutes.
- Computes query frequency counts and updates the top-K lists at each trie node.
- The update is applied atomically: build a new trie, swap the pointer. No locking needed (readers on old trie finish naturally).

**Handling trending queries:** Maintain a separate 'trending' trie that uses a time-decayed frequency (recent searches weighted more). Merge trending results with the all-time popular results at query time. This is how Twitter and Google surface trending autocomplete suggestions quickly."

</Accordion>

<Accordion title="Phase 3: Interviewer Probes">

**Interviewer:** "The user types 'twt' meaning 'twitter'. Your trie has no path for 'twt'. How do you handle this?"

**Strong answer:**
"This is the fuzzy matching problem. Several approaches, in order of complexity:

1. **Edit distance pre-computation:** For the top 1 million most popular queries, pre-compute all strings within edit distance 1 (single character deletion, insertion, substitution). Store these as aliases in the trie. 'twiter', 'twttier', 'twtter' all point to 'twitter'. At 1M queries with ~50 edit-distance-1 variants each, this adds 50M trie entries -- manageable.

2. **Phonetic matching (Soundex/Metaphone):** Convert both the query and stored strings to phonetic representations. 'twt' and 'twitter' have similar phonetic fingerprints. Useful for name searches.

3. **Character n-gram index:** Break queries into character trigrams. 'twitter' becomes `twi, wit, itt, tte, ter`. 'twt' matches on `twt` -- no overlap. But 'twittr' would match on `twi, wit, itt, ttr` -- high overlap with 'twitter'. This approach works well for longer fuzzy queries but poorly for very short ones.

For a search engine autocomplete, I would use approach 1 (edit distance pre-computation) for the top queries combined with a fallback to approach 3 for tail queries. This gives fast fuzzy matching for common misspellings without blowing up memory."

**Red flag answer:** "I would use Elasticsearch fuzzy queries." This is not wrong as a technology choice, but it avoids the actual design question. The interviewer wants to hear you reason about data structures and trade-offs, not name a product.

---

**Interviewer:** "100K requests/second, and every keystroke triggers a request. That is a lot of wasted work for queries the user never finishes typing. How do you reduce load?"

**Strong answer:**
"Several techniques, and in practice you use all of them:

1. **Client-side debouncing:** Wait 100-200ms after the last keystroke before sending the request. If the user is typing quickly, you skip intermediate keystrokes. This alone can reduce traffic by 50-70%.

2. **Client-side caching:** If the user typed 'app' and got suggestions, then types 'appl', the client can filter the 'app' results locally without a server round-trip if the server returned enough candidates (say 50). Only hit the server if the local filter yields fewer than 10 results.

3. **Request cancellation:** If a new keystroke arrives before the previous request returns, cancel the in-flight request. The user does not need results for 'ap' if they have already typed 'app'.

4. **Server-side caching:** Popular prefixes like 'fac', 'goo', 'ama' are queried thousands of times per second. Cache the results in Redis or even at the CDN edge. TTL of 5-15 minutes is fine since the trie is only updated every 15 minutes anyway.

With all four techniques, the actual load hitting the trie servers drops from 100K QPS to perhaps 20-30K QPS."

</Accordion>

### Scenario 6: "Design a Distributed Locking Service Like ZooKeeper"

**Interviewer:** "Design a distributed lock service that provides mutual exclusion guarantees. Multiple services need to coordinate access to shared resources -- for example, ensuring only one instance processes a given job at a time."

<Accordion title="Phase 1: Requirements Clarification">

**Candidate should ask:**

- "How many lock acquisitions per second?" -- "50,000 per second across the cluster."
- "What is the maximum acceptable latency for acquiring a lock?" -- "Under 10ms for uncontested locks."
- "What happens if the lock holder crashes? Should the lock be automatically released?" -- "Yes, via TTL/lease. Cannot hold a lock forever."
- "Do we need read-write locks or just exclusive locks?" -- "Start with exclusive, discuss read-write as extension."
- "What are the consistency requirements?" -- "Linearizable. If a lock is granted, it must be globally visible. Two processes must never believe they hold the same lock simultaneously."
- "Fault tolerance -- how many node failures should we survive?" -- "Survive any minority of nodes failing. Standard quorum."

"This is fundamentally a consensus problem. The core challenge is: in a distributed system with network partitions and node failures, how do you guarantee that at most one client holds a lock at any time?"

</Accordion>

<Accordion title="Phase 2: Architecture and Consensus">

**Strong answer:**

"I would build this on top of a replicated state machine using the Raft consensus protocol (or Paxos, but Raft is easier to reason about and implement correctly).

**Architecture:**
```
Clients --> Lock Service API
              |
         Raft Cluster (5 nodes)
         [Leader] [Follower] [Follower] [Follower] [Follower]
              |
         Persistent Log (WAL on each node)
```

**How lock acquisition works:**
1. Client sends `ACQUIRE(lock_name, client_id, lease_duration)` to the Lock Service.
2. The request is routed to the Raft leader.
3. The leader proposes a log entry: 'Client X holds lock Y until time T'.
4. Once a majority of nodes (3 out of 5) acknowledge the log entry, it is committed.
5. The leader responds to the client with a 'lock acquired' confirmation and a lease token.
6. The client must renew the lease before time T, or the lock is automatically released.

**Why Raft and not just Redis `SETNX`:**
Redis `SETNX` with TTL (the Redlock algorithm) has known safety issues. Martin Kleppmann published a detailed analysis showing that Redlock can violate mutual exclusion during clock skew or GC pauses. Specifically: Client A acquires the lock, then has a long GC pause. The TTL expires, Client B acquires the lock. Client A's GC pause ends, and now both clients believe they hold the lock.

Raft avoids this because:
- Lock state is replicated across a majority of nodes.
- The lease is enforced by the consensus group, not by client-side timeout.
- A fencing token (monotonically increasing) is issued with each lock grant. Even if a stale client tries to use an old lock, the fencing token lets downstream services reject the request.

**Fencing tokens (the critical detail most candidates miss):**
- Every lock grant includes a fencing token: a monotonically increasing integer.
- When Client A gets the lock, it receives fencing token 33.
- Client A passes token 33 to the shared resource (e.g., a database write with `WHERE fencing_token >= 33`).
- If Client A's lock expires and Client B gets token 34, and then Client A's delayed request arrives with token 33, the resource rejects it because 33 < 34.
- This is the only way to guarantee safety in the presence of process pauses and network delays."

</Accordion>

<Accordion title="Phase 3: Deep Dive -- The Hard Problems">

**Interviewer:** "What happens during a network partition? Say the 5-node cluster splits into a group of 3 and a group of 2."

**Strong answer:**
"This is exactly what Raft is designed for.

**If the leader is in the group of 3:**
- Business as usual. The leader can still reach a majority (itself + 2 others = 3 out of 5). New lock requests are processed normally.
- The 2 isolated followers cannot serve reads or writes. Clients connected to them get errors and must reconnect to the majority partition.

**If the leader is in the group of 2:**
- The leader can no longer reach a majority. Its writes will time out because it cannot get 3 acknowledgments.
- The group of 3 will elect a new leader (after the election timeout, typically 150-300ms).
- The new leader serves all lock requests. Any locks that the old leader granted but did not replicate to the majority are NOT committed and will be rolled back.
- Clients holding locks granted by the old leader will fail to renew their leases (since the old leader can no longer commit renewals). Their locks expire, and the new leader can grant those locks to other clients.

**The key safety guarantee:** At no point do two clients hold the same lock, because the lock grant requires majority consensus, and there can be at most one majority in any partition."

**Red flag answer:** "We would make both partitions read-only until the partition heals." This is overly conservative and shows a lack of understanding of quorum-based systems. The majority partition should continue operating normally -- that is the whole point of running an odd number of nodes.

---

**Interviewer:** "50K lock acquisitions per second across a 5-node Raft cluster. Is that realistic?"

**Strong answer:**
"Honestly, a single Raft group at 50K ops/sec is pushing it. ZooKeeper (which uses ZAB, similar to Raft) tops out at around 10-20K writes/sec because every write requires disk fsync on a majority of nodes.

To handle 50K/sec, I would use **lock partitioning:**
- Hash the lock name to one of N lock groups (say 100).
- Each lock group is its own independent Raft cluster (or a partition within a multi-raft system like TiKV or CockroachDB).
- Locks in different groups are completely independent -- no cross-group coordination needed.
- Each Raft group handles ~500 ops/sec, well within limits.

This is exactly how etcd (used by Kubernetes) and CockroachDB scale -- they shard the keyspace across multiple Raft groups, each running independent consensus.

The trade-off: more Raft groups means more nodes to manage, more leader elections, and no atomic operations across groups. If a client needs to acquire two locks in different groups atomically, you need a two-phase commit protocol on top. But for the common case of single-lock acquisition, this scales linearly."

</Accordion>
