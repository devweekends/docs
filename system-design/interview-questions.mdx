## Overview

This is a curated list of system design interview questions organized by difficulty and topic. Use this to practice and assess your readiness. Each sample question includes deep-dive answers, follow-up chains, red flag indicators, and what the interviewer is actually probing for beneath the surface.

<Tip>
**Interview Strategy**: Start with medium difficulty questions. If you can design 3-4 different systems confidently, you're ready for most interviews. The key insight: system design questions recycle the same 8-10 core patterns (caching, sharding, pub/sub, fan-out, consistent hashing, rate limiting, event sourcing, CQRS). Once you internalize these patterns, every new question is just a recombination of familiar building blocks.
</Tip>

## Sample Interview Questions with Deep-Dive Answers

### Question 1: Design a URL Shortener

<Accordion title="Requirements to clarify">
- How many URLs per day? (100M writes, 10B reads -- 100:1 read-to-write ratio)
- How long should URLs be valid? (configurable, default forever)
- Custom short URLs allowed? (yes, but as a premium feature)
- Analytics needed? (click counts, geographic distribution, referrer tracking)
- What is the expected total number of URLs over the system lifetime? (100B over 5 years = ~6-7 character short codes in Base62)
- Latency requirements? (redirect in under 50ms at p99)
</Accordion>

<Accordion title="Key design points">
1. **Encoding**: Base62 encoding (`a-z`, `A-Z`, `0-9`) gives 62^7 = 3.5 trillion possible URLs with 7 characters. This is enough for decades of traffic. Avoid Base64 because `+` and `/` cause URL encoding issues.
2. **ID Generation**: Two viable approaches. **Counter-based**: Use a distributed counter (like a Snowflake-style ID generator or a pre-allocated range per app server), then Base62 encode. Guarantees uniqueness without collision checks. **Hash-based**: MD5/SHA256 the long URL, take first 7 characters. Simpler, but requires collision handling -- on collision, append a salt and re-hash or try the next hash segment.
3. **Storage**: Write the `shortCode -> longURL` mapping to a persistent NoSQL store (DynamoDB or Cassandra). The access pattern is pure key-value lookup, so relational features are wasted. At 100B records with ~500 bytes per record, that is about 50TB of storage.
4. **Caching**: Put a Redis cluster in front of the DB for hot URLs. The 80/20 rule applies heavily here: 20% of URLs receive 80% of traffic. A Redis cluster with 256GB can cache ~500M mappings, which covers virtually all hot URLs. Set TTL to 24-48 hours.
5. **Read path**: `Client -> Load Balancer -> App Server -> Redis (hit?) -> return 301/302`. On cache miss, read from DB, populate cache, then redirect. Use 301 (permanent) for SEO-friendly redirects, 302 (temporary) if you need analytics on every click.
6. **Write path**: `Client -> LB -> App Server -> generate shortCode -> write to DB -> return shortURL`. For custom aliases, check DB for conflict first.
7. **Database choice**: DynamoDB with `shortCode` as the partition key. At 100M writes/day (~1,200 writes/sec average, 5K peak), DynamoDB handles this without breaking a sweat. For even cheaper options, Cassandra on commodity hardware.
</Accordion>

<Accordion title="What interviewers are really testing">
This question seems simple, but interviewers use it to test whether you can methodically work through a problem without overengineering. They want to see: can you calculate scale (back-of-envelope math), make reasonable database choices with justification, explain hash collision handling, and discuss caching strategy. The hidden trap is candidates who jump straight to microservices and Kafka for what is essentially a key-value lookup service.
</Accordion>

<Accordion title="Red flag answers">
- "I would use MySQL with auto-increment IDs" -- shows no awareness of how sequential IDs are predictable and how SQL becomes a bottleneck at this write volume without sharding.
- "Just hash the URL with MD5" without mentioning collision handling -- in a real system with billions of URLs, collisions are guaranteed, not theoretical.
- Immediately drawing 15 microservices for a fundamentally simple system -- signals inability to match complexity to requirements.
- Not discussing the read-to-write ratio at all -- the 100:1 ratio is the single most important architectural driver here.
</Accordion>

<Accordion title="Follow-up questions">
1. **"How do you handle hash collisions at scale?"** -- A strong candidate explains bloom filters for fast collision checking, or a counter-based approach that avoids collisions entirely. A great candidate mentions that Twitter's URL shortener (t.co) uses a counter-based approach for exactly this reason.
2. **"Your Redis cache just went down. What happens to latency?"** -- Tests whether they designed the system to degrade gracefully. Expect discussion of fallback to DB, impact on p99 latency (jumps from 5ms to 50-100ms), and whether they have a second cache tier or can tolerate the temporary latency spike.
3. **"A single URL is getting 1M clicks per second (it went viral on Twitter). How do you handle the hot key?"** -- Tests hot partition awareness. Strong answers mention local in-memory caching at the app server layer, replicating the hot key across multiple Redis shards, or CDN-level caching for the redirect.
</Accordion>

### Question 2: Design Twitter Timeline

<Accordion title="Requirements to clarify">
- Active users? (300M DAU, 1B total registered)
- Tweet rate? (500M tweets/day, ~6K tweets/sec average, 50K peak)
- Follow limits? (5,000 following per user)
- Celebrity accounts? (top 1% have millions of followers -- Lady Gaga has 80M+)
- Timeline freshness? (users expect tweets within 5 seconds of posting)
- Features in scope? Likes, retweets, replies, media, or just the core timeline?
</Accordion>

<Accordion title="Key design points">
1. **Fan-out on write (push model)**: When a user tweets, immediately push that tweet ID into the timeline cache (Redis sorted set) of every follower. For a user with 1,000 followers, that is 1,000 Redis writes. At 6K tweets/sec with average 200 followers, that is 1.2M Redis writes/sec. Manageable with a Redis cluster.
2. **Fan-out on read (pull model)**: When a user opens their timeline, query all accounts they follow, fetch recent tweets, merge, rank, and return. For a user following 500 accounts, this means 500 DB lookups. Too slow for every timeline load, but fine for edge cases.
3. **Hybrid approach (what Twitter actually does)**: Use fan-out on write for 99% of users. For celebrity accounts with 10M+ followers, do NOT fan-out on write (that would be 10M Redis writes per tweet). Instead, at read time, merge the pre-computed timeline with fresh tweets from followed celebrities. The user's app server fetches the cached timeline (fast) and celebrity tweets (a handful of queries) then merges them in memory.
4. **Caching layer**: Redis sorted sets keyed by `userId`, scored by timestamp. Store the last 800 tweet IDs per user. Cache hit rate should exceed 99% since users mostly view recent tweets. Total memory: 300M users * 800 tweet IDs * 8 bytes = ~1.9TB Redis, spread across a cluster.
5. **Ranking**: Chronological as the base, with engagement signals (likes, retweets, reply count) as a secondary ranking factor. The ranking model runs at read time, re-ordering the cached tweet IDs before fetching full tweet objects.
6. **Tweet storage**: Tweets table in a sharded MySQL/Vitess cluster, sharded by `tweetId`. Store `tweetId`, `authorId`, `content`, `timestamp`, `mediaUrls`. Separate `user_timeline` table is not needed -- the Redis cache IS the timeline.
7. **Media handling**: Images/videos uploaded to S3, transcoded asynchronously. Tweet object stores only the S3 URL. CDN serves media.
</Accordion>

<Accordion title="What interviewers are really testing">
This is the canonical fan-out question. The interviewer wants to see if you understand the fundamental tradeoff between write amplification (fan-out on write) and read amplification (fan-out on read). The celebrity problem is the real test -- it separates candidates who memorized a diagram from those who understand why the hybrid approach exists. They also want to see you do back-of-envelope math on the fan-out volume to justify the hybrid cutoff threshold.
</Accordion>

<Accordion title="Red flag answers">
- "Fan-out on write for everyone" without addressing the celebrity problem -- means they have not thought about the write amplification of pushing one tweet to 80M timelines (that is 80M Redis writes for a single tweet, taking minutes to propagate).
- "Just use a message queue" without explaining what the consumer does -- shows cargo-cult architecture.
- No discussion of data model or storage -- jumping to architecture boxes without explaining what data lives where.
- "Use GraphQL" or any specific API technology as if it solves the system design problem -- confuses interface with architecture.
</Accordion>

<Accordion title="Follow-up questions">
1. **"How do you decide the cutoff between push and pull? Is it 10K followers? 1M?"** -- A strong candidate explains this is tunable and depends on write throughput capacity. At Twitter's scale, the threshold was around 10K-50K followers. You measure the marginal cost of one more fan-out write vs. one more read-time merge, and set the threshold where the economics cross over.
2. **"A user follows 3,000 accounts and opens the app. Walk me through the exact sequence of operations to render their timeline."** -- Tests whether the candidate can trace a request end-to-end: app server receives request, fetches pre-computed timeline from Redis (the 800 cached tweet IDs), fetches fresh celebrity tweets from a separate query, merges and ranks, fetches full tweet objects for the top 20, hydrates user profiles and media URLs, returns the response. Under 200ms total.
3. **"What happens when a tweet goes viral and gets 500K retweets in an hour? How does this affect timeline delivery?"** -- Tests understanding of cascading fan-out. Each retweet triggers its own fan-out to that user's followers. The system needs backpressure mechanisms: queue depth monitoring, shedding fan-outs for inactive users, prioritizing online users.
</Accordion>

### Question 3: Design WhatsApp

<Accordion title="Requirements to clarify">
- Users? (2B registered, 500M DAU)
- Message rate? (100B messages/day, ~1.15M messages/sec average)
- Group size? (1,024 members max, but 99% of groups have under 50 members)
- Message types? (text, image, video, voice notes, documents)
- End-to-end encryption required? (yes, this affects architecture)
- Delivery guarantees? (at-least-once delivery, exactly-once display using deduplication at the client)
- Message retention? (on-device storage, server only holds until delivered)
</Accordion>

<Accordion title="Key design points">
1. **Connection layer**: WebSocket connections for real-time bidirectional communication. Each user maintains a persistent WebSocket to a gateway server. At 500M DAU, that is 500M concurrent connections. Each gateway server handles ~500K connections (using epoll/kqueue), so you need ~1,000 gateway servers. Use consistent hashing to route a user to a specific gateway.
2. **Message flow (1:1)**: `Sender -> Gateway A -> Message Service -> Message Queue -> Gateway B -> Recipient`. If the recipient is online, the message is delivered directly through their WebSocket connection. If offline, the message is persisted to a message store (Cassandra or HBase) and delivered when the user reconnects. The server sends back an ACK to the sender upon persistence (the single checkmark).
3. **Message flow (group)**: Sender sends to the group service. The group service looks up all group members, then fans out individual messages to each member through their respective gateway servers. For a 50-person group, this is 49 individual deliveries. For groups with 1,000 members, batch the fan-out and process asynchronously.
4. **Delivery status**: Three states modeled as separate ACK messages. **Sent** (single gray check): server acknowledged receipt. **Delivered** (double gray check): recipient device acknowledged receipt. **Read** (double blue check): recipient app reported the message was displayed on screen. Each status update is itself a small message flowing back through the system.
5. **Offline handling**: Messages queue in a per-user inbox in Cassandra, keyed by `(recipientId, timestamp)`. When the user reconnects, the gateway pulls all pending messages from the inbox and delivers them in order. After delivery confirmation, messages are deleted from the server (WhatsApp's privacy model: messages are transient on the server).
6. **End-to-end encryption**: Signal Protocol. The server never sees plaintext. Key exchange happens during connection setup. This means the server cannot index, search, or moderate message content -- an important architectural constraint.
7. **Media handling**: Large files (images, videos) are uploaded to a media service (S3-backed), encrypted client-side before upload. The message itself contains only a URL + decryption key. The recipient downloads and decrypts locally. This keeps the message channel lightweight.
8. **Presence system**: Heartbeat every 30 seconds from each client. "Last seen" timestamp stored in Redis. "Online" status computed by checking if the last heartbeat was within the last 60 seconds. Presence updates are pushed to contacts currently viewing the chat.
</Accordion>

<Accordion title="What interviewers are really testing">
The core challenge here is connection management at massive scale -- 500M concurrent WebSocket connections is an infrastructure problem most candidates have never faced. The interviewer wants to see how you handle connection routing (how does a message find the right gateway server for the recipient), offline message queuing, and the nuances of delivery guarantees (at-least-once delivery with client-side deduplication). Bonus points for discussing encryption constraints on architecture.
</Accordion>

<Accordion title="Red flag answers">
- "Use HTTP polling" -- fundamentally wrong for a real-time chat system at this scale. Polling 500M users every second is 500M HTTP requests/sec for presence alone.
- "Store all messages in MySQL" -- shows no understanding of the write volume. 1.15M messages/sec into a relational database is not viable without extreme sharding, and Cassandra/HBase are purpose-built for this workload.
- Not mentioning offline message delivery -- a chat system that only works when both users are online is not a chat system.
- "End-to-end encryption means the server encrypts messages" -- no, it means the server CANNOT decrypt messages. This is a fundamental distinction.
</Accordion>

<Accordion title="Follow-up questions">
1. **"User A is on Gateway Server 5, User B is on Gateway Server 12. How does a message from A reach B?"** -- Tests understanding of service discovery and connection routing. The message service needs a mapping of `userId -> gatewayServerId`, typically stored in Redis. When a message arrives for User B, the service looks up which gateway holds their connection and routes there. If User B disconnects and reconnects to Gateway Server 8, the mapping updates.
2. **"Gateway Server 12 crashes. 500K users lose their connections. What happens?"** -- Tests fault tolerance thinking. Users reconnect automatically (clients have retry with exponential backoff), land on new gateway servers, mapping updates, pending messages are delivered from the inbox queue. The key insight: no messages are lost because they were persisted before the ACK was sent to the sender.
3. **"How do you handle message ordering in a group chat when messages arrive from different senders through different gateway servers?"** -- Tests distributed ordering. Use a Lamport-style logical clock or rely on server-side timestamps assigned by the group service (single serialization point per group). Accept that exact ordering across concurrent messages is a "best effort" -- two messages sent within the same millisecond can appear in any order, and that is acceptable for chat.
</Accordion>

### Question 4: Design Netflix

<Accordion title="Requirements to clarify">
- Users? (200M paid subscribers, 100M concurrent at peak)
- Content library? (15,000+ titles, total ~200PB of video after transcoding all formats/qualities)
- Concurrent streams? (10M peak, each consuming 3-25 Mbps depending on quality)
- Quality levels? (480p, 720p, 1080p, 4K HDR, with adaptive bitrate switching)
- Global presence? (190+ countries, content licensing varies by region)
- Latency requirements? (play should start within 2 seconds of clicking, no buffering during playback)
</Accordion>

<Accordion title="Key design points">
1. **Video ingestion pipeline**: When a studio uploads a master file (often ProRes 4444, 50+ GB), it enters a transcoding pipeline. Netflix transcodes each title into ~1,200 different versions: combinations of resolution (480p to 4K), codec (H.264, H.265/HEVC, VP9, AV1), bitrate, and audio formats. This is a massive batch processing job running on thousands of EC2 instances. Each title takes 12-48 hours to fully encode. The output is stored in S3-compatible object storage.
2. **Adaptive bitrate streaming (ABR)**: Videos are split into 2-4 second chunks. Each chunk exists at multiple quality levels. The client player measures available bandwidth in real-time and requests the appropriate quality for the next chunk. Protocols: HLS (Apple devices) or MPEG-DASH (everything else). If your bandwidth drops from 20Mbps to 5Mbps mid-movie, the player seamlessly switches from 4K to 720p without interruption.
3. **CDN architecture (Open Connect)**: Netflix runs its own CDN called Open Connect with 17,000+ servers in 6,000+ locations worldwide. During off-peak hours (2-8 AM local time), Netflix pre-positions popular content onto ISP-embedded servers. When a user in Mumbai clicks play, the video streams from a server physically inside their ISP's data center, not from a Netflix data center in Virginia. This reduces internet backbone traffic by 90%+ and keeps latency under 20ms.
4. **Content discovery and recommendations**: The homepage is entirely personalized. Netflix uses ~1,000 ML models for personalization: "what to show", "how to rank", "which thumbnail to display" (yes, different users see different thumbnails for the same movie). The recommendation service runs on a real-time feature store (data from viewing history, search, browse, time of day, device type) and serves results in under 100ms.
5. **Search**: Elasticsearch cluster for title search, with fuzzy matching, synonym handling, and multi-language support. Also handles browse by genre, actor, director. The search index is much smaller than the recommendation system since it only covers metadata, not viewing behavior.
6. **Playback service**: When a user clicks play, the client calls the playback service which returns a manifest file (list of chunk URLs at each quality level). The manifest URLs point to the nearest CDN node. The playback service also handles DRM license delivery (Widevine for Android/Chrome, FairPlay for Apple, PlayReady for Windows/Xbox).
7. **Architecture philosophy**: Netflix uses a microservices architecture (~1,000 services), runs on AWS (except Open Connect which is custom hardware), and embraces chaos engineering (Chaos Monkey, Chaos Kong) to ensure resilience. The control plane (user accounts, billing, recommendations) runs on AWS. The data plane (actual video bytes) runs on Open Connect.
</Accordion>

<Accordion title="What interviewers are really testing">
This question separates engineers who think about "serving video files" from those who understand content delivery as a global infrastructure problem. The key insight is that the hard problem is not streaming video (that is well-understood protocol work), but getting the right bytes to the right edge location before the user even clicks play. Interviewers want to see CDN strategy, transcoding pipeline design, and adaptive bitrate understanding. Senior candidates should discuss the tension between storage cost (1,200 versions per title * 15K titles) and user experience.
</Accordion>

<Accordion title="Red flag answers">
- "Just use a CDN like CloudFront" -- shows no understanding that at Netflix's scale, they operate their own CDN because commercial CDNs would cost billions per year and give less control over performance.
- "Store videos in the database" -- a fundamental misunderstanding of object storage vs. databases.
- No mention of adaptive bitrate -- serving a single quality level means either wasting bandwidth for users on slow connections or providing terrible quality for users on fast connections.
- "Transcode on the fly when the user requests" -- real-time transcoding of 4K video is computationally expensive and would add 30+ seconds of latency. Pre-transcoding is mandatory.
</Accordion>

<Accordion title="Follow-up questions">
1. **"Netflix releases a new season of Stranger Things and 50M users try to watch episode 1 simultaneously. How does the system handle this?"** -- Tests understanding of predictive content placement. Netflix knows release schedules and pre-positions content on edge servers days in advance. The Open Connect servers are sized for peak capacity. Even so, a major release can saturate ISP links -- Netflix's traffic engineering team coordinates with ISPs beforehand.
2. **"How does Netflix decide which 1,200 encoding combinations to generate for each title? Why not just 10?"** -- Tests understanding of per-title encoding optimization. Netflix's "per-title encode" system analyzes each title's visual complexity (animation needs fewer bits than live action) and generates a custom bitrate ladder. A visually simple animated show might look great at 1080p/2Mbps, while a dark action scene needs 1080p/8Mbps. The 1,200 versions cover device/codec/quality/audio combinations -- not arbitrary.
3. **"A user on a train goes through a tunnel and loses connectivity for 30 seconds. What happens to their stream?"** -- Tests client-side buffering strategy. The player maintains a buffer of 30-120 seconds of pre-fetched chunks. During the tunnel, it plays from the buffer. When connectivity returns, it aggressively fills the buffer at lower quality first (fast fill), then switches back to high quality once the buffer is healthy. If the tunnel lasts longer than the buffer, playback pauses and resumes from the last chunk.
</Accordion>

### Question 5: Design Uber

<Accordion title="Requirements to clarify">
- Users? (100M riders, 5M active drivers)
- Rides per day? (20M completed rides)
- Location update frequency? (every 4 seconds from drivers, so 5M * 0.25 Hz = 1.25M location updates/sec)
- Matching criteria? (distance, ETA, driver rating, vehicle type, surge pricing zone)
- Geographic coverage? (10,000+ cities globally, each with different regulations)
- Latency for matching? (rider should see a driver match within 10-15 seconds of requesting)
</Accordion>

<Accordion title="Key design points">
1. **Location tracking and indexing**: Drivers send GPS coordinates every 4 seconds. These updates flow to a location service that maintains a real-time spatial index. Two viable approaches: **GeoHash** (encode lat/lng into a string prefix, use Redis sorted sets -- `GEOADD` and `GEORADIUS` commands) or **QuadTree** (in-memory tree structure that recursively subdivides geographic space). Uber actually uses Google S2 cells (a hierarchical spatial index based on a Hilbert curve). At 1.25M updates/sec, you need a sharded spatial index -- partition by geographic region (city or S2 cell level).
2. **Matching algorithm**: When a rider requests a ride, the system queries the spatial index for available drivers within a radius (start with 2km, expand if no matches). For each candidate driver, compute a score based on: straight-line distance, estimated ETA (using routing service), driver rating, driver acceptance rate, and vehicle type match. Sort by score, send the request to the top-ranked driver. If they decline or timeout (15 seconds), try the next driver.
3. **Trip state machine**: Every ride follows a state machine: `REQUESTED -> MATCHED -> DRIVER_EN_ROUTE -> ARRIVED -> TRIP_IN_PROGRESS -> COMPLETED` (or `CANCELLED` from several states). Each state transition triggers events: notifications, billing calculations, ETA updates. The trip service owns this state machine and persists it in a database (PostgreSQL with strong consistency -- you do not want eventual consistency for billing-critical state).
4. **Dynamic pricing (surge)**: Divide the city into hexagonal zones (H3 cells). For each zone, track the supply/demand ratio in real-time (number of available drivers vs. number of pending ride requests). When demand exceeds supply by a threshold, apply a surge multiplier (1.5x, 2x, etc.). This is computed every 30 seconds by a streaming pipeline (Kafka Streams or Flink) consuming ride request and driver availability events.
5. **Real-time communication**: WebSocket connections for both rider and driver apps. The rider sees the driver's location updating in real-time on the map. The driver sees the pickup location and turn-by-turn navigation. All location updates flow through the WebSocket connections, not polling.
6. **Dispatch service architecture**: The dispatch service is the brain of the system. It receives ride requests, queries the location service, runs the matching algorithm, and manages the offer flow (send to driver, wait for accept/decline, retry). This service must be highly available and low-latency. It is sharded by city -- a ride in NYC is processed by a completely independent set of servers from a ride in London. This provides natural fault isolation.
7. **Payment and billing**: After trip completion, the billing service calculates the fare (base fare + time + distance + surge multiplier - promotions). Payment is processed asynchronously. The rider sees the final amount immediately (estimated during the trip), but actual charge may adjust for tolls or route corrections. Uses idempotency keys to prevent double charges.
</Accordion>

<Accordion title="What interviewers are really testing">
The Uber question tests spatial indexing (a topic many candidates have never implemented), real-time systems design, and complex state management. The hidden depth is in the matching algorithm -- it is not just "find the nearest driver" but a multi-factor optimization problem with time pressure (the rider is waiting). Senior candidates should discuss the geographic sharding strategy, surge pricing as a distributed streaming computation, and how the dispatch service handles failures (what happens if a driver's phone dies mid-match).
</Accordion>

<Accordion title="Red flag answers">
- "Store driver locations in MySQL and query with `WHERE lat BETWEEN x AND y`" -- this is O(n) on the full table. Spatial indexing exists for a reason. At 5M drivers, you cannot scan the table on every ride request.
- "Match the rider with the closest driver" without discussing ETA -- the closest driver by straight-line distance might be across a river with a 20-minute detour. ETA (using the road network) is the correct metric.
- No mention of what happens when the matched driver declines -- the retry logic, timeout handling, and expanding search radius are critical to user experience.
- Ignoring the geographic sharding aspect -- a system that processes all rides through a single cluster will collapse under the latency of cross-continent requests.
</Accordion>

<Accordion title="Follow-up questions">
1. **"It is New Year's Eve in Manhattan. There are 50,000 ride requests in a 10-minute window and only 5,000 available drivers. Walk me through what happens."** -- Tests understanding of surge pricing, queue management, and graceful degradation. The surge multiplier shoots up (3-5x), discouraging marginal demand. Ride requests queue with a timeout. The dispatch service prioritizes based on rider wait time and willingness to pay surge. Some riders cancel and take the subway. The system should NOT crash -- it should degrade gracefully by increasing prices and wait times.
2. **"Driver A accepts the ride but their phone loses connectivity in a tunnel. The rider is waiting. What happens?"** -- Tests fault tolerance in stateful flows. The system has a heartbeat mechanism. If no heartbeat for 30 seconds, the trip moves to `DRIVER_UNREACHABLE` state. After 60 seconds, the trip is reassigned to another driver. The original driver, upon reconnecting, sees the trip was reassigned. The rider is notified of the driver switch with an updated ETA.
3. **"How do you prevent drivers from gaming the surge pricing system -- like coordinating to go offline in a zone to trigger surge, then coming back online?"** -- Tests awareness of adversarial behavior in marketplace systems. Detection: analyze patterns of coordinated offline/online behavior using ML anomaly detection. Prevention: surge calculations use a rolling average, not instantaneous supply. Smoothing with a 5-minute window makes it impossible to trigger instant surge by briefly going offline.
</Accordion>

### Question 6: Design a Rate Limiter

<Accordion title="Requirements to clarify">
- Where does the rate limiter sit? (API gateway level, per-service, or both)
- Rate limiting granularity? (per user, per IP, per API key, per endpoint)
- Rate limits? (e.g., 100 requests/minute per user, 1000 requests/minute per IP)
- Distributed or single-node? (distributed -- multiple API gateway instances need a shared view)
- What happens when rate limited? (return HTTP 429 with `Retry-After` header)
- Hard limit or soft limit? (hard: reject immediately; soft: allow some burst with degraded priority)
</Accordion>

<Accordion title="Key design points">
1. **Token Bucket algorithm**: The most commonly used in production (used by AWS, Stripe, and most API gateways). Each user has a bucket with capacity `N` tokens. Tokens are added at a fixed rate (e.g., 10 tokens/second). Each request consumes one token. If the bucket is empty, the request is rejected. Advantages: allows bursts (up to bucket capacity) while enforcing an average rate. Implementation: store `(lastRefillTimestamp, tokenCount)` per user in Redis. On each request, calculate how many tokens to add since last refill, subtract one, and check if `tokenCount >= 0`.
2. **Sliding Window Log**: Store the timestamp of every request in a sorted set. On each new request, remove timestamps older than the window, count remaining, and compare to the limit. Precise but memory-intensive -- for a 1-minute window with 100 req/min limit, you store up to 100 timestamps per user. At 10M users, that is 1B entries in the worst case. Use Redis sorted sets (`ZADD`, `ZREMRANGEBYSCORE`, `ZCARD`).
3. **Sliding Window Counter** (hybrid): Combine the current window count and the previous window count, weighted by the overlap. For example, if the current window is 30% elapsed: `count = previousWindowCount * 0.7 + currentWindowCount`. Approximation, but memory-efficient (only two counters per user per window) and accurate within 1-2% in practice. This is what most production systems use.
4. **Redis as the backing store**: All three algorithms need shared state across API gateway instances. Redis is the standard choice: single-digit millisecond latency, atomic operations (`INCR`, `EXPIRE`), and built-in TTL for automatic cleanup. Use Redis Cluster for availability. Set up Redis in the same availability zone as your API gateways to minimize latency overhead (1-2ms per rate limit check).
5. **Failure mode**: If Redis goes down, you have two options: **fail open** (allow all requests -- risky, but keeps the service running) or **fail closed** (reject all requests -- safe, but causes an outage). Most systems fail open with local in-memory rate limiting as a fallback (less accurate since each instance tracks independently, but prevents total outage).
6. **Response format**: Return HTTP 429 with headers: `X-RateLimit-Limit: 100`, `X-RateLimit-Remaining: 0`, `X-RateLimit-Reset: 1620000060` (Unix timestamp), `Retry-After: 30` (seconds). These headers let well-behaved clients back off gracefully.
7. **Multi-tier rate limiting**: Production systems often apply multiple limits simultaneously: 100 requests/minute per user AND 1000 requests/minute per IP AND 10,000 requests/minute per API key. The request is rejected if ANY limit is exceeded. Implement as multiple independent rate limiters checked in sequence.
</Accordion>

<Accordion title="What interviewers are really testing">
This question tests algorithmic understanding (can you explain multiple rate limiting algorithms and their tradeoffs), distributed systems awareness (how do you share state across instances), and production judgment (what happens when the rate limiter itself fails). The real differentiator is whether the candidate discusses the distributed aspect -- a single-node rate limiter is a trivial data structure exercise, but a distributed one across 50 API gateway instances is a real systems problem.
</Accordion>

<Accordion title="Red flag answers">
- Only knowing one algorithm and not being able to compare tradeoffs -- if you can only describe token bucket, you are reciting, not understanding.
- "Store rate limit counters in the application database" -- this adds 5-50ms of latency to every API call. Rate limiting must be sub-millisecond or low single-digit milliseconds.
- Not addressing the distributed case -- "just use a counter in memory" ignores that you have 50 instances of the API gateway.
- No discussion of what happens when the rate limiter infrastructure fails -- in production, your rate limiter will eventually be the cause of an outage if you do not plan for its failure.
</Accordion>

<Accordion title="Follow-up questions">
1. **"Your rate limiter adds 2ms of latency to every request. Your p99 API latency is 50ms. Product says that is unacceptable. How do you reduce it?"** -- Tests optimization thinking. Options: move to local in-memory rate limiting (loses accuracy across instances), use a local cache with periodic sync to Redis, pipeline multiple Redis commands, or use a sidecar proxy (like Envoy) with a local rate limit filter that syncs asynchronously.
2. **"An attacker is distributing requests across 10,000 different IP addresses to bypass your per-IP rate limit. How do you handle this?"** -- Tests understanding of rate limit evasion. Solutions: rate limit by additional signals (user agent fingerprint, API key, geographic cluster), implement a global rate limit (total requests to a specific endpoint regardless of source), use behavioral analysis (ML-based anomaly detection on request patterns), or deploy a WAF with bot detection (Cloudflare, AWS WAF).
3. **"You need to rate limit a webhook delivery system. Third-party servers have different capacity -- some can handle 100 req/sec, others only 5. How do you design adaptive rate limiting?"** -- Tests creative application. Use an adaptive token bucket where the refill rate adjusts based on response codes. If you get 429s or 503s from the destination, reduce the rate. If requests succeed, gradually increase. This is essentially TCP congestion control applied to HTTP -- exponential backoff on failure, additive increase on success.
</Accordion>

### Question 7: Design a Distributed Cache

<Accordion title="Requirements to clarify">
- Scale? (10M requests/sec, 50TB total data, sub-millisecond p99 latency)
- Consistency model? (eventual consistency is acceptable for cache -- stale reads are tolerable within a TTL window)
- Eviction policy? (LRU, LFU, or TTL-based?)
- Availability requirements? (99.99% -- cache downtime causes cascading failures to the backing database)
- Client library or transparent proxy? (client-side consistent hashing vs. proxy-based routing)
- Data types? (simple key-value, or complex types like lists, sets, sorted sets?)
</Accordion>

<Accordion title="Key design points">
1. **Consistent hashing for data distribution**: Distribute keys across N cache nodes using consistent hashing. Each node owns a range of the hash ring. When a node is added or removed, only `1/N` of keys are remapped (compared to modular hashing where ALL keys remap). Use virtual nodes (150-200 per physical node) to ensure even distribution. Without virtual nodes, real-world hash functions produce 10-30% imbalance across nodes.
2. **Replication for availability**: Replicate each key to `k` successor nodes on the hash ring (typically k=2 or k=3). Writes go to the primary node and are asynchronously replicated. If the primary fails, reads and writes failover to the first replica. This is the Dynamo-style approach used by Redis Cluster and Memcached with replication.
3. **Eviction policies**: **LRU** (Least Recently Used) is the default for most caches -- evict the key that hasn't been accessed for the longest time. Implementation: doubly-linked list + hash map, O(1) for both get and put. **LFU** (Least Frequently Used) is better when access frequency matters more than recency (e.g., CDN caches). **TTL-based** -- every key has an expiration time; stale keys are lazily evicted on access or proactively by a background sweep.
4. **Cache-aside pattern** (most common): Application checks cache first. On miss, reads from DB, writes to cache, returns to client. The application owns the caching logic. **Write-through**: Every write goes to cache AND DB synchronously. Ensures cache is always consistent but adds write latency. **Write-behind** (write-back): Writes go to cache immediately, and asynchronously flush to DB. Lower write latency but risk of data loss if cache crashes before flush.
5. **Hot key mitigation**: A single viral cache key can overwhelm the node that owns it. Solutions: replicate hot keys to multiple nodes (client randomly picks one), use a local in-process cache (L1 cache) in front of the distributed cache (L2 cache), or shard within the key (e.g., `popular_tweet_12345_shard_0` through `_shard_9`, pick one randomly).
6. **Cache stampede prevention**: When a hot key expires, hundreds of concurrent requests see a cache miss and all hit the database simultaneously. Solutions: **Locking** -- only one request fetches from DB while others wait. **Probabilistic early expiration** -- each request that reads the key has a small probability of refreshing it before actual expiration (`currentTime + random() * beta * computeTime > expiry`). **Never-expire + background refresh** -- cache entries never expire, but a background process refreshes them periodically.
7. **Memory management**: Use a slab allocator (like Memcached) to avoid memory fragmentation. Pre-allocate memory chunks of fixed sizes (64B, 128B, 256B, ..., 1MB). Each key-value pair is stored in the smallest slab that fits. This eliminates external fragmentation at the cost of some internal fragmentation (~15-20% wasted space).
</Accordion>

<Accordion title="What interviewers are really testing">
This is a deep infrastructure question that tests whether the candidate has actually operated caches in production or just read about them. The interviewer wants consistent hashing explained correctly (with virtual nodes -- anyone who skips virtual nodes has only read the Wikipedia summary), a thoughtful eviction strategy discussion, and awareness of operational nightmares (cache stampede, hot keys, memory fragmentation). Staff-level candidates should discuss the tradeoffs between cache-aside, write-through, and write-behind patterns with concrete scenarios.
</Accordion>

<Accordion title="Red flag answers">
- "Use consistent hashing" without explaining virtual nodes or why -- consistent hashing without virtual nodes has severe imbalance problems in practice.
- "Just use Redis" -- this is an implementation choice, not a design. The question is asking you to design the cache system itself, including distribution, replication, and eviction.
- Not discussing what happens when a cache node fails -- if a node with 10% of your keys dies and all those requests hit the database, that is a thundering herd that can take down your DB.
- "Cache everything with no TTL" -- shows no understanding of memory constraints or staleness issues.
</Accordion>

<Accordion title="Follow-up questions">
1. **"Node 7 in your 20-node cache cluster just died. Walk me through exactly what happens over the next 60 seconds."** -- A strong answer covers: consistent hashing redirects Node 7's key range to the next node on the ring (which already has replicas if replication is configured), clients using the cache library detect the failure via health checks, cache misses spike for Node 7's keys (cold start), the database sees a load spike, and you should have provisioned enough DB headroom to absorb this. The failed node's key range is served by replicas within seconds, but those replicas now have double the load.
2. **"Your cache hit rate dropped from 95% to 60% overnight. The cache cluster is healthy, no nodes failed. What happened?"** -- Tests debugging intuition. Common causes: a deployment changed cache key format (invalidating all existing keys), TTLs were misconfigured in a new release, a new feature introduced a high-cardinality key space that evicted existing hot keys (e.g., per-session caching with millions of sessions), or the data access pattern fundamentally changed (new feature that queries different data).
3. **"How would you implement a multi-region cache with data that is written in US-East and read globally?"** -- Tests global architecture thinking. Options: independent cache per region with cache-aside from a replicated database (each region warms its own cache), or a primary cache in US-East with async replication to other regions (lower miss rate but replication lag means stale reads). The right answer depends on staleness tolerance -- for user profiles (stale by seconds is fine), async replication works. For financial data (balance must be current), read-through from the primary region.
</Accordion>

### Question 8: Design a Payment System

<Accordion title="Requirements to clarify">
- Transaction volume? (10M transactions/day, peak 1,000 TPS during flash sales)
- Payment methods? (credit cards, debit cards, bank transfers, digital wallets)
- Multi-currency? (yes, with real-time exchange rates)
- Regulatory requirements? (PCI-DSS compliance, regional regulations like PSD2 in EU)
- Idempotency guarantees? (critical -- a payment must never be processed twice)
- Settlement timeline? (real-time authorization, T+1 settlement with acquiring banks)
- Refund support? (full and partial refunds, chargeback handling)
</Accordion>

<Accordion title="Key design points">
1. **Idempotency as the foundational principle**: Every payment request includes a client-generated idempotency key (UUID). The server stores this key with the payment result. If the same key is received again (due to network retry, client retry, or duplicate submission), the server returns the stored result without reprocessing. This is implemented as a unique constraint on `idempotency_key` in the payments table. Without this, a network timeout followed by a client retry can charge a customer twice -- this is a $0-cost bug that becomes a million-dollar lawsuit.
2. **Payment flow (happy path)**: `Client -> API Gateway -> Payment Service -> (validate) -> Payment Processor (Stripe/Adyen) -> (authorize) -> Payment Service -> (persist) -> Return success`. The critical ordering: authorize with the processor FIRST, then persist the result. If you persist "payment initiated" and then the authorization fails, you have an inconsistent state. If you authorize first and the persist fails, the idempotency key ensures you can recover by retrying the persist.
3. **Double-entry ledger**: Every financial transaction creates TWO ledger entries: a debit and a credit. A $50 payment creates: `DEBIT: customer_account -$50` and `CREDIT: merchant_account +$50`. The sum of all ledger entries must always be zero. This is not optional -- it is a regulatory requirement and the only way to ensure your books balance. Use an append-only ledger table, never update or delete entries. Corrections are made by adding reversing entries.
4. **Reconciliation**: At the end of each day (T+1), run a reconciliation job that compares your internal ledger against the payment processor's settlement file. Mismatches (usually 0.01-0.1% of transactions) are flagged for manual review. Common causes: timeouts where you think the payment failed but the processor actually charged the card, or currency rounding differences. This job is critical -- if you do not reconcile daily, discrepancies compound and become unresolvable.
5. **Handling failures**: Payment processing has multiple failure modes. **Timeout**: You sent the authorization request but never got a response. Is the payment charged or not? You MUST query the processor's API with the idempotency key to check status before retrying. **Processor down**: Circuit breaker pattern -- after 3 consecutive failures, stop sending to that processor for 30 seconds and try a backup processor (multi-acquirer routing). **Partial failure**: The authorization succeeded but your database write failed. On retry, the idempotency check at the processor returns "already authorized" and you can safely persist.
6. **PCI-DSS compliance**: Never store raw card numbers (PAN) in your systems. Use tokenization: the client sends card details directly to the payment processor (via their SDK/iframe), receives a token, and sends the token to your backend. Your servers never see the actual card number. This reduces your PCI scope from SAQ-D (hundreds of security controls) to SAQ-A (minimal controls). The cost difference in compliance audits alone is $100K+/year.
7. **Saga pattern for complex payments**: For a checkout with multiple steps (authorize card, reserve inventory, apply discount, create order), use a Saga: a sequence of local transactions with compensating actions. If the inventory reservation fails after the card was authorized, execute a compensating action (void the authorization). Each step is idempotent and has a corresponding compensation. Orchestrator-based sagas (a central coordinator) are easier to debug than choreography-based (event-driven) for payment flows.
</Accordion>

<Accordion title="What interviewers are really testing">
Payment systems are where "distributed systems theory" meets "you will lose real money if you get this wrong." The interviewer is testing whether you understand idempotency (not just the word, but how to implement it), double-entry bookkeeping (non-negotiable for financial systems), and failure handling (because payment processors WILL timeout, and the question "did the charge go through or not?" is the scariest question in payments). Staff-level candidates discuss reconciliation, PCI compliance, and multi-acquirer routing.
</Accordion>

<Accordion title="Red flag answers">
- "Retry the payment if it times out" -- without checking idempotency or querying the processor for status first. This is how you double-charge customers.
- "Store credit card numbers encrypted in our database" -- even encrypted, storing PANs puts you in PCI scope. The correct answer is tokenization -- never touch card numbers.
- Not mentioning the double-entry ledger -- a payment system that tracks balances as a single number (not debit/credit pairs) will have reconciliation nightmares.
- "Use eventual consistency for payments" -- strong consistency is non-negotiable for financial transactions. A user seeing "payment successful" that later rolls back to "failed" is unacceptable.
</Accordion>

<Accordion title="Follow-up questions">
1. **"Your payment processor returns a timeout after 30 seconds. The customer's card may or may not have been charged. What do you do?"** -- The only safe answer: query the processor's status API using your idempotency key. If the status is "authorized," persist and return success. If "not found," the charge did not go through -- retry. If the status API also times out, put the transaction in a "PENDING_REVIEW" state and resolve it via the reconciliation job. NEVER guess.
2. **"How do you handle a flash sale where 100K users try to pay for 1,000 items simultaneously?"** -- Tests understanding of contention. Use optimistic locking on inventory (decrement with a WHERE clause: `UPDATE inventory SET count = count - 1 WHERE item_id = X AND count > 0`). Payment authorization happens AFTER successful inventory reservation. The 99K users who fail the inventory check never even hit the payment processor, saving you processing fees and reducing contention.
3. **"A merchant reports that they received $50,000 in payments yesterday but your system only shows $48,000. How do you investigate?"** -- Tests reconciliation and debugging skills. Pull the settlement file from the processor, compare transaction IDs with your ledger, identify missing entries. Common causes: failed webhook deliveries (processor charged the card but the notification never reached your system), timezone mismatches in the reporting window, or currency conversion rounding. This is why daily reconciliation is not optional.
</Accordion>

### Question 9: Design a Web Crawler

<Accordion title="Requirements to clarify">
- Scale? (crawl 1B pages/day, that is ~11,500 pages/sec)
- Freshness? (re-crawl popular pages every 24 hours, less popular every 7-30 days)
- Politeness? (respect robots.txt, do not overwhelm any single domain)
- Content types? (HTML primarily, but detect and handle PDFs, images, JS-rendered pages)
- Storage? (raw HTML + extracted metadata + link graph)
- Duplicate detection? (URL deduplication and content deduplication)
</Accordion>

<Accordion title="Key design points">
1. **URL frontier (priority queue)**: The heart of the crawler is the URL frontier -- a priority queue of URLs to crawl. Priority is based on: page importance (PageRank-like score), freshness requirement (when was it last crawled), and domain-level scheduling. The frontier is sharded by domain to enforce per-domain crawl rates. Implementation: multiple queues in Redis or a custom on-disk queue (the frontier is too large for memory -- 1B URLs at 200 bytes each = 200GB).
2. **Politeness engine**: For each domain, enforce a minimum delay between requests (typically 1-5 seconds, or whatever `Crawl-delay` in robots.txt specifies). Maintain a `(domain, nextAllowedCrawlTime)` map. A dedicated "domain scheduler" dequeues URLs only when the domain's cooldown has passed. This prevents your crawler from DDoS-ing websites. Google's crawler respects this -- and so should yours, or you will get IP-banned.
3. **DNS resolver and caching**: DNS resolution for 11,500 URLs/sec is a bottleneck if done naively. Run a local DNS cache (like a local unbound resolver) with aggressive TTL caching. Batch DNS lookups. Pre-resolve domains in the frontier asynchronously before they are needed for crawling.
4. **Content fetching**: Use an async HTTP client (libcurl multi, or Python's aiohttp, or Go's goroutine-per-request model) to fetch pages concurrently. Maintain a pool of 10K-50K concurrent connections. Set aggressive timeouts: 10 seconds connect timeout, 30 seconds download timeout. Handle redirects (up to 5 hops), handle gzip/brotli compression, handle character encoding detection.
5. **Deduplication**: Two levels. **URL deduplication**: Normalize URLs (lowercase, remove fragments, sort query params), then check against a bloom filter (~10 bytes per URL, 0.1% false positive rate). At 10B known URLs, the bloom filter is ~10GB -- fits in memory. **Content deduplication**: Compute SimHash or MinHash of page content. Similar pages (same article on HTTP vs HTTPS, or paginated versions) are detected and only one version is stored.
6. **Link extraction and frontier feeding**: Parse downloaded HTML, extract all `<a href>` links, normalize them, check the bloom filter for dedup, and add new URLs to the frontier. This creates a cycle: crawl page -> extract links -> add to frontier -> crawl those pages. The frontier grows faster than you can crawl, so priority is critical.
7. **Storage**: Raw HTML stored in a distributed object store (S3/HDFS). Extracted metadata (title, text, links, timestamps) stored in a structured store (BigTable, HBase). The link graph (which page links to which) is stored separately for PageRank-style analysis. At 1B pages/day with average 100KB per page, that is 100TB/day of raw storage. Compress aggressively (gzip typically achieves 10x compression on HTML).
8. **Handling JavaScript-rendered pages**: Modern SPAs require a headless browser (Puppeteer/Playwright) to render. This is 100-1000x more expensive than simple HTTP fetches. Strategy: try simple HTTP fetch first, detect if the page has minimal content (under 1KB of text), and only send to the headless rendering pipeline if needed. Allocate a separate pool of rendering workers (~5% of total capacity).
</Accordion>

<Accordion title="What interviewers are really testing">
This question tests whether you can design a system that runs continuously at scale with external constraints you do not control (website response times, robots.txt, DNS). The core challenge is scheduling and prioritization -- you have billions of URLs and limited crawl budget, so you must be smart about what to crawl and when. The interviewer wants to see deduplication strategy, politeness handling, and an understanding of the link-extraction feedback loop. Senior candidates discuss content deduplication (SimHash), adaptive re-crawl scheduling, and handling JavaScript rendering.
</Accordion>

<Accordion title="Red flag answers">
- "Just use a BFS/DFS on links" -- this is a graph traversal question in an algorithms class, not a systems design question. At 1B pages/day, you need distributed scheduling, priority queues, and politeness constraints.
- Not mentioning robots.txt or politeness -- your crawler will get blocked and potentially trigger legal issues.
- "Store all URLs in a database and check for duplicates with a SELECT" -- at 10B URLs, this query takes seconds. Bloom filters exist for exactly this use case.
- No discussion of fault tolerance -- a crawler that loses its state on restart means re-crawling billions of pages.
</Accordion>

<Accordion title="Follow-up questions">
1. **"You discover that 30% of your crawl bandwidth is spent on near-duplicate pages (product pages with different sort orders, paginated listings). How do you fix this?"** -- Tests content deduplication. Use URL pattern detection to identify parameterized URLs (e.g., `?sort=price` vs `?sort=date` for the same product listing), canonicalize them, and crawl only the canonical version. For content-level dedup, compute SimHash fingerprints and skip pages with >95% similarity to already-crawled content.
2. **"A major news site publishes 1,000 articles per day and you need them indexed within 5 minutes. How do you prioritize freshness for some domains?"** -- Tests adaptive scheduling. Subscribe to the site's RSS feed or sitemap.xml for real-time change detection. Assign high priority in the frontier. Use a separate "real-time crawl" pipeline with lower politeness delays (if the site's robots.txt allows it). Some sites explicitly support this via the IndexNow protocol.
3. **"Your crawler has been running for 6 months and the bloom filter's false positive rate has degraded from 0.1% to 5%. What do you do?"** -- Tests operational understanding. The bloom filter is full -- it was sized for fewer URLs than you have discovered. Options: resize by creating a new larger filter and re-populating from the URL store (expensive but thorough), use a counting bloom filter that supports deletion of expired URLs, or switch to a cuckoo filter which supports deletion natively.
</Accordion>

### Question 10: Design Google Docs (Real-Time Collaboration)

<Accordion title="Requirements to clarify">
- Concurrent editors per document? (up to 100 simultaneous)
- Latency? (edits should appear on other users' screens within 200ms)
- Conflict resolution? (edits should never be lost, even if two users edit the same word simultaneously)
- Document size? (up to 1M characters, ~1MB)
- Offline editing? (yes, with sync on reconnect)
- History/versioning? (full edit history, ability to revert to any point)
</Accordion>

<Accordion title="Key design points">
1. **Conflict resolution -- OT vs CRDT**: The fundamental challenge of collaborative editing is resolving concurrent edits. Two established approaches:
   - **Operational Transformation (OT)**: Used by Google Docs. Each edit is an "operation" (insert char at position 5, delete chars 3-7). When two operations arrive concurrently, the server transforms one against the other to produce equivalent results regardless of application order. Example: User A inserts "X" at position 3, User B inserts "Y" at position 3 simultaneously. OT transforms B's operation to "insert Y at position 4" because A's insert shifted the positions. Requires a central server to determine canonical ordering.
   - **CRDT (Conflict-free Replicated Data Type)**: Used by Figma, Yjs, and Automerge. Each character gets a unique ID (based on the user who inserted it and a logical timestamp). Operations reference character IDs, not positions, so they commute naturally without transformation. More complex data structure but enables true peer-to-peer collaboration without a central server.
2. **Architecture for OT (Google Docs model)**: `Client -> WebSocket -> Collaboration Server -> (transform) -> broadcast to other clients`. Each document has a single collaboration server that serializes all operations. Clients send operations optimistically (apply locally, send to server, server transforms and broadcasts). The server maintains the document state and an operation log.
3. **WebSocket connection management**: Each client maintains a WebSocket to the collaboration server for the document they are editing. At Google's scale (hundreds of millions of documents, but only thousands actively being edited concurrently), you shard collaboration servers by document ID. Each server handles ~10K concurrent documents.
4. **Cursor and selection sync**: Each user's cursor position and text selection are broadcast to other users. These are ephemeral (not persisted) and sent as lightweight messages over the same WebSocket. Update frequency: throttle to every 50ms to avoid flooding the network.
5. **Persistence and versioning**: The operation log is the source of truth. Periodically (every 100 operations or 30 seconds), create a snapshot of the full document state. To reconstruct any historical version, load the nearest snapshot and replay operations forward. Store operations in a time-series database (Cassandra) and snapshots in blob storage (S3).
6. **Offline editing**: Client stores operations locally while offline. On reconnect, send all pending operations to the server. The server transforms them against any operations that happened while the client was offline. This can result in complex multi-way transformation. CRDTs handle this more gracefully than OT because CRDT operations commute without server-side transformation.
7. **Undo/Redo**: Challenging in a collaborative context. "Undo" means undo MY last action, not the last action globally. Implement as "inverse operations": if I inserted "hello", undo generates a "delete hello" operation, which is then transformed against any concurrent operations. This is one of the hardest parts of collaborative editing to get right.
</Accordion>

<Accordion title="What interviewers are really testing">
This is the hardest question on the list for most candidates. The interviewer wants to see if you understand the fundamental CAP-like tradeoff in collaborative editing: you want low latency (apply edits immediately), eventual consistency (all users see the same document), and conflict resolution (no edits lost). OT and CRDT are the two solutions, and a strong candidate should explain at least one in detail with a concrete example of how two concurrent edits are resolved. Staff-level candidates compare OT vs CRDT tradeoffs.
</Accordion>

<Accordion title="Red flag answers">
- "Use locking -- only one user can edit a section at a time" -- this defeats the entire purpose of real-time collaboration. Google Docs explicitly rejected this approach.
- "Last write wins" -- unacceptable for a document editor. User A types a paragraph, User B's cursor click "wins" and deletes it? No.
- Describing OT without explaining transformation with a concrete example -- if you cannot walk through two concurrent inserts being transformed, you do not understand OT.
- "Just merge the changes like Git" -- Git-style three-way merge works for asynchronous collaboration (pull requests) but not for real-time character-by-character editing.
</Accordion>

<Accordion title="Follow-up questions">
1. **"User A inserts 'Hello' at position 10, User B simultaneously deletes characters 8-12. Walk me through how OT resolves this."** -- Tests deep understanding. The server receives both operations. If A's insert is ordered first: insert "Hello" at 10, then B's delete needs transformation -- the range 8-12 now contains A's inserted text. The transformed delete becomes: delete chars 8-9 (original text before A's insert) and chars 15-17 (original text after A's insert), preserving A's "Hello". The exact transformation depends on the OT algorithm used (Jupiter, GOTO, etc.).
2. **"You chose OT with a central server. That server is now a single point of failure for every document. How do you make this fault-tolerant?"** -- Tests distributed systems thinking. Each document's collaboration server has a hot standby that receives replicated operations. If the primary fails, the standby takes over. The operation log in Cassandra survives the failover. Clients reconnect, replay from the last acknowledged operation sequence number. Key insight: you only need one active server per document, not per cluster -- the blast radius of a server failure is limited to the documents it was hosting.
3. **"How would you add 'suggesting mode' (like Google Docs' suggestion feature) where edits are proposals that the document owner must accept or reject?"** -- Tests ability to extend the design. Suggestions are a separate operation type stored in a parallel stream. They are rendered as overlays on the document (strikethrough for deletions, colored text for insertions) but do not modify the canonical document state. Accepting a suggestion generates a real operation. This requires the UI to render two layers simultaneously: the canonical document and the suggestion overlay.
</Accordion>

### Question 11: Design an API Rate Limiting and Authentication Gateway

<Accordion title="Requirements to clarify">
- Traffic volume? (500K requests/sec across all APIs)
- Authentication methods? (API keys, OAuth 2.0 JWT tokens, mTLS for service-to-service)
- Rate limiting granularity? (per API key, per endpoint, per user, tiered plans: free=100/min, pro=1000/min, enterprise=10K/min)
- Latency budget? (gateway must add less than 5ms to each request at p99)
- Protocol support? (HTTP/1.1, HTTP/2, gRPC, WebSocket upgrade)
- Deployment? (multi-region, each region handles its local traffic)
</Accordion>

<Accordion title="Key design points">
1. **Request flow**: `Client -> DNS/Anycast -> Edge Load Balancer -> API Gateway -> Backend Service`. The gateway is the single entry point for all API traffic. Every request passes through: TLS termination, authentication, rate limiting, request transformation, routing, and response transformation, in that order.
2. **Authentication layer**: For JWT tokens: validate the signature locally using the cached public key (no network call needed -- this is why JWTs are stateless). For API keys: look up in a local cache (Redis or in-memory) that syncs from the API key database every 30 seconds. For mTLS: validate the client certificate chain against a trusted CA bundle. Key insight: authentication MUST be fast because it runs on every request. Anything requiring a synchronous database lookup adds unacceptable latency.
3. **Rate limiting integration**: Use the sliding window counter algorithm (described in the rate limiter question) with Redis as the backing store. The gateway checks rate limits AFTER authentication (so you know which API key/user to rate limit against). Implement tiered limits: a `free` plan API key gets 100 req/min, `pro` gets 1,000, `enterprise` gets 10,000. Limits are stored as configuration per API key in the same cache used for authentication.
4. **Request routing**: Route based on URL path, HTTP method, headers, or request body. Configuration is loaded from a routing table (stored in etcd or Consul) that maps `(path pattern, method) -> backend service + load balancing policy`. Support canary routing (send 5% of traffic to a new version), A/B testing (route based on user segment), and circuit breaking (stop routing to unhealthy backends).
5. **Resilience patterns**: Circuit breaker per backend service (open after 5 consecutive failures, half-open after 30 seconds). Retry with exponential backoff for 5xx responses (but NOT for 4xx -- those are client errors). Timeout enforcement: if the backend does not respond within the configured timeout, return 504 Gateway Timeout.
6. **Observability**: Log every request with: request ID (UUID), API key hash, endpoint, response code, latency, rate limit remaining. Emit metrics: requests/sec by endpoint, error rate, p50/p99 latency, rate limit rejections. These are critical for debugging and capacity planning.
7. **Deployment topology**: Deploy the gateway as a stateless horizontally-scalable service. Each instance can handle any request (all state is in Redis/external config). Run 3+ instances per availability zone behind a load balancer. Use blue-green deployments for gateway updates since any downtime means ALL APIs are down.
</Accordion>

<Accordion title="What interviewers are really testing">
This question combines multiple system design concepts (authentication, rate limiting, load balancing, resilience) into one cohesive system. The interviewer wants to see if you can design a critical-path component that touches every request without becoming a bottleneck or a single point of failure. The hidden test is latency awareness -- every millisecond the gateway adds is multiplied by every request in your system.
</Accordion>

<Accordion title="Red flag answers">
- "Validate JWTs by calling the auth service on every request" -- defeats the purpose of JWTs being stateless. The whole point of JWTs is local validation without network calls.
- "Store rate limit state in the gateway's local memory" -- since you have multiple gateway instances, each instance has a partial view. Users can exceed limits by distributing requests across instances.
- "Use a single gateway instance" -- a single point of failure for all API traffic is unacceptable. The gateway must be horizontally scalable.
- Not discussing what happens when the gateway itself crashes -- if all traffic flows through the gateway, its failure mode is the system's failure mode.
</Accordion>

<Accordion title="Follow-up questions">
1. **"Your gateway adds 8ms of latency. Your target is 5ms. How do you profile and optimize?"** -- Expect discussion of: connection pooling to backends (avoid TCP handshake per request), response caching for idempotent GET endpoints, moving rate limit checks to an in-process data structure (approximation acceptable), and using HTTP/2 multiplexing to reduce connection overhead.
2. **"A compromised API key is being used to exfiltrate data. How does your gateway help detect and stop this?"** -- Tests security thinking. The gateway logs all requests per API key. Anomaly detection (unusual traffic volume, new source IPs, accessing endpoints the key normally does not access) flags suspicious activity. Immediate revocation updates the API key cache within 30 seconds (or faster with a push-based cache invalidation via pub/sub).
3. **"How do you handle API versioning at the gateway level?"** -- Tests practical API management. Route based on URL prefix (`/v1/users` vs `/v2/users`) or `Accept` header (`application/vnd.api.v2+json`). The gateway can also transform requests between versions (add default values for new fields, rename fields) to extend the life of old API versions without burdening backend teams.
</Accordion>

### Question 12: Design a Flash Sale / Inventory System

<Accordion title="Requirements to clarify">
- Scale? (1M users hitting the buy button within a 10-second window for 10,000 items)
- Consistency requirement? (CRITICAL -- never sell more items than available, i.e., no overselling)
- Latency? (respond within 500ms -- users expect near-instant feedback)
- Fairness? (first-come-first-served, or randomized lottery?)
- What happens after purchase? (payment processing, order creation, notification)
- Post-sale analytics? (sold-out time, conversion rate, user funnel)
</Accordion>

<Accordion title="Key design points">
1. **The core challenge**: 1M requests in 10 seconds = 100K requests/sec for a single resource (inventory count). This is not a scaling problem -- it is a contention problem. You cannot use a traditional database row lock because 100K concurrent transactions trying to lock the same row will create a lock convoy that takes minutes to resolve.
2. **Solution 1 -- Redis atomic decrement**: Store `inventory:item123 = 10000` in Redis. Each purchase request runs `DECR inventory:item123`. If the result is >= 0, the purchase succeeds. If negative, the item is sold out (increment back to undo, or just return "sold out"). Redis processes 100K single-key operations per second trivially. The entire hot path is one Redis command with no locks.
3. **Solution 2 -- Pre-allocated inventory tokens**: Before the sale starts, generate 10,000 tokens and load them into a Redis list. Each purchase request runs `LPOP` to grab a token. If the list is empty, sold out. This avoids even the theoretical race condition of `DECR` going negative (though Redis `DECR` is atomic, so it is safe too). The token approach makes post-sale reconciliation easier.
4. **Request funnel (shed traffic early)**: At 1M requests, most will fail (only 10K can succeed). Shed traffic as early as possible. Layer 1: CDN-level response for sold-out items (once sold out, serve a static "sold out" page from the CDN, zero load on your servers). Layer 2: Load balancer rate limiting (only allow 50K requests through to the app servers). Layer 3: App server checks Redis for remaining inventory before doing any expensive work (database queries, payment processing).
5. **Decouple purchase from payment**: The purchase request only reserves the inventory (Redis `DECR`). Payment processing happens asynchronously. If the user does not complete payment within 10 minutes, release the inventory (Redis `INCR`). This keeps the hot path (inventory check) blazing fast and isolates it from the slower payment flow.
6. **Queue-based architecture for fairness**: Instead of letting all 1M requests hit the system, put them in a queue when they arrive. Process the queue in order (FIFO = first-come-first-served). The user sees "You are in line, position 45,321" with real-time updates. This dramatically reduces contention because the inventory service processes requests sequentially from the queue at a controlled rate. This is the approach used by Ticketmaster, Supreme, and most flash sale platforms.
7. **Bot prevention**: Flash sales attract bots. Defenses: CAPTCHA before the sale, browser fingerprinting, rate limiting per IP and per account, analyzing mouse movement patterns (bots have non-human mouse trajectories), and requiring account age > 30 days. None are perfect individually -- use defense in depth.
</Accordion>

<Accordion title="What interviewers are really testing">
This tests whether you can handle extreme contention on a single resource. The naive approach (SQL transaction with `SELECT ... FOR UPDATE`) will collapse. The interviewer wants to see you identify contention as the bottleneck (not throughput), propose an appropriate data structure (Redis atomic operations), and design a request funnel that sheds losing requests early. Senior candidates discuss the queue-based approach, bot prevention, and the decoupling of inventory reservation from payment processing.
</Accordion>

<Accordion title="Red flag answers">
- "Use a database transaction with `SELECT ... FOR UPDATE`" for the hot inventory check -- at 100K TPS on a single row, this creates a lock convoy. The database connection pool exhausts in seconds.
- "Just scale horizontally" -- you cannot scale away contention on a single shared counter. Adding more app servers makes it worse (more contenders for the same lock).
- No mention of fairness or what happens to the 990K users who do not get an item -- the user experience for the failure case matters as much as the success case.
- "Prevent overselling by checking inventory in the application code" -- without atomic operations, a check-then-decrement has a race condition where 100 users all read `inventory = 1` and all proceed to purchase.
</Accordion>

<Accordion title="Follow-up questions">
1. **"The sale started and your Redis instance just died. 5,000 items are already sold. What happens?"** -- Tests disaster recovery. If using Redis with AOF persistence, replay the log on a new instance (data loss of at most 1 second of operations). If using Redis Cluster with replicas, the replica promotes automatically. The key question: can you reconstruct the inventory count? Yes, if you log every successful purchase to a durable store (Kafka, database) before returning success to the user. Count the logs to determine how many items remain.
2. **"Users are complaining that the system said 'purchased successfully' but then they received a 'payment failed' email 5 minutes later. How do you handle this?"** -- Tests the decoupled inventory/payment design. The "purchased successfully" message should say "reserved" not "purchased." If payment fails, release the inventory and notify the user. Better UX: hold the user on a payment confirmation screen until payment actually clears (~3-5 seconds for credit cards). The tradeoff: holding 10K users on a blocking payment screen means you need 10K concurrent connections for potentially 30 seconds each.
3. **"Your CEO wants to run 50 flash sales simultaneously across different products. How does the system change?"** -- Tests whether the design generalizes. Each product has its own Redis key, so there is no cross-product contention. The challenge shifts to total system throughput: 50 sales * 100K req/sec each = 5M req/sec total. Scale the load balancer and app server fleet, use Redis Cluster with one shard per product, and pre-warm the CDN with static "sold out" pages for each product.
</Accordion>

### Question 13: Design a Distributed Task Scheduler

<Accordion title="Requirements to clarify">
- Scale? (10M scheduled tasks, 100K task executions/minute)
- Task types? (one-time scheduled tasks, recurring cron-style tasks, delayed tasks)
- Reliability? (every task must execute at least once -- at-least-once semantics)
- Latency? (tasks should fire within 1 second of their scheduled time)
- Task duration? (short tasks: <1 minute, long tasks: up to 1 hour)
- Failure handling? (retry with backoff, dead-letter queue after N failures)
</Accordion>

<Accordion title="Key design points">
1. **Task storage**: Store task definitions in a persistent database (PostgreSQL or Cassandra). Schema: `taskId`, `scheduledTime`, `cronExpression` (for recurring), `payload`, `status` (`PENDING`, `RUNNING`, `COMPLETED`, `FAILED`), `retryCount`, `maxRetries`, `nextRetryTime`. Index on `(status, scheduledTime)` for efficient polling.
2. **Scheduler service (the clock)**: Multiple scheduler instances poll the database for tasks where `scheduledTime <= now() AND status = PENDING`. Use `SELECT ... FOR UPDATE SKIP LOCKED` (PostgreSQL) to avoid multiple schedulers picking up the same task. Alternatively, partition tasks by time bucket (each scheduler owns a range of time buckets) using a leader election mechanism (ZooKeeper, etcd lease). The scheduler does not execute tasks -- it enqueues them for workers.
3. **Worker pool**: Workers pull tasks from a queue (Kafka, SQS, or Redis Streams). Each worker executes the task payload, reports success/failure back to the scheduler. Workers are stateless and horizontally scalable. For long-running tasks, workers send heartbeats to indicate they are still alive. If a heartbeat is missed (worker crashed), the task is re-queued.
4. **Exactly-once execution (at-least-once + idempotency)**: The scheduler guarantees at-least-once execution (a task may be dispatched twice if the scheduler crashes after dispatch but before marking it as dispatched). Workers must be idempotent: executing the same task twice produces the same result. Each task has a unique `executionId`. Workers check a deduplication store before executing.
5. **Cron scheduling**: For recurring tasks, after each execution, compute the next execution time from the cron expression and create a new task entry. Use a cron parser library (not hand-rolled -- cron expressions have edge cases like leap years, daylight saving time, and "last day of month"). Store both `cronExpression` and the materialized `nextExecutionTime` for efficient querying.
6. **Time-wheel optimization**: For high-volume scheduling (100K tasks/minute), polling the database every second is expensive. Use a hierarchical timing wheel (Kafka uses this internally): a circular buffer where each slot represents a time interval (e.g., 1 second). Tasks are placed in the slot corresponding to their scheduled time. A pointer advances each second, firing all tasks in the current slot. This gives O(1) insertion and O(1) firing, compared to O(log N) for a priority queue.
7. **Failure handling**: On task failure, increment `retryCount`, compute `nextRetryTime` with exponential backoff (base * 2^retryCount, with jitter), and set status to `PENDING`. After `maxRetries` failures, move to `DEAD_LETTER` status and alert an operator. Maintain a dead-letter dashboard for manual inspection and replay.
8. **Monitoring**: Track metrics: tasks scheduled/min, tasks executed/min, execution latency (scheduled time vs. actual fire time), failure rate, retry rate, queue depth, worker utilization. Alert on: task execution latency > 5 seconds (scheduler falling behind), failure rate > 5%, queue depth growing (workers cannot keep up).
</Accordion>

<Accordion title="What interviewers are really testing">
This question tests reliability engineering. The core challenge is: how do you guarantee that every task fires exactly once (or at least once) in a distributed system where schedulers and workers can crash at any time? The interviewer wants to see leader election or distributed locking for scheduler coordination, at-least-once semantics with idempotent workers, and operational awareness (monitoring, alerting, dead-letter queues). Staff-level candidates discuss timing wheel optimization and the subtleties of cron scheduling across time zones.
</Accordion>

<Accordion title="Red flag answers">
- "Use `setInterval` in Node.js" or "use `cron` on a single Linux server" -- this is a single point of failure with no persistence. If the server restarts, all scheduled tasks are lost.
- "Poll the database every second" as the only approach at scale -- at 10M tasks, this query is expensive. Need to discuss partitioning, indexing, or timing wheels.
- Not discussing what happens when a worker crashes mid-execution -- the task is lost unless you have heartbeat monitoring and re-queuing.
- "Use Kubernetes CronJob" -- while valid for small scale, K8s CronJobs have second-level precision issues, no built-in dead-letter handling, and limited visibility into failures at scale.
</Accordion>

<Accordion title="Follow-up questions">
1. **"You have 100K tasks scheduled for exactly midnight. How do you avoid a thundering herd at 00:00:00?"** -- Tests operational thinking. Add jitter: spread tasks across a window (midnight +/- 30 seconds). Or use a pre-fetch mechanism: start loading midnight tasks at 23:59:50 and queue them for workers, so by midnight the workers are already processing. The timing wheel naturally handles this by spreading tasks across 1-second slots.
2. **"A recurring task has been silently failing for 3 days. Nobody noticed. How do you prevent this?"** -- Tests monitoring design. Track "expected vs. actual executions" per recurring task. If a task that should run hourly has not succeeded in 3 hours, fire an alert. Also track the ratio of successful runs: if it drops below 90%, alert. The dead-letter queue should have a dashboard with aging metrics -- tasks sitting in dead letter for > 1 hour get escalated.
3. **"Two scheduler instances both pick up the same task due to a database replication lag. The task charges a customer $100. What prevents the double charge?"** -- Tests understanding of idempotency in practice. The worker must check an idempotency key (the `taskId` + `executionId`) against a deduplication store before executing. If the key exists, skip execution. The deduplication store (Redis with TTL or a database unique constraint) must be checked BEFORE the side effect (the charge). This is the same pattern used in payment systems.
</Accordion>

## Self-Assessment Checklist

Before your interview, make sure you can:

### Fundamentals
- [ ] Calculate QPS, storage, and bandwidth requirements without a calculator -- practice until "100M DAU, 10 actions each" becomes "~12K QPS" in your head instantly
- [ ] Choose between SQL and NoSQL with justification -- not just "NoSQL scales better" but "our access pattern is single-key lookups with no joins, write volume is 50K/sec, and eventual consistency is acceptable for this data"
- [ ] Explain CAP theorem and when to prioritize what -- more importantly, explain why most real systems are CP or AP and what "partition tolerance" actually means in practice (hint: you do not choose it, it happens to you)
- [ ] Design a basic caching strategy -- including cache invalidation, TTL selection, and what happens on cache miss
- [ ] Draw a load-balanced architecture and explain the difference between L4 and L7 load balancing

### Intermediate
- [ ] Design for horizontal scalability -- know when to shard vs replicate and how to choose shard keys
- [ ] Explain consistent hashing and why it matters when adding/removing nodes (not just the textbook definition, but the actual problem it solves)
- [ ] Choose between push vs pull models with concrete reasoning -- fan-out on write vs fan-out on read is the same concept applied to Twitter, notifications, and news feeds
- [ ] Design for fault tolerance -- circuit breakers, bulkheads, retry with backoff and jitter, graceful degradation
- [ ] Handle data partitioning/sharding and explain hot partition mitigation strategies

### Advanced
- [ ] Design for global scale (multi-region) -- including data residency requirements, cross-region replication lag, and conflict resolution
- [ ] Handle distributed transactions -- Saga pattern, two-phase commit (and why it is usually avoided), outbox pattern
- [ ] Implement exactly-once delivery -- or more precisely, explain why it is impossible in theory and how to achieve "effectively-once" with idempotency keys
- [ ] Design real-time systems -- WebSocket vs SSE vs long polling, connection management at scale, presence systems
- [ ] Handle hot partitions -- detect them, split them, or use write-behind buffers to absorb spikes

## Pattern Recognition: Cross-Cutting Themes

The same design patterns appear across different system design questions. Recognizing them is the difference between solving problems from scratch and applying proven solutions:

| Pattern | Appears In | Core Idea |
|---------|-----------|-----------|
| **Fan-out on write** | Twitter timeline, notification system, group chat | Pre-compute results at write time to make reads O(1) |
| **Fan-out on read** | Celebrity tweets, search results, recommendation feeds | Compute at read time to avoid write amplification |
| **Event sourcing** | Payment ledger, order processing, collaborative editing | Append-only log of state changes, reconstruct state by replaying |
| **CQRS** | E-commerce catalog, social feeds, analytics | Separate read and write models optimized for their respective access patterns |
| **Consistent hashing** | Distributed cache, database sharding, load balancing | Minimize data movement when nodes join or leave |
| **Circuit breaker** | Payment routing, notification delivery, API gateway | Fail fast when downstream services are unhealthy |
| **Saga pattern** | E-commerce checkout, ride matching, booking systems | Coordinate distributed transactions with compensating actions |
| **Rate limiting** | API gateway, notification system, login attempts | Protect systems from overload using token bucket or sliding window |
| **Pub/Sub** | Real-time messaging, event-driven architecture, cache invalidation | Decouple producers from consumers for scalability and flexibility |
| **Idempotency** | Payment processing, message delivery, API retries | Make operations safely repeatable without side effects |
