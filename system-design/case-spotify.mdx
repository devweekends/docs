## Problem Statement

Design a music streaming service like Spotify that handles:
- Music catalog with millions of songs
- Real-time audio streaming
- Personalized playlists and recommendations
- Search and discovery
- Offline playback
- Social features (following, sharing)

## Requirements

### Functional Requirements

```
Core Features:
├── Music Playback
│   ├── Stream songs in real-time
│   ├── Queue management
│   ├── Shuffle and repeat
│   ├── Offline downloads
│   └── Cross-device sync
│
├── Content Discovery
│   ├── Search (songs, artists, albums, playlists)
│   ├── Browse by genre/mood
│   ├── Personalized recommendations
│   └── Radio stations
│
├── Library Management
│   ├── Save songs/albums/playlists
│   ├── Create playlists
│   ├── Recently played
│   └── Listening history
│
└── Social Features
    ├── Follow artists/users
    ├── Collaborative playlists
    └── Share music
```

### Non-Functional Requirements

```
Scale:
├── 400M monthly active users
├── 180M premium subscribers
├── 80M songs in catalog
├── 4B playlists
└── 1M concurrent streams

Performance:
├── Playback start < 200ms
├── Search results < 100ms
├── No buffering during playback
└── Seamless song transitions

Availability:
├── 99.99% uptime for streaming
├── Graceful degradation
└── Offline playback support
```

## Capacity Estimation

```
Storage:
├── Songs: 80M × 5MB (avg) = 400TB audio files
├── Multiple quality levels: 400TB × 4 = 1.6PB
├── Metadata: 80M × 10KB = 800GB
├── User data: 400M × 50KB = 20TB
└── Playlists: 4B × 1KB = 4TB

Bandwidth:
├── Concurrent streams: 1M
├── Bitrate: 160kbps (average)
├── Bandwidth: 1M × 160kbps = 160Gbps
│
├── Daily streams: 400M users × 2 hours × 60min/15 songs
├── = 3.2B song plays per day
└── = 37,000 song plays per second
```

```python
from dataclasses import dataclass
from typing import Optional, List, Generator
from enum import Enum
import asyncio

class AudioQuality(Enum):
    LOW = "96"
    NORMAL = "160"
    HIGH = "320"
    VERY_HIGH = "320"  # Lossless for premium

@dataclass
class AudioSegment:
    song_id: str
    segment_index: int
    start_time_ms: int
    duration_ms: int
    url: str
    
@dataclass
class PlaybackSession:
    session_id: str
    user_id: str
    device_id: str
    song_id: str
    quality: AudioQuality
    position_ms: int
    is_playing: bool


class AudioStreamingService:
    """
    Handles audio streaming with adaptive bitrate.
    """
    
    SEGMENT_DURATION_MS = 10_000  # 10 seconds per segment
    PREFETCH_SEGMENTS = 3
    
    def __init__(
        self, 
        audio_storage,
        cdn_service,
        session_store,
        analytics
    ):
        self.audio_storage = audio_storage
        self.cdn = cdn_service
        self.sessions = session_store
        self.analytics = analytics
    
    async def start_playback(
        self,
        user_id: str,
        device_id: str,
        song_id: str,
        quality: AudioQuality = AudioQuality.NORMAL,
        start_position_ms: int = 0
    ) -> PlaybackSession:
        """
        Initialize a new playback session.
        """
        # Create session
        session = PlaybackSession(
            session_id=generate_session_id(),
            user_id=user_id,
            device_id=device_id,
            song_id=song_id,
            quality=quality,
            position_ms=start_position_ms,
            is_playing=True
        )
        
        # Store session
        await self.sessions.save(session)
        
        # Get song metadata
        song = await self.audio_storage.get_song_metadata(song_id)
        
        # Generate CDN URLs for initial segments
        start_segment = start_position_ms // self.SEGMENT_DURATION_MS
        segments = await self._get_segment_urls(
            song_id,
            quality,
            start_segment,
            self.PREFETCH_SEGMENTS
        )
        
        # Track playback start
        await self.analytics.track_play_start(
            user_id=user_id,
            song_id=song_id,
            device_id=device_id,
            quality=quality.value
        )
        
        return {
            "session": session,
            "song": song,
            "segments": segments,
            "total_segments": song["duration_ms"] // self.SEGMENT_DURATION_MS + 1
        }
    
    async def get_next_segments(
        self,
        session_id: str,
        current_segment: int
    ) -> List[AudioSegment]:
        """
        Get next segments for continuous playback.
        Called by client as they approach end of buffered content.
        """
        session = await self.sessions.get(session_id)
        if not session:
            raise SessionNotFoundError(session_id)
        
        return await self._get_segment_urls(
            session.song_id,
            session.quality,
            current_segment + 1,
            self.PREFETCH_SEGMENTS
        )
    
    async def _get_segment_urls(
        self,
        song_id: str,
        quality: AudioQuality,
        start_segment: int,
        count: int
    ) -> List[AudioSegment]:
        """
        Generate signed CDN URLs for audio segments.
        """
        segments = []
        
        for i in range(count):
            segment_index = start_segment + i
            
            # Generate signed URL (expires in 1 hour)
            path = f"audio/{song_id}/{quality.value}/{segment_index}.ogg"
            signed_url = await self.cdn.get_signed_url(
                path,
                expires_in=3600
            )
            
            segments.append(AudioSegment(
                song_id=song_id,
                segment_index=segment_index,
                start_time_ms=segment_index * self.SEGMENT_DURATION_MS,
                duration_ms=self.SEGMENT_DURATION_MS,
                url=signed_url
            ))
        
        return segments
    
    async def update_position(
        self,
        session_id: str,
        position_ms: int
    ):
        """
        Update playback position for resume and analytics.
        """
        session = await self.sessions.get(session_id)
        if session:
            session.position_ms = position_ms
            await self.sessions.save(session)
    
    async def handle_quality_switch(
        self,
        session_id: str,
        new_quality: AudioQuality,
        current_segment: int
    ) -> List[AudioSegment]:
        """
        Handle adaptive bitrate switch.
        """
        session = await self.sessions.get(session_id)
        session.quality = new_quality
        await self.sessions.save(session)
        
        # Return new quality segments
        return await self._get_segment_urls(
            session.song_id,
            new_quality,
            current_segment,
            self.PREFETCH_SEGMENTS
        )
    
    async def sync_playback_state(
        self,
        user_id: str,
        device_id: str
    ) -> Optional[PlaybackSession]:
        """
        Get current playback state for cross-device sync.
        """
        # Find active session for user
        active_session = await self.sessions.get_active_for_user(user_id)
        
        if active_session and active_session.device_id != device_id:
            # Transfer playback to this device
            return await self.transfer_playback(
                active_session.session_id,
                device_id
            )
        
        return active_session
```

```javascript
const AudioQuality = {
    LOW: '96',
    NORMAL: '160',
    HIGH: '320',
    VERY_HIGH: '320'
};

class AudioStreamingService {
    static SEGMENT_DURATION_MS = 10_000;
    static PREFETCH_SEGMENTS = 3;
    
    constructor(audioStorage, cdnService, sessionStore, analytics) {
        this.audioStorage = audioStorage;
        this.cdn = cdnService;
        this.sessions = sessionStore;
        this.analytics = analytics;
    }
    
    async startPlayback({
        userId,
        deviceId,
        songId,
        quality = AudioQuality.NORMAL,
        startPositionMs = 0
    }) {
        const session = {
            sessionId: generateSessionId(),
            userId,
            deviceId,
            songId,
            quality,
            positionMs: startPositionMs,
            isPlaying: true
        };
        
        await this.sessions.save(session);
        
        const song = await this.audioStorage.getSongMetadata(songId);
        
        const startSegment = Math.floor(
            startPositionMs / AudioStreamingService.SEGMENT_DURATION_MS
        );
        
        const segments = await this.getSegmentUrls(
            songId,
            quality,
            startSegment,
            AudioStreamingService.PREFETCH_SEGMENTS
        );
        
        await this.analytics.trackPlayStart({
            userId,
            songId,
            deviceId,
            quality
        });
        
        return {
            session,
            song,
            segments,
            totalSegments: Math.ceil(
                song.durationMs / AudioStreamingService.SEGMENT_DURATION_MS
            )
        };
    }
    
    async getNextSegments(sessionId, currentSegment) {
        const session = await this.sessions.get(sessionId);
        if (!session) {
            throw new SessionNotFoundError(sessionId);
        }
        
        return this.getSegmentUrls(
            session.songId,
            session.quality,
            currentSegment + 1,
            AudioStreamingService.PREFETCH_SEGMENTS
        );
    }
    
    async getSegmentUrls(songId, quality, startSegment, count) {
        const segments = [];
        
        for (let i = 0; i < count; i++) {
            const segmentIndex = startSegment + i;
            const path = `audio/${songId}/${quality}/${segmentIndex}.ogg`;
            
            const signedUrl = await this.cdn.getSignedUrl(path, {
                expiresIn: 3600
            });
            
            segments.push({
                songId,
                segmentIndex,
                startTimeMs: segmentIndex * AudioStreamingService.SEGMENT_DURATION_MS,
                durationMs: AudioStreamingService.SEGMENT_DURATION_MS,
                url: signedUrl
            });
        }
        
        return segments;
    }
    
    async handleQualitySwitch(sessionId, newQuality, currentSegment) {
        const session = await this.sessions.get(sessionId);
        session.quality = newQuality;
        await this.sessions.save(session);
        
        return this.getSegmentUrls(
            session.songId,
            newQuality,
            currentSegment,
            AudioStreamingService.PREFETCH_SEGMENTS
        );
    }
}

module.exports = { AudioStreamingService, AudioQuality };
```

```python
from dataclasses import dataclass
from typing import List, Dict, Tuple
import numpy as np
from sklearn.cluster import KMeans

@dataclass
class Song:
    id: str
    name: str
    artist_id: str
    audio_features: Dict[str, float]  # tempo, energy, danceability, etc.

@dataclass
class UserTaste:
    cluster_id: int
    seed_tracks: List[str]
    feature_centroid: Dict[str, float]


class RecommendationEngine:
    """
    Multi-strategy recommendation engine.
    """
    
    def __init__(
        self,
        song_vectors,      # Pre-computed song embeddings
        user_vectors,      # Pre-computed user embeddings
        audio_features,    # Song audio features
        listening_history, # User listening data
        social_graph       # Follow relationships
    ):
        self.song_vectors = song_vectors
        self.user_vectors = user_vectors
        self.audio_features = audio_features
        self.history = listening_history
        self.social = social_graph
    
    async def get_personalized_recommendations(
        self,
        user_id: str,
        context: Dict = None,
        limit: int = 50
    ) -> List[Song]:
        """
        Get personalized recommendations combining multiple signals.
        """
        # Get user's recent listening
        recent = await self.history.get_recent_plays(user_id, days=30)
        
        # Get candidates from different sources
        candidates = []
        
        # 1. Collaborative filtering (40% weight)
        cf_candidates = await self._collaborative_filtering(user_id, limit=100)
        candidates.extend([(s, 0.4) for s in cf_candidates])
        
        # 2. Content-based from recent plays (30% weight)
        for song_id in recent[:10]:
            similar = await self._content_based_similar(song_id, limit=20)
            candidates.extend([(s, 0.3) for s in similar])
        
        # 3. Audio feature matching (20% weight)
        taste_profile = await self._get_audio_taste_profile(user_id)
        audio_similar = await self._audio_feature_matching(
            taste_profile, 
            limit=50
        )
        candidates.extend([(s, 0.2) for s in audio_similar])
        
        # 4. Discovery/exploration (10% weight)
        discovery = await self._get_discovery_tracks(user_id, limit=20)
        candidates.extend([(s, 0.1) for s in discovery])
        
        # Aggregate scores
        song_scores = {}
        for song, weight in candidates:
            if song.id not in song_scores:
                song_scores[song.id] = {"song": song, "score": 0}
            song_scores[song.id]["score"] += weight
        
        # Apply context modifiers
        if context:
            song_scores = self._apply_context_boost(song_scores, context)
        
        # Filter already played
        played_ids = set(recent)
        final = [
            item for item in song_scores.values()
            if item["song"].id not in played_ids
        ]
        
        # Sort by score and return top N
        final.sort(key=lambda x: x["score"], reverse=True)
        return [item["song"] for item in final[:limit]]
    
    async def _collaborative_filtering(
        self,
        user_id: str,
        limit: int
    ) -> List[Song]:
        """
        Find songs that similar users liked.
        Using pre-computed user embeddings.
        """
        # Get user's embedding
        user_vector = await self.user_vectors.get(user_id)
        
        # Find similar users
        similar_users = await self.user_vectors.find_similar(
            user_vector,
            k=50
        )
        
        # Get songs they liked that this user hasn't heard
        user_songs = set(await self.history.get_all_plays(user_id))
        candidates = []
        
        for similar_user_id, similarity in similar_users:
            their_songs = await self.history.get_recent_plays(
                similar_user_id,
                days=30
            )
            for song_id in their_songs:
                if song_id not in user_songs:
                    song = await self.audio_features.get_song(song_id)
                    candidates.append((song, similarity))
        
        # Sort by weighted score
        candidates.sort(key=lambda x: x[1], reverse=True)
        return [song for song, _ in candidates[:limit]]
    
    async def _content_based_similar(
        self,
        song_id: str,
        limit: int
    ) -> List[Song]:
        """
        Find songs similar to a given song.
        Using pre-computed song embeddings.
        """
        song_vector = await self.song_vectors.get(song_id)
        similar = await self.song_vectors.find_similar(song_vector, k=limit)
        
        return [
            await self.audio_features.get_song(s_id)
            for s_id, _ in similar
        ]
    
    async def generate_daily_mix(
        self,
        user_id: str,
        mix_index: int  # 1-6
    ) -> List[Song]:
        """
        Generate personalized daily mix playlist.
        Each mix focuses on a different taste cluster.
        """
        # Get user's taste clusters
        clusters = await self._cluster_user_taste(user_id, n_clusters=6)
        
        if mix_index > len(clusters):
            return []
        
        cluster = clusters[mix_index - 1]
        
        # Get seed tracks from this cluster
        seeds = cluster.seed_tracks[:5]
        
        playlist = []
        
        # 60% similar to seeds
        for seed_id in seeds:
            similar = await self._content_based_similar(seed_id, limit=5)
            playlist.extend(similar[:3])
        
        # 20% matching audio features
        audio_matches = await self._audio_feature_matching(
            cluster.feature_centroid,
            limit=10
        )
        playlist.extend(audio_matches)
        
        # 20% discovery (new artists in similar space)
        discovery = await self._get_discovery_in_cluster(
            user_id,
            cluster,
            limit=10
        )
        playlist.extend(discovery)
        
        # Order for good listening flow
        playlist = self._order_for_flow(playlist)
        
        return playlist[:50]
    
    def _order_for_flow(self, songs: List[Song]) -> List[Song]:
        """
        Order songs for smooth listening experience.
        Consider tempo, energy transitions.
        """
        if len(songs) <= 1:
            return songs
        
        ordered = [songs[0]]
        remaining = songs[1:]
        
        while remaining:
            last = ordered[-1]
            # Find song with smallest "distance" in audio space
            best_next = min(
                remaining,
                key=lambda s: self._audio_distance(last, s)
            )
            ordered.append(best_next)
            remaining.remove(best_next)
        
        return ordered
    
    def _audio_distance(self, song1: Song, song2: Song) -> float:
        """
        Calculate distance between songs in audio feature space.
        """
        features = ["tempo", "energy", "danceability", "valence"]
        
        distance = 0
        for f in features:
            diff = song1.audio_features.get(f, 0) - song2.audio_features.get(f, 0)
            distance += diff ** 2
        
        return distance ** 0.5
```

```javascript
    }
    
    async getPersonalizedRecommendations(userId, context = null, limit = 50) {
        const recent = await this.history.getRecentPlays(userId, { days: 30 });
        const candidates = [];
        
        // 1. Collaborative filtering (40%)
        const cfCandidates = await this.collaborativeFiltering(userId, 100);
        candidates.push(...cfCandidates.map(s => ({ song: s, weight: 0.4 })));
        
        // 2. Content-based (30%)
        for (const songId of recent.slice(0, 10)) {
            const similar = await this.contentBasedSimilar(songId, 20);
            candidates.push(...similar.map(s => ({ song: s, weight: 0.3 })));
        }
        
        // 3. Audio feature matching (20%)
        const tasteProfile = await this.getAudioTasteProfile(userId);
        const audioSimilar = await this.audioFeatureMatching(tasteProfile, 50);
        candidates.push(...audioSimilar.map(s => ({ song: s, weight: 0.2 })));
        
        // 4. Discovery (10%)
        const discovery = await this.getDiscoveryTracks(userId, 20);
        candidates.push(...discovery.map(s => ({ song: s, weight: 0.1 })));
        
        // Aggregate scores
        const songScores = new Map();
        for (const { song, weight } of candidates) {
            if (!songScores.has(song.id)) {
                songScores.set(song.id, { song, score: 0 });
            }
            songScores.get(song.id).score += weight;
        }
        
        // Apply context
        if (context) {
            this.applyContextBoost(songScores, context);
        }
        
        // Filter and sort
        const playedIds = new Set(recent);
        const final = [...songScores.values()]
            .filter(item => !playedIds.has(item.song.id))
            .sort((a, b) => b.score - a.score)
            .slice(0, limit);
        
        return final.map(item => item.song);
    }
    
    async generateDailyMix(userId, mixIndex) {
        const clusters = await this.clusterUserTaste(userId, 6);
        
        if (mixIndex > clusters.length) return [];
        
        const cluster = clusters[mixIndex - 1];
        const seeds = cluster.seedTracks.slice(0, 5);
        
        const playlist = [];
        
        // 60% similar to seeds
        for (const seedId of seeds) {
            const similar = await this.contentBasedSimilar(seedId, 5);
            playlist.push(...similar.slice(0, 3));
        }
        
        // 20% audio feature matches
        const audioMatches = await this.audioFeatureMatching(
            cluster.featureCentroid, 
            10
        );
        playlist.push(...audioMatches);
        
        // 20% discovery
        const discovery = await this.getDiscoveryInCluster(
            userId,
            cluster,
            10
        );
        playlist.push(...discovery);
        
        // Order for flow
        return this.orderForFlow(playlist).slice(0, 50);
    }
    
    orderForFlow(songs) {
        if (songs.length <= 1) return songs;
        
        const ordered = [songs[0]];
        const remaining = [...songs.slice(1)];
        
        while (remaining.length > 0) {
            const last = ordered[ordered.length - 1];
            
            let bestIndex = 0;
            let bestDistance = Infinity;
            
            for (let i = 0; i < remaining.length; i++) {
                const distance = this.audioDistance(last, remaining[i]);
                if (distance < bestDistance) {
                    bestDistance = distance;
                    bestIndex = i;
                }
            }
            
            ordered.push(remaining[bestIndex]);
            remaining.splice(bestIndex, 1);
        }
        
        return ordered;
    }
    
    audioDistance(song1, song2) {
        const features = ['tempo', 'energy', 'danceability', 'valence'];
        
        let distance = 0;
        for (const f of features) {
            const diff = (song1.audioFeatures[f] || 0) - 
                        (song2.audioFeatures[f] || 0);
            distance += diff * diff;
        }
        
        return Math.sqrt(distance);
    }
}
```

## Interview Tips

<Tip>
**Key Discussion Points:**

1. **Audio delivery**: How to ensure no buffering? The key insight is that audio files are small compared to video. A 3-minute song at 320kbps is only ~7MB. Prefetching the next 2-3 segments (30 seconds ahead) ensures seamless playback even on variable network conditions. Spotify uses a 10-second segment size and prefetches 3 segments ahead -- meaning ~30 seconds of buffer. This is far less demanding than video streaming.

2. **Personalization**: How to build recommendations? Discuss the multi-signal approach: collaborative filtering (users like you), content-based (similar audio features), and context-aware (time of day, activity). Spotify's Discover Weekly uses a combination of all three, trained on billions of listening events. Mention that "skip" is a stronger negative signal than "not played" -- it means the user actively rejected the song.

3. **Offline mode**: How to handle DRM? Downloaded songs are encrypted with device-specific keys and can only be played within the app. The client must go online every 30 days to verify the subscription is still active and refresh encryption keys. If the subscription lapses, all downloaded content becomes unplayable. This is a licensing requirement, not just a technical choice.

4. **Cross-device sync**: How to sync playback state? Use a centralized session service that stores current song, position, and queue per user. When a user switches devices, the new device fetches the latest session state. Spotify Connect allows controlling playback on one device from another -- this requires a real-time event bus between devices via the server.

5. **Royalty tracking**: How to track plays accurately? A "play" typically counts after 30 seconds of listening (industry standard). This means the analytics pipeline must track play duration, not just play starts. Royalty calculations are per-play, per-territory, and per-rights-holder, making the data pipeline enormously complex. Accuracy here directly affects artist payments -- errors mean financial disputes.
</Tip>

### Key Trade-offs

| Decision | Option A | Option B | Recommendation |
|----------|----------|----------|----------------|
| Audio delivery | Whole-file download | Segment-based streaming | **Segment-based streaming with aggressive prefetching.** A 4-minute song at 320kbps is only ~10MB -- small enough to cache entirely, but segment-based delivery (10-second chunks) enables instant playback start (<200ms) and adaptive quality switching on poor networks. Prefetch 3 segments ahead (~30 seconds of buffer) for gapless playback. The trade-off vs whole-file: slightly more HTTP requests, but the latency-to-first-byte advantage is decisive for user experience. Spotify uses this approach and also pre-downloads the first segment of the next song in the queue while the current song is still playing. |
| Audio codec | OGG Vorbis | AAC / Opus | **OGG Vorbis for desktop, AAC for web, Opus for low-latency.** Spotify uses OGG Vorbis at 96/160/320 kbps tiers because it provides excellent quality-per-bit and has no licensing fees (unlike AAC). AAC is used for the web player because browser support for OGG is inconsistent. Opus is superior to both for low-bitrate scenarios (mobile on 3G) but has less hardware decoder support on older devices. The trade-off: supporting multiple codecs means transcoding each of the 80M songs into 4+ formats, totaling 1.6PB of storage. Storage cost (~$30K/month at S3 rates) is trivial compared to the bandwidth savings from optimal codec selection per device. |
| Recommendation engine | Collaborative filtering only | Multi-signal hybrid | **Multi-signal hybrid.** Collaborative filtering alone fails on cold start (new users, new songs) and creates filter bubbles (recommending only what you already like). Spotify's Discover Weekly blends collaborative filtering (40% weight), content-based audio analysis (30%), neural network audio embeddings (20%), and deliberate exploration (10%). The trade-off: a hybrid system is significantly more complex to build and tune. But a single-strategy approach fails for ~30% of users (new users, eclectic listeners). Mention the 30-second skip signal -- it is the strongest negative feature, far more informative than "not played." |
| Catalog storage | SQL (PostgreSQL) | NoSQL (Cassandra) | **Cassandra for the song catalog, PostgreSQL for user data.** 80M songs with metadata is a read-heavy workload that maps naturally to Cassandra's wide-column model (partition by artist_id, cluster by album and track). Write volume is low (~100K new songs/month) but read volume is enormous (37K song plays/sec, each requiring metadata lookup). PostgreSQL handles user accounts, subscription state, and payment data where ACID transactions matter. The trade-off: Cassandra lacks full-text search, so a separate Elasticsearch cluster handles the search use case. |
| CDN caching | Uniform caching | Popularity-tiered caching | **Tiered caching by popularity.** Tier 1 (top 10K songs, ~20% of plays) -- cached on every edge PoP worldwide. Tier 2 (top 1M songs) -- cached at regional PoPs. Tier 3 (remaining 79M songs) -- cached on-demand at the nearest PoP on first request. The trade-off: the long tail's first play incurs 100-300ms of origin-fetch latency, but subsequent plays at that PoP are cached. An origin shield layer prevents thundering herd: even a globally cache-cold song results in only one origin fetch, not one per PoP. Pre-caching all 80M songs at every PoP would require 1.6PB per PoP -- impractical. |

### Common Candidate Mistakes

```
Mistake 1: Treating audio streaming like video streaming
  Video streaming requires adaptive bitrate switching mid-stream
  because quality differences are visually noticeable. Audio files
  are 10-50x smaller than video. A 320kbps audio stream requires
  only 40KB/sec -- even 3G networks handle this. The challenge is
  not bandwidth; it is latency to first byte and gapless playback
  between songs.

Mistake 2: Not discussing the "30-second rule" for royalties
  Spotify pays royalties only after a user listens for 30+ seconds.
  This means the analytics pipeline must track actual listen
  duration per song, not just "play started" events. A skip at 15
  seconds does not count as a play. This has architectural
  implications: the client must report listen duration, and the
  pipeline must handle deduplication (replaying a song = two plays).

Mistake 3: Ignoring gapless playback
  Users notice even a 100ms gap between songs in a playlist or
  album. Achieving gapless playback requires: prefetching the next
  song before the current one ends, decoding the next song's first
  segment while the current song's last segment is playing, and
  crossfading at the audio level. This is a client-side challenge
  but affects the server design (segment-level access, not
  whole-file downloads).

Mistake 4: Single recommendation strategy
  Candidates who only mention collaborative filtering miss the
  "cold start" problem: new users have no listening history, and
  new songs have no play data. You need multiple strategies:
  content-based (audio features), editorial curation (human-curated
  playlists), demographic defaults (popular in user's country),
  and collaborative filtering that kicks in once you have data.

Mistake 5: Not discussing the catalog update pipeline
  80M songs with frequent additions (100K+ new songs/month). Each
  new song must be: transcoded to multiple quality levels, analyzed
  for audio features, indexed in search (Elasticsearch), and made
  available globally via CDN. This pipeline must complete in under
  an hour for time-sensitive releases (album drops at midnight).
```

  <Accordion title="Q1: How would you design the CDN strategy for audio streaming to support 1 million concurrent streams with no buffering, including handling the 'long tail' of less popular songs?" icon="server">
    **What the interviewer is really testing:** Whether you understand CDN cache economics -- that popular content is cheap to serve (always cached) but the long tail (80M songs where most are played rarely) is where CDN design gets interesting.

    **Strong Answer:**
    - Audio streaming CDN strategy differs from video CDN in one critical way: audio files are tiny (a 4-minute song at 320kbps is about 10MB) compared to video (the same duration at 1080p is 300-500MB). This means you can cache far more content per edge node, and cache misses are faster to fill from origin.
    - **Tiered caching policy based on popularity:** Tier 1 (top 10K songs, ~20% of all plays) -- cache on every edge PoP (Point of Presence) worldwide with long TTLs. These songs represent a disproportionate share of traffic and should always be a cache hit. Tier 2 (top 1M songs) -- cache at regional PoPs, let popularity-based eviction manage the rest. Tier 3 (remaining 79M songs) -- cache on demand at the nearest PoP when requested. First play incurs an origin fetch (add ~100-300ms latency), but subsequent plays at that PoP are cached.
    - **Multi-CDN architecture:** Use at least two CDN providers (e.g., Fastly + Akamai) for redundancy and performance. Route users to the CDN with the lowest latency from their location using DNS-based or anycast routing. If one CDN has an outage in a region, seamlessly failover to the other. Spotify actually uses Google's CDN infrastructure as its primary and supplements with other providers.
    - **Origin shield:** Place a caching layer between the CDN edge and the origin storage (S3). This prevents cache misses from thousands of edge nodes from all hitting S3 simultaneously (the "thundering herd" on origin). The shield caches the long-tail content so that even a globally cache-cold song only results in one origin fetch, not one per PoP that requests it.
    - **Pre-warming for major releases:** When a popular artist drops an album at midnight, millions of users will request the same new songs within minutes. Pre-push these files to all edge PoPs before the release time. This is coordinated with the label's release schedule.
    - **Example:** Spotify processes 3.2 billion song plays per day. Their top 1000 songs account for a huge percentage of plays but a tiny fraction of their catalog. The real engineering challenge is making the 60-millionth most popular song start playing within 200ms for the one user in Indonesia who requests it. The origin shield pattern is what makes this possible without provisioning origin bandwidth for worst-case cache-miss scenarios.

    **Red flag answer:** Treating all 80M songs equally in the caching strategy, or suggesting all audio files should be cached at all edge locations (that would require 1.6PB of storage at every PoP).

    2. How do you handle the signed URL expiration for CDN segments? If a user pauses for 2 hours and then resumes, their segment URLs may have expired. What is the user experience and how does the client handle this?

    **Strong Answer:**
    - Discover Weekly needs to feel personal ("this was made for me") while also introducing new music the user has not heard before. The balance is roughly 80% familiar-ish territory (artists/genres the user already likes) and 20% true discovery (new artists/genres they might like but have not explored).

  <Accordion title="Q3: How do you synchronize playlist state across multiple devices and handle concurrent edits to collaborative playlists?" icon="arrows-rotate">
    **What the interviewer is really testing:** Whether you understand distributed state synchronization, conflict resolution strategies, and the specific challenges of ordered-list data structures under concurrent modification.

    **Strong Answer:**
    - Playlist sync has two distinct problems: (1) **Single-user, multi-device sync** -- the user adds a song on their phone and expects to see it on their desktop. (2) **Multi-user collaborative playlists** -- two users add different songs at the same time, possibly at the same position in the list.
    - For single-user sync: use a centralized playlist service with a version counter. Each device fetches the playlist state with a version number. When a device makes a change, it sends the mutation (e.g., "insert song X at position 5") along with the version it is based on. If the version matches the server's current version, the mutation is applied and the version is incremented. If not, the device must re-fetch the latest state and rebase its local change. This is optimistic concurrency control.
    - For collaborative playlists, the same pattern works but conflicts are more likely. Two users adding songs at the same position simultaneously creates a conflict. Resolution strategies: (a) **Last-writer-wins** -- simplest, but the user whose write arrives second silently "loses" their intended position. (b) **Operational Transformation (OT)** -- transform concurrent operations so both can be applied. If User A inserts at position 5 and User B inserts at position 5, transform B's operation to position 6 so both songs appear in the list. (c) **CRDTs (Conflict-free Replicated Data Types)** -- model the playlist as an ordered set where inserts are always mergeable without conflicts. This is more complex to implement but guarantees convergence.
    - In practice, playlists are low-conflict workloads (users rarely edit the same playlist at the exact same second), so a simpler approach works: use server-authoritative state with operational messages. Each edit is an operation (`ADD song_id at position`, `REMOVE song_id`, `REORDER song_id to position`). The server applies operations sequentially and broadcasts the updated state to all subscribed devices via WebSocket or server-sent events.
    - **Example:** Spotify Connect allows real-time control of playback across devices. When you switch from your phone to your smart speaker, the phone sends the current playback state (song, position, queue) to the server, and the speaker fetches it. This uses a pub/sub mechanism -- each device subscribes to the user's session channel, and state changes are broadcast in real-time. For collaborative playlists, Spotify uses a simpler model: edits are serialized on the server, and all collaborators see the same order.

    **Red flag answer:** Suggesting each device maintains its own playlist state and you "merge" periodically (this is a recipe for data loss), or not addressing what happens when two people edit a collaborative playlist simultaneously.

    **Follow-ups:**

    2. Spotify shows a "listening activity" feed where you can see what friends are playing in real-time. How would you design this feature to scale to 400M users without overwhelming the server with position-update events?

  <Accordion title="Q4: Design the offline playback feature. How do you balance DRM requirements, storage constraints on the device, and the need to verify subscription status?" icon="cloud-arrow-down">
    **What the interviewer is really testing:** Whether you understand that offline playback is a DRM and licensing problem as much as a technical one, and that the constraints come from music label contracts, not just engineering preferences.

    **Strong Answer:**
    - Offline playback requires downloading encrypted audio files to the device so they can be played without a network connection, while ensuring that (a) the files cannot be extracted and pirated, (b) playback stops if the subscription lapses, and (c) royalties are still tracked accurately.
    - **Encryption:** Each downloaded file is encrypted with a device-specific key derived from the device ID and a server-side secret. The key is stored in the platform's secure enclave (iOS Keychain, Android Keystore). The audio can only be decrypted by the Spotify app on that specific device. If the user copies the encrypted file to another device, it is unplayable.
    - **Key rotation and subscription verification:** The device must go online at least once every 30 days. During this "check-in," the client: (a) verifies the subscription is still active, (b) receives a refreshed decryption key or key-derivation token, (c) syncs play events that accumulated offline (for royalty tracking). If the subscription has lapsed, the server does not refresh the key, and the locally stored key expires. All downloaded content becomes unplayable. This 30-day window is typically a contractual requirement from music labels.
    - **Storage management:** Devices have limited storage. The download manager must: (a) download only the user's preferred quality level (not all four), (b) allow the user to set a maximum storage limit (e.g., 5GB), (c) automatically evict least-recently-played downloads when the limit is reached, (d) support download-over-WiFi-only to prevent cellular data charges.
    - **Offline play tracking:** When the user plays a downloaded song offline, the client logs the play event locally (song ID, timestamp, duration listened). When the device reconnects, these events are batched and sent to the analytics pipeline for royalty calculation. The 30-second rule still applies -- only plays over 30 seconds count. Deduplication logic on the server prevents replayed events from inflating play counts.
    - **Example:** Spotify allows premium users to download up to 10,000 songs on each of up to 5 devices. The 30-day online check-in is enforced not because of a technical limitation but because label contracts require periodic verification that the user is still paying. Netflix uses a similar pattern for offline video -- downloads expire after a window, and the app must phone home to refresh them.

    **Red flag answer:** Describing offline playback as simply "download the MP3 to the device" without addressing encryption, DRM, subscription verification, or royalty tracking.

    **What the interviewer is really testing:** Whether you understand that royalty tracking is a financial data pipeline with strict accuracy requirements -- errors mean artists get paid incorrectly, which leads to lawsuits and lost label partnerships.

    **Strong Answer:**
    - A "play" in the music industry counts only after 30 seconds of continuous listening. A skip at 15 seconds does not count. Replaying a song counts as a new play. This means the client must report actual listen duration, not just "play started" events.
    - **Client-side reporting:** The client sends play events with: `user_id`, `song_id`, `timestamp_start`, `duration_listened_ms`, `context` (playlist, album, radio, search), `device_type`, `country`, and `skipped` (boolean). These are batched and sent every 30-60 seconds during active use, and on app background/close.
    - **Pipeline architecture:** Play events flow into Kafka (partitioned by user ID for ordering guarantees), then into a stream processing layer (Flink or Spark Streaming) that: (a) Filters out plays under 30 seconds. (b) Deduplicates events (same user + song + timestamp within a window = one play). (c) Enriches with rights-holder data (which label/publisher/distributor owns this song in this territory). (d) Writes validated play records to a data warehouse (e.g., BigQuery or a custom data lake).
    - **Royalty calculation complexity:** Payment is not simply "plays x rate." It is a pool-based model: Spotify's total subscription revenue in a given period is divided among rights holders based on their share of total plays. This means you need accurate *global* play counts, broken down by territory and rights holder. A song may have different rights holders in different countries (the US distributor may differ from the EU distributor). The calculation runs monthly and generates payment reports for thousands of rights holders.
    - **Accuracy requirements:** Even a 0.1% error rate on 3.2 billion daily plays means 3.2 million misattributed plays per day. At scale, this creates financial discrepancies that labels will dispute. The pipeline needs end-to-end exactly-once semantics (using idempotent writes and deduplication) and periodic reconciliation against raw event logs.
    - **Example:** Spotify has faced multiple lawsuits over royalty calculation accuracy. Their engineering blog describes a system that processes 600+ billion events per month through a multi-stage pipeline with checkpointing and reconciliation at each stage. They re-run historical calculations when bugs are discovered and issue corrections to rights holders -- this is called a "true-up" in the industry.

    **Red flag answer:** Only tracking "play started" events without considering duration, or not understanding that royalty attribution is per-territory and per-rights-holder.

  <Accordion title="Q6: How does Spotify achieve gapless, seamless playback between songs, and what are the client-server coordination challenges?" icon="forward">
    **What the interviewer is really testing:** Whether you understand that gapless playback is a client-side audio engineering problem with server-side prerequisites, and that users are extremely sensitive to even tiny audio gaps.

    **Strong Answer:**
    - Gapless playback means there is zero perceivable silence or click between consecutive songs in a playlist or album. Users notice gaps as short as 50ms, especially in live albums, classical music, or DJ mixes where songs are meant to flow into each other.
    - **Client-side technique:** The client maintains two audio decoder instances. While Song A is playing, the client pre-fetches and begins decoding Song B's first segments in the background. When Song A reaches its final audio samples, the client crossfades to Song B's first samples at the audio buffer level (not the file level). This requires the client to know the exact sample count of Song A so it can seamlessly append Song B's samples.
    - **Server-side prerequisites:** (1) **Segment-based access:** Songs are stored as segments (e.g., 10-second chunks), and the client can request specific segments by index. The client requests Song B's first 2-3 segments when Song A reaches its last 30 seconds. (2) **Metadata about next track:** The server must know the user's queue/playlist so it can tell the client what song comes next well before Song A ends. The playback session includes the current queue, and the server pre-computes signed CDN URLs for the next song's initial segments and returns them with the current song's segment response. (3) **Audio normalization:** Songs from different albums have different loudness levels. Without normalization, the transition is jarring. Spotify pre-computes a "ReplayGain" value for each song during ingestion and applies loudness normalization on the client before playback.
    - **Crossfade vs. gapless:** Crossfade (Song A fades out while Song B fades in, overlapping for 1-5 seconds) is a user preference. True gapless means Song B starts at full volume the instant Song A's last sample ends. Both require prefetching, but crossfade also requires parallel decoding and mixing.
    - **Edge cases:** (a) User skips rapidly through songs -- the prefetch logic must cancel pending requests and start fresh for the new song. (b) Network drops during prefetch of the next song -- the client must handle gracefully (brief buffer spinner rather than a hard stop). (c) Shuffle mode means the "next song" is not known until the current song is near its end, which compresses the prefetch window.
    - **Example:** Spotify's client uses a custom audio pipeline (not the platform's default media player) precisely to control crossfading and gapless playback at the sample level. Their mobile clients decode audio in a background thread and feed samples to the audio output buffer, allowing them to splice Song A and Song B's waveforms with no gap. Apple Music initially launched without gapless playback and received significant user complaints, which shows how critical this feature is.

    **Red flag answer:** Suggesting you simply download the entire next song before the current one ends (wastes bandwidth, delays playback start, and does not solve the audio-level gap problem), or not understanding the difference between file-level and sample-level transitions.

    **Follow-ups:**
    1. The user is on a slow 3G connection and the prefetch of the next song's segments fails. Song A is about to end. What does the client do -- pause with a spinner, play silence, or skip to whatever is available?
    2. How do you handle gapless playback for "continuous mixes" (DJ sets or concept albums where songs truly have no gap in the original recording)? Is segment-based delivery compatible with this, or do you need special handling?
  </Accordion>

  <Accordion title="Q7: How would you design the rights management system that determines which songs are available in which countries, and how does this affect the overall architecture?" icon="scale-balanced">
    **What the interviewer is really testing:** Whether you understand that music licensing is territorial, that the catalog is not uniform across countries, and that this creates pervasive complexity across search, playback, recommendations, and offline features.

    **Strong Answer:**
    - Music rights are licensed per territory (country or region). A song available in the US may not be available in Germany if Spotify does not have a license agreement with the rights holder in that territory. This means the "catalog" is not a single global entity -- it is 180+ territory-specific catalogs that overlap significantly but differ at the edges.
    - **Data model:** Each song has an `availability` map: a set of territory codes where it is currently licensed. This is stored alongside the song metadata (in Cassandra). When a user plays, searches, or browses, the system must filter results to include only songs available in the user's territory. The user's territory is determined by their account registration country (not their current IP), though Spotify allows temporary access when traveling.
    - **Impact on search:** Elasticsearch must filter search results by territory. This means either: (a) indexing a `territories` field on each document and adding a filter clause to every search query, or (b) maintaining separate search indexes per territory (more storage, but faster queries). In practice, option (a) with a filter is sufficient because Elasticsearch handles filter clauses efficiently via bitsets.
    - **Impact on recommendations:** The recommendation engine must not suggest songs that are unavailable in the user's territory. This filtering happens at the final ranking stage, after candidate generation. If collaborative filtering suggests a song popular in Japan to a US user but it is not licensed in the US, it is silently dropped.
    - **Impact on playlists:** A collaborative playlist created by a user in the US may contain songs unavailable in India. When an Indian user opens this playlist, unavailable songs must be marked as grayed-out/unplayable. The playlist still shows them (to preserve the creator's intent) but playback is blocked.
    - **Rights changes over time:** Licenses expire and are renegotiated. When a song's rights change (added or removed from a territory), the system must: (a) update the metadata store, (b) invalidate relevant search index entries, (c) invalidate CDN caches for affected territories, (d) handle in-progress streams (usually allowed to finish, but no new plays). This requires an event-driven pipeline triggered by rights-change events from the licensing team's management system.
    - **Example:** In 2023, Spotify temporarily lost Universal Music Group songs in several countries during a licensing dispute. Within hours, millions of songs had to be made unplayable in specific territories. This required their rights management pipeline to bulk-update availability maps and propagate changes to search, playback, and offline systems. Songs already downloaded for offline playback became unplayable at the next subscription check-in.

    **Red flag answer:** Assuming the entire 80M song catalog is available everywhere, or treating rights as a one-time configuration rather than a continuously changing state that permeates every system.

    1. A user in the US downloads 200 songs for offline playback, then travels to a country where 50 of those songs are not licensed. Should those 50 songs be playable offline during travel? How does Spotify's "travel grace period" work architecturally?
    2. How would you handle a scenario where a song's rights change mid-stream -- the user started playing it when it was available, but the license expired during the 4-minute song? Do you cut off playback or let it finish?

  <Accordion title="Q8: How does Spotify handle adaptive bitrate switching during audio playback, and why is the approach different from video adaptive streaming (like HLS/DASH)?" icon="signal">
    **What the interviewer is really testing:** Whether you understand why audio streaming is fundamentally different from video streaming and can articulate the specific engineering trade-offs for audio.

    **Strong Answer:**
    - Video adaptive streaming (HLS/DASH) continuously switches between quality levels every few seconds based on measured bandwidth, because video quality differences are immediately visible (pixelation, resolution drops). Audio streaming does not need this level of granularity because: (a) audio bitrates are 10-50x lower than video (320kbps vs. 5-15Mbps), so even poor networks can usually sustain the highest audio quality, and (b) mid-song quality switching in audio is more noticeable (artifacts at the switch point) and more annoying to listeners than a brief video quality drop.
    - **Spotify's approach:** Quality is selected once at playback start based on: (a) user's quality preference setting (Low/Normal/High/Very High), (b) current network type (WiFi vs. cellular), (c) subscription tier (free users capped at 160kbps, premium up to 320kbps). The quality generally does not change mid-song. If network conditions degrade severely during playback, the client can switch quality at a segment boundary (every 10 seconds) to prevent buffering, but this is a last resort, not continuous adaptation.
    - **Buffering strategy:** Because audio files are small, the client can aggressively prefetch. At 320kbps, 30 seconds of audio is about 1.2MB. Prefetching 3 segments (30 seconds ahead) means the client has a 30-second buffer. Even if the network drops entirely for 25 seconds, playback continues without interruption. Video cannot do this because 30 seconds of 1080p video is 50-100MB.
    - **When adaptation matters:** The edge cases where audio bitrate switching matters are: (a) users transitioning from WiFi to cellular (e.g., leaving home), (b) users in areas with fluctuating coverage (trains, subways), (c) users on very slow connections (2G, satellite). For these cases, the client monitors download speed and, if it cannot sustain the current bitrate, it drops to a lower quality at the next segment boundary.
    - **Client-side buffering metrics:** The client tracks: current buffer depth (seconds of audio already downloaded), download throughput (rolling average over last N seconds), and estimated time to buffer underrun. If the estimated underrun is less than 5 seconds away, it triggers a quality switch.
    - **Example:** Spotify's early architecture actually downloaded entire songs as single files (not segments). They moved to segment-based delivery to enable mid-song quality switching, faster start times (start playing after the first segment downloads, not the entire song), and better CDN cache efficiency (popular song segments are cached independently). However, they still prefer to deliver entire songs at a consistent quality when bandwidth permits, unlike video where constant adaptation is the norm.

    **Red flag answer:** Describing HLS/DASH video adaptive streaming and applying it identically to audio, or suggesting you need the same level of continuous quality switching for a 320kbps audio stream as for a 10Mbps video stream.

    **Follow-ups:**
    1. A user starts playing a song on a fast WiFi connection at 320kbps, then walks out of range and the connection drops to 50kbps. The client has 15 seconds of buffer remaining. What is the optimal strategy -- switch to 96kbps immediately, or drain the buffer and switch only if needed?
    2. How do you handle the transition between WiFi and cellular seamlessly? There is often a brief period of no connectivity during the handoff. How does the buffer strategy protect against this?
  </Accordion>
